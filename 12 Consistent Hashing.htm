Consistent Hashing

How to distribute keys across a changing set of nodes so that adding or removing a node remaps only k/n keys instead of breaking every mapping.

The bottleneck Finite capacity. spreading keys across nodes that come and go costs a hashing scheme that limits reshuffling.

In this lesson

  • What is Consistent Hashing?
  • How it works?

Distributed Hash Table (DHT) is one of the fundamental component used in distributed scalable systems. Hash Tables need key, value and a hash function, where hash function maps the key to a location where the value is stored.

Suppose we are designing a distributed caching system. Given ‘n’ cache servers, an intuitive hash function would be ‘key % n’. It is simple and commonly used. But it has two major drawbacks:

  1. It is NOT horizontally scalable. Whenever a new cache host is added to the system, all existing mappings are broken. It will be a pain point in maintenance if the caching system contains lots of data. Practically it becomes difficult to schedule a downtime to update all caching mappings.
  2. It may NOT be load balanced, especially for non-uniformly distributed data. In practice, it can be easily assumed that the data will not be distributed uniformly. For the caching system, it translates into some caches becoming hot and saturated while the others idle and almost empty.

In such situations, consistent hashing is a good way to improve the caching system.

What is Consistent Hashing?

Consistent hashing is a very useful strategy for distributed caching system and DHTs. It allows distributing data across a cluster in such a way that will minimize reorganization when nodes are added or removed. Hence, making the caching system easier to scale up or scale down.

In Consistent Hashing when the hash table is resized (e.g. a new cache host is added to the system), only k/n keys need to be remapped, where k is the total number of keys and n is the total number of servers. Recall that in a caching system using the ‘mod’ as the hash function, all keys need to be remapped.

In consistent hashing objects are mapped to the same host if possible. When a host is removed from the system, the objects on that host are shared by other hosts; and when a new host is added, it takes its share from a few hosts without touching other’s shares.

How it works?

As a typical hash function, consistent hashing maps a key to an integer. Suppose the output of the hash function is in the range of [0, 256). Imagine that the integers in the range are placed on a ring such that the values are wrapped around.

Here’s how consistent hashing works:

  1. Given a list of cache servers, hash them to integers in the range.
  2. To map a key to a server,
    • Hash it to a single integer.
    • Move clockwise on the ring until finding the first cache it encounters.
    • That cache is the one that contains the key. See animation below as an example: key1 maps to cache A; key2 maps to cache C.

To add a new server, say D, keys that were originally residing at C will be split. Some of them will be shifted to D, while other keys will not be touched.

To remove a cache or if a cache failed, say A, all keys that were originally mapping to A will fall into B, and only those keys need to be moved to B, other keys will not be affected.

For load balancing, as we discussed in the beginning, the real data is essentially randomly distributed and thus may not be uniform. It may make the keys on caches unbalanced.

To handle this issue, we add “virtual replicas” for caches. Instead of mapping each cache to a single point on the ring, we map it to multiple points on the ring, i.e. replicas. This way, each cache is associated with multiple portions of the ring.

If the hash function is “mixes well,” as the number of replicas increases, the keys will be more balanced.

Principal-level deep dive

Virtual nodes, bounded loads, and real-world usage

The classic ring picture — hash each node to one point on a 0..232-1 circle, walk clockwise to find the owner — is the right mental model but a misleading production model. With N physical nodes placed at N random points, the gaps between consecutive points follow an exponential-like distribution. The largest arc is on average O(log N / N) of the ring while the expected arc is 1/N, so the heaviest node can own several times the mean. With one point per node you see load imbalance on the order of ±30–40%, and removing a node dumps its entire arc onto a single successor (a hot-spot, not a smear).

Virtual nodes (vnodes): smoothing by oversampling

The fix is to place each physical node at many points. If each of N machines owns V tokens (vnodes), the ring has N·V points and each physical node's total load is the sum of V independent arcs. By concentration, the relative standard deviation of per-node load shrinks as ~1/√V. The classic result: with V ≈ 100–200 vnodes per node you bound the imbalance to roughly ±5%.

  • Heterogeneity for free: a machine with 2× the capacity simply gets 2× the tokens. Weighting is just token-count assignment.
  • Smooth rebalancing: removing a node redistributes its V arcs across V different successors instead of one, so no single neighbor inherits the whole load. This is the property that makes online membership changes survivable.
  • Cost: a larger token map to store and binary-search, and more metadata to gossip. Cassandra historically defaulted to num_tokens=256, then lowered the recommendation to 16 (with allocation-aware token selection) precisely because high V hurt streaming, repair, and availability math.
Why V trades against availabilityWith more vnodes per node, any given node shares replica responsibility with more peers. Past a point, a multi-node failure is more likely to take out all replicas of some token range. This is why "more vnodes is always better" is wrong — token allocation strategy matters as much as the count.

Consistent hashing with bounded loads (Google, 2017)

Vnodes reduce variance but assume keys are uniform. Real workloads have hot keys and churn, so a node can still exceed capacity. Consistent Hashing with Bounded Loads (Mirrokni, Thorup, Zadimoghaddam) adds a hard cap: given total load L and n bins, set capacity c = ⌈(1+ε)·L/n⌉. A key hashes to its ideal node as usual, but if that node is already at capacity, you walk forward to the next node with spare room. The theorem guarantees no node exceeds (1+ε) times the average while the expected number of keys that need relocation on an insert/delete is O(1/ε²) — so a small ε (say 0.25) buys a tight cap at modest movement cost. Vimeo's groupcache-style balancer and HAProxy's bounded-load mode use exactly this.

Rendezvous (HRW) hashing — the no-ring alternative

Highest Random Weight hashing skips the ring entirely. For key k, compute w(k, s) = hash(k, s) for every server s and pick the argmax. Properties:

  • Minimal disruption: adding/removing a server only reassigns the keys for which that server was (or becomes) the max — exactly 1/n of keys move, the same guarantee as consistent hashing.
  • No vnodes needed for balance; the max-of-uniform distribution is naturally even. Weighting uses the WRH formula -weight / ln(uniform_hash).
  • Cost is O(n) per lookup vs. O(log n) for a sorted ring — fine for tens/hundreds of backends (CARP, many CDN/cache tiers), worse for thousands. Skeleton/hierarchical HRW restores O(log n).

Replica placement and the preference list

In Dynamo/Cassandra, the coordinator for key k is its ring successor, and the next RF-1 distinct physical nodes walking clockwise form the preference list. Two subtleties a principal must get right: (1) skip vnodes that map back to a physical node already in the list, so all RF replicas live on distinct machines; (2) make the preference list rack/AZ-aware (Cassandra's NetworkTopologyStrategy) so replicas don't all land in one failure domain. During a transient failure, writes go to the next healthy node as a hinted handoff and are replayed when the owner returns — this is the availability lever, paired with anti-entropy repair to fix what hints miss.

Jump consistent hash — when you never delete arbitrary nodes

Google's jump hash (Lamping & Veach) maps (key, num_buckets) to a bucket in ~ln(num_buckets) arithmetic steps with zero storage and near-perfect balance. The catch: it only supports buckets numbered 0..n-1, so you can shrink/grow at the tail but cannot remove bucket 17 out of 100 without remapping. That makes it ideal for sharded storage where shard count changes by resizing the whole set (e.g., a fixed shard map, or a stateless cache where you accept reshuffling), and unsuitable for a fluid membership of named, independently-failing servers.

Where each actually ships

  • Dynamo / Cassandra / Riak / ScyllaDB: ring + vnodes + preference-list replication, gossip-propagated token map.
  • Memcached / Redis clients: client-side consistent hashing (ketama uses 160 points per server with MD5) so adding a cache node invalidates only 1/n of entries instead of the whole cache. Redis Cluster itself uses 16384 fixed hash slots — explicit slot assignment rather than a hash ring, which is effectively bounded, manually-rebalanced sharding.
  • Load balancers: HAProxy/Envoy/Maglev. Google's Maglev uses a permutation-based lookup table for even split and connection stickiness; Envoy ships both ring-hash and Maglev plus the bounded-load cap.
  • CDN request routing: consistent hashing on the cache-key keeps a given object pinned to the same edge cache to maximize hit rate while tolerating PoP changes.

Mastery check

  • With plain hash(key) % N, why does bumping N from 4 to 5 remap roughly 4/5 of all keys instead of just 1/5 — and why is that catastrophic for a cache cluster (thundering-herd misses to the origin)?
  • Why does the basic hash ring only move K/N keys when a node joins or leaves, and exactly which keys move (those between the new node and its predecessor on the ring)?
  • Without virtual nodes, why does a single physical node added to a 3-node ring tend to land badly (one node can own ~50% of the keyspace), and how do 100–200 vnodes per server drive the variance down toward an even split?
  • How does the bounded-load variant (Mendelson/Vahdat — used by Google's load balancers) cap any node at (1+ε)·average by walking forward on the ring to the next under-capacity node, and what does it trade away (some keys no longer land on their "natural" owner)?
  • Why does Dynamo/Cassandra layer a replication factor on top of the ring by walking the next RF distinct physical nodes clockwise, rather than just storing one copy at the primary owner?
  • Can you explain why Maglev hashing (permutation lookup table) is preferred over a literal ring for L4 load balancers — constant-time O(1) lookup, near-perfect even split, and minimal disruption when a backend drops?
  • When would you reach for rendezvous (HRW) hashing instead of a ring, and why does it avoid storing/maintaining ring state at the cost of O(N) hashing per lookup?
