Consensus and Raft

How a set of unreliable machines agrees on a single value — the foundation under every replicated database, lock service, and config store.

The consensus problem

Consensus is the problem of getting a group of processes to agree on a single value, even when some of those processes crash, messages are delayed or reordered, and the network partitions. It is the hardest and most fundamental problem in distributed systems, because almost every other coordination primitive — leader election, distributed locks, atomic commit, replicated state machines, group membership — reduces to it.

A correct consensus algorithm must satisfy four properties:

  • Agreement: no two correct nodes decide different values.
  • Validity (integrity): the decided value was proposed by some node.
  • Termination (liveness): every correct node eventually decides.
  • Stability: once a node decides, it never changes its decision.

The replicated state machine model is the reason this matters in practice: if every replica starts in the same state and applies the same sequence of deterministic commands in the same order, every replica ends in the same state. Consensus is precisely how the cluster agrees on that command order. A consensus-backed log is therefore the spine of systems like etcd, ZooKeeper, Consul, CockroachDB, and Spanner.

FLP impossibility and why we relax it

The FLP result (Fischer, Lynch, Paterson, 1985) proves that in a fully asynchronous system — no bound on message delay — no deterministic consensus algorithm can guarantee termination if even a single process may crash. The problem is that you cannot distinguish a crashed node from a merely slow one.

Real systems escape FLP not by violating it but by weakening the assumptions:

  • Partial synchrony: assume the network is "eventually" timely. Raft and Paxos use timeouts — they remain safe always, and become live once the network behaves.
  • Randomization: algorithms like Ben-Or terminate with probability 1.
  • Failure detectors: an external oracle that eventually identifies crashed nodes.
Key insightThe practical lesson: production consensus algorithms never sacrifice safety (agreement) for liveness. During a partition they will stall rather than risk a split-brain. This is the CAP trade-off made concrete — consensus systems choose CP. See CAP Theorem and Consistency Models & PACELC.

Raft: designed for understandability

Raft (Ongaro & Ousterhout, 2014) deliberately decomposes consensus into three sub-problems so it can be reasoned about and implemented correctly. Every node is in one of three states: follower, candidate, or leader. Time is divided into terms, each beginning with an election; a term is a logical clock that lets nodes detect stale leaders.

1. Leader election

Followers expect periodic heartbeats from the leader. If a follower's randomized election timeout (e.g. 150–300 ms) elapses without a heartbeat, it increments the term, becomes an engineer, votes for itself, and requests votes from peers. A candidate wins if it gathers votes from a majority. Randomized timeouts make split votes rare and self-correcting.

2. Log replication

Clients send commands to the leader. The leader appends the command to its log and sends AppendEntries RPCs to followers. Once a majority have persisted the entry, the leader marks it committed, applies it to its state machine, and returns to the client. Each entry carries the term in which it was created, which is the basis of the safety argument.

3. Safety

The crucial guarantee is the Leader Completeness property: a leader for a given term contains all entries committed in previous terms. Raft enforces this with the election restriction — a node will only grant its vote to an engineer whose log is at least as up-to-date as its own. This means a node missing committed entries can never win an election, so committed entries are never lost.

// Simplified AppendEntries consistency check on a follower
if (args.prevLogTerm != log[args.prevLogIndex].term) {
    return REJECT;          // logs diverged; leader will back up and retry
}
appendOrOverwrite(args.entries);
if (args.leaderCommit > commitIndex) {
    commitIndex = min(args.leaderCommit, lastNewEntryIndex);
}

Raft also handles membership changes safely via joint consensus (Cnew,old overlap so no two disjoint majorities can form) and bounds log growth with snapshotting — the state machine is serialized and the log prefix discarded.

Trade-offRaft funnels all writes through one leader. This makes it easy to reason about but caps write throughput at one node and adds a leader-election stall (a few hundred ms) on failover. Systems needing higher write parallelism shard the keyspace and run one Raft group per shard (CockroachDB, TiKV).

Paxos and Multi-Paxos

Paxos (Lamport) is the original and most-studied consensus algorithm. Single-decree Paxos agrees on one value through two phases — prepare/promise and accept/accepted — mediated by proposers, acceptors, and learners. A proposal number imposes a total order on attempts; an acceptor promises not to accept anything older.

Multi-Paxos amortizes the cost for a stream of values by electing a stable leader so phase 1 is skipped for steady-state operations — the same optimization Raft bakes in by design. Paxos is famously hard to implement correctly, which is exactly why Raft was created. Functionally they are equivalent; Raft trades a small amount of generality for a large amount of clarity. EPaxos and Flexible Paxos are notable variants that relax the leader bottleneck or the quorum-intersection requirement.

Quorums and the R + W > N rule

Consensus relies on overlapping majorities. With N replicas, any two majorities of size ⌊N/2⌋ + 1 must share at least one node, which guarantees that a new leader sees the latest committed write. The same intersection logic generalizes to quorum reads/writes in Dynamo-style systems: if a write must reach W replicas and a read must reach R, then R + W > N guarantees the read set overlaps the write set, giving strong consistency for that key.

ConfigurationEffect
W = N, R = 1Fast reads, slow/fragile writes (any replica down blocks writes)
W = 1, R = NFast writes, slow reads; good for write-heavy logs
W = R = ⌊N/2⌋+1Balanced, tolerates ⌊N/2⌋ failures — the usual choice

Why an odd N? Five nodes tolerate two failures; six also tolerate only two but cost more — even sizes buy no extra fault tolerance while widening the quorum.

Where it shows up in practice

  • etcd / Consul — Raft-backed KV stores for Kubernetes config and service discovery.
  • ZooKeeper — uses ZAB, a consensus protocol closely related to Multi-Paxos. See Leader Election & Coordination.
  • Spanner / CockroachDB / TiKV — one Raft/Paxos group per data range for horizontally scalable strong consistency.
  • Kafka — KRaft mode replaced the ZooKeeper dependency with an internal Raft quorum. See Kafka & Log-based Messaging.

Principal-level mastery check

What strong engineers doThey reach for consensus only where it is actually required — metadata, leader election, config, small critical state — and keep it off the high-throughput data path. They know consensus is expensive (a majority round-trip per decision) and bound its blast radius by sharding into many small groups.

Probing questions to ask engineers:

  • "You said you'll use a leader. What happens to in-flight writes during failover, and how long is the unavailability window?"
  • "Why is the cluster size 3 or 5 and not 4 or 6?"
  • "Your design puts every user write through Raft. What's the throughput ceiling, and how would you raise it?"
  • "How does a newly elected leader know it has all committed entries?"
