Designing a Distributed Locking Service

Sometimes exactly one process must do a thing at a time — run a leader, touch a shared resource, avoid a double-charge. A distributed lock coordinates that across machines that can crash, pause, and partition. It looks simple ("acquire, do work, release") and is one of the most subtly dangerous things to build — because a held lock never truly guarantees exclusivity, which is the heart of the famous Kleppmann–antirez debate.

The crux Mutual exclusion across unreliable machines — in exchange for the truth that no lock alone guarantees exclusivity, so correctness needs fencing tokens enforced by the protected resource. A lock service lets one process proceed while others wait. But a holder can be paused or partitioned and lose the lock without knowing — so you pay by requiring a fencing token the resource checks, and by accepting that "I hold the lock" is never enough on its own.

1. Requirements

Functional: acquire(lockName) grants the lock to at most one client; release frees it; locks auto-expire if the holder dies (no permanent deadlock); support fencing tokens for correctness-critical use; ideally fair (FIFO) waiting.

Non-functional: safety — at most one holder (or, more precisely, at most one effective holder, see sub-problem 1); liveness (a crashed holder's lock is eventually released); highly available (the lock service itself isn't a SPOF); low latency (acquire/release are fast); correct under crashes, pauses, and partitions.

2. Capacity estimate

Lock services are about correctness, not throughput — but the numbers still shape the design:

  • Lock rate: say tens of thousands of acquire/release per second across the system. Consequence: modest — a small consensus cluster (3–5 nodes) handles it; lock services don't need to scale to millions of QPS, they need to be correct.
  • Cluster size: the lock store must survive node failure without two holders, so it's consensus-backed (3 or 5 nodes, majority quorum). Consequence: every lock grant is a consensus write — a majority round-trip — which bounds latency and is why you keep the cluster small and don't put locks on ultra-hot paths.
  • The real constraint is correctness under failure: the hard part isn't load, it's that holders pause (GC), clocks drift, and networks partition — so the design is dominated by safety under adverse timing, not capacity. Consequence: spend the design budget on fencing and lease semantics, not sharding.

3. The API

acquire(lockName, ttl) -> { acquired: bool, fencingToken: int }   // token is monotonic
renew(lockName, fencingToken, ttl) -> bool                         // extend the lease
release(lockName, fencingToken) -> bool

The fencing token — a monotonically increasing number returned on each acquisition — is the most important part of the API and the thing most naive lock services omit. The client must pass it to the protected resource on every operation; the resource rejects stale tokens (sub-problem 1). ttl makes the lock a lease that auto-expires if not renewed.

4. High-level design

The lock store is a small, strongly-consistent, consensus-backed key-value service (etcd / ZooKeeper / Chubby). acquire(lockName) is an atomic create-if-not-exists on a key, replicated via consensus so only one client can win even across failures — the majority quorum guarantees a single winner. The key holds the owner and a lease (TTL); the holder renews (heartbeats) to keep it. If the holder stops renewing (crash/pause), the lease expires and the lock becomes acquirable again. Each acquisition returns a fencing token (the key's monotonically increasing version), which the holder presents to the protected resource. The consensus backing is what makes the store itself safe (no split-brain granting two holders).

5. Lock model — lease + owner + fencing token

The fencing token is typically the key's version/revision number from the consensus store (etcd's mod-revision, ZooKeeper's zxid) — guaranteed to increase with each change, so a later acquisition always has a strictly higher token than an earlier one.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the GC pause that broke mutual exclusion (the Kleppmann argument) Client A acquires the lock and starts writing to the shared resource. Mid-operation, A suffers a long GC pause. Its lease expires; client B legitimately acquires the lock and writes. Then A wakes up — still believing it holds the lock — and writes too. Two writers, corruption. What single mechanism makes A's stale write impossible, and why does no timeout value fix this?
Scenario 2 — the holder crashed and the lock is stuck forever A client acquires a lock and crashes before releasing it. With a naive lock (acquire/release, no expiry), the lock is held forever — every other client waits indefinitely. How do you ensure a dead holder's lock is freed, and what new risk does that introduce?
Scenario 3 — Redlock: is a multi-Redis lock safe? (the real debate) A popular pattern (Redlock) acquires a lock across several independent Redis nodes for "high availability." Martin Kleppmann argued it's unsafe for correctness; antirez (Redis's creator) rebutted. What's the disagreement, and when is a Redis-based lock acceptable vs dangerous?

Scenario 1 is the held lock is never truly exclusive (fencing). Scenario 2 is leases & liveness. Scenario 3 is the lock store's correctness. Each maps to a sub-problem.

The four hard sub-problems

The acquire/release flow is trivial; the correctness is brutal. These four are why distributed locks are dangerous — each with a nested failure.

Sub-problem 1 — A held lock is never truly exclusive

The fundamental truth: between the moment a client checks "do I hold the lock?" and the moment it acts on the resource, it can be paused (GC), descheduled, or partitioned for arbitrarily long — long enough for its lease to expire and another client to acquire. So a client can believe it holds the lock when it no longer does. No lock service can prevent this, because it can't control the holder's pauses. The only robust defense is a fencing token: the resource itself enforces ordering.

The failure that happens when you get this wrong — the zombie holder This is Scenario 1, and the core of the Kleppmann argument. A paused holder (A, token 33) wakes after its lease expired and B (token 34) already acquired and wrote — A then writes with its stale belief, corrupting data. No timeout value fixes this: too long and a real failure stalls everyone; too short and a normal GC pause makes you a zombie — and the pause can always exceed any chosen timeout. The only robust fix is the fencing token: each acquisition gets a monotonically increasing token; the client sends it with every write; the protected resource records the highest token it has seen and rejects any write with a lower token. So when zombie A writes with token 33, the resource (having seen 34) rejects it — corruption impossible, even though A still "holds" the lock. The crucial insight: the lock service alone cannot guarantee exclusivity; the resource must validate the token. A lock without fencing is unsafe for anything correctness-critical.

Sub-problem 2 — Leases & liveness (avoiding deadlock)

If a lock is held until explicitly released, a crashed holder deadlocks everyone forever. So locks are leases: they auto-expire after a TTL unless the holder renews (heartbeats). A dead holder stops renewing, the lease expires, and the lock frees up — liveness without a human. The tension: the TTL must be long enough to outlast normal pauses but short enough for timely recovery.

The failure that happens when you get this wrong — the held-forever lock (and the slow-vs-dead tension) This is Scenario 2. Without a lease/TTL, a holder that crashes (or just forgets to release) holds the lock forever — permanent deadlock for all waiters. The lease + renewal fixes liveness: no renewal → expiry → lock freed. But this creates the zombie risk (Scenario 1): a holder that's merely slow (long GC) can have its lease expire while it's still alive and "thinks" it holds the lock. So leases trade deadlock-prevention for the zombie problem — which is exactly why fencing (sub-problem 1) is mandatory, not optional, once you use leases. Size the TTL above your worst expected pause, require frequent renewals, and never rely on the lease alone for correctness. (This is the same lease/heartbeat pattern as the job scheduler and leader election.)

Sub-problem 3 — The lock store's own correctness

The lock service itself must never grant the same lock to two clients — even during its own node failures or partitions. This requires the store to be strongly consistent / consensus-backed (etcd/ZooKeeper/Chubby), so that a partition can't let two halves each grant the lock (no split-brain in the lock store). A lock built on a store without this guarantee can hand out two holders during a failure — defeating the entire purpose.

The failure that happens when you get this wrong — an unsafe lock store (the Redlock debate) This is Scenario 3. Kleppmann's critique of Redlock: acquiring a lock across N independent Redis nodes (a majority) for "HA" makes timing assumptions (bounded clocks, bounded network/process delays) that real systems violate — and it provides no fencing token. So under clock drift, GC pauses, or partitions, Redlock can grant two holders and can't protect the resource. antirez's rebuttal: it's acceptable for efficiency locks (where an occasional double-grant just wastes work) and the timing assumptions are practically fine. The resolution both sides imply: a single-node Redis lock or Redlock is fine for "efficiency" (best-effort) locking, but for "correctness" locking you need a consensus-backed store (etcd/ZooKeeper) AND fencing tokens. The deeper lesson: a lock service's safety depends entirely on its store's consistency model — build correctness-critical locks on consensus, not on a store that can split-brain.

Sub-problem 4 — Efficiency locks vs correctness locks

Crucially, not all locks need the same rigor. An efficiency lock just avoids redundant work (e.g. "only one worker should regenerate this cache entry") — if it occasionally fails and two workers run, you waste a little work, no harm. A correctness lock protects against damage (e.g. "only one process may write this file / charge this card") — a double-grant corrupts data or money. These have very different requirements.

The failure that happens when you get this wrong — using the wrong lock grade Matching the lock to the stakes is itself a design decision. Over-engineering an efficiency lock (full consensus + fencing for "don't recompute the cache twice") wastes complexity and latency — a cheap best-effort Redis lock is fine, since the worst case is duplicated work. Under-engineering a correctness lock (a fencing-less Redis lock guarding money or a shared file) is the dangerous one — a GC pause or partition leads to two holders and real corruption (Scenarios 1 + 3). The rule: ask "what happens if two clients hold this at once?" — if the answer is "wasted work," use a cheap lock; if it's "data corruption / lost money," use a consensus-backed store + fencing token enforced by the resource (and ideally, redesign so you don't need a distributed lock at all — e.g. make the operation idempotent or use a single-writer/partition design). Distributed locks are a last resort for correctness; prefer designs that don't need them.

The rule in one lineA held lock is never truly exclusive (the holder can pause), so for correctness you need a consensus-backed store (no split-brain grants) plus a fencing token the resource enforces (rejects stale writers) plus a lease (no deadlock). For mere efficiency, a cheap best-effort lock suffices. And the best correctness lock is often no lock — use idempotency or single-writer partitioning instead.

The takeawayThe common production correctness-lock is the DynamoDB lock client: a conditional write claims the lock, a TTL/lease auto-expires it, and the item's version number serves as the fencing token — which the protected resource still must check. For efficiency locks, a single-node Redis SET NX is fine. But before reaching for any distributed lock on a correctness path, ask whether idempotency or a single-writer design removes the need — it usually does, and it's safer.

Famous implementations

Same facets — what it is, the model, and the failure to watch.

etcd / ZooKeeper / Chubby (correctness locks)

  • What it is. Consensus-backed coordination kernels with lock recipes (covered in Leader Election).
  • Model. Atomic create-if-not-exists + ephemeral/lease keys + watches; the key's revision (etcd mod-revision / ZK zxid) is a natural fencing token.
  • The failure to watch. The lock is safe only if the resource checks the fencing token; minority partitions go read-only (a feature); session-timeout vs GC tuning.

Redis / Redlock (efficiency locks)

  • What it is. A lock via Redis SET NX EX (single node) or Redlock (across N nodes).
  • Model. Fast, lease-based; Redlock acquires a majority of independent Redis nodes for HA.
  • The failure to watch. No fencing token and timing assumptions (Kleppmann) — fine for efficiency, unsafe for correctness; a single-node Redis lock loses the lock if that node fails over.

Database / optimistic concurrency (often the better answer)

  • What it is. Use the database's own atomicity instead of a separate lock — conditional updates, compare-and-swap on a version column.
  • Model. "Update row WHERE version = X" succeeds for exactly one writer (the version is a fencing token); no separate lock service.
  • The failure to watch. Contention on a hot row; but for many "one writer at a time" needs this is simpler and safer than a distributed lock.

Build it yourself

Work each before opening the approach.

Add fencing tokens to a lease-based lock

Take a lease-based lock and make it safe against a GC-paused zombie holder. Define what the lock returns and exactly what the protected resource must do.

Reveal the approach

On each successful acquire, return a monotonically increasing fencing token (use the consensus store's key version — etcd mod-revision / ZK zxid — guaranteed to increase per acquisition). The client must send this token with every write to the protected resource (the database, file store, etc.). The resource records the highest token it has accepted and rejects any write carrying a lower token. Now the GC-pause scenario is safe: A acquires (token 33), pauses; A's lease expires; B acquires (token 34) and writes — the resource records 34. When zombie A wakes and writes with token 33, the resource sees 33 < 34 and rejects it (Scenario 1). The non-negotiable insight: the lock service can't enforce exclusivity — only the resource can, by validating the token. No lease/timeout value substitutes for this. If the resource can't check a token (a dumb third-party API), you can't make the lock truly correct — redesign (idempotency / single-writer) instead.

Avoid the lock entirely

You're tempted to use a distributed lock so "only one worker processes each order." Design an alternative that needs no lock and is safer.

Reveal the approach

Distributed locks for correctness are a last resort — prefer designs that eliminate the need. Two common alternatives: (1) Single-writer partitioning — route all work for a given order to one consumer by partitioning on order_id (e.g. Kafka key), so by construction only one worker ever handles that order; no lock needed, no zombie risk. (2) Idempotency + optimistic concurrency — let multiple workers attempt the work but make it idempotent (dedup on order id) and/or guard the state transition with a conditional update (UPDATE order SET state='processing' WHERE id=X AND state='new') — exactly one succeeds; the database's atomicity is your lock + fencing token (the version column). Both avoid the held-lock-never-exclusive problem entirely. Reach for an actual distributed lock only when you genuinely need cross-resource mutual exclusion that these can't express — and then use consensus + fencing (Scenario's sub-problem 4).

Design-it-yourself: the open-ended ones

  • Build leader election on the lock. Use acquire + lease + watch so exactly one node leads and failover is fast — with fencing so a zombie leader can't act. → Leader Election.
  • Classify your locks. For three real needs (cache regen, leader election, ledger write), decide efficiency vs correctness and the right implementation for each. → Payment System.
  • Make a correctness lock survivable. Your consensus lock store goes read-only on a minority partition. Design the system so that stall degrades gracefully. → Consensus, Resilience.
