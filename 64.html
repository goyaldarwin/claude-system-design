Designing a Distributed Locking Service

Mutual exclusion across machines that crash and partition — why locks need consensus, why TTLs alone are unsafe, and why fencing tokens are the real fix.

The bottleneck No shared truth. mutual exclusion across machines, paid in a consensus-backed lease and fencing tokens.

Requirements and scope

The brief: design a service that lets independent processes — running on different machines, in different datacenters, with no shared memory — coordinate exclusive access to a resource. This is the role played by Google Chubby, Apache ZooKeeper, etcd, and HashiCorp Consul in real fleets.

Functional requirements:

  • Mutual exclusion: at most one client holds a named lock at any instant.
  • Acquire / release / try-acquire with optional blocking and a wait queue (fairness).
  • Shared/exclusive (read/write) modes are a common extension.
  • Lock ownership metadata — who holds it, since when — for debugging and observability.

Non-functional requirements — these dominate the design:

  • Safety: never grant the same exclusive lock to two clients. This is non-negotiable and must survive crashes, GC pauses, and partitions.
  • Liveness (no deadlock): if a lock holder crashes, the lock must eventually be reclaimable. A lock you can never release is worse than no lock.
  • Fault tolerance: the service survives the loss of a minority of nodes.
  • Low, predictable latency for acquire/release; locks sit on the hot path.
PitfallSafety and liveness pull in opposite directions. To guarantee liveness you must time out crashed holders; but a "crashed" holder can just be paused (GC, VM migration, network blip) and wake up still believing it holds the lock. Reconciling these two is the entire substance of this problem.

Do you actually need a distributed lock?

A principal should open by interrogating the requirement, because distributed locks are reached for far too often and are a frequent source of correctness bugs. There are two legitimate purposes, and they have very different bars:

  • Efficiency: avoid doing duplicate work (e.g. only one worker regenerates a cache entry). A rare double-execution is merely wasteful. A weak, single-node lock is fine here.
  • Correctness: two clients writing the same resource would corrupt data or double-charge a customer. Here a single violation is a catastrophe, and you need the full machinery below.
What strong engineers doThey ask "efficiency or correctness?" first. If correctness, they push back on whether a lock is even the right tool: often the better answer is to make the operation idempotent, use an atomic compare-and-swap / conditional write on the resource itself, partition work so a single owner is structurally guaranteed (e.g. Kafka partition assignment), or use a database transaction. A lock is a coordination band-aid layered outside the system of record; pushing the invariant into the system of record is almost always safer.

This connects directly to Designing a Distributed Job Scheduler (ensuring a task runs once) and Leader Election and Coordination (a "lock held forever by one process" is just leader election).

Why a lock needs consensus underneath

The naive design — a single server holding an in-memory hash of lockName → owner — gives perfect mutual exclusion until that server dies, at which point every lock is lost and liveness is gone. To be fault tolerant you must replicate the lock state, and the instant you replicate state across machines you are forced to solve agreement: which replica's view of "who holds lock L" is authoritative when they disagree?

That is the consensus problem. A distributed lock service is, at its core, a thin API over a replicated state machine driven by a consensus log — Raft in etcd and Consul, Zab (a Paxos relative) in ZooKeeper, Paxos in Chubby. The lock state is just an entry in a consistently-replicated, linearizable key-value store. "Acquire lock L" becomes a CAS append to the consensus log: create key /locks/L if it does not exist. Because consensus gives linearizability, every client sees a single agreed history of acquisitions and releases.

Key insightYou cannot build a correct distributed lock on top of a store that lacks linearizable, fault-tolerant compare-and-set. That is why locks are built on Chubby/ZooKeeper/etcd and not on an eventually-consistent store or a single Redis node. The lock service inherits the CP nature of its consensus core (see Consensus and Raft and CAP Theorem): during a partition the minority side cannot grant locks, by design.

The consensus layer also gives a clean answer to ordering and fairness. Because every operation is sequenced in the replicated log, the service can hand out locks in request order and there is a single global timeline of events (relevant when reasoning about happens-before; see Time, Clocks and Ordering).

Lease-based locks with TTL

To prevent a crashed holder from deadlocking the lock forever, locks are granted as leases: a lock is held only for a bounded time-to-live (TTL), and the holder must renew (heartbeat) before expiry to keep it. If the holder crashes, it stops renewing, the lease expires, and the service reclaims the lock automatically.

// Client side: acquire then keep alive
lease = locks.acquire("L", ttl=15s)        // CAS create in consensus store
spawn renewer:
    every 5s: locks.renew(lease)            // extend TTL while alive
do_work_under_lock()
locks.release(lease)
stop renewer

The TTL is a tension knob. Short TTL = fast crash recovery but more renewal traffic and higher risk of false expiry under load. Long TTL = cheap, but a crashed holder blocks others for a long time. etcd implements this with a Lease object that keys attach to; ZooKeeper achieves the same effect via session timeouts (below).

Trade-offTime-based leases require bounded clock drift and bounded scheduling delay to be safe. Neither is guaranteed in practice: a stop-the-world GC pause, a hypervisor freezing a VM for live migration, or a long disk stall can exceed any TTL you pick. This is exactly where TTL-only locks become unsafe, which motivates fencing tokens.

The fencing-token problem

This is the heart of the lesson and Martin Kleppmann's well-known critique. Consider the failure timeline that any TTL-based lock permits:

  1. Client 1 acquires lock L with a 15s lease and begins writing to a shared store.
  2. Client 1 hits a stop-the-world GC pause (or VM migration, or a swapped-out page) lasting 20s.
  3. The lease expires at 15s. The lock service, correctly, reclaims L and grants it to Client 2.
  4. Client 2 begins writing to the shared store.
  5. Client 1 resumes from its pause — still believing it holds the lock — and issues its delayed write.

Now two clients write concurrently. The lock service did nothing wrong; the lock was correctly exclusive at every instant according to its own clock. The bug is that the client's belief about holding the lock is stale, and the protected resource has no way to know that. No TTL, however carefully tuned, closes this hole — the pause can always exceed it, and you cannot detect "this node is paused" from inside the node.

PitfallAdding a "check the lock again right before writing" step does not fix this. The pause can occur in the window between the check and the write. Mutual exclusion on the lock does not imply mutual exclusion on the resource; the gap between them is unbounded.

The fix: monotonic fencing tokens

The lock service returns, with every grant, a fencing token — a number that strictly increases on each successful acquisition (a monotonic counter maintained in the consensus log; ZooKeeper's zxid / per-node version, etcd's revision, serve this role). The client passes this token on every write to the protected resource, and the resource (or its storage layer / gateway) rejects any write carrying a token lower than the highest it has already seen.

// Lock service
token = consensus.next_monotonic_token()    // strictly increasing, never reused
grant(client, lease, token)

// Protected storage, on every write
def write(data, fencing_token):
    if fencing_token <= self.max_seen_token:
        reject("stale token")                // an old, paused holder is fenced out
    self.max_seen_token = fencing_token
    apply(data)

Replaying the timeline: Client 1 holds token 33, Client 2 gets token 34. When Client 2 writes, the store records max_seen=34. When the paused Client 1 finally writes with token 33, the store sees 33 ≤ 34 and rejects it. The stale holder is fenced.

Key insightThe lock service alone can never guarantee correctness, because correctness depends on what the resource does. Fencing pushes the final enforcement into the resource itself: the lock orders intentions, the fencing token lets the resource reject out-of-order effects. A correctness-critical design that uses a lock without a fencing mechanism on the protected resource is, by Kleppmann's argument, simply broken — and a strong engineer will name this explicitly.

The catch: this only works if the protected resource can check and store a token. Many can (databases via a WHERE token > ? conditional update, object stores via conditional writes / preconditions). Some cannot — an arbitrary side-effecting API with no idempotency or versioning support cannot be safely fenced, which is itself a signal that a lock is the wrong tool for that interaction.

Sessions and ephemeral nodes (ZooKeeper / Chubby)

ZooKeeper offers an elegant lock primitive built on two features. A session is a client–cluster connection with a negotiated timeout; the client heartbeats to keep it alive, and if heartbeats stop the session expires server-side. An ephemeral znode is a node tied to a session — when the session ends (crash, network loss, expiry) the ZooKeeper cluster automatically deletes the node. This is the lease mechanism realized as data, with crash-cleanup handled by the cluster rather than the client.

The canonical recipe avoids the herd effect (every waiter waking on each release) by using sequential ephemeral nodes and each waiter watching only its immediate predecessor:

// ZooKeeper exclusive lock recipe
1. create("/lock/req-", EPHEMERAL | SEQUENTIAL)   // -> /lock/req-0000000017
2. children = sorted(getChildren("/lock"))
3. if my node is the lowest sequence number: LOCK ACQUIRED
4. else: set a watch on the ONE node just below mine; wait
5. on watch fire (predecessor gone): go to step 2

This gives FIFO fairness, no thundering herd, and automatic release on crash. The sequence number doubles as a coarse ordering, and ZooKeeper's zxid can serve as the fencing token. Chubby (Google) is philosophically similar — a small, Paxos-replicated, strongly consistent filesystem-like store whose primary use turned out to be name resolution, not the coarse-grained locking it was built for.

Trade-offZooKeeper/Chubby deliberately target coarse-grained locks — held for minutes to hours, low acquisition rate (leader election, config ownership, shard assignment). They are not built for fine-grained, high-churn locks taken millions of times per second; the consensus write per acquisition makes that prohibitively expensive. If you need fine-grained locking, redesign so the contention lives inside a sharded data store, not a global coordinator.

The Redlock debate

Redis is fast and tempting, so it is constantly used for locking. The key distinctions:

  • Single-node Redis lock — SET resource token NX PX 15000 — is fine for the efficiency use case but unsafe for correctness. Redis replication is asynchronous: if the master grants a lock and crashes before replicating to its replica, the promoted replica has no record of the lock, and a second client can acquire it. Mutual exclusion is silently violated.
  • Redlock is Redis's multi-node algorithm: acquire the lock on a majority of N independent Redis masters within a time bound. The argument is that majority acquisition tolerates a minority of node failures.

The debate. Kleppmann argued Redlock is unsafe for correctness because it relies on wall-clock timing assumptions rather than consensus: clock jumps (NTP steps), GC pauses, and process stalls can cause two clients to believe they hold the lock simultaneously, and Redlock provides no fencing token to backstop that. Antirez (Redis's author) rebutted that the timing assumptions are reasonable in practice and that fencing can be layered on top. The principal-level takeaways are durable regardless of who "won":

  1. For correctness-critical locks, prefer a system whose safety rests on consensus (etcd, ZooKeeper, Consul), not on synchronized clocks.
  2. Whatever lock you use, if correctness matters you still need fencing tokens on the resource — the lock service alone is never sufficient. This is the through-line that makes the Redlock argument secondary.
  3. For pure efficiency locks, a single-node Redis lock is pragmatic, cheap, and entirely defensible. Match the mechanism to the cost of a violation.

A reference design and API

Putting it together: a lock service is a stateless API tier in front of a 3- or 5-node consensus cluster (Raft). The API tier routes writes to the current Raft leader; the cluster tolerates the loss of a minority (1 of 3, 2 of 5).

OperationSemanticsReturns
acquire(name, ttl, mode)CAS-create lock key via consensus; block or queue if heldlease handle + fencing token
tryAcquire(name, ttl)Non-blocking; fail fast if heldlease | null
renew(lease)Extend TTL; fails if lease already expired/lostok | LOST
release(lease)Delete lock key via consensusok
watch(name)Notify on release (avoids polling / herd)stream

Critical client-side contract: the renew call must return LOST if the lease expired, and the application must stop work immediately and not write if it ever observes a lost lease — but because of the pause problem, it must also carry the fencing token on every protected write rather than trusting its local belief. Observability matters too: emit metrics for lock hold time, contention/wait time, renewal failures, and lease expirations — a sudden spike in expirations usually means GC or network trouble, not crashes (see Observability).

What strong engineers doThey sketch the full chain — consensus core for safety, leases for liveness, fencing tokens for the GC-pause hole — and they state which guarantee each layer provides and which failure each one does not cover. They size the lock granularity to the contention rate, and they keep returning to "what breaks if a holder pauses for 30 seconds?" as the litmus test for every design choice.

Principal-level mastery check

What strong engineers doThey refuse to treat "distributed lock" as a black box. They distinguish efficiency from correctness up front, recognize that a fault-tolerant lock is a consensus problem in disguise, and — the discriminator — they raise the fencing-token / GC-pause problem unprompted and explain that the lock service can never guarantee correctness on its own; the protected resource must reject stale tokens. They treat Redlock skeptically for correctness-critical paths and reach for idempotency or conditional writes before reaching for a lock at all.

Sharp questions to test your own understanding:

  • "Your lock holder pauses for 30 seconds in GC while holding a 10-second lease. Walk me through what happens, and how you prevent data corruption." (Forces fencing tokens; TTL-only answers fail here.)
  • "Why can't I just use SETNX on a single Redis node for this? When is that actually fine, and when is it dangerous?"
  • "Your lock service is built on a 3-node Raft cluster and the network partitions 1 from 2. Can clients on the minority side acquire or renew locks? Should they? What does the majority side do?"
  • "We're acquiring this lock 500k times per second. Is ZooKeeper the right choice? If not, how do you redesign so you don't need a global lock at all?"
  • "How do you choose the TTL? What inputs go into that number, and what fails if you pick it too short or too long?"
  • "The protected resource is a third-party REST API with no versioning or idempotency support. Can you make this lock correct? If not, what do you tell the team?"
