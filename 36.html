Leader Election and Coordination

The coordination kernel — ZooKeeper, etcd, and Chubby — and why almost every “distributed lock” you’ve written is subtly broken.

The bottleneck No shared truth & partial failure. electing one coordinator, paid in a consensus dependency and failover windows.

Why a coordination kernel exists

Once you have consensus, you rarely want every team to re-implement it. Coordination services package a consensus-replicated log behind a small, opinionated API so that application engineers can do leader election, locking, config distribution, and membership without writing a single Paxos line. The insight from Google’s Chubby paper (Burrows, 2006) was that developers asked for a lock service, not a consensus library — a familiar lock/file abstraction is what people actually reach for, and centralizing it means one well-audited implementation of the hard part.

These systems share a design shape: an odd-sized ensemble (3 or 5), a strongly-consistent replicated tree or key space, sessions with liveness tied to heartbeats, and watches/notifications so clients react to changes without polling. They are deliberately CP — during a partition the minority side stops serving rather than risk divergence. See CAP Theorem.

Key insightA coordination service is not a database. It is tuned for small amounts of critical metadata read far more than written — leader pointers, locks, config, membership. Putting bulk data or high write rates through it is the most common abuse and will collapse the ensemble.

ZooKeeper: ZAB, znodes, sessions, watches

ZooKeeper exposes a hierarchical namespace of znodes — like a filesystem where each node holds a small (≤ 1 MB, practically a few KB) data payload and can have children. The whole tree is replicated across the ensemble and kept consistent by ZAB (ZooKeeper Atomic Broadcast), a primary-backup atomic broadcast protocol closely related to Multi-Paxos and discussed alongside Raft.

ZAB in one paragraph

One server is the leader; followers forward writes to it. ZAB proceeds in epochs (like Raft terms). On leader failure it runs leader election + recovery: the new leader must hold the highest zxid (a 64-bit id = epoch << 32 | counter) so no committed transaction is lost, then syncs followers up to it. Writes are totally ordered by zxid and committed once a quorum acks — identical safety logic to Raft’s majority commit.

Znode flavours that make recipes work

  • Persistent — survives client disconnect; used for config and stable structure.
  • Ephemeral — automatically deleted when the creating session ends. This is the linchpin of leader election and membership: liveness becomes a side effect of node existence.
  • Sequential — ZooKeeper appends a monotonically increasing counter to the name (/lock/req-0000000042). Server-assigned, so it’s a cluster-wide unique, ordered token — and conveniently a fencing token (see below).

Sessions and the timing reality

A client opens a session with a negotiated timeout and keeps it alive with heartbeats. If heartbeats stop for the timeout, the server expires the session and deletes all its ephemeral nodes. The dangerous subtlety: the client’s local view of “I still hold the session” can lag the server’s decision to expire it. A GC pause, a clock jump, or network blip can mean the ensemble has already revoked your ephemeral lock node while your process still believes it’s the leader. This gap is the root of nearly every distributed-lock bug.

Watches

A watch is a one-shot trigger: a client registers interest in a znode, and on the next change ZooKeeper fires exactly one notification, after which the client must re-register. Watches are ordered relative to the state change (you never see the new data before the watch fires), which is what makes them safe to build recipes on. The one-shot design deliberately avoids the “herd” of holding millions of standing subscriptions.

PitfallNaive election where every contender watches the single /leader znode creates a herd effect: when the leader dies, all N−1 followers wake, re-read, and stampede the ensemble. The correct recipe (below) has each contender watch only the one node immediately ahead of it.

etcd and Chubby

etcd

etcd is the modern Go-based equivalent and the brain of Kubernetes (it stores all cluster state). It uses Raft directly, exposes a flat sorted key space (not a tree) over gRPC, and replaces ephemeral nodes with explicit leases: a key attached to a lease vanishes when the lease isn’t renewed. Election and locking are first-class — the clientv3/concurrency package gives you Mutex and Election built on lease + revision. Critically, every etcd key carries a monotonic revision (mod_revision), which is a ready-made fencing token. etcd also supports MVCC and multi-key transactions (compare-and-swap via Txn), making it more ergonomic than classic ZooKeeper for compare-then-write coordination.

Chubby

Chubby is Google’s internal lock service and the intellectual ancestor of both. Key design choices worth knowing: it uses Paxos for the replicated log; clients hold coarse-grained locks (held for hours/days, e.g. “I am the GFS master”) rather than fine-grained per-request locks, because coarse locks dramatically reduce load on the lock server and tolerate brief outages; and Chubby explicitly issues a sequencer — a lock-generation token a client passes to downstream services so a stale lock holder is rejected. That sequencer is exactly the fencing token the open-source world keeps rediscovering.

Leader election recipes

The canonical, herd-free ZooKeeper election:

// 1. Create my candidacy node (ephemeral + sequential)
myNode = create("/election/n-", EPHEMERAL_SEQUENTIAL)   // e.g. /election/n-0000000007

// 2. List children, sort by sequence number
children = sortBySeq(getChildren("/election"))

// 3. Am I the lowest? Then I'm the leader.
if (myNode == children[0]) {
    becomeLeader()
} else {
    // 4. Watch ONLY the node immediately before me (no herd)
    pred = children[indexOf(myNode) - 1]
    exists(pred, watch=true)   // when pred's session dies, its ephemeral node
                               // disappears -> my watch fires -> recheck step 3
}

Because each contender watches exactly one predecessor, a leader failure wakes exactly one client. etcd’s Election.Campaign() implements the same idea using the lowest create-revision among keys sharing a prefix.

PitfallWinning the election is not the same as being the leader. Between your watch firing and you re-reading the children, the world can change. And a leader can be deposed by a session expiry it hasn’t noticed yet. Treat “I am leader” as a revocable lease, never a permanent fact — and carry a fencing token into every side effect.

Distributed locks and the fencing-token problem

A lock recipe is the election recipe stopping at “lowest sequence number holds the lock.” The hard part is not acquiring the lock — it’s what happens when the holder thinks it still holds the lock but doesn’t. Consider:

  1. Client A acquires the lock and starts writing to a shared store.
  2. A pauses (stop-the-world GC, or the host is descheduled) for longer than the session timeout.
  3. The coordination service expires A’s session, deletes its ephemeral node; client B acquires the lock and begins writing.
  4. A wakes up, unaware it lost the lock, and completes its write — corrupting state. Two writers, one lock.

No lock service can prevent step 4 by itself, because the danger is in A’s already-in-flight effects. The fix is a fencing token: every lock acquisition returns a monotonically increasing number, and the protected resource itself must reject any write carrying a token lower than the highest it has seen.

// Storage layer enforces monotonicity of the token
write(data, token):
    if token < this.lastToken:
        reject("stale lock holder")     // A (token=33) is fenced out
    this.lastToken = token
    apply(data)

// A acquired with token 33, paused. B acquired with token 34.
// B's write bumps lastToken to 34. A's late write(token=33) is rejected.
Key insightA distributed lock is only as safe as the resource it guards. The lock service gives you the token (zxid / mod_revision / sequencer); the resource must enforce it. A lock without a fencing-aware resource provides at most mutual exclusion under the optimistic assumption that no one ever pauses — which is false in any real system.

The Redlock debate

A naive single-instance Redis lock — SET lockkey uuid NX PX 30000 — is fast and usually fine for efficiency (avoid duplicate work), but it is not safe for correctness because (a) Redis replication is asynchronous, so a failover can lose the lock key while a client still holds it, and (b) it provides no fencing token. Two clients can both believe they hold the lock.

Redlock was Redis’s proposed answer: acquire the lock on a majority of N independent Redis masters with the same TTL. Martin Kleppmann’s critique (2016) is the canonical principal-level reading:

  • Redlock relies on bounded clock drift and bounded pauses across all nodes. A GC pause or NTP step on a client invalidates the TTL reasoning — the lock can expire while the client thinks it’s held.
  • It still issues no fencing token, so it cannot protect a resource against a delayed writer regardless of how many Redis nodes agree.
  • It conflates an efficiency lock (best-effort, occasional double-work is fine) with a correctness lock (must never allow two writers). Only the latter needs the full machinery, and for that you want a consensus-backed service (ZooKeeper/etcd) plus fencing.

Antirez’s rebuttal argued the model is sound under reasonable operational assumptions. The principal takeaway is not who “won”: it’s that a lock’s safety is a property of the whole system — client pauses, clock assumptions, and the resource’s fencing — not of the lock algorithm in isolation. We build a real one in Designing a Distributed Locking Service.

Trade-offRedis lock: ~sub-millisecond, no extra infra, but unsafe for correctness. ZooKeeper/etcd lock: a consensus round-trip (single-digit ms within a region), needs an ensemble, but gives ordered fencing tokens and survives failover. Choose by asking: does a double-acquire corrupt data, or merely waste work?

Service discovery, config, and membership

The same primitives power three of the most common production uses:

Service discovery

Each service instance registers an ephemeral node / lease under a path (/services/payments/inst-42) carrying its host:port. Consumers watch the parent and get a live, self-healing roster — a crashed instance’s ephemeral node vanishes on session expiry, automatically deregistering it. This is the historical pattern; modern stacks often push discovery into the platform — Kubernetes Services + endpoints (backed by etcd) and the data-plane sidecars of a service mesh.

Dynamic configuration

Store config in znodes/keys; clients watch for changes and hot-reload without redeploy — feature flags, sharding maps, rate-limit thresholds (cf. API Rate Limiter). The strong consistency matters: all consumers converge on the same config version in the same order.

Membership and failure detection

Ephemeral nodes give you cluster membership for free, with failure detection delegated to session timeouts. The trade-off is sensitivity: short timeouts detect failures fast but flap on transient blips and GC pauses; long timeouts are stable but slow to react. For large, churny fleets where routing membership through one consensus ensemble doesn’t scale, systems instead use decentralized gossip-based membership (SWIM) — eventually consistent, but vastly more scalable, and the right tool when you don’t need a linearizable view. Scheduling systems lean on these primitives heavily; see Designing a Distributed Job Scheduler.

PitfallRouting every service’s health check and discovery lookup through a 3-node ZooKeeper ensemble is a classic scaling cliff. Reads scale (followers serve them), but watch fan-out and write load (registrations, heartbeats) hit the leader. Beyond a few tens of thousands of watches, you need caching, observer nodes, or a gossip layer in front.

Principal-level mastery check

What strong engineers doThey never say “I’ll use a distributed lock” without immediately asking whether it’s an efficiency lock or a correctness lock, and if correctness, they reach for fencing tokens enforced at the resource. They treat “I am the leader” as a revocable lease, account for GC pauses and clock skew, and keep the coordination service off the high-throughput data path — using it only for small, critical, mostly-read metadata.

Probing questions to ask engineers:

  • “Your leader holds a ZooKeeper lock, then GC-pauses for 40 seconds past the session timeout. Walk me through exactly what goes wrong and how a fencing token prevents it.”
  • “Why is a single-instance Redis SETNX lock unsafe for correctness? Would Redlock fix it? What does Redlock still not give you?”
  • “Design a herd-free leader election in ZooKeeper. Why watch the predecessor node and not /leader?”
  • “You have 50,000 instances doing service discovery. Why would a 5-node etcd ensemble fall over, and what would you do instead?”
  • “Compare ephemeral nodes, etcd leases, and gossip-based membership for failure detection. When does each win?”
  • “Why do these systems stop serving the minority side of a partition? When would that be the wrong choice for your use case?”
