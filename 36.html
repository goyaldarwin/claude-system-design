Leader Election & Coordination

Almost every distributed system needs to answer "who's in charge right now?" — which node is the primary, who holds this lock, what's the current config. Doing that correctly is so hard (it's consensus underneath) that you don't build it per-app; you stand up one coordination kernel — ZooKeeper, etcd, Consul — and everything else asks it. This lesson is the recipes that kernel makes possible, and the one bug (the stale lock) that keeps biting people.

The crux A small, strongly-consistent source of truth for "who/what is current" — in exchange for putting a consensus cluster on your critical path and accepting that a lock alone never guarantees mutual exclusion. A coordination kernel gives you correct leader election, locks, config and membership. You pay with a consensus cluster's latency and fault ceiling, and you must add fencing tokens because a held lock can always go stale.

What a coordination kernel is

A coordination kernel is a small, strongly-consistent, highly-available store whose job is to hold the handful of facts a distributed system must agree on: who is the current leader, who holds a lock, what is the live cluster membership, what is the current configuration. ZooKeeper, etcd, and Consul are the canonical ones. Under the hood each runs a consensus protocol (ZAB, Raft, Raft respectively), so the hard part — agreement under failure — is solved once, in one place, by experts.

You interact with it through a tiny set of primitives, and build all the higher-level recipes from them:

  • Consistent small data — a key/value or hierarchical namespace (ZooKeeper's znodes) that every client sees identically.
  • Ephemeral keys / sessions — data that vanishes when the client that created it disconnects (the basis of liveness).
  • Watches — "notify me when this key changes," so clients react instead of poll.
  • Compare-and-swap / create-if-absent — the atomic primitive that makes "exactly one winner" possible.
The one-line frameA coordination kernel trades a consensus cluster on your critical path for a correct, single source of truth about "who/what is current." You centralize the hardest problem so your apps don't reinvent it — but you inherit consensus's latency and fault ceiling, you must keep the kernel small and reliable, and you must remember that a lock it grants is only safe with a fencing token.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the Redlock debate, Kleppmann vs antirez (real, ongoing) Martin Kleppmann published a teardown of Redis's "Redlock" distributed-lock algorithm; antirez (Redis's creator) rebutted. Kleppmann's core point: a client acquires a lock, then pauses for a long GC — the lock's lease expires, another client legitimately grabs it, and then the first client wakes up still believing it holds the lock and writes to the shared resource. Two writers, corruption. What single mechanism makes this corruption impossible, and why does no timeout value alone fix it?
Scenario 2 — the thundering herd A 500-node cluster does leader election by having every node watch the same "leader" key, so when the leader dies, all 499 wake up and stampede to grab it at once. How do you elect a new leader so that exactly one is chosen and the other 498 barely stir?
Scenario 3 — the frozen primary Your database primary is elected via an ephemeral key in the coordination kernel. The primary's host freezes (GC, or a network hiccup) for longer than its session timeout. The kernel declares its session dead, deletes the ephemeral key, and a standby is elected primary. Then the original host un-freezes. For a brief window, how many nodes believe they're the primary, and what stops the zombie from doing damage?

Scenarios 1 and 3 are the same deep bug — the stale lock / zombie leader, fixed by fencing tokens. Scenario 2 is the election recipe that avoids the herd. Each maps to an idea below.

The five ideas to understand

Theory, not knobs — ideas to hold correctly, each guarded by a misconception.

Idea 1 — The kernel and its primitives

The reason a coordination kernel exists: correct leader election / locking is consensus, and consensus is hard to implement correctly (FLP, partitions, fencing). So you run one battle-tested consensus cluster and let every service build its coordination on its primitives, rather than embedding a buggy half-Paxos in each app. ZooKeeper pioneered this with its znode namespace + ephemeral nodes + watches; etcd offers the same idea as a flat key/value store with leases and watches.

The misconception to avoid — "I'll just implement leader election in my service" Hand-rolled election (ping each other, highest-IP-wins, etc.) breaks under exactly the cases that matter: a partition gives you two leaders; a GC pause gives you a zombie; a lost message gives you none. Delegate to a kernel that has solved consensus. The only thing you build is the recipe on top of its primitives — and even that you usually take from a library (Curator for ZooKeeper, etcd's concurrency package).

Idea 2 — The leader-election recipe (and the herd)

The standard ZooKeeper recipe: each candidate creates an ephemeral sequential znode under a common parent (/election/n_0001, /n_0002, …). The lowest sequence number is the leader. Crucially, each node watches only the node immediately ahead of it, not the leader. When a node dies, only its single successor is notified — it checks if it's now lowest, and if so becomes leader. One leader, one notification.

The misconception to avoid — "everyone watches the leader key" This is Scenario 2. If all N nodes watch the same key, a leader death wakes all N at once — the thundering herd — and they hammer the kernel. The sequential-znode + watch-your-predecessor trick turns an O(N) stampede into O(1) notifications: a death wakes exactly one node. The same idea makes fair, queued distributed locks (each waiter watches the holder ahead of it).

Idea 3 — Locks and the fencing-token problem

A distributed lock from the kernel grants you the lock — but holding a lock does not guarantee you actually have exclusive access to the protected resource, because you can be paused or partitioned and lose the lock without knowing. The fix is a fencing token: the lock service hands each acquisition a monotonically increasing number. The client includes this token with every write to the shared resource, and the resource rejects any write carrying a token lower than the highest it has seen. A zombie holder's stale (lower) token is refused.

The misconception to avoid — "if I hold the lock, I have exclusive access" This is Scenarios 1 and 3, and the heart of the Kleppmann/antirez debate. No timeout saves you: too long and a real failure stalls everyone; too short and a GC pause makes you a zombie. The only robust fix is fencing — the protected resource must enforce the token order, so even if two clients both believe they hold the lock, the one with the stale token gets rejected. Kleppmann's verdict on Redlock: without fencing tokens (and with its timing assumptions), it's unsafe for correctness-critical locking. The lock service can't make a resource safe; the resource must check the token.

Idea 4 — Sessions, leases, and watches

Liveness in the kernel is built on sessions/leases: a client maintains a session by sending heartbeats; if it stops (crash, freeze, partition), the session expires and the kernel automatically deletes that client's ephemeral keys. That's what makes "the leader's key disappears when the leader dies" work without anyone polling. Watches let waiting clients get a callback the instant a key changes, so failover is event-driven and fast.

The misconception to avoid — "my lock is held until I explicitly release it" A session-backed (ephemeral) lock is held only as long as your session is alive — a long pause past the session timeout silently releases it (exactly Scenario 3). This is a feature (a dead client can't hold a lock forever) and a hazard (a slow-but-alive client loses its lock without noticing). You must size session timeouts against your worst expected pause, expect to lose leadership on a long stall, and — for correctness — back it with fencing. Watches also fire once in ZooKeeper (you must re-arm them) and can lag reality slightly.

Idea 5 — Service discovery, configuration, and membership

Beyond election and locks, the kernel is the natural home for three more coordination facts: service discovery (instances register an ephemeral key on startup; it vanishes when they die, so the directory is self-cleaning), dynamic configuration (store config in a key; watchers get pushed updates without a redeploy), and membership / failure detection (the live set of nodes, kept current by sessions).

The misconception to avoid — "the coordination kernel is a general database" The kernel is strongly consistent and therefore expensive and small — every write is a consensus round-trip, and the whole dataset typically lives in memory. It's for small, critical, low-write metadata, not bulk data, high-write counters, or large blobs. Overload it (huge values, churny writes, thousands of watches) and you put a consensus stall on everyone's critical path. For high-volume service discovery many teams now prefer gossip-based systems (Consul's Serf layer) over hammering a strongly-consistent store.

Self-check — can you reconstruct the three scenarios?

You read the answers above. Close the page and rebuild them.

  • Scenario 1 (Redlock/GC pause): Why does no lease timeout alone prevent two writers, and exactly how does a fencing token make the corruption impossible? Where must the token be checked?
  • Scenario 2 (thundering herd): How does the sequential-znode + watch-your-predecessor recipe turn an O(N) stampede into O(1) notifications?
  • Scenario 3 (frozen primary): During the window, how many nodes think they're primary, what mechanism created that window, and what stops the zombie from writing?
  • Why is correct leader election really a consensus problem, and why shouldn't you implement it in your app?
  • Why must the coordination kernel stay small and low-write?

The shape to remember: 5 ideas — delegate consensus to one kernel, elect with sequential znodes (no herd), a lock needs a fencing token, sessions/leases give automatic liveness, keep the kernel small & critical-only — each guarded by a misconception.

Famous implementations

The kernels made concrete. Same facets — what it is, the coordination model, and the failure to watch.

ZooKeeper

  • What it is. The original coordination kernel — a hierarchical znode store on ZAB consensus.
  • Coordination model. Ephemeral + sequential znodes + one-shot watches; the Curator library packages the election/lock/barrier recipes.
  • The failure to watch. One-shot watches must be re-armed; session-timeout tuning vs GC pauses (Scenario 3); JVM/operational heaviness.

etcd

  • What it is. A flat, gRPC key/value store on Raft — the backing store of all of Kubernetes.
  • Coordination model. Leases (TTL keys) + streaming watches + atomic compare-and-swap; the concurrency package gives election & locks.
  • The failure to watch. The Cloudflare-2020 lesson — a partial partition can cause endless elections and a cluster-wide stall; keep it small (3/5 nodes) and low-latency.

Consul / Chubby

  • What it is. Consul = Raft KV + native service catalog + Serf gossip for membership; Chubby = Google's coarse-grained lock service (Paxos), the system that inspired ZooKeeper.
  • Coordination model. Sessions + health checks + KV watches; Consul splits strongly-consistent KV from gossip-based membership so discovery scales without hammering Raft.
  • The failure to watch. Same consensus fault ceiling for the KV side; mixing the strongly-consistent store with high-churn discovery is the anti-pattern Serf exists to avoid.

Build it yourself

These click when you design them. Work each before opening the approach.

Build leader election with no thundering herd

Using only create-if-absent, ephemeral+sequential keys, and watches, design leader election for a 500-node cluster so exactly one leads, failover is under a few seconds, and a leader death wakes only one node.

Reveal the approach

Each node creates an ephemeral sequential key under /election/ — the kernel assigns n_0001, n_0002, …. The node with the lowest sequence is leader. Each non-leader watches only the key immediately below its own (its predecessor), not the leader. When any node dies, its session expires, its ephemeral key auto-vanishes, and only its single successor is notified; that successor checks whether it's now the lowest — if yes it leads, if no it re-watches its new predecessor. Failover time = session timeout + one notification. One leader guaranteed (lowest sequence is unique), O(1) wake-ups, fair FIFO order. The same structure gives a fair distributed lock.

Add fencing tokens to a distributed lock

Take a lease-based lock and make it safe against a GC-paused zombie holder (Scenario 1). Define what the lock returns and what the protected resource must do.

Reveal the approach

On each successful acquire, the lock service returns a monotonically increasing fencing token (e.g. ZooKeeper's znode version, etcd's mod-revision, or a counter). The client must send this token with every write to the shared resource (the database, the file store). The resource records the highest token it has accepted and rejects any write with a lower token. Now if client A (token 33) pauses, its lease expires, and client B acquires (token 34) and writes, the resource has seen 34; when zombie A wakes and writes with token 33, the resource rejects it — corruption impossible. The key insight: the lock service alone can't enforce exclusivity; the resource must validate the token. No timeout value substitutes for this.

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Make a consensus stall survivable. Your coordination kernel can go unavailable on the minority side of a partition. Design the surrounding system so losing the kernel briefly degrades gracefully instead of taking everything down. → Consensus & Raft, Resilience Patterns.
  • Pick a lock for correctness vs efficiency. Decide where you need a fully-fenced, correctness-grade lock versus a cheap "best-effort" lock (Redis), and justify each by what a stale lock would cost. → Distributed Locking.
  • Scale service discovery off the kernel. Thousands of instances are churning your strongly-consistent registry. Redesign discovery (gossip layer, caching, TTLs) so it doesn't put a consensus write on every registration. → Anti-Entropy & Gossip, Microservices & Service Mesh.
