Batch & Big Data Processing

When the data is too big for one machine and you need a complete, exact answer — recompute a recommendation model, reconcile a day's payments, build a search index from scratch — you process it in batch: run a finite job over a bounded, complete dataset across a cluster. The ideas that make this work at petabyte scale — MapReduce's shuffle, Spark's in-memory DAG, columnar storage — are the foundation of modern data platforms.

The trade-off Exact, complete results over enormous datasets — in exchange for latency (minutes to hours) and the cost of moving data across the cluster. Batch gives you correctness and the ability to do complex historical joins over everything. You pay in latency — the answer is hours stale — and in the shuffle, the expensive all-to-all data movement that dominates big-data jobs.

What batch processing is

Batch processing runs a finite job over a bounded, complete dataset and produces a result, then stops — the opposite of streaming's never-ending computation. You reach for it when you need an exact answer over all the data (not a provisional one), when the computation is too complex or expensive to do per-event, or when latency simply doesn't matter (a nightly report, a weekly model retrain). "Big data" is just batch when the dataset exceeds one machine, forcing the work across a cluster — which introduces the central problem: moving data between machines is expensive.

Why it still matters in a streaming world: streaming gives freshness, batch gives completeness, exactness, and cost-efficiency at huge scale — reprocessing history, training models, reconciliation, and building derived datasets.

The one-line frameBatch trades latency for completeness, exactness, and throughput-per-dollar. You wait minutes-to-hours, but you get an exact answer over the entire dataset with arbitrary complexity — and the dominant cost you're managing is the shuffle, the all-to-all data movement that the whole field is organized around minimizing.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the job that's slow in the middle A big-data job reads fast, computes fast at the start, then spends 80% of its wall-clock time in a phase where the network is saturated and CPUs sit idle — between the "map" and "reduce" parts. What is that phase, why is it the bottleneck of nearly every big-data job, and what reduces it?
Scenario 2 — the analytics query that scans 200 columns to read 3 Your daily revenue report needs 3 columns out of a 200-column events table, over a billion rows. Stored as rows, the query reads all 200 columns off disk for every row. What storage layout would let it read only the 3 columns it needs, and roughly how much I/O does that save?
Scenario 3 — the same logic written twice (Lambda) A team wants both a real-time approximate count and a nightly exact recomputation. They build a streaming pipeline and a separate batch pipeline implementing the same aggregation logic in two different codebases. Six months later the two give subtly different numbers. What architecture is this, what's its core maintenance hazard, and what's the alternative?

Scenario 1 is the shuffle. Scenario 2 is columnar formats. Scenario 3 is Lambda vs Kappa. Each maps to an idea below.

The five ideas to understand

Five ideas carry the field. Each has a named failure nested under it.

Idea 1 — MapReduce and the shuffle

MapReduce (Google, 2004) is the foundational model: map each input record to key-value pairs in parallel across the cluster; shuffle moves all pairs with the same key to the same machine; reduce aggregates each key's group. Map and reduce are embarrassingly parallel — the magic and the cost are in the shuffle, which groups data by key across the whole cluster.

The failure that happens when you get this wrong — the shuffle bottleneck This is Scenario 1. The shuffle is an all-to-all data movement over the network plus disk spills — the most expensive part of nearly every big-data job, and the thing the entire discipline optimizes around. You reduce it by: filtering and aggregating before the shuffle (map-side combine / pushing predicates down), partitioning data so related keys are already co-located (avoiding a re-shuffle on joins), and avoiding unnecessary wide transformations. A job that shuffles the full dataset when it could have pre-aggregated is the classic big-data performance bug.

Idea 2 — Spark: DAGs and in-memory execution

MapReduce writes intermediate results to disk between every stage — slow for multi-step jobs (and disastrous for iterative ones like ML training). Spark models a job as a DAG (directed acyclic graph) of transformations on distributed datasets (RDDs/DataFrames) and keeps intermediate data in memory across stages, only spilling to disk when it must. It also tracks lineage, so a lost partition is recomputed from its inputs rather than from a checkpoint. Result: often 10–100× faster than MapReduce for multi-stage and iterative work.

The failure that happens when you get this wrong — spilling and skew Spark's speed assumes data fits in memory. When a stage's data exceeds memory it spills to disk, collapsing the in-memory advantage; and a skewed key (one key with vastly more rows than others) sends a huge partition to one executor while the rest finish — the job's tail is one straggler task. Fixes: size executor memory/partitions to the data, and for skew, salt the hot key (split it into sub-keys and combine after) or use a broadcast join for a small side. Skew is the Spark analog of the hot shard.

Idea 3 — Data lake, warehouse, lakehouse

Where does the data live? Data warehouse — structured, schema-on-write, optimized for SQL analytics (Snowflake, BigQuery). Data lake — raw files of any shape in cheap object storage, schema-on-read (S3 + Parquet). Lakehouse — the modern hybrid: lake storage + a table layer (Delta Lake, Apache Iceberg) that adds warehouse features (ACID transactions, schema enforcement, time travel) on top of open files.

The failure that happens when you get this wrong — the data swamp A data lake with no schema enforcement, catalog, or governance degrades into a data swamp: terabytes of files nobody can find, trust, or query, with no idea what's current or correct. Schema-on-read becomes schema-on-never. The lakehouse table formats exist precisely to fix this — they put a transactional, schema-enforced, discoverable table layer over the cheap files, so you get the lake's cost and flexibility without the swamp.

Idea 4 — Columnar formats and ELT

Analytics reads a few columns across many rows; transactional systems read whole rows. Columnar formats (Parquet, ORC) store each column contiguously, so a query reads only the columns it needs and compresses far better (similar values adjacent). This is why warehouses are column stores. And the modern pattern is ELT over ETL: load raw data into the warehouse/lake first, then transform in it with SQL (leveraging the warehouse's compute), rather than transforming before loading.

The failure that happens when you get this wrong — row storage for analytics This is Scenario 2. A row-oriented store forced to serve analytics reads every column of every row even when the query needs 3 of 200 — reading ~60× more data off disk than necessary, with poor compression. A columnar format reads only the 3 needed columns' contiguous blocks (and skips row groups via min/max stats), often cutting I/O by 50×+. Using a row store (or an OLTP database) for big analytical scans is a classic cost-and-latency mistake; analytics belongs in a columnar warehouse/lakehouse.

Idea 5 — Lambda vs Kappa architecture

How do you serve both real-time and exact results? Lambda architecture runs two layers: a speed layer (streaming, fast, approximate) and a batch layer (slow, exact, periodically recomputes from scratch), serving a merged view. Kappa architecture drops the batch layer: a single streaming pipeline over a replayable log handles both — to "recompute history" you just replay the log through the same code.

The failure that happens when you get this wrong — two codebases that drift This is Scenario 3. Lambda's core hazard is maintaining the same logic twice — once in the streaming layer, once in the batch layer — in different frameworks. They inevitably drift, and now your "real-time" and "corrected" numbers disagree with no clear source of truth. Kappa avoids it by using one replayable stream and one codebase: reprocessing is just replay. Lambda still wins when batch genuinely needs a different engine (huge historical joins streaming can't do); but if a single replayable pipeline can serve both, prefer Kappa to kill the dual-maintenance bug.

Famous implementations

The engines and stores made concrete. Same facets — what it is, the model, and the failure to watch.

Apache Spark

  • What it is. The dominant general-purpose big-data engine; batch, SQL, streaming, and ML in one framework.
  • Model. In-memory DAG over RDDs/DataFrames with lineage-based recovery; Catalyst optimizer pushes filters/aggregations before shuffles.
  • The failure to watch. Memory spills and data skew (Scenario 2's straggler); over-shuffling from poorly-planned joins.

Warehouses — BigQuery / Snowflake

  • What it is. Serverless/elastic columnar SQL warehouses that separate storage from compute.
  • Model. Columnar storage + massively-parallel query execution; you write SQL, they handle shuffle and parallelism; billed by scanned data or compute time.
  • The failure to watch. SELECT * and unpartitioned scans bill for the whole table; design schemas columnar and partition/cluster on filter columns.

Lakehouse — Delta Lake / Iceberg / Hudi

  • What it is. Open table formats that add ACID transactions, schema enforcement, and time travel over Parquet files in object storage.
  • Model. A transaction log over immutable data files gives consistent reads, upserts/deletes, and "query as of yesterday" — warehouse features on lake economics.
  • The failure to watch. Small-file proliferation needs periodic compaction; without the table layer's governance you're back to a swamp (Scenario's Idea 3).

Build it yourself

These click when you reason about the data movement. Work each before opening the approach.

Write a word count as MapReduce, then spot the shuffle

Express "count occurrences of each word across a terabyte of text" as map / shuffle / reduce, and identify where the network cost is and how to cut it.

Reveal the approach

Map: each node reads its chunk and emits (word, 1) for every word — fully parallel, no network. Shuffle: all (word, 1) pairs for the same word are moved to the same reducer — the all-to-all network cost. Reduce: each reducer sums the 1s for its words. The cut: add a map-side combiner that pre-sums each word locally before the shuffle — emit (word, localCount) instead of millions of (word, 1). For a terabyte of text with repeated words, this shrinks shuffle volume by orders of magnitude. Pre-aggregating before the shuffle is the single most important big-data optimization.

Fix a skewed join

You join a billion-row events table to a users table on user_id, but one bot account has 40% of all events. One Spark task runs for an hour while the rest finish in minutes. Fix it.

Reveal the approach

The bot's user_id is a hot key — all 40% of rows hash to one partition / one task (the straggler). Two fixes: (1) Salt the hot key — append a random suffix (0–N) to the bot's user_id on the events side and replicate the bot's user row N times with matching suffixes, so the join spreads across N tasks; combine results after. (2) If the users table is small, use a broadcast (map-side) join — ship the whole users table to every executor so there's no shuffle of the huge events table at all. Both attack the same root: don't let one key's data land on one machine. This is the hot-shard problem inside a batch job.
