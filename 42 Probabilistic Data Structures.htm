Probabilistic Data Structures

Trading a sliver of exactness for orders-of-magnitude less memory — the sketches that make web-scale membership, cardinality, and frequency queries feasible.

The bottleneck Finite capacity. trading exactness for massive space savings, paid in tunable false-positive/error rates.

Why approximate? The space–accuracy trade

An exact answer to "have I seen this URL?", "how many distinct users today?", or "what are the top-100 search terms?" requires storing every element — a hash set, a sorted dictionary, an exact counter per key. At billions of elements that is gigabytes of RAM, and RAM is the scarce resource on the hot path. Probabilistic data structures (sketches) answer the same questions with a bounded, tunable error while using kilobytes to a few megabytes that does not grow with the number of distinct elements.

The deal is always the same: you give up exactness and gain (1) sublinear or constant space, (2) constant-time updates and queries, and (3) usually mergeability — sketches built on different shards can be unioned, which makes them perfect for distributed and streaming systems. The discipline as a principal is knowing which error each structure tolerates and whether your use case can live with it.

Bloom filters & the sizing math

A Bloom filter is a bit array of m bits and k independent hash functions. To insert x, set the bits at positions h1(x) … hk(x). To test membership, check whether all k bits are set. If any is zero, x is definitely not present (no false negatives). If all are set, x is probably present — it could be a collision with other elements (a false positive).

insert(x):
    for i in 1..k:
        bits[ h_i(x) mod m ] = 1

contains(x):           # false positives possible, false negatives never
    for i in 1..k:
        if bits[ h_i(x) mod m ] == 0: return DEFINITELY_NOT
    return PROBABLY_YES

In practice you do not need k distinct hash functions — Kirsch–Mitzenmacher double hashing derives all k from two: gi(x) = h1(x) + i·h2(x), with negligible accuracy loss. This is what RocksDB, Cassandra, and Redis bitmaps actually do.

The sizing math you must be able to derive

Given n expected elements and a target false-positive probability p, the optimal parameters are:

m  =  - n * ln(p) / (ln 2)^2          # bits
k  =  (m / n) * ln 2                  # ≈ 0.693 * m/n  (round to int)
p  ≈  (1 - e^(-k n / m))^k          # achieved FP rate

The headline number to memorize: about 9.6 bits per element for a 1% false-positive rate, and roughly +4.8 bits per element for each 10× reduction in p. So 0.1% costs ~14.4 bits/element, 0.01% ~19.2 bits/element. A filter for 1 billion URLs at 1% FP is therefore ~1.2 GB — versus tens of GB for an exact set of the URLs themselves.

Key insightOptimal k sets the fill ratio to exactly 1/2 — half the bits set. That is the maximum-entropy point. Using more hash functions than optimal increases the FP rate because you saturate the array faster than the per-test stringency helps.
PitfallYou must size for the real n. A Bloom filter degrades silently: insert 5× your planned n and the array saturates toward all-ones, driving the FP rate toward 100% with no error thrown. Standard Bloom filters also cannot resize or delete. Production answers are scalable Bloom filters (a chain of filters with geometrically tightening p) or just rebuilding periodically.

Counting, cuckoo, and quotient filters

A plain Bloom filter cannot delete — clearing a bit might unset a bit shared with another element, creating false negatives. Three structures address this:

Counting Bloom filter

Replace each bit with a small counter (typically 4 bits). Insert increments, delete decrements. This supports deletion at ~4× the space of a standard Bloom filter. Counters can saturate; 4 bits (max 15) is empirically safe for normal load factors.

Cuckoo filter

Stores a short fingerprint of each element in a cuckoo hash table with two candidate buckets. Supports deletion, is more cache-friendly (one or two bucket reads vs. k scattered probes), and at FP rates below ~3% is more space-efficient than Bloom. The catch: inserts can fail when buckets fill and the cuckoo eviction chain hits its limit, so it has a hard load-factor ceiling (~95%).

Quotient filter

Stores fingerprints in a compact open-addressed table with metadata bits, giving better locality than Bloom (sequential memory access, so it works well on SSD/flash) and supporting merges and resizes. Used in some LSM and storage contexts where Bloom's random access pattern is the bottleneck.

Trade-offBloom: simplest, no delete, scattered reads. Counting Bloom: delete, 4× space. Cuckoo: delete + better locality + smaller at low FP, but insert can fail. Quotient: best locality & mergeable, slightly more complex. For an immutable per-SSTable filter (the common case in LSM storage engines) the lack of delete is irrelevant, so plain Bloom usually wins.

HyperLogLog: cardinality estimation

HyperLogLog (HLL) answers "how many distinct elements?" in fixed memory regardless of cardinality. The intuition: hash each element to a uniform bit string. The probability of seeing a hash with ρ leading zeros is 2-(ρ+1), so if the longest run of leading zeros you have ever observed is R, the stream probably held on the order of 2R distinct items. A single estimator is wildly noisy, so HLL uses stochastic averaging: the first p bits of the hash pick one of m = 2p registers, and each register keeps the max leading-zero count seen for its bucket. The harmonic mean across registers, with a bias-correction constant αm, yields the estimate.

add(x):
    h = hash64(x)
    idx = h >> (64 - p)              # top p bits select a register
    w   = h << p                     # remaining bits
    registers[idx] = max(registers[idx], leading_zeros(w) + 1)

estimate():                          # standard error  ≈ 1.04 / sqrt(m)
    Z = sum( 2^(-registers[j]) )
    E = alpha_m * m^2 / Z
    # small/large-range corrections apply; HLL++ uses 64-bit hash + bias tables

With m = 214 = 16384 registers of 6 bits each, HLL uses about 12 KB and delivers a standard error of 1.04/√16384 ≈ 0.81% — for cardinalities up into the billions. That is the basis of Redis's PFADD / PFCOUNT / PFMERGE (Redis implements the improved HLL++ with a sparse representation for small sets, capping at 12 KB dense). HLLs are losslessly mergeable by taking the element-wise max of registers, which is why they are ideal for "unique visitors across 1000 shards, last 30 days" — each shard/day keeps its own HLL and you union them at query time.

PitfallHLL gives cardinality, not the elements, and it cannot do reliable intersections — inclusion–exclusion (|A∪B| = |A|+|B|-|A∩B|) compounds the relative errors and explodes for similar-sized sets. If you need set intersection, reach for MinHash / theta sketches, not HLL.

Count-Min Sketch: frequency & heavy hitters

A Count-Min Sketch (CMS) estimates the frequency of each key in a stream using a 2-D array of counters: d rows (one hash function each) × w columns. To add x with weight c, increment count[i][hi(x) mod w] in every row. To query, take the minimum across rows — the min is the tightest bound because collisions can only ever add phantom counts.

add(x, c):
    for i in 0..d-1:
        count[i][ h_i(x) % w ] += c

estimate(x):                         # always ≥ true count (overestimate)
    return min over i of count[i][ h_i(x) % w ]

The guarantees: with w = ⌈e/ε⌉ and d = ⌈ln(1/δ)⌉, the estimate exceeds the true count by at most ε·N (N = total stream weight) with probability 1−δ. So error is relative to the total mass — CMS is accurate for heavy keys and noisy for the long tail, which is exactly the right bias for heavy-hitter detection. Pair CMS with a small top-k heap to track the most frequent items: on each update, query CMS and conditionally update the heap. This is how you find trending hashtags, hot keys, or DDoS source IPs in one pass over a firehose.

Key insightCMS only over-estimates, never under-estimates — the dual of a Bloom filter's one-sided error. The conservative-update variant (increment only the rows currently holding the minimum) cuts overestimation substantially in practice. A close cousin, the Count sketch, uses signed updates and the median estimator to give unbiased estimates with two-sided error — better for tail keys.

t-digest & HdrHistogram: percentiles

Computing exact percentiles (p50, p99, p99.9) over a latency stream requires sorting all samples — unbounded memory. Two structures solve this for monitoring and SLO work (see SLIs, SLOs & error budgets).

HdrHistogram

High Dynamic Range Histogram uses fixed log-linear buckets covering a configured range (e.g. 1µs to 1 hour) at a chosen number of significant figures. Recording a value and reading any percentile are O(1)/O(buckets), allocation-free on the hot path — which is why it is the standard for low-latency JVM/Go services and for coordinated-omission-corrected benchmarking. Error is bounded relative per value (e.g. 0.1% with 3 sig-figs). Histograms are trivially additive, so you merge per-host histograms into a fleet-wide one before computing p99.

t-digest

t-digest (Ted Dunning) adaptively clusters samples with much finer resolution near the tails (q→0 and q→1) than the middle, using a "scale function" to bound cluster sizes. It gives excellent accuracy at p99/p99.9 in a few kilobytes, is mergeable, and handles arbitrary unbounded value ranges (no need to pre-declare a max). It powers percentile aggregation in Elasticsearch, Spark, and many metrics backends.

PitfallYou cannot average percentiles. Taking the mean of per-host p99s is statistically meaningless and routinely wrong by large margins. Always merge the underlying histograms/digests, then compute the percentile. Mergeability is the whole reason these structures exist for distributed monitoring.

Where each shows up in real systems

  • LSM read path Every SSTable in RocksDB / Cassandra / HBase / LevelDB carries a Bloom filter so a point lookup can skip files that definitely lack the key — turning a multi-file disk read into a single bit check. This is the single most important use of Bloom filters in databases. See Storage Engines: LSM vs B-Tree.
  • Crawler dedup A web crawler uses a Bloom filter for the "URL seen?" set — a false positive merely skips re-crawling a page (acceptable), while no false negatives means you never loop. Billions of URLs in ~GB of RAM instead of TB.
  • Cache & CDN "One-hit-wonder" filtering: only admit a key to the cache on its second request (tracked by a Bloom filter) to avoid polluting it with single-access objects. Also used as a negative cache to short-circuit lookups for keys known absent.
  • Analytics & unique counts HLL backs "unique visitors / DAU / distinct queries" dashboards everywhere — Redis PFCOUNT, Presto/Trino approx_distinct, BigQuery APPROX_COUNT_DISTINCT, Druid. Per-shard HLLs merged at query time make distinct counts cheap across partitioned data.
  • Real-time aggregation In ad-click aggregation and stream processing, CMS finds heavy hitters (top advertisers, hot keys, fraud bursts) and HLL gives unique impressions/users per campaign — both in bounded memory in the stream layer (see Stream Processing & CDC).
  • Networking & security CMS for per-flow byte counts and DDoS source detection; Bloom filters in routers for forwarding tables; both in spam/abuse pipelines.

Principal-level mastery check

What strong engineers doThey reach for a sketch only after stating the exact-solution cost and naming the error the use case can absorb — "a false positive here just causes a redundant DB read, so a 1% Bloom filter is fine; a false negative would corrupt the index, so Bloom's one-sided error is exactly what I want." They quote the ~9.6 bits/element-per-1% rule of thumb and size for real n. They know which structure answers which question (membership vs. cardinality vs. frequency vs. quantile) and never confuse them. Crucially, they exploit mergeability for distributed rollups and call out the traps: you can't intersect HLLs, can't average percentiles, and can't delete from a plain Bloom filter.
  • A teammate proposes a Bloom filter to deduplicate financial transactions before writing them. What is wrong with that, and what does the one-sided error mean for correctness here?
  • You provisioned a Bloom filter for 100M elements but traffic grew to 500M. What happens to the false-positive rate, will you see an error, and how do you remediate without downtime?
  • Derive the optimal number of hash functions for a Bloom filter and explain why more hashes can make accuracy worse.
  • A dashboard shows "unique users" by averaging the p99 of per-region HLL counts. Identify both statistical errors in that sentence.
  • You need the top-100 search terms per minute over a 2M-QPS stream with bounded memory. Walk through CMS + heap, the error bound, and why you'd pick conservative update or a Count sketch for the tail.
  • For an LSM engine, would you put the per-SSTable filter in a Bloom, cuckoo, or quotient filter? Justify in terms of delete support, memory, and access pattern against flash.
