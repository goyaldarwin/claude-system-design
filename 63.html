Designing Ad Click Aggregation

Count ad clicks — billions per day — and aggregate them into near-real-time dashboards and exact billing. Two things make this hard and high-stakes: the scale (a firehose of events that no single database can ingest) and the fact that clicks are money — a few percent of miscounted clicks is millions of dollars, so duplicates and lost events are unacceptable. It's the canonical stream-processing design with an exactly-once requirement.

The crux Fast, accurate aggregation of a massive event firehose — in exchange for solving exactly-once counting (dedup), the event-time vs processing-time problem, and the fast-but-approximate vs slow-but-exact tension. You get live metrics and correct billing over billions of events. You pay by deduplicating to count each click once (money), handling late/out-of-order events with event-time windows, and reconciling a fast approximate path against a slow exact one.

1. Requirements

Functional: ingest ad-click events (adId, userId, timestamp, context); aggregate counts (and spend) per ad / campaign / time-window; serve near-real-time dashboards (clicks in the last minute/hour) and exact aggregates for billing; support filtering (by region, etc.).

Non-functional: massive scale (billions of clicks/day — a firehose), accuracy (each click counted exactly once — it's money), low query latency for dashboards, fault-tolerant (no lost events), handles late/out-of-order events (mobile clicks arrive late).

2. Capacity estimate

Say 10 billion clicks/day, each event ~100 bytes, with bursty peaks. Work it:

  • Average rate: 10B / 86,400s ≈ ~115,000 clicks/sec average; peak (prime time) maybe 3–5× → ~500,000/sec. Consequence: no single database can ingest this directly — you need a buffer (a log) and a parallel stream processor.
  • Raw volume: 10B × 100 B = 1 TB/day of raw events. You keep raw events (for re-aggregation / exact recompute / audit) in cheap object storage, but you don't query raw — you query pre-aggregated rollups. Consequence: store raw cheaply, serve from rollups.
  • Query shape: dashboards want "clicks per ad per minute" — tiny compared to raw. Pre-aggregating into per-minute (and per-hour/day) rollups turns a billion-row scan into a handful of rows. Consequence: aggregate on write (streaming), not on read.
  • The money constraint: at this scale, a 1% counting error = ~100M miscounted clicks/day = potentially millions in mis-billing. Consequence: exactly-once is a hard requirement, not a nicety — it drives the whole design.

3. The API

ingest({ clickId, adId, userId, eventTime, context })   // clickId = dedup key
query({ adId | campaignId, from, to, granularity })      -> [{ window, clicks, spend }]

ingest is high-throughput, fire-and-forget into the stream. The clickId is the idempotency/dedup key — the linchpin of exactly-once counting (sub-problem 2). eventTime is the time the click happened (not when ingested), driving event-time windowing. query reads pre-aggregated rollups, not raw events.

4. High-level design

The classic streaming pipeline: (1) click events are written to a durable log (Kafka/Kinesis) — this absorbs the firehose and decouples ingest from processing. (2) A stream processor (Flink/Spark Streaming) consumes the log, deduplicates by clickId, and aggregates clicks into time-windowed rollups (per ad, per minute), writing them to a fast query store. (3) Dashboards query the rollup store; raw events are also archived to object storage for exact re-aggregation/audit. The stream processor checkpoints its state so a crash doesn't lose or double-count in-flight aggregates (exactly-once stream state).

5. Data model — raw events + rollups

StoreHoldsUsed for
Event log (Kafka)Raw click events, partitioned by adId, retained daysBuffer + replay/reprocess
Rollup store (time-series / OLAP)Pre-aggregated counts per (adId, window) at minute/hour/day granularityFast dashboard & billing queries
Raw archive (object storage)All raw events, columnar (Parquet)Exact batch re-aggregation, audit
Dedup storeSeen clickIds (windowed/Bloom)Exactly-once counting

Rollups are typically stored hierarchically (minute → hour → day) so a query for a day reads daily rollups, not 1,440 minutes — query cost stays flat regardless of range.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the same click counted twice = a wrong bill A client retries a click event after a network blip, and the stream processor reprocesses a redelivered message after a restart — so one real click is counted as two or three. Multiply by billions and a 1% error is millions in over-billing. Why are duplicates inevitable in this pipeline, and how do you count each click exactly once at this scale?
Scenario 2 — the mobile click that arrives 5 minutes late A user clicks an ad at 10:00:30 on their phone, which is offline; the event reaches your pipeline at 10:05. If you bucket it by arrival time, the 10:00 minute is undercounted and 10:05 is overcounted — and advertiser reports are wrong. Which notion of time should you bucket by, and how do you know when a minute's count is final?
Scenario 3 — the dashboard says 1.2M, the bill says 1.18M Your real-time dashboard (fast, approximate) shows 1.2M clicks for an ad; the billing system (slow, exact) computes 1.18M. They disagree, and advertisers see both. Why do the fast and exact paths differ, what architecture deliberately runs both, and which one do you bill on?

Scenario 1 is exactly-once counting. Scenario 2 is event time & late clicks. Scenario 3 is fast vs exact (Lambda/Kappa). Each maps to a sub-problem.

The four hard sub-problems

The pipeline shape is standard streaming. These four are where ad aggregation is hard (and where money rides) — each with a nested failure.

Sub-problem 1 — Ingesting the firehose

500k events/sec can't go straight into a database — you'd overwhelm it and couple ingest to processing speed. The fix is a durable log in front: it accepts writes at firehose rate (sequential appends, partitioned), buffers bursts, and lets the stream processor consume at its own pace — with replay if the processor falls behind or needs reprocessing.

The failure that happens when you get this wrong — overwhelming the store Writing clicks directly to your aggregate database (or even your stream store synchronously from the ad server) means a traffic spike either drops events (lost money) or backs up and stalls the ad-serving path. The log decouples them: ad servers fire events into Kafka/Kinesis (which is built for this throughput and buffers durably), and the aggregation runs asynchronously downstream. Partition the log by adId so per-ad events are ordered and the load spreads. If the processor lags, events wait safely in the log (retained for days) rather than being lost — and you can replay to reprocess. Direct-to-database ingest is the scale failure; the durable log is the absorber.

Sub-problem 2 — Exactly-once counting (deduplication)

Duplicates come from everywhere: the ad client retries, the log delivers at-least-once, the stream processor reprocesses after a crash. Since clicks are money, each must be counted once. The mechanism: each click carries a unique clickId; before counting, check it against a dedup store — if seen, drop it. At billions/day, the dedup store uses a windowed approach (recent clickIds) often fronted by a Bloom filter for a cheap "definitely new?" check.

The failure that happens when you get this wrong — miscounting = money lost This is Scenario 1. Without dedup, the inevitable duplicates inflate counts — and at this scale a small duplicate rate is a large mis-billing (advertisers over-charged, or you under-counting and losing revenue). "Exactly-once delivery" is impossible, so you build exactly-once counting: dedup on clickId. At billions/day you can't keep every id forever, so use a time-windowed dedup (most duplicates arrive close together — keep ids for a window like 24h) backed by a Bloom filter (a fast "probably seen?" check; on "maybe seen" do an exact check) and the stream processor's exactly-once state (checkpoints so a restart doesn't re-aggregate already-counted events). The combination — dedup key + windowed store + Bloom + checkpointed processing — gives exactly-once counting without storing every id forever. This is the single highest-stakes sub-problem.

Sub-problem 3 — Event time & late clicks

Aggregating "clicks per minute" requires deciding which minute — and the answer must be event time (when the click happened, stamped at the source) not processing time (when you got it). Late and out-of-order events (mobile, retries) are normal, so you use event-time windows with a watermark to decide when a window is "done enough" to emit, plus an allowed-lateness grace.

The failure that happens when you get this wrong — wrong-window counts This is Scenario 2. If you bucket by processing time, the 10:00:30 click that arrives at 10:05 lands in the 10:05 minute — so the 10:00 minute is undercounted and 10:05 overcounted, and a replay or backlog scrambles every number (advertiser reports become untrustworthy). Bucketing by event time puts the click in the 10:00 window regardless of arrival — correct and replay-stable. The cost: you can't finalize the 10:00 window the instant the clock hits 10:01, because late clicks may still arrive. A watermark ("event time has advanced to T") plus an allowed-lateness window (e.g. accept clicks up to a few minutes late, emit corrections) resolves it — finalize after the grace period, send very-late stragglers to a correction path. Event time + watermarks is non-negotiable for correct time-series counts.

Sub-problem 4 — Fast vs exact (the Lambda/Kappa decision)

You want both a real-time approximate count (dashboards, in seconds) and an exact count (billing, can be minutes-to-hours late). These have different correctness/latency needs, which is the classic Lambda vs Kappa choice: run two layers (speed + batch) or one replayable stream that you reprocess for exactness.

The failure that happens when you get this wrong — the approximate becomes the billed number This is Scenario 3. The fast path (streaming, maybe with approximate counts / not-yet-final windows) and the exact path (full reprocessing of raw events with all late data and dedup) legitimately differ — the fast number is provisional. The danger is treating the approximate dashboard number as the billing number, or having two pipelines (Lambda's speed + batch layers) that drift because they're separate codebases. Resolution: be explicit that dashboards are provisional (live, approximate) and billing uses the exact reprocessed figure (after all late data + dedup), computed from the raw archive. Prefer Kappa where you can — one replayable stream + reprocessing — so the "exact" path is just the same code replayed over the durable log, eliminating the two-codebase drift. Bill on the exact, reconciled number; show the approximate one clearly labeled as live. Never bill on the provisional count.

Famous implementations

Same facets — what it is, the model, and the failure to watch.

Flink (exactly-once streaming)

  • What it is. The go-to stream processor for exactly-once, event-time aggregation.
  • Model. Event-time windows + watermarks; distributed checkpoints give exactly-once state; keyed aggregation per adId.
  • The failure to watch. Large dedup/window state needs tuning; exactly-once within Flink + idempotent sink, but external dedup (clickId) is still your design.

Kappa pipelines

  • What it is. A single replayable streaming pipeline serving both live and (by reprocessing) exact results.
  • Model. One codebase over a durable log; "recompute exactly" = replay the log through the same processor.
  • The failure to watch. Requires the log to retain enough history to reprocess; if exact needs a fundamentally different engine, Lambda may still be warranted.

OLAP rollup stores (Druid / ClickHouse / BigQuery)

  • What it is. Columnar stores built for fast aggregate queries over huge event volumes.
  • Model. Ingest pre-aggregated (or raw) rollups; serve dashboard queries in ms via columnar + pre-rollup.
  • The failure to watch. Querying raw at billions of rows is slow/costly — pre-aggregate into hierarchical rollups; mind cardinality of group-by dimensions.

Build it yourself

Work each before opening the approach.

Count each click exactly once at billions/day

Duplicates arrive from client retries and stream reprocessing. Design dedup that counts each clickId once without storing every id forever.

Reveal the approach

Use the clickId as the dedup key. Most duplicates arrive within a short window of the original (retries, redelivery), so keep a time-windowed set of seen ids (e.g. last 24h) rather than all-time. Front it with a Bloom filter: check the filter first — "definitely not seen" → it's new, count it and add it; "maybe seen" → do an exact lookup in the windowed store to confirm. The Bloom filter makes the common "new click" path cheap (one in-memory check) and bounds memory; the exact store resolves the rare "maybe." Combine with the stream processor's checkpointed exactly-once state so a restart doesn't re-count events between the last checkpoint and the crash. For the exact billing recompute, dedup over the full raw archive (a batch job can afford an exact all-ids dedup). Result: exactly-once counting on the live path (windowed dedup) and provably-exact on the billing path (full dedup) — without storing every id forever in the hot path.

Aggregate per-minute by event time with late data

Clicks can arrive minutes late. Build a per-minute count keyed by event time that's correct and knows when a minute is final.

Reveal the approach

Window by event time (the click's source timestamp) into 1-minute tumbling windows — so a click at 10:00:30 always lands in the 10:00 window no matter when it arrives (Scenario 2). Track a watermark = max-seen-event-time minus an allowed-lateness slack (sized to your real tail, say 2–5 min). Emit a minute's count when the watermark passes the minute's end; keep accepting late clicks within the allowed-lateness window and emit updated corrections; route very-late stragglers (beyond the window) to a separate correction/late-data path that adjusts the rollup. So the per-minute count is provisionally available quickly and finalized after the grace period — correct, replay-stable, and explicit about finality. The billing recompute over the raw archive captures all late data exactly. This is the event-time + watermark pattern (40) applied to clicks.
