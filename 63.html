Designing Ad Click Aggregation

A money-correct, near-real-time pipeline that ingests billions of click events a day, tolerates late and duplicate events, and serves both sub-minute dashboards and audited billing totals.

The bottleneck Finite capacity & latency. huge write volume + near-real-time counts, paid in streaming + batch + probabilistic structures.

Problem framing & why it’s hard

An ad click aggregation system sits between two unforgiving consumers. Advertisers and dashboards want fresh numbers — clicks, impressions, click-through rate, spend — with sub-minute latency so they can pause a campaign before it burns budget. Billing and finance want the same numbers to be exactly right, auditable, and stable months later. Those two goals pull in opposite directions: speed favors approximate, in-memory, eventually-consistent aggregation; correctness favors deterministic, replayable batch computation.

This is the canonical problem for understanding stream/batch duality. A weak answer reaches for “just count in Redis.” A strong answer treats the raw event log as the source of truth, builds fast approximate views on top, and keeps a slower authoritative recompute path for the money.

Key insight The hard part is not throughput — Kafka and a stream processor handle billions of events trivially. The hard part is correctness under failure: events arrive late, out of order, and duplicated, and the billing total must still tie out to the penny.

Requirements & scale estimation

Functional

  • Ingest click (and impression) events: {ad_id, campaign_id, advertiser_id, user_id, click_id, ts, ip, country, device}.
  • Serve aggregated metrics by dimension (ad, campaign, advertiser, country) over time buckets (per-minute, per-hour, per-day).
  • Return top-N ads/campaigns by clicks (heavy hitters) and unique-user counts (distinct).
  • Near-real-time dashboards: freshness target ~1 minute.
  • Authoritative billing totals: recomputed and reconciled, correct to the exact count.

Non-functional

  • High write volume, low read volume. Writes dominate by orders of magnitude.
  • Tolerate late events (mobile devices offline for hours) and duplicates (client retries, at-least-once delivery).
  • Durable: no event silently dropped. Replayable: any time window can be recomputed from raw.
  • Available for reads even during a region failure.

Event schema & query API

The producer-supplied click_id is the linchpin of dedup — it must be generated once at click time (see Unique ID Generator) and carried through retries unchanged.

// Ingested event (Avro)
ClickEvent {
  click_id:    string   // idempotency key, stable across retries
  ad_id:       long
  campaign_id: long
  advertiser_id: long
  user_id:     long      // hashed
  event_ts:    long      // EVENT time (when click happened, ms)
  ingest_ts:   long      // PROCESSING time (when we received it)
  country:     string
  device:      string
  ip:          string
}

// Read API
GET /v1/metrics?dimension=campaign&id=42
     &metric=clicks&from=2026-06-12T00:00&to=2026-06-12T23:59&granularity=1h
GET /v1/topn?dimension=ad&metric=clicks&window=1h&n=100
GET /v1/distinct?dimension=campaign&id=42&metric=unique_users&window=1d
Pitfall Distinguish event time (when the click physically happened) from processing time (when your pipeline saw it). Aggregating by processing time is trivial but wrong for billing — an event that arrives 3 hours late must land in the hour it occurred, not the hour it arrived. Everything downstream keys on event_ts.

Ingestion: Kafka as the source of truth

Collectors are thin, stateless, horizontally scaled behind a load balancer. Their only job: validate, enrich (geo from IP), serialize, and produce to Kafka. Kafka is deliberately the durable, replayable source of truth — not the collector’s memory, not the database. Retain raw events 7–30 days in Kafka and indefinitely in cold object storage (S3/HDFS) partitioned by event_hour.

  • Partitioning: partition by a key that spreads load and co-locates aggregation. Keying by ad_id lets a single stream task own all clicks for an ad (cheap local aggregation) but invites skew when one ad goes viral (see below). A common compromise: partition by hash(ad_id) for spread, accept that a stream operator must shuffle/rekey before aggregating.
  • Replication factor 3, acks=all so a broker loss never drops an event. For billing, durability beats latency on the write path.
  • Producer idempotence + transactional writes (Kafka enable.idempotence=true, transactional producer) eliminate broker-side duplicates from producer retries — but not application-level duplicate clicks. Those are handled later.
Key insight Persisting raw events to a log before any aggregation is the single most important decision. It decouples ingest from processing, absorbs spikes, lets you replay to fix bugs, and gives the batch path an authoritative input. This is the same log-as-source-of-truth principle from Stream Processing and CDC.

Stream processing: windows & watermarks

The speed path uses a stream processor (Flink is the reference; Kafka Streams or Spark Structured Streaming work too) to maintain windowed aggregates keyed by dimension. Three concepts do the heavy lifting:

Tumbling event-time windows

Aggregate into fixed, non-overlapping buckets keyed on event_ts — e.g. 1-minute tumbling windows per (campaign_id, country). Output incremental counts as windows fire.

Watermarks

A watermark is the processor’s assertion that “I believe I have seen all events with event_ts ≤ W.” It is computed as max(event_ts seen) − allowedLateness. When the watermark passes a window’s end, the window fires and emits its result. Watermarks are how a stream system reasons about completeness in the face of out-of-order arrival.

stream
  .assignTimestampsAndWatermarks(
     forBoundedOutOfOrderness(Duration.ofMinutes(2)))   // tolerate 2 min lateness
  .keyBy(e -> e.campaign_id)
  .window(TumblingEventTimeWindows.of(Time.minutes(1)))
  .allowedLateness(Time.hours(1))      // late events still update the bucket
  .sideOutputLateData(lateTag)          // VERY late events -> correction stream
  .aggregate(new CountAggregator());
Trade-off Watermark lateness is a latency-vs-completeness dial. A 2-minute bound means windows fire 2 minutes after their end — fresh, but events later than 2 min are “late.” A larger bound captures more stragglers but delays every result. There is no setting that is both perfectly fresh and perfectly complete; that gap is exactly why the batch path exists.

Mobile clients buffered offline can deliver clicks hours late. You cannot hold a window open for hours. Practical answer: emit a best-effort result at watermark time, route very-late events to a correction stream that issues delta updates to the store, and rely on the nightly batch recompute to make the final number authoritative. The deeper treatment of windowing, watermarks, and late-data handling lives in Stream Processing and CDC.

Exactly-once & the duplicate-click problem

Two distinct duplication sources, two distinct mechanisms:

1. Pipeline-internal exactly-once

If a stream task crashes and restarts, will it double-count? Flink’s checkpoint + two-phase-commit sink gives end-to-end exactly-once for the pipeline: operator state and Kafka offsets are checkpointed atomically; on recovery the processor rewinds to the last checkpoint and replays, and the transactional sink ensures emitted results are committed exactly once. Combined with Kafka transactional reads/writes this closes the internal gap.

2. Application-level duplicate clicks

The user’s device retried the beacon; the same physical click arrives as N events. Exactly-once processing will faithfully count all N — they are genuinely distinct messages. The defense is idempotency on click_id:

  • Streaming dedup window: keep a set of recently-seen click_ids in operator state (RocksDB-backed, keyed, TTL’d to e.g. 24h). First sighting counts; later sightings are dropped. Bounded memory because state is TTL’d.
  • Probabilistic pre-filter: front the exact set with a Bloom filter to cheaply reject definite-new IDs without a state lookup. A Bloom filter never yields false negatives, so a “not seen” answer is trustworthy; only “maybe seen” needs the exact check. (See Probabilistic Data Structures.)
  • Batch dedup: the recompute path deduplicates by GROUP BY click_id (or DISTINCT click_id) over the full raw log — exact and immune to window-boundary edge cases. This is the authoritative dedup; the streaming filter is best-effort for freshness.
Pitfall A duplicate that arrives after the streaming dedup window has expired (e.g. the same click_id 26 hours later) will slip past the stream filter. This is acceptable only because the nightly batch recompute over the immutable raw log is the source of truth for billing. Never bill directly off streaming counts.

Distinct counts & heavy hitters

Two metrics are infeasible to compute exactly at scale in real time, so we use sketches:

Unique users — HyperLogLog

Counting distinct users per campaign exactly would require holding every user_id in a set — gigabytes per high-traffic campaign. HyperLogLog estimates cardinality in ~1–2 KB with a few-percent error, and crucially HLL sketches are mergeable: per-minute sketches union into per-hour and per-day sketches without re-reading raw data. Store the sketch in each rollup bucket, not just the scalar count.

Top-N ads — Count-Min Sketch

Top-N by clicks over a sliding window is a heavy-hitters problem. A Count-Min sketch tracks approximate per-key frequencies in fixed memory (it over-counts, never under-counts), paired with a small heap of current top-N candidates to surface the top-N. Exact top-N would need a counter per ad_id — fine for thousands of ads, prohibitive when there are tens of millions of long-tail creatives.

Trade-off Sketches trade bounded memory and mergeability for bounded error. That is the right trade for dashboards and trend detection. It is the wrong trade for billing, where ~2% error on spend is unacceptable — billing-critical counts (total clicks) are kept exact via the batch path. Know which metrics tolerate approximation and which do not.

Time-series storage & rollups

Serve queries from a columnar OLAP / time-series store — Apache Druid, ClickHouse, or Apache Pinot — not from the operational database. These engines are built for exactly this access pattern: high-cardinality dimensional filters over time ranges, columnar compression, and ingestion-time rollup.

  • Pre-aggregation / rollups: the stream writes per-minute granules; a rollup job (or the store’s native rollup) folds minute → hour → day. A “clicks last 30 days” query reads ~30 day-granules, not ~43 K minute-rows.
  • Retention tiers: minute granularity for ~7 days, hour for ~90 days, day forever. Raw events live only in the cold log.
  • Store sketches alongside scalars so distinct-count and top-N queries remain mergeable across the rollup hierarchy.

Why not a generic SQL or NoSQL store? An RDBMS chokes on this write rate and full-scan aggregation; a plain KV store can hold pre-aggregated counters but can’t answer ad-hoc dimensional roll-ups efficiently. A purpose-built OLAP engine with columnar indexing wins decisively.

Lambda vs Kappa: reconciliation for billing

This is the architectural crux. Two patterns:

LambdaKappa
PathsSpeed (stream) + Batch (recompute)Stream only; recompute by replaying the log
Correctness fixNightly batch overwrites approximate streaming resultsRe-run the stream job over historical log offsets
CostTwo codebases / engines to maintainOne engine; reprocessing is just replay
Best whenStrict audited correctness needed (billing)Logic is purely stream-expressible; ops simplicity prized

For ad billing the pragmatic answer is Lambda, or Kappa-with-replay. The speed path serves dashboards within a minute (approximate, dedup best-effort, sketch-based distincts). A batch recompute (Spark over the raw S3/HDFS log, partitioned by hour) runs on a schedule and produces the authoritative, exact, fully-deduplicated totals. Reconciliation overwrites the streaming numbers for closed time windows; billing reads only reconciled buckets.

-- Batch recompute (authoritative), run hourly/nightly over raw log
SELECT campaign_id, date_trunc('hour', event_ts) AS bucket,
       COUNT(DISTINCT click_id)        AS exact_clicks,   -- dedup is exact here
       approx_count_distinct(user_id)  AS unique_users
FROM   raw_clicks
WHERE  event_ts >= :window_start AND event_ts < :window_end
GROUP  BY campaign_id, bucket;
-- result UPSERTs into the serving store, marking bucket = FINALIZED
Key insight Modern practice trends toward Kappa: keep one streaming codebase, and when you need a correctness recompute, replay the immutable Kafka/object-store log through the same job with relaxed (or infinite) lateness. You get a single source of truth for logic. Whether you call it Lambda or Kappa, the non-negotiable is: billing reads reconciled results derived from the immutable raw log, never live streaming state.

Hot partitions & skew

A Super Bowl ad or a celebrity tweet drives one ad_id to a huge fraction of total traffic. If you partition by ad_id, one Kafka partition and one stream task melt while the rest idle — the classic hot-partition problem.

  • Key salting: partition on (ad_id, salt) where salt ∈ [0, N), spreading a hot ad across N partitions, then sum the N partial counts in a second aggregation stage. Use a small N only for detected hot keys to avoid blowing up cardinality everywhere.
  • Two-phase (combine then reduce) aggregation: pre-aggregate locally per task (a local combiner cuts the shuffle volume by orders of magnitude), then merge partials by key. Skew on the final merge for one key is small because the input is already partial sums, not raw events.
  • Adaptive rebalancing: detect hot keys at runtime (a Count-Min sketch of key frequencies works) and dynamically increase their salt fan-out.
  • Backpressure: Kafka’s durable buffer plus the processor’s backpressure absorb spikes so the collector never has to drop — the log is the shock absorber. See Resilience Patterns.
Pitfall Consistent-hash partitioning (Consistent Hashing) spreads keys evenly but does not save you from a single hot key — all events for one key still hash to one partition. Skew on a single key needs salting or two-phase aggregation, not just a better hash function.

Principal-level mastery check

What strong engineers do They immediately separate the fast-approximate path from the slow-exact path and explain why billing cannot trust streaming counts. They reason in event time vs processing time and name watermarks as the completeness mechanism. They distinguish pipeline-exactly-once (checkpoints + transactional sinks) from application-level duplicate clicks (idempotency on click_id). They reach for HLL/Count-Min only where approximation is acceptable and keep billing counts exact. They anticipate hot partitions and propose salting / two-phase aggregation before being prompted. They treat the raw log as the immutable source of truth that makes every correction a replay.
  • Your dashboard shows 1,000,000 clicks for a campaign at 11:59; the nightly batch recompute says 985,000. Walk me through every reason the numbers differ, and which one is correct for billing — and why.
  • A mobile SDK buffers clicks offline and flushes them 6 hours later. Trace one such event through your pipeline. Where does it land, what fires, and what gets corrected?
  • How do you guarantee a click retried 4 times is counted once in real time? Once in billing? Are those the same mechanism — and what happens if the retry arrives after your dedup window expires?
  • One ad gets 40% of all traffic during a live event. Show me exactly where the system buckles and how you keep aggregation balanced without losing per-ad accuracy.
  • You found a bug that mis-attributed clicks for the last 3 days. How do you recompute correct totals, and what does your architecture need for that to be cheap and safe?
  • Why not skip the stream processor and just batch every 5 minutes? Why not skip batch and serve billing straight off the stream? Defend the dual path on cost.
