Proxies

Proxy servers as intermediaries — types, caching, collapsed forwarding, and where they sit in a system.

1. What is a proxy server?

A proxy server is an intermediary piece of hardware or software that sits between a client and the back-end (or origin) server. It receives requests from clients and relays them to the origin server. Typically, proxies are used to filter requests, log requests, or sometimes transform requests — by adding or removing headers, encrypting or decrypting, or compressing a resource. Because the proxy stands in the middle of every request and response, it is a natural place to enforce policy, observe traffic, and reshape payloads without changing either the client or the origin.

Another advantage of a proxy server is that its cache can serve a lot of requests. If multiple clients access a particular resource, the proxy server can cache it and serve all subsequent clients from that local copy without having to reach out to the remote server every time. This both speeds up the response for the client and reduces load on the origin.

Key insight A proxy is fundamentally about indirection: by inserting a hop between client and origin you gain a single, controllable choke point where caching, filtering, transformation, logging, and security can all be applied — without modifying the endpoints themselves.
2. Proxy server types

Proxies come in several flavours depending on where they sit and whose interests they serve. The two top-level categories are the open (forward) proxy and the reverse proxy.

Open proxy. An open proxy is a proxy server that is accessible by any user on the internet. In general, a proxy server only allows users within a network group to store and forward internet services such as DNS or web pages, in order to reduce and control the bandwidth used by the group. An open proxy, however, lets any user on the internet use this forwarding service. There are two well-known kinds of open proxies:

  • Anonymous proxy — This proxy reveals its identity as a server but does not disclose the initial IP address of the client. Although such a proxy server can be discovered fairly easily, it is beneficial for users who want to hide their IP address from the origin.
  • Transparent proxy — This proxy server identifies itself, and with the support of HTTP headers the initial IP address of the client can still be seen. The main benefit of using this kind of server is its ability to cache websites. A transparent proxy attaches the client’s real address in a header such as X-Forwarded-For, so the origin can still see who the original requester was — which is precisely what makes transparent proxies useful for caching with no client-side configuration.

Reverse proxy. A reverse proxy retrieves resources on behalf of a client from one or more servers. These resources are then returned to the client, appearing as if they had originated from the proxy server itself. In contrast to a forward proxy — which sits in front of clients and is an intermediary for their requests to any server — a reverse proxy sits in front of one or more origin servers and is an intermediary for requests coming into those servers. Reverse proxies are commonly used to:

  • Cache responses, so repeated requests for the same resource are served from the proxy rather than the origin.
  • Load balance incoming requests across a pool of origin servers (see Load Balancing).
  • Hide the origin — clients never address the back-end servers directly, which shields internal topology and IP addresses.
  • Terminate SSL/TLS — decrypting HTTPS at the edge so origin servers can deal in plaintext and offload the cryptographic cost.
Forward vs reverse proxy The distinction is whom the proxy represents. A forward proxy acts on behalf of the client — it sits at the edge of the client’s network, forwards their requests outward, and hides the client from the origin. A reverse proxy acts on behalf of the server — it sits at the edge of the server’s network, accepts requests inward, and hides the origin from the client. Same machinery, opposite direction, opposite party protected.
3. Caching and collapsed forwarding

Because a proxy is a shared choke point, it is an ideal place to cache. A proxy cache can absorb a large fraction of read traffic: the first client that requests a resource pays the cost of fetching it from the origin, and every subsequent client is served the cached copy until it expires. This is closely related to the broader caching strategies discussed in Caching — the proxy simply moves the cache to a network hop that is shared by many clients.

A particularly important proxy behaviour is collapsed forwarding (also called request coalescing). When many clients request the same uncached resource at nearly the same moment, a naive proxy would forward every one of those requests to the origin — a stampede that can overwhelm the back-end exactly when a popular item goes cold (the “thundering herd” or cache-stampede problem). With collapsed forwarding, the proxy detects that an identical request is already in flight, issues only a single request to the origin, and fans the one response back out to all the waiting clients. This collapses N concurrent identical requests into one origin fetch, protecting the origin from sudden spikes on hot keys.

What strong engineers do Strong engineers name collapsed forwarding / request coalescing explicitly when discussing how an edge layer protects the origin from a cache stampede, and they connect it to TTL choices and stale-while-revalidate. They also cleanly separate the two reasons to put a proxy in front of an origin: performance (caching, coalescing, compression) and control (SSL termination, header rewriting, auth, rate limiting).
4. Principal-level mastery check

At principal level, “add a proxy” is rarely the answer by itself — the interesting discussion is which proxy, where it sits, and what it owns versus the adjacent components. Be ready to reason about these probes:

  • Reverse proxy vs load balancer — are they the same thing? They overlap heavily and are often the same box (nginx, Envoy, HAProxy all do both), but conceptually a load balancer’s primary job is distributing connections across a healthy pool, while a reverse proxy’s primary job is mediating and transforming requests (caching, TLS, header rewriting). Where do you draw the boundary in your design, and why? See Load Balancing.
  • Where do you terminate TLS, and what are the consequences? Terminating SSL at the reverse proxy offloads crypto from origins and centralizes certificate management, but it means traffic between the proxy and origin is plaintext unless you re-encrypt (TLS passthrough vs TLS termination vs TLS re-origination). How does this interact with zero-trust and with regulatory requirements for end-to-end encryption?
  • How do you prevent a cache stampede on a hot key? Collapsed forwarding, request coalescing, stale-while-revalidate, and TTL jitter are the tools. Walk through exactly what happens at the proxy the instant a viral item expires — see Caching for the cache-side mechanics.
  • What is your cache key, and what makes a response cacheable? Method, path, query string, Vary headers, cookies, and auth state all affect cacheability. How do you avoid caching per-user responses, and how do you handle Cache-Control and X-Forwarded-For correctly behind multiple proxy hops?
  • What happens when the proxy itself fails or becomes the bottleneck? A reverse proxy is now in the critical path for every request. How do you make it redundant, how do you health-check it, and how do you stop it becoming a single point of failure — or a place where one slow origin exhausts all proxy connections (connection-pool / head-of-line blocking)?
