Storage Engines: LSM-Trees vs B-Trees

How databases actually put bytes on disk — the single decision that quietly dictates your system's write throughput, read latency, and storage bill.

Why the storage engine is the foundation

A storage engine is the component that turns logical operations (get(key), put(key, value), range scans) into physical I/O against disk and memory. Two families dominate: B-Trees (MySQL/InnoDB, PostgreSQL, most relational systems, plus key-value stores like BoltDB and LMDB) and LSM-Trees (Cassandra, ScyllaDB, RocksDB, LevelDB, HBase, and the storage layer under CockroachDB, TiKV, and many time-series databases).

This choice is not a detail you bolt on later. It determines whether your database is happiest under write-heavy or read-heavy load, how much SSD you burn, how predictable your tail latency is, and even how your operational pages look at 3am. When you design a system on top of a database, you are inheriting its storage engine's physics. A principal engineer should be able to reason about those physics directly rather than treating the database as a black box.

Key insight Both families exist to solve the same enemy: random I/O is slow, sequential I/O is fast. They make opposite bets. B-Trees mutate data in place (random writes, sequential-ish reads). LSM-Trees only ever append (sequential writes, scattered reads they then claw back with indexes, caches, and Bloom filters).

B-Trees: the read-optimized incumbent

A B-Tree (almost always a B+Tree in real systems) is a balanced, high-fanout tree stored as fixed-size pages (typically 4 KB, 8 KB, or 16 KB — InnoDB defaults to 16 KB). Internal pages hold keys plus pointers to child pages; leaf pages hold the actual rows (in a clustered index) or pointers to them. Fanout is high — a single 16 KB page can hold hundreds of keys — so even a billion-row table is only ~4 levels deep. That means a point lookup is roughly 4 page reads, with the upper levels almost always cached. This is closely tied to how database indexes are physically laid out.

In-place updates

The defining property of a B-Tree is that updates mutate the page where the key already lives. To update a row you find its leaf page, modify it, and write the page back. This keeps the tree authoritative and read-optimal — one place to look for any key — but it forces random writes: the page you must overwrite could be anywhere on disk.

The torn-page problem and the WAL

Overwriting a page is dangerous: a crash mid-write can leave a half-written page (a torn page). B-Trees solve this with a write-ahead log (WAL, called the redo log in InnoDB). Every mutation is appended to the WAL before the page is modified, so recovery can replay or roll back. PostgreSQL additionally writes full page images to the WAL after each checkpoint to defend against torn pages.

// B-Tree insert (simplified, ignoring concurrency)
write_ahead_log.append(redo_record(key, value))   // durable first
page = find_leaf(root, key)                        // ~log_b(N) page reads
if page.has_room():
    page.insert(key, value)                        // mutate in place
else:
    new_page = split(page)                          // structural change
    propagate_split_key_to_parent(new_page)         // may cascade up
mark_dirty(page)                                    // flushed later by checkpoint
Trade-off B-Trees give you excellent read latency and trivial range scans (leaf pages are linked in key order), at the cost of write amplification from page splits, random write placement, and the WAL doubling every write. Reads are O(log N) and crucially each key lives in exactly one place.

Fragmentation

Over time, splits and deletes leave pages partially full. A page that's 60% full wastes 40% of its I/O bandwidth on every read. This is why VACUUM (Postgres) and OPTIMIZE TABLE (MySQL) exist — they rebuild pages to reclaim space and restore density. B-Trees trade a small, bounded space overhead for in-place locality.

LSM-Trees: the write-optimized challenger

A Log-Structured Merge-Tree refuses to do random writes at all. Every write is an append. The structure has three moving parts:

  1. Memtable — an in-memory sorted structure (typically a skip list or balanced tree). Writes go here, kept sorted by key. Fast: it's RAM.
  2. WAL — before touching the memtable, the write is appended to an on-disk commit log so an unflushed memtable survives a crash.
  3. SSTables (Sorted String Tables) — when the memtable hits a size threshold (e.g. 64 MB), it is frozen and flushed to disk as an immutable, sorted file. New writes go to a fresh memtable.

Immutability changes everything

SSTables are never modified. An update to an existing key is just a newer entry written to the current memtable; it will land in a newer SSTable. A delete is a special marker called a tombstone — you can't erase a key that lives in an immutable file, so you write a record that says "this key is dead." The newest value always wins, determined by which SSTable (or memtable) is more recent.

// LSM write path -- always sequential, always fast
write_ahead_log.append(key, value)        // durable, sequential append
memtable.put(key, value)                  // in-memory sorted insert
if memtable.size() >= FLUSH_THRESHOLD:
    sstable = memtable.freeze().flush()   // sequential write of a sorted file
    memtable = new Memtable()
    wal.rotate()

// LSM read path -- newest-first, must check multiple layers
def get(key):
    if key in memtable: return memtable[key]
    for sstable in sstables_newest_to_oldest():
        if bloom_filter(sstable).might_contain(key):   // skip most files
            v = sstable.lookup(key)                     // binary search + index
            if v is not None: return v
    return NOT_FOUND

The read path is the price you pay. A key could live in the memtable or in any of dozens of SSTables. Without help, a read becomes "check everywhere," which is why LSM engines lean hard on Bloom filters, sparse per-file indexes, and block caches. This pattern — cheap writes, work shifted to read time and background merges — is the heart of designing a key-value store.

What strong candidates do They explain why LSM writes are fast in physical terms: a write is an in-memory insert plus a sequential log append, with no read-modify-write of a disk page. On an SSD, sequential append also produces friendlier wear patterns and fewer flash-translation-layer rewrites than scattered in-place updates.

The three amplifications

Every storage engine is a balancing act between three competing costs. Articulating this triangle is the single most important thing in this lesson.

AmplificationDefinitionHurts
WriteBytes physically written to disk ÷ bytes the application asked to writeSSD endurance, write throughput
ReadDisk reads (or pages touched) per logical readRead latency, IOPS budget
SpaceBytes on disk ÷ bytes of live logical dataStorage cost, cache efficiency

You cannot minimize all three at once — the RUM conjecture (Read, Update, Memory) formalizes that optimizing two corners worsens the third. The engines sit at different points:

  • B-Trees: low read amplification (one place to look), moderate space amplification (fragmentation), and meaningful write amplification — a single row update can rewrite a full 16 KB page plus its WAL record. Random-write heavy.
  • LSM-Trees: very low write amplification at ingest, but reads must consult multiple SSTables (read amplification) and a key may exist in several files before compaction (space amplification from stale versions and tombstones).
Pitfall "LSM has lower write amplification" is only true at ingest. Compaction rewrites data repeatedly as it merges levels — leveled compaction can produce write amplification of 10–30x over a record's lifetime. The amplification didn't disappear; it moved from the foreground write path to a background process. The win is that it's now sequential and off the critical path, not that it's free.

Compaction strategies

SSTables accumulate. Without merging them, reads degrade (more files to check) and space balloons (stale versions and tombstones never get reclaimed). Compaction is the background process that merges SSTables, discards overwritten values, and drops tombstones whose deletes are now globally settled. The strategy you pick is the central LSM tuning knob.

Size-tiered compaction (STCS)

Merge SSTables of similar size into a larger one. When you have N files of roughly the same size, merge them into one file N times bigger. This is the Cassandra default and is friendly to write-heavy workloads — fewer, larger merges.

  • Pro: low write amplification, great for ingest-heavy workloads.
  • Con: high space amplification (during a major compaction you may need ~2x the data size in free disk), and a key can live in many overlapping large files, hurting reads.

Leveled compaction (LCS)

Organize SSTables into levels (L0, L1, L2…), each ~10x larger than the previous. Within a level (except L0), SSTables have non-overlapping key ranges — so a given key exists in at most one file per level. RocksDB and LevelDB default to this.

  • Pro: bounded read amplification (~one file per level ≈ 7–10 files worst case) and low, predictable space amplification (~10%).
  • Con: high write amplification — merging into the next level rewrites overlapping data, often 10–30x.
Size-tieredLeveled
Write amplificationLowHigh
Read amplificationHigh (overlapping files)Low (one file/level)
Space amplificationHigh (~2x spikes)Low (~10%)

Hybrids exist: ScyllaDB's incremental and hybrid strategies, RocksDB's universal compaction, and Cassandra's time-window compaction (TWCS) for time-series data, where old time windows never need re-merging.

Pitfall Compaction competes with foreground traffic for disk IOPS and CPU. Mistuned compaction is one of the most common causes of LSM tail-latency blowups and "write stalls" — if flushes outrun compaction, L0 fills up and the engine throttles or blocks writes. Tombstones are a classic landmine: in time-series or queue-like access patterns you can scan thousands of tombstones to return zero rows, and a tombstone can't be dropped until you're sure every replica has seen the delete (the gc_grace_seconds window in Cassandra).

Bloom filters & avoiding disk reads

The LSM read amplification problem — "the key might be in any SSTable" — is mitigated by a Bloom filter per SSTable. A Bloom filter is a compact probabilistic set membership structure: ask "is key X in this file?" and it answers either "definitely not" or "probably yes." A negative is authoritative, so you skip that file entirely without touching disk. See Probabilistic Data Structures for the mechanics.

With a ~1% false-positive rate (about 10 bits/key), the vast majority of SSTables are eliminated for any given lookup, collapsing read amplification from "every file" to "the handful of files that might genuinely have it, plus an occasional false positive." This is what makes point lookups on a large LSM store practical at all.

Key insight Bloom filters only help point lookups. They do nothing for range scans, because a range query can't ask "is key X present" — it must inspect every SSTable whose key range overlaps the query. This is why range-scan-heavy workloads (analytics, time-series rollups) favor leveled compaction's non-overlapping ranges or sometimes a B-Tree outright. Engines also keep a per-file sparse index (one key per block) and a block cache so that even a "probably yes" only reads one data block.

WAL, page cache, and durability

Both engine families share two underpinnings worth making explicit.

Write-ahead logging

The WAL is the durability backbone for both. The contract is the same: append the intended change to a sequential log and fsync it before acknowledging the write, so a crash never loses an acknowledged operation. In a B-Tree the WAL protects against torn pages and lets dirty pages flush lazily at checkpoints. In an LSM tree the WAL protects the in-memory memtable, which would otherwise vanish on crash. Either way, group commit batches many transactions into one fsync to amortize the cost — the difference between thousands and tens of thousands of writes per second. The WAL also underpins replication: ship the log to followers and they reconstruct identical state.

Page cache

The OS page cache (and the engine's own buffer pool, like InnoDB's) keeps hot pages in RAM. For a B-Tree, the upper tree levels and hot leaves stay cached, so a "4 page read" lookup is usually zero disk reads. For an LSM tree, the block cache holds frequently read SSTable blocks. Caching is what hides most of the engine's worst-case I/O behavior from your latency graphs — until your working set exceeds RAM, at which point the engine's true read amplification surfaces brutally.

Pitfall Capacity planning that ignores working-set-vs-RAM is a trap. An LSM store benchmarks beautifully while everything fits in cache, then falls off a cliff when the dataset outgrows RAM and reads start hitting all those SSTables. Always benchmark with a dataset several times larger than memory.

Choosing an engine in practice

The decision flows almost entirely from your read/write ratio and access pattern.

  • Write-heavy ingest, point/recent reads (event logging, metrics, IoT telemetry, message queues, write-mostly KV): LSM. Sequential writes and Bloom-filtered point reads play to its strengths. This is why Cassandra, RocksDB, and time-series stores are LSM-based.
  • Read-heavy, transactional, lots of range scans and updates-in-place (OLTP, anything with rich secondary indexes and strong consistency): B-Tree. Predictable latency, one authoritative location per key, mature transactional support. This is the InnoDB/Postgres home turf.
  • Mixed / "it depends on the table": increasingly the line blurs. MyRocks puts an LSM (RocksDB) under MySQL for write-heavy fleets (Facebook's user-DB tier). MongoDB's WiredTiger offers both a B-Tree and an LSM mode. CockroachDB and TiKV build distributed SQL on top of RocksDB. The storage engine is becoming a per-workload choice, not a per-product one.

This decision interacts with everything above it: it shapes how you do sharding (LSM's sequential writes tolerate hot shards differently), it factors into the SQL vs NoSQL conversation (most NoSQL write-scalable stores are LSM under the hood), and it constrains the consistency and replication options discussed in the CAP theorem and replication lessons.

Trade-off A clean mental model: B-Trees optimize for reads and pay at write time; LSM-Trees optimize for writes and pay at read time and in background compaction. Neither is "faster" — they relocate cost to wherever you've decided you can afford it. The principal-level skill is naming which cost your workload can absorb.

Principal-level interview lens

What strong candidates do They never say "use NoSQL because it's faster." They connect the storage engine to the workload's physics: read/write ratio, working-set size vs RAM, access pattern (point vs range), and durability needs. They can sketch the LSM write path and read path from memory, explain the three amplifications and the RUM conjecture, and articulate why compaction strategy is the dominant LSM tuning knob. Crucially, they recognize that the engine choice constrains every layer above it — sharding, replication, consistency, and tail latency all inherit from it — and they name concrete systems (InnoDB, RocksDB, Cassandra, WiredTiger, MyRocks) to ground the discussion.

  • Your service does 50k writes/sec of immutable event records and reads only the most recent value per key. Walk me through which storage engine you'd pick and trace the exact write and read paths. Where does the latency hide, and what breaks first under load?
  • We chose an LSM store and our p99 read latency periodically spikes to 10x baseline. Give me three distinct root causes and how you'd distinguish them with metrics.
  • Explain write, read, and space amplification. For leveled vs size-tiered compaction, which amplification does each minimize and at whose expense? When would you deliberately choose the higher-write-amplification strategy?
  • A Bloom filter says a key is "probably present" but it isn't actually in that SSTable. What happens, what did it cost, and why are Bloom filters useless for range scans?
  • We're deleting millions of rows from a Cassandra table and reads are getting slower instead of faster. What's happening, and why can't the database just remove the data immediately?
  • Both B-Trees and LSM-Trees use a WAL. If they both log every write before acknowledging, where does the actual difference in write performance come from? Be specific about disk access patterns.
