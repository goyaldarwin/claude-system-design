Event-Driven Architecture

Instead of services calling each other directly ("do this now, and wait"), they publish events ("this happened") that other services react to, whenever they're ready. This decouples producers from consumers in time and identity — you can add a new reactor without touching the producer. The cost is that the system becomes asynchronous and eventually consistent, and a whole class of new failures (lost events, poison messages, ordering) becomes yours to handle.

The trade-off Loose coupling, independent scaling, and easy extensibility — in exchange for eventual consistency, harder debugging, and new failure modes (duplicates, poison messages, ordering). Producers don't know or wait for consumers, so you add reactors freely and absorb load spikes in the broker. You pay by giving up synchronous "it's done now" certainty, by losing the easy stack trace across a call, and by owning delivery semantics, retries, and dead-letter handling.

What event-driven means — "this happened," not "do this"

In a request-driven system, service A commands service B ("create this order") and waits for the result — A is coupled to B's availability and interface. In an event-driven system, A publishes a fact ("OrderPlaced") to a broker, and any number of services consume it on their own schedule. A doesn't know who's listening, doesn't wait, and isn't affected if a consumer is down or a new one is added. This is decoupling in time (async), identity (A doesn't know B), and load (the broker buffers spikes).

What it buys: add a new reactor (analytics, notifications, fraud check) to "OrderPlaced" without touching the order service; absorb traffic bursts in the broker; scale producers and consumers independently. What it costs: the system is now eventually consistent and asynchronous — there's no synchronous "it's all done" moment, debugging spans many services with no single stack trace, and you own delivery guarantees.

The one-line frameEvent-driven architecture trades synchronous certainty and easy debugging for loose coupling, extensibility, and load absorption. You stop asking "is it done?" and start designing for "it will be done, eventually, and possibly more than once" — which means idempotency, reliable publishing, dead-letter handling, and schema discipline are not optional extras but the core of the design.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the event that was published but the data wasn't saved Your order service saves the order to its database, then publishes "OrderPlaced" to Kafka. The process crashes between the two. Sometimes the reverse: the publish succeeds but the database transaction rolls back. Over weeks, the event stream and the database disagree. What's this problem called, and what mechanism makes the save and the publish effectively atomic?
Scenario 2 — the payment queue jammed for 6 hours (real-pattern incident) A payment consumer received one malformed message it could never process. With no dead-letter handling, it retried that message over 10,000 times across 6 hours — blocking every subsequent payment behind it in the partition. Recovery needed a manual reset. What is this single bad message called, and what two mechanisms keep one of them from halting the whole stream?
Scenario 3 — who runs the workflow? "Place order" must reserve inventory, charge payment, and schedule shipping across three services. Option A: each service listens for the previous one's event and emits the next (no central brain). Option B: one coordinator service drives all three and handles failures. What are these two styles called, and when does the brainless option become impossible to reason about?

Scenario 1 is reliable publishing (the outbox). Scenario 2 is poison messages & DLQs. Scenario 3 is choreography vs orchestration. Each maps to a decision below.

The five decisions that define an EDA design

Designing event-driven systems is these five choices. Each has a named failure nested under it.

Decision 1 — Event vs command, and how much state to carry

First, distinguish: a command is a directed instruction ("ChargeCard" — one recipient, expected to act); an event is a broadcast fact ("OrderPlaced" — whoever cares reacts). Then decide an event's payload along a spectrum: event notification (just "OrderPlaced, id=123" — thin; consumers call back for details), event-carried state transfer (the event includes the order's data — fat; consumers need no callback), or event sourcing (the event log is the source of truth; state is derived by replaying events).

The failure that happens when you get this wrong — the wrong event payload Too thin (notification only) and every consumer must call back to the producer for details — recreating tight coupling and load on the producer, and risking the producer's data having changed since the event. Too fat (carrying everything) and events bloat, leak the producer's internal model, and go stale. The guidance: carry the state consumers actually need to act independently (event-carried state transfer) to maximize decoupling, but keep it to the relevant facts, versioned. Choose event sourcing only when you genuinely want an auditable, replayable log as the system of record — it's powerful but a big commitment.

Decision 2 — Choreography or orchestration?

For a multi-step workflow across services: choreography — each service reacts to events and emits the next, no central coordinator (decentralized, loosely coupled). Orchestration — a central orchestrator (a saga coordinator, a workflow engine) explicitly drives each step and handles failures/compensation (centralized, visible).

The failure that happens when you get this wrong — choreography sprawl This is Scenario 3. Choreography is elegant for 2–3 steps but becomes impossible to reason about as it grows: the end-to-end flow exists in no single place — it's an emergent property of who-listens-to-what, scattered across services. Nobody can answer "what happens when an order is placed?" without tracing events through the whole system, and failure handling (what compensates what) is diffuse. For complex or failure-sensitive workflows, prefer orchestration (a saga with a coordinator) so the flow and its compensations live in one readable, debuggable place. Use choreography for simple fan-outs (one event, independent reactors), orchestration for real multi-step business processes.

Decision 3 — How do you publish events reliably?

The producer must update its database and publish an event — two systems, no shared transaction. The transactional outbox solves it: write the business data and an outbox event row in one local transaction (atomic for free), then a separate relay or CDC process publishes the outbox rows to the broker — at-least-once, so consumers must be idempotent.

The failure that happens when you get this wrong — the dual-write problem This is Scenario 1. Writing to the DB then publishing (or vice-versa) without a shared transaction means a crash between the two, or a rolled-back transaction whose event already published, silently desynchronizes your data and your stream. You cannot fix it by reordering (whichever is second can fail) or by a distributed transaction across DB+broker (slow, brittle — see 2PC). The outbox makes the data-write and the event-record one local transaction, then moves publishing to a reliable, retrying, idempotent step. This is the single most important reliability pattern in EDA — covered in depth in Distributed Transactions and CDC.

Decision 4 — What happens to messages that can't be processed?

Some messages will fail processing — transiently (a dependency is down: retry) or permanently (malformed, references deleted data: will never succeed). You need a retry strategy (exponential backoff + jitter for transient failures, with a cap) and a dead-letter queue (DLQ): after N failed attempts, move the message to a separate queue for inspection/replay rather than blocking the stream.

The failure that happens when you get this wrong — the poison message This is Scenario 2. A poison message is one the consumer can never process. Without a DLQ, the consumer retries it forever — in an ordered partition it blocks every message behind it (the payment processor: 10,000+ retries, 6 hours, all payments stuck). The fix: (1) retry with exponential backoff + jitter and a max-attempts cap (so transient failures recover without a synchronized retry storm); (2) after the cap, route the message to a DLQ — preserving it for analysis/replay while the stream keeps flowing. Also classify errors: retry timeouts/5xx, but send validation/schema errors straight to the DLQ (retrying them is pointless). Never let one bad message halt the pipeline.

Decision 5 — How do events evolve over time? (schema)

Events are a contract between producer and many consumers, often deployed at different times — so event schemas must evolve backward- and forward-compatibly. Use a schema registry (e.g. for Avro/Protobuf) to enforce that new versions only add optional fields and never break old consumers; producers and consumers validate against it.

The failure that happens when you get this wrong — the breaking event schema Because producers and consumers deploy independently and old events linger in the log (replayable!), a breaking schema change (removing/renaming a field, changing a type) silently breaks consumers that haven't upgraded — or breaks replay of historical events. The same backward-compatibility discipline as APIs: only additive changes in place, version when you must break, and enforce it with a schema registry that rejects incompatible changes at publish time. Forward compatibility matters too: a consumer must tolerate fields it doesn't yet understand (ignore unknowns), since a newer producer may emit them.

And make every consumer idempotent — at-least-once delivery is the norm.

  • Scenario 3 (who runs it): Choreography vs orchestration — when does choreography become unreasonable, and what do you switch to?

The shape to remember: EDA is 5 decisions — event vs command + payload, choreography vs orchestration, reliable publishing (outbox), poison messages + DLQ, schema evolution — each with a nested failure.

Famous implementations

The patterns made concrete. Same facets — what it is, the mechanism, and the failure to watch.

Event sourcing + CQRS

  • What it is. Store every state change as an immutable event (the log is the source of truth); derive current state by replaying. CQRS separates the write model (the event log) from read models (projections optimized per query).
  • Mechanism. Append events; build/refresh read models by consuming them; "time-travel" by replaying to any point.
  • The failure to watch. Big complexity commitment — schema evolution of old events, rebuilding projections, and eventual consistency between write and read models. Adopt only when audit/replay genuinely pays for it.

Transactional outbox + CDC

  • What it is. The reliable-publishing pattern (Decision 3) — the answer to the dual-write problem.
  • Mechanism. Business write + outbox row in one local transaction; CDC/relay publishes the outbox at-least-once.
  • The failure to watch. Consumers must dedupe (at-least-once); the relay/CDC connector must be monitored for lag.

Retry + dead-letter queue

  • What it is. The poison-message protection (Decision 4).
  • Mechanism. Bounded retries with exponential backoff + jitter; after the cap, move to a DLQ; alert and replay after a fix; classify transient vs permanent errors.
  • The failure to watch. A DLQ nobody monitors silently swallows failures — alert on DLQ depth and have a replay path; without jitter, retries synchronize into storms.

Build it yourself

These click when you design the flow. Work each before opening the approach.

Publish "OrderPlaced" without the dual-write

Your order service must persist an order and emit an "OrderPlaced" event with no lost or phantom events. Design tables and the publisher, and state the consumer guarantee.

Reveal the approach

In one local DB transaction, insert the order row and an outbox row (event type + payload + unique event id). A relay process — or CDC tailing the WAL — reads unsent outbox rows, publishes them to the broker, and marks them sent, retrying on failure. Because the relay can publish then crash before marking sent, delivery is at-least-once, so every consumer dedupes on the event id (idempotent). No event is lost (committed with the data) and none is phantom (only committed rows publish). To rebuild state, consumers replay from the log. This is the dual-write fix; no distributed transaction needed.

Design retry + DLQ for a flaky consumer

Your consumer calls a sometimes-slow external API and occasionally gets a malformed message. Design retry, backoff, and dead-lettering so transient failures recover but a poison message never blocks the stream.

Reveal the approach

Classify the error first. Transient (timeout, 5xx, rate-limit): retry with exponential backoff + jitter (e.g. 1s, 2s, 4s ± random), up to a max attempts cap — jitter prevents all consumers retrying in lockstep (a storm). Permanent (validation/schema/malformed, 4xx): don't retry — it'll never succeed. After the retry cap (or immediately for permanent errors), move the message to a dead-letter queue with the failure reason, and acknowledge the original so the stream advances (the poison message no longer blocks anything). Alert on DLQ depth, inspect/fix, and replay from the DLQ once resolved. This turns Scenario 2's 6-hour jam into "one message in the DLQ, payments still flowing."

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Choreography vs orchestration for checkout. Design a 4-step order workflow both ways and argue which is maintainable, including failure compensation. → Distributed Transactions (saga).
  • Design a notification fan-out. One "UserSignedUp" event must trigger email, analytics, and a welcome-flow — independently and resiliently. Choose payload style and delivery guarantees. → Notification System.
  • Make a system eventually consistent on purpose. Pick a feature where you deliberately accept eventual consistency via events, and define what the user sees during the lag. → CAP, Consistency Models.
