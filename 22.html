Designing Typeahead Suggestion

Autocomplete: as a user types each character, return the top few completions in under ~100 ms. The trick is that you cannot rank suggestions at query time at this latency — you pre-compute the best completions per prefix offline, store them in a structure built for prefix lookup, and serve from memory.

The crux Sub-100ms per keystroke means the ranking must be pre-computed offline; the query is a memory lookup, never a search. A trie (or prefix-keyed cache) holds the top-k completions per prefix; an offline pipeline rebuilds it from query logs. The query path does no ranking.

1. Requirements

Functional: given a prefix, return the top-k most relevant completions; relevance ~ popularity (historical query frequency); update suggestions as trends shift. Non-functional: very low latency (~100 ms or less — it fires on every keystroke), highly available, scale to billions of queries/day. Extended: personalization, typo tolerance, multi-language.

2. Capacity estimate

  • Query volume is huge and amplified: autocomplete fires per keystroke, so typing a 10-char query is ~10 requests. Billions of searches/day × several keystrokes each = tens of billions of suggestion requests/day.
  • Latency budget is tiny: the suggestion must beat the user's next keystroke — well under 100 ms — which rules out any per-query ranking or DB scan.
  • The corpus is large but the hot set is small: a relatively small number of prefixes/queries dominate; the structure fits in memory and replicates.

3. System API

suggest(prefix, k=10, lang?, user?) → [top-k completions]   // served from an in-memory trie/cache

Debounced on the client (don't fire on literally every keystroke), and the response is small — just k strings.

4. High-level architecture — two decoupled paths

  • Query path (online, read): request → in-memory trie (or prefix→top-k cache) on replicated suggestion servers → return top-k. No ranking, no DB. Behind a cache/LB.
  • Build path (offline, write): a batch pipeline reads aggregated query logs, computes popularity, builds the trie with pre-computed top-k at each node, and atomically swaps the new trie in for the old.
The shape in one sentenceAn offline pipeline turns query logs into a prefix structure with the answers pre-baked; the online path just walks to the prefix node and returns its stored top-k — the read does zero ranking.

5. Data model — the trie

A trie: each node is a character, each path is a query prefix, and — crucially — each node caches the pre-computed top-k completions for its prefix. Walking to the node for "be" gives you "best, better, beach…" instantly, no subtree scan. (At scale the trie is sharded by prefix and replicated.)

Three scenarios — the hard parts, as questions

Scenario 1 — finding completions for "be" A user types "be". You need all queries starting with "be", ranked. If completions live in a normal database, what query do you run — and why is it far too slow at 100 ms per keystroke?
Scenario 2 — ranking by popularity "be" has a million matching queries. You can only show 10. Sorting a million matches by popularity on every keystroke is impossible in the budget — so when does the ranking actually happen?
Scenario 3 — trends change A breaking-news term suddenly spikes in popularity. The suggestions should reflect it — but you can't update a million prefix nodes on every single query. How does the trie stay fresh without the write path crushing the read path?

Scenario 1 is the prefix structure. Scenario 2 is pre-computed ranking. Scenario 3 is offline rebuild.

6. The hard sub-problems

Sub-problem 1 — The prefix-lookup structure

Finding "all strings starting with P" is exactly what a trie is for: walk P character by character to a node in O(|P|), and every descendant shares that prefix. No scanning, no LIKE 'be%' table scan.

The failure that happens when you get this wrong — the scan-everything query This is Scenario 1. Storing completions in a row store and querying WHERE q LIKE 'be%' ORDER BY popularity LIMIT 10 means a scan (or at best an index range) over potentially millions of rows, then a sort — nowhere near 100 ms under load. A trie makes prefix navigation O(prefix length), independent of corpus size; held in memory, the walk is microseconds. The structure, not the database, is the answer.

Sub-problem 2 — Ranking the top-k

The trie finds the matching subtree, but you must return the best k by popularity. Ranking a million matches per keystroke is impossible — so you don't: each trie node stores its pre-computed top-k, baked in when the trie is built.

The failure that happens when you get this wrong — rank-at-query latency This is Scenario 2. Gathering all completions under a node and sorting them by popularity at query time blows the latency budget on popular prefixes (the ones with the most matches — exactly the hot path). The fix is to pre-compute top-k at every node offline, so the query returns a stored list with zero work. This is the same "move cost to the rare write" pattern as feed fan-out: do the ranking once when building the trie, not on every read.

Sub-problem 3 — Keeping it fresh at scale

Popularity shifts (trends, breaking news), so the trie must update — but updating top-k on millions of nodes per query is impossible. The answer is an offline batch rebuild: a pipeline aggregates query logs periodically, recomputes popularity and per-node top-k, builds a fresh trie, and atomically swaps it in place of the old one. Reads always hit a complete, immutable trie.

The failure that happens when you get this wrong — the write-hot trie This is Scenario 3. Trying to keep the live trie perfectly current — incrementing counts and re-sorting top-k on every query — turns the read structure into a write-contended mess and destroys read latency. Decouple them: the read trie is immutable; a separate offline job rebuilds it from logs and swaps it in (seconds-to-minutes staleness, which is fine for suggestions). For genuinely fast-moving trends, layer a small real-time boost on top, but the base ranking stays a periodic batch build. Replicate the (read-only) trie across servers for load and fault tolerance.

Famous implementations

Trie / prefix-hash serving

  • What it is. The classic in-memory trie with cached top-k; some systems flatten it to a prefix → top-k hash for even simpler O(1) lookup.
  • Key idea. Pre-baked answers + immutable read structure = trivial, fast reads.
  • The failure to watch. Memory size for huge corpora — shard the trie by prefix across servers.

Elasticsearch completion suggester

  • What it is. ES ships a completion suggester backed by an in-memory FST (a compressed trie variant).
  • Key idea. Off-the-shelf prefix suggestions with weights; good default before building your own.
  • The failure to watch. Rebuild/refresh semantics — updates aren't instant; same offline-rebuild reality. (Elasticsearch.)

Build it yourself

Implement a top-k trie

Build a trie that returns the top-10 completions for any prefix in O(prefix length). Where do you store the top-k, and how do you build them?

Reveal the approach

Each node represents one character; a path from root spells a prefix. At build time, walk the query log: for each (query, frequency), insert the query and, at every node along its path, maintain that node's top-k list (a small heap/sorted array of the most popular completions passing through it). At query time, walk the prefix to its node in O(|prefix|) and return the stored top-k — no subtree traversal, no sorting. The build is the expensive part (done offline, in batch); the query is a pointer walk. To keep memory bounded, cap k small and prune rare branches.

Design-it-yourself

  • Personalized suggestions. Blend global top-k with a user's own history at query time without blowing the budget.
  • Typo tolerance. Suggest for "recieve" → "receive". What structure handles fuzzy prefixes? → edit-distance / FST.

Let's design a real-time suggestion service, which will recommend terms to users as they enter text for searching.

Similar Services: Auto-suggestions, Typeahead search
Difficulty: Medium

What is Typeahead Suggestion?

Typeahead suggestions enable users to search for known and frequently searched terms. As the user types into the search box, it tries to predict the query based on the characters the user has entered and gives a list of suggestion to complete the query. Typeahead suggestions help the user to articulate their search queries better. It’s not about speeding up the search process but rather about guiding the users and lending them a helping hand in constructing their search query.

Requirements and Goals of the System

Functional Requirements: As the user types in their query, our service should suggest top 10 terms starting with whatever user has typed.

Non-functional Requirements: The suggestions should appear in real-time — to feel instant, the user should see them within ~100ms.

Basic System Design and Algorithm

The problem we are solving is that we have a lot of ‘strings’ that we need to store in such a way that users can search on any prefix. Our service will suggest next terms that will match the given prefix. For example, if our database contains following terms: cap, cat, captain, capital; and the user has typed in ‘cap’, our system should suggest ‘cap’, ‘captain’ and ‘capital’.

Since we’ve to serve a lot of queries with minimum latency, we need to come up with a scheme that can efficiently store our data such that it can be queried quickly. We can’t depend upon some database for this; we need to store our index in memory in a highly efficient data structure.

One of the most appropriate data structure that can serve our purpose would be the Trie (pronounced “try”). A trie is a tree-like data structure used to store phrases where each node stores a character of the phrase in a sequential manner. Should we’ve case insensitive trie? For simplicity and search use case let’s assume our data is case insensitive.

How to find top suggestion? Now that we can find all the terms given a prefix, how can we know what’re the top 10 terms that we should suggest? One simple solution could be to store the count of searches that terminated at each node, e.g., if users have searched about ‘CAPTAIN’ 100 times and ‘CAPTION’ 500 times, we can store this number with the last character of the phrase. So now if the user has typed ‘CAP’ we know the top most searched word under the prefix ‘CAP’ is ‘CAPTION’. So given a prefix, we can traverse the sub-tree under it, to find the top suggestions.

Given a prefix, how much time it can take to traverse its sub-tree? Given the amount of data we need to index, we should expect a huge tree. Even, traversing a sub-tree would take really long, e.g., the phrase ‘system design questions’ is 23 levels deep. Since we’ve very tight latency requirements, we do need to improve the efficiency of our solution.

Can we store top suggestions with each node? This can surely speed up our searches but will require a lot of extra storage. We can store top 10 suggestions at each node that we can return to the user. We’ve to bear the big increase in our storage capacity to achieve the required efficiency.

We can optimize our storage by storing only references of the terminal nodes rather than storing the entire phrase. To find the suggested term we’ve to traverse back using the parent reference from the terminal node. We will also need to store the frequency with each reference to keep track of top suggestions.

How would we build this trie? We can efficiently build our trie bottom up. Each parent node will recursively call all the child nodes to calculate their top suggestions and their counts. Parent nodes will combine top suggestions from all of their children to determine their top suggestions.

How to update the trie? Assuming five billion searches every day, which would give us approximately 60K queries per second. If we try to update our trie for every query it’ll be extremely resource intensive and this can hamper our read requests too. One solution to handle this could be to update our trie offline after a certain interval.

As the new queries come in, we can log them and also track their frequencies. Either we can log every query or do sampling and log every 1000th query. For example, if we don’t want to show a term which is searched for less than 1000 times, it’s safe to log every 1000th searched term.

We can have a Map-Reduce (MR) setup to process all the logging data periodically, say every hour. These MR jobs will calculate frequencies of all searched terms in the past hour. We can then update our trie with this new data. We can take the current snapshot of the trie and update it with all the new terms and their frequencies. We should do this offline, as we don’t want our read queries to be blocked by update trie requests. We can have two options:

  1. We can make a copy of the trie on each server to update it offline. Once done we can switch to start using it and discard the old one.
  2. Another option is we can have a master-slave configuration for each trie server. We can update slave while the master is serving traffic. Once the update is complete, we can make the slave our new master. We can later update our old master, which can then start serving traffic too.

How can we update the frequencies of typeahead suggestions? Since we are storing frequencies of our typeahead suggestions with each node, we need to update them too. We can update only difference in frequencies rather than recounting all search terms from scratch. If we’re keeping count of all the terms searched in last 10 days, we’ll need to subtract the counts from the time period no longer included and add the counts for the new time period being included. We can add and subtract frequencies based on Exponential Moving Average (EMA) of each term. In EMA, we give more weight to the latest data. It’s also known as the exponentially weighted moving average.

After inserting a new term in the trie, we’ll go to the terminal node of the phrase and increase its frequency. Since we’re storing the top 10 queries in each node, it is possible that this particular search term jumped into the top 10 queries of a few other nodes. So, we need to update the top 10 queries of those nodes then. We’ve to traverse back from the node to all the way up to the root. For every parent, we check if the current query is part of the top 10. If so, we update the corresponding frequency. If not, we check if the current query’s frequency is high enough to be a part of the top 10. If so, we insert this new term and remove the term with the lowest frequency.

How can we remove a term from the trie? Let’s say we’ve to remove a term from the trie, because of some legal issue or hate or piracy etc. We can completely remove such terms from the trie when the regular update happens, meanwhile, we can add a filtering layer on each server, which will remove any such term before sending them to users.

What could be different ranking criteria for suggestions? In addition to a simple count, for terms ranking, we have to consider other factors too, e.g., freshness, user location, language, demographics, personal history etc.

Permanent Storage of the Trie

How to store trie in a file so that we can rebuild our trie easily - this will be needed when a machine restarts? We can take snapshot of our trie periodically and store it in a file. This will enable us to rebuild a trie if the server goes down. To store, we can start with the root node and save the trie level-by-level. With each node we can store what character it contains and how many children it has. Right after each node we should put all of its children. Let’s assume we have following trie:

If we store this trie in a file with the above-mentioned scheme, we will have: “C2,A2,R1,T,P,O1,D”. From this, we can easily rebuild our trie.

If you’ve noticed we are not storing top suggestions and their counts with each node, it is hard to store this information, as our trie is being stored top down, we don’t have child nodes created before the parent, so there is no easy way to store their references. For this, we have to recalculate all the top terms with counts. This can be done while we are building the trie. Each node will calculate its top suggestions and pass it to its parent. Each parent node will merge results from all of its children to figure out its top suggestions.

Scale Estimation

If we are building a service, which has the same scale as that of Google, we can expect 5 billion searches every day, which would give us approximately 60K queries per second.

Since there will be a lot of duplicates in 5 billion queries, we can assume that only 20% of these will be unique. If we only want to index top 50% of the search terms, we can get rid of a lot of less frequently searched queries. Let’s assume we will have 100 million unique terms for which we want to build an index.

Storage Estimation: If on the average each query consists of 3 words, and if the average length of a word is 5 characters, this will give us 15 characters of average query size. Assuming we need 2 bytes to store a character, we will need 30 bytes to store an average query. So total storage we will need:

100 million * 30 bytes => 3 GB

We can expect some growth in this data every day, but we should also be removing some terms that are not searched anymore. If we assume we have 2% new queries every day and if we are maintaining our index for last one year, total storage we should expect:

3GB + (0.02 * 3 GB * 365 days) => 25 GB

Data Partition

Although our index can easily fit on one server, we can still partition it in order to meet our requirements of higher efficiency and lower latencies. How can we efficiently partition our data to distribute it onto multiple servers?

a. Range Based Partitioning: What if we store our phrases in separate partitions based on their first letter. So we save all the terms starting with letter ‘A’ in one partition and those that start with letter ‘B’ into another partition and so on. We can even combine certain less frequently occurring letters into one database partition. We should come up with this partitioning scheme statically so that we can always store and search terms in a predictable manner.

The main problem with this approach is that it can lead to unbalanced servers, for instance; if we decide to put all terms starting with letter ‘E’ into a DB partition, but later we realize that we have too many terms that start with letter ‘E’, which we can’t fit into one DB partition.

We can see that the above problem will happen with every statically defined scheme. It is not possible to calculate if each of our partitions will fit on one server statically.

b. Partition based on the maximum capacity of the server: Let’s say we partition our trie based on the maximum memory capacity of the servers. We can keep storing data on a server as long as it has memory available. Whenever a sub-tree cannot fit into a server, we break our partition there to assign that range to this server and move on the next server to repeat this process. Let’s say, if our first trie server can store all terms from ‘A’ to ‘AABC’, which mean our next server will store from ‘AABD’ onwards. If our second server could store up to ‘BXA’, next serve will start from ‘BXB’ and so on. We can keep a hash table to quickly access this partitioning scheme:

Server 1, A-AABC
Server 2, AABD-BXA
Server 3, BXB-CDA

For querying, if the user has typed ‘A’ we have to query both server 1 and 2 to find the top suggestions. When the user has typed ‘AA’, still we have to query server 1 and 2, but when the user has typed ‘AAA’ we only need to query server 1.

We can have a load balancer in front of our trie servers, which can store this mapping and redirect traffic. Also if we are querying from multiple servers, either we need to merge the results at the server side to calculate overall top results, or make our clients do that. If we prefer to do this on the server side, we need to introduce another layer of servers between load balancers and trie servers, let’s call them aggregator. These servers will aggregate results from multiple trie servers and return the top results to the client.

Partitioning based on the maximum capacity can still lead us to hotspots e.g., if there are a lot of queries for terms starting with ‘cap’, the server holding it will have a high load compared to others.

c. Partition based on the hash of the term: Each term will be passed to a hash function, which will generate a server number and we will store the term on that server. This will make our term distribution random and hence minimizing hotspots. To find typeahead suggestions for a term, we have to ask all servers and then aggregate the results. We have to use consistent hashing for fault tolerance and load distribution.

Cache

We should realize that caching the top searched terms will be extremely helpful in our service. There will be a small percentage of queries that will be responsible for most of the traffic. We can have separate cache servers in front of the trie servers, holding most frequently searched terms and their typeahead suggestions. Application servers should check these cache servers before hitting the trie servers to see if they have the desired searched terms.

We can also build a simple Machine Learning (ML) model that can try to predict the engagement on each suggestion based on simple counting, personalization, or trending data etc., and cache these terms.

Replication and Load Balancer

We should have replicas for our trie servers both for load balancing and also for fault tolerance. We also need a load balancer that keeps track of our data partitioning scheme and redirects traffic based on the prefixes.

Fault Tolerance

What will happen when a trie server goes down? As discussed above we can have a master-slave configuration, if the master dies slave can take over after failover. Any server that comes back up, can rebuild the trie based on the last snapshot.

Typeahead Client

We can perform the following optimizations on the client to improve user’s experience:

  1. The client should only try hitting the server if the user has not pressed any key for 50ms.
  2. If the user is constantly typing, the client can cancel the in-progress requests.
  3. Initially, the client can wait until the user enters a couple of characters.
  4. Clients can pre-fetch some data from the server to save future requests.
  5. Clients can store the recent history of suggestions locally. Recent history has a very high rate of being reused.
  6. Establishing an early connection with server turns out to be one of the most important factors. As soon as the user opens the search engine website, the client can open a connection with the server. So when user types in the first character, client doesn’t waste time in establishing the connection.
  7. The server can push some part of their cache to CDNs and Internet Service Providers (ISPs) for efficiency.

Personalization

Users will receive some typeahead suggestions based on their historical searches, location, language, etc. We can store the personal history of each user separately on the server and cache them on the client too. The server can add these personalized terms in the final set, before sending it to the user. Personalized searches should always come before others.

Principal-level deep dive

Deeper analysis: trade-offs & failure modes

The core lesson treats the trie as a black box: "store strings in a trie, precompute top-k, update offline." That recipe is correct, but the engineering substance lives in the second-order decisions — what shape the in-memory structure actually takes, how the read path stays under 100ms while the write path runs an hour behind, and what breaks when a single prefix becomes a hotspot. This is roughly the architecture behind Google's autocomplete, Elasticsearch's completion suggester, and the LinkedIn/Twitter typeahead services.

Why a trie at all — and why not a B-tree or a SQL LIKE 'cap%'

A prefix query is fundamentally a tree traversal problem. A trie reaches the node for prefix cap in O(len(prefix)) = O(3) pointer hops, completely independent of how many terms are indexed. A SQL WHERE term LIKE 'cap%' against a B-tree index is also fast for the prefix scan itself (it's a range seek), but it then has to scan and sort all matching rows by frequency at query time — which for a hot prefix like "a" could be millions of rows. The trie's win is that you precompute the answer and attach it to the prefix node, turning a sort-millions problem into a pointer dereference.

The real reason it's in memoryThe structure is small — the lesson estimates ~25 GB for a year of Google-scale terms, and that's before radix compression. 25 GB fits in RAM on a single modern box. There is no reason to pay a disk seek (even an SSD's ~100µs) on the read path when the whole index is RAM-resident. The trie isn't chosen for asymptotic cleverness; it's chosen because it lets you serve a prefix query without touching disk and without sorting at request time.

Radix tree, not a naive character trie

A node-per-character trie wastes enormous space: every node carries a child-pointer map (a 26-entry array or a hash map), and chains of single-child nodes (c→a→p→t→a→i→n) are pure overhead. Production systems use a radix tree (compressed trie / Patricia trie) that merges single-child chains into one edge labelled with a substring. This is exactly the merge the lesson shows in its second diagram — it's not an optional optimization, it's the default. It cuts node count by 5–10x and improves cache locality (fewer pointer chases per traversal).

Top-k: precompute vs. query-time — the central trade-off

This is the decision the whole design pivots on.

ApproachRead latencyMemory costWhen it wins
Query-time traversal (walk the whole subtree, collect terminals, sort by freq, take 10)O(subtree size) — unbounded; a short prefix like "s" means traversing millions of nodesMinimal — only frequency per terminalSmall / cold datasets, or when memory is the binding constraint
Precompute top-k per node (store 10 winning suggestions at every internal node)O(len(prefix)) — bounded, sub-millisecond~10 entries × every node; large multiplierGoogle-scale read traffic; this is what you ship

The storage trick the lesson mentions matters: don't store the full phrase at each node's top-10 list — store a reference to the terminal node plus a frequency. The phrase is reconstructed by walking parent pointers back up from the terminal. This turns "10 copies of a 30-char string per node" into "10 pointers + 10 ints per node," which is what makes the precompute approach affordable.

Failure mode: the unbounded short prefixIf you ship query-time traversal and forget to cap it, the prefix "" or "a" becomes an availability incident — a single keystroke triggers a multi-million-node traversal on the read path, blowing your latency budget and pinning a CPU. Either precompute top-k, or hard-cap traversal depth/breadth and accept approximate results for very short prefixes.

Updating the trie from query logs — strictly offline

At 60K QPS, mutating the live trie per search is a non-starter: writes would contend with reads on the same in-memory structure, and you'd be re-bubbling top-k lists up to the root on every keystroke. The discipline is read path and write path never share a mutable structure:

StageWhat happens
1. Log + sampleAppend queries to a log (this is exactly a Kafka topic in practice). Sample if you only care about terms above a frequency floor — logging every 1000th query means anything searched <1000× never qualifies, which is the cut you wanted anyway.
2. Aggregate (MapReduce / Spark / stream job)Compute per-term frequency deltas over the window (hourly). Use an Exponential Moving Average so fresh queries outweigh stale ones and old terms decay out naturally — better than a hard sliding window that needs you to subtract the expired tail.
3. Rebuild / patchBuild a new trie snapshot (or patch a copy) with updated frequencies and recomputed top-k lists, bottom-up so each parent merges its children's winners.
4. Atomic swapHot-swap the new trie in. Two patterns below.
The swap: copy-on-server vs. master/slavePattern A: each server builds an updated copy locally, then flips a pointer to it and GCs the old one — simple, but you double the memory of that server during the build (50 GB box to hold two 25 GB tries). Pattern B: a master/replica pair where you update the idle replica, promote it, then catch up the old master — no memory doubling per box, but more orchestration. The atomic pointer swap is what gives readers a consistent view with zero locking; a reader either sees the entire old trie or the entire new one, never a half-built one.

Sharding the trie by prefix

The index fits on one box today, but you replicate and shard for throughput, latency, and blast-radius reasons. Three strategies, in increasing sophistication:

  1. Static range by first letter (A→server1, B→server2). Dead simple, but guarantees skew — "s" and "c" prefixes dwarf "x" and "z", so servers are wildly unbalanced.
  2. Range by server capacity (server fills until full, then the next range starts on server 2: A–AABC, AABD–BXA, …). Balances storage but not load — a viral prefix still lands entirely on one shard and hotspots it.
  3. Hash of the term (consistent hashing). Scatters terms randomly, killing hotspots — but now every prefix query must fan out to every shard and an aggregator merges the partial top-10s. You've traded a hotspot for fan-out cost on every read.
The aggregator and the mergeWith range partitioning, a short prefix like "a" may span multiple shards (server 1 holds A–AABC, server 2 holds AABD onward), so even range sharding needs a fan-out + merge for short prefixes. An aggregator tier between the load balancer and trie servers collects partial top-10 lists and merges them into a global top-10. The merge is correct only because each shard returns its top-10 with frequencies — you can't merge ranked lists without the underlying scores.

Client-side: debouncing, cancellation, caching, and the latency budget

A surprising amount of perceived speed is won on the client before a request ever leaves the browser:

TechniqueWhat it buys
Debounce (wait ~50ms of keystroke silence before firing)Collapses a burst of fast typing into one request; a fast typist hitting 5 keys in 200ms generates 1 query, not 5.
Cancel in-flight requestsWhen a newer keystroke arrives, abort the stale request so you never render suggestions for a prefix the user already moved past.
Local caching of recent suggestionsRecent prefixes have very high reuse (backspacing, re-typing); serve from local cache with zero network round-trip.
Early connection + prefetchOpen the TCP/TLS connection on page load, so the first keystroke doesn't pay handshake latency. Prefetch likely next-prefix results.
Edge / CDN cache of head prefixesPush the hottest prefixes' answers to CDN/ISP edges so common queries never reach origin.
The <100ms latency budget — where it goesIf the target is "suggestions feel instant" (~100ms perceived), the budget is tight: ~debounce wait (50ms, overlaps typing) + network RTT (often 20–80ms alone) + server processing. That leaves the trie lookup itself only a few milliseconds of headroom — which is precisely why the index must be RAM-resident, top-k must be precomputed (no request-time sort), and the head prefixes must be cached at the edge. There is no budget for a disk seek, a database query, or a subtree scan. Every "expensive" decision in this design exists to defend single-digit-millisecond server time.
