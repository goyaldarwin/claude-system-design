Designing a Unique ID Generator

Every distributed system needs to mint unique identifiers — for tweets, orders, messages — at high rate, across many machines, with no coordination bottleneck. A single auto-increment column can't keep up and is a single point of failure; random UUIDs are unique but kill database performance. The elegant answer, Twitter's Snowflake, packs time + machine + sequence into 64 bits — and its one sharp edge is what happens when a clock moves backwards.

The crux Globally-unique, roughly-sortable, 64-bit IDs at scale with no central coordinator — in exchange for depending on loosely-synced clocks and a per-node sequence limit. Encode time + machine id + a counter so each node generates IDs alone, no lock or round-trip. The price: IDs are only as monotonic as your clocks, and a backwards clock or a sequence overflow must be handled explicitly.

1. Requirements

Functional:

  • Generate globally unique IDs (never a collision, ever).
  • IDs are 64-bit (fit a BIGINT / long — UUIDs' 128 bits are wasteful as a primary key).
  • IDs are roughly time-sortable (sortable by creation order — hugely useful for feeds, pagination, and B-tree insert locality).

Non-functional:

  • High throughput — tens of thousands of IDs/sec per node, millions across the fleet.
  • Low latency — generating an ID must be near-instant (ideally no network call).
  • Highly available & no single point of failure — a central allocator everyone calls is both.

2. Capacity estimate — work the bits

The capacity math here is a bit budget — how to spend 64 bits to meet the requirements. Let's derive it:

  • 1 bit — sign bit, left 0 (so the ID is a positive signed 64-bit integer). 63 bits to spend.
  • Timestamp: we want IDs sortable by time and the system to last decades. 41 bits of milliseconds = 241 ms ≈ 69 years. Using a custom epoch (start counting from your launch date, not 1970) gives the full 69 years from launch. → 41 bits.
  • Machine id: how many ID-generating nodes? 10 bits = 1,024 nodes. Plenty for a large fleet. → 10 bits.
  • Sequence: the remaining 12 bits = 4,096 values — this is how many IDs one node can mint per millisecond. So per node: 4,096 IDs/ms = ~4.096 million IDs/second; across 1,024 nodes, ~4 billion IDs/sec. → 12 bits.

1 + 41 + 10 + 12 = 64. Every bit is accounted for, and each allocation maps directly to a requirement: timestamp → sortability + lifespan, machine id → fleet size, sequence → per-node throughput. The consequence: if you need more than 4,096 IDs per node per millisecond, you've overflowed the sequence (sub-problem 4) — the bit budget is the design.

3. The API

Deliberately tiny — the whole point is a local call, no parameters, no network:

long nextId()        // returns the next unique 64-bit id

It runs in-process on each application node (or as a sidecar), so generating an ID is a few CPU instructions — no round-trip, which is what lets it hit millions/sec at sub-microsecond latency. Contrast a central "ID service" you'd call over the network: that adds latency and is a single point of failure.

4. High-level design

Each node independently builds an ID by composing three numbers it can produce locally:

  1. Read the current time in ms (minus the custom epoch) → the 41-bit timestamp.
  2. Use its assigned machine id → the 10-bit machine field (set once at startup).
  3. Take the next value of a per-millisecond counter → the 12-bit sequence (resets to 0 each new millisecond).

Bit-shift and OR them together into one 64-bit integer. No two nodes share a machine id, so even if two nodes generate at the same millisecond with the same sequence, the machine-id field makes the IDs differ — uniqueness without any coordination. And because the timestamp is the high-order bits, IDs sort by creation time.

5. The bit layout (the "data model")

BitsFieldMeaning
1signAlways 0 (keeps it a positive number)
41timestampMilliseconds since a custom epoch (~69 yrs) — high-order, so IDs sort by time
10machine idWhich node generated it (1,024 nodes)
12sequencePer-node counter within one millisecond (4,096/ms)

This is the Snowflake layout. The split is tunable: more machine-id bits (bigger fleet) means fewer sequence bits (lower per-node rate), and vice-versa — you spend the 63 bits to match your fleet size and throughput.

Scenario 3 — two nodes boot with the same machine id Through a config bug, two nodes start up with machine id 42. Now they can each independently produce the same (timestamp, machine, sequence) triple — a duplicate ID, silently. How do you assign machine ids so this can never happen, even as nodes autoscale up and down?

Scenario 1 is why not the obvious options. Scenario 2 is the clock-backwards problem. Scenario 3 is machine-id assignment. Each maps to a sub-problem below.

The four hard sub-problems

The design above is the easy 80%. These four are where it gets interesting — each with a nested failure.

Sub-problem 1 — Why not the obvious options?

Three tempting alternatives and why they fall short: (a) Database auto-increment — simple and sortable, but every insert needs the one database (a throughput bottleneck and single point of failure); sharded auto-increment with offsets is fragile. (b) UUID v4 (random 128-bit) — generated anywhere with no coordination, but 128 bits is twice the storage as a key, and being random they destroy B-tree insert locality. (c) Snowflake — 64-bit, coordination-free, time-sortable: the sweet spot.

The failure that happens when you get this wrong — random UUIDs kill the index This is Scenario 1. A random UUID as a primary key means each insert lands at a random position in the table's B-tree index — so the database constantly splits pages all over the tree, thrashes the buffer cache (the hot insertion point is everywhere), and fragments storage. Sequential-ish IDs (like Snowflake's time-ordered ones) insert at the end of the index — cheap, cache-friendly, no fragmentation. This is why "just use UUIDs" is a real performance problem at write-heavy scale, and why time-sortable IDs matter beyond convenience. (UUID v7, time-ordered, fixes much of this — the lesson is to keep IDs roughly sequential.)

Sub-problem 2 — The clock-backwards problem

Snowflake's uniqueness rests on the timestamp never decreasing. But wall clocks lie — NTP corrections and leap seconds can step a clock backwards. If the timestamp field goes back to a value the node already used, it can re-mint a (timestamp, machine, sequence) triple it already issued → a duplicate ID, the one thing that must never happen.

The failure that happens when you get this wrong — duplicate IDs on a backwards clock This is Scenario 2. The standard Snowflake defense: the generator remembers the last timestamp it used, and if the current clock reading is less than that, it knows time went backwards — so it refuses to generate (throws/blocks) until the clock catches back up to the last-seen time. Refusing is the safe choice: a brief generation stall (usually milliseconds) is vastly preferable to silently issuing a duplicate ID that corrupts your data. Mitigate the stall by keeping clocks tightly synced and, for small rewinds, briefly waiting; for large rewinds, alert — something is wrong with that node's clock. (This is exactly the "time can't go backwards" lesson from the leap-second outages, applied.)

Sub-problem 3 — Assigning machine IDs without collision

Uniqueness depends on every node having a distinct machine id — so how do nodes get one, especially when they autoscale (nodes come and go, and you can't hardcode)? Options: a coordination service (ZooKeeper/etcd) hands out an unused id from the 1,024-space on startup (and reclaims it when a node dies); or derive it from something already unique (a stable host index, an ENI/IP-based scheme). The id must be claimed atomically so two nodes can't grab the same one.

The failure that happens when you get this wrong — two nodes, same machine id This is Scenario 3. If two live nodes share a machine id, they can each generate the identical (timestamp, machine, sequence) triple — a silent duplicate, invisible until two rows collide. Prevent it with atomic id assignment: a node, on startup, claims an id from a coordination kernel using create-if-not-exists on an ephemeral key — so the id is held only while the node is alive and is freed (for reuse) the moment it dies, and two nodes can never claim the same id. Hardcoding or deriving ids from non-unique sources (a reused config, a non-stable hostname) is the trap. The 10-bit space (1,024) bounds the fleet — size that field for your max node count.

Sub-problem 4 — Sequence overflow within a millisecond

The 12-bit sequence allows 4,096 IDs per node per millisecond. What if a node needs to generate the 4,097th ID in the same millisecond? The sequence has overflowed.

The failure that happens when you get this wrong — running out of sequence The correct handling: when the sequence hits its max within a millisecond, wait (busy-spin) until the next millisecond, then reset the sequence to 0 and continue. This caps a single node at 4,096 IDs/ms (~4M/sec) — if you genuinely need more per node, you re-budget the bits (more sequence bits, fewer machine/timestamp bits) or scale out to more nodes. The bug to avoid is letting the sequence wrap around to 0 within the same millisecond without advancing time — that re-issues IDs already used this millisecond (duplicates). The wait-for-next-ms is a tiny stall that preserves correctness; it's the same "correctness over a brief stall" principle as the clock-backwards case.

The rule in one lineUse a Snowflake-style time+machine+sequence 64-bit ID for coordination-free, sortable, compact IDs — and handle its three sharp edges: refuse to generate on a backwards clock, assign machine ids atomically (coordination kernel), and wait for the next ms on sequence overflow.

The takeawayYou rarely need a bespoke service — a Snowflake library + a coordination kernel for machine ids, or your database's native UUID v7 (which bakes the time-ordering in), usually suffices. The clock-sync quality of your platform directly affects how often the backwards-clock guard trips.

Famous implementations

The approaches in real systems. Same facets — what it is, the mechanism, the failure to watch.

Twitter Snowflake

  • What it is. The originating 64-bit time|machine|sequence design (41|10|12 + sign bit).
  • Mechanism. Local generation, no runtime coordination; refuses to generate if the clock moves backward.
  • The failure to watch. Clock skew/rewind (the backwards-clock guard) and machine-id assignment — the two things that can produce duplicates.

Instagram (sharded Postgres)

  • What it is. A Snowflake-like scheme baked into Postgres: 41 bits time + shard id + per-shard sequence (mod 1024), generated by a stored procedure.
  • Mechanism. The shard id doubles as the machine id, so an ID also encodes where the row lives — combining ID generation with logical sharding.
  • The failure to watch. Per-shard-per-ms sequence cap (1,024); the scheme is tied to their shard layout.

Flickr ticket server

  • What it is. A pre-Snowflake approach: a dedicated MySQL with an auto-increment table that hands out IDs.
  • Mechanism. Two ticket servers (odd/even auto-increment offsets) for HA; clients batch-fetch blocks of ids to avoid a round-trip per id.
  • The failure to watch. It's still a service to run and a (mitigated) central dependency; batch-fetch loses strict ordering across clients.

Build it yourself

Work each before opening the approach.

Re-budget the bits for a different scale

You need IDs for a fleet of up to 4,000 nodes, each generating up to 1,000 IDs/ms, lasting 30+ years. Re-allocate the 63 bits and check it works.

Reveal the approach

4,000 nodes needs 12 machine-id bits (4,096 ≥ 4,000). 1,000 IDs/ms needs 10 sequence bits (1,024 ≥ 1,000). That leaves 41 bits for timestamp (63 − 12 − 10 = 41) = ~69 years of ms — comfortably ≥ 30. So 1(sign)+41(time)+12(machine)+10(seq) = 64. Note the trade: by giving more bits to machine id you took them from sequence, lowering per-node rate — that's the fundamental tension. If you needed both more nodes and more per-node rate, you'd have to shorten the timestamp (shorter lifespan or coarser-than-ms resolution) or accept a larger ID. The bit budget makes every requirement a quantitative trade-off — which is the whole lesson.

Handle a clock that jumps backward 5 seconds

A node's NTP correction yanks the clock back 5 seconds — far longer than a brief wait can absorb. Design the behavior so you never emit a duplicate and the system degrades sanely.

Reveal the approach

The generator tracks lastTimestamp. On each request, read the clock: if it's ≥ lastTimestamp, proceed normally. If it's behind by a tiny amount (a few ms), busy-wait until it catches up — cheap, invisible. If it's behind by a large amount (5s), don't block for 5 seconds: instead refuse to generate and raise an alert / take the node out of rotation — a 5-second backward jump means that node's clock is unhealthy and shouldn't mint IDs at all. Other nodes (with good clocks) keep serving, so the fleet stays available. Some designs keep generating using lastTimestamp (not the wall clock) and a continuing sequence until real time catches up — valid as long as you never reuse a (time, machine, seq) triple. The non-negotiable: correctness (no duplicates) over availability of that one node.

Design-it-yourself: the open-ended ones

  • IDs that encode their shard. Like Instagram, design IDs so the ID itself tells you which shard the row lives on — and discuss the coupling cost. → Sharding.
  • Short, user-facing IDs. A URL shortener needs short base-62 codes, not 64-bit numbers. Adapt the approach and discuss the trade-offs. → URL Shortener.
  • No-coordination machine ids. You can't run ZooKeeper. Design machine-id assignment from cloud primitives (instance metadata, IP) that still guarantees uniqueness. → Coordination.
