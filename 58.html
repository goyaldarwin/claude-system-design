Designing a Unique ID Generator

How to mint billions of globally unique, roughly time-ordered 64-bit identifiers per day with no central coordinator on the hot path.

The bottleneck No shared truth. unique IDs without coordination, paid in clock-skew handling and roughly-sortable bit layouts.

Requirements and back-of-envelope

The problem: "a service that generates unique IDs in a distributed system." Before sketching anything, pin down the functional and non-functional requirements — the right answer is entirely determined by them.

  • Uniqueness: no two IDs ever collide, across all data centers, forever.
  • Size: typically 64-bit integers. They fit a BIGINT / long, index densely, and are half the width of a 128-bit UUID in every B-tree page, network frame, and cache line.
  • Sortable / k-sorted: IDs should be roughly time-ordered so that newer rows sort after older ones. This is a soft requirement that has outsized impact on storage-engine performance (see below).
  • Throughput & latency: e.g. 10,000+ IDs/sec per node, p99 < 1 ms, ideally generated locally with no network round-trip.
  • No single point of failure: the generator must not depend on a central server on the critical path.
Key insightThe whole problem is a tension between two goals that pull in opposite directions: uniqueness wants global coordination, while throughput and availability want zero coordination. Every approach below is a different point on that trade-off curve. The winning designs push coordination off the hot path — they coordinate rarely (a machine-id lease) so they can generate IDs locally (no round-trip per ID).

Sanity numbers

A 64-bit unsigned space is ~1.8×1019 values. Twitter at peak tweeted < 104/sec; even a hyperscaler minting 106 IDs/sec burns through only ~3×1013/year — you will exhaust the timestamp field (a 41-bit ms clock lasts ~69 years) long before the space. The real scarcity is bits, not IDs.

Approach 1: UUIDs

A UUID (RFC 4122) is a 128-bit value. Two variants matter here:

  • UUIDv4 — 122 random bits. Collision probability is negligible (you'd need ~1018 UUIDs for a 50% chance). Fully decentralized, zero coordination, generate anywhere offline. But it is not sortable — consecutive inserts scatter randomly across the index.
  • UUIDv1 — 60-bit timestamp + 14-bit clock sequence + 48-bit MAC/node. Time-ordered in principle, but the timestamp is laid out in a byte order that is not lexicographically sortable, and it leaks the MAC address (privacy / security concern). UUIDv6/v7 (2024 RFC 9562) fix the ordering — v7 is a Unix-ms timestamp + random, and is the modern recommendation when you can afford 128 bits.
PitfallThe killer problem with random UUIDs (v4) as a primary key is write amplification. In an InnoDB B-tree (clustered index) or any B-tree storage engine, random keys cause inserts to land in random pages — constant page splits, a cold buffer pool, and poor fill factor. Benchmarks routinely show 2–10× slower bulk inserts and a 30–40% larger index versus a monotonic key. See Storage Engines: LSM vs B-Tree. Also: 128 bits is double the storage and, because secondary indexes in InnoDB embed the PK, that bloat multiplies.

verdict Great when you need offline / client-side generation and don't care about 64-bit or sort order. For a system requiring a sortable 64-bit ID, UUIDv4 is a wrong answer; UUIDv7 is acceptable only if the 128-bit width is fine.

Approach 2: DB auto-increment

The naive answer: a single table with an AUTO_INCREMENT / SERIAL column. Insert a row, read back the generated key.

It gives you small, perfectly monotonic, sortable IDs — but it has two fatal flaws at scale:

  • SPOF and write ceiling: a single primary DB is one machine's worth of write throughput and one failure domain. Every ID generation is a network round-trip plus a durable write.
  • Doesn't shard: the moment you partition the database (see Sharding / Data Partitioning), each shard's counter is independent and they collide.

The classic workaround is stepped sequences: configure N database servers with auto_increment_increment = N and distinct auto_increment_offset (server 1 → 1,3,5,7…; server 2 → 2,4,6,8… for N=2). This removes the single SPOF but is brittle: adding a node requires re-planning the modulus, IDs are only weakly time-ordered across servers, and you've coupled ID generation to DB topology.

Approach 3: Ticket / segment servers (Flickr)

Flickr's well-known design uses dedicated ticket servers — tiny MySQL boxes whose only job is to vend IDs via REPLACE INTO on a single-row table:

REPLACE INTO Tickets64 (stub) VALUES ('a');
SELECT LAST_INSERT_ID();

Two ticket servers run with offset 1/2 and increment 2 (odd vs even) so a failure of one doesn't stop ID issuance. The crucial scaling refinement is the segment (range) allocator, used by systems like Meituan's Leaf: instead of one ID per round-trip, a service requests a block of IDs (e.g. 1,000) in a single transaction, then hands them out from local memory.

-- atomic range claim
UPDATE id_alloc SET max_id = max_id + 1000
  WHERE biz_tag = 'orders';
SELECT max_id FROM id_alloc WHERE biz_tag = 'orders';
-- server now owns [max_id-1000, max_id), serves locally
What strong engineers doThey immediately propose double buffering on the segment allocator: prefetch the next block asynchronously when the current block is ~80% consumed, so a client never blocks on the DB round-trip. This makes p99 latency independent of the DB and tolerates short DB outages (you keep serving from the buffer). It is the difference between a textbook answer and a production one.
Trade-offSegment allocation trades density for availability: on every restart you discard the unused tail of the current block, so IDs have gaps. That's fine (gaps are allowed; only uniqueness matters) but it means IDs are k-sorted across servers, not globally monotonic — two servers holding adjacent blocks can interleave.

Approach 4: Twitter Snowflake

Snowflake is the canonical answer for a distributed, coordination-free, sortable 64-bit ID. Every node generates IDs locally by composing three fields into a single 64-bit integer. The original Twitter layout:

 0 | 41-bit timestamp (ms since epoch) | 10-bit machine id | 12-bit sequence
 ^   ^                                   ^                    ^
 |   |                                   |                    +-- 4096 ids / ms / node
 |   |                                   +-- 1024 nodes (5 dc + 5 worker, or flat)
 |   +-- ~69 years from a custom epoch
 +-- sign bit, always 0 -> id stays positive in signed long
FieldBitsCapacityPurpose
Sign1always 0keeps the value positive in a signed 64-bit long
Timestamp41~69 yrs of mshigh bits → sort order; use a custom epoch to extend lifespan
Machine id101024 nodesguarantees cross-node uniqueness without coordination per ID
Sequence124096 / msdisambiguates IDs minted in the same millisecond on one node

Because the timestamp occupies the high-order bits, the integer ordering of two IDs matches their time ordering — this is what makes Snowflake IDs sortable. The generation logic:

synchronized long nextId() {
  long now = currentTimeMillis();
  if (now < lastTs)                 // clock moved backwards!
      throw new ClockBackwardException(lastTs - now);
  if (now == lastTs) {
      seq = (seq + 1) & 0xFFF;       // 12-bit mask
      if (seq == 0) now = waitNextMillis(lastTs);  // burned 4096, spin
  } else {
      seq = 0;
  }
  lastTs = now;
  return ((now - EPOCH) << 22)       // 10 + 12 = 22 bits below timestamp
       | (machineId << 12)
       | seq;
}

Field widths are negotiable per workload: if you have only a few nodes but huge per-node throughput, steal bits from machine-id and give them to the sequence. Discord, Instagram, and Sonyflake all ship Snowflake variants with different splits (Instagram embeds a shard id; Sonyflake uses 10 ms ticks to last 174 years with fewer sequence bits).

The clock problem: skew and rollback

Snowflake's correctness rests entirely on the wall clock being monotonically non-decreasing. This is its single biggest operational hazard.

  • Clock skew: two nodes' clocks differ by tens of ms. This is harmless for uniqueness (machine-id keeps them distinct) but means IDs are only k-sorted — globally ordered within the skew window, not strictly. Don't promise strict total order across nodes.
  • Clock rollback: NTP slews or steps the clock backwards, or a VM is restored from a snapshot. If now < lastTs, naive code could re-mint an already-issued (timestamp, machine-id, seq) triple — a collision.
PitfallNever run NTP in step mode on an ID-generation host. Use chrony in slew-only mode (maxslewrate, no makestep) so the clock only ever speeds up or slows down, never jumps back. A backward jump is the one thing the algorithm cannot tolerate.

Mitigation strategies, in increasing robustness:

  1. Refuse and wait: if the rollback is small (< a few ms), block until the clock catches up to lastTs. Simple, but stalls.
  2. Refuse and alarm: if the rollback exceeds a threshold, throw and let the node take itself out of rotation. Availability over a silent collision.
  3. Persist the high-water mark: periodically write lastTs to durable storage / ZooKeeper. On restart, refuse to issue IDs until the clock passes the persisted watermark. Baidu's UidGenerator and Meituan Leaf-snowflake do exactly this against ZooKeeper.
  4. Logical clocks: decouple the high bits from wall time entirely — advance a monotonic counter on every backward jump. This trades strict time-meaning for guaranteed monotonicity (related ideas in Time, Clocks and Ordering).

Why sortable IDs matter: ULID, KSUID, index locality

Sortability is not a nice-to-have; it directly governs storage-engine efficiency. Time-ordered keys turn random I/O into sequential I/O:

  • B-trees: monotonic inserts always append to the rightmost leaf page — minimal page splits, high fill factor, hot pages stay in the buffer pool. Random keys (UUIDv4) thrash the whole tree.
  • LSM trees: sorted keys compact cleanly and, critically, enable efficient range scans — "fetch the last hour of events" becomes a contiguous read. Time-prefixed keys also bound which SSTables a read must touch.

This is why storage-engine choice and ID design must be discussed together. Two popular wider sortable formats sit alongside Snowflake:

FormatBitsLayoutNotable
Snowflake6441 ts(ms) + 10 node + 12 seqfits BIGINT; needs coordinated machine-id
ULID12848 ts(ms) + 80 randomCrockford base32, lexicographically sortable as text; no node coordination
KSUID16032 ts(s) + 128 randomsecond-granularity, huge random space, URL-safe
Trade-offSnowflake buys you 64 bits and tight ordering at the cost of coordination (you must assign unique machine-ids). ULID/KSUID/UUIDv7 buy you zero coordination (random low bits) at the cost of 128/160 bits and only millisecond/second-level ordering with random tie-breaking. For a sharded database PK where width matters, Snowflake wins; for client-generated IDs or where you can't run a coordinator, ULID/UUIDv7 win.

Machine-id assignment and coordination

Snowflake's Achilles heel: two nodes must never share a machine-id, or they will collide within the same millisecond. The 10-bit field gives only 1024 slots, so static config files break the moment you autoscale, replace, or restart pods. Options:

  • ZooKeeper sequential znodes: on startup a node creates an ephemeral sequential znode under /snowflake/workers/; the sequence number (mod 1024) is its machine-id. The original Twitter Snowflake used exactly this. Ephemeral means the slot is reclaimed on disconnect (related: Leader Election and Coordination).
  • Lease + persistence (Leaf): get a worker-id from ZooKeeper, then cache it locally and persist the clock watermark, so a brief ZK outage doesn't stop generation.
  • Derive from infra: map machine-id from the pod's stable ordinal in a Kubernetes StatefulSet, or from a DB-backed registry that hands out leases with a TTL. The DB-segment approach (Leaf-segment) sidesteps the machine-id problem entirely.
PitfallThe dangerous edge case: a node loses its ZK session (GC pause, network blip), ZK reclaims its ephemeral slot and hands it to a new node, but the old node hasn't noticed yet and keeps minting IDs with the now-shared id. Mitigate by having the generator stop issuing as soon as it detects a lost session, and by ensuring the session timeout exceeds your max GC pause. This is the same fencing problem covered in Designing a Distributed Locking Service.

Putting it together

A pragmatic production design:

  1. Default: Snowflake generated in-process inside each application instance — no ID service, no network hop, lowest latency. Machine-id leased from ZooKeeper/etcd at startup.
  2. If clients can't be trusted to generate IDs (mobile, third parties): run a small fleet of stateless ID servers behind a load balancer (Load Balancing), each with its own machine-id, returning Snowflake IDs over a thin RPC.
  3. If you can't run a coordinator at all: use UUIDv7/ULID and accept 128 bits.
  4. If you need guaranteed gapless small integers (e.g. human-facing invoice numbers): a DB segment allocator with double buffering — but understand it serializes per biz-tag.

This pattern underpins real designs in this course: short-code generation in Designing a URL Shortening service like TinyURL is the same problem with a different encoding, and tweet/message IDs in Designing Twitter are literally where Snowflake was born.

Principal-level mastery check

What strong engineers doThey start by extracting the exact requirements (64-bit? sortable? coordination allowed? gapless?) because each one eliminates whole approaches — they don't reflexively answer "Snowflake." They proactively raise the clock-rollback failure mode and propose NTP slew-mode plus a persisted watermark before being prompted. They correctly distinguish uniqueness (hard guarantee) from ordering (best-effort, k-sorted) and never overclaim strict global ordering across nodes. And they connect ID design to downstream storage cost — write amplification in B-trees, compaction in LSM — rather than treating the generator in isolation.
  • "You proposed Snowflake. Walk me through what happens, bit by bit, when one machine mints two IDs in the same millisecond — and what happens on the 4097th."
  • "NTP steps the clock back 200 ms on one node. Trace the exact failure. Now design three independent layers of defense, and tell me which one you'd ship first."
  • "How does a freshly autoscaled pod get a machine-id? What guarantees no two live pods share one, and what's the failure window if ZooKeeper reclaims a slot too early?"
  • "A teammate wants random UUIDv4 as the clustered primary key in MySQL for 'simplicity.' Quantify what that costs at 50k inserts/sec versus a Snowflake key."
  • "The product asks for strictly increasing, gapless order numbers. Why can't Snowflake give you that, and what's the throughput ceiling of the design that can?"
  • "Your IDs are 'roughly sortable.' Define the exact bound on how out-of-order two IDs can be, in terms of your clock-skew SLA."
