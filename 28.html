Designing Uber Backend

Old = the original Grokking walkthrough, preserved.

Match riders to nearby drivers in real time. It fuses two hard problems you've now seen separately: tracking millions of moving objects (the geo problem) and atomically matching a rider to exactly one driver (the concurrency problem) — all under a relentless firehose of GPS updates.

The crux Index millions of moving drivers geospatially, then match each request to exactly one driver without double-assigning. The location-update firehose, the nearby-driver query, and the atomic match are the three hard parts.

1. Requirements

Functional: drivers stream their location; a rider requests a ride; the system finds nearby available drivers and matches one; both track the trip live; fare + payment at the end. Non-functional: low match latency, scale to millions of concurrent drivers/riders, highly available, location updates eventually consistent but matching must be correct. Extended: ETA/routing, surge pricing, driver dispatch optimization.

2. Capacity estimate

  • Location firehose: millions of active drivers each sending GPS every ~4 seconds = millions of writes/sec to the location index — the dominant write load.
  • Match queries: each ride request triggers a geospatial "nearby available drivers" query — read load proportional to request rate.
  • Matching must be exact: a driver can be offered to only one rider at a time — correctness under concurrency, not just throughput.

3. System API

updateLocation(driver_id, lat, lng)            // high-frequency firehose
requestRide(rider_id, pickup, dest) → trip_id  // triggers nearby-search + match
// server pushes: onMatch, onDriverLocation (live tracking)

4. High-level architecture

  • Location-ingest path: driver apps stream GPS → an in-memory geospatial index of available drivers (the write-heavy core).
  • Matching service: on a request, query the index for nearby available drivers, then atomically claim one.
  • Trip service + store: trip lifecycle, live tracking (push), fare, payment.
The shape in one sentenceA firehose of GPS updates keeps an in-memory geo index of available drivers current; a match request queries that index for nearby drivers and atomically claims one — geo + concurrency, fused.

Three scenarios — the hard parts, as questions

Scenario 1 — a million GPS pings a second Every active driver sends location every ~4 seconds. That's millions of writes/sec, each potentially moving a driver between geo cells. What kind of store survives that, and what consistency can you relax?
Scenario 2 — "find me a driver" A rider requests a ride downtown. You need available drivers within ~2 km, fast. Over millions of constantly-moving drivers, how do you answer that without scanning them all — and what lesson does this reuse?
Scenario 3 — two riders, one driver Two ride requests near the same driver arrive in the same instant. Both nearby-searches return that driver as the best match. What must NOT happen, and how do you guarantee the driver is offered to exactly one rider?

Scenario 1 is the location firehose. Scenario 2 is the nearby query (reuses Yelp's geo index). Scenario 3 is atomic matching.

6. The hard sub-problems

Sub-problem 1 — Ingesting driver locations

Millions of drivers × a ping every few seconds is a colossal write rate, and each ping may re-bucket a driver into a new geo cell. This must not melt the index.

The failure that happens when you get this wrong — the location write storm This is Scenario 1, the moving-object problem from Nearby Friends, at Uber scale. Writing every GPS ping to a strongly-indexed durable store is impossible. The answer: an in-memory geo index sized for write throughput, eventual consistency on positions (a driver's dot being 1–2s stale is fine), only re-bucket on cell-boundary crossing (not every sub-meter jitter), and treat the location stream as ephemeral (you don't need durable history of every ping for matching). The durable store holds trips, not the firehose.

Sub-problem 2 — Finding nearby drivers

A ride request needs available drivers near the pickup — exactly the geospatial proximity query, over a moving, filtered (available-only) set.

The failure that happens when you get this wrong — the geo scan This is Scenario 2. Scanning millions of drivers per request is hopeless; the fix is the same cell index (geohash/quadtree/S2) from Yelp: convert pickup to a cell, look up available drivers in that cell + neighbors, compute exact distance/ETA on those few, rank. The only difference from Yelp is the index is constantly updated (Sub-problem 1) and filtered to available drivers. Dense-city skew is handled the same way (adaptive cells). This is why Uber, Yelp, and Tinder all reduce to the same geo-index core.

Sub-problem 3 — Atomic matching

The genuinely new problem versus Yelp: a driver can be offered to only one rider. Two simultaneous requests must not both claim the same driver.

The failure that happens when you get this wrong — the double-assigned driver This is Scenario 3. If two matching requests both read "driver X available" and both assign X, you've double-booked — one rider gets a no-show. This is a concurrency problem, not a geo one. The fix is to make the claim atomic: when matching picks driver X, acquire a distributed lock on X (e.g. Redis SET driver:X NX with a TTL) before offering the ride; if the lock fails, X was just taken — pick the next candidate. The TTL auto-releases X if the driver doesn't accept in time. (Same family as BookMyShow's seat hold and the general distributed lock.) Mark X unavailable in the geo index on claim so subsequent searches skip them.

Famous implementations

Uber's geo + dispatch (H3, Ringpop)

  • What it is. Uber built H3 (a hexagonal global grid) for its geo-indexing and Ringpop for sharded, consistent-hashed dispatch.
  • Key idea. Hexagons have uniform neighbor distance (no corner/edge ambiguity of squares); dispatch is sharded by geography.
  • The failure to watch. Cell granularity + dense-city skew; matching consistency across shards.

Redis (geo + distributed lock)

  • What it is. One store doing double duty: GEOADD/GEOSEARCH for the live index and SET NX PX for the match lock.
  • Key idea. In-memory speed for both the firehose and the atomic claim. (Distributed Locking.)
  • The failure to watch. Lock correctness under Redis failover (see the distributed-lock lesson's caveats).

Build it yourself

Design the atomic match

Two riders' nearby-searches both return driver X as best. Design the claim so exactly one gets X and the other smoothly gets the next driver.

Reveal the approach

Matching produces a ranked candidate list per request (nearest/soonest first). Before offering a driver, attempt an atomic claim: SET lock:driver:X rider NX PX 15000 (set-if-not-exists, 15s TTL). If it succeeds, you hold X — offer the ride and mark X unavailable in the geo index. If it fails, X was just claimed by the other request, so move to the next candidate in your list. The TTL ensures that if the driver doesn't accept (or the claimer crashes), X is auto-released and becomes matchable again. So two simultaneous requests for X: one wins the NX, the other transparently falls through to its second choice — no double-assignment, no manual coordination. (Mind lock correctness across Redis failover — lesson 64.)

Design-it-yourself

  • Surge pricing. Detect demand > supply per region in real time and adjust price. → Ad Click Aggregation (windowed counts).
  • ETA / routing. Estimate arrival times over a live road graph.

Let's design a ride-sharing service like Uber, which connects passengers who need a ride with drivers who have a car.

Similar Services: Lyft, Didi, Via, Sidecar etc.

What is Uber?

Uber enables its customers to book drivers for taxi rides. Uber drivers use their personal cars to drive customers around. Both customers and drivers communicate with each other through their smartphones using Uber app.

Requirements and Goals of the System

Let's start with building a simpler version of Uber.

There are two kinds of users in our system 1) Drivers 2) Customers.

  • Drivers need to regularly notify the service about their current location and their availability to pick passengers.
  • Passengers get to see all the nearby available drivers.
  • Customer can request a ride; nearby drivers are notified that a customer is ready to be picked up.
  • Once a driver and customer accept a ride, they can constantly see each other's current location, until the trip finishes.
  • Upon reaching the destination, the driver marks the journey complete to become available for the next ride.

Capacity Estimation and Constraints

  • Let's assume we have 300M customer and 1M drivers, with 1M daily active customers and 500K daily active drivers.
  • Let's assume 1M daily rides.
  • Let's assume that all active drivers notify their current location every three seconds.
  • Once a customer puts a request for a ride, the system should be able to contact drivers in real-time.

Basic System Design and Algorithm

We will take the solution discussed in Designing Yelp and modify it to make it work for above-mentioned "Uber" use cases. The biggest difference we have is that our QuadTree was not built keeping in mind that there will be frequent updates to it. So, we have two issues with our Dynamic Grid solution:

  • Since all active drivers are reporting their locations every three seconds, we need to update our data structures to reflect that. If we have to update the QuadTree for every change in the driver's position, it will take a lot of time and resources. To update a driver to its new location, we must find the right grid based on the driver's previous location. If the new position does not belong to the current grid, we have to remove the driver from the current grid and move/reinsert the user to the correct grid. After this move, if the new grid reaches the maximum limit of drivers, we have to repartition it.
  • We need to have a quick mechanism to propagate current location of all the nearby drivers to any active customer in that area. Also, when a ride is in progress, our system needs to notify both the driver and passenger about the current location of the car.

Although our QuadTree helps us find nearby drivers quickly, a fast update in the tree is not guaranteed.

Do we need to modify our QuadTree every time a driver reports their location? If we don't update our QuadTree with every update from the driver, it will have some old data and will not reflect the current location of drivers correctly. If you recall, our purpose of building the QuadTree was to find nearby drivers (or places) efficiently. Since all active drivers report their location every three seconds, hence there will be lot more updates happening to our tree than querying for nearby drivers. So, what if we keep the latest position reported by all drivers in a hash table and update our QuadTree a little less frequent? Let's assume we guarantee that a driver's current location will be reflected in the QuadTree within 15 seconds. Meanwhile, we will maintain a hash table that will store the current location reported by drivers; let's call this DriverLocationHT.

How much memory we need for DriverLocationHT? We need to store DriverID, their present and old location in the hash table. So we need total 35 bytes to store one record:

  1. DriverID (3 bytes - 1 million drivers)
  2. Old latitude (8 bytes)
  3. Old longitude (8 bytes)
  4. New latitude (8 bytes)
  5. New longitude (8 bytes) Total = 35 bytes

If we have 1 million total drivers, we need the following memory (ignoring hash table overhead):

1 million * 35 bytes => 35 MB

How much bandwidth our service will consume to receive location updates from all drivers? If we get DriverID and their location, it will be (3+16 => 19 bytes). If we receive this information every three seconds from one million drivers, we will be getting 19MB per three seconds.

Do we need to distribute DriverLocationHT onto multiple servers? Although our memory and bandwidth requirements do not dictate that, since all this information can easily fit on one server, but for scalability, performance, and fault tolerance we should distribute the hash table onto multiple servers. We can distribute based on the DriverID to make the distribution completely random. Let's call the machines holding DriverLocationHT, Driver Location server. Other than storing driver's location, each of these servers will do two things:

  1. As soon as the server receives an update for a driver's location, they will broadcast that information to all the interested customers.
  2. The server needs to notify respective QuadTree server to refresh the driver's location. As discussed above, this can happen every 10 seconds.

How can we efficiently broadcast driver's location to customers? We can have a Push Model, where the server will push the positions to all the relevant users. We can have a dedicated Notification Service that can broadcast the current location of drivers to all the interested customers. We can build our Notification service on publisher/subscriber model. When a customer opens the Uber app on their cell phone, they query the server to find nearby drivers. On the server side, before returning the list of drivers to the customer, we will subscribe the customer for all the updates from those drivers. We can maintain a list of customers (subscribers) interested in knowing the location of a driver and whenever we have an update in DriverLocationHT for that driver, we can broadcast the current location of the driver to all subscribed customers. This way our system makes sure that we always show driver's current position to the customer.

How much memory we need to store all these subscriptions? As we have estimated above we will have 1M daily active customers and 500K daily active drivers. On the average let's assume that five customers subscribe one driver. Let's assume we store all this information in a hash table so that we can update it efficiently. We need to store driver and customer IDs to maintain the subscriptions. Assuming we will need 3 bytes for DriverID and 8 bytes for CustomerID, we will need 21MB of memory.

(500K * 3) + (500K * 5 * 8 ) ~= 21 MB

How much bandwidth we need to broadcast the driver's location to customers? For every active driver we have five subscribers, so total subscribers we have:

5 * 500K => 2.5M

To all these customers we need to send DriverID (3 bytes) and their location (16 bytes) every second, so we need the following bandwidth:

2.5M * 19 bytes => 47.5 MB/s

How can we efficiently implement Notification service? We can either use HTTP long polling or push notifications.

How the new publishers/drivers will get added for a current customer? As we have proposed above that customers will be subscribed to nearby drivers when they open the Uber app for the first time, what will happen when a new driver enters the area the customer is looking at? To add a new customer/driver subscription dynamically, we need to keep track of the area the customer is watching. This will make our solution complicated, what if instead of pushing this information, clients pull it from the server?

How about if clients pull information about nearby drivers from the server? Clients can send their current location, and the server will find all the nearby drivers from the QuadTree to return them to the client. Upon receiving this information, the client can update their screen to reflect current positions of the drivers. Clients can query every five seconds to limit the number of round trips to the server. This solution looks quite simpler compared to the push model described above.

Do we need to repartition a grid as soon as it reaches the maximum limit? We can have a cushion to let each grid grow a little bigger beyond the limit before we decide to partition it. Let's say our grids can grow/shrink extra 10% before we partition/merge them. This should decrease the load for grid partition or merge on high traffic grids.

How would "Request Ride" use case work?

  1. The customer will put a request for a ride.
  2. One of the Aggregator servers will take the request and asks QuadTree servers to return nearby drivers.
  3. The Aggregator server collects all the results and sorts them by ratings.
  4. The Aggregator server will send a notification to the top (say three) drivers simultaneously, whichever driver accepts the request first will be assigned the ride. The other drivers will receive a cancellation request. If none of the three drivers respond, the Aggregator will request a ride from the next three drivers from the list.
  5. Once a driver accepts a request, the customer is notified.

Fault Tolerance and Replication

What if a Driver Location server or Notification server dies? We would need replicas of these servers, so that if the primary dies the secondary can take control. Also, we can store this data in some persistent storage like SSDs that can provide fast IOs; this will ensure that if both primary and secondary servers die we can recover the data from the persistent storage.

Ranking

How about if we want to rank the search results not just by proximity but also by popularity or relevance?

How can we return top rated drivers within a given radius? Let's assume we keep track of the overall ratings of each driver in our database and QuadTree. An aggregated number can represent this popularity in our system, e.g., how many stars a driver gets out of ten? While searching for top 10 drivers within a given radius, we can ask each partition of the QuadTree to return top 10 drivers with maximum rating. The aggregator server can then determine top 10 drivers among all the drivers returned by different partitions.

Advanced Issues

  1. How to handle clients on slow and disconnecting networks?
  2. What if a client gets disconnected when it was a part of a ride? How will we handle billing in such a scenario?
  3. How about if clients pull all the information as compared to servers always pushing it?
Principal-level deep dive

Deeper analysis: trade-offs & failure modes

The classic write-up above is essentially "Yelp + a hash table for moving objects." That framing is correct but it hides the parts that actually break in production. Uber's real architecture (publicly described in their engineering blog around the Ringpop, Schemaless, and H3 eras) solved several problems the textbook version waves away. Below are the trade-offs a principal engineer is expected to reason about.

Core tensionThe geo-index needs to be fresh (drivers move every few seconds) AND cheap to query AND cheap to update. A single data structure cannot win all three. The whole design is a series of choices about which axis to sacrifice where.

1. Location ingestion: the firehose problem

The estimate of "1M drivers x update every 3s = ~333K writes/sec" undersells reality. Uber's production peak is millions of GPS pings per second once you count rider apps reporting too, plus the fact that a single "location update" is rarely one row — it triggers ETA recompute, supply heatmap updates, fraud/spoofing checks, and trip-tracking writes. Key decisions:

  • Do not write GPS pings to a transactional DB. They go into an in-memory store (the DriverLocationHT) and/or a stream (Kafka). The hash table is the source of truth for "where is driver X right now"; the durable store only needs the trip's path, sampled.
  • Decouple ingest from index update. The lesson's trick — keep the latest ping in a hash table, refresh the QuadTree only every 10–15s — is the single most important idea on the page. It converts 333K random tree mutations/sec into a much smaller, batched, amortized re-index. The cost is bounded staleness: a queried driver position can be up to ~15s old. For a 2-minute pickup that is acceptable; for live trip tracking it is not (so trip tracking bypasses the QuadTree and reads the hash table / a direct socket).
  • GPS is noisy. Raw pings jitter, drop, and lie (spoofing). Real systems run map-matching / Kalman filtering before the location is trusted, otherwise drivers teleport across the index and trigger spurious re-partitions.
Failure modeIf ingest and indexing are coupled synchronously, a GC pause or hot-shard slowdown on the index backs up the ingest path. Drivers' apps retry, amplifying load — a classic retry storm. Always put a buffer (queue/stream) between the firehose and any stateful index. See 40 Stream Processing.

2. Why moving objects break a static geo-index

A QuadTree (and a geohash prefix tree) is built for static points of interest — restaurants don't move. Three things go wrong when the points move continuously:

ProblemStatic index (Yelp)Moving index (Uber)
Update costRare; rebuild offlineEvery point moves every 3s — re-insert = remove + find new cell + insert
Cell crossingNeverConstant; a driver near a cell boundary flaps between cells
Re-partitionRare, batchA busy downtown cell hits its capacity limit and splits/merges repeatedly
ConcurrencyRead-mostly, easy to cacheWrite-heavy tree mutation needs locking — contention on hot cells

QuadTree rebalancing for moving objects is the crux. When a dense urban cell exceeds its driver limit it must split into four children; when drivers leave, siblings merge back. Under constant movement this thrashes. Mitigations the lesson hints at and production extends:

  • Hysteresis (the "10% cushion"). Don't split exactly at the limit; let a cell grow ~10% over before splitting and shrink ~10% under before merging. This stops oscillation around the threshold — the same idea as a thermostat's deadband.
  • Batched re-indexing. Because the hash table absorbs every ping, the tree only sees position deltas every 10–15s, so most boundary flapping is filtered out before it ever touches the tree.
  • Drop the tree entirely for a flat geo-grid. Uber's H3 (hexagonal hierarchical index) and Google's S2 map a lat/long to a fixed-resolution cell ID with O(1) math — no tree, no rebalancing. "Find nearby" becomes "fetch the buckets for this cell + its 6 neighbors." This trades adaptive density-balancing for dead-simple, lock-free, shardable updates. For a write-dominated workload this is almost always the better trade — which is why production Uber uses H3, not a QuadTree.
Why hexagons (H3)?Squares have two distance classes to neighbors (edge-adjacent vs corner-adjacent); hexagons have one — every neighbor is equidistant. That makes "expand the search ring" uniform and makes supply/demand heatmaps and surge zones behave consistently. The trade-off: hexagons don't subdivide cleanly into smaller hexagons, so H3's hierarchy is approximate, not exact.

3. Matching: "nearest available driver" is not just nearest

The page's Request-Ride flow (ask QuadTree for nearby, sort by rating, ping top 3) is a fine starting point but the real objective function is global, not greedy-local:

  • Nearest by straight-line distance ≠ nearest by ETA. A driver 200m away across a river with no bridge is "far." Matching must use road-network ETA (routing engine), not haversine distance.
  • Greedy per-request matching is locally optimal, globally bad. Assigning the closest driver to each request as it arrives can strand a slightly-farther rider whose only nearby driver just got taken. Uber moved to batched matching: collect requests over a short window (~seconds) within a region and solve an assignment problem (bipartite matching / Hungarian-style optimization) minimizing total wait time across all pairs. This raised match rates measurably.
  • The "ping top 3, first-accept-wins" race. Sending the offer to several drivers concurrently and taking the first acceptance requires a way to atomically assign the ride and cancel the losers. This is a distributed mutual-exclusion problem — see ride-state consistency below.

4. ETA computation

ETA appears trivially ("just route it") but is a whole subsystem. It feeds matching, pricing, and the rider's pickup countdown. Naive shortest-path over a live road graph for every request doesn't scale, so production systems precompute contraction hierarchies / partitioned shortest-path tables and layer a learned model on top to predict real travel time from historical + live traffic (the road says 6 min, history says it's 11 at 5pm on Friday). ETA is therefore a prediction, not a fact, and its error directly degrades pickup experience and surge accuracy.

5. Surge pricing

Surge is a feedback controller over the geo-grid, not a feature bolted on at the end. For each cell (H3 hexagon), the system continuously computes a supply/demand ratio from the same location + request streams and raises the multiplier when demand outstrips nearby supply. Subtleties:

  • It must be eventually consistent and quoted-then-honored. The price shown at request time is locked for that trip even though the underlying multiplier moves — otherwise riders see the fare change mid-tap. That means a short-lived price quote token, not a live read.
  • It is a control loop and can oscillate. High surge suppresses demand and attracts supply, which drops surge, which brings demand back — smoothing/damping is needed or prices flap minute to minute.
  • Cell boundaries leak. A rider can walk across a hex boundary to a cheaper cell; smoothing across neighboring cells reduces sharp price cliffs at edges.

6. Consistency of ride state

Driver locations can be stale and approximate — that's fine. Ride state cannot. A trip moving through requested → accepted → arrived → in_progress → completed → paid is a state machine where a double-booked driver or a double-charged rider is a real-money, real-trust bug. This is the part of Uber that needs strong consistency, in contrast to the eventually-consistent location layer.

SubsystemConsistency needStorage choice
Driver locationEventual, bounded staleness OKIn-memory HT + stream
Nearby-driver queryBest-effort, approximateGeo-index (H3/QuadTree)
Ride / trip stateStrong; single-writer per tripDurable store, linearizable per-trip
Payments / billingStrong + idempotent + auditableTransactional, see 59 Payment System
  • One trip = one authoritative writer. The accept-race ("first of 3 drivers wins") must be serialized: a single coordinator (or a lock keyed by tripId/driverId) accepts exactly one driver and rejects the rest. This is leader/locking territory — see 62 Distributed Locking and 36 Leader Election.
  • A driver can be offered two rides at once. Without a guard, two riders' "top 3" lists can both include the same driver and both get accepted. Conditional/compare-and-set assignment (assign only if driver still available) prevents the double-book.
  • Disconnect during a trip (Advanced Issue #2). Billing must survive a dropped client. The driver app buffers the trip path locally and reconciles on reconnect; the server treats trip events as idempotent (event with a stable ID applied once) so replays don't double-charge. This is a saga / eventual-settlement problem — see 32 Distributed Transactions.
Mental modelUber is two systems glued together: a soft real-time, eventually-consistent location/matching plane (lose a ping, who cares) and a hard, strongly-consistent transactional plane for trips and money (lose a state transition, someone gets charged wrong). Most design mistakes come from applying one plane's guarantees to the other.

7. Scaling location updates to millions every few seconds

  • Shard by geography, not just by driver ID. The lesson shards DriverLocationHT by DriverID for even distribution — good for writes. But spatial queries ("who's near here?") want co-located data. Real systems shard the geo-index by cell/region so a nearby-query hits one shard, and accept that a busy downtown cell becomes a hot shard needing finer splitting. The two sharding schemes coexist: driver-keyed for the location HT, cell-keyed for the index.
  • Stateful routing to the owning node. Uber's Ringpop used consistent hashing + a gossip membership ring so any node could route a request to the node currently owning that driver/region, with rebalancing as nodes join/leave. See 12 Consistent Hashing and 34 Gossip.
  • Persistent connections beat polling at this scale. The lesson debates push vs. 5-second pull. Pull is simpler but at millions of clients the constant reconnect/query overhead dominates; long-lived WebSocket/SSE channels with server push (and a fallback to polling on bad networks) is what's actually used. See 13 Long-Poll / WS / SSE.
  • Bound the fan-out. The "5 subscribers per driver" estimate is optimistic; in a dense area one driver's movement may matter to far more viewers.
