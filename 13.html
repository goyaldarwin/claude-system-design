Long-Polling vs WebSockets vs SSE

HTTP is request-response: the client asks, the server answers. But sometimes the server needs to push — a new message, a live score, an LLM token. These four techniques are the ways to fake or enable that, each trading latency, direction, and proxy-friendliness differently. Choosing is a how-to-pick problem driven by a few dimensions.

The trade-off Server-push freshness in exchange for connection state, proxy fragility, and scaling cost. Real-time delivery means holding connections open or polling repeatedly — each costs memory, pins clients to a backend, and (for WebSocket) fights corporate proxies. You pick the lightest technique that meets the freshness and direction your feature actually needs.

The problem: HTTP can't push

Plain HTTP is pull: the client requests, the server responds, the connection closes. That's perfect for loading a page, useless for "tell me the instant a new message arrives." The four techniques here are the ways to get server→client freshness over (or alongside) HTTP — from faking it with repeated requests to upgrading to a true bidirectional socket.

The one-line frameThere's no single "real-time" answer — there's a ladder. Pick the lightest technique that satisfies your direction, frequency, proxy, and scale needs. Reaching for WebSocket when SSE would do buys you proxy fragility and connection state you didn't need.

Three scenarios — decide before you read

Scenario 1 — the LLM token stream You're streaming an AI assistant's response token-by-token to the browser as it's generated. The flow is purely server→client. WebSocket or something simpler — and why is the "obvious" bidirectional choice the wrong one here?
Scenario 2 — chat behind a corporate firewall You need push notifications to users on locked-down corporate networks where proxies strip the WebSocket Upgrade header. What's the universal fallback that works anywhere HTTP works?
Scenario 3 — one million idle connections Your chat server holds 1,000,000 mostly-idle WebSocket connections. CPU is near zero, but the box is in trouble. What resource is actually the bottleneck, and what breaks when you deploy a new version?

Scenario 1 is SSE (one-way, HTTP-native). Scenario 2 is long-polling (the fallback). Scenario 3 is the connection-state scaling failure.

The four dimensions that decide

Run any feature through these four and the right transport falls out.

Direction — one-way or two-way?

If only the server pushes (tickers, notifications, log tails, LLM tokens), you want SSE — simpler than WebSocket and HTTP-native. If the client also sends frequently (chat, games, collaborative editing), you need WebSocket's full duplex.

Frequency — rare events or a firehose?

Rare events tolerate long-polling or even polling; a high-frequency, low-latency stream needs a held connection (SSE/WebSocket) so you're not paying HTTP headers per message.

Proxy-friendliness — must it cross hostile middleboxes?

Polling, long-polling, and SSE are plain HTTP and traverse almost anything. WebSocket is the weakest — it needs the Upgrade/Connection handshake to survive every hop, and corporate proxies and old L7 LBs often strip it (WSS/TLS tunnels through far better).

Scale — how many concurrent connections?

Every held connection costs memory and a slot regardless of activity — the binding constraint for any push technique (see the failure below).

Famous implementations — the four techniques

Same facets each — what it is, when it wins, the catch.

Polling (Ajax)

  • What. Client re-requests on a timer; every request is a fresh HTTP call.
  • Wins when. Staleness > ~10s is fine and you want zero server connection state (a CDN can even cache the response) — e.g. an unread-count badge.
  • Catch. Wasted requests and up-to-one-interval staleness; tune the interval to the freshness SLA.

Long-polling

  • What. Client sends a request the server holds open until an event fires (or it times out), then the client immediately re-issues.
  • Wins when. You need push but must traverse hostile proxies — the universal fallback, works anywhere HTTP works, no upgrade.
  • Catch. A held request pins a server worker unless you use async I/O; near-push latency but more overhead than a true stream.

SSE (Server-Sent Events)

  • What. One long-lived HTTP stream (text/event-stream), server→client only, via the browser's EventSource.
  • Wins when. One-directional server push: live scores, tickers, log tailing, progress bars, LLM token streaming. Built-in auto-reconnect + Last-Event-ID replay; rides HTTP/2 multiplexing. Criminally underused.
  • Catch. Text only (base64 for binary); some proxies buffer the stream (disable with X-Accel-Buffering: no).

WebSocket

  • What. A TCP connection upgraded from HTTP (101 Switching Protocols); full-duplex, tiny per-frame overhead, native binary.
  • Wins when. Both sides talk often and latency matters: chat, multiplayer games, collaborative editing, trading dashboards.
  • Catch. Stateful connections, manual reconnect/resync, and the worst proxy-friendliness.

The failure: scaling connection state

The hard part of every push technique isn't CPU — it's that an idle connection still costs memory and a slot.

Why a million idle sockets is hard — the C10k → C10M problem This is Scenario 3. The classic C10k problem was solved by event loops (epoll/kqueue/IOCP — nginx, Node, Netty, Go). The frontier is C10M (10M/box), bounded by memory: at ~10–50 KB of buffers per socket, 1M connections is 10–50 GB of RAM before any app state. Connection state is the bottleneck, not CPU. Two more consequences:
  • LB stickiness — long-lived connections pin to one backend; you can't rebalance like stateless HTTP. A deploy that rolls a backend drops every socket on it, and clients reconnect at once (a thundering herd) — mitigate with jittered backoff and connection draining.
  • Fan-out backplane — a publisher and a subscriber may be on different boxes, so you need a Pub/Sub layer (Redis, Kafka, NATS) to route a message to whichever node holds the target connection. This is the connection-registry + delivery split you design in Messenger.
Large systems terminate millions of sockets on a thin gateway tier (Go/Rust/Erlang — WhatsApp famously ran ~2M connections per Erlang box) and keep business logic on stateless services behind it.

The decision table — which transport, by need

Always design the degradation ladderLibraries like Socket.IO/SignalR negotiate the best available transport and silently fall back: WebSocket → SSE/long-poll → polling. Design the ladder, not a single transport — corporate proxies and ancient middleboxes still break WebSocket in the wild.

Self-check — can you reconstruct the three scenarios?

  • Scenario 1 (LLM stream): Why is SSE the right call for one-directional token streaming, and what does it give you for free that raw WebSocket doesn't?
  • Scenario 2 (corporate firewall): Why is long-polling the universal fallback, and what does WebSocket need that proxies strip?
  • Scenario 3 (1M sockets): Why is memory (not CPU) the bottleneck, what does a deploy do to held connections, and what's a backplane for?
  • Name the four deciding dimensions and which transport each points to.
  • Why design a fallback ladder rather than commit to one transport?

The shape to remember: pick by 4 dimensions — direction, frequency, proxy-friendliness, scale — among 4 transports, and the universal failure is connection-state scaling (C10k→C10M).

Build it yourself

Design fan-out across a fleet

User A (connected to box 1) sends a message to user B (connected to box 3). Neither box knows about the other's connections. How does B get the message?

Reveal the approach

You need a connection registry (which user is on which box) and a backplane. Box 1 receives A's message, looks up that B is on box 3 (via the registry, e.g. Redis), and publishes the message to a Pub/Sub channel for box 3 (or for B's user ID). Box 3, subscribed, receives it and pushes it down B's socket. At larger scale you skip the per-box routing and just publish to a topic keyed by recipient; every gateway subscribed to its users' topics delivers locally. This registry + delivery split is exactly the Messenger design.

Design-it-yourself

  • Graceful deploys for sockets. Roll a new version without a reconnect storm. → connection draining + jittered backoff.
  • Pick transport per feature. For a product, choose a transport for: typing indicators, an unread badge, a live dashboard, an LLM chat. Defend each by the four dimensions.
