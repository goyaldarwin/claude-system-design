Conflict-free Replicated Data Types (CRDTs)

Data structures that converge to the same state under concurrent, conflict-free, multi-master writes — no consensus, no coordination, no last-writer wipes your edit.

Why CRDTs exist

Strongly consistent systems force every write through a single ordering point — a leader, a quorum, a consensus log. That gives you linearizability but costs you availability and latency: during a partition the minority side must reject writes (see Consensus and Raft and the CP corner of CAP Theorem). For a huge class of workloads — shopping carts, presence, counters, collaborative documents, multi-region caches, offline-first mobile apps — that is the wrong trade. You want every replica to accept writes locally, immediately, even while partitioned, and you want them to automatically reconcile when they reconnect.

The naive AP approach — last-writer-wins on a whole object using wall-clock timestamps — silently discards concurrent updates and is at the mercy of clock skew (see Time, Clocks and Ordering). Dynamo-style systems exposed sibling versions and pushed conflict resolution onto the application, which is error-prone. CRDTs (Shapiro, Preguiça, Baquero, Zawirski, 2011) are the principled answer: data types whose merge function is mathematically guaranteed to converge regardless of message order, duplication, or delay — so reconciliation is built into the type, not bolted on by the app.

Key insightA CRDT trades away the ability to enforce global invariants (e.g. "balance never goes negative", "username is unique") in exchange for coordination-free availability. It guarantees Strong Eventual Consistency (SEC): any two replicas that have received the same set of updates — in any order — are in the same state. No rollbacks, no conflicts surfaced to the user.

The math: join-semilattices and monotonic merge

The convergence guarantee is not a heuristic; it falls out of order theory. A join-semilattice is a set S with a partial order ≤ and a binary join (least upper bound) operation ⊔ that is:

  • Commutative: a ⊔ b = b ⊔ a — order of merges does not matter.
  • Associative: (a ⊔ b) ⊔ c = a ⊔ (b ⊔ c) — grouping does not matter.
  • Idempotent: a ⊔ a = a — receiving the same state twice changes nothing.

Those three properties (often abbreviated ACI) are exactly the algebra of an unreliable network: messages get reordered (commutativity handles it), batched (associativity), and duplicated/redelivered (idempotence). If every local update only moves a replica's state upward in the lattice (a monotonic inflation) and merges compute the join, then all replicas climb toward the same least upper bound. Convergence is a theorem, not a hope.

// The whole contract for a state-based CRDT
state   : element of a join-semilattice S
update  : S -> S          // must be an inflation: s ≤ update(s)
merge   : S x S -> S      // merge(a,b) = a ⊔ b  (LUB; commutative, assoc, idempotent)
query   : S -> V          // pure read, no side effects

Example: a single integer under max. max is commutative, associative, idempotent; updates only increase the value. Merge two replicas by taking the max and they converge. Most CRDTs are clever compositions of these primitive lattices (max, union, products, maps) so the join distributes elementwise.

Two flavors: state-based (CvRDT) vs operation-based (CmRDT)

There are two dual ways to build a CRDT, and a principal engineer should be able to argue the trade-off between them on demand.

State-based — CvRDT (convergent)

Replicas apply updates locally, then periodically ship their entire state to peers, who merge it via the join. Correctness only requires the join to be ACI; the transport may lose, reorder, or duplicate messages freely. This pairs naturally with the gossip / anti-entropy machinery in Anti-Entropy and Gossip — you just exchange states on a gossip round and merge.

  • + robust tolerates any message loss/dup/reorder; transport need not be reliable.
  • + simple the only proof obligation is "merge is a join".
  • − heavy shipping full state is expensive as objects grow. Delta-state CRDTs (δ-CRDTs, Almeida et al.) fix this by shipping only the join-irreducible deltas instead of the whole state.

Operation-based — CmRDT (commutative)

Replicas broadcast each operation (not state). Each replica applies operations locally and to inbound ops. Correctness requires concurrent operations to commute, and the delivery layer to provide reliable causal broadcast with exactly-once delivery (causal so that, e.g., a remove sees the matching add).

  • + compact only the small op travels, not the whole object.
  • − demanding needs a reliable, exactly-once, causally-ordered delivery layer — the network burden moves up the stack.
Trade-offState-based pushes complexity into the data type (idempotent merges) and asks little of the network. Op-based pushes complexity into the network (causal exactly-once delivery) and keeps messages tiny. The two are theoretically equivalent in expressive power — you can emulate either with the other — but they fail differently in production. Pure op-based breaks badly if a message is duplicated; that is why most real engines (Automerge, Yjs, Riak) use delta-state or op-based-with-dedup hybrids.

A catalog of concrete CRDTs

Counters

G-Counter (grow-only): a vector of per-replica counts, one slot per replica id. Increment bumps only your own slot. Merge is elementwise max; value is the sum of all slots. Because each replica owns its slot, two concurrent increments never collide.

// G-Counter at replica i, with N replicas
P : map<replicaId, int>        // all zero initially
increment():        P[i] += 1
value():            return sum(P.values())
merge(other):       for r in replicas: P[r] = max(P[r], other.P[r])

PN-Counter: two G-Counters, P for increments and N for decrements; value is sum(P) - sum(N). This is how you build a decrementable counter (a like/unlike count, inventory tally) that still converges — you cannot subtract directly because decrement is not an inflation, so you track removals in a separate grow-only structure.

Key insightThe OR-Set is the workhorse and the one to reach for in practice. It resolves the classic add-wins semantics for concurrent add/remove: if one replica adds x and another concurrently removes x, the add wins, because the remove only tombstones the tags it has actually seen — it cannot delete a tag created concurrently. This matches user intuition far better than 2P-Set's "removal is forever".

Registers

LWW-Register: holds a single value plus a timestamp; merge keeps the value with the higher timestamp (ties broken deterministically by replica id). Simple, but it silently discards concurrent writes — the loser is gone. Use a logical clock or hybrid logical clock, never raw wall-clock, to bound the damage from skew. MV-Register (multi-value) instead keeps all concurrently-written values (using a version vector to detect concurrency) and surfaces the conflict set to the application — this is the Dynamo "siblings" model expressed as a CRDT.

Sequence CRDTs for collaborative text

Ordered lists (the text of a shared document) are the hard case: concurrent inserts at "the same position" must end up in a stable, agreed order on every replica without a central server. Naive index-based positions break instantly — remote ops shift the indices out from under you.

The trick is to give every character/element a stable, globally-unique, densely-orderable identifier so order is intrinsic to the element, not a mutable index:

  • RGA (Replicated Growable Array): each element references the id of the element it was inserted after; concurrent inserts at the same anchor are ordered by a per-element timestamp/id. Deletes leave tombstones.
  • LSEQ / Logoot / Treedoc: assign each position a fractional / variable-length path identifier between its neighbours, so you can always mint a new id strictly between any two existing ones (dense order). LSEQ adapts the allocation strategy to keep identifiers from growing too long under interleaved editing.
  • YATA (used by Yjs) and Automerge's sequence type are the modern, production-tuned descendants.
PitfallTwo well-known costs of sequence CRDTs: (1) tombstone accumulation — deleted characters linger as metadata; long-lived documents need garbage collection that itself requires knowing all replicas have seen the delete. (2) interleaving anomalies — some early algorithms (and a famous bug class in Logoot) can interleave two users' concurrently-typed words into garbage like "HWeolrllod". RGA-family algorithms with proper origin tracking avoid this; verify any sequence CRDT against the interleaving test.

CRDTs vs Operational Transformation

Operational Transformation (OT) — the algorithm behind Google Docs — solves the same collaborative-editing problem differently. OT transforms each incoming operation against concurrent ones (an index 5 insert becomes index 6 after a remote insert at 3) so they apply consistently. CRDTs instead make operations commute by construction via unique ids, so no transformation is needed.

Trade-offThe industry has been migrating toward CRDTs for new collaborative and local-first products (Figma, Linear, Apple Notes, many Y.js apps) precisely because they drop the central-server requirement and support offline editing cleanly. OT still wins on memory footprint and is entrenched where a server already mediates every edit (Google Docs). See Designing Collaborative Editing for the full system design.

Real systems

  • Riak shipped the first mainstream CRDT library ("Riak DT") — counters, sets, maps, registers — built on its Dynamo-style vector-clock foundation, letting the database auto-resolve siblings instead of returning conflicts to the app.
  • Redis (Active-Active / CRDB) in Redis Enterprise implements geo-distributed multi-master replication where Redis types (strings as LWW/MV registers, counters as PN-counters, sets as OR-Sets, sorted sets) are backed by CRDTs so each region accepts local writes and converges. Plain open-source Redis is single-master and is not this.
  • Automerge and Yjs are the dominant JSON/document CRDT libraries powering local-first and collaborative apps; they use op-based/delta encodings with binary compaction and run in the browser and on mobile.
  • Azure Cosmos DB multi-region writes and various caches expose CRDT-style merge policies; CDNs and presence/online-status systems lean on grow-only sets and counters.
  • Amazon's original shopping-cart conflict resolution is the canonical motivating example: merging carts as an OR-Set-like union means a deleted-then-re-added item resurfaces — an availability-favoring choice over losing a customer's add.
PitfallCRDTs are not a universal substitute for consensus. They cannot enforce cross-key invariants: "total inventory ≥ 0", "this seat is booked by exactly one person", "balance never overdraws". Those require coordination (locks, consensus, or compensating reservation patterns). A common architecture splits the system — CRDTs for the high-volume, conflict-tolerant 95% (carts, presence, edits) and a consensus/transactional core for the invariant-critical 5% (payment capture, seat allocation). See Distributed Transactions and Consistency Models & PACELC.

Principal-level mastery check

What strong engineers doThey lead with the algebra — "I need a merge that's commutative, associative, idempotent so it survives reorder/dup/redelivery" — rather than memorized type names. They pick the right type for the semantics (OR-Set for add-wins, PN-Counter for likes, MV-Register when conflicts must be surfaced) and they name the costs out loud: tombstone GC, metadata growth, and the inability to enforce global invariants. They explicitly draw the boundary between the CRDT-able part of the system and the part that still needs consensus, and they choose state-based vs op-based based on what the network layer can guarantee.
  • Walk me through why a G-Counter cannot just be a single integer with += and merge-by-max. Why does each replica need its own slot? What breaks otherwise?
  • A user adds an item to a shared list on their phone (offline) while another device removes the same item. With a 2P-Set, an LWW-Set, and an OR-Set, what does each converge to — and which matches user expectation?
  • You have a PN-Counter for available inventory across 5 regions. Each region decrements on a sale. Explain precisely why this can oversell, and what minimal coordination you would add to prevent going negative without globally serializing all sales.
  • State-based vs op-based: your replication runs over an at-least-once message bus with no ordering guarantees. Which flavor do you choose and why? What changes if you instead have a reliable causal-broadcast layer?
  • Tombstones accumulate forever in a long-lived collaborative document. Design a safe garbage-collection scheme. What do you need to know about every replica before you can physically delete a tombstone, and how does that reintroduce a coordination problem?
  • Compare CRDT-based and OT-based collaborative text editing for a product that must work fully offline for days. Which do you pick, what's your metadata budget, and how do you test for interleaving anomalies?
