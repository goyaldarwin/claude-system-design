Observability

Monitoring tells you that something is wrong; observability lets you ask why — including questions you never thought to pre-define. In a distributed system where one user request fans out across dozens of services, you can't debug by SSH-ing into a box. The three pillars (metrics, logs, traces) plus correlation are how you reconstruct what happened — and the hidden tax that breaks observability budgets is cardinality.

The trade-off The ability to ask arbitrary questions about your running system — in exchange for the storage, ingestion, and cardinality cost of the telemetry you emit. Rich telemetry lets you diagnose novel failures without redeploying. You pay in data volume: metrics, logs, and traces cost money to store and query, and high-cardinality labels can explode that cost exponentially — so observability is a deliberate sampling and dimensionality budget, not "log everything."

What observability is — vs monitoring

Monitoring watches a fixed set of pre-defined signals and tells you when one crosses a threshold — it answers questions you knew to ask in advance ("is CPU > 80%?"). Observability is the property of being able to ask arbitrary, new questions about your system's internal state from its outputs — to debug a failure you've never seen before, without shipping new code to investigate. Monitoring is a subset; observability is the broader capability you need when systems are too complex to predict every failure mode.

It rests on three pillars, each answering a different question:

  • Metrics — cheap, aggregated numbers over time (request rate, error rate, latency percentiles). Great for "is something wrong, and roughly where?" Low storage cost, but limited detail.
  • Logs — discrete, timestamped event records (ideally structured/JSON). Great for "what exactly happened in this event?" High detail, higher cost.
  • Traces — the path of a single request across all the services it touched (a trace), with timing per hop. Great for "where did this request spend its time / where did it fail?" in a distributed system.
The one-line frameObservability trades telemetry cost (storage, ingest, cardinality) for the ability to diagnose the unknown. The skill isn't emitting everything — it's choosing metrics with bounded cardinality, sampling traces/logs intelligently, and correlating the three pillars so that when something breaks you can go from "users are erroring" to "this line in this service" in minutes, not hours.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the pillars compared.

Scenario 1 — the metrics bill that 100×'d overnight A developer adds a label to a metric — user_id — to "make debugging easier." The next day the observability bill explodes and queries crawl. The metric that had a few hundred time-series now has millions. What is the property that exploded, why does adding that one label multiply cost, and what should carry per-user detail instead?
Scenario 2 — the request that vanished across services A user reports a checkout that failed. You have logs in all 12 services the request touched, but no way to find which log lines belonged to that user's request — they're interleaved with millions of others, and the failure happened three services deep. What single mechanism would let you follow one request's entire journey across every service, and what must be passed for it to work?
Scenario 3 — the error that got sampled away To control cost, you sample 1% of traces at the point each request enters the system. A rare, important error happens in 0.1% of requests — and almost none of those error traces survive sampling, so you can never find an example to debug. Why did head-based sampling lose exactly the traces you needed, and what sampling strategy keeps the interesting ones?

Scenario 1 is metrics & the cardinality budget. Scenario 2 is traces & context propagation. Scenario 3 is sampling (head vs tail). Each maps to a decision below.

The four decisions that define an observability design

Designing observability is these four choices. Each has a named failure nested under it.

Decision 1 — Metrics: what to count, and the cardinality budget

Metrics are stored as time-series: a unique series exists for every combination of label values. Use proven frameworks to pick what to count: RED (Rate, Errors, Duration — for request-driven services) and USE (Utilization, Saturation, Errors — for resources like CPU/disk). The critical constraint is cardinality: each label multiplies the series count, so labels must be low-cardinality (status code, endpoint, region — tens of values), never unbounded (user IDs, request IDs — millions).

The failure that happens when you get this wrong — the cardinality explosion This is Scenario 1. Time-series count = the product of every label's distinct values. Add a high-cardinality label like user_id (millions of values) and you multiply your series count by millions — storage and ingestion cost explode, and queries slow to a crawl. This "label explosion" is the single biggest cost driver in metrics-based observability. The rule: metrics labels must be bounded (status, route, instance — the small set you'd group-by). Per-request or per-user detail belongs in logs or traces (which are indexed differently), not in metric labels. Watch out: even an endpoint label can explode if your routes contain IDs (/users/12345) — normalize to /users/:id.

Decision 2 — Logs & traces: connecting the dots

In a distributed system, the killer capability is following one request across every service. That requires distributed tracing: assign each incoming request a trace ID, and propagate it (plus a per-hop span ID) through every downstream call — in HTTP headers, message metadata, etc. Each service emits spans (timed operations) tagged with the trace ID; a tracing backend stitches them into one waterfall. Tag your structured logs with the same trace ID so logs, traces, and metrics correlate.

The failure that happens when you get this wrong — the un-correlatable request This is Scenario 2. Without a propagated trace ID, your logs and metrics are per-service islands — you can see that some requests failed in service X, but not which user-request, nor its path before/after X. Debugging becomes archaeology across 12 log streams. The fix is context propagation: generate a trace ID at the edge and pass it through every call (and into async messages), so one ID ties together the whole journey across services and pillars. The hard part is making propagation universal — one service that drops the header breaks the trace — which is why standards like OpenTelemetry exist to do it consistently.

Decision 3 — How much do you sample?

Tracing every request at full detail is expensive at scale, so you sample. Head-based sampling decides at the start of a request (e.g. keep 1%) — cheap and simple, but blind to what happens later. Tail-based sampling decides at the end, after seeing the whole trace — so you can keep all errors and slow requests and sample the boring fast-success ones — more useful but needs buffering and more infrastructure.

The failure that happens when you get this wrong — sampling away the evidence This is Scenario 3. Head-based sampling at a flat rate discards traces before it knows they're interesting — so a rare error (0.1% of requests) sampled at 1% leaves you almost no error traces to debug, exactly when you need them most. Tail-based sampling fixes this: buffer the full trace, then keep it if it errored or was slow, and sample down only the uninteresting successes. You retain the signal (errors, tail latency) while still cutting volume on the noise. The cost is the buffering infrastructure; the payoff is that your sampled data actually contains the failures you'll be asked to explain.

Decision 4 — What pages a human?

Alerting policy determines whether on-call is effective or burned out. Two rules: alert on symptoms, not causes (page when users are affected — high error rate, SLO burn — not on every internal cause like "CPU high," which may not affect anyone), and make every page actionable and documented (a runbook link; if there's nothing a human can do, it shouldn't page). Tie alerts to your SLO burn rate so pages mean "users are being hurt fast."

The failure that happens when you get this wrong — cause-based alert noise Cause-based alerting — paging on every internal condition (CPU spike, a single retry, disk 70% full) — floods on-call with pages that often correspond to no user impact, because the system absorbed the cause (autoscaling kicked in, a replica took over). The result is alert fatigue: real pages get lost in the noise and ignored, and the genuinely-bad incident is missed. Alert on symptoms (the user-facing SLI is breaching / burning budget fast) so a page always means real harm; route cause signals to dashboards/tickets for investigation, not to a pager. Every page must be urgent, real, and have a documented action — otherwise delete it.

The takeawayOpenTelemetry (OTel) is the emerging standard for emitting metrics/logs/traces in a vendor-neutral way — instrument once, ship anywhere — and it handles context propagation consistently across languages. Managed backends (Datadog, Honeycomb, Cloud Trace) bill heavily on cardinality and volume, so the cardinality budget (Decision 1) and sampling (Decision 3) are directly cost-control decisions, not just engineering ones.

Famous implementations

The tools made concrete. Same facets — what it is, the model, and the failure to watch.

Metrics — Prometheus

  • What it is. The de-facto open-source metrics system; pull-based scraping of time-series with a label model and PromQL.
  • Model. Each label combination is a series; histograms enable percentile queries; alertmanager + recording rules drive alerts.
  • The failure to watch. Cardinality explosion (Scenario 1) — a high-cardinality label can OOM the server; long-term storage needs Thanos/Cortex/Mimir.

Traces — Jaeger / OpenTelemetry

  • What it is. Distributed tracing — Jaeger/Tempo as backends, OpenTelemetry as the instrumentation + propagation standard.
  • Model. Trace ID + spans propagated via standard headers (W3C Trace Context); head- or tail-based sampling.
  • The failure to watch. One un-instrumented hop breaks the trace; head sampling drops rare errors (Scenario 3) — prefer tail-based for error retention.

Stacks — ELK / Datadog / Grafana

  • What it is. Unified observability: ELK/Loki for logs, Grafana for dashboards, Datadog/Honeycomb/New Relic as all-in-one SaaS.
  • Model. Correlate the three pillars by trace ID; high-cardinality querying (Honeycomb's strength) for exploratory debugging.
  • The failure to watch. Volume/cardinality-based billing makes "log everything" financially ruinous; sample and structure deliberately.

Build it yourself

These click when you instrument something. Work each before opening the approach.

Instrument a service with RED, with bounded cardinality

Add metrics to an HTTP service so you can see its health and debug issues, without blowing up cardinality. What do you emit, with what labels?

Reveal the approach

Emit RED: Rate (requests/sec counter), Errors (error count / error ratio), Duration (a latency histogram, so you can query p50/p99). Label with low-cardinality dimensions only: route (normalized — /users/:id, never the raw ID), method, status_class (2xx/4xx/5xx), instance. That's maybe a few hundred series — cheap and groupable. Do not add user_id, request_id, or full URLs as labels (Scenario 1 — cardinality explosion). When you need per-request detail, attach it to the trace/structured log keyed by trace ID, not to a metric. From these RED metrics you can detect and roughly localize any problem; traces/logs take it the rest of the way.

Propagate a trace across three services

A request hits service A, which calls B, which calls C (and also enqueues a message processed later by D). Design how you follow the whole thing as one trace.

Reveal the approach

At the edge (A), generate a trace ID (or adopt an incoming one). For each outbound call, A creates a span and injects the trace context (trace ID + current span ID) into the request — HTTP headers (W3C traceparent) for A→B→C, and into the message metadata for the async A→D hop. Each service starts a child span with the received parent, does its work, and reports its span (with timing, status) to the tracing backend, which stitches them into one waterfall by trace ID. Tag every structured log line with the trace ID too, so logs join the trace. Now one user request — sync and async parts — is a single navigable timeline (Scenario 2's fix). Use OpenTelemetry so propagation is automatic and consistent; the one rule is that every hop must forward the context or the trace breaks there.

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Cut an observability bill in half. Your telemetry cost doubled. Find the cardinality and volume offenders and redesign labels + sampling without losing debuggability. → Cost-Aware System Design.
  • Design tail-based sampling. You must keep all error/slow traces but sample successes to 1%. Design the buffering and decision pipeline. → Stream Processing.
  • Build a symptom-based alert set. Replace a noisy cause-based alert list with a small set tied to SLO burn rate, each with a runbook. → SLI/SLO/SLA.
