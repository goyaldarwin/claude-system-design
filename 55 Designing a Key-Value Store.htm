Designing a Key-Value Store

Build a Dynamo-class distributed key-value store from first principles: consistent hashing, quorum replication, vector clocks, anti-entropy, and an LSM storage engine — the canonical problem that exercises every distributed-systems primitive at once.

The bottleneck Finite capacity & no shared truth. building Dynamo — partitioning + tunable-quorum consistency + conflict resolution.

Why this problem matters

The distributed key-value store is the Rosetta Stone of distributed system design. A single, faithful design forces the engineer to make concrete decisions about partitioning, replication, consistency, conflict resolution, failure detection, and storage — the entire distributed-systems syllabus collapsed into one artifact. Amazon's 2007 Dynamo paper remains the reference architecture; Cassandra, Riak, Voldemort, and (loosely) DynamoDB all descend from it.

What separates a principal-level answer from a mid-level one is not naming consistent hashing — everyone does that. It is reasoning precisely about why Dynamo chose an AP, leaderless, eventually-consistent design over a CP, leader-based one (see Consensus and Raft), and being able to articulate the failure modes each choice introduces: divergent replicas, sibling versions, sloppy quorums that violate R+W>N guarantees during partitions, and the operational cost of anti-entropy.

Key insight Dynamo is "always writeable." It pushes conflict resolution to read time (and often to the application) so that writes never block on coordination. That single decision — availability over consistency under partition (CAP) — cascades into every other component: vector clocks, read-repair, Merkle trees, and gossip all exist to clean up the divergence that "always writeable" creates.

Requirements and scope

Functional

  • get(key) → value (or a set of conflicting values + a context token)
  • put(key, context, value) → store value; context carries causality (vector clock)
  • Values are opaque byte arrays (Dynamo's model). No secondary indexes, no range scans, no multi-key transactions.

Non-functional (the interesting part)

PropertyTargetImplication
Availability"Always writeable" — 99.9%+ on writes even during failuresLeaderless, AP design; sloppy quorums
Latencyp99 < 10–15 ms for get/putSSD-backed LSM, in-region replication, no synchronous cross-region writes
ScaleHorizontal to thousands of nodes, petabytesConsistent hashing, no central coordinator
ConsistencyTunable per-operationQuorum knobs N, R, W exposed to the caller
DurabilityNo data loss on single-node failureN>1 replicas, write-ahead log in storage engine
What strong engineers do They state the SLA in the customer's terms: Dynamo's SLA was framed at the 99.9th percentile, not the mean, because Amazon's service-oriented architecture fans out one request into hundreds of downstream calls — the tail dominates. Pin this number early; it justifies rejecting designs that add a coordination round-trip (e.g., synchronous leader election) on the write path.

Capacity estimation

Suppose 10 KB average value, 1 million writes/sec peak, 5:1 read:write ratio, 1-year retention, N=3.

  • Write throughput: 1M/s × 10 KB × 3 replicas = 30 GB/s of replica writes. Spread over 1000 nodes ≈ 30 MB/s/node — comfortable for NVMe.
  • Storage: 1M/s × 10 KB × 86400 × 365 ≈ 315 PB/yr logical, ×3 = ~945 PB raw. With LSM compaction overhead and bloom-filter/index metadata, budget ~1.2× on top.
  • Read throughput: 5M reads/s. With R=2, that's 10M replica reads/s. The block cache and bloom filters keep most reads off disk (see Caching).

These numbers justify sharding (data partitioning) as non-negotiable and rule out anything that requires a global view of the keyspace on the hot path.

The API

// get returns potentially-conflicting versions plus an opaque context
GetResult get(Key key, ConsistencyLevel level)
  -> { values: [Value], context: Context }   // context = serialized vector clock(s)

// put MUST echo back the context from a prior get to express causality
void put(Key key, Context context, Value value, ConsistencyLevel level)

// ConsistencyLevel maps to R/W: ONE, QUORUM, ALL  (Cassandra-style)

The context is the crux of the API contract. A blind put without a context creates a new, causally-unrelated version (a sibling). A read-modify-write loop must pass the context it received so the system can supersede the old version rather than fork it.

Partitioning: consistent hashing with virtual nodes

Keys are mapped onto a fixed ring (e.g., MD5 of the key → 128-bit ring position). A node owns the arc of the ring ending at its position. This is consistent hashing: adding or removing a node only reshuffles keys between it and its ring neighbors, not the whole keyspace.

Why naive consistent hashing isn't enough

  • Non-uniform load. Random node positions create uneven arc sizes; some nodes own 3× the data of others.
  • Heterogeneous hardware. A beefier node should own more of the ring.
  • Slow rebalancing. When a node joins, only one neighbor donates data — a single-source bottleneck.
Key insight Virtual nodes (vnodes) fix all three. Each physical node is assigned T positions on the ring (Dynamo: T≈100–256). Load is now the average of T arcs (variance ↓ by √T). A bigger machine gets more tokens. When a node joins, it takes T small slices from many donors in parallel, so rebalancing is fast and spread out. Cassandra defaults to 256 vnodes per node for exactly this reason.
ring: sorted list of (token, vnode) where token = hash(node_id + ":" + i), i in [0,T)
owner(key):
    pos = hash(key)
    return first vnode clockwise from pos on the ring

Replication and the preference list

Each key is replicated to N nodes. Starting at the key's owning vnode, walk clockwise and collect the next N distinct physical nodes (skipping vnodes that map to a node already in the list). This ordered list is the preference list for the key.

Pitfall If you naively take the next N vnodes, you can land multiple replicas on the same physical node — defeating the purpose of replication. You must skip duplicates, and ideally enforce that replicas span multiple racks / availability zones so a single fault domain failure doesn't take out a quorum. Cassandra encodes this in a NetworkTopologyStrategy; Dynamo built rack-awareness into the preference list directly.

The first N nodes in the preference list are the natural replicas. Because of failures, the list is longer than N — extra nodes downstream serve as fallback targets for hinted handoff. See also Redundancy and Replication and the deeper Replication Deep Dive.

Tunable consistency: N, R, W quorums

Dynamo exposes three knobs per operation:

  • N — replication factor (how many nodes hold each key).
  • W — write quorum: a put succeeds once W replicas acknowledge.
  • R — read quorum: a get waits for R replicas to respond, then reconciles.
Key insight R + W > N guarantees the read and write quorums overlap on at least one node, so a read is guaranteed to see the latest acknowledged write — strong-ish consistency without a leader. W > N/2 additionally prevents two concurrent writes from both succeeding on disjoint sets. Common configs: N=3,W=2,R=2 (balanced); N=3,W=3,R=1 (read-optimized, fragile to writes); N=3,W=1,R=1 (max availability, eventual only).
ConfigReadsWritesUse case
W=1, R=1Fast, may be staleAlways writeableShopping cart, session store
W=2, R=2 (N=3)Overlapping quorumSurvives 1 node downGeneral purpose
W=N, R=1Fast readsSlow, blocks if any replica downRead-heavy, rarely written config
Pitfall R+W>N does not make Dynamo linearizable. With a sloppy quorum (writes accepted by the first N healthy nodes, not the natural N, to stay available during partitions), the overlap guarantee can be violated: a write satisfied by fallback nodes and a read satisfied by natural nodes may not intersect. This is the price of "always writeable." For true linearizability you need a consensus protocol (Raft) and accept reduced availability — see Consistency Models and PACELC, whose ELC clause says even when there's no partition, you trade latency for consistency.

Versioning and conflict resolution

Because writes can hit different replicas during partitions, two versions of the same key can diverge. Dynamo must (a) detect divergence and (b) resolve it.

Vector clocks for causality

Each version carries a vector clock: a map {node_id: counter}. On a write, the coordinating node increments its own entry. Given two clocks, you can decide their causal relationship (see Time, Clocks and Ordering):

compare(A, B):
    if every A[i] <= B[i] and A != B:  A happened-before B  (B wins, discard A)
    if every B[i] <= A[i] and A != B:  B happened-before A  (A wins)
    else:                              CONCURRENT -> siblings, must reconcile

If one clock dominates the other, the system silently keeps the newer version. If they're concurrent, both are returned to the next reader as siblings.

Pitfall Vector clocks grow unbounded if many nodes coordinate writes to the same key. Dynamo truncates the oldest entries past a threshold (e.g., 10), which can produce false-concurrency and unnecessary siblings. In practice this is why Riak and Cassandra steered users toward last-write-wins or CRDTs instead of raw vector clocks for most workloads.

Resolution strategies

  • Semantic reconciliation (application-level). Dynamo's shopping cart merges sibling carts by union — the famous "resurrected deleted item" anomaly is the trade-off. The store hands siblings to the app; the app decides.
  • Last-write-wins (LWW). Attach a wall-clock timestamp; highest timestamp wins. Simple, but silently drops concurrent writes and is vulnerable to clock skew. Cassandra defaults to LWW per-cell.
  • CRDTs. Choose data types whose merge is commutative/associative/idempotent so siblings always converge automatically — no application logic, no lost updates. See CRDTs.

Handling failures: hinted handoff, read-repair, anti-entropy

Three layered mechanisms keep replicas converging, operating at increasing timescales.

Hinted handoff (seconds)

If a natural replica is down at write time, the coordinator writes to the next healthy node in the preference list with a hint (the intended recipient). When the original node recovers, the holder replays the hinted writes to it, then deletes the hint. This is what makes Dynamo "always writeable" during transient failures — the write isn't rejected, just temporarily re-homed.

Read-repair (on every read)

When a coordinator gathers R responses and detects that some replicas have a stale version, it asynchronously pushes the reconciled value back to the laggards. Reads thus opportunistically heal the data they touch — cheap for hot keys, useless for cold ones.

Anti-entropy via Merkle trees (background)

Cold keys never get read-repaired, and hinted handoff can be lost if the holder also dies. The backstop is a periodic full reconciliation between replicas using Merkle trees: each node builds a hash tree over its key ranges. Two replicas exchange root hashes; if equal, they're in sync and stop — O(1). If not, they descend only the differing subtrees, transferring just the divergent keys. This makes the comparison cost proportional to the difference, not the dataset size. Covered in depth in Anti-Entropy and Gossip.

Trade-off The three repair mechanisms form a cost/coverage ladder: read-repair is nearly free but only covers read keys; hinted handoff covers transient failures but loses hints on cascading failure; Merkle-tree anti-entropy covers everything but is I/O- and CPU-heavy and must be rate-limited so it doesn't compete with foreground traffic. A mature system runs all three and tunes anti-entropy frequency against the acceptable inconsistency window.

Membership and failure detection: gossip

With thousands of nodes and no central master, every node must learn the ring topology and who's alive. Dynamo uses a gossip protocol: each second, each node picks a random peer and exchanges its view of (membership, token assignments, node liveness). State converges across the cluster in O(log N) rounds.

  • Seed nodes bootstrap membership so the cluster doesn't logically partition into disjoint gossip groups.
  • Failure detection is local and probabilistic: node A marks B as down when B stops responding to A's requests, rather than relying on a global consensus that B is dead. This avoids flapping and false positives from transient network blips — an accrual failure detector (phi-accrual) outputs a suspicion level rather than a boolean.
Key insight Membership in Dynamo is decentralized and gossip-driven precisely to avoid a coordinator that would become a SPOF and a scaling bottleneck. Contrast this with systems that do centralize coordination via a consensus group — see Leader Election and Coordination and a distributed locking service. The deliberate refusal to have a leader is what lets the store survive partitions while remaining writeable.

The storage engine: LSM trees

The per-node storage engine must absorb a write-heavy, high-throughput stream on SSDs. The dominant choice is the Log-Structured Merge tree (LSM), not a B-tree.

Write path

  1. Append to a commit log / WAL on disk (durability).
  2. Insert into an in-memory sorted structure (memtable, e.g., a skip list).
  3. When the memtable fills, flush it immutably to disk as a sorted SSTable.
  4. Background compaction merges SSTables, discarding superseded/tombstoned keys.

Read path

Check the memtable, then SSTables newest-to-oldest. A per-SSTable bloom filter (a probabilistic data structure) cheaply rules out SSTables that can't contain the key, so a read usually touches one file. A sparse index locates the block within it.

Trade-off LSM trades read amplification and space amplification (a key may exist in several SSTables until compaction) for sequential, append-only writes — ideal for SSDs and write-heavy KV workloads. A B-tree gives O(1) read locality but suffers random-write amplification and in-place page updates. The full comparison, including leveled vs. size-tiered compaction and the RUM conjecture, is in Storage Engines: LSM vs B-Tree.
Pitfall Deletes in an LSM are tombstones, not in-place removals. Combined with the leaderless replication model, this creates the classic deleted-data-resurrection bug: if a tombstone is garbage-collected (after gc_grace_seconds) before anti-entropy propagates it to a replica that missed the delete, that replica's stale live value re-replicates the row back. The fix is to ensure repair completes within the grace window — a real operational constraint, not a theoretical one.

Dynamo-style vs. Raft-style: which and why

The single most important framing to internalize is when this architecture is the wrong choice.

DimensionDynamo-style (this design)Raft/Paxos-style (e.g., Spanner, etcd)
TopologyLeaderless, every replica accepts writesPer-shard leader, followers replicate
ConsistencyEventual / tunable, not linearizableLinearizable per key/range
Availability under partitionAP — minority side still writeableCP — minority side rejects writes
Conflict handlingPush to read time (siblings, vector clocks)None — leader serializes, no conflicts
Write latencyNo coordination round-tripOne round-trip to a quorum of followers
Best forHigh-availability, write-heavy, conflict-tolerant dataAnything needing read-your-writes / transactions
What strong engineers do They refuse to treat "eventually consistent" as a free lunch. They name the concrete anomalies the application must tolerate — lost updates, sibling explosion, resurrected deletes, monotonic-read violations — and either justify why the use case tolerates them (carts, session, metrics) or pivot to a Raft-backed design (Distributed Transactions) when it can't. The same primitives reappear when you design a distributed cache; the difference is durability requirements and whether stale reads are acceptable.

Principal-level mastery check

What strong engineers do They drive the design from the availability SLA at p99 and let that justify every leaderless choice, rather than reciting Dynamo features in isolation. They are precise about where consistency guarantees actually hold (and where sloppy quorums break R+W>N), they connect each repair mechanism to the failure timescale it covers, and they proactively surface the application-visible anomalies the design imposes. When pushed, they can redesign the same store as a CP, Raft-backed system and articulate exactly what changes and what it costs in latency and availability.
  • Quorum nuance: "You set N=3, R=2, W=2 and call it strongly consistent. Walk me through a network partition with sloppy quorums where a reader misses a just-acknowledged write despite R+W>N. How would you detect that this happened?"
  • Conflict explosion: "Your vector clocks are growing past the truncation limit on a hot key written by 50 coordinators. What anomalies does truncation introduce, and would you switch to LWW or a CRDT? Defend the data-loss implications of each."
  • Repair economics: "Anti-entropy is consuming 40% of disk I/O and hurting p99. How do you reduce the inconsistency window without that cost — and what's the relationship between gc_grace_seconds, repair cadence, and resurrected deletes?"
  • Hot partitions: "A single key (a celebrity's record) takes 30% of traffic. Virtual nodes don't help — the key still maps to N nodes. What do you actually do?" (Answer should reach for request-coalescing, a read-through cache layer, or splitting the value; honest engineers admit consistent hashing doesn't solve single-key hotspots.)
  • The pivot: "Product now needs read-your-writes and conditional puts (compare-and-set). Can you bolt that onto a leaderless Dynamo, or must you re-leader the shards with Raft? What breaks?"
  • Cross-region: "Extend this to three regions with a 100 ms RTT. Where do you put the quorum boundary, and how does the consistency model degrade?" (Tie to Multi-Region and Disaster Recovery and PACELC's latency clause.)
