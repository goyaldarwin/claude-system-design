Time, Clocks and Ordering

Why "what time is it?" is the wrong question in a distributed system — and how logical clocks, vector clocks, HLC and TrueTime let you order events without trusting wall clocks.

The bottleneck No shared truth. there is no global clock — ordering events costs logical/vector clocks or bounded-uncertainty waits.

In this lesson

Why physical wall-clock time lies

Every server has a quartz oscillator driving its clock. Quartz crystals drift — typically tens to hundreds of parts per million. At even 50 ppm, two machines diverge by ~4.3 seconds per day if left uncorrected. To fight this, machines run NTP (Network Time Protocol), which estimates offset from upstream stratum servers over a jittery network. NTP corrects drift but cannot eliminate uncertainty: typical synchronization is good to a few milliseconds inside a datacenter and tens to hundreds of milliseconds across the internet.

This produces a series of failure modes that bite distributed systems:

  • Clock skew — two nodes report different times for the same instant. Node A's "now" might be 80 ms ahead of Node B's.
  • Backward jumps — when NTP corrects a fast clock, it may step time backward. A timestamp generated after another event can be numerically smaller. Code that assumes monotonic timestamps (e.g. expiry, last-write-wins) breaks.
  • Leap seconds — UTC inserts a leap second to track Earth's rotation. Naive handling caused the 2012 outages at Reddit, LinkedIn, and Mozilla. Google's response was the leap smear: spread the extra second across a 24-hour window so no discontinuity exists.
  • VM pauses and GC — a process can be frozen for hundreds of ms (STW GC) or seconds (VM migration, host swap). When it resumes, its notion of "now" is stale, but it doesn't know that.
Pitfall The classic bug: last-write-wins (LWW) keyed on System.currentTimeMillis(). With skew, the write that physically happened second can carry the smaller timestamp and be silently discarded. Cassandra's LWW conflict resolution has exactly this hazard — a node with a fast clock can make its writes "win" forever, masking later correct writes. This is data loss with no error and no log line.

The core lesson: physical time is an external reference, not an ordering mechanism. You can use it to display a date or measure latency, but you cannot use a bare wall-clock timestamp to decide which of two events happened first across machines. For that you need logical time. This thread runs through Consistency Models and PACELC, Replication Deep Dive, and CRDTs.

The happens-before relation

Leslie Lamport's 1978 paper "Time, Clocks, and the Ordering of Events in a Distributed System" defines a partial order over events called happens-before, written a → b. It is defined by three rules:

  1. If a and b are in the same process and a occurs before b, then a → b.
  2. If a is the sending of a message and b is the receipt of that message, then a → b.
  3. Transitivity: if a → b and b → c, then a → c.

If neither a → b nor b → a, the events are concurrent (written a || b). Concurrency here does not mean "at the same wall-clock instant" — it means causally independent: no chain of process steps or messages connects them. This is the right abstraction because it captures only the orderings that could possibly matter for correctness: causality flows through messages, and nothing else.

Key insight Happens-before is a partial order, not a total one. Two events can be unordered. Any system that needs a single total order — a replicated log, a consensus sequence — must impose one on top of the partial order, breaking ties arbitrarily but consistently. That is precisely what Consensus and Raft does.

Lamport logical clocks

A Lamport clock is a single integer counter L per process, updated by these rules:

// each process maintains an integer L, initially 0

on local event:
    L = L + 1

on send(msg):
    L = L + 1
    msg.timestamp = L

on receive(msg):
    L = max(L, msg.timestamp) + 1

The guarantee is one-directional: if a → b then L(a) < L(b). This is the clock condition. By taking max on receive, causal dependencies always push the receiver's counter past the sender's.

To build a total order (useful for a distributed mutual-exclusion algorithm or a state-machine replication log), break ties on equal L with a fixed rule like process ID: order by (L, processId). This gives a consistent total order across all nodes.

Pitfall The implication does not run backward. L(a) < L(b) does not imply a → b. Two concurrent events can have L=5 and L=7 purely by coincidence of local activity. A Lamport clock therefore cannot detect concurrency — it cannot tell you whether two writes conflicted or one truly preceded the other. For conflict detection you need vector clocks.

So Lamport clocks are great for imposing an order (they're enough for total-order broadcast tie-breaking) but useless for discovering whether a real causal dependency exists. That distinction drives the next step.

Vector clocks and version vectors

A vector clock replaces the single counter with a vector of counters, one entry per process. With N nodes, each event carries an N-element vector V.

// process i holds vector V[0..N-1], all zero initially

on local event:
    V[i] = V[i] + 1

on send(msg):
    V[i] = V[i] + 1
    msg.vc = copy(V)

on receive(msg):
    for k in 0..N-1:
        V[k] = max(V[k], msg.vc[k])
    V[i] = V[i] + 1

Compare two vectors element-wise:

ConditionMeaning
VA[k] ≤ VB[k] for all k, and VA ≠ VBa → b (a causally precedes b)
VB dominates VA symmetricallyb → a
Each dominates on some indexa || b — concurrent / conflicting

Now the implication runs both ways: a → b if and only if VA < VB. The "neither dominates" case precisely identifies concurrent updates — exactly the writes that conflict and need reconciliation. This is why Amazon's Dynamo (and Riak after it) attach a vector clock to every object version. On read, if two siblings are concurrent, the store returns both and lets the application (or a CRDT-style merge) resolve them instead of silently dropping one. See Designing a Key-Value Store and CRDTs for the reconciliation side.

A version vector is the same machinery applied to replicas of a data item rather than processes exchanging messages: one counter per replica, incremented on each update at that replica, merged on replication. The terms are often used interchangeably; the formal distinction (Baquero/Almeida) is that version vectors track object replica state while vector clocks track event causality.

Trade-off Vector clocks give you exact causality but cost O(N) space per value, where N is the number of writer nodes — and that set grows over time. Riak caps growth with pruning (drop oldest entries by timestamp once the vector exceeds a size/age threshold), which can introduce false-concurrency but bounds memory. Dynamo bounds it by truncating at ~10 entries. The unbounded-growth problem is the practical reason vector clocks don't appear everywhere.

Hybrid Logical Clocks (HLC)

Lamport clocks lose the connection to real time (the counter is meaningless as a date). Pure physical clocks lose causality. Hybrid Logical Clocks (Kulkarni et al., 2014) combine both: a timestamp is a pair (l, c) where l is a physical-time component (close to NTP wall time) and c is a small logical counter that breaks ties and absorbs causality when physical time stalls or goes backward.

// pt() = local physical clock reading
on send or local event:
    l_prev = l
    l = max(l_prev, pt())
    if l == l_prev: c = c + 1
    else:           c = 0
    return (l, c)

on receive(m):       // m = (l_m, c_m)
    l_prev = l
    l = max(l_prev, l_m, pt())
    if l == l_prev and l == l_m: c = max(c, c_m) + 1
    elif l == l_prev:            c = c + 1
    elif l == l_m:               c = c_m + 1
    else:                        c = 0
    return (l, c)

Properties that make HLC attractive:

  • It satisfies the happens-before clock condition like a Lamport clock: causally ordered events get ordered timestamps.
  • The l component stays within a bounded distance of NTP wall-clock time, so timestamps are meaningful and queryable ("give me writes around 10:05").
  • It is monotonic — never goes backward — even when the underlying NTP clock steps backward, because of the max and the logical counter.
  • It is fixed-size (typically 64 bits: ~48 bits physical millis + ~16 bits logical), unlike vector clocks.

HLC is used as the timestamp mechanism in CockroachDB, YugabyteDB, and MongoDB's cluster time. CockroachDB uses HLC plus a configured max_offset (default 500 ms) bound: if it observes a timestamp from another node further ahead than max_offset, it crashes that node rather than risk a causality violation. HLC is the pragmatic middle ground when you don't have specialized hardware — it gives near-physical timestamps with logical-clock safety.

Key insight HLC does not detect concurrency the way a vector clock does — it produces a single comparable value, so it gives you a usable total-ish order, not conflict detection. Choose vector clocks when you must surface conflicts (Dynamo-style), HLC when you want monotonic, near-real timestamps for ordering and MVCC.

TrueTime, commit-wait and Spanner

Google Spanner takes a radically different stance: instead of accepting clock uncertainty, bound it and expose it. The TrueTime API does not return a single instant. It returns an interval:

TT.now() -> { earliest, latest }   // guaranteed: true time in [earliest, latest]
TT.after(t)  -> true if t has definitely passed
TT.before(t) -> true if t has definitely not yet arrived

TrueTime achieves a small interval — typically ε ≈ 1–7 ms — using GPS receivers and atomic clocks in every datacenter, with a time-master daemon polling them and disciplining local clocks. The hardware gives a tight, provable error bound rather than NTP's best-effort estimate.

The payoff is commit-wait, which delivers external consistency (linearizability of transactions, the strongest model). When a read-write transaction commits, Spanner:

  1. Picks a commit timestamp s = TT.now().latest at the moment of commit.
  2. Waits until TT.after(s) is true — i.e. sleeps for ~2ε until the chosen timestamp is guaranteed to be in the past for every node on Earth.
  3. Only then releases locks and acknowledges the commit.

Because every transaction waits out the uncertainty window before becoming visible, if transaction T1 commits before T2 begins (in real time), then s1 < s2 is guaranteed. The total order of commit timestamps matches real-time order — that is the definition of external consistency. The cost is paid directly in write latency: every commit eats roughly 2ε (~10–14 ms historically) of mandatory wait, which is why Google invests so heavily in shrinking ε — smaller uncertainty literally means faster commits.

Trade-off TrueTime trades capital and operational cost (GPS + atomic clocks in every region, a hard dependency Spanner cannot run without) for the ability to do globally-consistent transactions and lock-free, externally-consistent snapshot reads. HLC-based systems (CockroachDB, Yugabyte) get most of the way there on commodity hardware but must use stricter offset checks and occasionally restart transactions on uncertainty (the "uncertainty restart" window), paying in tail latency and occasional retries instead of a hardware bill.
Pitfall Don't claim "Spanner has perfectly synchronized clocks." It does not — it has bounded uncertainty and an algorithm (commit-wait) that tolerates the residual error by waiting it out. The cleverness is in turning an unsolvable problem (perfect sync) into a solvable one (bounded sync + wait). Conflating the two signals shallow understanding.

Choosing a clock for your design

MechanismDetects concurrency?SizeTracks real time?Where used

A principal-level heuristic:

  • Need to display a time or expire a key → physical clock, accept skew.
  • Need a total order over events but no conflict detection → Lamport (or just use the consensus log from Raft, which gives you one for free).
  • Need to detect and surface concurrent writes for application/CRDT merge → vector clocks or version vectors. See CRDTs and Replication Deep Dive.
  • Need monotonic, queryable timestamps for MVCC on commodity hardware → HLC.
  • Need external consistency for global transactions and can afford the infra → TrueTime-style bounded clocks + commit-wait. Relates to Distributed Transactions.

This decision interacts directly with your consistency target. As Consistency Models and PACELC argues, in the normal (no-partition) case you trade latency against consistency — commit-wait makes that trade explicit and visible in p50 write latency.

Principal-level mastery check

What strong engineers do They never reach for System.currentTimeMillis() to order distributed events, and they say why unprompted — skew, backward steps, GC pauses. They distinguish "ordering events" (logical clocks) from "knowing the date" (physical clocks). They can state the one-way nature of the Lamport guarantee and immediately reach for vector clocks when conflict detection is needed. They explain TrueTime as "bounded uncertainty plus commit-wait," not "perfect clocks," and they connect the choice of clock back to the consistency model and the latency budget the design can afford.

  • Your service uses last-write-wins keyed on wall-clock time. Walk me through a concrete sequence where this silently loses a write. How would you detect it in production, and what would you change?
  • When is a Lamport clock sufficient, and when must you upgrade to a vector clock? Give me a system that genuinely needs the vector and one where it would be over-engineering.
  • Vector clocks grow with the number of writers. How does Dynamo or Riak bound that growth, and what correctness anomaly does the bounding introduce?
  • Explain commit-wait. Where exactly is the latency paid, and what would halving TrueTime's ε do to your write throughput and SLOs?
  • You're building a multi-region store on commodity hardware (no GPS clocks). How do HLCs let you approximate Spanner's guarantees, and what do you give up — concretely — versus a true TrueTime deployment?
  • Two replicas show concurrent versions of the same object with non-comparable vector clocks. What are your options for resolving the conflict, and how does the choice differ for a shopping cart versus a bank balance?
