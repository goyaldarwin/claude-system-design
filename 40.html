Stream Processing & Change Data Capture

Batch processing waits for all the data, then computes. Stream processing computes continuously over data that never ends — clicks, sensor readings, database changes — producing answers within seconds. The catch is that "now" is slippery: events arrive late and out of order, so you must reason carefully about which time you mean. And the most common stream source isn't a queue at all — it's your database's own change log, via CDC.

The trade-off Continuous, low-latency answers over unbounded data — in exchange for approximate completeness, stateful operators you must checkpoint, and reasoning about event time vs processing time. Streaming gives you results in seconds instead of hours. You pay by accepting that an answer is always "as of late-arriving data so far," by managing operator state and checkpoints, and by handling the gap between when an event happened and when you process it.

What stream processing is

Stream processing computes over an unbounded data set — a sequence of events with no end — producing continuously updated results, typically within seconds of an event arriving. Contrast with batch processing, which runs over a bounded, complete dataset and finishes. The streaming mindset: you never have "all the data," so every answer is provisional, and the central difficulty is time.

The core pieces:

  • Source — an unbounded input: a Kafka topic, a Kinesis stream, or a database change feed (CDC).
  • Operators — transformations (map, filter, join, aggregate). Some are stateful (a running count, a windowed sum) and must remember data across events.
  • Windows — how you slice an endless stream into finite chunks to aggregate (per-minute counts, per-session totals).
  • Sink — where results go: a database, a dashboard, another stream.
The one-line frameStreaming trades completeness and simplicity for freshness. You get answers in seconds, but every answer is "as of the data seen so far," you must checkpoint operator state to survive failures, and you must decide whether you mean the time an event happened or the time you processed it. Get the time question wrong and your counts are simply incorrect.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the mobile event that arrives 5 minutes late You count "purchases per minute" from a mobile app. A user buys something at 10:00:30, but their phone is offline; the event reaches your pipeline at 10:05:10. Your counter, keyed on arrival time, files it under the 10:05 minute. Which minute's count is now wrong — and which notion of time should you have used instead?
Scenario 2 — the dual-write that drifts (real-world pattern, Netflix/LinkedIn) A service writes an order to its database and publishes an "order placed" event to Kafka for downstream consumers. Over weeks, the database and the event stream slowly disagree — some events never published, some published for writes that rolled back. What's the name of this problem, and what mechanism turns the database itself into the reliable event source?
Scenario 3 — the operator that crashes mid-aggregation A streaming job maintains a running per-user total in memory. The machine running that operator crashes. What happens to the in-flight totals, how do you recover them exactly (not double-counting), and why is this harder than in a stateless service?

Scenario 1 is event time vs processing time. Scenario 2 is CDC and the outbox. Scenario 3 is state and checkpointing. Each maps to an idea below.

The five ideas to understand

Five ideas carry the whole subject. Each has a named failure nested under it.

Idea 1 — Event time vs processing time

Every event has two timestamps: event time (when it actually happened, stamped at the source) and processing time (when your pipeline got to it). They differ because of network delay, offline buffering, retries, and backlogs. For any correctness-sensitive aggregation — counts, billing, analytics — you almost always want event time, so the same event always lands in the same window no matter when it arrives or how backed-up you are.

The failure that happens when you get this wrong — the late event miscounted This is Scenario 1. Using processing time, the 10:00:30 purchase that arrives at 10:05 is counted in the 10:05 minute — the 10:00 minute is undercounted, the 10:05 minute is overcounted, and a backlog or replay would scramble all your numbers. Using event time, the event's own 10:00:30 timestamp puts it in the 10:00 window regardless of arrival — correct and replay-stable. The cost: you can't close the 10:00 window the instant the clock hits 10:01, because late events may still arrive. That's what watermarks handle.

Idea 2 — Windowing and watermarks

To aggregate an endless stream you slice it into windows: tumbling (fixed, non-overlapping — "each minute"), sliding (fixed-size, overlapping — "last 5 minutes, updated every minute"), or session (grouped by activity gaps — "a user's burst of clicks"). A watermark is the engine's estimate of "event time has advanced to T, so windows ending before T can be finalized" — while still allowing a bounded grace period for late stragglers.

The failure that happens when you get this wrong — windows that never close (or close too soon) Without a watermark you face a dilemma: wait forever for possibly-late events (the window never emits, results never appear, state grows without bound) or close immediately (and drop every late event, undercounting). Watermarks resolve it: emit when the watermark passes the window's end, plus a bounded allowed lateness after which stragglers are dropped or sent to a side-output. Set the lateness too tight and you drop real data; too loose and you hold state and delay results. It's a freshness-vs-completeness dial.

Idea 3 — State, checkpointing, and exactly-once

Stateful operators (counts, joins, windows) hold data in memory that must survive failures. Engines periodically take a checkpoint — a consistent snapshot of all operator state plus the input offsets it corresponds to — to durable storage. On failure, the job restores the last checkpoint and replays the source from the matching offsets. Combined with idempotent or transactional sinks, this gives exactly-once processing of the streaming computation.

The failure that happens when you get this wrong — lost or double-counted state This is Scenario 3. A stateless service just restarts; a stateful operator that crashes without checkpoints loses every in-flight aggregate — or, if you naively replay the input without restoring state, double-counts. The fix is the checkpoint + offset pairing: restore state and rewind the source to exactly the offsets that state reflects, so replay neither loses nor duplicates. This is why streaming engines (Flink's distributed snapshots, Kafka Streams' changelog topics) make checkpointing a first-class feature — you don't hand-roll it.

Idea 4 — Change Data Capture and the outbox (the best stream source)

Change Data Capture (CDC) turns a database into an event stream by tailing its write-ahead log — every insert/update/delete becomes an event, in commit order, with no change to the application. It's how you feed search indexes, caches, data warehouses, and other services from a system of record without dual writes. The outbox pattern pairs with it: write business data and an outbox event row in one local transaction, and let CDC publish the outbox — so the event is committed atomically with the data.

The failure that happens when you get this wrong — the dual-write drift This is Scenario 2. Writing to the database and separately publishing to Kafka has no shared transaction (the dual-write problem): a crash between them, or a rolled-back transaction whose event already published, makes the stream and the database drift — silently, over time. CDC fixes it by making the database's own committed log the single source of truth: you only ever write to the database, and every committed change becomes the event. Netflix, LinkedIn, and others run exactly this (Debezium → Kafka) to replace fragile dual-writes. Delivery is at-least-once, so consumers must still be idempotent.

Idea 5 — Backpressure

A streaming pipeline is a chain of operators at different speeds. When a downstream operator (or sink) can't keep up with the rate upstream is producing, the system must push back — signal upstream to slow down — rather than let data pile up. Good engines propagate backpressure all the way to the source (Kafka simply pauses consumption; the log buffers durably).

The failure that happens when you get this wrong — unbounded buffering → OOM Without backpressure, a slow sink causes records to accumulate in in-memory buffers between operators; the buffers grow until the process runs out of memory and crashes — often cascading as the restarted job faces an even bigger backlog. The fix is to make the slowness flow backward: block or slow the upstream operators, ultimately pausing reads from the durable source so the backlog sits safely in the log (disk) rather than in process memory. A bounded, backpressured pipeline degrades by getting slower; an unbounded one degrades by dying.

Famous implementations

The engines made concrete. Same facets — model, time/state handling, and the failure to watch.

Apache Flink

  • Model. True per-event stream processor (not micro-batch); rich event-time semantics.
  • Time/state. First-class event time, watermarks, all window types; distributed checkpoints (Chandy-Lamport snapshots) give exactly-once with offset rewind.
  • The failure to watch. Large state needs careful state-backend + checkpoint tuning; checkpoints competing with throughput.

Kafka Streams

  • Model. A library (not a cluster) embedded in your app; reads/writes Kafka topics; stream–table duality (a changelog is a table).
  • Time/state. Event-time windows; local state stores (RocksDB) backed by Kafka changelog topics for fault-tolerant exactly-once within Kafka.
  • The failure to watch. Inherits Kafka's rebalance behavior; state restore from changelog can be slow on large state.

Debezium (CDC)

  • Model. Tails database WALs (MySQL binlog, Postgres logical replication, MongoDB oplog) and emits change events to Kafka.
  • Time/state. Commit-ordered change stream; the implementation of the outbox/CDC pattern (Scenario 2's fix).
  • The failure to watch. At-least-once (consumers must dedupe); snapshot phase + WAL pressure under bursty load; connector lag drift if not monitored.

Build it yourself

These click when you design the pipeline. Work each before opening the approach.

Count events per minute, correctly, with late data

Build a per-minute purchase counter from a mobile stream where events can arrive minutes late and out of order. Which time, which window, and how do you decide when a minute's count is final?

Reveal the approach

Use event time (each event's source timestamp) with tumbling 1-minute windows, so every purchase lands in the minute it actually happened regardless of arrival. Track a watermark = max-seen-event-time minus an allowed-lateness slack (say 2 minutes, sized to your real tail). Emit a minute's count when the watermark passes the end of that minute; keep accepting late events for the allowed-lateness window and emit updates, then drop (or side-output) anything later. Result: correct, replay-stable counts, finalized after a bounded delay — the freshness-vs-completeness trade made explicit. Contrast with processing-time windows, which would be wrong the moment anything arrived late (Scenario 1).

Stream database changes to a search index with CDC

Your orders live in Postgres; a search service needs them indexed within seconds, and they must never silently drift from the database. Design the pipeline without dual writes.

Reveal the approach

Run CDC (Debezium) tailing Postgres's logical-replication WAL → emit each committed change to a Kafka topic in commit order → a consumer applies them to the search index. The application only writes to Postgres; the index is derived from the committed log, so it can't diverge from the source of truth (no dual write). If you also need to emit semantic events (not raw row changes), use the outbox: write an outbox row in the same transaction as the order, and have CDC publish that. Delivery is at-least-once, so make the index-apply idempotent (upsert by primary key). To rebuild the index, replay the topic from the start — the log is the history.

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Design real-time + correct (Lambda/Kappa). You need a fast approximate count now and an exact count later. Decide whether to run a streaming + batch layer (Lambda) or a single replayable stream (Kappa). → Batch & Big Data, Ad Click Aggregation.
  • Handle a bursty source without OOM. A producer occasionally 10×es its rate. Design the pipeline so the backlog sits durably and operators backpressure instead of crashing. → Resilience Patterns.
  • Build event sourcing + CQRS. Store all state changes as an event log and derive read models from it. Design the write log, the projections, and how you rebuild a read model. → Event-Driven Architecture.
