Indexes

An index makes reads fast by keeping a sorted, pointer-laden copy of a column — and you pay for it in storage and slower writes, because every write must now update the index too. Every decision here is about which reads you're buying speed for, and what that costs the write path.

The trade-off Fast reads in exchange for extra storage and slower writes. An index turns "scan every row" into "jump straight to it." In return every write must maintain the index, and every index is more disk — so an unindexed query is a disaster and an over-indexed table is slow to write.

What an index is

An index is a separate, sorted data structure that maps a column's values to the location of the matching rows — a table of contents for your data. Without it, finding a row means a sequential scan: read every row and check. With it, the database jumps to the value in O(log n) instead of O(n). On a 10-million-row table, that's the difference between ~24 reads and 10 million.

The catch is symmetric: the index is a second copy that must be kept in sync. Every insert, update, or delete to an indexed column must also update the index — so indexes speed reads and slow writes, and consume storage. That tension drives all four decisions.

The one-line frameThe skill isn't "add indexes" — it's matching indexes to how the data is actually queried. The right index turns a 10M-row scan into 10 reads; the wrong (or missing) one leaves a query scanning the whole table under load, and a pile of unused indexes just taxes every write.

Three scenarios — decide before you read

Don't scroll past these. Decide your answer first — the rest of the lesson is the answer. If you already know all three cold, skip to the decision table.

Scenario 1 — the query that was fine yesterday A query filters WHERE email = ? on a 10-million-row users table with no index on email. It was fast in dev with 1,000 rows. In production, under load, the whole site slows to a crawl. What is the database doing on every one of those queries, and how many rows is it reading?
Scenario 2 — the write-heavy ingest You're ingesting 100,000 sensor readings per second, each with a random key, into a classic B-Tree-indexed table. Writes start fast, then degrade badly as the table grows. What is the B-Tree doing on each random-key insert that makes high-volume random writes its worst case — and what engine is built for exactly this?
Scenario 3 — the composite index that isn't used You created an index on (country, city, age). A query filters only on city and runs slow — the index seems ignored. Another filters country = ? AND age > ? and only half-uses it. Why does the database skip or partially use an index you built specifically for these columns?

Scenario 1 is the full-table scan from a missing index. Scenario 2 is the wrong-engine write wall (B-Tree vs LSM). Scenario 3 is the leftmost-prefix rule. Each maps to a decision below.

Indexing is defined by four decisions

Make each well and reads fly without crushing writes; make one wrong and you get a named failure, nested beneath it.

Decision 1 — What do you index (and, just as important, not)?

Index the columns your queries actually filter, join, and sort on — and resist indexing everything. Each index speeds some reads but taxes every write and costs storage. The signal for "needs an index" is a query plan showing a sequential scan on a large table; the signal for "too many indexes" is write latency climbing while indexes sit unused.

The failure that happens when you get this wrong — the full-table scan This is Scenario 1. With no index on the filtered column, the database falls back to a sequential scan — reading every row to find matches. On 10M rows that's 10M reads that should have been ~24, and under concurrent load those scans saturate disk and CPU and cascade into a site-wide slowdown. Missing indexes are the most common cause of "the app suddenly got slow." The fix: find it with EXPLAIN ANALYZE (look for "Seq Scan"), add the index, confirm the plan flips to an index scan. The inverse failure is over-indexing: every redundant index slows all writes and wastes space for reads nobody runs.

Decision 2 — Which storage engine backs the index: B-Tree or LSM?

Almost every index resolves to one of two families, and the choice dictates write throughput, read tail, and disk footprint:

  • B-Tree (update-in-place). Keys sorted in fixed-size pages (InnoDB 16 KB) in a shallow high-fanout tree; a lookup touches O(log n) pages — ~3–4 reads for a billion rows, mostly cached. The default for read-heavy OLTP (InnoDB, Postgres). Cost is on writes: an update rewrites a whole page in place (a random write), splits pages on overflow, and writes the WAL.
  • LSM-Tree (log-structured merge). Never updates in place: writes append to an in-memory memtable + sequential WAL; full memtables flush as immutable sorted SSTables; background compaction merges them. Ingestion is sequential and fast. The engine behind RocksDB, Cassandra, ScyllaDB.

The honest comparison is the RUM conjecture: optimize at most two of read, update, and space amplification. B-Tree = low read amplification, high write amplification under random keys. LSM = low write amplification at ingest, higher read amplification (probe multiple SSTable levels), mitigated by Bloom filters.

The failure that happens when you get this wrong — the wrong-engine write wall This is Scenario 2. High-volume random-key inserts are the B-Tree's worst case: each insert seeks a random leaf page, rewrites the whole 16 KB page for a tiny logical change (write amplification), and splits pages as they fill — so ingest throughput collapses as the table grows. An LSM-tree turns those same writes into sequential appends, which is why time-series and high-ingest workloads pick it. (Caveat: "LSM is write-optimized" is only true at ingest — sustained compaction can push lifetime write amplification above a B-Tree's. Full treatment: Storage Engines.)

Decision 3 — Clustered vs secondary, covering, and column order

Two layout choices dominate query plans:

  • Clustered index (InnoDB primary key) stores the full row in the leaf — a PK lookup needs no second hop.
  • Secondary index leaves store only the PK, so reading non-indexed columns costs an extra lookup back into the clustered index — the classic bookmark lookup.
  • Covering index includes every column the query reads, so the engine answers entirely from the index (an "index-only scan") — no second hop.
  • Composite index (a, b, c) obeys the leftmost-prefix rule: it serves filters on a, a,b, and a,b,c — but a query filtering only on b can't use it. And equality columns must come left of range columns, because the first range predicate stops later columns from narrowing the scan.
The failure that happens when you get this wrong — dead index / bookmark lookup This is Scenario 3. A query filtering only on city can't use an index on (country, city, age) — city isn't the leftmost column, so the database scans instead (the index looks "ignored"). And country = ? AND age > ? only uses the country part: once a range (age >) is hit, the column after it can't narrow further — here city sits between them, so age can't be used to seek. Order composite columns to your access patterns (equality first, then range), not alphabetically. Separately, a non-covering secondary index forces a bookmark lookup per row — add the read columns to make it covering when the query is hot.

Decision 4 — How do you add an index to a live, large table?

Creating an index isn't free — the database must read the whole table to build the sorted structure. On a large, busy table the naive CREATE INDEX can lock writes (or the whole table) for the duration, which on a multi-terabyte table is an outage.

The failure that happens when you get this wrong — the locking migration A plain index build on a huge table holds a lock and saturates I/O scanning every row — turning a "quick fix" into downtime. The fixes: use the engine's non-blocking path (Postgres CREATE INDEX CONCURRENTLY, MySQL online DDL), or for the hardest cases an online-schema-change tool (gh-ost, pt-online-schema-change) that builds a shadow copy and swaps it in. Always build indexes off-peak and throttled. The irony to remember: adding an index to fix slow reads can itself cause an outage if you do it the naive way on a large table.

Self-check — can you reconstruct the three scenarios?

Close the page and rebuild them.

  • Scenario 1 (missing index): What does the DB do without an index on a filtered column, how many rows on 10M, and how do you diagnose + fix it?
  • Scenario 2 (random-key ingest): Why is high-volume random insertion the B-Tree's worst case, and what does an LSM do instead? When is "LSM is write-optimized" misleading?
  • Scenario 3 (composite index): State the leftmost-prefix rule and why equality columns go before range columns. Why is the (country, city, age) index useless for a city-only filter?
  • Clustered vs secondary vs covering — which avoids the bookmark lookup, and how?
  • What's the RUM conjecture, and where do B-Tree and LSM each land on it?
  • Why can adding an index cause an outage, and how do you add one to a live large table safely?

The shape to remember: indexing is 4 decisions — what to index, B-Tree vs LSM, layout & column order, adding one live — with named failures: full scan, wrong-engine wall, dead index, locking migration.

Famous implementations

Same facets each — what it is, engine, key idea, the failure to watch.

InnoDB (MySQL)

  • Engine. B+Tree, update-in-place, 16 KB pages.
  • Key idea. The table is the clustered primary-key index — rows live in the PK leaves, so PK lookups are one hop and secondary indexes store the PK (not a physical pointer).
  • The failure to watch. A random or large PK (e.g. a UUID) scatters inserts across pages and bloats every secondary index — prefer a small, monotonic PK.

PostgreSQL

  • Engine. B-Tree by default, plus rich index types (GIN for full-text/JSONB, GiST, BRIN for huge append-only tables).
  • Key idea. Heap-organized (rows aren't clustered by PK), so it leans on index-only scans via covering indexes (INCLUDE) and CREATE INDEX CONCURRENTLY for online builds.
  • The failure to watch. Bloat from its MVCC model — indexes need maintenance (VACUUM) or they degrade.

RocksDB

  • Engine. LSM-tree; the embedded storage engine behind many systems (used in CockroachDB, Kafka Streams, MySQL's MyRocks, and more).
  • Key idea. Append-only writes + background compaction + per-SSTable Bloom filters to skip levels on point lookups — tuned for write-heavy ingest.
  • The failure to watch. Compaction is a background I/O tax that competes with foreground traffic; range scans can't use Bloom filters, so they're sensitive to compaction health.

Build it yourself

Work each before opening the approach.

Read a query plan and fix the scan

A query on a big table is slow. Walk the steps to confirm it's a missing index and prove the fix worked — what command, what do you look for, what changes?

Reveal the approach

Run EXPLAIN ANALYZE <query>. Look for a Seq Scan (Postgres) / full table scan with a high row count and time. Identify the filtered/joined/sorted columns, create an index matching them (equality columns first, then a range column), re-run EXPLAIN ANALYZE, and confirm the plan now shows an Index Scan (or Index-Only Scan if covering) with a tiny row estimate and far lower time. The discipline: never "add an index because it feels slow" — let the plan tell you, and verify the plan flipped.

Design the composite index for a set of queries

Given three queries on a table — filter by status; filter by status + sort by created_at; filter by user_id + status — design the fewest indexes that serve them all efficiently.

Reveal the approach

An index on (status, created_at) covers query 1 (leftmost prefix status) and query 2 (filter status, then ordered by created_at — the sort comes free). Query 3 needs (user_id, status) since user_id isn't a prefix of the first index. So two composite indexes, ordered equality-first, serve all three — instead of five single-column indexes that each tax writes. The skill is reusing leftmost prefixes across queries to minimize index count.
