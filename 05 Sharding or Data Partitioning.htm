Sharding / Data Partitioning

Breaking a large database into smaller parts across machines to improve manageability, performance, availability and load balancing — and the trade-offs that come with it.

Tax paid Finite capacity. data won't fit/serve from one box — splitting it costs cross-shard joins, hot shards, and rebalancing. Why this exists → lesson 01

Data partitioning (also known as sharding) is a technique to break up a big database (DB) into many smaller parts. It is the process of splitting up a DB/table across multiple machines to improve the manageability, performance, availability and load balancing of an application. The justification for data sharding is that, after a certain scale point, it is cheaper and more feasible to scale horizontally by adding more machines than to grow it vertically by adding beefier servers.

1. Partitioning Methods

There are many different schemes one could use to decide how to break up an application database into multiple smaller DBs. Below are three of the most popular schemes used by various large scale applications.

a. Horizontal partitioning: In this scheme, we put different rows into different tables. For example, if we are storing different places in a table, we can decide that locations with ZIP codes less than 10000 are stored in one table, and places with ZIP codes greater than 10000 are stored in a separate table. This is also called a range based sharding, as we are storing different ranges of data in separate tables.

The key problem with this approach is that if the value whose range is used for sharding isn’t chosen carefully, then the partitioning scheme will lead to unbalanced servers. In the previous example, splitting location based on their zip codes assumes that places will be evenly distributed across the different zip codes. This assumption is not valid as there will be a lot of places in a thickly populated area like Manhattan compared to its suburb cities.

b. Vertical Partitioning: In this scheme, we divide our data to store tables related to a specific feature to their own server. For example, if we are building Instagram like application, where we need to store data related to users, all the photos they upload and people they follow, we can decide to place user profile information on one DB server, friend lists on another and photos on a third server.

Vertical partitioning is straightforward to implement and has a low impact on the application. The main problem with this approach is that if our application experiences additional growth, then it may be necessary to further partition a feature specific DB across various servers (e.g. it would not be possible for a single server to handle all the metadata queries for 10 billion photos by 140 million users).

c. Directory Based Partitioning: A loosely coupled approach to work around issues mentioned in above schemes is to create a lookup service which knows your current partitioning scheme and abstracts it away from the DB access code. So, to find out where does a particular data entity resides, we query our directory server that holds the mapping between each tuple key to its DB server. This loosely coupled approach means we can perform tasks like adding servers to the DB pool or change our partitioning scheme without having to impact your application.

2. Partitioning Criteria

a. Key or Hash-based partitioning: Under this scheme, we apply a hash function to some key attribute of the entity we are storing, that yields the partition number. For example, if we have 100 DB servers and our ID is a numeric value that gets incremented by one, each time a new record is inserted. In this example, the hash function could be ‘ID % 100’, which will give us the server number where we can store/read that record. This approach should ensure a uniform allocation of data among servers. The fundamental problem with this approach is that it effectively fixes the total number of DB servers, since adding new servers means changing the hash function which would require redistribution of data and downtime for the service. A workaround for this problem is to use Consistent Hashing.

b. List partitioning: In this scheme, each partition is assigned a list of values, so whenever we want to insert a new record, we will see which partition contains our key and then store it there. For example, we can decide all users living in Iceland, Norway, Sweden, Finland or Denmark will be stored in a partition for the Nordic countries.

c. Round-robin partitioning: This is a very simple strategy that ensures uniform data distribution. With ‘n’ partitions, the ‘i’ tuple is assigned to partition (i mod n).

d. Composite partitioning: Under this scheme, we combine any of above partitioning schemes to devise a new scheme. For example, first applying a list partitioning and then a hash based partitioning. Consistent hashing could be considered a composite of hash and list partitioning where the hash reduces the key space to a size that can be listed.

3. Common Problems of Sharding

On a sharded database, there are certain extra constraints on the different operations that can be performed. Most of these constraints are due to the fact that, operations across multiple tables or multiple rows in the same table, will no longer run on the same server. Below are some of the constraints and additional complexities introduced by sharding:

a. Joins and Denormalization: Performing joins on a database which is running on one server is straightforward, but once a database is partitioned and spread across multiple machines it is often not feasible to perform joins that span database shards. Such joins will not be performance efficient since data has to be compiled from multiple servers. A common workaround for this problem is to denormalize the database so that queries that previously required joins can be performed from a single table. Of course, the service now has to deal with all the perils of denormalization such as data inconsistency.

b. Referential integrity: As we saw that performing a cross-shard query on a partitioned database is not feasible, similarly trying to enforce data integrity constraints such as foreign keys in a sharded database can be extremely difficult.

Most of RDBMS do not support foreign keys constraints across databases on different database servers. Which means that applications that require referential integrity on sharded databases often have to enforce it in application code. Often in such cases, applications have to run regular SQL jobs to clean up dangling references.

c. Rebalancing: There could be many reasons we have to change our sharding scheme:

  1. The data distribution is not uniform, e.g., there are a lot of places for a particular ZIP code, that cannot fit into one database partition.
  2. There are a lot of load on a shard, e.g., there are too many requests being handled by the DB shard dedicated to user photos.

In such cases, either we have to create more DB shards or have to rebalance existing shards, which means the partitioning scheme changed and all existing data moved to new locations. Doing this without incurring downtime is extremely difficult. Using a scheme like directory based partitioning does make rebalancing a more palatable experience at the cost of increasing the complexity of the system and creating a new single point of failure (i.e. the lookup service/database).

Principal-level deep dive

Partitioning at scale: rebalancing and hot shards

The original lesson covers why we partition; the hard part in production is changing a partitioning scheme while traffic is live, and surviving the inevitable skew. A partition map is not a static decision — it is a control plane that has to move data without violating consistency or blowing through SLOs. This is where most designs that look fine on a whiteboard fall apart.

Why hash(key) % N is a trap

Modulo-by-node-count is the partitioning scheme everyone reaches for first, and it is almost always wrong. The mapping partition = hash(key) % N ties the key's home directly to the cluster size N. The moment you go from 8 nodes to 9, the modulus changes for nearly every key: empirically about (N-1)/N of all keys remap — roughly 89% at N=9. That triggers a near-total reshuffle, saturating the network and cache layer exactly when you were trying to add capacity to relieve pressure. It is a positive-feedback failure mode.

Two fixes dominate real systems:

  • Consistent hashing with virtual nodes. Keys and nodes map onto a hash ring; adding a node only steals keys from its ring neighbors, so only ~K/N keys move (K = total keys). Plain consistent hashing has its own skew problem — random placement gives ~O(log N) load imbalance — so production systems (Dynamo, Cassandra, Riak) assign each physical node 128–256 vnodes to smooth the distribution. See Consistent Hashing.
  • Fixed partition count (decoupling). Pick a large fixed number of partitions up front (e.g. 1024 or 4096), far more than nodes, and assign whole partitions to nodes. Rebalancing becomes "move partition P from node A to node B" — an integer reassignment in the partition map, never a rehash of keys. This is how Kafka, Elasticsearch, and Riak's ring_creation_size work. The cost is choosing the count wisely: too few caps your max node count, too many adds metadata and per-partition overhead.
Principal heuristicNever let the number of partitions equal the number of nodes. Decouple them. Partitions are a logical unit you rebalance; nodes are physical capacity you scale. Conflating them is what makes % N toxic.

Range vs hash partitioning

This is the most consequential trade-off, and it is not symmetric.

DimensionHash partitioningRange partitioning
DistributionEven, by constructionProne to skew; needs split/merge
Range scans (WHERE ts BETWEEN ...)Scatter-gather across all partitionsHits one / few contiguous partitions
Hot-spot riskLow for random keysHigh for monotonic keys (timestamps, autoinc IDs)
Used byDynamo, Cassandra (Murmur3 token)HBase, Bigtable, CockroachDB, Spanner

The classic failure: range-partitioning on a timestamp gives you clean range scans but funnels every write into the "latest" partition — a single hot region. Cassandra-style hashing kills the hot spot but makes a recent-events query a scatter-gather over the whole ring. CockroachDB and Bigtable split the difference: range partitions plus automatic split-on-load, so a hot range gets cleaved in two and the halves migrate apart.

Hot shards and skew mitigation

Even with hashing, individual keys can be hot — a celebrity user, a viral object, a tenant 100x larger than the median. Hashing distributes keys across partitions but cannot split a single hot key. Tactics, in order of escalation:

  • Key salting / decomposition. Append a bounded random suffix (key#0..key#9) to spread one logical key across 10 partitions. Writes fan out trivially; the cost is reads must now gather and merge all 10 — so reserve this for known hot keys, not blanket application.
  • Split the hot partition. Detect a partition exceeding a load threshold and dynamically split its key range, moving one half elsewhere. Requires range partitioning or a split-aware directory.
  • Request hedging & caching. For read-hot keys, a small front cache (or per-key request coalescing) absorbs the storm without touching the partition layout at all — often the cheapest fix.
  • Tenant isolation. Pin oversized tenants to dedicated partitions/nodes so their skew can't starve neighbors (noisy-neighbor control).

Secondary indexes in a partitioned world

Once data is partitioned by primary key, "find all rows where color = red" becomes a distributed problem with two distinct designs:

  • Local (document-partitioned) secondary index. Each partition indexes only its own documents. Writes are cheap — one local index update, same partition. Reads are expensive: the query must fan out to every partition and merge (scatter-gather), because matching rows can live anywhere. This is Elasticsearch and Cassandra's default. Good for write-heavy workloads where read latency tail is tolerable.
  • Global (term-partitioned) secondary index. The index itself is partitioned by the indexed term, so all color = red entries sit together and a read hits one partition. Reads are fast and targeted. Writes pay: a single document update may touch multiple index partitions (one per indexed field), so writes become distributed and are usually made asynchronous — meaning the global index is eventually consistent with the base data. DynamoDB's GSIs are exactly this, and Amazon documents their async, eventually-consistent nature.
The real trade-offLocal index = cheap writes, expensive scatter-gather reads. Global index = fast reads, expensive distributed writes that are usually eventually consistent. You are choosing which operation pays the cross-partition tax.

Directory-based / lookup-table partitioning

Instead of computing a key's home from a function, store the mapping explicitly in a directory service (the partition map). This is what fixed-partition systems effectively do, and it is the foundation of Bigtable/HBase (the META region maps key ranges to tablet servers) and Vitess (the topology + VSchema). The directory gives you maximum flexibility — arbitrary, non-algorithmic placement, and the ability to move any partition anywhere — at the cost of a lookup hop and a piece of critical, highly-available infrastructure. The directory must not be a single point of failure: it is replicated and aggressively cached on clients, with cache invalidation on map changes. Most production lookup paths are "client cache hit → go straight to the right node; on a stale-shard error, re-fetch the map."

Resharding a live system with zero downtime

The endgame: changing the partition map under production traffic. The proven playbook (used by Vitess resharding, MySQL-at-scale teams, and DynamoDB adaptive capacity) is a dual-write / backfill / cutover dance:

  • 1. Prepare the new partitions and start double-writing — every write goes to both the old and new layout. The application now tolerates either map.
  • 2. Backfill historical data from old to new partitions in the background, throttled to protect the live workload. Track a high-water mark.
  • 3. Verify by comparing checksums/row counts between old and new for each key range; reconcile drift caused by races during backfill.
  • 4. Cutover reads partition-by-partition, not all at once — flip the directory to point new reads at the new layout, one key range at a time, so you can roll back a single range if verification fails.
  • 5. Stop double-writing and decommission the old partitions only after reads have been stable on the new map for a safe soak period.

The non-obvious hard parts are ordering and idempotency: during double-write, a backfill read can race a live write and stomp newer data, so backfill must be idempotent and last-writer-wins by version/timestamp, and you should replay the change log (CDC / binlog) over the backfilled snapshot to catch in-flight mutations. Spanner and CockroachDB hide most of this behind transactional range splits, but if you are sharding MySQL or a key-value store yourself, you own this state machine.
