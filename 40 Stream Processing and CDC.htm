Stream Processing and Change Data Capture

Computing continuously over unbounded data — getting correct answers when records arrive late, out of order, and forever.

Unbounded data and the streaming mindset

Batch processing assumes a bounded dataset: you have all the input, you run a job, it finishes, you get an answer. Stream processing assumes the input is unbounded — events arrive forever, the job never "completes," and you must emit results incrementally before you have seen everything. This single difference cascades into every hard problem in the field: when is a result final? what does "the answer" even mean for an infinite input? how do you recover state without re-reading an infinite log?

The modern reframing (popularised by the Google Dataflow model and implemented in Apache Flink, Kafka Streams, and Spark Structured Streaming) is that batch is just a special case of streaming where the input happens to be finite. A well-designed streaming engine answers three orthogonal questions: what is being computed (the aggregation), where in event time it is computed (windowing), and when in processing time results are materialised (triggers and watermarks). Keeping these axes separate is what lets you reason about correctness.

Key insight"Streaming vs batch" is the wrong axis. The real axis is bounded vs unbounded and latency tolerance. The same Flink program can run over a Kafka topic (unbounded, low latency) or a historical S3 dataset (bounded, reprocessing). This is the Lambda-to-Kappa evolution — see Batch and Big Data Processing.

Event time vs processing time, and watermarks

Two clocks matter and they are never the same:

  • Event time: when the event actually occurred at the source (embedded in the record, e.g. the timestamp a user clicked).
  • Processing time: when the event reaches the operator that handles it.

The gap between them — skew — comes from network delay, queueing in Kafka, mobile clients that were offline for hours, and operator backlog. If you aggregate by processing time you get answers that are easy to compute but wrong: a "9:00–9:01 clicks" count that includes a click from 8:47 that arrived late. Correct analytics almost always require event-time processing.

But event time creates a completion problem: at 9:05 processing time, can you close the 9:00–9:01 window? Maybe a straggler is still in flight. The engine's answer is the watermark — a monotonically increasing assertion that "I believe I have now seen all events with event time ≤ T." A watermark of 9:02 means windows ending at or before 9:02 can fire.

Watermarks are heuristics, not guarantees. A common strategy is bounded out-of-orderness: watermark = max_event_time_seen − allowed_lateness. Set the lag too small and you drop legitimate late data; set it too large and every window result is delayed by that lag. This is the fundamental latency–completeness trade-off of stream processing.

Trade-offTight watermark = low latency, more dropped/late records. Loose watermark = high completeness, higher latency. There is no free lunch — you choose where on the curve to sit, and you handle the tail with allowed lateness (re-fire the window) or a side output for dropped records so nothing is silently lost.

Windowing: tumbling, sliding, session

Aggregating an infinite stream requires slicing it into finite chunks. The three canonical window types:

TypeDefinitionOverlapTypical use
TumblingFixed size, contiguous, non-overlapping (e.g. 1-minute buckets)None — each event in exactly one windowPer-minute metrics, billing periods
Sliding (hopping)Fixed size, advances by a smaller step (e.g. 5-min window every 1 min)Yes — an event lands in multiple windowsMoving averages, "last 5 min" dashboards
SessionData-driven; closes after a gap of inactivity (e.g. 30 min idle)None, but variable length per keyUser sessions, fraud bursts

Sliding windows multiply state and output: a 1-hour window sliding every minute means each event is replicated into ~60 windows. Session windows are the trickiest because a late event can merge two previously separate sessions, forcing the engine to retract and recompute. The ad-click counting problem in Designing Ad Click Aggregation is essentially a windowing-plus-watermark design exercise: count clicks per ad per minute in event time, tolerate late clicks, and stay exactly-once for billing.

// Flink event-time tumbling window, 1-minute click counts per ad
clicks
  .assignTimestampsAndWatermarks(
      WatermarkStrategy
        .<Click>forBoundedOutOfOrderness(Duration.ofSeconds(30))
        .withTimestampAssigner((e, ts) -> e.eventTimeMillis))
  .keyBy(c -> c.adId)
  .window(TumblingEventTimeWindows.of(Time.minutes(1)))
  .allowedLateness(Time.minutes(5))     // re-fire if stragglers arrive
  .sideOutputLateData(droppedTag)        // anything later than 5 min -> audit
  .aggregate(new CountAgg());

State, checkpointing, and exactly-once

A stateless stream operator (a filter, a map) is trivial to recover — just replay. The hard part is stateful operators: a running count, a window buffer, a join's hash table, a dedup set. This state can be gigabytes to terabytes per operator and must survive failures without corrupting results.

Flink's mechanism is asynchronous barrier snapshotting (a variant of Chandy–Lamport). Periodically the source injects a checkpoint barrier into the stream. As the barrier flows through the dataflow graph, each operator snapshots its state when the barrier passes. When the barrier reaches all sinks, a globally consistent snapshot exists across every operator. State is persisted to durable storage (RocksDB local + S3/HDFS for the checkpoint). On failure the whole job rewinds to the last completed checkpoint and source offsets are reset to the recorded positions.

Pitfall"Exactly-once" almost never means each event is physically processed once. It means exactly-once effect on state — the observable result is as if each event were processed once. Replaying after a checkpoint reprocesses events; correctness comes from rewinding state in lockstep with input offsets, plus idempotent or transactional sinks.

End-to-end exactly-once requires the sink to cooperate. Two patterns:

  • Idempotent writes: the sink uses a deterministic key (e.g. adId:minute) so reprocessing overwrites rather than double-counts. Simplest and most robust.
  • Two-phase commit (transactional sink): Flink's TwoPhaseCommitSinkFunction pre-commits output on checkpoint and commits when the checkpoint completes, so partial output is never visible. Kafka transactions implement this for stream-to-stream pipelines. This is a distributed-transaction protocol layered on the checkpoint — see Distributed Transactions.

Compare delivery semantics with the broker layer in Kafka and Log-based Messaging: the consumer offset commit and the producer transaction together are what make a Kafka→Flink→Kafka pipeline exactly-once.

Stream–table duality

One of the most powerful mental models in this space: a stream and a table are two views of the same data.

  • A table is a snapshot of state at a point in time — the current value for each key.
  • A stream is the sequence of changes — the changelog — that produced that table.

Replaying a stream and aggregating it (a "table" of latest-value-per-key) is the aggregation direction; emitting a row every time a table changes is the changelog direction. Kafka encodes this directly: a compacted topic keyed by primary key is a table (it retains the latest value per key), while a normal topic is the change stream. Kafka Streams' KTable vs KStream and Flink's dynamic tables are this duality made into an API. The deep consequence: a database's replication log and a Kafka topic are the same abstraction, which is exactly what makes CDC possible.

Event sourcing and CQRS

Event sourcing inverts the usual persistence model. Instead of storing current state and mutating it in place, you store the immutable log of events that occurred (OrderPlaced, ItemAdded, OrderShipped), and current state is a left-fold over that log. The log is the source of truth; any state view is a derived, rebuildable projection.

Benefits: a complete audit trail for free, the ability to reconstruct state at any past time ("temporal queries"), and the ability to build new read models retroactively by replaying history. Costs: events are immutable so schema evolution and GDPR-style deletion are genuinely hard (crypto-shredding is the usual answer), and naive replay of a long log is slow — you need periodic snapshots to bound replay cost.

CQRS (Command Query Responsibility Segregation) frequently pairs with it: separate the write model (validates commands, appends events) from one or more read models (denormalised, query-optimised projections built by consuming the event stream). The read side is eventually consistent with the write side — a deliberate trade you must surface in the UX. This is a core pattern in Event-Driven Architecture.

Trade-offEvent sourcing + CQRS buys auditability, replayability, and independently scalable reads. It costs you eventual consistency on the read side, harder deletes/corrections, and significant operational complexity. Reach for it where the audit log is the business (payments, ledgers, trading) — not as a default architecture.

Change Data Capture and the outbox pattern

Most companies already have their truth in an OLTP database, not in a Kafka log. Change Data Capture is how you turn that database's mutations into a stream — the bridge between the table world and the stream world. Two approaches, and the distinction is critical:

Query-based (polling)Log-based (WAL tailing)
MechanismPoll WHERE updated_at > last_seenRead the DB's replication log (WAL/binlog/redo)
Captures deletes?No (row is gone)Yes
Captures intermediate states?No (only last value between polls)Yes (every change)
Source loadRepeated table scansLow — reads sequential log
LatencyPoll intervalNear real-time
ToolingKafka Connect JDBC, cron jobsDebezium, Maxwell, AWS DMS

Log-based CDC is strongly preferred at scale. Debezium connects as a replication client — it reads the Postgres WAL (via a logical decoding slot), MySQL binlog, or MongoDB oplog — and emits a structured change event (before, after, op, source metadata) per row mutation onto Kafka. Because it reads the same log the database uses for its own replicas, it sees every insert/update/delete in commit order with minimal source overhead. This is the stream–table duality at work: the WAL is the changelog of the database table.

PitfallA logical replication slot in Postgres pins WAL until the consumer acks. If Debezium falls behind or dies, WAL accumulates and can fill the disk and take the primary down. Monitor replication slot lag as a first-class SLO and alert aggressively.

The dual-write problem and the outbox pattern

A service that does "write to DB, then publish to Kafka" has a dual-write problem: the two writes are not atomic. If the process crashes between them, the DB and Kafka diverge — an order exists with no OrderPlaced event, or vice versa. Distributed transactions across a DB and a broker (XA) are slow, poorly supported, and operationally fragile.

The transactional outbox pattern solves this elegantly using CDC. In the same local DB transaction as the business write, insert a row into an outbox table. That single ACID transaction guarantees the business change and the intent-to-publish commit or roll back together. A CDC connector (Debezium) then tails the outbox table and publishes those rows to Kafka. The broker may receive duplicates on retry, so consumers must be idempotent — but no event is ever lost and none is published for a rolled-back transaction.

BEGIN;
  UPDATE orders SET status = 'PLACED' WHERE id = 42;
  INSERT INTO outbox (aggregate_id, event_type, payload)
    VALUES (42, 'OrderPlaced', '{"orderId":42,...}');
COMMIT;
-- Debezium tails the WAL, sees the outbox INSERT,
-- publishes 'OrderPlaced' to Kafka. At-least-once -> idempotent consumers.

CDC also powers broader DB-to-stream pipelines: replicating an OLTP database into a data lake or warehouse, keeping a search index (Elasticsearch) or a cache in sync, and feeding real-time materialised views. It decouples the system of record from the proliferating downstream consumers without coupling them to the application code — the database becomes a streaming source.

Backpressure

When a downstream operator (or sink) cannot keep up with its input rate, the system must do something. The wrong answer is to buffer unboundedly — that just relocates the failure to an OOM crash. Backpressure is the mechanism by which a slow consumer slows down its producers all the way back to the source.

Flink propagates backpressure naturally through its bounded network buffers and credit-based flow control: when a downstream task's input buffers fill, it stops granting credits, the upstream task blocks on the network, and the slowdown cascades to the Kafka source, which simply stops polling new records. Because Kafka is a durable log, the data isn't lost — it stays in the topic, consumer lag grows, and the job catches up when capacity frees up. Contrast this with a push-based system over an unbounded in-memory queue, where a slow consumer causes data loss or crashes.

Key insightA durable log decouples producer and consumer rates and turns backpressure into a lag metric rather than a failure. "Consumer lag" is the single most important streaming health signal — rising lag means you are falling behind real time; flat lag means you are keeping up. See Queues and Observability.

Principal-level mastery check

What strong engineers doThey separate the three Dataflow questions — what / where in event time / when in processing time — instead of conflating windowing with triggering. They reach for event time and watermarks by default for analytics, and can articulate the latency–completeness trade-off with a concrete watermark strategy. They know "exactly-once" means exactly-once effect, achieved via checkpointing plus idempotent or transactional sinks, not magic. They propose log-based CDC + the outbox pattern to kill the dual-write problem rather than hand-waving "and then publish to Kafka." And they treat consumer lag and replication-slot lag as first-class operational SLOs.
  • You're counting unique users per minute over a stream where mobile clients can be offline for 6 hours. How do you set watermarks, and what happens to a click that arrives 6 hours late? Walk me through allowed lateness vs side outputs vs dropping.
  • Your Flink job claims exactly-once but the downstream database shows duplicate rows after a failover. Where would you look, and what makes a sink truly exactly-once?
  • Design the order-service write path so that a crash can never leave the database and Kafka inconsistent. Why not just use a distributed transaction across both?
  • Contrast log-based and query-based CDC. When would query-based actually be the right call, and what's the operational risk of a Postgres logical replication slot?
  • You're adding a new read model to an event-sourced system with two years of history. How do you build it, and how do you bound replay time?
  • Consumer lag on one partition is climbing while others are flat. What does that tell you, and how does backpressure relate to lag?
