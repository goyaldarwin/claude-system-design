Designing Collaborative Editing (Google Docs)

Many people edit the same document at the same time and everyone sees a consistent result within milliseconds — no "merge conflict" dialogs, no lost keystrokes. The core problem is reconciling concurrent edits to the same shared state, which has exactly two serious answers (Operational Transformation and CRDTs), each with sharp trade-offs — and a real-world third way (server-authoritative) that companies like Figma actually chose.

The crux Real-time concurrent editing that always converges to the same document — in exchange for a hard conflict-resolution algorithm (OT or CRDT), per-character bookkeeping, and presence/cursor sync. Everyone edits freely and the document stays consistent. You pay with one of two intricate algorithms to merge concurrent ops, metadata to make positions stable under concurrent inserts, and the machinery to broadcast edits and presence in real time.

1. Requirements

Functional: multiple users edit one document concurrently; each sees others' changes in real time (sub-second); the document always converges to the same state for everyone; show presence (who's here) and live cursors/selections; support offline editing that syncs on reconnect; keep history (undo, version restore).

Non-functional: low latency (your own keystrokes feel instant; others' appear in <1s), strong eventual convergence (no permanent divergence, no lost edits), scalable to many concurrent editors per document and many documents, available (keep working through brief disconnects).

2. Capacity estimate

Say 10M active documents, average 5 concurrent editors on active ones, an editor emits an op every ~0.5s while typing. Work it:

  • Op rate per doc: 5 editors × 2 ops/s = 10 ops/s/doc, each broadcast to the other 4 → ~40 messages/s/doc. Small per doc — the challenge is the number of docs and live connections, not per-doc throughput.
  • Connections: millions of concurrent WebSocket connections (persistent, one per editor). Consequence: you need a connection layer that holds millions of long-lived connections, and a way to route an op to exactly the editors of that document.
  • The real wall — editors per doc: conflict-resolution and per-character metadata cost grows with concurrent editors on one doc. Etherpad got inconsistent past ~150 editors on a single pad; Figma engineered for ~200. Consequence: there's a practical ceiling on editors-per-document set by the algorithm and metadata, and it's a deliberate design target, not infinity.

3. The API

// over a persistent WebSocket connection per editor:
join(docId)                      -> initial document state + version
sendOp(docId, op, baseVersion)   // op = insert/delete/format at a position/id
// server broadcasts:
applyOp(docId, transformedOp, version)   // others' ops, ready to apply
presence(docId, [{userId, cursor}])      // who's here + cursors

Edits travel as small operations (insert "x" at position p; delete range; apply bold) with a base version (what the client had seen), not as whole-document snapshots — ops are tiny and are what the conflict algorithm transforms/merges.

4. High-level design

Each client holds a local copy and applies its own edits optimistically (instant local feedback), sending ops to a sync server over a persistent WebSocket. The sync server is the per-document coordination point: it orders incoming ops, resolves concurrency (via OT or CRDT), and broadcasts the resolved ops to all other editors of that document, who apply them. Documents are sharded across sync servers (by docId); a coordination layer routes each doc's editors to the same server (its owner) so there's a single ordering authority per doc. Ops are persisted to a durable op log; periodic snapshots bound replay time.

5. Document model — ops, not snapshots

A document is represented as an initial state plus a sequence of operations — this is what makes real-time merge and history possible. The crucial sub-decision is how an op addresses a position: a naive numeric index (insert at character 5) breaks under concurrency (everyone's indices shift differently). The robust answer (used by CRDTs and modern editors) is to give each character a stable, unique, densely-orderable identifier so an op references an identity, not a shifting offset. Formatting is often modeled as attributes over ranges/ids.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — two people type at the same spot The document is "Hello world". User A inserts "X" at index 5; simultaneously User B inserts "Y" at index 5. Both ops say "insert at 5," but after A's insert, B's "5" no longer means the same place. If you naively apply both, the two clients end up with different documents. What are the two algorithms that fix this, and how does each think about it?
Scenario 2 — Etherpad past 150 editors / Figma's choice (real) Etherpad (OT-based) became inconsistent past ~150 concurrent editors on one pad despite no resource pressure. Figma evaluated pure CRDTs and OT, rejected both, and built a server-authoritative system with CRDT-inspired last-writer-wins registers, scaling to ~200 editors. Why is OT hard to get right at scale, why are pure CRDTs heavy, and why did having a server change the calculus?
Scenario 3 — you edited offline, then reconnected A user edits on a flaky connection / fully offline for 10 minutes, making 50 edits, while others changed the doc too. On reconnect, all those edits must merge in without losing any — theirs or others'. How do you guarantee no edit is lost and everyone converges, and what does the client need to track while offline?

Scenario 1 is concurrent edits (OT vs CRDT). Scenario 2 is convergence + the algorithm trade-off. Scenario 3 is transport, presence & offline. Each maps to a sub-problem.

The four hard sub-problems

The transport is the easy part. These four are the real difficulty — each with a nested failure.

Sub-problem 1 — Resolving concurrent edits: OT vs CRDT

Two genuine approaches to merging concurrent ops on shared ordered text:

  • Operational Transformation (OT) — transform each incoming op against the concurrent ops that already applied, adjusting its position so the result is consistent (A's "insert at 5" and B's "insert at 5" become "insert at 5" and "insert at 6"). Powers Google Docs. Needs a central server to order ops and is famously tricky to prove correct (many transform-function edge cases).
  • CRDTs — give each character a stable unique id with a dense order, so concurrent inserts reference identities, not offsets, and merge commutatively with no transform. Powers Yjs/Automerge. No central transform needed (works peer-to-peer), but carries more metadata.
The failure that happens when you get this wrong — the position clash This is Scenario 1. If you treat edits as "apply this op at this numeric index" and just relay them, concurrent inserts/deletes make the indices mean different things on different clients → the documents diverge (A sees "HelloXY world", B sees "HelloYX world"). Both OT and CRDT solve it, differently: OT rewrites the op's position relative to concurrent ops (server-coordinated); CRDT avoids positions entirely by using stable per-character ids with a dense total order, so "where" is unambiguous regardless of concurrency. Pick the model first — everything else (transport, persistence) depends on it. Numeric indices over a relayed channel is the naive trap that never converges.

Sub-problem 2 — The convergence guarantee (and choosing the algorithm)

The non-negotiable property: after all ops propagate, every client's document is identical (strong eventual convergence), with no lost edits. OT guarantees it via correct transform functions + a server ordering; CRDTs guarantee it mathematically (commutative/associative/idempotent merge — see CRDTs). The choice between them (and a third option) is the central engineering decision.

The failure that happens when you get this wrong — replicas diverge (and the OT-vs-CRDT-vs-server trade) This is Scenario 2. If convergence isn't guaranteed, clients permanently disagree about the document — the worst possible bug. The trade-offs that decide the algorithm: OT is memory-light but the transform functions are notoriously hard to get correct — subtle concurrency cases cause divergence (Etherpad's inconsistency past ~150 editors is this class of problem). Pure CRDTs are provably convergent but carry heavy per-character metadata (ids + tombstones), costly for large/long-lived docs. The third way — server-authoritative (Figma): because you have a trusted central server (which most products do), you can use the simplest CRDT-flavored scheme — property-level last-writer-wins registers with fractional indexing — letting the server be the ordering authority and skipping both OT's transform complexity and pure-CRDT's full machinery. Figma rejected both pure approaches for exactly this reason and scaled to ~200 editors. The lesson: pure peer-to-peer CRDTs shine when there's no trusted server (true local-first); when you have a server, lean on it to simplify.

Sub-problem 3 — Real-time transport, presence & offline

Edits and presence flow over a persistent WebSocket (low-latency bidirectional). The client applies its own ops optimistically for instant feedback, then reconciles with the server's ordering. Presence/cursors are broadcast as ephemeral state. And the client must survive disconnects: buffer local ops while offline and sync on reconnect.

The failure that happens when you get this wrong — lost edits on reconnect This is Scenario 3. If the client doesn't durably buffer its offline ops (with their base versions/ids), a disconnect drops them — the user's 10 minutes of work vanishes. The fix: the client queues local ops while offline (persisted locally), and on reconnect replays them to the server, which merges them via the chosen OT/CRDT mechanism against whatever changed meanwhile — so neither the returning user's edits nor the others' are lost (convergence handles the merge). This is exactly where CRDTs are strong (offline-first merges are their native case). Also handle: optimistic local apply + server reordering (the user might briefly see their op move as the authoritative order arrives), and presence cleanup (a disconnected user's cursor must disappear). Treating the connection as always-on, or not buffering offline ops, is the lost-work trap.

Sub-problem 4 — Persistence & history

The document must be durable, restorable to any past version (history/undo), and loadable fast for a new joiner. Store the op log (the sequence of operations — the source of truth, enabling history and replay) plus periodic snapshots (a materialized document state at a point in time) so loading doesn't require replaying from the beginning.

The failure that happens when you get this wrong — unbounded growth / slow loads The op log grows forever, and CRDT documents accumulate tombstones (deleted characters linger as metadata). Without management, loading a long-lived document means replaying millions of ops (slow joins) and carrying ever-growing metadata (Scenario 2's "CRDTs are heavy"). The fix: take periodic snapshots so a new editor loads "snapshot + ops since" instead of the full history; compact/garbage-collect old ops and CRDT tombstones once they're safely behind a snapshot all clients have seen; and keep the full op log (or version checkpoints) separately for history/restore if you need long-term undo. The tension — keep history vs bound size — is resolved by snapshotting + GC, not by keeping everything forever in the live structure.

The rule in one linePick the convergence model first: OT (server-coordinated, light, hard to prove), pure CRDT (P2P/offline-first, convergent, heavy), or — if you have a trusted server, which you usually do — a server-authoritative CRDT-inspired LWW scheme (Figma) that's simplest. Then layer optimistic local apply + WebSocket sync + offline op buffering for UX, and snapshots + GC for persistence. There's a real editors-per-doc ceiling — design for a target (≈100–200), not infinity.

The takeawayManaged real-time platforms (Liveblocks, PartyKit, Ably) and CRDT libraries (Yjs, Automerge) let you adopt the hard convergence machinery without building it — especially the server-authoritative CRDT-inspired path. They handle the WebSocket fan-out, presence, and merge; you decide the document model and persistence/GC strategy. For a Google-Docs-scale custom build, OT + a sharded sync-server tier is the classic route; for most products, a CRDT library + a managed connection layer is far less risky.

Famous implementations

Same facets — approach, mechanism, and the failure to watch.

Google Docs (OT)

  • Approach. Operational Transformation with a central server ordering ops.
  • Mechanism. Clients send ops + revision; server transforms against concurrent ops and broadcasts; optimistic local apply.
  • The failure to watch. Transform-function correctness is genuinely hard; the server is required for ordering (no true P2P).

Figma (server-authoritative, CRDT-inspired)

  • Approach. A central server is the authority; document objects use property-level last-writer-wins registers with fractional indexing for ordering — CRDT ideas, simplified by the server.
  • Mechanism. Clients send changes; server resolves per-property LWW and broadcasts; scales to ~200 concurrent editors.
  • The failure to watch. LWW means a true simultaneous edit to the same property has one winner (fine for design objects, not for a counter); relies on the trusted server.

Yjs / Automerge (CRDT)

  • Approach. Pure CRDT sequence types; convergent with or without a central server.
  • Mechanism. Stable per-character ids, commutative merge; great for offline-first and P2P.
  • The failure to watch. Metadata/tombstone growth on long-lived docs — needs snapshotting + GC (Scenario's sub-problem 4).

Build it yourself

Work each before opening the approach.

Make two concurrent same-spot inserts converge

"Hello world"; A inserts "X" at index 5 and B inserts "Y" at index 5 concurrently. Show how OT and how a CRDT each make both clients end up identical.

Reveal the approach

OT: the server picks an order (say A then B). A's op applies as "insert X at 5". B's op was written against the pre-A document, so the server transforms it against A's op: since A inserted one char at ≤ B's position, B's "insert at 5" becomes "insert at 6". Result on everyone: "HelloXY world" — both clients converge because the same transformed ops apply in the same order. CRDT: there are no numeric positions — each existing character has a stable id, and a new character gets a fractional id between two neighbors. A's "X" and B's "Y" each get an id between the ids of "o"(4) and " "(5); when they tie on the same gap, a deterministic tiebreak (e.g. by author id) orders them consistently — so every replica independently computes the same order ("HelloXY world" or "HelloYX world", but the same everywhere) with no transform and no server coordination. Both converge; OT does it by rewriting positions, CRDT by never using positions.

Sync 10 minutes of offline edits without loss

A user made 50 edits offline while others changed the doc. Design the client + server behavior so nothing is lost on reconnect.

Reveal the approach

While offline, the client applies edits locally (instant UX) and durably queues each op (persisted to local storage, with the version/ids it was based on) — so a crash or refresh doesn't lose them. On reconnect, the client replays its queued ops to the server in order; the server merges each via the chosen OT/CRDT mechanism against everything that changed meanwhile, and broadcasts the results. Because the convergence algorithm guarantees no lost edits, both the returning user's 50 edits and the others' concurrent edits all survive and everyone converges (Scenario 3). CRDTs make this especially clean (offline merge is their native case — ops commute regardless of when they arrive). The client also reconciles its optimistic local state with the authoritative order (it may briefly see its edits shift). The essentials: persist offline ops locally, replay on reconnect, let convergence merge — never assume the connection was up, and never discard unsynced ops.

Design-it-yourself: the open-ended ones

  • Choose the convergence model for your product. A design tool (Figma-like) vs a P2P offline notes app — pick OT / pure CRDT / server-authoritative for each and justify. → CRDTs.
  • Scale past one sync server. Route millions of docs across a sync-server tier so each doc has one ordering authority; handle a server's failure. → Coordination, Sharding.
  • Add rich history / version restore. Design op-log + snapshot storage so a user can restore any past version and undo across sessions, without unbounded growth. → Event sourcing.
