Designing a Distributed Message Queue

Build a Kafka/SQS-class system from scratch — the partitioned append-only log, replication and ISR, delivery semantics, ordering, and the cost of every guarantee.

Requirements and back-of-envelope

Start by pinning down what kind of queue this is. The "design a message queue" problem hides at least three very different products. Force the requirements out before drawing boxes.

Functional

  • Produce: append messages to a named topic (or queue), keyed or unkeyed.
  • Consume: one or many independent consumers read messages; support fan-out (pub/sub) and competing consumers (work queue).
  • Durability: an acknowledged message survives broker crashes and a configurable number of node failures.
  • Ordering: at least per-key or per-partition ordering.
  • Retention: keep messages for a time/size window even after delivery (the "replayable log" requirement), or delete on ack (the "transient queue" requirement).

Non-functional & sizing

Suppose 1 M messages/sec, average 1 KB each → 1 GB/s write throughput, ~86 TB/day before replication. With replication factor 3 that is ~260 TB/day of disk and ~2 GB/s of inter-broker replication network (the leader copies the 1 GB/s ingress to RF−1 = 2 followers). No single machine handles this, so the system must partition and the design conversation is really about partitioning + replication, not about a clever in-memory ring buffer.

Key insightThe single most important architectural fork: is this a transient queue (delete-on-consume, SQS/RabbitMQ-style) or a durable replayable log (Kafka/Kinesis/Pulsar-style)? Almost every later trade-off — offset tracking, ordering, redelivery, exactly-once — follows from this choice. State it explicitly at minute one. See Queues and Kafka and Log-based Messaging.

Two models: broker-managed queue vs. log

DimensionBroker-managed queue (SQS, RabbitMQ)Append-only log (Kafka, Kinesis)
State of "read"Broker tracks per-message acks; deletes on ackConsumer tracks an offset; broker is dumb
RedeliveryBroker redelivers un-acked messages after visibility timeoutConsumer rewinds its offset
Fan-outNeeds N copies / N queuesFree — N consumer groups read the same log
OrderingHard at scale (per-message acks reorder)Strict within a partition by construction
Throughput ceilingPer-message bookkeeping is the bottleneckSequential I/O; very high
Replay / time-travelNo (message gone after ack)Yes (rewind to any retained offset)

The log model wins for high-throughput, multi-consumer, replayable workloads, and is what most "Kafka clone" designs require. The queue model is simpler and better when you need per-message visibility timeouts, dead-letter handling, and selective ack/nack. A principal answer names both and justifies picking one. We design the log model below because it is strictly harder and subsumes the queue model (a queue is a log with retention = "until acked" and a single consumer group).

The append-only partitioned log

A topic is split into partitions; each partition is an ordered, immutable, append-only sequence of records. Partitions are the unit of parallelism, ordering, and placement. A message's position in a partition is its offset — a monotonically increasing 64-bit integer assigned by the leader.

Why append-only is fast

Writes are sequential appends to the end of a file, which a modern disk (even spinning rust, certainly NVMe) handles at hundreds of MB/s to GB/s because there are no random seeks. Reads are also sequential and ride the OS page cache, so a recently-written record is often served from RAM without a disk hit. Kafka deliberately keeps data in the kernel page cache and uses zero-copy (sendfile()) to move bytes from page cache straight to the socket, bypassing user space. This is the same insight behind LSM-tree storage — turn random writes into sequential ones. See Storage Engines: LSM vs B-Tree.

On-disk layout

Each partition is a directory of segment files (e.g. 1 GB each). The active segment is appended to; older segments are immutable. Per segment we keep:

  • .log — the records, framed with length + CRC + offset + timestamp.
  • .index — a sparse offset→byte-position index (every few KB) for O(log n) lookup, then a short scan.
  • .timeindex — timestamp→offset for time-based seeks and retention.

Retention deletes whole old segments, which is O(1) and never fragments the active write path.

partition-0/
  00000000000000000000.log     # base offset 0
  00000000000000000000.index
  00000000000000524288.log     # base offset 524288 (new segment)
  00000000000000524288.index
  ...active segment appended here...

Partition placement and routing

Producers route a record to a partition by hash(key) % numPartitions (keyed) or round-robin (unkeyed). Keyed routing is what gives you per-key ordering: all records for a customer/user land in one partition and are consumed in order. The number of partitions is a capacity knob set at topic creation; growing it later breaks key–partition affinity, so size for peak. See Sharding and Consistent Hashing.

Pitfallhash(key) % N is not consistent hashing — changing N reshuffles almost every key. Repartitioning a live topic is a migration, not a config tweak. Either over-provision partitions up front or use a scheme (like Pulsar's bundle splitting) that decouples logical topics from physical placement.

Producer / consumer APIs and offsets

Producer

send(topic, key, value, {acks: "all", maxInFlight: 5})
  → resolves with (partition, offset) once acked

// Batching: producer accumulates records per partition and
// flushes when batch.size OR linger.ms is hit — amortizes
// per-request overhead and is the single biggest throughput lever.

The producer talks to the partition leader only. acks controls durability vs. latency (covered below). maxInFlight > 1 pipelines requests for throughput but can reorder on retry unless idempotence is enabled.

Consumer — pull, not push

Consumers pull via fetch(partition, offset, maxBytes). Pull (Kafka, Kinesis) lets a slow consumer set its own pace and naturally batches; push (RabbitMQ, classic SQS long-poll is a hybrid) needs broker-side flow control to avoid overwhelming consumers. Pull is the standard answer for a high-throughput log because it makes backpressure trivial — a busy consumer simply fetches less often.

Offset tracking

The consumer's read position is just an integer per partition. Where to store it is a real design decision:

  • In the broker (committed offsets): Kafka stores committed offsets in an internal compacted topic __consumer_offsets, keyed by (group, topic, partition). Survives consumer restarts.
  • In the consumer / external store: the consumer owns the offset and commits it transactionally with its output (the foundation of exactly-once into a database).

The commit cadence dictates delivery semantics: commit after processing → at-least-once; commit before processing → at-most-once.

Consumer groups and rebalancing

A consumer group is the abstraction that delivers both work-queue and pub/sub semantics from one log:

  • Each partition is assigned to exactly one consumer within a group → competing-consumer work queue, scales by adding consumers up to the partition count.
  • Multiple groups each get the full stream independently → pub/sub fan-out.

The partition count is therefore the maximum parallelism of any single group. Ten partitions → at most ten useful consumers; the eleventh sits idle.

Rebalancing

A group coordinator (a broker) tracks membership via heartbeats. When a consumer joins, leaves, or its session.timeout elapses, the coordinator triggers a rebalance — partitions are reassigned across the surviving members.

PitfallNaive ("stop-the-world" / eager) rebalancing revokes all partitions from all consumers and reassigns from scratch — the whole group stalls. With large groups and frequent scaling this is a real outage class. Modern designs use cooperative / incremental rebalancing (only the moving partitions pause) and static membership (a restarting pod keeps its assignment, skipping the rebalance entirely). Mentioning this distinguishes a senior answer.

This is itself a consensus/coordination problem — group membership and leader/coordinator election sit on the same primitives discussed in Leader Election and Coordination and Consensus and Raft.

Replication, ISR and durability

Each partition has a leader and RF−1 followers on different brokers. Producers and consumers talk only to the leader; followers replicate by fetching from it — this is leader-based replication, the same model analyzed in Replication Deep Dive and Redundancy and Replication.

In-Sync Replicas (ISR)

The ISR is the set of replicas caught up to the leader within a lag bound (replica.lag.time.max.ms). The leader advances the high-water mark (the highest offset a consumer may read) only once all ISR members have the record. A follower that falls behind is ejected from the ISR; it must re-catch-up before rejoining.

The durability knobs

KnobEffectCost
acks=0Fire-and-forget, no ackLose unflushed data on crash
acks=1Leader persists, acks before replicatingLose data if leader dies before followers catch up
acks=all + min.insync.replicas=2Ack only after the full ISR has itHigher latency; rejects writes if ISR shrinks below 2
Trade-offacks=all alone is not durable if min.insync.replicas=1 — a single in-sync leader satisfies "all", and losing it loses data. The durable configuration is acks=all plus min.insync.replicas≥2 on RF=3, which tolerates one node loss with zero data loss but rejects producer writes (chooses availability-loss over data-loss) when two replicas are down. That is a deliberate CP choice — see CAP Theorem and Consistency Models and PACELC.

Leader failover and unclean elections

When a leader dies, the coordinator elects a new leader from the ISR — safe, because every ISR member has all committed records. If the ISR is empty you face unclean leader election: promote an out-of-sync replica (available but loses data) or block until an ISR member returns (consistent but unavailable). The default is to block. Older Kafka stored this metadata in ZooKeeper; KRaft now runs a built-in Raft quorum for the metadata log, removing the external dependency.

Delivery semantics: at-least / at-most / exactly-once

There are three classic guarantees, and the honest framing is that true exactly-once delivery over an unreliable network is impossible — what systems deliver is exactly-once processing built from at-least-once delivery + idempotence or transactions.

At-most-once

Commit the offset before processing. A crash mid-process loses the message. Acceptable for metrics/telemetry where loss < duplication cost.

At-least-once (the default, sane choice)

Commit the offset after processing. A crash after processing but before commit replays the message → duplicates. The consumer must be idempotent (dedupe by message ID, or use upserts keyed by a natural/idempotency key).

Exactly-once processing

Achieved by one of two mechanisms:

  1. Idempotent producer: each producer gets a PID; each record carries a per-partition sequence number. The broker rejects a record whose sequence it has already seen, eliminating duplicates from producer retries (which are the common source of dupes).
  2. Transactions / atomic offset commit: the consume–process–produce–commit cycle is wrapped in a transaction so that the output writes and the consumed-offset commit either all happen or none do. Kafka does this with a transaction coordinator and a two-phase commit over the partitions plus the offsets topic. The downstream consumer must read with isolation.level=read_committed to skip aborted records.

The other route — effectively-once into an external store — commits the offset in the same database transaction as the side effect, or writes with an idempotency key so replays are no-ops. This is the realistic pattern in most services and ties into the Stream Processing and CDC outbox/transactional-messaging discussion.

Pitfall"Exactly-once" is a marketing-adjacent term. It only holds inside the system's transactional boundary. The moment an effect leaves that boundary — an email sent, a charge made, an HTTP call to a third party — you are back to at-least-once and must make that effect idempotent yourself. Say this out loud; it is the crux.

Ordering guarantees and their cost

A log gives total order within a partition and no order across partitions. Every ordering requirement maps onto this:

  • Global total order: requires a single partition → throughput capped at one leader, no parallel consumption. Almost never worth it; push ordering down to a key instead.
  • Per-key order (the usual real requirement): route by key so each key lands in one partition. Order is preserved per key while different keys parallelize across partitions. This is the design sweet spot.
  • No order: round-robin for maximum spread.

Ordering is fragile at three points: (1) producer retries with maxInFlight>1 can reorder — fix with the idempotent producer, which preserves order even with pipelining; (2) repartitioning breaks key affinity; (3) a multi-threaded consumer that processes one partition's records concurrently destroys the order the log gave you — keep per-partition processing single-threaded (or key-affine) if order matters.

Trade-offOrdering and parallelism are in direct tension. More partitions = more parallelism but a coarser ordering domain. The principal move is to choose the narrowest ordering scope the business actually needs (usually per-entity, not global) and partition by that key. Ordering is also why logical clocks matter when you reconstruct cross-partition causality — see Time, Clocks and Ordering.

Retention, compaction and backpressure

Retention

Time- or size-based retention (e.g. 7 days, 1 TB) deletes whole old segments. Because consumers track their own offsets and the broker is stateless about delivery, retention is fully decoupled from consumption — that is what makes replay and adding a new consumer group "from the beginning" possible.

Log compaction

For changelog/state topics you want "keep the latest value per key forever, not the full history." Compaction rewrites segments keeping only the last record per key (a tombstone deletes a key). This turns the log into a durable, replayable materialized view — the backbone of event-driven architectures, event sourcing, and Kafka's __consumer_offsets itself. It is conceptually identical to LSM-tree compaction (Storage Engines).

Backpressure and failure handling

  • Slow consumers: with pull-based fetch, a lagging consumer just falls behind (visible as growing consumer lag, the key SLO to alarm on). It catches up later or is replaced — the broker is unaffected. Contrast with push systems that need credit-based flow control.
  • Producer overload: bound the producer's in-flight buffer; when full, either block the caller (backpressure upstream) or drop with an error. Never silently buffer unbounded — that just moves the OOM.
  • Poison messages: a record that always fails processing stalls its partition (you cannot advance the offset). Use a retry policy with a bounded count then route to a dead-letter topic/queue; never block the partition forever.
  • Consumer crash: session timeout → rebalance → another consumer resumes from the last committed offset (hence at-least-once and the idempotency requirement).
Key insightThe whole system's elegance comes from one decision: make the broker dumb and the consumer smart. The broker only appends and serves byte ranges; consumers own offsets, ordering-per-key, dedupe, and retry. This is what lets a log scale to GB/s, support unlimited replay, and add consumers for free — at the cost of pushing delivery-semantics complexity into clients.

Principal-level mastery check

What strong engineers doThey state the queue-vs-log fork in the first two minutes and justify the choice from the requirements, not from familiarity with Kafka. They derive partition count from throughput and the desired consumer parallelism, and they connect every guarantee to a concrete config: per-key ordering → key-based routing; zero data loss → acks=all + min.insync.replicas≥2; exactly-once → idempotent producer + transactional offset commit. They are honest that "exactly-once" is exactly-once processing within a transactional boundary and degrades to at-least-once at any external side effect, so consumers must be idempotent. They name the real failure modes — stop-the-world rebalances, unclean leader election, poison messages stalling a partition, repartitioning breaking key affinity — and have a mitigation for each. They quantify with consumer lag as the operational SLO.
  • You require strict ordering of all events for a given account but also 1 M events/sec across accounts. How do you get both, and what is your maximum throughput per account?
  • Walk me through exactly what happens to an in-flight, acknowledged message when the partition leader's broker is hard-killed. Under what config do you lose it, and under what config don't you?
  • A consumer in the group is stuck on a poison message. What is the blast radius, how do you detect it, and how do you recover without losing or reordering the good messages behind it?
  • You need exactly-once delivery of a payment charge to a third-party API that has no idempotency support. Is that achievable? What do you build on the message-queue side and what must live elsewhere?
  • Traffic 5x'd and you must double the partitions on a live, keyed topic. What breaks, and how do you migrate without violating per-key ordering for in-flight keys?
  • Compare storing committed offsets in the broker vs. in the consumer's own datastore. Which one makes exactly-once-into-a-database possible, and why?
