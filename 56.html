Designing a Distributed Message Queue

Producers need to hand off work to consumers without waiting, survive bursts, and not lose messages if a consumer crashes — across many machines. We design a durable, partitioned message queue (think Kafka/SQS from scratch): how messages are stored and ordered, how delivery is guaranteed, and the two hardest realities — you can't have exactly-once delivery, and one bad message can jam everything.

The crux Durable, scalable async hand-off that decouples producers from consumers — in exchange for at-least-once delivery (duplicates), ordering only within a partition, and poison-message handling. A queue lets producers fire-and-forget and absorbs bursts so consumers process at their own pace. You pay by accepting duplicates (and making consumers idempotent), getting order only per-partition, and building dead-letter handling so one unprocessable message can't block the rest.

1. Requirements

Functional: producers send messages to a named queue/topic; consumers receive and ack them; messages are durable (survive broker restart); support multiple consumers and (for a log) multiple independent consumer groups; configurable retention.

Non-functional: high throughput (millions of messages/sec), durable (no acknowledged message lost), highly available (survive a broker failure), scalable (add brokers/partitions for more capacity), low end-to-end latency.

2. Capacity estimate

Say 1,000,000 messages/sec, average message 1 KB, retained 7 days. Work it:

  • Write throughput: 1M msg/s × 1 KB = 1 GB/sec of writes. A single broker disk can't take that, so we must partition — spread across, say, 100 partitions = 10 MB/s each (easy for sequential disk writes). Consequence: partitioning is mandatory, which is what limits ordering to per-partition.
  • Storage: 1 GB/s × 86,400 s/day × 7 days ≈ ~600 TB (before replication). With 3× replication ≈ 1.8 PB — spread across the broker fleet. Consequence: retention × throughput drives your storage bill; tune retention deliberately.
  • Replication bandwidth: every write is copied to 2 followers → 2 GB/s of inter-broker replication traffic on top of the 1 GB/s ingest. Consequence: the network, not just the disk, is a capacity constraint.

3. The API

send(topic, key, payload)            // key decides the partition (ordering scope)
receive(group, maxMessages) -> [msg] // pull a batch for a consumer group
ack(group, offset)                   // mark processed up to this offset

A pull model (consumers ask for messages) lets each consumer go at its own pace and is naturally back-pressured — a slow consumer just pulls less, rather than being overwhelmed by a push. The key on send selects the partition (so same-key messages stay ordered), and consumers track an offset (how far they've processed) that they ack.

4. High-level design

A topic is split into partitions, each an independent append-only log on a broker. Producers append to the tail of a partition (chosen by key hash); each message gets a monotonically increasing offset within that partition. Consumers in a group divide the partitions among themselves (each partition to one consumer) and read forward, committing offsets. Each partition is replicated to other brokers (one leader, N followers) for durability and failover.

Reading doesn't delete (in the log model) — messages stay until retention expires, so many consumer groups read independently and you can replay. (A traditional queue like SQS instead deletes on ack and uses a visibility timeout — we contrast both.)

5. Storage model — append-only segments

Each partition log is stored as a sequence of segment files on disk, written sequentially (sequential disk I/O is dramatically faster than random — the key to high throughput). An index maps offset → file position for fast seeks. Retention deletes whole old segments (or compacts them to keep the latest value per key). The OS page cache serves recent reads from memory; the broker can even use zero-copy to send bytes straight from page cache to the network.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the acknowledged message that vanished A producer sends a message and the broker confirms "received" as soon as the leader has it — before any replica copies it. The leader broker then crashes. Where did the "successfully sent" message go, and what durability setting would have prevented it — at what cost?
Scenario 2 — the payment processed twice A consumer reads "charge $50," charges the card, then crashes before acking. On restart it re-reads the same message and charges again. Which delivery guarantee causes this, why is "exactly-once delivery" not the fix, and what is?
Scenario 3 — one bad message jams the whole queue (real-pattern incident) A malformed message can't be processed. The consumer retries it — and because it's an ordered partition, retrying that one message blocks every message behind it. (A real payment processor retried one bad message 10,000+ times over 6 hours, halting all payments.) How do you keep one unprocessable message from stopping the line?

Scenario 1 is durability. Scenario 2 is delivery semantics. Scenario 3 is poison messages. Each maps to a sub-problem.

The four hard sub-problems

Walking the design was the easy part. These four decide whether the queue is trustworthy — each with a nested failure.

Sub-problem 1 — Durability: how (and when) a message is safe

A message is durable only once it's on stable storage on enough replicas. Each partition has a leader and followers; the producer's durability setting decides when a send is "successful": leader-only (fast, riskier) or all in-sync replicas confirmed (safe, slower). This is the synchronous-vs-asynchronous replication knob, expressed for a queue.

The failure that happens when you get this wrong — ack-before-replicate loss This is Scenario 1. If the broker acks the producer as soon as the leader has the message (before replication), a leader crash loses it — the producer was told "sent" but no surviving replica has it. Requiring acknowledgment from a quorum of in-sync replicas (e.g. leader + 1 follower, with min.insync.replicas=2) guarantees an acknowledged message survives a single broker loss, at the cost of higher write latency (waiting for replicas). The trade is durability vs latency — for a payment queue you wait; for click telemetry you might not. (Same quorum-overlap logic as consensus.)

Sub-problem 2 — Delivery semantics

Three possible guarantees, set by when you ack relative to processing: at-most-once (ack before processing → a crash loses the message; rare), at-least-once (ack after processing → a crash re-delivers; the common default), and exactly-once. The crucial truth: exactly-once delivery across a network is impossible — what you actually achieve is exactly-once effect.

The failure that happens when you get this wrong — chasing exactly-once delivery This is Scenario 2. At-least-once (the sensible default — never lose a message) means a crash between processing and ack causes redelivery → duplicates → a double charge. The wrong fix is to demand "exactly-once delivery," which is provably impossible over an unreliable network. The right fix is idempotent consumers: derive a stable idempotency/dedup key from the message and make reprocessing a no-op (the payment system records executed keys and returns the original result on a repeat). For Kafka-to-Kafka pipelines, Kafka's transactions give exactly-once within Kafka — but an external side-effect (a card charge) still needs idempotency at that system. Build for at-least-once + idempotency = exactly-once effect.

Sub-problem 3 — Ordering vs consumer scaling

You scale consumers by adding partitions (each partition is consumed by one consumer in a group, so parallelism caps at the partition count). But ordering is guaranteed only within a partition. So there's an inherent tension: to keep related messages ordered they must share a partition (same key) — which limits how much you can parallelize them.

The failure that happens when you get this wrong — the order/parallelism conflict This is the design tension. If you demand global total order, you need a single partition — one consumer, no parallelism, a throughput ceiling. If you partition for parallelism, messages in different partitions have no order relative to each other. The resolution: order only what must be ordered, via the partition key. Example: key by user_id so each user's events stay in order (created→paid→shipped), while different users spread across partitions for parallelism. Pick a high-cardinality key to avoid a hot partition (one key dominating one partition). You almost never need global order — you need per-entity order, which the key gives you cheaply.

Sub-problem 4 — Poison messages & visibility

A message that can never be processed (malformed, references deleted data) must not block the queue. Two mechanisms: a visibility timeout (in queue-model brokers like SQS, a received-but-unacked message becomes invisible for a window, then reappears for retry if not acked — so a crashed consumer's message isn't lost) and a dead-letter queue (DLQ) for messages that fail too many times.

The failure that happens when you get this wrong — the jammed queue This is Scenario 3. A poison message that the consumer keeps failing on — especially in an ordered partition where it blocks everything behind it — can halt the whole pipeline (the real payment processor: one bad message, 10,000+ retries, 6 hours of stopped payments). The fix: bounded retries with exponential backoff + jitter, and after a max-attempts cap, move the message to a DLQ (preserving it for inspection/replay) and advance past it so the line keeps moving. Classify errors — retry transient failures (timeouts), but send permanent ones (validation/schema) straight to the DLQ; retrying them is pointless. Alert on DLQ depth so failures aren't silently swallowed. (Covered broadly in Event-Driven Architecture.)

At a glance — log vs traditional queue

FacetLog (Kafka-style)Traditional queue (SQS/RabbitMQ)
After ackRetained (replayable)Deleted
ConsumersMany independent groupsUsually one (work distribution)
OrderingPer-partitionOften none under concurrency (FIFO mode for order)
Failure handlingOffsets + retention; replayVisibility timeout + DLQ
Best forEvent streaming, replay, fan-outTask queues, decoupling work
The rule in one lineBuild a partitioned append-only log for throughput + replay; make writes durable via quorum acks; accept at-least-once + idempotent consumers (exactly-once delivery is impossible); order per-key not globally; and protect the line with retry + DLQ. Choose the log for streaming/fan-out, a traditional queue for simple work distribution (see Choosing a Message Broker).

Self-check — can you reconstruct the scenarios?

You read the answers above. Close the page and rebuild them.

  • Scenario 1 (vanished message): Why does leader-only ack lose data, and what does quorum-ack buy and cost?
  • Scenario 2 (double charge): Why is exactly-once delivery impossible, and how do idempotent consumers give exactly-once effect?
  • Scenario 3 (jammed queue): How do retry caps + a DLQ keep one poison message from blocking the line?
  • Why is ordering only per-partition, and how does the key resolve order-vs-parallelism?
  • Why is sequential append-only storage the key to throughput?

The shape to remember: 4 sub-problems — durability (quorum ack), delivery (at-least-once + idempotency), order vs scale (partition key), poison messages (retry + DLQ).

Famous implementations

Same facets — model, durability/ordering, and the failure to watch.

Kafka

  • Model. Partitioned, replicated commit log; pull consumers with committed offsets; retained & replayable.
  • Durability/ordering. acks=all + ISR for durability; per-partition order; transactions for exactly-once within Kafka.
  • The failure to watch. Rebalance storms on big consumer groups; hot partitions; under-replication on broker loss.

Amazon SQS

  • Model. Fully-managed queue; messages deleted on ack; visibility timeout for in-flight messages.
  • Durability/ordering. Replicated across AZs; standard = best-effort order + at-least-once; FIFO mode = ordering + dedup.
  • The failure to watch. Standard SQS can duplicate and reorder (design idempotent consumers); visibility-timeout tuning (too short → double-processing).

RabbitMQ

  • Model. Traditional broker with flexible routing (exchanges/bindings); push or pull; messages removed on ack.
  • Durability/ordering. Durable queues + publisher confirms; per-queue ordering; mirrored/quorum queues for HA.
  • The failure to watch. Throughput lower than a log; a backed-up queue (slow consumers) consumes broker memory — needs flow control / DLQ.

Build it yourself

Work each before opening the approach.

Make a consumer exactly-once-effective

Your consumer reads "charge $50" and calls a payment API; at-least-once delivery means it may re-read after a crash. Design so the card is charged at most once.

Reveal the approach

Don't rely on broker exactly-once — the side-effect is external. Derive a stable idempotency key from the message (e.g. payment_id or topic-partition-offset) and pass it to the payment API; the payment system records executed keys and, on a repeat, returns the original result instead of charging again. Commit/ack the message only after the charge call succeeds. Now redelivery resends the same key → recognized → no second charge. The pattern is at-least-once delivery + idempotent side-effect = exactly-once effect (Scenario 2). See Payment System for the ledger side.

Choose partitioning for order + scale

You process order-lifecycle events (created, paid, shipped) at 500k/sec and must process each order's events in order while spreading load. Design the partitioning.

Reveal the approach

Key by order_id: all events for one order hash to the same partition, preserving their order; different orders spread across partitions for parallelism. Choose partition count ≥ peak consumer count with headroom (a group can't have more active consumers than partitions) — e.g. 256 partitions for ~128 consumers. Avoid low-cardinality keys (status, region) that create a hot partition. You get per-order order (which is all you need) without paying for global order (which would force one partition and cap throughput). This is the order-vs-scale resolution (Scenario's sub-problem 3) in practice.

Design-it-yourself: the open-ended ones

  • Add delayed/scheduled delivery. Support "deliver this message in 1 hour." Design it on top of the log/queue without scanning. → Job Scheduler.
  • Make it multi-region. Producers in two regions, consumers everywhere. Decide replication topology and ordering across regions. → Multi-Region & DR.
  • Back a transactional outbox. Design how a service reliably publishes to this queue atomically with its DB write. → Distributed Transactions (outbox).
