Conflict-free Replicated Data Types (CRDTs)

Two users edit the same document offline, then reconnect. Two replicas accept conflicting writes during a partition. Normally you'd need a coordinator or a conflict resolver to sort it out. CRDTs make the conflict impossible by construction: data types whose merge operation is mathematically guaranteed to converge to the same result no matter what order updates arrive in. No coordination, no lost writes, no central authority.

The trade-off Automatic, coordination-free convergence with no lost updates — in exchange for restricting yourself to operations that are commutative, and paying in metadata. If your merge is commutative, associative, and idempotent, replicas always converge with zero coordination. The price: you can only use data types and operations that fit that algebra, and you carry extra metadata (tombstones, version info) to make it work.

What a CRDT is

A Conflict-free Replicated Data Type is a data structure replicated across nodes that can be updated independently and concurrently, without coordination, and is mathematically guaranteed to converge to the same value on every replica once they've exchanged updates. The conflicts that anti-entropy and vector clocks only detect, CRDTs prevent — the merge can't produce divergent results because of the algebra it obeys.

The everyday use cases:

  • Collaborative editing — Google-Docs-style shared documents where everyone edits at once.
  • Offline-first / local-first apps — mobile apps that work offline and sync later without "merge conflict" dialogs.
  • AP databases — leaderless stores using CRDTs as values so concurrent writes merge instead of one being lost.
The one-line frameA CRDT trades expressiveness and metadata for guaranteed coordination-free convergence. As long as every update is commutative/associative/idempotent under merge, replicas always agree with zero locking or consensus — but you're limited to operations that fit that algebra, and you carry extra bookkeeping (unique tags, tombstones, version vectors) to enforce it.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the lost increment A "likes" counter is replicated on 3 nodes. During a partition, replica A increments it twice and replica B increments it once. They use last-write-wins to resolve. What count do they converge to, why is it wrong, and what data structure would make all three increments survive without any coordination?
Scenario 2 — add and remove race A shopping cart is a replicated set. On replica A you add "milk." Concurrently, on replica B (which already had "milk") you remove "milk." They sync. Should "milk" be in the cart or not? Why is there no universally "correct" answer, and how do different set CRDTs decide?
Scenario 3 — Figma's 200 editors (real system) Figma supports up to ~200 people editing one design file live. They evaluated pure CRDTs and rejected them as too heavyweight, and rejected Operational Transformation as too complex — then built a server-authoritative system inspired by CRDT last-writer-wins registers with fractional indexing. Why would a company building exactly the canonical CRDT use case choose not to use a textbook CRDT?

Scenario 1 is the counter CRDT. Scenario 2 is the set CRDTs and their add/remove bias. Scenario 3 is the CRDT-vs-OT engineering reality. Each maps to an idea below.

The five ideas to understand

Theory, not knobs — ideas to hold correctly, each guarded by a misconception.

Idea 1 — The math: a join-semilattice and monotonic merge

A CRDT's convergence guarantee rests on its merge function forming a join-semilattice: the merge must be commutative (A merge B = B merge A — order doesn't matter), associative (grouping doesn't matter), and idempotent (merging the same update twice is harmless). Together these mean updates can arrive in any order, be duplicated, or be re-delivered — and every replica still reaches the same least upper bound. That's the whole magic; everything else is engineering around it.

The misconception to avoid — "any sensible merge will converge" A merge that seems reasonable but isn't commutative/idempotent will diverge. "Take the bigger number" works for a grow-only counter; "subtract on remove" does not (re-delivery double-subtracts). Last-write-wins is a valid CRDT (the LWW-register) but it loses the non-winning update — convergent, but not what you want for a counter. Convergence and correctness are different goals: the algebra guarantees all replicas agree; whether they agree on the value you want depends on choosing the right CRDT.

Idea 2 — Two flavors: state-based (CvRDT) vs operation-based (CmRDT)

  • State-based (CvRDT, "convergent") — replicas send their full state; the receiver merges via the semilattice join. Robust to lost/duplicate/reordered messages (idempotent merge handles it), but sending whole state is bandwidth-heavy. Pairs naturally with gossip/anti-entropy.
  • Operation-based (CmRDT, "commutative") — replicas broadcast just the operations (e.g. "increment by 1"). Cheap on the wire, but requires reliable, exactly-once, causal delivery — because operations must be commutative and not lost or duplicated.
The misconception to avoid — "the two flavors are interchangeable" They make opposite assumptions about the network. State-based tolerates a flaky channel (gossip it; idempotent merge cleans up) but wastes bandwidth. Op-based is bandwidth-efficient but requires a reliable causal-broadcast layer underneath — if you can't guarantee exactly-once causal delivery, an op-based CRDT silently breaks. Modern designs (delta-CRDTs) blend them: ship small state deltas to get state-based robustness at op-based bandwidth.

Idea 3 — The catalog: counters, sets, registers

"CRDT" is a family, each member solving a specific shape:

  • G-Counter (grow-only) — one per-node sub-counter; value = sum; merge = element-wise max. PN-Counter = two G-Counters (increments and decrements) to allow subtraction. Solves Scenario 1.
  • Sets — G-Set (add only), 2P-Set (add + remove once, never re-add), OR-Set (observed-remove: each add carries a unique tag, remove only deletes the tags it has seen — so a concurrent add survives a remove). Solves Scenario 2.
  • Registers — LWW-Register (keep the value with the highest timestamp) or MV-Register (keep all concurrent values for the app to resolve).
The misconception to avoid — "a CRDT is one fixed thing" You choose the CRDT to match your desired conflict semantics. Scenario 1 needs a PN-Counter (all increments survive). Scenario 2 has no objectively right answer — an OR-Set with add-wins keeps "milk," a remove-wins variant drops it; you pick the bias your product wants. Picking the wrong member (an LWW-register for a counter) converges but loses data. The skill is matching the CRDT's built-in conflict resolution to your business rule.

Idea 4 — Sequence CRDTs for collaborative text

Editing text concurrently is the hard case: if two people insert at "position 5" at once, naive indices collide. Sequence CRDTs (RGA, Logoot, the engines behind Yjs/Automerge) give every character a stable, globally-unique, densely-orderable identifier (often a fractional index between its neighbors) instead of a numeric position. Inserts and deletes reference these stable IDs, so concurrent edits merge deterministically without renumbering.

The misconception to avoid — "merging text is just merging a list" Positional indices break the moment two inserts race — everyone's offsets shift differently. The fix is to stop using positions and use stable identifiers with a dense order (you can always mint a new ID between any two). The cost is metadata: every character carries an ID and deleted characters often linger as tombstones, so a long-lived document accumulates overhead — which is exactly the "too heavyweight" complaint in Scenario 3.

Idea 5 — CRDT vs Operational Transformation (OT)

OT is the older approach (it powers Google Docs): instead of conflict-free data, it transforms each operation against concurrent ones so they can apply in any order with the right result (your "insert at 5" becomes "insert at 6" because someone else inserted earlier). OT needs a central server to order operations and is famously tricky to get correct. CRDTs need no central transform — convergence is built into the data — but carry more metadata.

The misconception to avoid — "CRDTs always beat OT" This is Scenario 3. Figma — the poster child for real-time collaboration — rejected both pure CRDTs (too much memory/metadata for a huge design tree) and OT (combinatorial transform complexity), and built a server-authoritative hybrid: CRDT-inspired property-level last-writer-wins registers with fractional indexing, scaling to ~200 concurrent editors. The lesson: CRDTs are a powerful tool, not a silver bullet. A central server (which you often have anyway) lets you use the simplest CRDT-flavored scheme — LWW registers — and skip the heavyweight sequence machinery. Pure peer-to-peer CRDTs shine when there's no trusted server (true local-first / offline-first).

Cloud & managed building blocks

CRDTs show up as built-in data types and as sync libraries:

NeedManaged / library option
CRDT data types in a databaseRedis Enterprise (Active-Active / CRDB), Riak (data types), Azure Cosmos DB multi-region writes (LWW or custom conflict resolution)
Collaborative-editing sync engineYjs / Automerge (open-source libraries), Liveblocks / PartyKit (managed), Ditto (offline-first mobile)
Multi-region active-active writesDynamoDB global tables (LWW conflict resolution), Cosmos DB multi-master
The takeawayMost managed "multi-region active-active" databases resolve write conflicts with an LWW-register by default — convergent but lossy. If silently dropping the losing write is unacceptable (counters, sets), you must either choose a richer CRDT type (Redis CRDB, Riak) or supply a custom conflict resolver. Know your store's default before you trust it with mergeable data.

Famous implementations

CRDTs made concrete. Same facets — what it provides, the mechanism, and the failure to watch.

Riak / Redis (database CRDTs)

  • Provides. Server-side CRDT data types — counters, sets, maps, registers — that merge concurrent writes automatically.
  • Mechanism. State-based CRDTs reconciled via the store's anti-entropy; Redis Enterprise CRDB replicates them active-active across regions.
  • The failure to watch. Metadata growth (per-node counter entries, OR-Set tombstones) and the need to actually use the CRDT type rather than a plain value with LWW.

Automerge / Yjs (local-first libraries)

  • Provides. Client-side CRDTs for documents, lists, and text — the engines behind many offline-first and collaborative apps.
  • Mechanism. Sequence CRDTs with stable IDs; sync any-to-any (peer-to-peer or via a relay), no central transform needed.
  • The failure to watch. Document size grows with edit history and tombstones; large or long-lived docs need compaction/garbage collection.

Figma (CRDT-inspired, server-authoritative)

  • Provides. Real-time multiplayer design editing for ~200 concurrent users per file.
  • Mechanism. Property-level last-writer-wins registers with fractional indexing for ordering — CRDT ideas simplified by a central authoritative server, deliberately not full CRDTs or OT.
  • The failure to watch. The simplification relies on having a trusted server; the LWW property means a true simultaneous edit to the same property has one winner — acceptable for design objects, not for a counter.

Build it yourself

CRDTs click when you implement the merge yourself. Work each before opening the approach.

Build a G-Counter, then a PN-Counter

Design a counter replicated on N nodes that never loses an increment under concurrency. Then extend it to allow decrements. Define the state, the increment op, and the merge.

Reveal the approach

G-Counter: state is a vector of per-node counts [c_A, c_B, c_C]; node X's increment bumps only c_X; the value is the sum; merge = element-wise max. Max is commutative, associative, and idempotent, so any delivery order converges — and Scenario 1's three increments (A+2, B+1) merge to [2,1,0] = 3, correct. PN-Counter: keep two G-Counters, P (increments) and N (decrements); value = sum(P) − sum(N). You can't decrement a single max-merged counter (it'd break idempotence), so you model "subtract" as "increment the decrement counter" — turning a non-commutative op into a commutative one.

Build an OR-Set (add wins over concurrent remove)

Design a replicated set where a concurrent add and remove of the same element resolve with add winning, and re-delivery of messages is harmless.

Reveal the approach

Tag every add with a globally unique id: the element's state is the set of live tags. Add "milk" → insert (milk, tag₁). Remove "milk" → record the specific tags you've observed for milk as removed — it cannot delete a tag it hasn't seen. So in Scenario 2, B's remove only kills the tags it saw; A's concurrent add carries a new tag B never observed, which survives the merge — milk stays (add-wins). The element is "present" iff it has at least one live (added-but-not-removed) tag. Unique tags make merge idempotent (re-adding the same tag is a no-op) and give the add-wins bias; swap the rule to get remove-wins. The cost: removed tags linger as tombstones.
