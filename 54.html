Security in System Design

Security isn't a feature you bolt on at the end — it's a dimension of every design decision, and the cost of getting it wrong is measured in millions of leaked records and destroyed companies. The principal-level view isn't memorizing attacks; it's understanding the layers of defense — who you are, what you can do, how data is protected, and how you survive the inevitable breach of any single layer.

The trade-off Protecting data and access against attackers — in exchange for friction, latency, key/secret management overhead, and the discipline of assuming every layer will eventually be breached. Security buys you confidentiality, integrity, and the ability to limit damage when (not if) something is compromised. You pay with added auth steps, encryption overhead, the operational burden of managing secrets and patches, and designing as though no single control can be trusted alone.

Security as a dimension — the layers of defense

Security is a property of the whole system, woven through every layer, not a module you add. It rests on the classic CIA triad — Confidentiality (only authorized parties see data), Integrity (data isn't tampered with), Availability (it stays up — the link to resilience and rate limiting). The design question is how to defend each layer, knowing any one can fail:

  • Identity & access — prove who a caller is (authentication) and decide what they may do (authorization).
  • Data protection — encrypt data in transit and at rest so a network sniff or a stolen disk yields nothing.
  • Secrets — manage the keys/passwords/tokens that protect everything else.
  • Depth — layer controls (network, app, data) and grant least privilege, so one breach isn't game over.
  • Operations — patch known vulnerabilities, threat-model, minimize and protect PII.
The one-line frameSecurity trades friction, overhead, and operational discipline for confidentiality, integrity, and damage limitation. The mindset that separates principals: assume breach — design so that compromising any single layer (a stolen credential, a leaked key, an unpatched server) does not hand the attacker everything. Defense in depth and least privilege are how you make a breach survivable instead of catastrophic.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the checklist.

Scenario 1 — Equifax, 2017 (real breach) A critical vulnerability in Apache Struts (CVE-2017-5638, CVSS 10.0) was disclosed and patched on 7 March 2017. Equifax was told to patch it — but the patch wasn't applied. Two months later attackers exploited the still-unpatched server and exfiltrated 147 million people's data over 78 days. Worse, the breach went undetected because an expired SSL certificate had disabled their network monitoring. Which security failures compounded here, and which single discipline would have prevented the whole thing?
Scenario 2 — the API key in the GitHub repo A developer hardcodes a database password (or cloud access key) into the source to "make it work," and it gets committed to a repository. Months later the repo is exposed (or an insider/leak occurs) and the credential grants full access to production data. Where should secrets live instead, and what limits the damage when one inevitably leaks?
Scenario 3 — the service that trusts the network An internal microservice assumes "if a request reached me from inside our network, it must be legitimate" — so it does no authentication on internal calls. An attacker who breaches one machine on the network can now call every internal service freely and move laterally to the crown jewels. What security model rejects "inside the network = trusted," and how do services verify each other?

Scenario 1 is patching & operational discipline (+ defense in depth). Scenario 2 is secrets management (+ least privilege). Scenario 3 is zero-trust / defense in depth. Each maps to a layer below.

The five layers of defense

Security in a design is these five layers. Each has a named failure nested under it.

Layer 1 — Authentication vs authorization

Two distinct questions, constantly conflated. Authentication (AuthN) = who are you? (verify identity — password + MFA, a token, a certificate). Authorization (AuthZ) = what are you allowed to do? (check permissions — RBAC/ABAC, scopes). The modern stack: OAuth2 for delegated authorization (grant an app limited access without sharing your password), OIDC for authentication on top of it, and JWT as the signed token carrying identity/claims.

The failure that happens when you get this wrong — conflating AuthN and AuthZ Authenticating a user but not properly authorizing each action is one of the most common (and severe) real-world bugs — broken access control tops the OWASP list. The classic form: you verify who the user is, then trust client-supplied IDs without checking they're allowed that resource — so user A requests /orders/123 (B's order) and gets it (an IDOR — insecure direct object reference). Always enforce AuthZ server-side on every request, scoped to the authenticated principal — never assume "logged in" means "allowed," and never trust the client to scope itself.

Layer 2 — Encryption in transit and at rest

Two places data must be protected. In transit — TLS (HTTPS) encrypts data on the wire so a network eavesdropper sees ciphertext, and authenticates the server (and optionally the client, via mTLS). At rest — encrypt stored data (databases, disks, backups, object storage) so a stolen disk or leaked backup is useless without the key. Both are table stakes; the interesting decisions are key management and what threat each actually defends against.

The failure that happens when you get this wrong — plaintext exposure Skipping either leaves a gap an attacker walks through. No TLS → credentials and data sniffable on any shared network. No encryption at rest → a leaked backup, a decommissioned-but-not-wiped disk, or a compromised storage bucket exposes everything in the clear. A subtle one: encryption at rest only helps if the keys aren't stored next to the data — "encrypted with the key in the same bucket" is theater. Encrypt everywhere, manage keys separately (Layer 3), and for the most sensitive fields (passwords) don't encrypt at all — hash with a salted, slow algorithm (bcrypt/argon2) so even a full DB dump can't reveal them.

Layer 3 — Secrets management

Everything above depends on secrets — API keys, DB passwords, signing keys, TLS certs. They must never live in source code, config files in the repo, or environment variables baked into images. Instead use a dedicated secret manager / KMS (Vault, AWS Secrets Manager, KMS): secrets are stored encrypted, fetched at runtime by authenticated services, access-audited, and rotated regularly so a leaked secret has a short useful life.

The failure that happens when you get this wrong — the hardcoded secret This is Scenario 2. A credential committed to a repo (or baked into an image/config) is a time bomb: it leaks via a repo exposure, a fired employee, a misconfigured bucket, or a dependency that scrapes the filesystem — and a hardcoded secret is usually long-lived and over-privileged, so the leak is total and lasting. Fixes: store secrets in a secret manager fetched at runtime (never in code/repo — scan repos to enforce it), rotate them so a leak expires quickly, and pair with least privilege so even a leaked credential can only touch the little it needs. Short-lived, scoped, externally-managed — the opposite of hardcoded-forever.

Layer 4 — Defense in depth & least privilege

The core architectural principles. Defense in depth: layer independent controls (network firewall/segmentation, app-level auth, data encryption, monitoring) so an attacker who breaches one still faces others — no single point of total failure. Least privilege: every user, service, and credential gets the minimum access it needs and no more, so a compromised component can do limited damage. Together they assume breach and contain it.

The failure that happens when you get this wrong — the single soft layer (trusting the network) This is Scenario 3. The traditional "hard shell, soft center" model — a strong perimeter firewall but full mutual trust inside — means one breached machine lets an attacker move laterally to everything (no internal auth, over-broad access). The modern answer is zero-trust: trust no network location; every request is authenticated and authorized regardless of origin, services verify each other (mTLS), and access is least-privilege and segmented. Combined with defense in depth, a single compromise is contained — the attacker hits another locked door at every step instead of having free rein. "Inside the network = trusted" is the assumption that turns a small breach into a total one.

Layer 5 — Patching, threat modeling & PII handling

The operational layer. Patch known vulnerabilities promptly (track CVEs in your dependencies — software composition analysis). Threat-model (systematically ask "how could this be attacked?" — STRIDE, the OWASP Top 10) and design against injection, broken access control, etc. Minimize and protect PII: collect only the personal data you need, encrypt/tokenize it, restrict access, and honor regulations (GDPR, CCPA).

The failure that happens when you get this wrong — the unpatched known CVE (Equifax) This is Scenario 1, the most preventable kind of catastrophe. Equifax's breach exploited a vulnerability that had a patch available two months before the attack — the failure wasn't a clever zero-day, it was not applying a known fix. And defense in depth failed too: an expired certificate had silently disabled the monitoring that should have caught the exfiltration, so 147M records leaked over 78 days unnoticed. The lessons compound: patch known CVEs urgently (have an inventory and a process, not goodwill), and don't rely on one control — layered monitoring + least privilege would have limited the damage even with the unpatched server. Most breaches exploit known, fixable weaknesses, not exotic attacks.

Self-check — can you reconstruct the three scenarios?

You read the answers above. Close the page and rebuild them.

  • Scenario 1 (Equifax): Which failures compounded (unpatched known CVE + expired-cert monitoring gap), and which discipline would have prevented the whole thing? Why does defense in depth matter even with an unpatched server?
  • Scenario 2 (key in repo): Where should secrets live, and what (rotation + least privilege) limits the damage when one leaks?
  • Scenario 3 (trusts the network): What's wrong with "inside = trusted," and how does zero-trust + mTLS fix it?
  • AuthN vs AuthZ — the difference, and why broken access control / IDOR is so common?
  • Why hash (not encrypt) passwords, and why must encryption-at-rest keys live separately from the data?

The shape to remember: 5 layers — AuthN/AuthZ, encryption, secrets, depth + least privilege, patching/threat-model/PII — each with a nested failure; the meta-principle is assume breach.

The takeawayThe cloud provides every layer as a managed service — identity, KMS/secret managers, WAF/DDoS, vulnerability scanning, mesh-based mTLS — so you rarely build crypto or auth from scratch (and shouldn't). What the cloud can't do for you is the discipline: actually patching what the scanner flags (Equifax had the info), authorizing every request in your code, and not hardcoding the secret. The tools are necessary; the operational rigor is the part that fails.

Famous implementations

The standards made concrete. Same facets — what it is, the mechanism, and the failure to watch.

OAuth2 / OIDC / JWT

  • What it is. The standard stack for delegated auth: OAuth2 (authorization — grant an app scoped access), OIDC (authentication layered on OAuth2), JWT (the signed token).
  • Mechanism. Flows issue scoped, expiring tokens; services verify a JWT's signature locally (no issuer round-trip) and check scopes/claims per request.
  • The failure to watch. JWTs can't be revoked before expiry (keep them short-lived + use refresh tokens); never put secrets in the (base64, not encrypted) JWT payload; validate the signature and aud/exp, don't just decode.

mTLS & zero-trust

  • What it is. Mutual TLS — both sides present certificates — as the basis of service-to-service identity in a zero-trust network.
  • Mechanism. A service mesh issues and rotates per-service certs and enforces mTLS on every call, so services authenticate each other regardless of network location.
  • The failure to watch. Certificate lifecycle (expiry breaks comms — and, as Equifax showed, an expired cert elsewhere can silently disable monitoring); the mesh must automate rotation.

Secret managers / KMS

  • What it is. Dedicated systems for storing, distributing, auditing, and rotating secrets and encryption keys (Vault, AWS Secrets Manager + KMS, Cloud KMS, Key Vault).
  • Mechanism. Secrets stored encrypted; fetched at runtime by authenticated workloads; access logged; automatic rotation; envelope encryption so data keys are themselves protected by a master key in an HSM.
  • The failure to watch. Over-broad access policies (defeats least privilege); not rotating; bootstrapping the first secret (the workload's own identity) — use the platform's instance identity, not a hardcoded token.

Build it yourself

These click when you design the controls. Work each before opening the approach.

Design authentication + authorization for an API

A multi-tenant API serves users from many organizations. Design how you authenticate callers and ensure one tenant can never access another's data — including the common IDOR trap.

Reveal the approach

AuthN: users authenticate (OIDC + MFA) and receive a short-lived JWT carrying their identity, tenant/org id, and scopes; services verify the signature + exp/aud on every request (no issuer round-trip). AuthZ: on every request, derive the principal from the token (never trust a client-supplied user/tenant id) and enforce, server-side, that the requested resource belongs to the caller's tenant and that their role permits the action — e.g. GET /orders/123 must check that order 123's tenant == token's tenant, or return 404/403 (defeating IDOR, Scenario's Layer 1). Use RBAC/ABAC for the role check. Crucially, scope every data query by tenant id at the data layer too (defense in depth), so a missed app-level check still can't leak across tenants. The rule: authenticate from the token, authorize every action against the authenticated principal, never the client's claims.

Threat-model a file-upload endpoint

Users upload files that your service stores and later serves. Walk through how it could be attacked and the controls at each layer.

Reveal the approach

Ask "how could this be abused?" systematically: AuthN/AuthZ — only authenticated users upload, and only to their own space (avoid IDOR on the resulting object key). Injection/malware — validate type/size, scan for malware, never trust the filename, serve user content from a separate domain (avoid stored XSS). Resource abuse — rate-limit uploads and cap size (avoid DoS / cost attacks); use presigned URLs so bytes don't flow through (and can't overwhelm) your app. Data protection — encrypt at rest, serve private files via signed URLs with expiry, not public buckets. Secrets/keys — the signing keys live in a KMS. Defense in depth — even if one check is bypassed, tenant-scoping + monitoring + least-privilege storage credentials contain it. The method (STRIDE / OWASP) is to enumerate threats per layer rather than hope.

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Make a credential leak survivable. Assume a service's secret will leak. Design rotation + least privilege + monitoring so the blast radius is tiny and short-lived. → secret managers.
  • Adopt zero-trust for internal services. Replace "internal = trusted" with mutual authentication on every internal call. Design the identity + mTLS scheme. → Microservices & Service Mesh.
  • Handle PII for GDPR. Design data minimization, encryption/tokenization, access controls, and deletion ("right to be forgotten") for a user-data store. → Multi-Region (residency).
