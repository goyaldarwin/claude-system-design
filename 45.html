API Design: REST, gRPC, GraphQL

An API is a contract you can almost never take back — once clients depend on it, every change risks breaking someone. The three dominant styles (REST, gRPC, GraphQL) make different bets about who controls the response shape, how data is encoded, and how change is managed. The principal-level skill isn't picking one; it's designing any of them to be idempotent, paginated, and evolvable from day one.

The trade-off A stable, evolvable contract many clients can build on — in exchange for the discipline of versioning, compatibility, and never being able to silently change a shape clients rely on. A good API lets clients integrate once and keep working for years, and lets you add features freely. You pay by committing to backward compatibility, idempotency, and pagination up front — retrofitting them after launch is painful or impossible.

What an API contract is — and why it's hard to take back

An API is the published contract through which clients use your service: the operations, the request and response shapes, the error semantics. The thing that makes API design a senior concern is irreversibility: the moment external (or even internal, cross-team) clients depend on a field or behavior, you can't change it without breaking them — and you often can't force them to upgrade. So decisions you make at launch (how retries work, how results paginate, how the response is shaped) are effectively permanent.

The three dominant styles, in one line each:

  • REST — resources addressed by URLs, manipulated with HTTP verbs (GET/POST/PUT/DELETE); the server defines the response shape. Ubiquitous, cacheable, human-readable.
  • gRPC — typed remote procedure calls over HTTP/2 with Protocol Buffers (compact binary); fast, strongly-typed, great for service-to-service.
  • GraphQL — the client specifies exactly the fields it wants in one query; eliminates over/under-fetching, at the cost of server complexity.
The one-line frameAPI design trades the freedom to change things later for a contract clients can build on. The style choice matters, but the durable skill is baking in idempotency, cursor pagination, and backward-compatible evolution from the start — because retrofitting them onto a live API with real clients ranges from painful to impossible.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the timeout that charged twice A client calls POST /charges to charge a card $50. The network times out before the response arrives — the client doesn't know if it worked, so it retries. The charge actually succeeded both times. What single mechanism lets the client safely retry without double-charging, and why can't you rely on "exactly-once delivery" instead?
Scenario 2 — the page that skips and repeats rows Your API paginates with ?offset=40&limit=20. A client reads page 3, but between requests new rows are inserted near the top. Page 4 now repeats some rows from page 3 and skips others. Why does offset pagination break on changing data, and what pagination scheme is stable under inserts?
Scenario 3 — Stripe vs GitHub on breaking changes (real contrast) Stripe has kept every API version working since 2011 — a 7-year-old integration still runs — using pinned versions and internal transform layers. GitHub once shipped a change to workflow_dispatch with no warning and broke production systems that depended on the old behavior. What's the discipline that lets Stripe evolve constantly without breaking anyone, and what counts as a backward-compatible change?

Scenario 1 is idempotency. Scenario 2 is pagination. Scenario 3 is compatibility & versioning. Each maps to a decision below.

The four decisions that define an API design

Designing any API is these four choices. Each has a named failure nested under it.

Decision 1 — Which style? (REST / gRPC / GraphQL)

Match the style to the consumer and traffic shape. REST for public/web APIs and CRUD-shaped resources — cacheable, universally supported, easy to debug. gRPC for internal service-to-service calls where latency and type-safety matter — binary Protobuf over HTTP/2, with code-generated clients and streaming. GraphQL when many diverse clients (mobile, web) need different subsets of a rich, interconnected data graph and you want to avoid maintaining dozens of bespoke endpoints.

The failure that happens when you get this wrong — style/use-case mismatch Each style is wrong somewhere. GraphQL for a simple CRUD service adds a resolver layer, query-cost concerns, and caching headaches you didn't need. gRPC for a public browser API fights the web (limited browser support, not human-readable, needs a proxy). REST for a chatty mobile screen causes over-fetching (downloading fields you ignore) or under-fetching (many round-trips) — the exact pain GraphQL exists to solve. Pick by consumer: public+web → REST, internal+performance → gRPC, many clients+flexible shape → GraphQL.

Decision 2 — How do you handle retries? (idempotency)

In any networked API, clients will retry — timeouts make "did my request succeed?" unanswerable. An operation is idempotent if doing it twice has the same effect as once. GET/PUT/DELETE are naturally idempotent; POST is not (each creates/charges again). The fix is an idempotency key: the client sends a unique key with the request; the server records the key and its result, and a retry with the same key returns the original result instead of re-executing.

The failure that happens when you get this wrong — the duplicate side-effect This is Scenario 1. Without idempotency keys, a retried POST /charges charges twice — and you cannot solve this with "exactly-once delivery," which is impossible across a network. The client attaches a unique idempotency key (a UUID); the server, on first receipt, processes and stores (key → result); on any retry with that key it returns the stored result without re-charging. Stripe scopes keys to the account and remembers them for ~24h–30 days. Every mutating endpoint that has a real side-effect needs this — design it in from the start, because clients retry whether you planned for it or not.

Decision 3 — How do clients page large result sets?

You can't return a million rows in one response. Two schemes: offset pagination (?offset=N&limit=M — simple, supports jump-to-page, but unstable on changing data and slow for deep offsets) and cursor pagination (?after=<opaque-cursor> — the cursor encodes "where you left off," typically a stable sort key like an id/timestamp; stable under inserts and efficient at any depth). Cursor is the right default for feeds and large/changing datasets.

The failure that happens when you get this wrong — offset pagination drift This is Scenario 2. offset means "skip the first N rows," but "the first N" shifts when rows are inserted or deleted between page requests — so a client paging a live feed repeats and skips rows, and deep offsets force the database to scan-and-discard N rows (slow). Cursor pagination fixes both: the cursor points to a stable position ("everything after id X"), so new inserts don't shift it, and the query is an indexed range scan regardless of depth. Use cursors for anything ordered, large, or changing; reserve offset for small, static, jump-to-page admin tables.

Decision 4 — How do you evolve the API without breaking clients?

The contract must change — new fields, new behavior — without breaking existing integrations. The discipline: only make additive (backward-compatible) changes in place, and version when you must break. Backward-compatible = adding a new optional field, a new endpoint, a new optional parameter, a new enum value clients can ignore. Breaking = removing/renaming a field, changing a type, making an optional field required, changing semantics. For breaking changes, version (URL /v2, header, or pinned-version-per-account) and support old versions for a long, announced window.

The failure that happens when you get this wrong — the unannounced breaking change This is Scenario 3. GitHub's silent change to workflow_dispatch broke clients that had no warning — the cardinal API sin. Stripe's opposite approach: pinned versions + internal transform layers — each account is pinned to the API version it integrated against, and Stripe translates between the latest internal model and each pinned version, so a 2011 integration keeps working while the API evolves daily. The rules: never remove or repurpose a field clients use; add, don't mutate; if you must break, ship a new version and keep the old one alive for an announced deprecation window (GitHub now promises ≥24 months). Treat "would this break a client who ignores the change?" as the test for every modification.

The rule in one lineChoose REST for public/web CRUD, gRPC for internal performance-critical calls, GraphQL for many clients needing flexible data — then, whatever you pick, make it idempotent, cursor-paginated, and backward-compatible from day one. The style is reversible-ish; those three properties are not.

Famous implementations

The three styles made concrete. Same facets — what it is, the strength, and the failure to watch.

REST — Stripe (the gold standard)

  • What it is. Resource-oriented HTTP/JSON API; the widely-cited reference for API design.
  • Strength. Idempotency keys on all mutating calls, cursor pagination, and rock-solid versioning (pinned per account, transform layers) — every integration since 2011 still works.
  • The failure to watch. REST's inherent over/under-fetching on rich screens; Stripe mitigates with expandable fields rather than switching styles.

gRPC + Protocol Buffers

  • What it is. Typed RPC over HTTP/2 with Protobuf binary encoding; generated clients in many languages; supports streaming.
  • Strength. Low latency, small payloads, strong typing, and schema evolution via numbered fields — add fields by new number, never reuse a number, and old/new clients interoperate.
  • The failure to watch. Poor browser support (needs grpc-web/a proxy) and opacity (binary, harder to debug); reusing or renumbering a Protobuf field number silently corrupts compatibility.

GraphQL

  • What it is. A single endpoint where clients send queries specifying exactly the fields/graph they want.
  • Strength. Eliminates over/under-fetching; one flexible API serves diverse clients without bespoke endpoints; strongly-typed schema.
  • The failure to watch. The N+1 problem (a query for 100 items naively fires 100 sub-queries — fix with batching/DataLoader) and query-cost abuse (a malicious/expensive nested query — fix with depth/cost limits). Caching is harder than REST's HTTP caching.

Build it yourself

These click when you design the contract. Work each before opening the approach.

Design an idempotent POST for payments

Design POST /charges so a client can safely retry after a timeout without ever double-charging. Specify what the client sends and what the server stores and returns.

Reveal the approach

Require an Idempotency-Key header (client-generated UUID, one per logical charge attempt). On receipt, the server looks up the key: if unseen, it begins processing, records the key with a "pending" then final result atomically (e.g. a unique constraint on the key in the same transaction as the charge), and returns the result; if seen and complete, it returns the stored response without re-charging; if seen and still pending (a concurrent retry), it returns a "request in progress" or waits. Scope keys to the account and expire them after a window (Stripe: ~24h–30d). Now a timeout-driven retry with the same key is a safe no-op. Note this gives exactly-once effect via idempotency — not via delivery guarantees, which are impossible (32, 61).

Evolve a response schema without breaking clients

You need to rename a field name to full_name and split it into first_name/last_name, on a live API with thousands of integrations. Do it without breaking anyone.

Reveal the approach

Never rename in place — that's a breaking change. Instead: add the new fields (first_name, last_name, full_name) alongside the old name, populating all of them. Old clients keep reading name; new clients read the new fields. Announce name as deprecated with a long sunset window. Only after telemetry shows no client reads name (or after the announced window for a major version bump) do you remove it — in a new API version, with the old version still served. For gRPC you'd add new numbered fields and leave the old number reserved forever. The principle: additive changes are safe; removals/renames require versioning + a deprecation window (the Stripe discipline, not the GitHub mistake).

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Pick a style for a mobile + web + partner API. One backend serves a React web app, a mobile app, and third-party partners. Decide REST/gRPC/GraphQL (possibly a mix + BFF) and justify each. → Microservices.
  • Design a public API's rate limiting & auth. At the gateway, design how you authenticate, rate-limit per client, and version — the cross-cutting contract concerns. → Rate Limiter, Security.
  • Kill the GraphQL N+1. A query for a list of posts each with their author fires one author query per post. Design batching so it's one query for all authors. → Indexes.
