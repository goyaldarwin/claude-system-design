Consistency Models & PACELC

"Consistency" is the most overloaded word in distributed systems. A consistency model is a precise contract between a data store and its clients: given concurrent reads and writes across replicas, which results are legal and which are forbidden? Pick a strong model and you reason simply but pay in latency; pick a weak one and you go fast but must handle surprises. PACELC names that trade-off in both the failure and the normal case.

The trade-off A simpler mental model and fewer surprising reads — in exchange for coordination latency on every operation. The stronger the model, the more replicas must agree before a read or write returns, so the more it costs. Weaker models return faster but allow stale, out-of-order, or conflicting reads you must design around.

What a consistency model is

A consistency model is a contract: it defines, for a system with multiple replicas serving concurrent clients, exactly which read results are legal. It answers "if I write X and then read, am I guaranteed to see X? What if someone else reads at the same time — can they see an older value? Can two readers see writes in different orders?"

Models form a ladder from strongest (most constraining, most expensive) to weakest (most permissive, cheapest):

  • Linearizable — the system behaves as if there's one copy and every operation happens atomically at a single instant in real time. The gold standard, the costliest.
  • Sequential — all clients see operations in the same order, but that order need not match real time.
  • Causal — operations that are causally related (this write read that one) are seen in order by everyone; unrelated (concurrent) operations may be seen in different orders. Often the practical sweet spot.
  • Eventual — if writes stop, all replicas eventually converge. Says nothing about what you read in the meantime.
The one-line frameA consistency model trades coordination latency for predictability. Strong models make your code simple — you reason as if there's one copy — but force replicas to agree before answering. Weak models answer instantly from the nearest replica but hand you stale, reordered, or conflicting reads to cope with. The skill is choosing the weakest model your feature can tolerate.

Three scenarios — decide before you read

Don't scroll past these. Decide your answer first. If you know all three cold, skip to classifying real systems.

Scenario 1 — Jepsen vs MongoDB 4.2.6 (real finding) Kyle Kingsbury's Jepsen tested MongoDB 4.2.6 at its strongest read and write concern. Under a network partition, transactions that updated several documents were acknowledged as successful, then the writes vanished — and in some cases a read saw the result of a write that "happened later." Which consistency guarantee was being violated, and what's the difference between a system that's slow and one that lies about acknowledged writes?
Scenario 2 — "read your own write" A user updates their profile photo, the write succeeds, they reload the page — and see the old photo. The system is "eventually consistent" and their read hit a replica that hadn't caught up. Which weaker-than-strong guarantee would fix exactly this, without paying for full linearizability?
Scenario 3 — the healthy-network tax Your network has no partition — everything's fine, which is true 99.9% of the time. A strongly consistent read still takes longer than an eventually consistent one because replicas must coordinate. CAP says nothing about this case. What are you still choosing on every single read?

Scenario 1 is linearizability vs serializability — what "strong" really promises. Scenario 2 is session guarantees. Scenario 3 is PACELC's else-branch. Each maps to an idea below.

The five ideas to understand

This is theory, not knobs to configure — a set of ideas to hold correctly, each guarded by a misconception.

Idea 1 — The strength ladder is a spectrum, not a switch

Between "strong" and "eventual" lies a whole ladder (listed above). Each rung adds a constraint and a cost. Causal consistency is the most-loved middle rung: it guarantees that if operation B depends on A (someone read A before writing B), everyone sees A before B — while letting genuinely concurrent operations be reordered. It's strong enough to kill the most jarring anomalies (comments appearing before the post they reply to) yet cheap enough to stay available under partition.

The misconception to avoid — "the choice is strong vs eventual" Framing it as a binary throws away the rung you usually want. Most "we need strong consistency" requirements are really "we need causal" or "we need read-your-writes" — far cheaper guarantees that still feel correct to users. Name the exact anomaly you can't tolerate (stale read? reordered writes? lost update?) and pick the weakest rung that forbids it.

Idea 2 — Linearizability is about recency; serializability is about transactions

These two "strong" words mean different things and are often conflated:

  • Linearizability (a consistency model) — about single operations on single objects: once a write completes, every later read sees it; the system acts like one copy in real-time order.
  • Serializability (an isolation level) — about transactions grouping multiple operations: the result is equivalent to some serial order of the transactions, but that order need not match real time.
  • Strict serializability = both at once (serializable + real-time order). This is what Spanner offers and what you informally mean by "fully consistent."
The misconception to avoid — "strongest settings can't lose data" This is Scenario 1. MongoDB at max concern was supposed to give snapshot isolation; Jepsen showed it didn't — acknowledged writes disappeared and a read saw a "future" write (a retrocausal anomaly). The lesson: a model is only as strong as the implementation actually delivers, and "we set it to the strongest level" is not the same as "it is linearizable / strictly serializable." A correct strong system may be slow or unavailable under partition; it must never acknowledge a write and then lose it. Verify with tests like Jepsen, not vendor labels.

Idea 3 — Session (client-centric) guarantees

Full linearizability is global and expensive. Often you only need guarantees relative to one client's own session — far cheaper because they involve no global agreement:

  • Read-your-writes — you always see your own latest write (fixes Scenario 2).
  • Monotonic reads — you never see time go backwards (a later read won't show an older value than an earlier one).
  • Monotonic writes — your writes apply in the order you issued them.
  • Writes-follow-reads — a write you make after reading X is ordered after X everywhere.
The misconception to avoid — "clients observe one global state" This is Scenario 2. The user doesn't need the whole world to be consistent — they need their own session to make sense. Routing a client's reads to a replica that has its recent writes (sticky routing, or read-from-primary-after-write) delivers read-your-writes at a fraction of linearizability's cost. Most "consistency bugs" users actually notice are session-guarantee violations, not global ones.

Idea 4 — PACELC: the trade-off has two halves

CAP only describes the rare partition. Daniel Abadi's PACELC (2010) spells out both regimes:

Partition → choose Availability or Consistency; Else (normal operation) → choose Latency or Consistency.

The ELC leg is the part that runs 99.9% of the time. Even with a perfectly healthy network, a strongly consistent system must coordinate — a quorum, a leader round-trip, a commit-wait — before it can answer, and that's latency. Relax consistency and you answer from the nearest replica instantly.

The misconception to avoid — "CAP covers consistency choices" This is Scenario 3. Leaning on CAP is the junior tell: it's silent about the latency-vs-consistency choice you make on every request, partition or not — the cost that actually dominates your latency budget. State systems in PACELC terms: "Spanner is PC/EC — consistent under partition and normally, paying latency for it" is a far sharper statement than "Spanner is CP."

Idea 5 — What strong consistency actually costs

The price of a rung up the ladder is concrete: more replicas on the critical path of every operation. Linearizable reads can't be served from a stale local replica — they need a quorum read or a leader round-trip. This is why low-latency global apps lean on weaker models plus session guarantees, and reserve linearizability for the few operations that truly need it (account balances, inventory counts, lock acquisition).

The misconception to avoid — "just make everything strongly consistent" Blanket strong consistency turns every read into a coordination event — you've paid global-agreement latency for a "likes" counter where a few seconds of staleness is invisible. Consistency is per-operation, not per-database. Tunable stores let you dial it per request precisely so you spend the latency only where correctness demands it.

Classifying real systems (PACELC)

The two-letter class — one letter per leg — is how engineers actually describe a store:

SystemPACELC classReading
Google SpannerPC/ECStrictly serializable under partition and normally; pays TrueTime commit-wait latency to stay consistent everywhere.
DynamoDB / CassandraPA/EL (tunable)Available under partition, low-latency normally — per-request quorum knobs dial toward consistency when needed.
MongoDBPC/ECMinority primary steps down under partition; normally reads from primary for consistency (with the Jepsen caveat that the implementation must actually deliver it).
Default async-replicated SQL (read replicas)PC/ELPrimary failover under partition; normally serves fast, possibly-stale reads from replicas.
There is no "CA" classEvery networked store has a P-clause answer whether or not its designers chose one. The interesting, ever-present leg is E: what do you do on the 99.9% of requests with no partition?

Self-check — can you reconstruct the three scenarios?

You read the answers above. Close the page and rebuild them.

  • Scenario 1 (Jepsen/MongoDB): What's the difference between linearizability and serializability, and what does it mean that a system "labeled strong" lost acknowledged writes? Why is "slow" acceptable but "lost an acked write" never is?
  • Scenario 2 (old photo): Which session guarantee fixes "I don't see my own write," and how do you implement it cheaply without full linearizability?
  • Scenario 3 (healthy-network tax): What does PACELC's ELC leg add over CAP, and why does it dominate your latency budget?
  • Name the consistency ladder strongest to weakest, and the rung that's usually the practical sweet spot.
  • Why is consistency a per-operation choice, not a per-database one?

The shape to remember: 5 ideas — it's a ladder, not a switch, linearizable ≠ serializable, session guarantees are the cheap win, PACELC has two legs, strong consistency costs a round-trip — each guarded by a misconception.

The takeawayCosmos DB is the clearest teaching tool here — it makes you pick one of five named rungs of the ladder explicitly, and bills/latency-charges you accordingly. Most other systems give you the two endpoints (eventual / strong) plus a quorum dial.

Famous implementations

The models made concrete. Same facets — the model offered, the mechanism, the cost, and the failure to watch.

Strong — Google Spanner

  • Model offered. Strict serializability (linearizable + serializable) globally.
  • Mechanism. TrueTime — GPS/atomic-clock-bounded time — lets every node agree on global ordering; a transaction waits out the clock uncertainty ("commit-wait") before acknowledging, then runs Paxos per shard.
  • The cost. A few milliseconds of commit-wait on every write, plus a quorum round-trip. PC/EC.
  • The failure to watch. Latency is non-negotiable — you can't make Spanner writes faster than its consistency guarantee allows; if you don't need the guarantee, you're overpaying.

Tunable — Cassandra / DynamoDB

  • Model offered. Anything from eventual to strong, chosen per request.
  • Mechanism. The quorum rule R + W > N — if read replicas + write replicas exceed total replicas, every read overlaps the latest write. QUORUM/QUORUM = strong-ish; ONE/ONE = fast and eventual.
  • The cost. Whatever you dial — you pay latency only on the requests where you ask for consistency.
  • The failure to watch. There's no single right setting; a wrong knob silently gives you staleness or latency you didn't intend. Read-your-writes needs deliberate quorum or routing.

Causal — COPS-style stores

  • Model offered. Causal consistency — related operations ordered everywhere, concurrent ones free.
  • Mechanism. Track causal dependencies (often via version vectors) and don't expose a write until its dependencies are visible locally.
  • The cost. Metadata overhead to track dependencies; far cheaper than global coordination, and stays available under partition.
  • The failure to watch. Causal is the academic sweet spot but rare as a turnkey product — in practice you often approximate it with session guarantees in a tunable store.

Build it yourself

Models stick when you apply them. Work each before opening the approach.

Pick the weakest model per feature

For a social app, choose a consistency model for each: the account balance in its wallet, the feed of posts, the follower count, the user's own profile edits. Justify each by naming the anomaly you can't tolerate.

Reveal the approach

Wallet balance: linearizable / strict serializable — a stale or lost balance is money; pay the latency. Feed: eventual — a few seconds of staleness is invisible. Follower count: eventual — off-by-a-few is fine. Own profile edits: read-your-writes (a session guarantee) — the user must see their own change immediately, but the world doesn't need it instantly. The skill is naming the intolerable anomaly per feature and picking the cheapest rung that forbids it — not blanketing everything in "strong."

Work the quorum math

For N=3 replicas, which (R, W) give strong consistency and which give low latency? Why does R + W > N guarantee a read sees the latest write?

Reveal the approach

W=2, R=2 → R+W=4 > 3: any read quorum and any write quorum share at least one replica, so the read is guaranteed to touch a node holding the latest write — strong, but every op waits for 2 nodes. W=1, R=1 (R+W=2 ≤ 3): fastest, but read and write sets may not overlap, so reads can be stale. The overlap is the trick — the same quorum-overlap idea behind consensus, used here as a per-request consistency dial.

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Implement read-your-writes cheaply. Your app reads from async replicas (fast, stale). Design how a user always sees their own just-made write without making every read strongly consistent. → Replication.
  • Defend a "we need strong consistency" requirement. A product manager insists the whole app be strongly consistent. Decompose it feature-by-feature and show which parts genuinely need it. → CAP, Choosing a Database.
  • Resolve concurrent writes. Two replicas accept conflicting writes to the same key under an AP model. Design conflict resolution that doesn't silently lose data. → CRDTs, version vectors.
