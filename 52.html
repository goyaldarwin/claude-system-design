Deployment Strategies

Change is the leading cause of outages — most incidents start with a deploy. So the goal of a deployment strategy isn't to deploy faster; it's to make change safe: roll out gradually, detect problems while the blast radius is small, and roll back instantly. The deepest move is decoupling deploy (ship the code, dormant) from release (turn it on), so the two riskiest moments don't happen at once.

The trade-off Safe, reversible change with a small blast radius — in exchange for running multiple versions at once, deployment infrastructure, and the discipline of backward-compatible steps. Gradual rollout + instant rollback turns a risky big-bang change into a controlled, observable one. You pay by temporarily running old and new versions side by side (which forces backward compatibility, especially in the database) and by building the deploy/flag machinery.

Why deployment is a reliability problem

Steady-state systems rarely fail on their own — they fail when something changes, and the change you make most often is a deploy. So deployment strategy is really change-safety strategy, and it has three jobs: (1) limit the blast radius of a bad change (don't expose it to everyone at once), (2) detect problems automatically while exposure is small, and (3) recover instantly (roll back) when you do.

The foundational idea that enables all three: separate deploy from release. Deploying means getting the new code onto servers (it can be present but dormant); releasing means making it active for users. Decoupling them — via feature flags — means you ship code continuously and turn features on independently, gradually, and reversibly, without a redeploy.

The one-line frameA deployment strategy trades the simplicity of all-at-once and the cost of running multiple versions for a small blast radius, early detection, and instant rollback. The price — old and new versions coexisting during a rollout — forces every change (especially database changes) to be backward-compatible, which is the discipline that makes safe deploys possible at all.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the strategies.

Scenario 1 — Knight Capital, 1 August 2012 (real incident) Knight Capital deployed new trading code to its 8 servers — but the deploy missed one server, which kept running old code. Worse, the new code repurposed an old, dormant feature flag that, on the stale server, activated long-dead "test" logic. At market open that one server began firing millions of unintended orders. In 45 minutes Knight lost $440 million and was effectively destroyed. Which deployment discipline failures combined here, and how would each have prevented the disaster?
Scenario 2 — the bug that hit 100% of users instantly You deploy a new version to all servers at once. It has a subtle bug that only manifests under real production traffic. Within seconds, every user is affected, error rates spike across the board, and you're now doing an emergency rollback under maximum pressure. What rollout approach would have exposed the bug to 1% of users first, caught it automatically, and stopped before it reached everyone?
Scenario 3 — the migration that broke the old version mid-deploy Your deploy renames a database column from name to full_name. During the rolling deploy, old-version instances (still querying name) and new-version instances (querying full_name) run simultaneously — and the old instances start throwing errors because the column they need is gone. Why does a rename break a rolling deploy, and what's the multi-step pattern that changes a schema with zero downtime?

Scenario 1 is deploy discipline (and decoupling deploy/release). Scenario 2 is canary rollout. Scenario 3 is zero-downtime DB migration. Each maps to a decision below.

The four decisions that define a deploy strategy

Designing safe change is these four choices. Each has a named failure nested under it.

Decision 1 — Decouple deploy from release

Deploying code and releasing a feature are different acts with different risks — conflating them means the moment you ship is also the moment users are exposed, with no independent control. Decouple them with feature flags: deploy the new code dark (present but off), then release by flipping a flag — for 1% of users, then 10%, then all — and roll back instantly by flipping it off, no redeploy. This also enables dark launches (run new code against real traffic without showing results) and shadow traffic (mirror production requests to the new version to test it).

The failure that happens when you get this wrong — the all-at-once flip (and undisciplined deploys) This is Scenario 1. Knight Capital is the canonical catastrophe, and it stacks several failures: (a) inconsistent deploy — one of 8 servers didn't get the new code, so old and new ran together with no detection (an automated, verified deploy would have caught the missed server); (b) repurposing an old flag — reusing a dormant feature flag's name meant the new code reactivated dead logic on the stale server (never repurpose flags; remove dead ones); (c) no gradual release — it went live to the full market instantly with no canary to catch the runaway orders; (d) no kill switch — no fast way to disable the misbehaving code. Decoupling deploy from release, automated consistent deploys, flag hygiene, and a kill switch each independently would have bounded a $440M loss to a near-miss.

Decision 2 — How do you roll out? (rolling, blue-green, canary)

Three strategies for getting new code live:

  • Rolling update — replace instances in batches (some old, some new during the roll). No extra capacity, but old+new coexist (forcing compatibility) and rollback means rolling back.
  • Blue-green — stand up a full parallel "green" environment, switch all traffic at once, keep "blue" ready for instant rollback. Clean and instantly reversible, but needs double the capacity briefly.
  • Canary — route a small % of traffic to the new version, watch its metrics, and only proceed (or auto-rollback) based on what you see. The safest for catching production-only bugs early.
The failure that happens when you get this wrong — no early detection This is Scenario 2. A big-bang rollout (all instances at once) exposes every user to a bad change simultaneously — the blast radius is 100% before you have any signal, and you discover the problem from a flood of user-facing errors, mid-emergency. A canary exposes ~1% first and compares the canary's error rate / latency / business metrics against the baseline (automated canary analysis); a regression auto-halts and rolls back before most users ever see it. The principle: find the bug with 1% of users, not 100% — gradual exposure converts a potential outage into a contained blip.

Decision 3 — How do you change the database safely?

Schema changes are the hardest deploys because the database is shared state that old and new code both use during a rolling deploy. The pattern is expand-contract (parallel-change): expand (add the new schema additively — new column/table — without removing the old), migrate (backfill data; deploy code that writes both and reads new), then contract (once nothing uses the old schema, remove it) — each step backward-compatible so old and new code coexist safely.

The failure that happens when you get this wrong — the breaking migration This is Scenario 3. A destructive schema change in one step (rename/drop a column, change a type, add a NOT NULL) breaks whichever code version doesn't match — during a rolling deploy, old instances querying the old column error out the instant it's gone. Expand-contract avoids it: e.g. to rename name→full_name: (1) add full_name (keep name); (2) deploy code that writes both; backfill full_name from name; (3) deploy code that reads full_name; (4) once no code reads name, drop it. Every intermediate state is backward-compatible, so old and new versions coexist without error, and any step can be rolled back. (Same discipline as API evolution — additive, then remove later.) Also: avoid locking DDL on large tables (use online/concurrent variants) so the migration itself doesn't cause an outage.

Decision 4 — How do you undo? (rollback vs roll-forward)

When a deploy goes bad, you either roll back (revert to the previous known-good version) or roll forward (ship a fix on top). Rollback is faster and safer when available — so design for it: keep the previous version deployable, and ensure changes are backward-compatible so reverting code doesn't conflict with already-migrated data. Roll-forward is necessary when you can't roll back (e.g. an irreversible data migration already ran), which is exactly why you make migrations reversible/expand-contract.

The failure that happens when you get this wrong — stranded with no way back The worst place to be during an incident is having shipped a bad change you can't reverse — because a destructive migration already dropped the old column, or the new version wrote data the old version can't read. Now your only option is to fix-forward under maximum pressure, with users hurting the whole time. Design every deploy to be rollback-able: keep the prior version one click away (blue-green / flag flip makes rollback instant), and make data changes expand-contract so reverting code never conflicts with the data. A deploy you can't undo is a deploy you shouldn't make without extreme care — reversibility is the safety net everything else assumes.

Famous implementations

The strategies made concrete. Same facets — what it is, the mechanism, and the failure to watch.

Canary + automated analysis

  • What it is. Route a small % of traffic to the new version and decide go/no-go from its metrics (Spinnaker/Kayenta pioneered automated canary analysis).
  • Mechanism. Compare canary vs baseline on error rate, latency, and business KPIs over a window; auto-promote on pass, auto-rollback on regression.
  • The failure to watch. Too-short analysis windows or too-few canary requests miss the bug; canary must see representative traffic.

Feature flags / dark launch / shadow

  • What it is. Runtime switches that decouple release from deploy (LaunchDarkly, Unleash, in-house).
  • Mechanism. Deploy code dark; flip flags per cohort/% to release; dark-launch (execute new path, hide output) and shadow (mirror traffic) to test under real load; instant kill switch.
  • The failure to watch. Flag sprawl and stale/repurposed flags (Knight Capital) — treat flags as code with lifecycle and cleanup.

Expand-contract migrations

  • What it is. The zero-downtime schema-change pattern (parallel-change).
  • Mechanism. Add new (expand) → dual-write + backfill → switch reads → remove old (contract); every step backward-compatible.
  • The failure to watch. Skipping a step (dropping the old column too early) breaks coexisting versions; locking DDL on big tables causes its own outage — use online/concurrent operations.

Build it yourself

These click when you sequence the steps. Work each before opening the approach.

Add a NOT NULL column with zero downtime

You must add a required email_verified boolean to a 100M-row users table, while old and new code versions run during a rolling deploy. Sequence the migration so nothing breaks.

Reveal the approach

Never add a NOT NULL column with a default in one shot on a huge table (locks/old-code break). Expand-contract: (1) Add the column as nullable (fast, non-locking; old code ignores it). (2) Deploy code that writes the column on new/updated rows. (3) Backfill existing rows in batches (not one giant UPDATE that locks the table). (4) Once every row has a value and all instances write it, add the NOT NULL constraint (validate separately from the lock-heavy step where the DB supports it). (5) Old code never needed the column, so it coexisted safely throughout. Each step is backward-compatible and individually reversible — the opposite of a one-step breaking migration (Scenario 3). Use online/concurrent DDL so the schema change itself doesn't lock out traffic.

Design a canary with automated rollback

You deploy a new version of a checkout service. Design a canary rollout that catches a production-only regression at ~1% and rolls back automatically, then promotes safely if healthy.

Reveal the approach

Deploy the new version alongside the old; route ~1% of real traffic to it (the canary), 99% to the stable baseline. Run automated canary analysis: over a window (long enough for statistical signal — minutes, with enough requests), compare the canary's error rate, latency percentiles, and key business metrics (checkout success rate) against the baseline. If the canary regresses beyond thresholds, auto-halt and roll back (shift traffic back to baseline) — the bug hit ~1% of users for a few minutes, not 100% (Scenario 2 fix). If healthy, progressively shift more traffic (1%→10%→50%→100%), re-analyzing at each step, until full. Tie the gate to your SLO/error budget, and keep the baseline running until 100% so rollback stays instant. This makes the rollout self-driving and self-protecting.

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Build a flag-driven release process. Design how features ship dark, release gradually by cohort, get a kill switch, and get cleaned up — with the Knight Capital failures explicitly designed out. → Microservices.
  • Migrate a live high-traffic table. Re-shard or re-type a column on a table taking thousands of writes/sec with zero downtime. Sequence expand-contract + backfill + dual-write. → Sharding, Replication.
  • Gate deploys on the error budget. Wire your deploy pipeline so a depleted error budget automatically freezes risky releases. → SLI/SLO/SLA.
