Object Storage and CDN

Object stores and CDNs — getting large and static content off your application's critical path.

In this lesson

  • Why bytes don't belong in your app tier
  • Object storage semantics
  • Durability, consistency & the upload path
  • Storage classes, lifecycle & versioning
  • CDNs: edge caching mechanics
  • Cache control, invalidation & private content
  • Design patterns that compose them
  • Principal-level mastery check

Why bytes don't belong in your app tier

The single most common mistake in a media-heavy design is to stream a 4 GB video, a 20 MP photo, or a user's PDF through the application servers. A stateless app process that should be answering a request in single-digit milliseconds is instead held open for minutes, its memory and socket buffers consumed copying bytes it does nothing with. Concurrency collapses: an autoscaling group sized for 50,000 lightweight API calls per node now serves a few hundred uploads. Worse, you are paying premium compute prices to do a job that a purpose-built byte-mover does for a fraction of the cost.

The principal-level instinct is to recognize two fundamentally different workloads and route them to two fundamentally different substrates. Metadata and transactions — who owns the file, its name, its ACL, its version pointer — are small, mutable, queryable, and belong in a database. The blob itself — large, write-once, read-many — belongs in an object store, and its delivery belongs to a CDN. Your app tier touches neither byte stream; it only mints the URLs that let clients talk to those systems directly.

Object storage semantics

Object storage (S3, GCS, Azure Blob) is not a filesystem and not a block device. It is a flat namespace of objects — an opaque blob of bytes plus a key (the full path string), plus metadata and a version. There are no directories; the slashes in users/42/avatar.png are just characters in the key, and the "folder" view your console shows is a prefix-listing illusion. This flatness is what makes object stores scale to trillions of objects across a fleet without a central inode table to bottleneck on.

Contrast the three storage models, because the boundaries are easy to get wrong:

ModelUnit / APIMutabilityUse case
Block (EBS)Fixed-size blocks, read/write offsetsRandom in-place writesDatabase volumes, raw disks for a single attached host
File (NFS/EFS)POSIX paths, directories, locksPartial in-place editsShared mounts, legacy apps expecting a filesystem
Object (S3)PUT/GET/DELETE by key over HTTPWhole-object replace; no append/edit-in-placeMedia, backups, static assets, data-lake files

The key semantic consequence: objects are effectively immutable. You cannot patch byte 5,000 of an object — you overwrite the whole key with a new version. This write-once, read-many shape is exactly what allows the aggressive caching and replication that give object stores their durability and CDN-friendliness. It also means object storage is the wrong home for anything needing frequent small mutations, like a hot counter or a transactional row.

Durability, consistency & the upload path

Object stores advertise eleven nines of durability (99.999999999%) — the expected probability that a stored object survives a year. That is not a slogan; it is engineered. Each object is split and stored with redundancy, classically via erasure coding: an object is chopped into k data fragments plus m parity fragments (say 10+4), spread across independent disks, racks, and availability zones, so any 4 of the 14 fragments can be lost and the object still reconstructs. Erasure coding gives the same fault tolerance as 3× replication at roughly 1.4× the storage cost — the reason it dominates at scale.

Consistency model

For years S3 was famously eventually consistent for overwrites and list operations, which forced ugly workarounds (read-after-write retries, external manifests). Since December 2020, S3 provides strong read-after-write consistency for all operations. Concretely: the moment a PUT returns 200, any subsequent GET from any client, in any AZ of that region, is guaranteed to return the new bytes — never a stale version, never a 404 for an object that was just written. The same now holds for overwrites and for LIST reflecting newly added keys. This eliminates an entire class of "I just wrote it but the next service can't see it" race bugs. Note this is per-object; it is not a multi-object transaction — there is no atomic "rename" across two keys.

Multipart upload

A single PUT tops out around 5 GB and risks restarting the whole transfer if the connection drops at 99%. Multipart upload solves both: the client initiates an upload, sends parts (5 MB–5 GB each, up to 10,000 parts) in parallel, each retryable independently, then sends a CompleteMultipartUpload that stitches them server-side into one object. This is how you upload terabyte-scale objects, saturate bandwidth with parallelism, and resume after a flaky mobile connection drops.

Presigned URLs

The mechanism that gets the app server off the byte path is the presigned URL. Your backend, holding the real credentials, computes a signed URL that grants a specific operation (PUT to one key, or GET of one object) for a bounded time. The client then talks straight to the object store. The app never sees the bytes; it only authorizes.

// Backend: mint a 15-minute upload grant, then hand the URL to the client
url = s3.presign(
    method   = "PUT",
    bucket   = "user-uploads",
    key      = "u/42/" + uuid() + ".jpg",
    expires  = 900,                      // seconds
    conditions = { "content-length-range": [0, 10_000_000],
                   "Content-Type": "image/jpeg" }
);
// Client uploads directly to S3 with this URL; app tier never touches the blob.
Key insightDurability (will the bytes survive?) is not availability (can I reach them right now?). Eleven nines of durability still sits behind a regional service with its own availability SLA — and a single-region bucket is a single-region failure domain. If the object data is business-critical, cross-region replication is a separate, deliberate decision. See Multi-Region & DR.

Storage classes, lifecycle & versioning

Not all objects are accessed equally, and storage cost dominates the bill for large archives. Object stores expose storage classes that trade retrieval latency and cost: a hot Standard tier for frequently read objects; Infrequent-Access tiers that are cheaper to store but charge a retrieval fee; and cold archive tiers (Glacier-style) where storage is nearly free but a restore can take minutes to hours. An Intelligent-Tiering option moves objects between tiers automatically based on observed access.

Lifecycle policies automate this: "transition objects to Infrequent-Access after 30 days, to archive after 90, expire after 365." This is the difference between a log/media archive that costs sane money and one that bankrupts the team a year in. Versioning, when enabled, keeps every overwrite and delete as a prior version (a delete just writes a delete-marker), giving you accidental-deletion recovery and an audit trail — at the cost of storing every revision, which is precisely why lifecycle rules to expire old non-current versions matter.

CDNs: edge caching mechanics

A Content Delivery Network is a globally distributed fleet of cache servers (edge PoPs) that sit between users and your origin. The win is twofold: latency — serving from a PoP 10 ms away instead of an origin 150 ms across an ocean — and offload — the origin (your object store or app) sees a tiny fraction of total traffic because the edge absorbs the rest.

Requests are steered to the nearest healthy PoP by anycast routing: the same IP is announced from every PoP, and BGP delivers the packet to the topologically closest one. This is conceptually the geographic sibling of the load balancing you do inside a region — see Load Balancing. On a request, the edge checks its local cache:

  • Cache hit — the object is present and fresh; served immediately from the edge. No origin contact.
  • Cache miss — not present (or stale); the edge fetches from origin, stores a copy, and serves it. Subsequent requests for that key in that PoP hit.

A subtle but critical optimization is origin shielding (a.k.a. a mid-tier or "shield" cache): rather than every one of hundreds of PoPs independently missing and hammering the origin on a cold object, misses funnel through a designated regional shield PoP. The origin sees at most one fetch per object instead of one per PoP — this is what prevents a viral video or a fresh deploy of static assets from creating a thundering herd against your bucket. It is the same dogpile/stampede problem covered in Caching, solved at the network edge.

Two delivery models exist. A pull CDN (the default and overwhelmingly common one) lazily fetches from origin on the first miss — zero pre-work, ideal for large catalogs where you don't know what will be hot. A push CDN has you proactively upload content to the edge ahead of demand — useful for a small set of guaranteed-hot assets (a launch-day video, a software release) where you can't afford the cold-miss latency for the first viewer.

Cache control, invalidation & private content

Freshness is governed by HTTP headers the origin sets, and getting these right is most of the skill:

  • Cache-Control: max-age=N — the TTL. For how many seconds may the edge (and browser) serve this without re-checking origin? public allows shared caches to store it; private restricts to the browser; no-store forbids caching entirely.
  • ETag — an opaque content fingerprint. After TTL expires, the edge sends a conditional If-None-Match; if unchanged the origin replies 304 Not Modified with no body, revalidating cheaply instead of re-shipping the bytes.
  • stale-while-revalidate=N — serve the stale copy instantly while asynchronously refreshing it in the background. The user never waits on a revalidation round-trip; the cache self-heals. Pair with stale-if-error to keep serving stale content when the origin is down — a free resilience win.
Cache invalidation is hard"There are only two hard things in computer science." Once an object is replicated across hundreds of PoPs, recalling it is slow and expensive — explicit purges can take seconds to minutes to propagate globally, and aggressive purging defeats the point of the cache. The robust answer is to avoid invalidation entirely: treat URLs as immutable via content-hashed / versioned filenames (app.9f3c1a.js, logo.v2.png). New content gets a new URL, so there is nothing to invalidate; old URLs simply age out. Reserve explicit purges for emergencies (leaked content, legal takedown).

For private content — a paid video, a user's private document — you cannot rely on an unguessable URL. CDNs support signed URLs / signed cookies: the edge itself validates a cryptographic signature (and expiry, and optionally source-IP) before serving. This pushes authorization to the edge so the byte path still never touches your app, while a leaked link stops working after its short expiry. This is the same presigned-URL idea from the object store, enforced one layer further out at the CDN.

Design patterns that compose them

These primitives combine into three patterns you should reach for reflexively:

  1. Direct-to-S3 upload via presigned URL. Client asks the app for an upload grant; app returns a presigned PUT URL and records intended metadata as pending; client uploads straight to the bucket; an S3 event notification (or the client's completion callback) flips the metadata to ready. The app tier handled two tiny JSON requests and zero bytes. This is the backbone of file upload in Dropbox and avatar/photo upload in any social product.
  2. CDN-in-front-of-object-store for static assets. Point the CDN's origin at the bucket, set long max-age on versioned filenames, and serve all images, JS/CSS bundles, and downloads from the edge. The origin bucket sees near-zero traffic after warm-up; the app tier serves only dynamic API calls.
  3. Metadata-in-DB + blob-in-store split. The relational/NoSQL row holds the small queryable truth (owner, title, ACL, storage_key, version, size, checksum); the object store holds the bytes. Queries, joins, permission checks, and feeds all run against the cheap, indexable database; the heavy bytes live where bytes are cheap. This split is the data-modeling heart of YouTube / Netflix (catalog and watch-history in databases, transcoded video renditions in the object store, delivered via CDN) and of essentially every media product.
Trade-offThe DB-plus-store split introduces a two-system consistency gap: the row and the blob are written separately and can diverge (an orphaned blob whose row never committed, or a row pointing at a key that failed to upload). You reconcile this with the pending→ready state machine plus an async sweeper that garbage-collects orphaned objects and retries or expires stuck-pending rows. You are trading a clean single-system transaction for scale and cost — a trade worth making, but only if you actually build the reconciliation.

Principal-level mastery check

What strong engineers doThey get bytes off the app tier in the first two minutes — presigned upload, CDN in front of the store — without being prompted, and they reason explicitly about the metadata/blob split and the consistency gap it creates. They reach for versioned-URL caching over purge-based invalidation, and they distinguish durability from availability and from access control rather than treating "we'll use S3" as a complete answer.

Probing questions to ask engineers:

  • "A user uploads a 3 GB video from a phone on a flaky connection. Walk me through every hop — what touches the bytes, and what happens when the connection drops at 80%?"
  • "You're caching avatars at the edge with a one-year TTL. A user changes their avatar. How does anyone ever see the new one?"
  • "How do you serve a paid movie so a leaked link can't be shared, without your app tier ever streaming the bytes?"
  • "Your DB row says the file exists but the GET 404s. How did that happen and how does your design prevent or repair it?"
  • "Eleven nines of durability — does that mean the data is highly available? What's the difference, and what would you add for a region-failure scenario?"
  • "Why erasure coding instead of three full copies, and what does that buy you in storage cost versus rebuild time?"
