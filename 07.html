Proxies

A proxy is an intermediary you insert between client and origin to get one controllable choke point — for caching, filtering, TLS, and observability — without changing either endpoint. You pay for it with an extra hop and one more thing to operate. The decisions are about which direction it faces and what work it does there.

The trade-off A single point of control in exchange for an extra hop and a component to run. Inserting an intermediary gives you one place to cache, filter, transform, terminate TLS, and observe traffic — without touching client or origin. In return every request takes one more hop, and the proxy is now infrastructure you must keep alive.

What a proxy is

A proxy server is an intermediary — hardware or software — that sits between a client and the origin (back-end) server. It receives requests and relays them, and because it stands in the middle of every request and response, it's a natural place to cache, filter, log, transform, encrypt/decrypt, and compress — all without modifying either endpoint.

The core idea is indirection: insert one controllable hop and you gain a single point where policy, observation, and reshaping all apply. That same hop is also the cost — latency and a component to operate.

The one-line frameA proxy gives you a choke point. Everything good about it (shared cache, central policy, TLS offload, origin hiding) and everything risky about it (an extra hop, a thing that can overload or fail) follow from that one fact.

Three scenarios — decide before you read

Decide your answer first — the rest is the answer.

Scenario 1 — whom does it protect? One team wants to hide clients' identities as they browse the internet; another wants to hide the back-end servers from clients and balance load across them. Both say "put a proxy in front" — but these need opposite kinds of proxy. Which is which, and what does each hide?
Scenario 2 — the hot resource goes cold A popular cached resource expires. In the next 50 ms, 10,000 clients request it simultaneously — all misses. What does a naive proxy do, and what happens to the origin?
Scenario 3 — the choke point chokes You route 100% of traffic through one proxy for TLS termination and caching. What have you just (re)introduced, and how do you keep the proxy from becoming the thing that takes you down?

Scenario 1 is forward vs reverse. Scenario 2 is the cache stampede (and collapsed forwarding). Scenario 3 is the choke-point risk.

A proxy is defined by three decisions

Fewer than a database, but each matters.

Decision 1 — Which direction does it face? (forward vs reverse)

The whole taxonomy reduces to whom the proxy represents:

  • Forward proxy acts on behalf of the client — sits at the edge of the client's network, forwards their requests outward, and hides the client from the origin. Used for egress control, content filtering, anonymity, and client-side caching. (An open proxy is a forward proxy any internet user can use; anonymous hides the client IP, transparent passes it in X-Forwarded-For and is great for caching with no client config.)
  • Reverse proxy acts on behalf of the server — sits in front of one or more origins, accepts requests inward, and hides the origin from clients. Used to cache responses, load balance, hide internal topology, and terminate TLS.
The failure that happens when you get this wrong — the wrong-direction proxy This is Scenario 1. "Put a proxy in front" is ambiguous until you say which party it protects. Hiding browsing clients needs a forward proxy (client-side, outward). Hiding and balancing back-ends needs a reverse proxy (server-side, inward). Same machinery, opposite direction, opposite party protected — pick the wrong one and you solve the wrong problem (e.g. exposing your origin while trying to anonymize clients).

Decision 2 — What work does it do at the choke point?

Two cleanly separable reasons to insert a proxy — name which you're after:

  • Performance: caching (serve repeated reads from the proxy), request coalescing, compression.
  • Control: TLS termination, header rewriting, authentication, rate limiting, logging.
The failure that happens when you get this wrong — the choke point overloads (or fails) This is Scenario 3. The proxy sees every request, so it's both a throughput ceiling and a single point of failure — the very SPOF you may have added it to remove. Defenses are the same as for a load balancer: run it as a redundant pool (active-active behind a VIP, or many nodes via ECMP), size it for peak, and keep TLS/crypto cost in mind since termination is CPU-heavy. A proxy you can't lose isn't an optimization — it's a liability.

Decision 3 — How does it protect the origin from spikes?

Because a proxy is a shared choke point, it's the ideal place to absorb read traffic — the first client pays the origin fetch, everyone after is served the cached copy. The key behavior is collapsed forwarding (request coalescing).

The failure that happens when you get this wrong — the cache stampede This is Scenario 2. When a hot resource expires and thousands of identical requests miss at once, a naive proxy forwards every one to the origin — a thundering herd that can flatten the back-end exactly when an item goes hot. Collapsed forwarding fixes it: the proxy detects an identical request already in flight, sends only one to the origin, and fans the single response back to all waiters — collapsing N requests into 1. Pair it with sensible TTLs and stale-while-revalidate. (Same problem and family of fixes as the caching lesson's stampede.)

The shape to remember: a proxy is 3 decisions — direction, work it does, origin protection — with failures: wrong-direction, choke-point overload, cache stampede.

Famous implementations

NGINX

  • What it is. The ubiquitous reverse proxy / web server / TLS terminator.
  • Key idea. Event-driven, extremely efficient; the default edge for a huge share of the web and Kubernetes Ingress.
  • The failure to watch. Classic config-reload model — dynamic back-ends need an Ingress controller or NGINX Plus.

Envoy

  • What it is. Modern L7 proxy, the data plane behind most service meshes.
  • Key idea. Dynamic config via a control plane (xDS); ships retries, circuit breaking, mTLS, and collapsed-forwarding-style protections.
  • The failure to watch. Per-hop latency + operational complexity of the control plane.

Squid

  • What it is. The classic forward/caching proxy — the canonical example of client-side caching and egress control.
  • Key idea. Transparent caching with X-Forwarded-For, collapsed forwarding for hot objects.
  • The failure to watch. An open forward proxy left unsecured becomes an abuse relay.

Build it yourself

Implement collapsed forwarding

Many clients request the same uncached key at once. Write the proxy logic that sends only one request to the origin and serves all waiters from it. What's the data structure, and what's the edge case when the in-flight request fails?

Reveal the approach

Keep a map from key → an in-flight "promise"/future. On a miss: if the key is already in the map, await that future; otherwise create the future, fire the single origin request, store it, and on completion resolve all waiters and remove the entry. This is exactly single-flight. The edge case: if the origin request fails, you must fail (or retry) all waiters and clear the map entry — otherwise a poisoned entry blocks the key. Add a short timeout so one slow origin call doesn't stall every waiter forever.

Design-it-yourself

  • Edge TLS + origin protection. Design a reverse-proxy edge that terminates TLS, caches, and shields the origin from spikes. → CDN, Caching.
  • Make the proxy HA. One proxy is a SPOF. Design the redundant tier. → Load Balancing.
