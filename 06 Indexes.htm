Indexes

How indexes trade storage and write cost for fast reads, and why they become essential at scale.

Indexes are well known when it comes to databases; they are used to improve the speed of data retrieval operations on the data store. An index makes the trade-offs of increased storage overhead, and slower writes (since we not only have to write the data but also have to update the index) for the benefit of faster reads. Indexes are used to quickly locate data without having to examine every row in a database table. Indexes can be created using one or more columns of a database table, providing the basis for both rapid random lookups and efficient access of ordered records.

An index is a data structure that can be perceived as a table of contents that points us to the location where actual data lives. So when we create an index on a column of a table, we store that column and a pointer to the whole row in the index. Indexes are also used to create different views of the same data. For large data sets, this is an excellent way to specify different filters or sorting schemes without resorting to creating multiple additional copies of the data.

Just as to a traditional relational data store, we can also apply this concept to larger data sets. The trick with indexes is that we must carefully consider how users will access the data. In the case of data sets that are many TBs in size but with very small payloads (e.g., 1 KB), indexes are a necessity for optimizing data access. Finding a small payload in such a large data set can be a real challenge since we can’t possibly iterate over that much data in any reasonable time. Furthermore, it is very likely that such a large data set is spread over several physical devices—this means we need some way to find the correct physical location of the desired data. Indexes are the best way to do this.

Principal-level deep dive

How indexes are really stored: LSM vs B-Tree

The original lesson treats an index as an abstract sorted lookup structure. At principal level the more consequential question is which storage engine physically backs that index, because that single decision dictates your write throughput, read latency tail, disk footprint, and even how compaction interacts with your SLOs. Almost every index you will ever build resolves to one of two families: a B-Tree (update-in-place) or an LSM-Tree (log-structured merge). Knowing why, and when each flips from advantage to liability, is the actual skill.

B-Trees: update-in-place, read-optimized

A B+Tree keeps keys sorted in fixed-size pages (typically 4–16 KB; InnoDB defaults to 16 KB) arranged in a shallow, high-fanout tree. A lookup or range scan touches O(log_f N) pages — for a billion rows with fanout ~500 that is roughly 3–4 page reads, most of which are cached. This is why B-Trees are the default for read-heavy OLTP: MySQL InnoDB and PostgreSQL both build their primary structures on B+Trees. The cost is on the write path. An update locates the leaf page and rewrites it in place, which means a random write, page splits when a leaf overflows, and a redo/WAL write plus (in InnoDB) a doublewrite to guard against torn pages. Under random-key insertion you get write amplification from rewriting whole pages for tiny logical changes, plus fragmentation that degrades scan locality over time.

LSM-Trees: buffer, flush, compact

An LSM-Tree never updates in place. Writes land in an in-memory sorted structure (the memtable, usually a skiplist) plus a sequential WAL for durability. When the memtable fills, it is flushed as an immutable, sorted SSTable on disk. Reads must now consult the memtable and potentially several SSTables, so a background compaction process merges SSTables, discards overwritten/deleted keys (tombstones), and keeps the level count bounded. This is the engine behind RocksDB, Cassandra, ScyllaDB, and the LevelDB lineage. Because every write is an append, ingestion is sequential and fast — the workload that crushes a B-Tree (high-volume random-key inserts) is exactly where LSM shines.

The three amplifications — and how the choice flips

The honest way to compare engines is the RUM conjecture: you can optimize at most two of read, update, and memory/space amplification. The numbers below are characteristic, not guarantees.

DimensionB-TreeLSM-Tree
Read amplificationLow — one path to one leafHigher — may probe multiple SSTable levels (mitigated by Bloom filters)
Write amplificationHigh under random keys — full-page rewrites + WAL + doublewriteLower at ingest, but compaction re-rewrites data (leveled compaction ~10–30x over its lifetime)
Space amplificationModerate — fragmentation, ~half-full pages after splitsLow with leveled compaction; high with size-tiered (stale versions linger)
Sweet spotRead-heavy, range-scan-heavy, mixed OLTPWrite-heavy ingest, time-series, append-mostly logs
The flipLeveled compaction (RocksDB default) trades write amplification for low space amplification and good reads; size-tiered compaction (Cassandra's classic mode) trades space for lower write amplification. The "LSM is write-optimized" maxim is only true at ingest — sustained, compaction can push lifetime write amplification well above a B-Tree's. The right answer depends on whether your bottleneck is the foreground write or the background I/O budget.

Bloom filters: skipping SSTables you don't need to read

The LSM read penalty would be fatal without a way to avoid touching SSTables that cannot contain the key. Each SSTable carries a Bloom filter — a probabilistic bit array (see Probabilistic Data Structures) that answers "is this key possibly here?" with no false negatives and a tunable false-positive rate (~1% at ~10 bits/key). A point lookup checks each level's filter and skips the disk read entirely on a negative, collapsing a multi-SSTable read into typically one real I/O. Range scans cannot use Bloom filters (they test exact keys only), which is why LSM range performance is more sensitive to compaction health than point lookups.

Clustered vs secondary, covering, and column order

Beyond the engine, two index-layout choices dominate query plans. A clustered index (InnoDB primary key) stores the full row in the leaf, so a PK lookup needs no second hop. A secondary (non-clustered) index leaf stores only the PK; resolving non-indexed columns costs an extra lookup back into the clustered index — the classic "bookmark lookup." A covering index eliminates that second hop by including every column the query reads, so the engine answers entirely from the index (an "index-only scan" in Postgres). For a composite index (a, b, c), the leftmost-prefix rule governs usability: it serves predicates on a, a,b, and a,b,c, but a query filtering only on b cannot use it. Equality columns belong left of range columns, because the first range predicate stops further columns from narrowing the scan — order the columns to your access patterns, not alphabetically.
