CAP Theorem

A distributed system, when the network splits, can keep answering or stay correct — not both. CAP names that forced choice. The catch every junior misses: it only bites during a partition, and partitions are rare — so CAP is necessary but not sufficient.

The theorem Under a network partition, you must choose Consistency or Availability — you cannot have both. Conjectured by Eric Brewer (PODC 2000), proved by Gilbert & Lynch (MIT, 2002). It describes degradation under failure, not steady-state capability — which is exactly why you also need PACELC.

What CAP says

CAP is about three properties a distributed data store might offer:

  • Consistency (C) — every read returns the most recent write (or an error). All nodes agree on the data at the same instant. (In the formal proof this means linearizability.)
  • Availability (A) — every request to a non-failing node gets a non-error response. But an available answer carries no guarantee it reflects the latest write.
  • Partition tolerance (P) — the system keeps working despite a network partition.

The theorem (Brewer's 2000 conjecture, proved by Gilbert & Lynch in 2002 via an asynchronous-network adversary argument): you cannot have all three at once.

The one-line frameCAP is not a menu you order from once at design time. The trade-off only forces itself on you when a partition actually occurs. When the network is healthy, a good system offers both strong consistency and high availability. CAP describes how a system degrades under failure, not what it can do normally.

Three scenarios — decide before you read

Don't scroll past these. Decide your answer first — the rest of the lesson is the answer. If you already know all three cold, skip to classifying real systems.

Scenario 1 — the cable is cut A switch fails. Two halves of your cluster can no longer talk. A write request arrives at a node on one half. It has exactly two options — what are they, and what does each one sacrifice?
Scenario 2 — "we're a CA system" A teammate says your distributed database is "CA — consistent and available, we don't need partition tolerance." Why is that statement meaningless for any system that talks over a network?
Scenario 3 — the 99.9% you forgot Your network is healthy — no partition — which is true the vast majority of the time. A strongly consistent read still costs more than an eventually consistent one. CAP says nothing about this case. What decision are you still making on every single request?

Scenario 1 is the forced choice. Scenario 2 is the "CA" trap. Scenario 3 is why you need PACELC — the latency-vs-consistency choice CAP is silent about. Each maps to an idea below.

The four ideas to understand

A theory lesson isn't decisions to configure or sub-problems to build — it's a small set of ideas you must hold correctly, each guarded by a common misconception. Get the idea, avoid the trap.

Idea 1 — The forced choice: C or A, but only during a partition

This is Scenario 1. When the network splits and a node can't reach its peers, a request lands and the node faces a binary decision with no third option:

  • Reject or block → preserve consistency (CP). If the node can't confirm it holds the latest data, it refuses the (possibly stale) read or the (un-replicable) write. It sacrifices availability.
  • Answer anyway → preserve availability (AP). The node responds with whatever it has and accepts the write locally to reconcile later. It sacrifices consistency.
The misconception to avoid — "you pick two at design time" The Venn-diagram teaching makes CAP sound like a one-time menu choice. It isn't. When the network is healthy, you get both C and A. The choice is forced only at the moment of partition — CAP is a statement about behavior under a specific failure, not a permanent property of your system.

Idea 2 — CP systems: correct, or unavailable

A CP system chooses consistency when partitioned: if it can't guarantee a read reflects the latest write, it returns an error or times out rather than serve stale data. The minority side of a partition effectively stops serving.

  • Banking / ledgers — you'd rather a transfer fail and be retried than let two partitions independently spend the same balance.
  • Coordination stores (ZooKeeper, etcd, Consul) — deliberately CP: they'd rather go unavailable than hand out conflicting answers about who holds a lock.
The misconception to avoid — "CA is a real option" This is Scenario 2. The diagram shows three pairs (CP, AP, CA), but CA is a trap. In any system that talks over a network, partitions will happen — switches fail, cables cut, GC pauses freeze nodes. Partition tolerance isn't optional, so the real choice is always C vs A. A "CA" system is either a single-node system (where the question doesn't arise) or a CP/AP system in denial — one whose partition behavior is whatever the code accidentally does, which is strictly worse than a deliberate choice.

Idea 3 — AP systems: always up, eventually right

An AP system chooses availability: every node keeps answering even when partitioned, returning its best local data and accepting writes to reconcile later (via versioning, vector clocks, or last-write-wins). Two clients on opposite sides may briefly see different values; once the partition heals, replicas converge — eventual consistency.

  • Cassandra, DynamoDB — the textbook AP stores (the Dynamo lineage, where staying available for shopping carts beat perfectly-current reads).
  • DNS, CDNs — everyday AP: serve a possibly-stale answer globally rather than refuse to resolve.

The trade-off in one line: CP sacrifices uptime to never lie about data; AP sacrifices freshness to never go down. A stale "likes" count is harmless; a stale account balance is not.

The misconception to avoid — "consistency is on or off" "Strong vs eventual" hides a whole ladder: linearizable (one copy, real-time order) → sequential (same order everywhere, not necessarily real-time) → causal (related ops ordered; concurrent ops may differ — often the sweet spot) → eventual (converges if writes stop). Choosing a rung on this ladder is the real decision; CAP's binary collapses it. Full ladder in Consistency Models.

Idea 4 — PACELC: the else-branch CAP forgets

This is Scenario 3, and it's what separates a junior answer from a principal one. CAP only describes the rare partition. Daniel Abadi's PACELC (2010) spells out both regimes:

Partition → choose Availability or Consistency (the CAP question); Else (normal operation) → choose Latency or Consistency.

The ELC leg is the common case CAP ignores. Even on a perfectly healthy network, a strongly consistent system must coordinate before acknowledging — a quorum write, a leader round-trip, a consensus append — and that costs latency. To go faster you relax consistency. So every read and write carries a latency-vs-consistency decision, partition or not.

The misconception to avoid — "CAP is enough" Over-reliance on CAP is the tell-tale junior answer. It's silent about the trade-off you make on 99.9% of requests — the latency cost that actually dominates your budget and user experience. Strong engineers reach for PACELC to state behavior in both regimes, and they never just say "it's CP" — they name the operation, the failure, and the user-visible consequence: "During a partition, writes to the minority side are rejected with a 503 so the client retries; majority-side reads stay consistent. We accept a few seconds of write unavailability per failover because double-spending is unacceptable."

Classifying real systems (PACELC)

PACELC gives a two-letter class, one letter per leg — this is how you actually talk about a store:

SystemPACELC classReading
Google SpannerPC/ECConsistent under partition and in normal operation; pays the latency (TrueTime commit-wait) to stay consistent everywhere.
DynamoDB / CassandraPA/EL (tunable)Available under partition, low-latency normally — but per-request quorum knobs dial toward consistency when needed.
MongoDBPC/ECUnder partition the minority primary steps down; normally routes reads to the primary for consistency.
There is no "CA" classPACELC has no notation for "ignores partitions," because a distributed system can't. Every system has a P-clause answer even if its designers never chose one — in which case the behavior is whatever the implementation accidentally does.

The shape to remember: CAP is 4 ideas — the forced C-or-A choice (only during a partition), CP, AP, PACELC (the normal-case else-branch) — each guarded by a misconception: design-time menu, CA-is-real, consistency-is-binary, CAP-is-enough.

Famous implementations

The theory made concrete in real stores. Same facets — what it is, its CAP/PACELC stance, the mechanism, and the failure to watch.

CP — ZooKeeper / etcd / Spanner

  • Stance. Consistency over availability under partition; pay latency for consistency normally (PC/EC).
  • Mechanism. Built on consensus (Raft/Paxos): any commit needs a majority quorum. When a partition splits the cluster, at most one side holds a majority — that side progresses, the minority stalls (refuses writes, and linearizable reads) rather than serve stale state. On heal, the minority replays the leader's log.
  • The failure to watch. The minority partition is unavailable by design — put a coordination store on the critical path only where you can tolerate that stall.

AP — Cassandra / DynamoDB

  • Stance. Availability over consistency under partition; low latency normally (PA/EL).
  • Mechanism. No single leader required; every replica accepts reads/writes and reconciles later via versioning / last-write-wins. Stays up on every partition; converges after heal.
  • The failure to watch. Concurrent writes on opposite sides of a partition create conflicts you must resolve — silently picking last-write-wins can lose data.

Tunable — Cassandra quorums (PACELC as a dial)

  • Stance. The ELC leg exposed as a per-statement knob — you choose consistency vs latency per query.
  • Mechanism. The quorum rule R + W > N (read replicas + write replicas > total): QUORUM/QUORUM buys consistency at higher latency; ONE/ONE buys latency at the cost of staleness.
  • The failure to watch. There's no single right setting for the whole datastore — choose per operation, and a wrong knob silently gives you staleness or latency you didn't intend.

Build it yourself

Theory sticks when you apply it. Work each one before opening the approach.

Classify a system the way a principal does

Take a system you know (say, a URL shortener's redirect path, or a payment ledger). State its behavior under partition and in normal operation — naming the operation, the failure, and the user-visible consequence. Don't just say "it's CP."

Reveal the approach

Pick the operation, then both regimes. Example (payment ledger): "Writes (transfers) are CP — during a partition the minority side rejects them with a 503 and the client retries; we accept seconds of write unavailability per region failover because double-spending is unacceptable. Reads of balance are also consistent (PC/EC). In normal operation we pay a quorum round-trip of latency on every write." Contrast with a likes-counter: AP/EL — always accept the increment, reconcile later, never block, because a briefly-stale count is harmless. The skill is naming op + failure + consequence in both the P and E regimes.

Work the quorum math

For a store with N=3 replicas, which (R, W) settings give you strong consistency, and which give you low latency? Why does R + W > N guarantee a read sees the latest write?

Reveal the approach

With N=3: W=2, R=2 gives R+W=4 > 3 — any read quorum and any write quorum overlap in at least one replica, so a read is guaranteed to touch a node that saw the latest write: strong consistency, but every op waits for 2 nodes. W=1, R=1 (R+W=2 ≤ 3) is fastest but the read and write sets may not overlap, so you can read stale data. The overlap is the whole trick: R + W > N forces a shared replica between any reader and the last writer. This is the ELC leg as arithmetic.

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Pick C or A per feature. For one product (e.g. a social app), classify three features — the feed, the follower count, the DM read-receipt — as CP or AP and justify each by "stale vs no answer, which is worse?" → Consistency Models.
  • Make a CP store's stall tolerable. Your metadata store (etcd) goes unavailable on the minority side during a partition. Design the system around that stall so a partition doesn't take the whole product down. → Consensus & Raft, Resilience Patterns.
  • Resolve AP conflicts. Two clients write the same key on opposite sides of a partition. Design conflict resolution that doesn't silently lose a write (vs naive last-write-wins). → CRDTs, Replication.
