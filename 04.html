Caching

A cache trades freshness for speed. Every decision in this lesson is about which staleness, which failure, and which data loss you are willing to accept in exchange — and what happens the day the cache is empty or gone.

The trade-off Lower latency in exchange for staleness, complexity, and a new failure. A cache keeps a copy closer and faster, so reads get quicker. In return you accept staleness (the copy can be wrong), invalidation complexity (keeping it from being wrong), and a new failure: the day the cache disappears, its full load lands on whatever was behind it.

What a cache is

A cache exploits locality of reference — recently or frequently requested data is likely to be requested again — by keeping a copy in a store that is faster or closer than the origin (the source of truth, usually a database). That is the entire upside. Everything hard about caching comes from the fact that you now have two copies of the truth, and they drift apart.

The one-line frameA cache you cannot safely lose is not a cache — it is an undocumented primary store. Design the miss path for the moment the cache is empty or down first; the hit path takes care of itself.

Three scenarios — decide before you read

Don't scroll past these. Decide your answer first — the rest of the lesson is the answer. If you already know all three cold, skip to the decision table.

Scenario 1 — the empty cache You run a product page at 50,000 req/s. 98% of reads are served from Redis; the 2% that miss go to the origin database (Postgres). At 02:00 the Redis cluster restarts for a kernel patch. What does Postgres see in the next 30 seconds, and does your service survive it?
Scenario 2 — the one hot key It's a flash sale. One SKU — the discounted games console — takes 40,000 req/s. Your cache is spread across 12 nodes using consistent hashing, so the keyspace is "perfectly balanced." How loaded is the node that owns the entry sku:console, and what breaks first — CPU, NIC, or the event loop?
Scenario 3 — the write you can't lose You're caching account balances to cut load on the origin database. A write comes in: a user pays $400. Where do you write it — cache, database, both, in what order — and what is the exact window in which a crash loses that $400?

Scenario 1 is a cache stampede. Scenario 2 is the hot-key problem. Scenario 3 is a write-policy choice. Each is what happens when one of the four decisions below is made wrong — and each has taken down a real system.

A cache is defined by four decisions

Every caching design — from a browser to a Redis cluster — is just these four choices. Make each one well and the cache works; make one wrong and you get a named, predictable failure. We take them in order, and each failure lives under the decision that causes it.

Decision 1 — Where does it live?

The classic taxonomy — local cache, distributed cache, global cache, CDN — is really one axis: how far is the copy from the request, and how many requesters share it? A closer, more private copy is faster and easier to keep correct, but serves fewer requests. A farther, more shared copy serves more requests, but keeping the shared copies in agreement becomes a distributed-systems problem. The layers, ordered from nearest the user to nearest the origin database:

LayerLives atBest forHow it is invalidated
Client / browserThe user's deviceStatic assets, user-specific immutable dataCache-Control max-age, ETag
CDN / edgeA PoP near the userStatic and cacheable dynamic responsesSurrogate keys, purge API
App near-cache (L1)Inside the app processHottest entries, sub-microsecond accessPub/sub invalidation, short TTL
Distributed cache (L2)A shared tier (e.g. a Redis cluster)Shared working set, session stateDelete-on-write, TTL
Origin database buffer poolThe database itselfRecently-read pagesManaged by the engine

One tension recurs everywhere: if each app node keeps its own local cache and the load balancer spreads requests randomly, the same entry lands on different nodes and the hit rate collapses. The two classic fixes — a distributed cache (use consistent hashing to give each node a slice of the keyspace) or a global cache (one shared tier everyone queries) — are the same locality-versus-shared-hit-surface trade you meet again in the load balancer lesson.

The failure that happens when you get this wrong — the hot key This is Scenario 2. Consistent hashing spreads entries evenly across nodes, but a single viral entry — the discounted console, a celebrity profile — concentrates all of its traffic on the one node that owns it. A "balanced" keyspace is irrelevant: 40,000 req/s for sku:console all land on one node's CPU and NIC, no matter how many nodes you have. Mitigations:
  • Key splitting — store the value under N suffixed copies sku:console#0 .. #N, and read one at random, spreading the load across N nodes.
  • An L1 near-cache — a small in-process cache in front of the shared tier, so the hottest entries never leave the app process and need no network round-trip.
  • Promote to a replicated set — serve the known hot entry from every node.

Decision 2 — How does a write propagate? (the write policy)

When data changes, where do you write it — cache, origin database, or both, and in what order? This is the write policy, and it is not stylistic: it determines exactly what data you can lose and what staleness window readers observe. This is the answer to Scenario 3.

PolicyMechanismConsistency / durabilityReach for it when…
Write-throughWrite cache and origin database synchronously before acknowledging the writerCache always matches the database; no data loss; write latency = database latencyRead-heavy, low write rate, can tolerate slow writes
Write-back (write-behind)Write cache, acknowledge immediately, flush to the database asynchronously (batched)Lowest write latency and highest throughput — but data loss if the cache node dies before the flushWrite-heavy, loss-tolerant: counters, metrics, session activity
Write-aroundWrite straight to the database, bypass the cache; the cache fills on the next read missAvoids polluting the cache with write-once data; the first read after a write is a missBulk loads, write-once-read-rarely data
Refresh-aheadProactively refresh hot entries before their TTL expiresHides refresh latency from readers; wasted work if the prediction is wrongPredictable hot entries with an expensive recompute
Scenario 3, answeredFor the $400 payment, write-back is the trap: you acknowledge the user, the cache node crashes before the asynchronous flush, and the money is gone with no record. A balance is a ledger — use write-through, or better, write to a durable log first. (This is why Redis offers AOF persistence, and why ledgers are append-only.) Reserve write-back for data you can lose a few seconds of: view counts, "last seen," metrics.

Decision 3 — How is a stale copy corrected? (invalidation)

"There are only two hard things in computer science: cache invalidation and naming things." — Phil Karlton

Once the cache and the origin database can disagree, you need a rule for fixing the cached copy. The difficulty is that the two diverge under concurrency. Three schemes, in increasing robustness:

  • TTL only — entries expire after a fixed time. Simple, but it bounds staleness rather than eliminating it: a 60-second TTL means up to 60 seconds of wrong answers.
  • Explicit invalidation (delete-on-write) — delete the entry whenever the database changes. But this opens a race: a reader can repopulate the cache with a stale value it read just before your database write, in the gap between your write and your delete. The cache is then wrong until the TTL backstop fires.
  • CDC-driven — a change-data-capture stream (for example, Debezium tailing the database's write log) invalidates deterministically off the committed write, instead of racing the application path. The robust answer when you need it.
The rule that surprises peopleOn a write, delete the entry — don't update it. Updating re-introduces the race (two concurrent writers can interleave their updates and leave the wrong one in place); deleting just means the next reader re-reads the database. Pair delete-on-write with a short TTL as a backstop, and for the hardest cases use versioned entries so a stale write cannot clobber a newer one.
The failure that happens when you get this wrong — the stampede This is Scenario 1, and it has caused some of the most spectacular cache-related outages. When a hot entry expires — or the whole cache restarts — thousands of requests miss at the same instant and all hit the origin database to recompute the same value. The origin saturates, latency cascades, and because failed requests often re-delete or re-miss, the storm feeds itself.
Real incident — Facebook, 24 September 2010 (~2.5-hour outage). An invalid configuration value was written to the database. Facebook's automated cache-refresh system saw the bad value and tried to refresh it from the database. But every cache client — the application code that reads the cache, not an end user — that hit a database timeout interpreted the failure as "this value is invalid," deleted the cached entry, and queried the database again. The database was soon taking hundreds of thousands of self-inflicted queries per second. Crucially, fixing the configuration value did not help — the loop was self-sustaining; engineers had to take the whole site offline to break it. The follow-on fix was a mechanism Facebook calls leases, described in their "Scaling Memcache at Facebook" paper (USENIX NSDI 2013): on a miss, the cache hands exactly one cache client a short-lived token (the lease) authorizing it to recompute the value, while every other client waits for that result — so a thundering herd becomes a single origin query. (This is the same idea as single-flight, the first fix below.) Reported to have cut peak database queries by ~92%.
There are four standard fixes, and you usually combine more than one:
  • Request coalescing / single-flight — only the first miss recomputes; concurrent misses for the same entry wait on that one computation and share its result. Go's singleflight and Caffeine's LoadingCache do this within one node; for the distributed tier you need a per-entry lock (Redis SET key val NX PX ttl). This is Facebook's lease.
  • Probabilistic early expiration — each reader rolls a probability (rising as the entry ages) to recompute early, so one unlucky reader refreshes before the mass expiry.
  • Stale-while-revalidate — on a miss, one reader recomputes while everyone else is served the last-known-good value instead of piling onto the origin.
  • TTL jitter + a circuit breaker — for the cold-restart case (Scenario 1), spread expiries so entries don't all lapse on the same second, and gate the miss path with a circuit breaker so a struggling origin sheds load instead of dying.
A second failure of this same decision — penetration Distinct from a stampede: repeated lookups for entries that do not exist (404s, missing users, failed logins, or an attacker requesting random non-existent keys) sail straight past the cache and hammer the origin database — a bigger cache does not help, because there is nothing to cache. It belongs to the invalidation decision because the fix is a lookup-and-store rule: negative caching (store the "not found" result with a short TTL) plus a Bloom filter in front to reject definitely-absent keys in O(1) without a network round-trip. (Bloom filters: lesson 42.)

Decision 4 — What gets evicted when the cache is full?

A cache has finite space, so when it fills, something must be thrown out. The classic policies (FIFO, LIFO, LRU, MRU, LFU, Random) all exist; the interesting question in practice is why plain LRU is often wrong and what modern caches do instead.

  • LRU is cheap but blind to frequency: a single scan of cold data (a batch job, an analytics sweep) evicts the genuinely hot working set — the scan-pollution failure.
  • LFU tracks frequency but never ages out entries that were hot long ago, and exact LFU is expensive to maintain.
  • W-TinyLFU (the default in Caffeine) combines a small LRU admission window with a frequency filter: a new entry is admitted only if a Count-Min Sketch estimates it is accessed more often than the entry it would evict, and counts are periodically halved (aging). Near-optimal hit ratios at O(1) cost.
  • ARC keeps recency and frequency lists plus "ghost" lists of recently-evicted entries, and self-tunes the split between them. Scan-resistant and tuning-free; patent history pushed much of open source toward W-TinyLFU instead.
When plain LRU is still the right callTiny caches, uniform access, or when you want something dead-simple and predictable. Don't reach for W-TinyLFU because it is fancy — reach for it when you have actually measured scan pollution or a low hit ratio on a skewed workload.

The decision table — make all four, per data class

The single most common mistake is choosing one caching strategy for the whole system. You make all four decisions together, and you make them per data class — a ledger and a view-counter want opposite answers. Here is how the three scenarios resolve:

Data classWrite policyInvalidationEvictionWhy
Financial ledger / balancesWrite-through (or log-first)Delete-on-write + versionLRU is fineZero tolerance for a lost or stale write
Metrics / counters / "last seen"Write-back (batched)TTLW-TinyLFUThroughput matters; a few seconds of loss is acceptable
Product catalog / read-mostlyWrite-around + read-throughDelete-on-write + short TTLW-TinyLFURead-heavy, skewed, scan-prone

Self-check — can you reconstruct the three scenarios?

You read the answers above. Close the page and see if you can rebuild them — that is the test, not recognition.

  • Scenario 1 (empty cache): Why does a cold cache or a single expiring hot entry produce a self-feeding stampede, and which combination of single-flight, probabilistic early expiry, stale-serve, TTL jitter, and a circuit breaker would you actually deploy — and which one is Facebook's lease?
  • Scenario 2 (hot key): Why is a "perfectly balanced" consistent-hash keyspace irrelevant to a single viral entry, and would you reach for key-splitting, an L1 near-cache, or a replicated set first — and what decides the order?
  • Scenario 3 (write you can't lose): Contrast write-through, write-back, and write-around on write latency, data-loss window, and the read-hit rate for a freshly written entry. Which fits a ledger, a metrics pipeline, and a write-heavy log?
  • Why delete rather than update the entry on a write — and what race does delete-on-write still leave open?
  • Why is cache penetration a different failure from a stampede, and why does a bigger cache fix neither?
  • When is plain LRU still the right choice over W-TinyLFU?

The shape to remember: a cache is 4 decisions — where it lives, how writes propagate, how stale copies are corrected, what gets evicted — and each wrong choice has a named failure: hot key, lost write, stampede / penetration, scan pollution.

Famous implementations

Three pieces of software cover most of what you'll meet in practice. Each is a different answer to the four decisions above. Each entry below breaks down the same facets so you can compare them directly — what it is, its concurrency model, what it stores, persistence, eviction, how it scales, the failure to watch, and when to reach for it.

Redis — the rich, persistent default

An in-memory data-structure server, used as the default distributed cache (L2) tier almost everywhere.

  • Concurrency model. Core command execution is single-threaded (one event loop), so commands are atomic without locks — but a single slow command or one hot entry can stall the whole instance. Since version 6.0 it adds threads for network I/O only, not command execution.
  • What it stores. Not just strings — rich values: hashes, lists, sets, sorted sets, streams, bitmaps. This is why Redis routinely doubles as a rate limiter, leaderboard, job queue, or pub/sub bus, not only a cache.
  • Persistence. Optional, two mechanisms: RDB point-in-time snapshots (compact, fast restart, can lose the last few minutes) and AOF, an append-only log of every write (more durable, larger, slower replay). Often run together. This is exactly what lets Redis back a write-through or log-first policy instead of being purely disposable.
  • Eviction. Built-in policies selectable per instance: LRU and LFU approximations, random, TTL-based, or noeviction (reject writes when full).
  • How it scales. Redis Cluster sharding: every key maps to one of 16,384 hash slots (CRC16 of the key, mod 16384), and slots are split across nodes server-side. Adding/removing a node moves slots, not the whole keyspace. Built-in primary–replica replication and automatic failover.
  • The failure to watch. The single-threaded core: one expensive command (a big KEYS, a large sorted-set range) or one hot key blocks every other client on that node.
Reach for Redis when…you need more than get/set — atomic counters, sorted sets, pub/sub for invalidation fan-out — or the cache must survive a restart, replicate, or fail over. The default choice unless you have a specific reason not to.

Memcached — the simple, fast original

The original distributed cache: deliberately minimal, and still excellent at the one thing it does.

  • Concurrency model. Multi-threaded — uses all CPU cores. On plain get/set under many concurrent connections it runs roughly 20–40% faster than Redis and handles very high connection counts gracefully.
  • What it stores. Opaque string/blob values by key only — no data structures. You serialize objects yourself.
  • Persistence. None. Purely in-memory; a restart loses everything. By design — it is a cache, not a store.
  • Eviction. LRU within a slab-allocated memory model (memory is carved into fixed size-classes, which can cause some waste, called slab calcification).
  • How it scales. Client-side: servers know nothing about each other; the client library spreads keys across them with consistent hashing. No built-in replication, failover, or rebalancing — that's the application's job.
  • The failure to watch. A lost node simply drops its share of the cache (those keys miss to the origin); without client-side consistent hashing, adding/removing a node reshuffles nearly everything and triggers a stampede.
Reach for Memcached when…your workload is pure, high-throughput get/set of simple values, you need neither persistence nor data structures, and raw multi-core speed on simple operations matters more than features.

Caffeine — the in-process L1

Not a server at all — a Java library for an in-process L1 cache, the successor to Google Guava's cache.

  • Concurrency model. Lock-free reads: the recency/frequency bookkeeping is buffered and replayed asynchronously, so maintaining eviction order never blocks a reader. This is what makes it fast under heavy contention.
  • What it stores. Arbitrary in-heap Java objects — no serialization, no network. It answers a hot entry in sub-microseconds.
  • Persistence. None — it lives and dies with the process. It is the L1 in front of a persistent L2 (Redis), not the source of truth.
  • Eviction. The reference implementation of W-TinyLFU (the policy from Decision 4): a Count-Min Sketch admission filter plus a small LRU window, giving near-optimal hit ratios at O(1) cost. Also supports size- and time-based expiry and refresh-ahead.
  • How it scales. It doesn't — it's per-process. Each application instance holds its own copy, so N instances mean N independent caches that can disagree.
  • The failure to watch. Coherence: because every process has its own copy, a write on one node doesn't invalidate the others. Bound staleness with a short TTL or fan out invalidations over a pub/sub channel.
Reach for Caffeine when…you want an in-process L1 in front of Redis/Memcached for the hottest entries (the hot-key mitigation), or a single-node app needs a fast local cache. Always pair it with a short TTL or pub/sub invalidation, since each process holds its own copy.

Build it yourself

Reading how a cache works is not the same as being able to build one. These are the implementation questions an interviewer (or a real design doc) will push on. Work each one out before you open the approach. The point is to discover why the well-known designs are shaped the way they are.

Implement an LRU cache with O(1) get and put

Build a fixed-capacity cache where reading or writing an entry is O(1), and when it's full you evict the least-recently-used entry — also O(1). What two data structures do you need, and why does either one alone fail?

Reveal the approach

A hash map + a doubly-linked list. The hash map gives O(1) lookup from key to its list node; the doubly-linked list keeps entries in recency order. On every access you unlink the node and move it to the head (O(1) because the map hands you the node directly — no scan). The tail is always the LRU victim, so eviction is O(1).

Why neither alone works: a hash map has no notion of order, so you can't find the victim; a list alone makes lookup O(n). The doubly-linked part matters because unlinking a node in O(1) needs both neighbours. (Java's LinkedHashMap with access-order is exactly this; many languages have an ordered-dict that gives it for free.)

Add per-entry TTL expiry to that cache

Now each entry also has a TTL and must not be returned after it expires. How do you avoid scanning every entry to find what's expired, and what's the trade-off between expiring lazily and expiring actively?

Reveal the approach

Lazy expiry: store an expiry timestamp on each entry and check it on read — if expired, treat it as a miss and evict. Zero background work, but dead entries waste memory until someone touches them. Active expiry: keep a min-heap or timing wheel keyed by expiry time and pop due entries in the background. Reclaims memory promptly but costs CPU and a second structure. Redis uses a hybrid: lazy on access plus a sampling sweep. The trade-off is memory (lazy holds garbage) versus CPU and complexity (active runs a reaper).

Implement an LFU cache in O(1)

Evict the least-frequently-used entry instead. The naive design needs a scan to find the minimum count. How do you make get, put, and eviction all O(1)?

Reveal the approach

Keep entries bucketed by access count: a map from frequency → a doubly-linked list of entries at that frequency, plus a map from key → node, plus a pointer to the current minimum frequency. On access, move the entry from its frequency-bucket to the (frequency+1) bucket in O(1); if its old bucket was the min and is now empty, increment the min pointer. Eviction removes the LRU entry from the min-frequency bucket. This is the standard O(1) LFU; notice it's strictly more bookkeeping than LRU — which is exactly why real caches reach for W-TinyLFU rather than exact LFU.

Make the cache thread-safe under high concurrency

Many threads hit the cache at once. A single global lock around get/put makes it correct but serializes everything — the lock becomes the bottleneck. How do you keep it correct without that contention?

Reveal the approach

Lock striping / sharding: split the keyspace into N independent segments, each with its own lock, so unrelated keys never contend (the idea behind Java's old ConcurrentHashMap segments). Better still, modern caches like Caffeine avoid locking the read path entirely: reads are lock-free and the recency/frequency bookkeeping is buffered and replayed asynchronously, so maintaining the eviction order doesn't block readers. The general lesson: don't protect the whole structure with one lock; either partition the contention or move the bookkeeping off the hot path.
