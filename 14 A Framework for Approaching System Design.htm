A Framework for Approaching System Design

A reusable seven-step method for reasoning through any real-world system design problem — from clarifying requirements through identifying and resolving bottlenecks.

Designing a large-scale system is hard for a few recurring reasons: 1) the work is open-ended — there is rarely one correct answer, only a set of decisions with different trade-offs; 2) the right design depends on scale and constraints that are not always stated up front; and 3) it is easy to jump to a solution before you understand the problem. A repeatable method protects you from all three. It forces you to surface the requirements, ground your decisions in numbers, and reason explicitly about the trade-offs instead of reaching for whatever architecture is fashionable.

The value of a framework is not that it produces a "standard" design — it does not. It is that it keeps you honest: every component you add should trace back to a requirement, and every requirement should be justified by an estimate. This is how strong engineers reason through a design, whether they are sketching it on a whiteboard, writing a design doc, or arguing for it in an architecture review.

Below are the seven steps, illustrated throughout with a Twitter-like service so you can see how each step feeds the next.

Step 1: Clarify requirements & scope

Start by pinning down the exact scope of the problem. Because design problems are open-ended, resolving ambiguity early is the single highest-leverage thing you can do. When you are building a real system, this means clarifying requirements with stakeholders — product owners, downstream teams, on-call engineers, whoever depends on the system. Time and engineering budget are finite, so you also need to decide which parts of the system you will focus on and which you will treat as out of scope (or as someone else's service).

Separate functional requirements (what the system does) from non-functional requirements (how well it must do it — latency, availability, consistency, durability, cost). For a Twitter-like service, the questions you would resolve before going further include:

  • Will users be able to post tweets and follow other people?
  • Do we need to create and display each user's timeline?
  • Will tweets contain photos and videos?
  • Are we building the backend only, or the front-end too?
  • Will users be able to search tweets?
  • Do we need to display hot trending topics?
  • Are push notifications required for new (or important) tweets?

Each answer reshapes the eventual design. Just as important is naming what you are not building: deciding to treat media storage as an external object store, or to defer trending topics to a later phase, is a legitimate scoping decision that keeps the design focused on where the real complexity lives.

Write the non-functional targets down. "Reads must return in under 200 ms at p99," "the timeline can be eventually consistent for a few seconds," "we target 99.9% availability." These targets are what later let you choose between a synchronous quorum read and a cached read, or between a strongly consistent store and an eventually consistent one — see Consistency Models & PACELC.
Step 2: Back-of-the-envelope estimation

Estimate the scale before you design anything. These numbers decide whether a single database primary is sufficient or laughable, whether you need sharding, how aggressively you need to cache, and how much you should worry about bandwidth. Designing capacity, partitioning, or caching without estimation is just guessing.

Three questions to answer:

  • Traffic / QPS: how many new tweets per second, how many timeline reads per second, what is the read-to-write ratio?
  • Storage: how much data per year? The answer changes dramatically once photos and videos are in scope.
  • Bandwidth: what ingress and egress are we expecting? This drives load-balancing and CDN decisions.

Worked example, the way you would do it on paper:

QuantityCalculationResult
Writes/day100M DAU × 10 writes/day~1B writes/day
Avg write QPS1B ÷ 86,400 s~12k writes/sec
Peak write QPS~5× average~60k writes/sec
Read:write ratiotimeline reads dominateoften 100:1 → ~1.2M reads/sec
Storage/day1B writes × 1 KB~1 TB/day
Storage/year1 TB × 365 (before replication)~365 TB/yr

That read:write ratio (~100:1 here) is one of the most consequential numbers you will produce — a read-heavy system is designed completely differently from a write-heavy one. It immediately tells you to invest in read replicas, caching layers, and possibly precomputed timelines rather than optimizing the write path.

Step 3: Define the APIs / interfaces

Define the APIs the system exposes. This does two things: it nails down the exact contract other components and clients depend on, and it acts as a check that you have not misread the requirements — if an interface is awkward to express, a requirement is probably underspecified. For the Twitter-like service:

postTweet(user_id, tweet_data, tweet_location, user_location, timestamp, …)

generateTimeline(user_id, current_time, user_location, …)

markTweetFavorite(user_id, tweet_id, timestamp, …)

At this point you also decide the interface style — REST, gRPC, GraphQL, or an event/message contract — based on the access patterns and consumers. That choice is a design decision in its own right; see API Design.

Step 4: Design the data model

Defining the data model early clarifies how data flows between components and sets up the later decisions about partitioning and storage. A useful discipline is to start from the access patterns — how the data will actually be read and written — rather than from a list of tables. The access patterns determine the partition key, whether you want a B-tree or LSM-based storage engine, and where hot keys will appear. Identify the entities, how they relate, and the concerns around storage, transport, and encryption. Core entities here:

User: UserID, Name, Email, DoB, CreationDate, LastLogin, etc.
Tweet: TweetID, Content, TweetLocation, NumberOfLikes, TimeStamp, etc.
UserFollows: UserID1, UserID2
FavoriteTweets: UserID, TweetID, TimeStamp

Then choose the storage technology to match. Does an eventually consistent, write-scalable NoSQL store like Cassandra fit the access patterns better than a relational MySQL-style database? What block or object storage holds the photos and videos? The SQL vs. NoSQL lesson and Choosing a Database walk through how to make that call deliberately rather than by habit.

Step 5: High-level architecture

Sketch a block diagram with five or six boxes for the core components — enough to solve the problem end-to-end, no more. The goal is a design you can explain in one breath, where each box exists for a reason you can state.

For Twitter, at a high level you would have multiple application servers handling read and write requests, with load balancers in front of them distributing traffic. Because the estimation in Step 2 told us reads vastly outnumber writes, it is worth splitting the read and write paths onto separate fleets so each can scale independently. On the backend you need a database that stores all tweets and supports a very high read volume, plus a distributed file/object store for photos and videos.

Let the numbers shape the boxes. The split between read and write fleets, the presence of a cache tier, and the choice of a separate media store are not defaults — each one is a direct consequence of a number from Step 2 (the read:write ratio, the QPS, the storage-per-year figure). If you cannot point to the estimate that justifies a component, question whether it belongs.
Step 6: Deep-dive the bottleneck components

Now go deep on the two or three components where the difficulty actually concentrates. Breadth across the whole diagram is the easy part; depth on the hard part is where good design happens. For each, lay out the alternative approaches, their pros and cons, and why you would choose one. There is no single right answer — what matters is reasoning about trade-offs while respecting the system constraints from Steps 1 and 2.

  • Given the data volume, how do we partition across multiple databases? Should all of a user's data live on one shard? What problems does that cause (hot shards, uneven load)?
  • How do we handle "hot" users who tweet heavily or follow huge numbers of accounts?
  • Since timelines surface the most recent and relevant tweets, should the storage layout be optimized to scan the latest tweets cheaply?
  • How much caching do we add, and at which layer? What is the expected hit rate, and what staleness does write-through vs write-back caching introduce?
  • Which components most need better load balancing?

This is where naming concrete machinery signals that your breadth is backed by real understanding: consistent hashing with virtual nodes for partition assignment, push vs pull timeline fan-out with a hybrid for celebrity accounts, read-repair vs synchronous quorum for replica consistency. Pick the deep-dive that the design's own constraints point to — the fan-out path for a feed, the exactly-once/dedup path for a payment system, the inverted-index update for search.

Step 7: Identify bottlenecks, failure modes & evolution

Finally, stress the design. Walk through as many bottlenecks and failure modes as you can and propose mitigations for each. Assume the network is unreliable and dependencies will be slow before they are down.

  • Is there a single point of failure? What removes it?
  • Do we have enough replicas of the data to keep serving users if we lose a few servers?
  • Do we run enough copies of each service that a few failures will not bring the whole system down?
  • How do we know when something breaks? Which signals are measured at which boundary, and what alert fires before users notice?

Reasoning about partial failure as a first-class concern — retries with exponential backoff and jitter to avoid retry storms, circuit breakers and bulkheads to cap blast radius, graceful degradation instead of cascading failure — is what separates a design that survives production from one that only works on the happy path. These map directly to the Resilience Patterns worth knowing by name, and they should be tied back to concrete SLOs & Error Budgets.

Two dimensions round out a mature design:

  • Cost. Sanity-check the bill. Cross-AZ replication traffic, the egress cost of a chatty microservice mesh, and the difference between keeping 365 TB hot in memory versus tiering cold data to object storage all have real dollar consequences that should inform the design.
  • Evolution. How does a system get from where it is today to the proposed architecture without a big-bang cutover? Dual-writes, backfill jobs, shadow traffic, the strangler-fig pattern, and a rollback plan are what make a design adoptable rather than academic.
Common anti-patterns to avoid.
  • No numbers. Designing capacity, sharding, or caching without estimation is the most common mistake.
  • Cargo-cult microservices. Reaching for Kafka, a service mesh, and a dozen services for a problem a single partitioned datastore solves — complexity with no justification.
  • Over-engineering for scale that does not exist. Multi-region active-active for a 10k-user internal tool.
  • Ignoring failure. Never asking "what happens when this dies?" or assuming the network is reliable.
  • Skipping scope. Building before deciding what is in scope, what is deferred, and what is someone else's service.
The method in one line. Clarify scope → estimate → define interfaces → model the data → sketch the architecture → go deep on the bottleneck → harden against failure and plan its evolution. Every component traces back to a requirement; every requirement is justified by a number. Run this loop on each problem in the rest of the course and it will become second nature.
