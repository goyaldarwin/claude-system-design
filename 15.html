Designing a URL Shortening Service like TinyURL

The simplest classic design — and a perfect lens on read-heavy systems. ~200 writes/sec, ~20,000 reads/sec: a 100:1 ratio. The whole design is about making the redirect a sub-millisecond cache hit, and the one genuinely interesting decision is how you mint short, unique, unguessable keys.

The crux Reads outnumber writes ~100:1, so the redirect path must be a cache hit, and key generation must never collide or bottleneck. Generating the short key is the one hard choice; everything else is caching the read path and sharding the store.

1. Requirements

Functional: shorten a long URL to a unique alias; redirect the alias to the original; optional custom alias; optional expiry.

Non-functional: highly available (downtime breaks every existing link), redirect latency near-zero, and short links not guessable (security). Extended: click analytics, public REST API.

2. Capacity estimate

  • Writes: assume 500M new URLs/month ≈ ~200 writes/sec.
  • Reads: at a 100:1 read:write ratio → 50B redirects/month ≈ ~20,000 reads/sec. This ratio is the whole design.
  • Storage: ~500 bytes/record × 500M/month × 60 months ≈ ~15 TB over 5 years — modest; the working set of hot links is far smaller and caches in RAM.
  • Key space: base62 ([a-z A-Z 0-9]) at 6 chars = 626 ≈ 56 billion combinations — plenty; 7 chars gives 3.5 trillion.

3. System API

shorten(api_key, long_url, custom_alias?, expiry?) → short_url
redirect(short_key) → HTTP 301/302 to long_url

(301 permanent caches in the browser — fewer hits but no analytics; 302 temporary always hits your server — enables analytics. Pick by whether you need click tracking.)

4. High-level architecture

  • App tier behind a load balancer — stateless, scales with reads.
  • Key-generation service or strategy (Sub-problem 1).
  • A cache (Redis) in front of the store — the redirect path should almost always be a cache hit at 20K reads/sec.
  • The store (key → long URL), sharded; read replicas for the long tail.

5. Data model

Almost trivially simple — which is the point: short_key (PK) → long_url, creator_id, created_at, expiry. No joins, no relationships. A key-value store fits perfectly (the access is pure get-by-key), though a relational store is fine at this scale too. (Choosing a Database.)

Three scenarios — the hard parts, as questions

Scenario 1 — two users, same instant Two requests to shorten different URLs hit two different servers in the same millisecond. How do you guarantee they get different short keys — and what's wrong with both "hash the URL" and "one global counter"?
Scenario 2 — the link goes viral Someone shortens a link and tweets it to 10 million followers. It gets 50,000 redirect requests/sec — all for one key. What absorbs that, and what happens the moment that one hot key expires from cache?
Scenario 3 — the links pile up After five years you have billions of records, most for links nobody ever clicks again. How do you keep the store from growing forever, and what do you do when someone hits an expired key?

Scenario 1 is key generation. Scenario 2 is the read-path cache. Scenario 3 is expiry/cleanup.

6. The hard sub-problems

Sub-problem 1 — Key generation

The one genuinely interesting decision. Three approaches:

ApproachHowTrade-off
Hash the URLMD5/SHA-256, take first 6–7 chars, base62Stateless, but collisions (two URLs → same prefix) need detection + retry; same URL maps to same key (no per-user analytics)
Counter + base62A distributed unique ID, base62-encodedZero collisions, but the counter is a dependency; keys are sequential and guessable
Random base62Generate a random 7-char key, check uniquenessUnguessable, no counter; collision probability negligible at this scale (one extra read to confirm)
The failure that happens when you get this wrong — collision or counter SPOF This is Scenario 1. Hashing collides — two different URLs can share a 6-char prefix — so you must detect the clash and rehash/append, adding a read on every write. A single global counter never collides but is a SPOF and a write bottleneck — except the myth here is that the counter is the problem: at ~200 writes/sec a Redis INCR (or a Snowflake-style ID, or handing each server a pre-allocated range of counter values) handles it trivially; a counter only bottlenecks at millions of writes/sec, which no URL shortener approaches. Since the requirement is unguessable, random base62 with a uniqueness check is usually the cleanest answer; counter+base62 with per-server ranges is the zero-collision alternative.

Sub-problem 2 — The redirect read path

20,000 reads/sec (50,000+ for a viral link) means the redirect must not hit the database each time. Put a cache (Redis) in front keyed by short_key; the hit rate is enormous because link popularity is heavily skewed (80-20). Cache misses fall through to read replicas.

The failure that happens when you get this wrong — the viral-link stampede This is Scenario 2. A single viral key is a hot key: 50K req/s all for one entry. The cache absorbs it — until that key expires from cache, when thousands of concurrent misses hit the store at once (a cache stampede). Fixes are the caching lesson's: single-flight on the miss, generous TTL with jitter, and stale-while-revalidate. The hot key itself is harmless to a cache (it's one entry everyone reads) — the danger is the synchronized miss.

Sub-problem 3 — Storage & expiry

Shard the store by short_key (hash-based — even distribution, and lookups are always by exact key so scatter-gather never arises). Expiry keeps it from growing forever: store a TTL, lazily treat expired keys as 404 on read, and run a background cleanup (or DB-native TTL) to reclaim space.

The failure that happens when you get this wrong — dead-link bloat This is Scenario 3. Without expiry, the store grows unbounded with links nobody clicks — paying storage and slowing everything. Lazy expiry (check on read, treat expired as gone) plus a throttled background sweep reclaims space without a giant scan. On an expired/missing key, return a clean 404 (or a "link expired" page), and serve it from cache too so a flood of hits on a dead viral link doesn't hammer the store.

Self-check — can you reconstruct the design?

  • The walkthrough: from 500M URLs/month and 100:1, derive writes/sec, reads/sec, and 5-year storage — and say why the ratio makes this a caching problem.
  • Scenario 1 (keys): Compare hash / counter / random base62 on collision, guessability, and bottleneck — and debunk the "counter is a bottleneck" myth.
  • Scenario 2 (viral): Why is a viral key safe for a cache but dangerous on expiry, and what prevents the stampede?
  • Scenario 3 (bloat): How do you bound storage growth and handle a hit on an expired key?
  • 301 vs 302 — which do you pick if you need click analytics, and why?

The shape to remember: walk it (requirements→estimate→API→high-level→model), then 3 hard parts — key generation, cached redirect, storage/expiry — all driven by the 100:1 read ratio.

Famous implementations

Bitly

  • What it is. The best-known commercial shortener; billions of links, heavy on click analytics.
  • Key idea. Unguessable keys + analytics on every resolve (so 302-style tracking), aggressive edge caching for the read path.
  • The failure to watch. Analytics writes per redirect become their own high-volume pipeline (see Ad Click Aggregation).

Snowflake-style ID generation

  • What it is. Time-ordered 64-bit IDs minted per-node, base62-encoded into keys.
  • Key idea. No central counter; IDs are roughly time-sortable and index-friendly. (Lesson 58.)
  • The failure to watch. Sequential-ish keys are guessable — add randomness if unguessability matters.

Build it yourself

Implement collision-free key generation

Generate a unique 7-char base62 key per shorten request across many servers, with no central bottleneck and no guessable sequence. What do you choose and why?

Reveal the approach

Random base62: generate 7 random chars (627 ≈ 3.5T space), attempt an insert with a unique constraint; on the rare collision, regenerate. At ~200 writes/sec the collision rate is negligible and the uniqueness check is one conditional write. If you want zero collisions and no per-write check, hand each app server a pre-allocated range of counter values (e.g. server A owns 1–1M, B owns 1M–2M) and base62-encode locally — no per-request coordination — but XOR/scramble the counter so keys aren't sequentially guessable. Both avoid the single-counter SPOF.

Design-it-yourself

  • Add click analytics. Count redirects per link at 20K/sec without slowing the redirect. → Ad Click Aggregation.
  • Custom + reserved aliases. Let users pick aliases without clashing with generated keys. How do you partition the namespace?
