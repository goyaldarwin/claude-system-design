SQL vs. NoSQL

The classic "relational vs non-relational" split is real but over-taught. The honest version: "SQL vs NoSQL" collapses three independent choices — query model, storage engine, consistency — into one misleading binary. A principal reasons about each axis separately and defaults to relational until a specific need forces otherwise.

The trade-off Query flexibility and integrity (SQL) vs. one access pattern done extremely well at scale (NoSQL). Relational gives you ad-hoc queries, joins, and ACID at the price of harder horizontal write-scaling. Each NoSQL family trades that general query power for a single pattern (get-by-key, document, wide-column…) done fast. You're choosing which flexibility to give up.

What the question really is

"SQL or NoSQL?" sounds like one decision. It's actually three, bundled together and then mislabeled:

  • Query model — relational (joins, ad-hoc queries) vs a NoSQL family (key-value, document, wide-column, graph).
  • Storage engine — B-tree (read-optimized) vs LSM-tree (write-optimized) — the axis that actually dominates performance.
  • Consistency model — strong vs tunable vs eventual.

A modern datastore mixes any combination: distributed SQL with serializable isolation, a key-value store on a B-tree, a wide-column store with per-query consistency. The label on the tin tells you almost nothing about the trade-offs that matter.

The one-line frameThe relational model gives you query generality and integrity, and pays in horizontal write-scaling effort. Each NoSQL family gives up generality to nail one access pattern. So the real question isn't "SQL or NoSQL" — it's "which flexibility am I willing to sacrifice for this workload?" Default to relational; justify every move away.

Three scenarios — decide before you read

Scenario 1 — "let's use NoSQL, it scales" A team picks a document store for a new app "because NoSQL scales and we'll grow." The data is full of relationships (users, orders, line items, payments). What goes wrong six months in, and what did "NoSQL scales" actually leave out?
Scenario 2 — the ledger that must not lose a cent You're storing financial transactions. Two transfers touch the same balance at the same instant. Which property must the store guarantee, and which side of the SQL/NoSQL line traditionally gives it — and is that still true today?
Scenario 3 — 500K writes/sec of sensor data You're ingesting half a million timestamped sensor readings per second, read back mostly as time-range scans. What single axis — not the SQL/NoSQL label — decides whether your store survives this, and which way does it point?

Scenario 1 is the label trap. Scenario 2 is the consistency axis (and NewSQL). Scenario 3 is the storage-engine axis.

The three axes the binary hides

Reason about each separately — that's the whole skill.

Axis 1 — Query model

Relational shines when data has relationships and you don't yet know every query — joins, transactions, ad-hoc queries, integrity via constraints. The NoSQL families each nail one shape: key-value (O(1) get-by-key), document (self-contained aggregates), wide-column (write-heavy keyed-by-partition), graph (multi-hop traversals). The short version: relational keeps query generality at the cost of harder write-scaling; NoSQL trades generality for one pattern done extremely well. (Full family guide: Choosing a Database.)

Axis 2 — Storage engine (the real performance axis)

Whether you write SQL or a document query is nearly irrelevant to behavior under load. What dominates is the on-disk structure:

DimensionB-tree (InnoDB, Postgres)LSM-tree (Cassandra, RocksDB)
WritesRandom, in-place; page-split write amplificationSequential to memtable then flush; high throughput
ReadsPredictable ~logₙ(n) page readsMay touch several SSTables; Bloom filters mitigate
Worst caseWrite-heavy random keysRead amplification + compaction stalls under write pressure

This is why Cassandra, RocksDB, HBase, ScyllaDB feel alike (all LSM) and MySQL/Postgres feel alike (both B-tree), regardless of API. (Deep dive: Storage Engines, Indexes.)

Axis 3 — Consistency (tunable, not a label property)

"NoSQL sacrifices consistency" is wrong as a blanket claim — consistency is a per-operation knob in serious distributed stores. Cassandra and DynamoDB let you trade latency for guarantees on every read/write (the R + W > N quorum dial). And the premise that you must drop ACID to scale out died with Google Spanner (2012): NewSQL shards data into ranges, runs consensus per range, and layers SQL on top — linear write scaling and serializable transactions. (Full ladder: CAP / PACELC.)

The failure: choosing by the label

The mistake — treating the binary as the decision This is Scenario 1. "NoSQL scales" picks a store by a marketing axis and ignores the query model: relational data in a document store means emulating joins in app code and duplicating data you then can't keep consistent. The inverse errors are just as common — "SQL can't scale" (false since NewSQL) and "NoSQL is faster" (depends entirely on the storage engine and workload). The 2018 "SQL = vertical, NoSQL = horizontal" framing is actively misleading because it collapses the three independent axes above into one. The cure: state the access pattern, scale, and consistency need first; let those pick the engine; never start from the label.

The decision table — reasons to pick each

Reach for…When…Because
SQL (relational)Data is relational; you need ACID; queries aren't all known yetJoins, integrity, query generality — the right default

Self-check — can you reconstruct the three scenarios?

  • Scenario 1 (label trap): Why does "NoSQL scales" fail for relational data, and what three axes does the SQL/NoSQL binary hide?
  • Scenario 2 (ledger): Which property does a ledger need, and why is "you must drop ACID to scale" no longer true?
  • Scenario 3 (sensor firehose): Why does the storage engine (not the label) decide this, and does it point to B-tree or LSM?
  • Name the three independent axes and give a store that mixes them unexpectedly.
  • Why is relational the right default, and what specifically justifies leaving it?

The shape to remember: "SQL vs NoSQL" is really 3 axes — query model, storage engine, consistency — and the failure is choosing by the label instead of the axes.

Famous implementations

Relational — PostgreSQL / MySQL

  • Axes. Relational query model, B-tree engine, strong consistency.
  • Key idea. Joins + ACID + ad-hoc queries; the default OLTP store. Postgres JSONB even absorbs document workloads.
  • The failure to watch. Single-primary write ceiling — past it you shard or move to NewSQL.

NoSQL families — DynamoDB / MongoDB / Cassandra

  • Axes. Key-value/document/wide-column model; LSM engine (mostly); tunable consistency.
  • Key idea. Each nails one access pattern with easy horizontal scale — pick by your dominant query shape.
  • The failure to watch. An unplanned query (off the key) is expensive or impossible; "schemaless" pushes schema into your app.

NewSQL — Spanner / CockroachDB / TiDB

  • Axes. Relational + SQL, distributed storage (Raft/Paxos per range), serializable consistency.
  • Key idea. The "ACID vs scale" trade was a single-node artifact; consensus-per-shard + 2PC removes it — Spanner uses TrueTime, CockroachDB uses hybrid logical clocks.
  • The failure to watch. Cross-region commit latency on every write — don't buy distributed transactions you don't need.

Build it yourself

Decompose a choice into the three axes

Someone says "use MongoDB." Re-express the decision as the three axes — query model, storage engine, consistency — and say which one actually settles it for a given workload.

Reveal the approach

Query model: is access whole-document-by-id (document fits) or relational with joins (it doesn't)? Storage engine: write-heavy random ingest (LSM helps) or read-heavy point/range (B-tree)? Consistency: do you need read-your-writes / multi-doc transactions (Mongo added these, but they're not its strength) or is eventual fine? For most "use MongoDB" proposals, the query model axis settles it: if the data is relational, Postgres (with JSONB for nested bits) beats a document store regardless of the other axes. Naming the axes turns a preference into a defensible decision.

Design-it-yourself

  • Pick stores for one product across workloads. Orders, sessions, search, clickstream — each by its axes. → Choosing a Database.
  • Justify (or kill) NewSQL. When does Spanner/CockroachDB beat a Postgres primary + replicas? → Consensus.
