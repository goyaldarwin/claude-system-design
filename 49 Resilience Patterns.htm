Resilience Patterns

How distributed systems survive partial failure on purpose — timeouts, retries, circuit breakers, bulkheads, backpressure, and the discipline to fail fast instead of cascading.

Designing for failure

At scale, failure is not an exceptional event — it is the steady state. A service with 100 dependencies, each at 99.9% availability, has a theoretical ceiling of ~90.5% availability if every dependency is on the critical path and treated as required. Resilience is the art of breaking that multiplicative trap: making most dependencies optional, isolating the ones that aren't, and ensuring that when a downstream component degrades, the blast radius stays contained instead of propagating upward.

The mental model: every remote call is a potential source of latency, errors, and resource exhaustion. The patterns in this lesson are the standard toolkit for bounding that exposure. They are not independent — they compose. A timeout makes a circuit breaker possible; a circuit breaker makes load shedding meaningful; idempotency makes retries safe. Strong engineers discuss them as a system, not a checklist.

Key insightResilience is fundamentally about converting one failure mode into a less harmful one. A timeout converts an indefinite hang into a fast error. A circuit breaker converts a slow error into an instant one. Load shedding converts total collapse into partial, predictable degradation. You are always trading something.

Timeouts: the foundation

The single most common cause of cascading failure is a missing or unbounded timeout. When a downstream service slows down, callers without timeouts hold their threads, connections, and memory hostage waiting for a response that may never come. Each waiting request consumes a slot in a finite pool. Within seconds the caller's thread pool or connection pool is exhausted, and now the caller is also "down" — even though its own code is healthy. The failure has propagated one hop upstream.

Setting timeouts correctly

  • Base them on the p99 (or p99.9), not the average. A timeout at the mean will time out half your slow-but-valid requests. A common starting point is the downstream's p99.9 latency plus a small margin.
  • Make timeouts shorter as you go up the stack, or use deadline propagation. If the user-facing API has a 2s budget and it calls service B which calls service C, C must finish well within B's remaining budget. Naive per-hop timeouts (each 2s) let a deep call chain blow the user budget many times over.
  • Distinguish connect timeout from read/request timeout. A connect timeout should be tight (tens to low hundreds of ms on a fast network); the request timeout covers actual work.
What strong engineers doThey reach for deadline propagation rather than fixed per-hop timeouts. The top-level request stamps an absolute deadline (e.g. deadline = now + 2s), passes it through headers or gRPC context, and every hop computes its remaining budget: remaining = deadline - now. If remaining is near zero, the hop fails fast instead of starting doomed work. gRPC does this natively; see API Design: REST, gRPC, GraphQL.
// Deadline propagation, conceptual
func handle(ctx Context, req Request) Response {
    // ctx carries an absolute deadline set by the edge
    remaining := ctx.Deadline().Sub(now())
    if remaining < minUsefulWork {
        return errDeadlineExceeded   // fail fast, don't start work we can't finish
    }
    // pass a *tightened* deadline downstream
    childCtx := ctx.WithTimeout(remaining - networkBudget)
    return downstream.Call(childCtx, req)
}
PitfallA timeout without an accompanying cancellation only protects the caller, not the callee. If you time out but the downstream keeps processing the now-abandoned request, you have wasted work piling up on an already-struggling service — you have made the overload worse. Propagate cancellation so the downstream can stop.

Retries, backoff, jitter, and the retry storm

Retries paper over transient failures — a dropped packet, a brief GC pause, a rolling restart. They are also one of the most dangerous patterns because they amplify load exactly when a system is least able to handle it.

The retry storm / thundering herd

Suppose a downstream service briefly degrades and starts returning errors. Every caller retries 3 times. Instantaneous load on the recovering service is now 4× normal. The extra load keeps it pinned, causing more errors, causing more retries — a self-reinforcing retry storm that turns a 2-second blip into a 20-minute outage. If every client also retries on the same schedule, they synchronize into a thundering herd that hammers the service in waves.

The required mitigations

  1. Exponential backoff: wait base * 2^attempt between retries, so load decays instead of staying constant.
  2. Jitter: randomize the backoff so clients de-synchronize. Full jitter — sleep = random(0, base * 2^attempt) — is the AWS-recommended default and outperforms equal/decorrelated jitter in most simulations.
  3. A retry budget / token bucket: cap retries to a fraction (e.g. 10%) of successful requests so a sustained outage can't multiply load. This is what gRPC's retry throttling and Envoy's retry budgets implement.
  4. Retry only idempotent or retry-safe operations (see idempotency), and only on retryable error classes (timeouts, 503, connection resets — not 400, 404, or 409).
// Exponential backoff with full jitter + retry budget
attempt := 0
for attempt <= maxAttempts {
    if !retryBudget.tryAcquire() { break }   // stop amplifying load under sustained failure
    resp, err := call()
    if err == nil || !isRetryable(err) { return resp, err }
    backoff := min(maxBackoff, base * (1 << attempt))
    sleep(random(0, backoff))                // FULL JITTER
    attempt++
}
PitfallRetries at every layer of a stack multiply. If the edge retries 3×, the mid-tier retries 3×, and the data layer retries 3×, a single user request can become 27 downstream requests during a partial outage. Decide on one layer that owns retries (usually closest to the failure, or the edge) and disable them elsewhere. This is the most underappreciated rule in retry design.

Circuit breakers

A circuit breaker stops sending requests to a dependency that is clearly failing, giving it room to recover and letting the caller fail fast. It is a state machine with three states:

StateBehaviorTransition
ClosedRequests flow normally; failures are counted (error rate over a rolling window).If error rate or count crosses threshold → Open.
OpenRequests fail immediately (no call made) — fast failure, no thread held, downstream gets relief.After a cooldown timer (e.g. 5–30s) → Half-open.
Half-openA limited number of trial requests are allowed through to probe recovery.Trials succeed → Closed. Any trial fails → back to Open.
Key insightA circuit breaker's value is failing fast. When open, the caller doesn't waste a timeout-worth of latency and a held thread on every request — it returns instantly (ideally to a fallback). This is what prevents thread-pool exhaustion from propagating. Trip on error rate over a rolling window, not raw counts, so low-traffic dependencies don't trip on a single error and high-traffic ones don't tolerate thousands.
PitfallTuning is hard and breakers can cause outages. A threshold that's too sensitive trips during normal latency variance and takes down a healthy dependency. A breaker shared across all clients of a service means one misbehaving caller's failures can open the circuit for everyone (an argument for per-client or per-endpoint breakers). And a breaker that opens during a partial outage can deny service to the healthy partition. Always pair with a fallback and good metrics.

Bulkheads and resource isolation

Named after a ship's watertight compartments: if one floods, the others keep the ship afloat. In software, a bulkhead partitions a finite resource (threads, connections, memory) so that exhaustion in one area can't sink the whole process.

  • Per-dependency thread/connection pools. Give each downstream its own bounded pool. If dependency X hangs, only X's pool exhausts; calls to Y, Z, and local-only endpoints keep flowing. Hystrix popularized this; a shared global pool is the classic anti-pattern that turns one slow dependency into total outage.
  • Per-tenant / per-customer isolation. A noisy tenant generating 10× load shouldn't starve everyone else. Partition capacity or use cell-based architecture (shuffle sharding) so each customer maps to a subset of capacity and a bad actor's blast radius is bounded.
  • Separate critical from non-critical paths. Don't let analytics writes share a connection pool with checkout. See Designing a Payment System for where this matters most.
Trade-offBulkheads improve isolation but reduce statistical multiplexing efficiency. Ten dedicated pools of 10 connections can't absorb a burst the way one shared pool of 100 can — you may reject requests while idle capacity sits in another partition. You're trading peak utilization for fault isolation. Size partitions for the dependency's real concurrency, and consider semaphore isolation (cheap, no extra threads) for low-latency dependencies where a thread pool's overhead isn't worth it.

Backpressure and load shedding

When demand exceeds capacity, a system has three choices: queue the excess, slow the producers, or drop work. Doing nothing — unbounded queuing — is the path to collapse.

Backpressure

Backpressure propagates "slow down" signals back to the source. TCP flow control is the canonical example. Reactive frameworks (Reactive Streams, gRPC flow control) make it explicit: a consumer requests N items, and the producer must not exceed that. The goal is to keep work in bounded buffers and let the slowest stage govern the rate, rather than letting fast producers build an unbounded backlog that eventually OOMs a consumer. This connects directly to Queues and Kafka, where bounded queues and consumer lag are the explicit backpressure mechanism.

Load shedding

When backpressure can't reach the source (e.g. external user traffic), the server must shed load — reject excess work fast so the requests it does accept succeed. The cardinal rule:

Key insightA system at overload should return a fast 503, never a slow timeout. Rejecting 30% of requests in 1ms so the other 70% succeed in 50ms is vastly better than accepting 100% and having all of them time out at 10s. Goodput (useful work completed) collapses when you try to serve everyone — the throughput-vs-offered-load curve bends downward past the knee. Load shedding keeps you at the knee.
  • Shed at the edge / load balancer when possible — cheapest place to drop. See Load Balancing.
  • Prioritize: shed low-value traffic first (background sync, retries, free tier) and protect high-value traffic (checkout, paying customers). Carry a priority class on requests.
  • Use queue length and latency as the signal. Adaptive techniques like controlled delay (CoDel) or Netflix's concurrency-limit (Little's Law–based) shed when latency climbs, without needing a hand-tuned RPS cap.

Rate limiting and throttling as protection

Rate limiting is load shedding made deliberate and contractual. It protects a service from being overwhelmed by any single client — whether malicious, buggy, or just successful. Common algorithms: token bucket (allows bursts up to bucket size, refills at a steady rate), leaky bucket (smooths to a constant rate), fixed window (simple but boundary-spiky), and sliding window (smoother, more state). The full design is in Designing an API Rate Limiter.

  • Throttling (returning 429 with Retry-After) is the polite cousin — it tells well-behaved clients to back off rather than dropping silently.
  • Per-client, per-endpoint, and global limits compose: a per-client limit enforces fairness; a global limit protects the service's absolute capacity.
  • Distributed rate limiting needs shared state (often Redis with a token-bucket Lua script) to enforce a global limit across many instances — a consistency vs. latency trade-off (see CAP Theorem).

Graceful degradation and fallbacks

When a dependency is unavailable (circuit open, timeout, shed), the question is: what do we serve instead? Graceful degradation means the system delivers reduced but useful functionality rather than an error page.

  • Serve stale-but-valid cached data (stale-while-revalidate) instead of failing. See Caching.
  • Return a sensible default — e.g. an empty recommendations list, generic ranking, or a "personalization unavailable" experience — rather than 500.
  • Disable non-essential features under load (feature flags), shedding the expensive optional work to protect the core flow.
  • Degrade write paths to async: accept the write into a durable queue and process later, instead of failing synchronously.
What strong engineers doThey classify every dependency as critical or optional up front. Optional dependencies must have a fallback so their failure can't take down the request. Critical dependencies get the heaviest investment in redundancy and isolation. This classification is the difference between a service that degrades gracefully and one that collapses on the first downstream hiccup.

Idempotency: the prerequisite for safe retries

Retries are only safe if re-executing an operation is harmless. Idempotency means applying an operation N times has the same effect as applying it once. Reads and overwrites (PUT name = "x") are naturally idempotent; increments, appends, and "create a new order" are not.

The standard mechanism is an idempotency key: the client generates a unique key per logical operation and sends it with every retry. The server records the key and the result; on a duplicate key it returns the stored result instead of re-executing.

// Idempotent create with a client-supplied key
func createPayment(key string, req PaymentReq) Result {
    if existing := store.get(key); existing != nil {
        return existing            // duplicate retry -> return prior result, no double-charge
    }
    res := process(req)
    store.putIfAbsent(key, res)    // atomic; handles concurrent duplicates
    return res
}
PitfallThe check-and-store must be atomic and durable, or two concurrent retries both pass the "does the key exist?" check and both execute — a double charge. Use a unique constraint, conditional write (putIfAbsent / INSERT ... ON CONFLICT), or a distributed lock (see Distributed Locking Service). This is exactly why payment systems treat idempotency as a first-class API contract.

The anatomy of cascading failure

Cascading failure is the macro-pattern these tactics defend against. The canonical sequence:

  1. One node or dependency slows down or dies (GC pause, deploy, hot shard).
  2. Its share of load redistributes to surviving nodes, pushing them past capacity.
  3. Surviving nodes slow down; callers without timeouts hold threads; retries amplify load.
  4. More nodes tip over. The healthy capacity that remains is now also overwhelmed — the system can't recover even after the original trigger is gone.

The insidious part is step 4: a metastable failure. The system is stuck in a bad equilibrium sustained by its own retry and load-redistribution feedback loops. Removing the original trigger doesn't help — you have to break the loop by shedding load, draining queues, or restarting with rate limits.

Key insightEvery pattern in this lesson breaks the cascade at a specific link: timeouts stop thread exhaustion from propagating; circuit breakers stop hammering a downed dependency; retry budgets + jitter stop load amplification; bulkheads contain the blast radius; load shedding keeps surviving capacity productive. Recovery from a metastable state almost always requires deliberately reducing load below normal until the system re-stabilizes.
PitfallBeware self-inflicted cascades: a thundering herd of cache misses after a flush (cache stampede — mitigate with request coalescing and staggered TTLs), or synchronized health-check failures that eject every node at once. Auto-scaling can also amplify a cascade if new instances need to warm caches or connect to an already-saturated database.

Chaos engineering to validate

Resilience patterns that aren't continuously exercised rot. A timeout config drifts, a fallback path breaks silently because no one tests it, a circuit breaker's threshold becomes wrong as traffic grows. Chaos engineering is the practice of injecting controlled failure in production (or production-like) environments to verify the system behaves as designed.

  • Start with a hypothesis and a steady-state metric (e.g. "checkout success rate stays >99.9% if recommendation service is killed"). Inject the failure; confirm the metric holds.
  • Inject realistic failures: latency, errors, instance termination, network partition, dependency unavailability, clock skew (relevant to Time, Clocks, and Ordering), region loss (see Multi-Region and DR).
  • Limit blast radius: start in staging, then a small production cell, with an automatic abort if the steady-state metric breaches. Netflix's Chaos Monkey / ChAP and Gremlin operationalize this.
  • You cannot validate resilience without observability. You need the metrics, traces, and SLO dashboards (see SLI/SLO/SLA) to even know whether an experiment passed.
What strong engineers doThey treat the fallback path as a first-class, continuously-tested code path, not an emergency hatch. The most common production surprise is discovering the fallback was itself broken — it called the same down dependency, or threw on the degraded input. GameDays and automated chaos experiments are how you find that before the incident.

Principal-level mastery check

What strong engineers doThey treat resilience as a system of composing trade-offs, not a checklist of buzzwords. They name the specific failure mode each pattern addresses, reason about how patterns interact (retries are dangerous without idempotency and budgets; circuit breakers are useless without fallbacks), and they always ask "what's the blast radius?" They distinguish critical from optional dependencies, default to deadline propagation over per-hop timeouts, pick one layer to own retries, and insist that fallback paths and timeout configs be continuously validated via chaos testing. Crucially, they understand metastable failure: that recovery often requires deliberately shedding load below normal.
  • A downstream dependency's p99 latency jumps from 50ms to 5s but it never returns errors. Walk me through exactly how this takes down your service, and which single pattern would have prevented it. (Listening for: thread/connection-pool exhaustion from missing timeouts; why a circuit breaker that trips only on errors wouldn't fire here; latency-based shedding or slow-call thresholds.)
  • You add retries with exponential backoff to improve reliability and availability gets worse during partial outages. Why, and how do you fix it without removing retries? (Retry storm / amplification; jitter; retry budgets; retries multiplying across layers; retrying only retryable, idempotent operations.)
  • Your service recovered from the triggering event 20 minutes ago but is still failing. What's happening and how do you get out? (Metastable failure; self-sustaining retry/load loops; recovery requires shedding load below normal, draining queues, rate-limited restart.)
  • How do you set timeouts across a 4-hop call chain so a 2-second user budget is never exceeded, and so a slow service doesn't waste downstream work? (Deadline propagation vs. fixed per-hop timeouts; remaining-budget computation; cancellation propagation so abandoned work stops.)
  • When is a bulkhead the wrong choice, and what does it cost you? (Loss of statistical multiplexing; rejecting requests while idle capacity sits in another partition; sizing; semaphore vs. thread-pool isolation for low-latency deps.)
  • How would you prove your resilience design actually works, and what's the most likely thing you'd discover is broken? (Chaos engineering with steady-state hypotheses and bounded blast radius; the fallback path being silently broken; dependence on observability to evaluate experiments.)
