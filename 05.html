Sharding / Data Partitioning

One machine eventually can't hold the data or absorb the writes. Sharding splits the data across many machines — and in exchange you give up cheap joins, easy transactions, and the freedom to query any way you like. Every decision here is about which of those you sacrifice, and how you survive moving the data while traffic is live.

The trade-off Unlimited horizontal capacity in exchange for cross-shard joins, transactions, and rebalancing pain. Splitting data across nodes lets you scale writes and storage past one machine. In return you accept that operations spanning shards are slow or impossible, and that changing the layout under live traffic is one of the hardest things you'll do.

What sharding is

Sharding (a.k.a. horizontal partitioning) splits one logical dataset into many smaller pieces — shards — each held on a different machine, so that no single machine has to store all the data or serve all the traffic. It's the answer when a single primary plus read replicas can no longer hold the working set or absorb the write rate.

It's worth separating two axes people conflate:

  • Horizontal partitioning (sharding): different rows on different machines (users A–M here, N–Z there). This is what "sharding" almost always means.
  • Vertical partitioning: different columns/features on different machines (profiles here, photos there). Simple, but each feature can still outgrow one machine.
The one-line frameA shard map is not a static decision — it's a control plane that must move data without violating consistency or blowing your latency budget. Most designs that look fine on a whiteboard fall apart at "how do you change it while live?" Design that first.

Three scenarios — decide before you read

Don't scroll past these. Decide your answer first — the rest of the lesson is the answer. If you already know all three cold, skip to the decision table.

Scenario 1 — the ninth node You map keys with shard = hash(key) % 8 across 8 nodes. Black Friday is coming, so you add a 9th node to relieve load. What fraction of your keys now live on the wrong node, and what does that do to your database and cache the instant you deploy?
Scenario 2 — the celebrity shard You shard users by hashing the user ID — a clean, even distribution. Then one user gets 50 million followers and every action they take is read millions of times an hour. Your ring is perfectly balanced, so why is one node on fire while the rest idle — and can adding nodes fix it?
Scenario 3 — the query you didn't shard for Your data is sharded by user ID. Product asks for "all orders placed in the last hour, across all users." Your shard key is the user, not the time — so how does this query run, and what does it cost?

Scenario 1 is the resize reshuffle (the hash%N trap). Scenario 2 is the hot shard. Scenario 3 is the scatter-gather cost of non-key queries. Each maps to a decision below.

Sharding is defined by four decisions

Every sharded system — Vitess, Cassandra, DynamoDB — is just these four choices. Make each well and it scales smoothly; make one wrong and you get a named, predictable failure, shown nested beneath it.

Decision 1 — How do you map a key to a shard?

Given a key, which shard owns it? Four schemes:

  • Hash-based — shard = hash(key) % N. Even distribution by construction, but (see below) ties the mapping to the node count.
  • Range-based — contiguous key ranges per shard (ZIP < 10000 here, ≥ 10000 there). Great for range scans, prone to skew and hot spots.
  • Directory-based — an explicit lookup service maps each key (or key range) to a shard. Maximum flexibility (move any partition anywhere), at the cost of a lookup hop and a critical piece of infrastructure.
  • List / composite — assign explicit value sets (Nordic countries → one shard), or combine schemes. Consistent hashing is effectively hash + list.
The failure that happens when you get this wrong — the resize reshuffle This is Scenario 1. hash(key) % N ties every key's home to the cluster size N. Go from 8 nodes to 9 and the modulus changes for nearly every key — empirically about (N−1)/N of all keys remap (~89% at N=9). A near-total reshuffle saturates the network and cache exactly when you were adding capacity to relieve pressure — a positive-feedback failure. Two fixes dominate:
  • Consistent hashing with virtual nodes — keys and nodes sit on a ring; adding a node only steals keys from its neighbors, so only ~1/N of keys move. Vnodes (Cassandra historically 256, newer 16) smooth the otherwise ~O(log N) imbalance.
  • Fixed partition count (decoupling) — pick a large fixed number of partitions up front (1024, 4096), far more than nodes, and assign whole partitions to nodes. Rebalancing becomes "move partition P from A to B" — an integer reassignment, never a rehash. This is how Kafka, Elasticsearch, and Instagram (below) do it.
The heuristic: never let the number of partitions equal the number of nodes. Partitions are a logical unit you rebalance; nodes are physical capacity you scale.

Decision 2 — What do you shard on? (choosing the key)

The shard key is the most consequential choice, and hash-vs-range isn't symmetric:

DimensionHash partitioningRange partitioning
DistributionEven by constructionProne to skew; needs split/merge
Range scans (WHERE ts BETWEEN…)Scatter-gather across all shardsHits one / few contiguous shards
Hot-spot riskLow for random keysHigh for monotonic keys (timestamps, autoinc IDs)
Used byDynamo, CassandraHBase, Bigtable, CockroachDB, Spanner

The classic failure: range-partitioning on a timestamp gives clean range scans but funnels every write into the "latest" partition — one hot region. Hashing kills the hot spot but makes a recent-events query a scatter-gather. CockroachDB and Bigtable split the difference with automatic split-on-load.

The failure that happens when you get this wrong — the hot shard This is Scenario 2. Hashing distributes keys evenly, but it cannot split a single hot key — a celebrity user, a viral object, a tenant 100× the median. A balanced ring is irrelevant when all the traffic is for one key on one node. Mitigations, in order of escalation:
  • Key salting — append a bounded suffix (key#0..key#9) to spread one logical key across 10 shards. Writes fan out; reads must gather all 10 — so reserve it for known hot keys.
  • Split the hot partition — detect a partition over a load threshold and split its range (needs range partitioning or a split-aware directory).
  • Front cache / request coalescing — for read-hot keys, a small cache absorbs the storm without touching the layout (often the cheapest fix — see Caching).
  • Tenant isolation — pin oversized tenants to dedicated shards so they can't starve neighbors.

Decision 3 — How do queries that aren't on the shard key work?

Once data is sharded by primary key, "find all rows where color = red" becomes a distributed problem with two designs:

  • Local (document-partitioned) secondary index — each shard indexes only its own rows. Writes are cheap (one local update); reads must fan out to every shard and merge (scatter-gather), because matches can live anywhere. Elasticsearch and Cassandra default to this. Good for write-heavy with a tolerable read tail.
  • Global (term-partitioned) secondary index — the index is itself partitioned by the indexed term, so all color = red entries sit together and a read hits one shard. Reads are fast; writes pay (one row update may touch several index partitions), so the global index is usually asynchronous and eventually consistent. DynamoDB's GSIs are exactly this.
The failure that happens when you get this wrong — scatter-gather This is Scenario 3. A query whose filter isn't the shard key can't be routed to one shard, so it fans out to all of them and merges — latency now tracks your slowest shard, and throughput drops as every query touches every node. The choice is which operation pays the cross-shard cost: local index = cheap writes, expensive scatter-gather reads; global index = fast reads, expensive (eventually-consistent) distributed writes. Also: cross-shard joins and transactions are slow or impossible — you denormalize (and accept the inconsistency) or enforce integrity in application code.

Decision 4 — How do you reshard a live system?

The endgame: changing the shard map under production traffic, with zero downtime. The proven playbook (Vitess resharding, MySQL-at-scale, DynamoDB adaptive capacity) is a dual-write / backfill / cutover dance:

  1. Double-write — every write goes to both old and new layouts; the app tolerates either map.
  2. Backfill historical data old→new in the background, throttled to protect live traffic; track a high-water mark.
  3. Verify by comparing checksums/row counts per key range; reconcile drift from races.
  4. Cutover reads partition-by-partition — flip the directory one key range at a time so you can roll back a single range.
  5. Stop double-writing and decommission the old shards only after a safe soak.
The failure that happens when you get this wrong — downtime or data loss Naive resharding (stop writes, copy data, switch over) means an outage proportional to your data size — hours for terabytes. Skip the verify step and races during backfill silently drop or duplicate rows. The dual-write playbook exists precisely so the cutover is incremental and reversible.
How Instagram made resharding painless — logical shards. Instagram put thousands of "logical shards" (4,096), each just a Postgres schema, onto a smaller number of physical servers. Because a logical shard is a schema, moving one to a new server is a standard pg_dump / restore + a config change — no rehashing, no application change. Their IDs encode the shard: a 64-bit ID = 41 bits timestamp + 13 bits shard ID + 10 bits sequence (1,024 IDs per shard per millisecond), so a row's ID tells you its shard directly. This is the "fixed partition count, decoupled from nodes" fix from Decision 1, made concrete. (Source: Instagram Engineering.)

Self-check — can you reconstruct the three scenarios?

Close the page and rebuild them — that's the test, not recognition.

  • Scenario 1 (ninth node): Why does hash%N remap ~89% of keys when you go 8→9, and how do consistent hashing and fixed-partition-count each avoid it?
  • Scenario 2 (celebrity): Why can't hashing split a single hot key, why doesn't adding nodes help, and which mitigation would you try first?
  • Scenario 3 (non-key query): Why does a query off the shard key become scatter-gather, and what's the local-vs-global secondary-index trade-off?
  • Why should the number of partitions never equal the number of nodes?
  • Walk the five steps of a zero-downtime reshard, and say what the verify step protects against.
  • Why are cross-shard joins and transactions slow or impossible, and what do you do instead?

The shape to remember: sharding is 4 decisions — key→shard mapping, the shard key, non-key queries, live resharding — with named failures: resize reshuffle, hot shard, scatter-gather, downtime.

Famous implementations

Same facets each — what it is, mapping scheme, key strength, the failure to watch.

Vitess (sharded MySQL)

  • What it is. A sharding middleware over MySQL (born at YouTube; runs Slack, GitHub). Presents many MySQL shards as one logical database.
  • Mapping. Directory-based: a topology service + VSchema map keyspace IDs to shards; the query router (vtgate) fans out and merges.
  • Key strength. Online resharding is a first-class feature — the dual-write/backfill/cutover dance is built in and battle-tested.
  • The failure to watch. The topology service is critical infrastructure; cross-shard queries still scatter-gather and cross-shard transactions are limited.

Cassandra

  • What it is. A wide-column store that shards by consistent hashing with virtual nodes; no central coordinator.
  • Mapping. Hash (Murmur3 token) of the partition key onto a ring; vnodes smooth distribution.
  • Key strength. Linear write scale, multi-region, no SPOF; great for even, write-heavy workloads.
  • The failure to watch. A bad partition-key choice (low cardinality, or a hot key) creates an unfixable hot shard; secondary indexes are local (scatter-gather).

DynamoDB

  • What it is. AWS's managed key-value/wide-column store; sharding and rebalancing are fully hidden.
  • Mapping. Hash of the partition key to internal partitions; AWS auto-splits and rebalances (adaptive capacity) for you.
  • Key strength. You never operate the shard map; it absorbs hot partitions automatically up to limits.
  • The failure to watch. You must design the partition key and GSIs around your queries up front; an unplanned access pattern is expensive or impossible, and GSIs are eventually consistent.

Build it yourself

Work each before opening the approach.

Implement key→shard mapping three ways

Map a key to one of N shards using (a) hash%N, (b) a consistent-hash ring, (c) a fixed-partition table. For each, what happens to the key→shard mapping when you add one node?

Reveal the approach

(a) hash(key) % N: O(1), zero state — but adding a node changes N, remapping ~(N−1)/N of keys. (b) Consistent hash: place nodes (and vnodes) on a ring by hashing their IDs; a key belongs to the next node clockwise. Adding a node inserts it on the ring and steals only the arc between it and its predecessor — ~1/N of keys move. (c) Fixed partitions: partition = hash(key) % P with P fixed and large (e.g. 4096), plus a small table mapping partition→node. Adding a node reassigns some partitions (a table edit); keys never rehash. The lesson: decouple the logical partition (stable) from the physical node (scales).

Design a zero-downtime reshard

You're moving from 4 shards to 8 under live traffic. Sketch the sequence that never drops a write and lets you roll back if verification fails.

Reveal the approach

Dual-write to old+new layouts so no write is lost during the move; backfill history old→new in the background, throttled, tracking a high-water mark; verify by checksumming row ranges and reconciling drift; cut reads over one key range at a time via the directory so a failed range rolls back alone; stop double-writing and retire old shards only after a soak. The two non-negotiables: double-write (no lost writes) and per-range cutover (reversible).

Design-it-yourself: the open-ended ones

  • Pick a shard key for a chat app. Messages, by conversation or by user? What query gets expensive either way? → Messenger.
  • Handle a known hot tenant. One enterprise customer is 100× the median. Keep them from starving everyone else. → Caching, tenant isolation.
  • Add search by a non-key attribute. Your store is sharded by ID; product wants filter-by-status. Local or global index — defend the choice. → Elasticsearch.
