Choosing a Database

A decision framework for picking a datastore by access pattern, consistency, scale, and operational fit — not by familiarity.

The bottleneck Method. choosing a store by which bottleneck dominates your access pattern — not by familiarity.

The decision dimensions

Choosing a database is the single most expensive-to-reverse decision in most systems. Data has gravity: once a few terabytes and a hundred services depend on a store's query semantics, migrating off it is a multi-quarter project. A principal engineer treats the choice as a structured trade-off across a fixed set of dimensions, not as a preference. Walk every candidate store through the same checklist.

DimensionWhat to askWhy it dominates the choice
Data model fitIs the data relational, a blob keyed by id, a nested document, a wide sparse row, a graph, free text, or a time series?A mismatch here forces you to fight the engine forever — joins emulated in app code, or denormalized copies you must keep in sync.
Access patternRead- vs write-heavy? Point lookup, range scan, full-text, or analytical aggregation?Storage engines are optimized for one shape. See Storage Engines: LSM vs B-Tree.
ConsistencyDo you need read-your-writes, linearizability, or is eventual fine?Strong consistency costs latency and availability under partition. See Consistency & PACELC.
ScaleHow big does data and throughput get? Read-scale via replicas, or write-scale via sharding?Read replicas are cheap; horizontal write scaling means a partition key on day one. See Sharding.
Latencyp99 target — sub-ms, single-digit ms, or seconds-OK analytics?In-memory beats on-disk beats cross-region quorum by orders of magnitude.
DurabilityCan you lose the last second of writes? Sync vs async replication, fsync policy.Durability and write latency trade directly against each other.
Transactions / isolationDo you need multi-row ACID, snapshot isolation, serializability?Multi-key transactions across shards are expensive and not all stores offer them.
Operational maturityDoes your team know how to run it? Managed offering? Backup, upgrade, observability story?The best-fit engine you cannot operate at 3 a.m. is the wrong choice.
Sequence mattersResolve the dimensions in roughly this order: model fit and access pattern eliminate most engines; consistency and scale narrow the survivors; latency, durability, and transactions break ties; operational fit is the final veto. Never start from operational fit (“we already run Postgres”) — that is the familiarity trap dressed up as pragmatism.

Data model fit

The data model is the first filter because it is the hardest to work around. Map your dominant entity and its access shape to a model family:

ModelShape it nailsCanonical access pattern
RelationalEntities with relationships, ad-hoc queries, strong invariantsJoins, transactions, secondary indexes, aggregation
Key-valueOpaque value addressed by a known keyO(1) point get/put by primary key
DocumentSelf-contained nested aggregates read as a unitFetch/update one document by id; some secondary indexing
Wide-columnMassive write volume keyed by a partition + clustering keyRange scan within a partition; write-optimized
GraphDensely connected data, multi-hop relationshipsTraversals: “friends of friends who like X”
SearchFree text, relevance ranking, facetingInverted-index full-text and aggregation queries
Time-seriesAppend-only timestamped metricsRange scans by time window, downsampling, rollups

The deeper SQL-vs-NoSQL trade-offs (schema flexibility, join cost, consistency defaults) are covered in SQL vs. NoSQL. The short version: relational gives you query flexibility and integrity at the price of harder horizontal write scaling; the NoSQL families each trade query generality for one access pattern done extremely well.

Family-by-family guide

Relational — PostgreSQL, MySQL

Nails: ad-hoc queries, joins across normalized entities, multi-row ACID transactions, strong invariants enforced by constraints.

  • Use when the data is relational and you do not yet know every query — the relational model preserves flexibility. This should be the default for most OLTP systems until a dimension proves otherwise.
  • Avoid when a single table’s write throughput exceeds what one primary plus read replicas can absorb and the working set no longer fits on one node — you then need application-level sharding or a NewSQL store. Postgres in particular is rich (JSONB, full-text, PostGIS, partitioning) and frequently postpones the need to add another engine.

Key-value — Redis, DynamoDB

Nails: sub-millisecond point reads/writes by a known key, predictable horizontal scale.

  • Use when access is purely “give me the value for this key” — sessions, caches, feature flags, rate-limit counters (Redis), or a high-scale partition-keyed workload with bounded query shapes (DynamoDB).
  • Avoid when you need to query by attributes other than the key or do ad-hoc joins. DynamoDB punishes unplanned access patterns; you must design the keys and GSIs around queries up front. Redis is also memory-bound — not your system of record unless persistence and cost are acceptable.

Document — MongoDB

Nails: aggregates that are read and written as a whole (a product with its variants, an order with line items), with a flexible evolving schema.

  • Use when the natural unit of access is one self-contained document and write/read patterns rarely cross document boundaries. Good for content, catalogs, user profiles.
  • Avoid when your data is highly relational and you find yourself emulating joins in app code or duplicating data across documents you then struggle to keep consistent. “Schemaless” does not mean “no schema” — it pushes schema enforcement into your application.

Wide-column — Cassandra, HBase

Nails: enormous write volume, linear horizontal scale, multi-region availability with tunable consistency.

  • Use when you have write-heavy, append-style workloads at scale — event logs, sensor data, messaging, time-ordered feeds — and you can model the table around the exact query (partition key + clustering columns). Cassandra is AP-leaning with tunable quorums; it survives node and region loss gracefully.
  • Avoid when you need ad-hoc queries, joins, or strong multi-row transactions. You must design one table per query pattern and accept denormalization; an unplanned query may be impossible without a full scan.

Graph — Neo4j

Nails: deep multi-hop traversals over densely connected data, where relationships are first-class.

  • Use when the questions are inherently about connections — fraud rings, social graphs, recommendations, dependency/lineage analysis — and the same query in SQL would mean many self-joins of unknown depth.
  • Avoid when connections are shallow (one or two hops) or relationships are incidental; a relational store with a join table is simpler, cheaper, and easier to operate. Graph databases are also harder to shard.

Search — Elasticsearch

Nails: full-text relevance ranking, faceting, fuzzy matching, and flexible aggregation over an inverted index.

  • Use when users type queries and expect ranked, typo-tolerant results, or you need analytics dashboards over semi-structured logs. Almost always a secondary index fed via change data capture, not the source of truth. See Search & Elasticsearch.
  • Avoid when you need it as a primary transactional store — it is near-real-time (refresh delay), lacks robust multi-document transactions, and its consistency model is not built for system-of-record duty.

Time-series — InfluxDB, Prometheus

Nails: high-volume append-only timestamped measurements, time-window range scans, downsampling and retention.

  • Use when data is metrics or events indexed primarily by time — monitoring, IoT telemetry, financial ticks. These engines give you cheap rollups and TTL-based expiry for free. See Designing a Metrics & Monitoring System.
  • Avoid when the access pattern is not time-centric, or you need to mutate historical rows — time-series engines assume an immutable append workload. Watch the cardinality: high-cardinality label sets (per-user or per-request IDs) blow up the in-memory index of engines like Prometheus and InfluxDB — it is their classic failure mode, not a strength (see Metrics & Monitoring).

NewSQL — Spanner, CockroachDB

Nails: the relational model and SQL plus horizontal write scaling and strong (often serializable) consistency across shards.

  • Use when you genuinely need both transactional integrity at relational-query richness and scale beyond a single primary, often with multi-region strong consistency — financial ledgers, global inventory.
  • Avoid when a single Postgres primary with read replicas would do. NewSQL adds cost, operational complexity, and cross-region commit latency (a consensus round-trip per write, plus two-phase commit on top of consensus for multi-shard transactions — see Consensus & Raft). Do not pay for distributed transactions you do not need.
The recurring trade-offEvery NoSQL family wins its niche by giving up query generality. Relational and NewSQL keep generality but pay in write-scaling effort (relational) or latency and cost (NewSQL). There is no free lunch: you are choosing which flexibility to sacrifice, not whether to sacrifice any.

Polyglot persistence & choosing per service

Polyglot persistence is the practice of letting each service own the datastore that fits its access pattern, rather than forcing one database to serve the whole company. A product-catalog service might use Postgres; the session store, Redis; the activity feed, Cassandra; the search box, Elasticsearch fed by CDC; the dashboards, a time-series store.

This aligns naturally with microservice boundaries — each service owns its data and exposes it only through its API, so the store is an implementation detail. But polyglot is not free:

  • Every additional engine is a new thing to operate, monitor, back up, patch, and staff expertise for.
  • Keeping derived stores (search indexes, caches, read models) in sync with the source of truth requires CDC or event pipelines, with their own consistency lag.
  • Cross-store transactions do not exist — you fall back to sagas and eventual consistency.
Discipline over enthusiasmAdopt a new datastore only when an existing one cannot meet a dimension that actually matters for that service. Three well-run engines beat seven half-understood ones. The cost of polyglot is operational sprawl; pay it only where the access-pattern win is real.

The familiarity trap

The most common failure mode — in real architecture reviews — is choosing by familiarity: “I’ll use MongoDB because I know it,” or reaching for Cassandra because it sounds web-scale. Familiarity is a legitimate tiebreaker under operational fit, but it must never be the first filter. The tell is an engineer who names the database before describing the access pattern.

The inverse failure is resume-driven design: picking the trendiest store to learn it on the company’s dime. Both errors skip the framework. The cure is the same: state the access pattern, consistency need, and scale first; let those eliminate engines; then let familiarity and managed-service availability break the remaining tie. A principal engineer can defend the choice by walking the dimensions, not by citing comfort.

A worked decision example

Suppose you are designing storage for an e-commerce platform. Rather than picking one database, decompose by workload and run each through the framework:

WorkloadAccess patternConsistencyChoiceWhy
Orders & paymentsMulti-row transactions, ad-hoc queriesStrong / serializablePostgreSQL (or CockroachDB if global)ACID, integrity, joins; default until scale forces NewSQL
Product catalogRead whole product aggregate by idRead-your-writes OKDocument store / Postgres JSONBSelf-contained nested aggregate; JSONB keeps it inside the relational default
Sessions & cartPoint get/put by user key, sub-msEventual fineRedisIn-memory KV, TTL expiry
Search boxFull-text, faceted, rankedNRT lag acceptableElasticsearch (CDC-fed)Inverted index, relevance
Clickstream / activityHuge write volume, time-orderedEventualCassandraWrite-optimized, linear scale
Ops metricsTime-window scans, rollupsEventualPrometheus / InfluxDBTime-series engine

Notice the source of truth (orders) is relational and strongly consistent; everything else is either a derived store (search) or a workload-specific optimization (sessions, clickstream, metrics). The decision was driven dimension-by-dimension, and each choice is defensible in one sentence. That is the standard.

Principal-level mastery check

What strong engineers doThey state the access pattern, consistency need, and scale before naming any database, then let those constraints eliminate engines. They default to relational and justify every move away from it. They reach for polyglot only when a dimension demands it, and they can name the operational cost they are taking on.

Probing questions to ask engineers:

  • “You picked MongoDB — what about this access pattern makes a document store better than Postgres with JSONB here?”
  • “What query would be slow or impossible on the store you chose, and how would you handle it when product asks for it?”
  • “You said Cassandra for scale — what’s your partition key, and what happens to a query that doesn’t include it?”
  • “This is your system of record. Walk me through the consistency and durability guarantees on a write. Can you lose the last second of data?”
  • “You introduced a fourth datastore. What’s the operational cost, and who keeps the derived store in sync with the source of truth?”
  • “If you had to ship with only one database, which would you drop first and why?”
