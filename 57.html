Designing a Key-Value Store

A distributed key-value store (think DynamoDB/Cassandra) must hold more data than one machine, survive node failures, and stay available — with simple get/put on a key. Designing one is a tour of the whole field: consistent hashing for placement, replication + quorums for durability and tunable consistency, anti-entropy to heal divergence, and the CAP choice made concrete. It's the Dynamo paper, rebuilt from requirements.

The trade-off A horizontally-scalable, always-available store — in exchange for (by default) eventual consistency, conflict resolution, and giving up rich queries/transactions. Spread data across many nodes with no single leader so you scale linearly and survive failures. You pay by accepting eventual consistency (tunable per request), handling write conflicts, and forgoing joins/SQL — you get a hash table, not a database.

1. Requirements

Functional: get(key) and put(key, value) — no joins, no secondary queries, no transactions across keys (that's the deal that makes it scale). Data is larger than one machine; values are opaque blobs.

Non-functional: highly available (always writable, even during failures — the AP choice, like Dynamo's shopping cart), horizontally scalable (add nodes for linear capacity/throughput), low latency, durable (replicated), no single point of failure (leaderless — every node is a peer), tunable consistency (per-request).

2. Capacity estimate

Say 100 TB of data, 500,000 reads/sec + 100,000 writes/sec, replication factor 3. Work it:

  • Storage: 100 TB × 3 (replicas) = 300 TB total. At ~10 TB usable per node → ~30 nodes minimum, more for headroom. Consequence: enough nodes that node churn is constant — placement must tolerate it.
  • Throughput: 600k ops/sec × 3 (each write hits 3 replicas; quorum reads hit 2–3) spread over ~30+ nodes ≈ tens of thousands of ops/sec/node — fine for an LSM engine. Consequence: writes amplify by the replication factor — budget for it.
  • Why leaderless: at this scale, demanding a single leader per key (and failover) adds latency and a coordination point. A leaderless quorum design stays available through node loss with no failover — the AP choice. Consequence: we owe conflict resolution and anti-entropy.

3. The API

get(key, consistency) -> (value, version)   // consistency: ONE | QUORUM | ALL
put(key, value, version, consistency)        // version enables conflict detection

The consistency level is per request — that's the tunable knob (PACELC's ELC leg as an API). version (a vector clock / version vector) lets the store detect concurrent writes. Deliberately no query, no join, no multi-key transaction.

4. High-level design

A ring of peer nodes, no leader. A client (or a coordinator node) hashes the key to find its position on the consistent-hashing ring and the N nodes that own its replicas (the next N clockwise). A put is sent to all N replicas; it succeeds once W of them ack. A get reads from replicas until R respond, returning the newest version. Nodes gossip membership and use anti-entropy (Merkle trees, read-repair, hinted handoff) to converge replicas in the background. Any node can coordinate any request — no special roles, no SPOF.

5. Data placement — consistent hashing + replicas

Consistent hashing (with virtual nodes for even distribution) maps each key to a position; its N replicas are the next N distinct physical nodes clockwise (the replication factor). Replicas are placed to span failure domains (different racks/AZs) so one failure doesn't take out multiple replicas. Adding/removing a node only shifts the keys in the affected arcs (~1/node), not the whole keyspace — essential when you have dozens of nodes churning.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — Discord, Cassandra → ScyllaDB (real migration) Discord stored trillions of messages in Cassandra but hit latency/operational pain — GC pauses, tail latencies, and a large node count. Migrating to ScyllaDB (a C++ Cassandra-compatible rewrite) cut p99 read latency from 40–125ms to ~15ms and shrank the fleet from 177 nodes to 72. What about a leaderless KV store's design causes these latency/operational pains, and what does this say about the storage engine choice?
Scenario 2 — the read that returns old data You write a value with consistency ONE (one replica acks), then immediately read with ONE from a different replica that hasn't received the write yet — and get the old value. Why did this happen, and what R/W settings guarantee a read sees the latest write?
Scenario 3 — two writes to the same key, one silently lost During a network partition, two clients write the same key on opposite sides. They reconcile by keeping the one with the larger timestamp — and the other write vanishes with no error. Why is timestamp last-write-wins dangerous, and how should concurrent writes be handled instead?

Scenario 1 is the storage engine (+ placement). Scenario 2 is tunable consistency / quorums. Scenario 3 is conflict resolution. Each maps to a sub-problem.

The four hard sub-problems

Walking the design was the easy part. These four are the Dynamo paper's hard core — each with a nested failure.

Sub-problem 1 — Placement & replication

Map keys to nodes so it survives node churn, and replicate each key to N nodes for durability/availability. Consistent hashing + virtual nodes handles placement; the N replicas (next N clockwise, across failure domains) provide redundancy. This is the foundation everything else (quorums, anti-entropy) builds on.

The failure that happens when you get this wrong — rehash on node change Naive hash % N placement remaps almost the entire keyspace when a node is added or fails — catastrophic in a 30+ node cluster where churn is routine, causing massive data movement and a flood of misses/repairs. Consistent hashing limits movement to the arcs around the changed node (~1/N of keys), and virtual nodes keep the distribution even so no node is overloaded. Replica placement must also span failure domains — if all 3 replicas of a key sit in one rack/AZ, one failure loses all of them. Placement that ignores topology turns one failure into data loss.

Sub-problem 2 — Tunable consistency via quorums

With N replicas, you choose how many must respond to a write (W) and a read (R). The rule R + W > N guarantees any read quorum overlaps any write quorum in at least one replica — so a read is guaranteed to see the latest write (strong-ish consistency). Lower R/W = lower latency + higher availability but possible staleness. You dial this per request: QUORUM for important reads, ONE for fast/tolerant ones.

The failure that happens when you get this wrong — silent stale reads This is Scenario 2. With W=1, R=1 (R+W=2 ≤ N=3), the replica you write to and the replica you read from may not overlap — so a read can hit a replica that never got the write and return stale data, with no indication anything's wrong. To guarantee read-your-writes / latest-value reads, choose R and W so R + W > N (e.g. W=2, R=2 with N=3): now every read quorum and write quorum share a replica, so the read sees the latest write — at the cost of waiting for 2 nodes each. There's no single right setting; pick per operation by how much staleness you can tolerate (this is PACELC's ELC leg as a knob). Assuming "it's usually consistent" is the trap.

Sub-problem 3 — Conflicts & healing divergence

Leaderless + available-under-partition means two clients can write the same key concurrently and both succeed — the replicas now disagree. You must (a) detect the conflict and (b) resolve it, and (c) heal replicas that drifted. Detection uses version vectors (which reveal whether two writes were causally ordered or truly concurrent); healing uses read-repair (fix on read), hinted handoff (catch up a node that was down), and Merkle-tree anti-entropy (background reconciliation).

The failure that happens when you get this wrong — last-write-wins data loss This is Scenario 3. Resolving conflicts by "keep the larger timestamp" (LWW) is simple but silently discards the losing write — and with unreliable clocks, the "loser" may actually be the write you wanted. For data where losing a write is unacceptable (a shopping cart, a counter, a set), use version vectors to detect concurrency and then either return both siblings to the application to merge (Dynamo's cart: union the carts) or use a CRDT that merges without loss. LWW is acceptable only when the data genuinely tolerates it (a "last seen" timestamp). Know your store's default — many default to LWW, which quietly loses data you may care about.

Sub-problem 4 — Storage engine & deletes

Each node needs a local engine to store its keys. Write-heavy KV stores use an LSM-tree (sequential writes, great write throughput) — which means deletes are tombstones (a delete writes a marker, not a removal) and reads may touch multiple SSTables (mitigated by Bloom filters). The engine choice shapes the store's entire latency and operational profile.

The failure that happens when you get this wrong — resurrected deletes & engine pain This is Scenario 1. Two engine-level traps. (1) Resurrected deletes: in an eventually-consistent LSM store, a delete is a tombstone that must propagate to all replicas; if a replica was down during the delete and tombstones are garbage-collected too early, that replica resurrects the deleted data on the next repair — so you must keep tombstones for a grace period and run repairs within it. (2) Engine/operational pain (Discord): an LSM store's background compaction competes with reads, and in a GC'd runtime (Cassandra/JVM) stop-the-world pauses spike tail latency — which is exactly why Discord moved to ScyllaDB (C++, no GC pauses, better compaction control), cutting p99 from ~125ms to ~15ms and the fleet from 177 to 72 nodes. The storage engine and its background machinery, not the distribution model, often dominate real-world latency and ops cost.

The rule in one linePlace with consistent hashing across failure domains, replicate N ways, tune consistency with R + W > N quorums per request, resolve conflicts with version vectors + merge/CRDT (never blind LWW), heal with anti-entropy, and store with an LSM engine whose compaction and tombstone discipline you respect — the engine often dominates real latency (Discord).

Famous implementations

Same facets — model, consistency/conflict handling, and the failure to watch.

Amazon Dynamo / DynamoDB

  • What it is. The originating leaderless design (the 2007 paper) and its managed descendant.
  • Consistency/conflicts. Tunable (eventual/strong reads); version vectors + app-level merge in classic Dynamo; managed DynamoDB hides much of it but global tables use LWW.
  • The failure to watch. LWW conflict resolution silently dropping concurrent writes; hot partitions.

Cassandra

  • What it is. Open-source leaderless wide-column store on the Dynamo model + an LSM engine.
  • Consistency/conflicts. Per-query consistency levels (ONE/QUORUM/ALL), R+W>N for strong; read-repair + hinted handoff + Merkle repair; tombstone deletes.
  • The failure to watch. JVM GC pauses spiking tail latency; compaction load; tombstone read-amplification and resurrected deletes if repairs lapse (Scenario 1).

ScyllaDB

  • What it is. A C++ rewrite of Cassandra (same model + protocol), built for low/predictable latency.
  • Consistency/conflicts. Same Cassandra semantics; a shard-per-core architecture and no GC.
  • The failure to watch. Still LSM (compaction/tombstone discipline applies); the win is removing GC-pause tail latency — why Discord migrated (177→72 nodes, p99 125→15ms).

Build it yourself

Work each before opening the approach.

Pick R, W, N for two workloads

For (a) a user-session store that must always be writable and tolerates slight staleness, and (b) an inventory count that must be read-correct, choose N/R/W and explain the trade.

Reveal the approach

Use N=3 for both. (a) Sessions: W=1, R=1 (or W=1, R=2) — fast, always-available writes; staleness on a read is harmless (you'll get a slightly old session). R+W=2 ≤ 3, so reads can be stale — acceptable here. (b) Inventory: W=2, R=2 — R+W=4 > 3, so every read quorum overlaps the latest write quorum, guaranteeing read-correct counts, at the cost of waiting for 2 nodes per op. The lesson: consistency is a per-operation dial, not a database-wide setting — spend the latency only where correctness demands it (Scenario 2). For inventory you might further use a CRDT counter to avoid lost decrements under concurrency.

Resolve a shopping-cart conflict without loss

Two clients add different items to the same cart on opposite sides of a partition. Design conflict handling so no added item is lost.

Reveal the approach

Don't use timestamp LWW — it would drop one client's additions (Scenario 3). Tag the cart with a version vector; when the two writes reconcile, the vectors are incomparable (neither happened-before the other) → a true concurrent conflict. Resolve by merging: for a cart, take the union of items (Dynamo's actual approach — better to occasionally re-add a removed item than to lose an added one). Even cleaner: model the cart as an OR-Set CRDT so the merge is automatic, commutative, and lossless — concurrent adds both survive, and removes only affect tags they've observed. The principle: detect concurrency with version vectors, then merge with logic that fits the data, never silently pick a winner.

Design-it-yourself: the open-ended ones

  • Add secondary indexes. Your KV store needs "find users by email." Design how, given there's no global query — and the consistency cost. → Indexes, Search.
  • Tune for write-heavy vs read-heavy. Choose the LSM compaction strategy and Bloom-filter sizing for each. → Storage Engines.
  • Make deletes safe. Design tombstone TTL + repair schedule so deletes never resurrect, without unbounded tombstone buildup. → Anti-Entropy.
