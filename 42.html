Probabilistic Data Structures

Sometimes an exact answer costs gigabytes and a "good enough" answer costs kilobytes. Probabilistic data structures make that trade deliberately: they answer questions like "have I seen this before?", "how many distinct items?", and "what's the 99th percentile?" using a tiny, fixed amount of memory — at the price of a small, bounded, tunable error. At scale, that trade is often the only thing that fits in RAM.

The trade-off Tiny, fixed memory and fast answers — in exchange for approximate results with a bounded, tunable error. These structures replace gigabytes of exact bookkeeping with kilobytes. You accept a known error rate (a false-positive chance, a ±2% cardinality estimate) that you can dial against memory — and you must understand exactly what kind of error each one can and cannot make.

Why approximate? The space–accuracy trade

Exact answers to "have I seen this URL?", "how many unique visitors today?", or "what's the p99 latency?" require storing every item or every sample — gigabytes at scale, far too much to keep in memory per node. Probabilistic data structures store a compressed, lossy summary instead: a fixed, tiny footprint (often kilobytes) that answers the question with a small, bounded, tunable error. The error isn't a bug — it's the feature you're buying memory savings with, and the key skill is knowing precisely which kind of error each structure makes.

The four questions and their tools:

  • Membership — "have I seen X?" → Bloom filter.
  • Cardinality — "how many distinct items?" → HyperLogLog.
  • Frequency — "how many times did X occur / who are the heavy hitters?" → Count-Min Sketch.
  • Percentiles — "what's the p99?" → t-digest / HdrHistogram.
The one-line frameYou trade exactness for a fixed, tiny memory footprint and speed. Each structure gives a specific, one-sided or bounded error you can tune against memory — and using the right one for the right question (and never trusting it beyond its guarantee) is the whole game.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — Chrome Safe Browsing (real system) Google Chrome checks every URL you visit against a huge blocklist of malicious sites — but it can't ship or query that multi-gigabyte list on every navigation. Instead each browser holds a small Bloom filter of the blocklist: it answers "is this URL definitely safe?" locally and instantly, and only when the filter says "maybe malicious" does Chrome make a network call to the exact list. Why does this design work, and what does the filter's possible error force Chrome to do on a "maybe"?
Scenario 2 — counting unique visitors at scale You need the count of distinct users who visited each of a million pages today. Storing every page's set of user IDs would be enormous. What structure gives each page's unique-visitor count in ~12 KB regardless of whether it had 100 or 100 million visitors, and what can it not tell you?
Scenario 3 — the p99 you can't average Each of your 50 servers reports its own p99 request latency. The dashboard shows the average of the 50 p99s as "the p99." A senior engineer says that number is meaningless. Why can't you average percentiles, and what structure lets you compute a correct global p99 from per-server summaries?

Scenario 1 is the Bloom filter. Scenario 2 is HyperLogLog. Scenario 3 is t-digest/HdrHistogram. Each maps to a structure below.

The four structures to know

Four structures cover the four questions. Each has a named misuse nested under it.

Structure 1 — Bloom filter (membership)

A Bloom filter is a bit array plus k hash functions. To add an item, hash it k ways and set those k bits. To test membership, hash and check those bits: if any is 0, the item is definitely absent; if all are 1, it's probably present. So it has no false negatives (never says "absent" wrongly) but can have false positives (says "present" for something it never saw). It can't store the items or delete them (basic form). The false-positive rate is set by the bit-array size and k relative to the number of items.

The misuse to avoid — trusting a "yes" This is Scenario 1. The asymmetry is the whole design: a Bloom filter's "no" is certain; its "yes" is a maybe. So you use it as a cheap negative filter and back a "yes" with an exact check. Chrome trusts "definitely safe" (the "no" path) to skip the network entirely, and on "maybe malicious" makes the authoritative lookup — the false positive only ever costs an extra check, never a missed threat. The same pattern protects an LSM-tree: "definitely not in this SSTable" skips a disk read; "maybe" reads the file. Never use a Bloom filter where a false positive is unsafe without a backing check.

Structure 2 — HyperLogLog (cardinality)

HyperLogLog (HLL) estimates the number of distinct elements in a stream using a fixed ~12 KB, with a typical error around ±2% — whether the true count is thousands or billions. It works by hashing each item and tracking the maximum number of leading zeros seen (a longer run of leading zeros implies more distinct items, by probability), bucketed for stability. Critically, HLLs are mergeable: union two HLLs to get the distinct count of the combined set — perfect for distributed counting.

The misuse to avoid — asking it the wrong question This is Scenario 2's caveat. HLL answers "how many distinct?" — it cannot tell you whether a specific item is in the set (that's a Bloom filter's job) nor list the items. People reach for it then ask "is user X here?" and get nonsense. Use HLL only for cardinality; for per-page unique counts, keep one mergeable HLL per page and union them across servers/days. ~12 KB/page × a million pages is tractable; exact sets would not be.

Structure 3 — Count-Min Sketch (frequency & heavy hitters)

A Count-Min Sketch estimates how many times each item has occurred, using a small 2-D array of counters and several hash functions: to count an item, increment one counter per row (the row's hashed column); to query, take the minimum across its rows. Hash collisions can only add to a counter, so the min is an over-estimate that's never an under-estimate — it never undercounts. It's the go-to for "top-K / heavy hitters" (which items are most frequent — heavy hitters) in a stream, in fixed memory.

The misuse to avoid — forgetting it overcounts (one-sided error) A Count-Min Sketch can over-estimate a frequency (collisions inflate counters) but never under-estimate — the error is one-sided. So it's safe for "is this at least N?" and heavy-hitter detection, but you must not treat its counts as exact, and rare items can be relatively badly overcounted (a low true count swamped by collisions from frequent items). Size width/depth to bound the error, and remember the guarantee is "true count ≤ estimate," nothing tighter.

Structure 4 — t-digest / HdrHistogram (percentiles)

Computing an exact percentile requires sorting all samples — impossible to keep at scale. HdrHistogram (fixed bucket boundaries, great for bounded ranges like latencies) and t-digest (adaptive, accurate in the tails) summarize a distribution in small fixed memory and answer "what's the p50/p99/p999?" with bounded error. Both are mergeable: combine per-server digests into a correct global distribution.

The misuse to avoid — averaging percentiles This is Scenario 3, and it's one of the most common metrics mistakes. You cannot average percentiles — the mean of 50 servers' p99s is not the global p99 (a percentile is a property of the whole distribution, not a number you can linearly combine). The correct approach: each server keeps a t-digest/HdrHistogram of its raw latencies, you merge the digests, and compute the p99 from the merged distribution. This is exactly why monitoring systems store histograms, not pre-computed percentiles, per host.

The rule in one lineMatch the structure to the exact question and respect its error shape: Bloom for membership (trust the "no"), HLL for distinct counts (mergeable), Count-Min for frequency (never undercounts), histograms for percentiles (never average them — merge them).

Famous implementations

The structures made concrete. Same facets — what it provides, the use, and the failure to watch.

Redis

  • Provides. Native HyperLogLog (PFADD/PFCOUNT/PFMERGE); Bloom/Cuckoo filters, Count-Min, Top-K via the RedisBloom module.
  • Use. Unique-visitor counts that merge across keys; dedup/membership guards; heavy-hitter detection — all in fixed memory at in-memory speed.
  • The failure to watch. Treating PFCOUNT as exact (it's ±2%) or using an HLL for membership tests.

Databases — LSM engines / Cassandra / HBase

  • Provides. Per-SSTable Bloom filters on the read path.
  • Use. Skip disk reads for keys "definitely not" in a file — the core of LSM read performance.
  • The failure to watch. Under-sized filters → more false positives → wasted disk reads (read amplification); it's a memory-vs-I/O dial.

Analytics & monitoring — BigQuery / Prometheus / Druid

  • Provides. Approx-distinct (HLL) functions; histogram metrics (t-digest/HdrHistogram) for percentiles.
  • Use. Dashboards over billions of rows without exact-distinct cost; correct global percentiles by merging per-host histograms.
  • The failure to watch. Averaging percentiles across hosts (Scenario 3) — store and merge histograms instead.

Build it yourself

These click when you do the math. Work each before opening the approach.

Size a Bloom filter from first principles

You'll store 10 million URLs and want a 1% false-positive rate. Derive the bit-array size and the number of hash functions, and explain the memory-vs-accuracy knob.

Reveal the approach

For n items and target false-positive rate p: bits m ≈ −n·ln(p) / (ln2)². With n=10,000,000 and p=0.01: −ln(0.01)=4.6, (ln2)²=0.48, so m ≈ 10M × 4.6 / 0.48 ≈ 95.8M bits ≈ 12 MB (~9.6 bits/item). Optimal hash count k ≈ (m/n)·ln2 ≈ 9.6 × 0.69 ≈ 7. The knob: each extra ~4.8 bits/item roughly squares-down the false-positive rate (10%→1%→0.1%), so you trade memory for fewer false positives. Note the storage is independent of how long each URL is — you store bits, not the URLs — which is why 12 MB beats storing 10M full URLs.

Estimate uniques across a fleet with HyperLogLog

Twenty servers each see part of today's traffic; you want the total count of distinct users with ~2% error and minimal coordination. Design it.

Reveal the approach

Each server maintains one HyperLogLog (~12 KB) of the user IDs it sees — just PFADD each ID, no central state, no per-ID storage. At report time, merge the 20 HLLs (union of registers, taking the max per bucket) into one and read the estimate — that's the distinct count of the union, at ~2% error, because HLLs are mergeable. Total memory moved: 20 × 12 KB, regardless of whether the true count is thousands or billions. You could not do this by averaging per-server distinct counts (overlapping users would be double-counted); the merge handles overlap correctly because it operates on the underlying registers, not the counts.

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Stop cache penetration with a Bloom filter. Attackers query keys that don't exist, missing the cache and hammering the database. Use a Bloom filter to reject definitely-absent keys before the DB. → Caching.
  • Dedup a billion-URL crawl frontier. A web crawler must avoid re-fetching seen URLs without storing them all. Design the membership check and handle the false-positive consequence. → Web Crawler.
  • Build a correct latency SLO dashboard. 100 services each emit latencies; you need accurate global p50/p99/p999. Choose the structure and the aggregation. → Metrics & Monitoring, SLI/SLO/SLA.
