Consistency Models and PACELC

A precise map of what "consistency" actually means, what it costs, and how to choose the right guarantee per use case.

In this lesson

Why "consistency" is an overloaded word

The word consistency is used for at least three different things, and conflating them is the single most common source of confusion in design discussions:

  • The C in ACID — a database keeps invariants (constraints, foreign keys, triggers) satisfied across a transaction. This is an application-level property and is mostly orthogonal to distributed-systems consistency.
  • The C in CAP — specifically linearizability, a strong single-object guarantee. See CAP Theorem.
  • A consistency model — a contract between a replicated/concurrent system and its clients about which orderings of reads and writes are observable. This lesson is about this third meaning.

A consistency model answers one question: given a set of concurrent operations across replicas, which results is a client allowed to see? Stronger models forbid more "surprising" outcomes but cost more in latency, availability, and coordination. Weaker models permit anomalies but are cheap and partition-tolerant. Everything below is a point on that trade-off curve.

Key insight Consistency models form a hierarchy ordered by strength. A system that provides a stronger model automatically provides all weaker ones below it. Your job as a designer is to pick the weakest model that still satisfies the application's invariants — not the strongest available.

The consistency spectrum

From strongest to weakest, the canonical models for replicated data:

ModelGuaranteeForbidsTypical cost
Strict serializabilityTransactions appear in a single total order consistent with real timeStale reads, lost updates, all serialization anomalies, real-time order violationsConsensus + synchronized clocks or global ordering
LinearizableEach single-object op appears instantaneous at some point between its invocation and response; total order respects real timeStale reads after a completed writeQuorum reads/writes or consensus per op
SequentialAll clients see the same total order; each client's ops keep program order. The order need NOT match real timeReordering within a single clientTotal order broadcast, no clock sync needed
CausalOperations related by happens-before are seen in that order by all; concurrent ops may be seen in any orderEffect visible before causeTrack causal metadata (vector/version clocks); no global coordination
EventualIf writes stop, all replicas converge to the same value eventuallyNothing about ordering or recencyAnti-entropy / gossip; cheapest, most available

Linearizability is the gold standard for single objects. Once a write completes (the client gets an ack), every subsequent read — by any client — must return that value or a newer one. There is no "stale read" window. The price is that reads and writes must coordinate: classically via a quorum (W + R > N) with read-repair, or via routing all ops through a consensus leader. See Consensus and Raft — Raft gives you a linearizable replicated log; reads must go through the leader (or use leader leases / ReadIndex) to stay linearizable.

Sequential consistency drops the real-time requirement. Everyone agrees on one order of operations and each process's own operations appear in the order it issued them, but that global order need not match wall-clock reality. A read issued after a write completed (in real time) might legally return the old value because the system placed the read "earlier" in the agreed order. This is strictly weaker than linearizability and cheaper, but the distinction trips up many engineers.

Causal consistency is the strongest model that remains available under partition (a key result). It only orders operations that have a happens-before relationship — if write B was made after the client observed write A, everyone must see A before B. Independent, concurrent writes can be observed in different orders by different clients. Causal metadata (version vectors) is tracked per key or per session. This is what most "collaborative" systems lean on; see Time, Clocks, and Ordering for the underlying logical-clock machinery and CRDTs for the convergence layer that pairs naturally with it.

Eventual consistency is the weakest commonly-named model: it says nothing about what you read now, only that absent new writes, replicas converge. Dynamo-style stores live here by default. The convergence is achieved through anti-entropy and gossip plus conflict resolution (last-writer-wins, vector clocks, or CRDTs).

Pitfall "Eventual consistency" with no further qualification is almost useless as a contract — "eventually" can mean milliseconds or, during a partition, hours. Strong candidates always ask: eventual, but with which client-centric guarantees layered on top? A pure eventual store can show a user their own write disappearing on the next request, which is unacceptable for most UX.

Client-centric (session) guarantees

The models above are data-centric — they describe the global behavior of the store. Client-centric guarantees describe what a single client/session observes across its own sequence of operations, and they can be layered on top of an eventually-consistent store cheaply (usually by routing a session to the same replica or carrying a version token). They are the practical glue that makes "eventual" usable:

  • Read-your-writes (read-after-write): after a client writes a value, its own subsequent reads see that value or newer. Prevents the "I posted a comment and it vanished" anomaly. Cheapest implementation: pin the session to the primary or to the replica that accepted the write; or pass the write's version and read only from replicas caught up to it.
  • Monotonic reads: if a client reads value v, later reads never return an older value. Prevents time going backwards. Implemented by pinning a session to one replica (or a replica set ordered by version).
  • Monotonic writes: writes from a single client are applied in the order they were issued. Without this, a "set name=A" then "set name=B" can land as B-then-A on a replica.
  • Writes-follow-reads (causal session): a write that depends on a value the client read is ordered after that value's write everywhere. This is the per-session building block of causal consistency.
Key insight Together, these four session guarantees compose into causal consistency. That is the bridge between the cheap eventual world and the expensive strongly-consistent world: causal+session guarantees give you most of what users perceive as "correct" without paying for global coordination.

Linearizability vs serializability (and strict serializability)

These two are routinely confused because both are "the strong one." They are different axes:

LinearizabilitySerializability
ScopeSingle object / single operationMulti-object transactions
PropertyRecency & real-time order of opsEquivalence to some serial execution of transactions
Real-time?Yes — respects wall-clock orderNo — any serial order is acceptable
FieldDistributed systemsDatabase isolation (the I in ACID)

Serializability says a set of concurrent transactions produces a result equivalent to running them one at a time in some order. But it does not require that order to match real time. Two serializable transactions T1 (commits at 10:00) and T2 (commits at 10:01) could be serialized as "T2 then T1" — legal under serializability, even though T2 happened later. That's fine for isolation correctness but can violate user expectations.

Strict serializability = serializability + linearizability's real-time constraint. It is the strongest practical model for a transactional distributed database: transactions appear in a single total order that is consistent with the real-time order in which they committed. This is what Google Spanner provides, and it is exactly why Spanner needs TrueTime — it must reason about real-time bounds to assign commit timestamps that respect actual ordering. See Distributed Transactions for the 2PC/commit-protocol mechanics underneath.

Trade-off Snapshot isolation (Postgres, MySQL-InnoDB default-ish, many distributed SQL engines) is weaker than serializability — it permits write-skew anomalies. It's popular because it gives non-blocking consistent reads and is far cheaper. The principal move is knowing your engine's default isolation level and whether your invariants survive write skew (e.g., "two on-call doctors, both check ≥1 other is on duty, both go off-duty" — classic write skew that SI allows and serializability forbids).

PACELC: CAP, refined

CAP is famously incomplete: it only tells you what happens during a network partition (choose Consistency or Availability), and partitions are rare. It says nothing about the steady-state trade-off that dominates day-to-day design. PACELC (Abadi, 2012) fixes this:

if (Partition) {
    choose between Availability (A) and Consistency (C)
} else {   // normal operation, the 99.99% case
    choose between Latency (L) and Consistency (C)
}

The "else" clause is the important addition. Even with zero partitions, a replicated system faces a fundamental tension: to guarantee a linearizable read you must contact a quorum (or the leader), which adds round trips and latency. To serve a read from the nearest replica with low latency, you risk returning stale data. You pay for consistency in latency on every single request, not just during failures.

PACELC classifies a system with a pair: its partition behavior and its normal-operation behavior. The two halves are independent — a system can be PA/EC (sacrifice consistency under partition, but prefer consistency over latency normally).

Key insight CAP made partition tolerance look like the central trade-off. PACELC reveals that for most systems most of the time, the real, continuous decision is latency vs consistency. Spanner's whole engineering effort (TrueTime, atomic clocks, commit-wait) is fundamentally about paying a small bounded latency tax to deliver EC — consistency in the no-partition case.

Classifying real systems

SystemPACELCBehavior
DynamoDB / DynamoPA/ELStays available under partition; prefers low latency normally (eventually consistent reads by default). Optional strongly-consistent reads flip it toward EC per-request.
CassandraPA/EL (tunable)Tunable per query via consistency levels. ONE = PA/EL; QUORUM/QUORUM with W+R>N approximates linearizability per key; LOCAL_QUORUM tunes the latency/consistency knob.
Riak / VoldemortPA/ELDynamo lineage; sloppy quorums + hinted handoff favor availability and latency.
Google SpannerPC/ECConsistency over availability under partition; consistency over latency normally. TrueTime + commit-wait + Paxos groups give strict serializability. Pays a latency tax (commit-wait ~ a few ms).
MongoDBPA/EC (default)Single-primary replica set; under partition the minority side loses its primary and goes read-only (availability sacrificed on the minority); normally favors consistency by routing reads to primary. Read/write concerns tune both axes.
HBase / BigTablePC/ECSingle region server owns a region; strongly consistent; partition can make a region unavailable until reassigned.

Pitfall Beware single-letter labels applied to a whole product. Cassandra, DynamoDB, and MongoDB all expose per-operation tuning (consistency levels, read/write concerns, consistent-read flags). The PACELC label describes the default or the dominant design bias — the right answer in an interview is "it depends on the consistency level configured for that query," with the default stated.

The cost of strong consistency

Strong consistency is never free. Where, precisely, does the cost show up?

  • Latency on the hot path. A linearizable read needs a quorum round trip or a leader hop. Cross-region, that's tens of milliseconds added to every read, not just writes. Spanner's commit-wait deliberately stalls a commit for the TrueTime uncertainty window (ε, typically a few ms) to guarantee timestamp ordering.
  • Availability under partition. A CP system rejects writes (and linearizable reads) on the minority side. If you require strong consistency, a partition that isolates a quorum makes part of your system unavailable — by design. See CAP Theorem.
  • Throughput ceiling. A single consensus leader (Raft/Paxos) is a serialization point. You scale by sharding into independent consensus groups (Spanner's "Paxos groups," CockroachDB's "ranges"), which then reintroduces the distributed-transaction problem across shards.
  • Coordination amplifies tail latency. Quorum operations are gated by the slowest of the required replicas. A single slow node (GC pause, disk hiccup) inflates p99. Dynamo-style sloppy quorums trade consistency to dodge exactly this.
// Quorum knobs (N replicas, W write quorum, R read quorum)
// Linearizable-ish per key requires:  W + R > N   AND   W > N/2
// Examples for N=3:
//   W=3, R=1  -> fast reads, slow/fragile writes (any node down blocks writes)
//   W=1, R=3  -> fast writes, slow reads, write survives 1 ack only
//   W=2, R=2  -> balanced, tolerates 1 node down, overlap guaranteed

The deeper point: strong consistency converts a physics problem (speed of light across regions) into a latency budget. There is no algorithm that removes the round trips; you can only hide them (caching, leases, local reads with bounded staleness) or push the consistency decision to the client. See Replication Deep Dive for sync vs async replication mechanics that sit underneath all of this.

Choosing per use case

The principal skill is decomposing one product into many consistency decisions. The same app rarely needs one global model.

Use caseRight modelWhy
Account balance / payments / inventory decrementStrict serializability (or at least linearizable + transactions)Double-spend and oversell are correctness failures, not UX warts. See Distributed Transactions.
Distributed lock / leader election / configLinearizable (consensus)Two leaders or a stale lock = split brain. Use etcd/ZooKeeper. See Leader Election and Coordination.
Social feed, likes, view countsEventual + read-your-writesA like count off by a few for a second is invisible; availability and latency matter far more.
Collaborative editing / chat orderingCausal consistency (+ CRDTs)Cause must precede effect; concurrent edits merge. See CRDTs.
Shopping cartEventual, merge on conflict (Dynamo's original use case)Adds should never be lost; "merge carts" beats "reject add." Availability prioritized.
User profile / settingsEventual + read-your-writes + monotonic readsUser must see their own edit immediately and never see it regress.
What strong candidates do They never say "we'll use strong consistency" as a blanket. They name the specific invariant that demands it (no oversell, no double-spend, exactly-one leader), apply the minimum sufficient model there, and explicitly relax everything else to eventual + session guarantees to recover latency and availability. They also state the PACELC posture per data domain, not per system.

Principal-level interview lens

What strong candidates do They treat consistency as a per-domain budget, not a global switch. They distinguish linearizability (single object, real-time) from serializability (multi-object isolation) without prompting, reach for causal + session guarantees as the pragmatic middle, and quantify the cost ("strong reads add a cross-region quorum hop, ~40 ms to p99"). When they say "eventually consistent" they immediately specify the client-centric guarantees layered on top, and they cite PACELC's else-branch as the real day-to-day trade-off rather than fixating on CAP partitions.

Sharp probing questions an interviewer should ask:

  • "You said the store is eventually consistent. A user updates their profile photo and refreshes — they see the old photo. Is that acceptable? If not, which exact guarantee fixes it, and what does it cost to implement?"
  • "Your design uses snapshot isolation. Walk me through a write-skew anomaly that could violate a business invariant here, and tell me whether you'd pay for serializability or guard it another way."
  • "Classify your primary datastore in PACELC and defend both letters. Now change the consistency level / read concern on the critical read path — how does the classification and the p99 move?"
  • "You need linearizable reads from a Raft cluster. Why isn't reading from the leader's local state automatically safe, and what mechanism (ReadIndex, lease, log barrier) makes it correct?"
  • "Two regions, 80 ms RTT apart. Product wants strong consistency and sub-20 ms reads in both regions. Explain why that's physically impossible and what you'd negotiate instead (bounded staleness? read-local writes-global? Spanner-style?)."
  • "Where in this design would causal consistency be both sufficient and strictly cheaper than linearizability, and how would you track the causal metadata at scale without it blowing up?"
