Storage Engines: LSM-Trees vs B-Trees

Under every database is a storage engine that decides how bytes hit the disk — and there are essentially two designs. B-Trees update data in place and are the read-optimized incumbent behind almost every relational database. LSM-Trees never update in place, only append and merge, and are the write-optimized challenger behind Cassandra, RocksDB, and most NoSQL. Which one your database uses shapes its entire performance profile.

The trade-off Write throughput — in exchange for read and space amplification, and a background compaction process that can stall you. An LSM-tree turns random writes into sequential appends, winning huge write throughput. You pay by reading from many files (read amplification), storing old versions until merged (space amplification), and running compaction that, if it falls behind, stalls writes. A B-tree is the inverse: fast point reads, predictable space, but every write does a random in-place update.

What a storage engine is

A storage engine is the layer that actually reads and writes your data to durable storage: how records are laid out on disk, how they're found again, and how durability survives a crash. It's the foundation under the query planner and the network layer — and it comes in two fundamental designs that make opposite bets about which operation should be cheap.

  • B-Tree — a balanced tree of fixed-size pages, updated in place. To change a value, you find its page and overwrite it. Reads are a handful of page lookups (shallow tree); writes are random-access disk I/O.
  • LSM-Tree (Log-Structured Merge-tree) — writes go to an in-memory memtable, which is flushed as an immutable sorted file (SSTable) when full. Files are never updated — new writes (and deletes, as tombstones) just append more files; a background compaction merges them.
The one-line frameThe two engines trade read performance and predictable space against write throughput. The B-tree makes reads cheap and writes do random in-place I/O. The LSM-tree makes writes cheap (sequential appends) and pays on the read side (check several files) and in background work (compaction). Everything else — amplifications, Bloom filters, compaction strategies — follows from that one choice.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the RocksDB write stall (real, common production failure) A write-heavy service on an LSM store hums along, then suddenly write latency spikes and throughput craters — without any change in load. Monitoring shows SSTable files piling up and the engine has begun stalling writes on purpose. What background process fell behind, why does the engine deliberately throttle your writes, and what would you tune?
Scenario 2 — the point lookup that reads ten files You ask an LSM store for one key by its exact ID. To answer, it might have to check the memtable plus several SSTables across levels, because the key could live in any of them. How does it avoid actually reading every file from disk for a key that isn't there, and what's the data structure that makes a "not found" cheap?
Scenario 3 — "we just need it faster," but which way? A team is choosing a database for a workload that is 95% writes (ingesting sensor events), rarely reads old data, and is cost-sensitive on disk. Another team has a workload that is 90% point reads of recent records with frequent updates. Which engine fits each, and which of the three "amplifications" does each team most need to minimize?

Scenario 1 is compaction and the write-stall. Scenario 2 is Bloom filters and the read path. Scenario 3 is the three amplifications. Each maps to an idea below.

The four ideas that separate them

Four ideas explain the whole performance difference. Each has a named failure nested under it.

Idea 1 — In-place update: the B-Tree's bet

A B-tree keeps data in fixed-size pages arranged as a shallow, balanced tree (typically 3–4 levels even for huge datasets). A point read is a few page reads; a range scan walks leaf pages in order. A write finds the page and overwrites it in place — random-access I/O, and possibly a page split if it's full. This makes reads fast and predictable and space tight (one copy of each row), at the cost of write-time random I/O.

The failure that happens when you get this wrong — the torn page If the machine crashes mid-write of a page, you can get a torn page — half old, half new bytes — corrupting the tree. The fix is the write-ahead log (WAL): write the change to an append-only log first, then update the page; on crash, replay the log to repair partial writes. Every B-tree engine has a WAL for exactly this reason — and it adds write I/O, narrowing the B-tree's gap to LSM on writes.

Idea 2 — Append + merge: the LSM's bet

An LSM-tree never overwrites. Writes hit the memtable (in memory, fast); when it fills, it's flushed as a new immutable SSTable. Because files are immutable, writes are pure sequential appends — far faster than random in-place updates, and crash-safe by construction (you never corrupt a file you never modify). The catch: old values and deletes accumulate across files, so a background compaction process continually merges SSTables, discarding superseded values and tombstones.

The failure that happens when you get this wrong — the compaction write-stall This is Scenario 1. Compaction is the price of immutability, and it competes with your writes for disk I/O. If the incoming write rate exceeds the rate compaction can merge files, SSTables pile up — reads get slower (more files to check) and eventually the engine deliberately stalls writes (RocksDB's "write stall") to let compaction catch up. The tuning levers: more/faster compaction threads, a different compaction strategy, faster disks, or simply not exceeding sustainable write throughput. The basic write path rarely fails; the background machinery is what causes LSM incidents.

Idea 3 — The three amplifications (and why you can't minimize all)

Every storage engine trades off three "amplifications" — and no configuration minimizes all three at once:

  • Write amplification — bytes actually written to disk per byte of logical write. LSM compaction rewrites data multiple times as it moves down levels; the WAL doubles B-tree writes.
  • Read amplification — disk reads per logical read. LSM may check many files for one key; B-tree is a few page reads.
  • Space amplification — disk used per byte of live data. LSM holds old versions/tombstones until compacted; B-tree keeps one copy but has page fragmentation/half-empty pages.
The failure that happens when you get this wrong — optimizing the wrong amplification This is Scenario 3. Compaction strategy is choosing which amplification to sacrifice. Leveled compaction (LCS) keeps few overlapping files → low read & space amplification, but high write amplification (data rewritten often) — good for read-heavy. Size-tiered compaction (STCS) merges similar-sized files less often → low write amplification, but high space (duplicate data) and read amplification — good for write-heavy. Picking LCS for a write-firehose (or STCS for a read-heavy, space-tight store) means you optimized the amplification you didn't have a problem with and made your real bottleneck worse.

Idea 4 — Bloom filters and the read path

Because an LSM key can live in any SSTable, a naive point lookup would read every file. The fix: each SSTable carries a Bloom filter — a tiny probabilistic structure that answers "is this key definitely not here?" with certainty (and "maybe here" otherwise). The engine consults the in-memory Bloom filter first and only touches a file's disk blocks if the filter says the key might be present. A point read for a non-existent key usually touches zero data files.

The failure that happens when you get this wrong — read amplification without a filter This is Scenario 2. Without Bloom filters, every point lookup — especially for keys that don't exist — scans block indexes across many SSTables, turning one logical read into many disk reads (crippling read amplification). The Bloom filter collapses "is this key absent?" to an in-memory check, so absent and rarely-present keys are nearly free. Its false-positive rate is a memory-vs-read-amplification knob (smaller filter → more false positives → more wasted disk reads). This is why sizing a Bloom filter is a core LSM operational skill.

The rule in one lineWrite-heavy ingest → LSM (and watch compaction); read-heavy / update-heavy OLTP → B-tree. Then tune the engine to minimize the one amplification your workload actually suffers from — not all three.

Self-check — can you reconstruct the three scenarios?

You read the answers above. Close the page and rebuild them.

  • Scenario 1 (write stall): What is compaction, why does the engine deliberately stall writes when it falls behind, and what do you tune?
  • Scenario 2 (10-file lookup): Why might a point read touch many SSTables, and how does a Bloom filter make an absent key nearly free?
  • Scenario 3 (which engine): Match each workload to an engine and name the amplification each must minimize. What does choosing LCS vs STCS trade?
  • Why does a B-tree need a WAL, and what does that cost it on writes?
  • State the three amplifications and why you can't minimize all three.

The shape to remember: 4 ideas — B-tree updates in place (needs a WAL), LSM appends & compacts (can stall), three amplifications, pick one to sacrifice, Bloom filters tame the LSM read path — each with a nested failure.

Cloud & managed building blocks — which engine you're really getting

You rarely pick a raw engine, but every managed database is one underneath — knowing which explains its performance:

EngineDatabases that use itManaged examples
B-TreeMySQL/InnoDB, PostgreSQL, SQL ServerRDS, Aurora, Cloud SQL, Azure SQL
LSM-TreeCassandra, RocksDB, LevelDB, ScyllaDB, HBase, BigTableDynamoDB, Keyspaces, Bigtable, Cosmos DB (internally)
LSM as embedded engineRocksDB under Kafka Streams, CockroachDB, TiKV, MyRocks(embedded; you inherit its compaction tuning)
The takeawayWhen a managed NoSQL store has mysterious latency spikes under heavy writes, suspect compaction (Scenario 1) — it's an LSM engine and the background merge is competing with your writes. When a relational database's write throughput plateaus, suspect the random-I/O + WAL ceiling of its B-tree. The engine choice is baked into the product; recognizing it tells you which failure mode to expect.

Famous implementations

The engines made concrete. Same facets — design, on-disk structure, and the failure to watch.

B-Tree — InnoDB (MySQL) / PostgreSQL

  • Design. Clustered B+Tree (InnoDB: data stored in the primary-key B-tree) with a WAL (redo log / WAL).
  • On-disk. Fixed-size pages updated in place; secondary indexes are separate B-trees.
  • The failure to watch. Write throughput limited by random page I/O + WAL flushes; page fragmentation wastes space; torn-page protection adds writes.

LSM — RocksDB / LevelDB

  • Design. The reference embeddable LSM engine (RocksDB is Facebook's fork of Google's LevelDB), used inside many other databases.
  • On-disk. Memtable → leveled SSTables, per-SSTable Bloom filters, configurable compaction (leveled/universal).
  • The failure to watch. Write stalls when compaction lags (Scenario 1); the three-amplification tuning is on you (or your host database).

LSM at scale — Cassandra

  • Design. Distributed LSM: each node runs an LSM engine; consistent hashing places data, anti-entropy heals it.
  • On-disk. Memtable + commit log (WAL) → SSTables; STCS or LCS compaction chosen per table.
  • The failure to watch. Tombstone buildup from delete-heavy patterns + wrong compaction strategy = read amplification and stalls; choose STCS vs LCS to match the table's read/write mix.

Build it yourself

These click when you trace the I/O. Work each before opening the approach.

Trace a write and a read through an LSM-tree

Walk a single write from arrival to durable disk, and a point read for a key that might exist. Where does each go, and what makes the write fast and the read potentially slow?

Reveal the approach

Write: append to the commit log (WAL) for durability, then insert into the in-memory memtable — both fast (sequential log append + memory write), acknowledge. When the memtable fills, flush it as a new immutable SSTable (sequential write). Read: check the memtable first; if absent, consult each SSTable's Bloom filter — skip files whose filter says "definitely not here"; for files that say "maybe," use the file's sparse index to read the right block. The newest occurrence of the key wins (later SSTables / memtable override older). The write is fast because it's append-only; the read can be slow because the key's location is unknown until you check filters — which is exactly why Bloom filters and compaction (fewer files) matter.

Size a Bloom filter for the read path

An SSTable holds 1,000,000 keys. You want a 1% false-positive rate. Roughly how many bits and hash functions, and what happens to read amplification if you halve the memory?

Reveal the approach

For false-positive rate p and n keys, bits m ≈ −n·ln(p)/(ln2)² ≈ 1,000,000 × 9.6 ≈ 9.6 million bits (≈1.2 MB, ~9.6 bits/key), and optimal hash functions k ≈ (m/n)·ln2 ≈ 7. Halving the memory (~4.8 bits/key) raises the false-positive rate to roughly 10%+, so ~10% of "maybe present" answers are wrong — each a wasted disk read into an SSTable that doesn't have the key, directly increasing read amplification. The Bloom filter is a memory-vs-read-I/O dial; full derivation in Probabilistic Data Structures.
