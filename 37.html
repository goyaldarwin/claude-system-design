Replication Deep Dive

Copying the same data to many machines — the topologies, the lag bugs that bite real users, and the failover decisions that cause outages or data loss.

Why replicate — and the three topologies

Replication keeps a copy of the same data on multiple nodes. It buys three distinct things that are easy to conflate: high availability (survive a node or datacenter failure), read scalability (serve reads from copies near the client), and latency reduction (place data geographically close to users). The hard part is never copying the data once — it is keeping the copies consistent while writes keep arriving.

Replication is distinct from partitioning: partitioning splits one dataset across nodes to scale capacity; replication copies the same dataset for availability and read throughput. Production systems do both — e.g. 1024 shards, each a 3-replica group. This lesson is the data-path companion to Redundancy and Replication (the why) and Consensus and Raft (the agreement protocol underneath strong replication).

There are exactly three replication architectures, distinguished by who is allowed to accept writes:

  • single-leader One node accepts all writes; replicas are read-only followers. (MySQL, PostgreSQL, MongoDB, Kafka partitions.)
  • multi-leader Multiple nodes accept writes and replicate to each other. (Multi-DC active/active, CouchDB, BDR.)
  • leaderless Any node accepts writes; the client (or a coordinator) writes to several and reads from several. (Dynamo, Cassandra, Riak, ScyllaDB.)

Single-leader replication

The leader (a.k.a. primary/master) is the single serialization point for writes. Every write is ordered by the leader, recorded in a replication log, and streamed to followers. Reads can be served by the leader or any follower. This is the default for the relational world because it gives a single, totally-ordered write history — which makes transactions and a clean consistency story tractable.

Its strengths: writes have a global order, conflict resolution is trivial (there are no concurrent conflicting writes — the leader serializes everything), and read replicas scale read-heavy workloads almost linearly. Its weaknesses: the leader is a write bottleneck and a single point of failure, so the system's availability is gated on fast, correct failover.

Key insightRead replicas scale reads, never writes. Adding followers does nothing for write throughput — every write still funnels through one leader. When write volume is the bottleneck, the answer is partitioning, not more replicas. Conflating the two is the single most common scaling-design mistake.

Synchronous vs asynchronous: the durability/latency knob

The defining choice in single-leader (and the leader-to-follower link generally) is when the leader acknowledges a write to the client relative to follower replication.

ModeLeader acks after…Durability on leader failureWrite latency
Synchronousat least one follower confirmsNo data loss if that follower survivesBounded by the slowest acked follower (and its RTT)
Asynchronouslocal persistence onlyWrites not yet shipped are lostLowest — no remote wait
Semi-synchronousone follower confirms, rest asyncOne durable copy; bounded loss windowOne RTT to the fastest replica

Fully synchronous replication to all followers is almost never used: a single slow or dead follower would block all writes, so availability collapses as you add replicas. The practical sweet spot is semi-synchronous — one synchronous follower for durability, the rest async for performance. PostgreSQL exposes this directly via synchronous_commit and synchronous_standby_names; MySQL via rpl_semi_sync_master_enabled plus a timeout that silently downgrades to async if no follower acks in time.

Trade-offAsync replication trades a durability window for latency: if the leader dies before shipping the tail of its log and you promote a follower, those acknowledged-to-client writes are gone. This is exactly the GitHub 2018 incident class. Synchronous replication closes that window but couples your write latency — and availability — to a remote node. There is no free lunch; this is the local face of PACELC's "Else Latency/Consistency" axis.

Replication lag and the bugs it causes

With async followers, a read against a follower may return stale data because the follower hasn't applied the leader's latest writes yet. Under normal load lag is milliseconds; under a write spike, a long-running follower query, or a network hiccup it can blow out to seconds or minutes. This eventual consistency is fine for many reads but produces three concrete, user-visible anomalies that a principal must name and fix:

Read-after-write (read-your-writes)

A user submits a comment (write → leader), then reloads the page (read → a lagging follower) and their comment is gone. The fix is the read-your-writes guarantee: a user always sees their own writes. Implementations:

  • Read from the leader for data the user may have just modified (e.g. their own profile), follower for everything else.
  • Track the client's last-write timestamp/log position (an LSN) and route reads to a follower that has caught up to it — or wait until it has.
  • Sticky routing for a short window after a write so the client hits a single, monotonically-advancing replica.

Monotonic reads

A user refreshes twice and sees a comment appear, then vanish — because the two reads hit followers with different lag (one ahead, one behind), making time appear to run backward. The monotonic reads guarantee says a user never sees the clock go backward. The standard fix is to pin each user to one replica (hash of user ID → replica), so their successive reads never regress.

Consistent prefix reads

Across partitions, an observer can see an answer before the question it answers, because the two writes landed on differently-lagged partitions. The consistent prefix guarantee preserves causal order. Hardest to provide once data is partitioned; relates directly to Time, Clocks and Ordering.

PitfallThese are client-centric consistency guarantees, weaker than linearizability and far cheaper. Strong engineers reach for them first. Do not promise "we'll just make it strongly consistent" by default — that means routing everything to the leader (killing read scale) or running full consensus on the read path (killing latency). Match the guarantee to the actual product requirement per read path.

How replication is implemented on the wire

The replication log can be encoded at four levels, each with sharp trade-offs that come up in real migrations and incidents:

MethodWhat shipsStrengthFailure mode
Statement-basedThe literal SQL (UPDATE…)CompactNondeterminism: NOW(), RAND(), auto-increment, triggers diverge across replicas
WAL / physicalByte-level changes to storage pagesExact, cheap to applyTightly coupled to storage format/version — blocks zero-downtime upgrades and cross-engine replication
Logical / row-basedLogical row changes (this PK, these column values)Decoupled from storage; enables version upgrades, CDC, heterogeneous targetsMore verbose; needs primary keys; some operations awkward to express
Trigger-basedApp-level rows captured by DB triggersFlexible, selectiveHigher overhead, more bug-prone — last resort

The industry trend is decisively toward logical/row-based logs (MySQL binlog_format=ROW, PostgreSQL logical decoding / wal_level=logical) precisely because the same log that drives replicas can be tapped for Change Data Capture — feeding search indexes, caches, data warehouses, and event streams. This is the bridge to Stream Processing and CDC and Kafka: a replication log is an ordered event stream. Physical WAL shipping remains common for hot-standby/DR where source and target are identical versions.

# MySQL semi-sync, row-based, with a bounded fallback window
binlog_format            = ROW
rpl_semi_sync_master_enabled        = 1
rpl_semi_sync_master_timeout        = 1000   # ms; downgrade to async if no ack
rpl_semi_sync_master_wait_for_slave_count = 1

# PostgreSQL: one sync standby, others async; quorum form also supported
synchronous_commit       = on
synchronous_standby_names = 'FIRST 1 (replica_a, replica_b)'
wal_level                = logical            # enables CDC / logical replication

Failover and split-brain

When the leader fails, a follower must be promoted. The steps — detect failure, choose the most up-to-date follower, reconfigure clients and remaining followers to the new leader — are deceptively dangerous. The hard problems:

  • Lost writes: with async replication, the promoted follower may be behind. Acknowledged writes that hadn't replicated are gone. If the old leader rejoins and external systems already saw newer IDs (e.g. a cache or another DB keyed off the old auto-increment), you get corruption that outlives the incident.
  • Detection ambiguity: you cannot distinguish a dead leader from a slow leader or a partitioned one (the FLP intuition from Consensus and Raft). Too aggressive a timeout causes needless failovers; too lax extends the outage.
  • Split-brain: two nodes both believe they are leader and both accept writes — the catastrophic failure. It diverges the dataset and there is no automatic merge for a single-leader system.
PitfallNaive "promote a replica when the leader looks down" logic causes split-brain. Defenses: fencing tokens (a monotonic epoch the storage layer checks so a deposed leader's writes are rejected — see Distributed Locking Service), STONITH ("shoot the other node in the head"), and routing the leadership decision through a consensus group / coordination service rather than ad-hoc heartbeats. This is why production systems delegate failover to etcd, ZooKeeper, or a built-in Raft group — see Leader Election and Coordination.

Note the difference between plain async followers (MySQL classic, vulnerable to lost writes on promotion) and consensus-replicated logs (Raft/Paxos groups in CockroachDB, etcd, MongoDB's replica-set election): the latter only ever commit a write once a majority has it, so a newly elected leader provably holds all committed entries. That is the Leader Completeness property — failover without data loss, at the cost of a majority round-trip per write.

Multi-leader and write conflicts

Multi-leader replication permits writes at several leaders, typically one per datacenter, replicating asynchronously to the others. It is attractive for multi-region active/active (each region writes locally with low latency and tolerates inter-region partitions) and for offline-capable clients (each device is effectively a leader). The price is that two leaders can concurrently modify the same record — a write conflict — which single-leader never has.

Conflict avoidance is the cleanest answer where feasible: route all writes for a given record to the same home leader, so concurrent conflicting writes never arise. When you cannot avoid them, you must resolve:

  • Last-write-wins (LWW): attach a timestamp/version, keep the highest. Simple but silently discards data and is hostage to clock skew (see Time, Clocks and Ordering). Cassandra's default; acceptable only when losing one of two concurrent writes is genuinely fine.
  • Application-defined merge: on conflict, hand both versions to the app (or surface siblings to the user, as Riak does) to merge with domain knowledge.
  • CRDTs: data types mathematically designed so concurrent updates always merge deterministically without losing intent — the principled fix. See CRDTs (counters, sets, and collaborative-editing text in Collaborative Editing).

Conflict detection itself relies on causality tracking — version vectors distinguish "B happened after A" (safe overwrite) from "A and B are concurrent" (true conflict). LWW timestamps cannot tell these apart, which is why naive LWW loses data even without clock skew.

Leaderless (Dynamo-style)

Leaderless replication abandons the leader entirely: the client (or a coordinator node) writes to several replicas and reads from several. Correctness comes from quorum overlap: with N replicas, requiring W acks on write and R responses on read, if W + R > N the read and write sets are guaranteed to intersect on at least one replica holding the latest value. A common config is N=3, W=2, R=2.

Because some replicas may have missed a write (they were down or slow), leaderless systems repair divergence in the background with two mechanisms, both covered in Anti-Entropy and Gossip:

  • Read repair: on a read that returns stale values from some replicas, the coordinator writes back the freshest value — opportunistic, fixes hot data.
  • Anti-entropy / Merkle trees: a background process compares replicas (via Merkle trees to find divergent ranges cheaply) and reconciles them — fixes cold data that's rarely read.
PitfallW + R > N is not linearizability. Sloppy quorums (writing to any N reachable nodes during a partition, with hinted handoff) break the overlap guarantee; concurrent writes still need conflict resolution; and there are well-known edge cases (a write that succeeds on fewer than W nodes but isn't rolled back). Treat Dynamo-style quorums as a strong eventual consistency knob, not a strict-consistency one. CAP and PACELC are the framing tools here.

The architectural pattern is: leaderless systems lean AP (stay writable during partitions, reconcile later), single-leader-with-consensus leans CP (stay consistent, sacrifice availability of the affected shard during failover). This maps directly onto the SQL vs NoSQL divide — Dynamo-style stores (Cassandra, DynamoDB, Riak) chose leaderless availability, while traditional RDBMS and NewSQL chose leader-based consistency.

Principal-level mastery check

What strong engineers doThey pick a topology from requirements, not habit: single-leader by default for its clean consistency story; multi-leader only when multi-region write latency or offline clients force it (and then they have a conflict-resolution answer ready); leaderless only when continuous write availability under partition is worth eventual consistency. They name the specific replication-lag anomaly each read path can suffer and apply the cheapest guarantee that fixes it — rather than blanket-routing to the leader. And they never confuse adding read replicas with scaling writes.

Probing questions to ask engineers:

  • "Your design serves reads from async followers. A user posts a comment and refreshes — it's gone. Which guarantee is violated, and exactly how do you implement the fix without sending all reads to the leader?"
  • "Walk me through what happens to the last 200ms of acknowledged writes when the leader crashes under async replication. How do you bound or eliminate that loss window, and what does it cost?"
  • "Two nodes both think they're leader. How did that happen, what damage does it do, and what mechanism prevents it — be specific about fencing tokens or consensus?"
  • "You chose multi-leader active/active across three regions. Two regions edit the same row concurrently. What's your conflict-resolution strategy, and what data could LWW silently destroy?"
  • "With N=3, W=2, R=2, give me a concrete sequence where a read still returns stale data. When does W+R>N fail to give you what people think it does?"
  • "Why would you switch from physical WAL shipping to logical replication, and what does that unlock beyond replication itself?"
