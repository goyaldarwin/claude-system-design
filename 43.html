Search & Elasticsearch

A database can find a row by its key in microseconds, but ask it "which of a billion documents contain these words, ranked by relevance?" and it falls over — that's a full scan. Search engines solve a fundamentally different problem with a fundamentally different data structure: the inverted index. Elasticsearch is the inverted index made distributed, near-real-time, and operable — and its design is a set of decisions about indexing, sharding, and querying.

The trade-off Fast full-text and ranked search over huge corpora — in exchange for a second copy of your data, near-real-time (not instant) visibility, and memory-hungry operation. An inverted index turns "find documents containing these terms, ranked" from an impossible scan into a fast lookup. You pay by maintaining a separate search store (kept in sync with your source of truth), accepting a ~1-second indexing delay, and feeding it a lot of RAM.

What a search engine is — the inverted index

A normal index (a B-tree) maps a key → its row: great for "give me document 42." Search asks the reverse: "give me every document containing the word 'distributed', ranked by relevance." The data structure that answers it is the inverted index: a map from each term → the list of documents containing it (the postings list). To find documents with two words, you intersect their postings lists — fast set operations, not a scan.

Getting from raw text to terms is analysis: tokenize into words, lowercase, remove stop-words, stem ("running"→"run"). The same analyzer must run at index time and query time, or your query terms won't match the indexed terms. Relevance ranking (BM25) then scores how well each matching document fits the query.

The one-line frameA search engine trades a second, derived copy of your data and near-real-time freshness for fast full-text, ranked, faceted queries a database can't do. It is almost never your system of record — it's a specialized read store you keep in sync with one, and its whole architecture (analysis, shards, segments) serves that one query shape.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to when to use it.

Scenario 1 — the search that returns nothing You index product titles and search for "iPhone" — zero results, even though "iPhone 15 Pro" is clearly in the index. The field was mapped as an exact keyword, not analyzed text, so the only indexed term is the whole literal title. What's the mismatch between how the data was indexed and how you're querying it, and what's the rule that prevents it?
Scenario 2 — Plaid, March 2019 (real incident) Plaid ran log analytics on hosted Elasticsearch. A few unbounded, memory-intensive queries (huge aggregations) caused data nodes to run out of heap and crash; as they died, load shifted to the survivors, which also OOM'd — the cluster went red over 10–15 minutes, repeatedly. Why is one expensive query able to take down a whole search cluster, and what guards against it?
Scenario 3 — "I just saved it, why isn't it searchable?" A user creates a document; a colleague searches for it half a second later and it's not there, then a second later it appears. Nothing is broken. Why is search near-real-time rather than instant, and what would forcing instant visibility cost?

Scenario 1 is analysis & mappings. Scenario 2 is it's a separate store you must protect. Scenario 3 is near-real-time & segments. Each maps to a decision below.

The five decisions that define a search design

Designing search is these five choices. Each has a named failure nested under it.

Decision 1 — The inverted index, mappings & analyzers

You decide, per field, how it's indexed: as analyzed text (tokenized, for full-text search) or an exact keyword (whole-value, for filters/aggregations/sorting) — and which analyzer (tokenizer + filters: lowercasing, stemming, stop-words, language rules) processes it. This mapping is the contract that makes search work: it determines what terms land in the postings lists.

The failure that happens when you get this wrong — analyzer mismatch (zero hits) This is Scenario 1. The iron rule: the analyzer at index time and the analyzer at query time must agree. If you index "iPhone 15 Pro" as a single keyword term but run a full-text match for "iphone," nothing matches — the index has the literal phrase, not the token "iphone." Likewise mismatched stemming/lowercasing silently kills hits. Map full-text fields as text with the same analyzer you query with (often the field stores both a text and a keyword sub-field, for searching and for exact filtering/sorting respectively). Most "search returns nothing" bugs are an analysis mismatch.

Decision 2 — Sharding & replicas

An index is split into shards (each a self-contained Lucene index) so it can exceed one node and search in parallel; each shard has replicas for availability and read throughput. A query scatters to all shards and gathers their top results. Shard count is fixed at index creation (you reshard by reindexing), so it's a sizing decision: enough shards to parallelize and grow, not so many that overhead dominates.

The failure that happens when you get this wrong — over-sharding Each shard carries fixed overhead (memory for its segments, file handles, cluster-state bookkeeping) and every query fans out to all of them. Hundreds of tiny shards means huge per-query coordination cost and wasted heap — a common cause of a struggling cluster. The guidance: keep shards in the tens-of-GB range, total shards proportional to data and node heap, and use time-based indices (roll over daily/weekly) for logs rather than one ever-growing index. Under-sharding (one giant shard) is the opposite mistake — no parallelism, can't grow.

Decision 3 — Query context vs filter context

Elasticsearch evaluates clauses in one of two modes. Query context asks "how well does this match?" and computes a relevance score (used for full-text ranking). Filter context asks "does this match, yes or no?" — no scoring, and the results are cacheable. Use query context for the relevance part (the search words); use filter context for exact constraints (status = active, date in range, category = X).

The failure that happens when you get this wrong — scoring what you should filter Putting exact constraints (a boolean flag, a date range, a category) into query context wastes work computing meaningless relevance scores for them and skips the filter cache — so every request recomputes what could be cached. On a high-QPS endpoint that's a large, needless CPU cost and worse latency. Rule: only the genuinely "how relevant" parts belong in query context; every yes/no constraint goes in filter context to be cached. Getting this split right is one of the biggest easy search-performance wins.

Decision 4 — Near-real-time indexing & segments

Lucene writes new documents into immutable segments. New docs aren't searchable until a refresh (default ~every 1 second) makes a new segment visible — this is why search is near-real-time, not instant. Many small segments accumulate, so a background merge combines them (like LSM compaction). Durability comes from a translog (a WAL) replayed on crash.

The failure that happens when you get this wrong — refresh & merge pressure This is Scenario 3. The ~1-second refresh delay is the price of efficient indexing — forcing instant visibility (refresh=true on every write, or a tiny refresh interval) creates a flood of tiny segments, hammering CPU/IO with merges and tanking indexing throughput. For bulk loads you raise the refresh interval (or disable it) to batch documents into fewer, larger segments, then refresh once. The near-real-time delay is a feature: accept ~1s visibility and you get far higher indexing throughput. (Like an LSM, merges compete with your writes.)

Decision 5 — Keeping it in sync (it's a copy, and you must protect it)

Elasticsearch is almost never your system of record — it's a derived read store fed from a database. You must decide how to keep it current (dual-write → drift; better: CDC/the outbox so every committed DB change flows to the index) and how to protect a memory-hungry, shared cluster from any one tenant or query.

The failure that happens when you get this wrong — drift, and the unbounded-query OOM Two related failures. Drift: writing to the DB and the index separately (a dual write) lets them diverge; feed the index from the DB's committed log via CDC instead, and rebuild by replaying. The OOM cascade (Scenario 2 — Plaid): Elasticsearch aggregations and deep pagination can use unbounded heap; one giant query crashes a data node, load shifts, survivors crash too, and the cluster goes red. Guards: circuit breakers (Elasticsearch's heap limits that reject a query before it OOMs the node), query/aggregation limits, search.max_buckets, banning deep from/size pagination (use search_after), and isolating heavy analytical workloads from interactive ones. Treat the cluster as shared, fragile, and in need of resource limits.

At a glance — when to use it (and when not)

Use Elasticsearch for…Don't use it as…
Full-text search with relevance rankingYour primary system of record / source of truth
Log/metrics analytics & aggregations (the ELK stack)A transactional store (no real ACID transactions)
Faceted / filtered search (e-commerce)A key-value cache (a cache is cheaper/faster for point lookups)
Geo / autocomplete / fuzzy matchingA relational join engine
The rule in one lineUse a search engine when the query is "find & rank documents by content/facets," keep it as a derived store synced from your real database (via CDC), accept ~1s near-real-time visibility, and protect it with resource limits — one unbounded query can take the cluster down.

Self-check — can you reconstruct the three scenarios?

You read the answers above. Close the page and rebuild them.

  • Scenario 1 (zero hits): What's the analyzer mismatch behind "iPhone" returning nothing, and the rule that prevents it? When do you use text vs keyword?
  • Scenario 2 (Plaid OOM): Why can one unbounded query take down the whole cluster, and what guards (circuit breakers, limits, isolation) prevent it?
  • Scenario 3 (not searchable yet): Why is search near-real-time, what's a segment/refresh, and what does forcing instant visibility cost?
  • Why does putting exact constraints in query context hurt, and what does filter context buy?
  • Why is shard count a sizing decision fixed at creation, and what's over-sharding?

The shape to remember: search is 5 decisions — inverted index & analysis, shards & replicas, query vs filter context, near-real-time segments, sync & protect the copy — each with a nested failure.

Famous implementations

The engines made concrete. Same facets — what it is, the model, and the failure to watch.

Elasticsearch / OpenSearch

  • What it is. Distributed search + analytics over Lucene; the heart of the ELK/Elastic and OpenSearch stacks.
  • Model. Sharded/replicated indices, near-real-time refresh, JSON query DSL, aggregations, circuit breakers.
  • The failure to watch. Unbounded aggregation/pagination OOM (Plaid, Scenario 2); over-sharding; analyzer mismatches; split-brain historically (fixed by the 7.x quorum-based master election — the same consensus idea).

Lucene / Solr

  • What it is. Lucene is the underlying Java search library (segments, postings, BM25); Solr is the older server built on it.
  • Model. Single-node inverted index (Lucene) that Elasticsearch/Solr distribute; immutable segments + merges.
  • The failure to watch. Segment merge pressure (Scenario 3) is a Lucene-level cost inherited by everything built on it.

Vector / semantic search (FAISS, k-NN, pgvector)

  • What it is. Search by embedding similarity (nearest-neighbor in vector space) rather than exact terms — powers semantic and AI retrieval.
  • Model. Approximate-nearest-neighbor indexes (HNSW, IVF) over high-dimensional vectors; increasingly bundled into Elasticsearch/OpenSearch alongside the inverted index.
  • The failure to watch. ANN is approximate (recall vs latency trade), and large vector indexes are very memory-hungry — the same RAM pressure as classic search, amplified.

Build it yourself

These click when you build the index by hand. Work each before opening the approach.

Build an inverted index by hand

Index three documents and answer a two-word query, then explain how relevance ranking and the analyzer fit in.

Reveal the approach

Docs: D1 "the quick fox", D2 "quick brown dog", D3 "lazy fox sleeps". Analyze each (lowercase, drop stop-word "the", stem): D1{quick,fox}, D2{quick,brown,dog}, D3{lazy,fox,sleep}. Inverted index (term → postings): quick→[D1,D2], fox→[D1,D3], brown→[D2], dog→[D2], lazy→[D3], sleep→[D3]. Query "quick fox": intersect postings of quick[D1,D2] and fox[D1,D3] → {D1}. Rank by BM25 (term frequency × inverse document frequency, length-normalized) — D1 scores high because it contains both query terms. Crucially the query must run the same analyzer (lowercase+stem) or "Quick" / "foxes" wouldn't match the indexed terms — the analyzer-agreement rule.

Keep search in sync with the source of truth

Your products live in Postgres; search is in Elasticsearch. Design how changes propagate so search is current within seconds and never silently drifts — and how you rebuild the index.

Reveal the approach

Don't dual-write. Use CDC (Debezium) tailing Postgres's WAL → a Kafka topic of committed changes → a consumer that upserts/deletes documents in Elasticsearch (idempotent by primary key, since delivery is at-least-once). The DB stays the single source of truth; the index is derived from its committed log, so it can't diverge. To rebuild after a mapping change or corruption, reindex: create a new index with the new mapping and replay the full change history (or a bulk export) into it, then atomically swap an alias from old to new index — zero-downtime. Protect the cluster during bulk reindex by raising the refresh interval and rate-limiting (Scenario 3 + Scenario 2 guards).
