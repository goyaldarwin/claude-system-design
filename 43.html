Search and Elasticsearch

How full-text search engines actually work, and the six dimensions that drive every Elasticsearch design decision.

The bottleneck Latency & finite capacity. fast full-text/relevance via inverted indexes + sharding, paid in near-real-time indexing.

In this lesson

  • 1. The inverted index — how search really works
  • 2. Shards & replicas

Almost every system that lets users search by free text eventually arrives at a search engine, and in practice that engine is Lucene wearing an Elasticsearch (or OpenSearch) coat. A principal-level answer is not "throw it in Elasticsearch." It is a defensible position across six axes: how the index is structured, how it is partitioned, how fields are mapped, how queries are scored, how writes become visible, and how the cluster is shaped. This lesson walks those six dimensions, each with the mechanism, the numbers, and the failure mode that matters.

1. The inverted index — how search really works

A relational B-tree index answers "find the row whose id = 42" or "rows where price BETWEEN 10 AND 20." It is built for looking up a document by an attribute value. Full-text search asks the inverse question: "find every document that contains the word consensus." A B-tree on a TEXT column cannot help — it indexes the whole string, not the words inside it, and LIKE '%consensus%' degrades to a full scan.

The data structure built for this is the inverted index. Instead of mapping document → terms, it maps term → the list of documents containing it. That per-term list is a postings list:

term        postings (docId : term-frequency : positions)
---------   ----------------------------------------------
"raft"      [ 3:2:(5,40), 17:1:(12), 88:1:(2) ]
"consensus" [ 3:1:(6),   17:3:(1,13,40) ]
"quorum"    [ 88:4:(...) ]

To answer "raft consensus" the engine fetches the two postings lists and intersects them — documents 3 and 17 contain both. Because postings lists are sorted by docId, intersection is a linear merge, and the lists themselves are delta-and-bitpack compressed so they stay cache-friendly. This is the same family of structures behind Twitter Search, where tweets are tokenized and fanned into postings as they arrive.

Two pieces make the magic work:

  • Tokenization / analysis: the text "Raft, the consensus algorithm!" is broken into terms (raft, consensus, algorithm), lowercased, punctuation stripped, and optionally stemmed (running → run). The same analyzer runs at index time and query time so the terms line up.
  • The term dictionary: Lucene stores the sorted set of terms in an FST (finite-state transducer), giving O(term length) lookups and cheap prefix/wildcard traversal — the basis of autocomplete and fuzzy matching.

Relevance comes from the numbers already in the postings: term frequency, document frequency, and field length feed the BM25 scoring function (Lucene's default since 6.0), so a rare word that appears many times in a short document ranks that document highly. This is why search returns a ranked result set rather than the unordered set a SQL WHERE returns.

Key insightA B-tree is optimized for "which value does this row have?" An inverted index is optimized for "which rows contain this value?" They are duals, not substitutes. That is the whole reason a search engine exists alongside your primary database rather than replacing it — see Storage Engines: LSM vs B-Tree.

2. Shards & replicas

An Elasticsearch index is split into primary shards, and each primary shard is an independent Lucene index that can live on any data node. This is how a single logical index scales beyond one machine — the same partitioning idea as Sharding / Partitioning, applied to search. A document is routed to a shard by shard = hash(routing_key) % number_of_primary_shards; by default the routing key is the document _id.

The consequence of that formula is the single most-tested fact about Elasticsearch: the primary shard count is fixed at index creation. Change the divisor and every document hashes to a different shard, so the routing breaks. To re-shard you must reindex into a new index with a new shard count (the _reindex or _split/_shrink APIs) and swap an alias over — you cannot mutate it in place.

Replica shards are full copies of a primary on different nodes. They do two jobs: high availability (a replica is promoted if its primary's node dies) and read throughput (searches round-robin across primary and replicas). Replica count, unlike primary count, can be changed live. So the design rule is:

  • Primaries — set for total data volume and write parallelism; pick carefully, you live with it.
  • Replicas — dial up for query QPS and resilience; cheap to change at runtime.

Shard sizing is where most clusters go wrong. Rules of thumb that hold up in production:

  • Target tens of GB per shard — roughly 10–50 GB, and keep shards under the ~2 billion-document Lucene hard limit well in advance.
  • Keep total shard count proportional to heap: aim for < 20 shards per GB of JVM heap on a node (a 30 GB-heap node tops out around 600 shards).
  • Avoid over-sharding: every shard is a separate Lucene index with its own memory overhead (segment metadata, FSTs, thread-pool slots). A thousand 50 MB shards is far worse than ten 5 GB shards — the cluster state balloons, the master node struggles, and every search fans out to a thousand tiny tasks.
Trade-offMore primary shards raise write parallelism and let you spread data wider, but every query must scatter to all of them and gather/merge the results — scatter-gather latency is bounded by the slowest shard. Fewer, larger shards lower coordination overhead but cap parallelism and make rebalancing chunkier. For time-series/log data, prefer one index per time window (e.g. daily) with a handful of shards each, managed by ILM, rather than one giant index.

3. Mappings & analyzers

The mapping is the schema: it declares each field's type and, for text, which analyzer processes it. The decision that trips up most people is text vs keyword:

TypeAnalyzed?Use for
textYes — tokenized into terms, full-text searchable, not usable for exact-match/sort/aggregationsBody, title, description — "find documents mentioning X"
keywordNo — stored as a single exact tokenStatus, tags, IDs, enums — exact filters, sorting, aggregations, term facets

The standard pattern is a multi-field: map a field as text for searching and add a .keyword sub-field for aggregating and sorting. You are designing the mapping for the query you intend to run — if you will never full-text search the field, make it keyword and save the analysis cost; if you will only ever match it loosely, you may not need the keyword variant.

An analyzer is a pipeline of three stages:

  1. Character filters — pre-process raw text (strip HTML, map & → and).
  2. Tokenizer — split into tokens (standard, whitespace, edge-ngram for autocomplete, pattern).
  3. Token filters — lowercase, stop-word removal, stemming, synonyms, ASCII-folding.

Choosing the analyzer is a product decision masquerading as a config knob: stemming makes running match run (good for recall) but can over-match; edge-ngrams enable type-ahead but multiply index size; language analyzers handle accents and compound words.

FootgunDynamic mapping auto-infers a type the first time it sees a field, which is convenient in a demo and dangerous in production. A field that arrives once as "12345" becomes text+keyword forever; a deeply nested JSON blob with unbounded keys triggers mapping explosion — thousands of fields bloat cluster state and can take the cluster down. Set "dynamic": "strict" (or false) on anything user-supplied, and define mappings explicitly.

4. Query vs filter context

Every clause in an Elasticsearch query runs in one of two contexts, and knowing which is the difference between a 5 ms and a 500 ms request:

  • Query context answers "how well does this document match?" — it computes a relevance _score via BM25. Use it for the free-text part the user actually cares about ranking by. It is more expensive and is not cached, because the score depends on the whole result set.
  • Filter context answers a yes/no: "does this document match?" — no scoring. Use it for structured constraints (status: active, created >= 2024-01-01, tenant_id: 42). Filters are cacheable.

The canonical shape is a bool query that puts the scored part in must and everything else in filter:

{ "query": { "bool": {
  "must":   [ { "match": { "body": "raft consensus" } } ],   // scored
  "filter": [ { "term":  { "status": "published" } },        // cached, no score
              { "range": { "created": { "gte": "2024-01-01" } } } ]
}}}

Filters hit the filter cache (a per-segment bitset of matching docs). Once status: published has been evaluated for a segment, the bitset is reused across every subsequent query on that segment — effectively free. The mistake to avoid is dumping exact-match constraints into must: they get needlessly scored and never cached, so a tenant-isolation clause that should be a free filter becomes a per-query CPU cost.

5. The indexing model — near-real-time & segments

Elasticsearch is near-real-time, not real-time. A document you index is not instantly searchable. Understanding why requires the Lucene segment model, which is structurally close to an LSM tree: writes accumulate in memory and are periodically flushed to immutable on-disk files.

  1. An indexed document lands in an in-memory buffer and is appended to the translog (a write-ahead log) for durability.
  2. On refresh — by default every 1 second — the buffer is written to a new in-memory/page-cache segment and becomes searchable. This refresh, not the write itself, is what makes the doc visible. Raising refresh_interval to 30s (or disabling it during a bulk load) dramatically cuts indexing cost at the price of staleness.
  3. Segments are immutable. An update writes a new document and marks the old one deleted (a tombstone); a delete is just a tombstone. Nothing is mutated in place.
  4. On flush, segments are fsync'd and the translog is truncated — this is the real durability boundary.
  5. Because each refresh births a new small segment, a background merge continually combines small segments into larger ones, physically dropping tombstoned docs. Merging is the source of write amplification — the same data is rewritten several times — and is the main consumer of disk I/O on a write-heavy cluster.

Two operational levers follow directly:

  • Bulk indexing. Index via the _bulk API in batches of a few MB / thousands of docs. One request per document wastes a network round-trip and a refresh cycle on each. For heavy backfills: set refresh_interval: -1, set number_of_replicas: 0, bulk-load, then restore both — this can be an order of magnitude faster.
  • Durability vs latency. The translog fsyncs every 5s by default (async), so a node crash can lose up to ~5 s of writes. index.translog.durability: request fsyncs on every request — safer, slower.
MechanismThe 1-second refresh is exactly why "I wrote it but search can't find it yet" is normal, not a bug. If a workflow needs read-your-write, either GET /index/_doc/{id} (the realtime get path reads the translog and bypasses refresh) or call ?refresh=wait_for on the write — never lower refresh_interval globally just to satisfy one path.

6. Cluster roles & scaling

An Elasticsearch cluster is not a uniform blob of nodes; nodes take roles, and separating them is how you scale and stay stable:

  • Master-eligible nodes — own the cluster state (mappings, shard locations, routing table) and elect a leader via a Raft-like quorum protocol. Run a dedicated, odd number (3 is standard) so a split brain cannot form — the same majority logic as Consensus & Raft. Keep them off the data path so a hot query can't destabilize cluster management.
  • Data nodes — hold shards, do the indexing and the actual search work. Scale these for storage and CPU.
  • Coordinating nodes — the scatter-gather brains: receive a search, fan it to the relevant shards, then merge and rank the partial results. Dedicated coordinators absorb the merge cost off the data nodes for high-fan-out queries.
  • Ingest nodes — run pre-processing pipelines (enrich, parse, geoip) before documents are indexed.

The principal framing is that indexing throughput and search throughput are separate scaling problems. Heavy writes are bottlenecked by merges and translog fsyncs on data nodes; heavy reads are bottlenecked by shard fan-out and merge cost. You scale write capacity by adding primaries/data nodes and tuning refresh; you scale read capacity by adding replicas and coordinators. Conflating them — "just add nodes" — is a junior answer.

For time-based data, hot / warm / cold tiers map index age to hardware cost: fresh, frequently-queried indices live on hot nodes (NVMe, lots of RAM); aging indices roll to warm (cheaper disks, fewer replicas) and then cold/frozen (object-storage-backed, searchable but slow), driven by Index Lifecycle Management. This is the search-engine version of the storage-tiering you would design for Object Storage.

On the consistency axis, Elasticsearch is a CP-leaning system for cluster metadata but tunable for data. A write succeeds when the primary and the configured number of replicas (wait_for_active_shards) acknowledge it; the default replication is synchronous to in-sync replicas, so a quorum acknowledges before the client sees success, but the refresh-then-visible delay means reads across replicas can momentarily diverge. The knobs you trade are exactly the ones from Consistency Models & PACELC: refresh interval vs freshness, translog durability vs write latency, replica acks vs availability.

When NOT to use Elasticsearch — and how to keep it in sync

The most important principal instinct: Elasticsearch is a search/analytics layer, not a system of record. Reasons not to make it your primary store:

  • No real transactions, no joins, no foreign keys — it is a denormalized document store.
  • The near-real-time visibility delay and the up-to-5 s translog window are wrong for "money must be durable the instant the API returns."
  • Reindexing to change a mapping or shard count is painful if it holds your only copy of the data.
  • Segment merges and analysis make it heavier per-write than a purpose-built OLTP store.

So the standard architecture is: keep the source of truth in a transactional database, and treat Elasticsearch as a derived read model. Avoid the dual-write anti-pattern — writing to the DB and to ES from application code in two steps — because a crash between the two leaves them permanently inconsistent with no clean recovery. Instead, drive ES from the database's commit log via Change Data Capture: a connector (Debezium, etc.) tails the WAL/binlog and streams changes into ES, typically through Kafka. This gives you a single durable ordering of changes, idempotent replay, and a clean rebuild path — the pattern covered in Stream Processing & CDC. Because the index is derived, you can blow it away and rebuild from the log at any time.

Finally, search and approximate structures often combine: a Bloom filter can short-circuit "does this term exist at all in this segment?" before touching the postings list, and HyperLogLog-style sketches back cardinality aggregations — useful when an exact distinct count over billions of docs would be too expensive.

Principal-level mastery check

What strong engineers doThey treat Elasticsearch as a derived read model fed by CDC, never as the system of record, and they justify it on the six axes — index structure, sharding, mapping, query/filter, indexing model, cluster shape — rather than reaching for it reflexively. They size shards in tens of GB, separate write-scaling from read-scaling, and put tenant/structured constraints in filter context for free caching.

Probing questions to ask engineers:

  • "You picked 30 primary shards. Justify the number — and what do you do in a year when the data has 5×'d?"
  • "A user creates a record and immediately searches for it but gets nothing. Walk me through why, and how you'd fix it without globally lowering refresh_interval."
  • "When would you map a field as keyword vs text, and what breaks if you sort or aggregate on a text field?"
  • "Your service writes to Postgres and to Elasticsearch in the same request. What's wrong with that, and what's the right pattern?"
  • "Indexing is keeping up but search latency spikes at p99. Where do you look first — shards, merges, query vs filter, or coordinator fan-out?"
  • "Why is the inverted index the right structure here and a B-tree the wrong one? When would you keep both?"
