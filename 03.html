Load Balancing

A load balancer spreads traffic across many servers so no single one is overwhelmed — and in doing so becomes the one box every request must pass through. Every decision here is about where to balance, how to choose a server, how to spot a dead one, and how to stop the balancer itself from being your single point of failure.

The trade-off Horizontal scale and resilience in exchange for an extra hop and a new critical component. Distributing load lets you add servers instead of buying bigger ones, and route around failures. In return you accept a routing hop on every request, the work of tracking which servers are healthy, and a new failure: the balancer is now on the path of every request.

What a load balancer is

A load balancer (LB) sits in front of a pool of identical servers (the backends) and spreads incoming requests across them according to some rule, so that no single backend takes all the traffic. It also continuously tracks which backends are healthy and stops routing to any that are down, slow, or erroring. You can place one at every tier — between users and the web servers, between web and application servers, between the application and the cache or database.

The one-line frameThe load balancer exists to remove the single point of failure that one server represents — so the cardinal sin is to build it in a way that makes the balancer itself the new single point of failure. Design Decision 4 first.

Three scenarios — decide before you read

Don't scroll past these. Decide your answer first — the rest of the lesson is the answer. If you already know all three cold, skip to the decision table.

Scenario 1 — the chatty gRPC client You front a fleet of gRPC services with a fast L4 load balancer. One client opens a single connection and sends 10,000 req/s over it; the other clients are quiet. Across your 8 backends, how is that load distributed — and why does adding more backends not help?
Scenario 2 — the stuck backend You balance with plain round-robin across 10 backends. One backend's disk is failing: it still accepts connections but every request to it takes 30 seconds instead of 50 ms. What fraction of requests does round-robin keep sending to it, and what does that do to your tail latency?
Scenario 3 — the routine deploy You roll out a new version: Kubernetes terminates old pods one at a time and starts new ones. During each swap, a burst of requests returns 502 Bad Gateway. The pod is being shut down gracefully — so where are those 502s coming from?

Scenario 1 is the L4 connection hot-spot. Scenario 2 is the stuck-backend / wrong-algorithm problem. Scenario 3 is the connection-draining race on deploy. Each is what happens when one of the four decisions below is made wrong — and each shows up in real production every day.

A load balancer is defined by four decisions

Every load-balancing setup — a hardware appliance, NGINX, or an Envoy sidecar — is just these four choices. Make each well and traffic flows smoothly; make one wrong and you get a named, predictable failure. We take them in order, and each failure lives under the decision that causes it.

Decision 1 — Where do you balance? (L4 vs L7)

The first choice is which layer of the network the balancer operates at — and it determines everything it can and cannot do.

  • An L4 (transport) load balancer decides using only the connection's 5-tuple (source IP and port, destination IP and port, protocol) and then forwards packets — it never reads the request. It is cheap, extremely fast (line-rate with hardware or eBPF), and protocol-agnostic: raw TCP, TLS passthrough, gRPC, anything. The catch: it pins a whole connection to one backend.
  • An L7 (application) load balancer terminates the connection, reads the request, and routes per request. This unlocks routing by URL path or header, retries on safe methods, per-request metrics, and TLS termination. The cost is CPU (it parses and decrypts) and a little latency.
The rule of thumbUse L4 for raw north-south throughput, TLS passthrough, and non-HTTP protocols. Use L7 when you need content-based routing, per-request fairness, or smart retries. Many real stacks layer both: an L4 edge in front of an L7 fleet (e.g. Envoy or NGINX).
The failure that happens when you get this wrong — the connection hot-spot This is Scenario 1. Because an L4 balancer pins an entire connection to one backend, and gRPC and HTTP/2 multiplex thousands of requests over one long-lived connection, all 10,000 req/s from that one chatty client land on the single backend its connection happens to be pinned to. Adding backends does nothing — the new ones get no share of that connection's traffic. The fix is to balance at L7, where the balancer sees individual requests (streams) and can spread them across the backend pool per request, not per connection.

Decision 2 — How do you pick a backend?

Once a request arrives, which backend gets it? Round-robin is the default, and it is almost never the best production choice, because it assumes every request is identical and every backend equally loaded — both false.

AlgorithmHow it picksWhen it wins / fails
Round-robin / weightedNext in rotation (weighted by capacity)Fine for uniform, short requests; blind to actual load — a stuck backend keeps getting its turn
Least-connectionsThe backend with the fewest active connectionsGood when request durations vary; can mislead if idle keep-alive connections aren't real work
Least-response-timeLowest (in-flight × recent latency)Best signal for uneven backends; needs accurate, low-noise latency telemetry
Power-of-two-choices (P2C)Sample 2 backends at random, send to the less loadedNear-optimal spread with almost no state; the modern default in service meshes
Consistent hashing (Maglev)Hash a key (client IP, session) to a backendPreserves affinity; minimal reshuffle when membership changes — but a skewed key can hot-spot
Why power-of-two-choices is the modern defaultPlain least-connections needs the balancer to track global load across every backend; with many balancer instances they all pile onto the same "least loaded" backend at once (a herd). P2C — sample two, pick the better — needs almost no shared state and gives a provably exponential improvement in worst-case load over pure random. Studies put it at roughly 23% lower average queue length and up to 80% lower worst-case hot-spotting versus round-robin. It degrades gracefully under stale information, which is why Envoy and Finagle ship it as standard. (Result from Mitzenmacher's "The Power of Two Choices in Randomized Load Balancing.")
The failure that happens when you get this wrong — the stuck backend This is Scenario 2. Round-robin is blind to load: it keeps sending every Nth request to the failing backend regardless of the fact that each one now takes 30 seconds. With 10 backends, ~10% of all requests get stuck behind the slow one, and because clients wait on them, your tail latency (p99) collapses even though 90% of backends are healthy. A load-aware algorithm — least-response-time or P2C — notices the backend's in-flight count and latency climbing and naturally steers away from it. (Passive health checking then ejects it entirely.)

Decision 3 — How do you know a backend is healthy?

A balancer is only as good as its picture of backend health. Two complementary mechanisms:

  • Active health checks — the balancer proactively probes each backend (e.g. GET /healthz every few seconds) with healthy/unhealthy thresholds and hysteresis (require N consecutive results before flipping state, so a single blip doesn't eject a good backend). Catches dead backends, but a shallow check can return 200 while the app's real dependency is down. Expose separate liveness vs readiness signals.
  • Passive health checks / outlier ejection — the balancer watches real traffic and ejects a backend that returns a burst of 5xx, connection failures, or high latency (this is how a stuck backend gets removed). Catches "grey" failures an active probe misses, and reacts in real time.
The failure that happens when you get this wrong — the ejection cascade Eject too aggressively and you cause the outage you were trying to prevent. If a shared dependency blips and every backend briefly returns 5xx, naive outlier ejection removes them all — routing 100% of traffic onto the last backend standing, which instantly collapses under the load, and now you have a total outage from a transient blip.
Why this is also a retry-storm problem. The same dynamic plays out with retries: a system handling 300 QPS sees latency cross the timeout threshold, every client retries, and offered load jumps to ~560 QPS — a documented amplification pattern (1 original + N retries can be a 6× multiplier). The infrastructure built for reliability becomes a failure multiplier. The guardrails: an ejection cap (max_ejection_percent — never eject more than, say, 50% of the pool), a "panic threshold" (when too many backends look unhealthy, assume the health signal is wrong and spray to everyone), retry budgets, and jittered backoff.

Decision 4 — How is the balancer itself not a single point of failure?

Drawing one LB box quietly reintroduces the SPOF the balancer was meant to remove, and makes it a throughput ceiling too. The mitigations, roughly by scale:

  • Active-active pair with a floating VIP — a heartbeat protocol (VRRP, via keepalived) lets a standby take the virtual IP if the active one dies. Cheap, but the survivor must absorb full load.
  • ECMP + many LB nodes — routers spread flows across a fleet of equal-cost balancers; with Direct Server Return the responses bypass the LB entirely. This is how hyperscale L4 tiers handle terabits.
  • GSLB — above one region you steer users by GeoDNS (simple, but slow failover because clients cache DNS records) or Anycast (near-instant failover, but BGP can break long-lived connections mid-flight).
The failure that happens when you get this wrong — deploy 502s (connection draining) This is Scenario 3. Rolling a deploy must not drop in-flight requests, but the naive setup does — and "graceful shutdown" is not enough on its own.
The Kubernetes endpoint-removal race. When a pod is terminated, two things start at the same time with no ordering guarantee: (1) the kubelet sends the pod SIGTERM and it begins shutting down, and (2) the endpoint controller starts removing the pod's IP from the load balancer's backend list. Because removal must propagate to every kube-proxy/LB node, the pod often exits before the LB stops sending it traffic — so requests land on a process that is gone, returning 502s. The fix is connection draining done in the right order: (1) flip readiness to fail so the LB stops sending new traffic, (2) keep serving in-flight requests, (3) wait for de-registration to propagate — a preStop sleep that outlasts the propagation delay — then let the process exit.

Self-check — can you reconstruct the three scenarios?

You read the answers above. Close the page and see if you can rebuild them — that is the test, not recognition.

  • Scenario 1 (chatty gRPC client): Why does an L4 balancer concentrate one multiplexed connection's traffic on a single backend, why does adding backends not help, and what does moving to L7 change?
  • Scenario 2 (stuck backend): Why does round-robin keep feeding a slow-but-accepting backend, what does that do to p99, and which load-aware algorithm steers around it — and how does P2C beat plain least-connections?
  • Scenario 3 (deploy 502s): What are the two parallel, unordered tracks during pod termination, why does that produce 502s despite graceful shutdown, and what is the correct draining sequence?
  • When do you choose L4 over L7, and why do hyperscale stacks layer both?
  • Why can aggressive outlier ejection cause the outage it was meant to prevent, and what caps prevent it?
  • How do you stop the balancer from being a SPOF — and at what scale do you move from a VIP pair to ECMP+DSR to GSLB?

The shape to remember: a load balancer is 4 decisions — where you balance (L4/L7), how you pick a backend, how you detect health, how it avoids being a SPOF — and each wrong choice has a named failure: connection hot-spot, stale-load herd, ejection cascade, deploy 502s.

Famous implementations

Three pieces of software cover most of what you'll meet. Each entry below breaks down the same facets so you can compare them directly — what it is, the layer it works at, configuration model, key strength, the failure to watch, and when to reach for it.

NGINX — the ubiquitous software LB / reverse proxy

Originally a web server, now the most widely deployed software reverse proxy and L7 load balancer.

  • Layer. Primarily L7 (HTTP/HTTPS/gRPC), with an L4 stream module for raw TCP/UDP.
  • Configuration model. Static config files, reloaded on change — simple and battle-tested, but dynamic backend changes traditionally need a reload (the commercial NGINX Plus and the Ingress controllers add dynamic upstreams).
  • Key strength. Extremely efficient event-driven core; doubles as web server, cache, and TLS terminator. The default front door for a huge share of the web and the most common Kubernetes Ingress.
  • Algorithms. Round-robin, weighted, least-connections, IP-hash.
  • The failure to watch. The classic config-reload model means service discovery has to be wired in separately; a stale upstream list keeps sending traffic to dead backends until reloaded.
Reach for NGINX when…you want a proven, low-overhead L7 front door, reverse proxy, or Kubernetes Ingress, and your backend set is relatively stable or managed by an Ingress controller.

HAProxy — the high-performance L4/L7 workhorse

A dedicated load balancer (not a web server) known for raw performance and reliability.

  • Layer. Both L4 (TCP) and L7 (HTTP) in one binary.
  • Configuration model. A single, expressive config file; a runtime API lets you change server state and weights without a full reload.
  • Key strength. Mature, rich health-checking and observability, very high throughput per core. A long-standing default where a pure, fast LB is wanted rather than a web server.
  • Algorithms. Round-robin, least-connections, source-hash, URI-hash, and more, with per-server weights.
  • The failure to watch. Like NGINX, it is a centralized middlebox — you still must make it highly available (a VIP pair) so it isn't the SPOF.
Reach for HAProxy when…you want a dedicated, high-throughput L4 or L7 balancer with deep health-check and stats tooling, and you don't need it to also serve content.

Envoy — the modern mesh data plane

A modern L7 proxy designed for cloud-native, east-west microservice traffic; the data plane behind most service meshes (Istio, Linkerd's earlier versions, Consul).

  • Layer. L7-first (HTTP/1.1, HTTP/2, gRPC), with L4 support.
  • Configuration model. Dynamic — driven by a control plane over APIs (the xDS protocol), so backends, routes, and policy update live with no restart. This is the key difference from the file-reload model.
  • Key strength. Ships P2C load balancing, outlier ejection, retries, circuit breaking, and mTLS out of the box, with first-class observability. Runs as a per-pod sidecar so the application just calls localhost.
  • Algorithms. P2C (default), round-robin, least-request, ring-hash / Maglev consistent hashing.
  • The failure to watch. Operational complexity and per-hop cost — a proxy beside every pod adds latency and resource overhead, and the control plane becomes critical infrastructure.
Reach for Envoy when…you're balancing east-west microservice traffic (especially gRPC), want dynamic config without restarts, and need built-in retries / circuit-breaking / mTLS — typically via a service mesh.

Build it yourself

Reading how a load balancer works is not the same as being able to build one. These are the implementation questions an interviewer (or a real design doc) will push on. Work each one out before you open the approach. The point is to discover why the well-known designs are shaped the way they are.

Implement round-robin, then weighted round-robin

Spread requests evenly across N backends, then handle the case where backends have different capacities (one is twice as powerful). What's the data you keep, and how do you make weighted distribution smooth rather than bursty (not "3 in a row to the big one, then 1 to the small one")?

Reveal the approach

Plain round-robin is a single index incremented mod N — O(1), no per-backend state. For weighted, the naive approach (repeat each backend weight-times in a list) is bursty. The standard smooth algorithm is smooth weighted round-robin (used by NGINX): each backend has a current_weight that you increase by its configured weight every step, pick the backend with the highest current weight, then subtract the total weight from the chosen one. This interleaves picks proportionally (for weights 5/1/1 you get a balanced spread, not five-in-a-row). Key insight: balance isn't just the right ratio, it's the right ordering.

Implement power-of-two-choices

You have N backends and want near-optimal load spread without tracking global state. Implement P2C — and explain why sampling two and picking the lighter is so much better than picking one at random, but sampling three is barely better than two.

Reveal the approach

Pick two backends uniformly at random, compare a cheap local load metric (active connections or in-flight requests), send to the lighter one. O(1), no global coordination, no herd effect. The why: with one random choice the busiest backend's load grows like log n / log log n; with two choices it drops to log log n — an exponential improvement — because the second sample lets you avoid the unlucky-busy backend. Going from two to three samples only shaves a constant factor, so the extra sampling cost isn't worth it. This is exactly the result behind Envoy's default policy.

Implement health checking with hysteresis

Mark a backend down when it's failing, back up when it recovers — without flapping in and out on a single blip. How do you avoid both (a) ejecting a backend over one transient error and (b) the ejection cascade that takes down the whole pool?

Reveal the approach

Use hysteresis: require N consecutive failures before marking a backend unhealthy and M consecutive successes before marking it healthy again — so one blip doesn't flip state. Combine active probes (periodic /healthz) with passive outlier detection (eject on a burst of real-traffic 5xx). Critically, cap the damage: a max_ejection_percent so you never remove more than (say) half the pool, and a panic threshold — if more than that fraction looks unhealthy, assume the health signal itself is wrong and route to everyone rather than hammering the survivors. That cap is the difference between shedding one bad backend and causing a total outage.

Design-it-yourself: the open-ended ones

No single right answer — these are the system-design prompts. Sketch a design, then follow the link to the lesson that goes deep.

  • Make the balancer highly available. Design an active-active LB pair that survives one node dying with no dropped traffic. What moves the traffic, and how fast? → Resilience Patterns.
  • Global load balancing. Steer users across three regions and fail a whole region out. Why might you pick Anycast over GeoDNS, and what does Anycast break? → Multi-Region & DR.
  • Sticky sessions vs stateless. A WebSocket app needs a client to keep hitting the same backend. Implement stickiness — then argue why externalizing session state (to a cache) and staying stateless is usually better. → Service Mesh.
