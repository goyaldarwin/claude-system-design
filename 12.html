Consistent Hashing

How do you spread keys across a changing set of nodes so that adding or removing one moves as few keys as possible? Plain hash % N remaps almost everything on a resize. Consistent hashing moves only ~1/N — at the cost of uneven load that you then have to smooth.

The trade-off Minimal key movement on membership change in exchange for load imbalance you must engineer away. Mapping keys to a ring means adding/removing a node only disturbs its neighbors (~1/N of keys), not the whole keyspace. In return, random placement gives lumpy load — so you add virtual nodes and bounded loads to even it out.

What consistent hashing is

Consistent hashing maps both keys and nodes onto the same circular hash space (say 0 to 232−1). To find a key's owner, hash the key to a point and walk clockwise to the first node you hit. The payoff is in what happens on a membership change:

  • Add a node — it lands somewhere on the ring and takes over only the arc between it and its predecessor. Just those keys move; everything else is untouched.
  • Remove a node — its keys fall to the next node clockwise. Only that node's keys move.

So a resize moves ~k/N keys (k = total keys, N = nodes), versus hash % N which remaps nearly all keys because the modulus changes. That's the whole reason it exists.

The one-line frameThe ring solves key movement but not load balance — randomly placed nodes own wildly different arc sizes. Everything below (vnodes, bounded loads) is the work of turning "few keys move" into "and the load is actually even."

Three scenarios — decide before you read

Scenario 1 — adding the ninth cache node You have 8 cache nodes mapped with hash % 8. You add a 9th. What fraction of keys now resolve to a different node — and how does consistent hashing change that number?
Scenario 2 — the lumpy ring You place each of 8 nodes at one random point on the ring. Will the 8 arcs be roughly equal? What does that mean for load, and what happens to a departed node's entire arc?
Scenario 3 — vnodes aren't enough You've added virtual nodes, so arcs are even. But one key is extremely hot, and its node is now over capacity even though the ring is balanced. How do you cap a node's load without re-hashing everything?

Scenario 1 is why consistent hashing exists. Scenario 2 is random skew (→ vnodes). Scenario 3 is bounded loads.

It comes down to three refinements

Each refinement fixes a failure of the previous one — that's the cleanest way to understand consistent hashing.

Refinement 1 — The bare ring

Hash nodes to points on the ring; a key belongs to the first node clockwise. This already gives the headline property: adding/removing a node moves only ~1/N of keys (Scenario 1: going 8→9 nodes moves ~1/9 of keys, vs hash%N remapping ~89%).

The failure of the bare ring — random skew This is Scenario 2. With N nodes at N random points, the arcs between them are not equal — the gaps follow an exponential-like distribution. The largest arc averages O(log N / N) of the ring while the mean is 1/N, so the heaviest node owns several times its fair share: empirically ±30–40% imbalance with one point per node. Worse, removing a node dumps its entire arc onto a single successor — a hot-spot, not a smear. The fix is the next refinement.

Refinement 2 — Virtual nodes (vnodes)

Place each physical node at many points (tokens) on the ring. With V tokens per node, each node's load is the sum of V independent arcs, and by concentration the relative imbalance shrinks as ~1/√V — so V ≈ 100–200 brings imbalance down to roughly ±5%. Two bonuses: heterogeneity for free (a 2×-capacity machine gets 2× the tokens), and smooth rebalancing (a departing node's V arcs spread across V successors, not one).

The failure of too many vnodes — availability math "More vnodes is always better" is wrong. With more tokens per node, each node shares replica responsibility with more peers — so past a point, a multi-node failure is likelier to wipe out all replicas of some token range. This is why Cassandra historically defaulted to num_tokens=256, then lowered the recommendation to 16 (with allocation-aware token selection): high V hurt streaming, repair, and the availability math. Token allocation strategy matters as much as the count.

Refinement 3 — Bounded loads (and ring-free alternatives)

Vnodes even out arcs, but assume keys are uniform. Real workloads have hot keys, so a node can still exceed capacity.

The failure vnodes don't fix — the hot key over capacity This is Scenario 3. Consistent Hashing with Bounded Loads (Mirrokni, Thorup, Zadimoghaddam — Google, 2017) adds a hard cap: with total load L over n nodes, set capacity c = ⌈(1+ε)·L/n⌉. A key hashes to its ideal node as usual; if that node is full, walk forward to the next with room. The theorem guarantees no node exceeds (1+ε)× the average, while only O(1/ε²) keys relocate per insert/delete — so a small ε (e.g. 0.25) buys a tight cap at modest movement. Vimeo's groupcache balancer and HAProxy's bounded-load mode use exactly this.

Two ring-free alternatives worth knowing:

  • Rendezvous (HRW) hashing — for each key, compute hash(key, node) for every node and pick the highest. No ring to store; naturally weighted; O(N) per lookup (fine for small N).
  • Jump consistent hash — a tiny, fast function mapping a key to one of N buckets with minimal movement when N grows — but only supports adding/removing the last bucket, so it fits sharding where you never remove arbitrary nodes.

The decision table — which variant

SituationUseWhy
Distributed cache / DHT, churnRing + vnodesMinimal movement, even load, weighting

Self-check — can you reconstruct the three scenarios?

  • Scenario 1: Why does hash%N remap ~89% of keys on 8→9, and why does the ring move only ~1/N?
  • Scenario 2: Why is a one-point-per-node ring skewed by ±30–40%, and how do vnodes (and the 1/√V law) fix it?
  • Scenario 3: Why don't vnodes fix a hot key, and how do bounded loads cap a node with little movement?
  • Why is "more vnodes is always better" wrong (the availability trade-off)?
  • When would you reach for rendezvous or jump hash instead of a ring?

The shape to remember: consistent hashing is 3 refinements — the bare ring (fixes resize movement), vnodes (fix random skew), bounded loads (fix hot keys) — each solving the previous one's failure.

Famous implementations

Dynamo / Cassandra / Riak (ring + vnodes)

  • What it is. The canonical ring with virtual nodes; the basis of the Dynamo lineage.
  • Key idea. Vnodes for even load + weighting; the key's ring position also defines its replica set (the next R nodes clockwise — the "preference list").
  • The failure to watch. Token count vs availability (Cassandra 256→16); a hot key still needs bounded loads or caching.

Maglev (Google L4 LB)

  • What it is. Google's consistent-hashing scheme for load balancers, building a lookup table.
  • Key idea. Near-perfect even distribution + minimal disruption on membership change, with O(1) table lookups — built for thousands of LB nodes agreeing without coordination.
  • The failure to watch. Table rebuild cost on large membership changes; tuned for the LB use case.

Jump consistent hash

  • What it is. A ~5-line function mapping a key to one of N buckets (Google, 2014).
  • Key idea. No memory, very fast, minimal movement as N grows.
  • The failure to watch. Can only add/remove the last bucket — no arbitrary node removal, so it fits fixed/growing shard sets, not churny caches.

Build it yourself

Implement the ring with virtual nodes

Build a structure that maps a key to a node, supports add/remove node, and keeps load even. What do you store, how do you look up, and how do vnodes plug in?

Reveal the approach

Keep a sorted structure (e.g. a tree / sorted array) of ring positions → node. For each physical node, insert V tokens: hash(node + "#" + i) for i in 0..V. Lookup: hash the key, binary-search for the first token ≥ that position (wrapping to the first if past the end), return its node — O(log(N·V)). Add a node: insert its V tokens (only keys in those arcs move). Remove: delete its tokens (its keys fall to successors). Tune V ~100–200 for ±5% balance; weight a bigger machine by giving it more tokens. That's the whole production data structure.

Design-it-yourself

  • Add bounded loads. Extend your ring so no node exceeds (1+ε)× average; what do you check on each placement? → Refinement 3.
  • Pick a scheme for a growing shard set. You only ever add shards, never remove. Ring, jump hash, or rendezvous? → decision table.
