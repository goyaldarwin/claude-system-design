Load Balancing

How load balancers distribute traffic across resources, where they sit in the stack, and the smart-client, hardware, and software approaches to implementing them.

The bottleneck Finite capacity. one server can't take all traffic — distributing it adds a routing hop and health-tracking.

In this lesson

  • 1. Smart Clients
  • 2. Hardware Load Balancers
  • 3. Software Load Balancers

Load balancer (LB) is another critical piece of any distributed system. It helps to distribute load across multiple resources according to some metric (random, round-robin, random with weighting for memory or CPU utilization, etc.). LB also keeps track of the status of all the resources while distributing requests. If a server is not available to take new requests or is not responding or has elevated error rate, LB will stop sending traffic to such a server.

To utilize full scalability and redundancy, we can try to balance the load at each layer of the system. We can add LBs at three places:

  • Between the user and the web server
  • Between web servers and an internal platform layer, like application servers or cache servers
  • Between internal platform layer and database.

There are many ways to implement load balancing.

1. Smart Clients

A smart client will take a pool of service hosts and balances load across them. It also detects hosts that are not responding to avoid sending requests their way. Smart clients also have to detect recovered hosts, deal with adding new hosts, etc.

Adding load-balancing functionality into the database (cache, service, etc.) client is usually an attractive solution for the developer. It looks easy to implement and manage especially when the system is not large. But as the system grows, LBs need to be evolved into standalone servers.

2. Hardware Load Balancers

The most expensive–but very high performance–solution to load balancing is to buy a dedicated hardware load balancer (something like a Citrix NetScaler). While they can solve a remarkable range of problems, hardware solutions are very expensive, and they are not trivial to configure.

As such, even large companies with large budgets will often avoid using dedicated hardware for all their load-balancing needs. Instead, they use them only as the first point of contact for user requests to their infrastructure and use other mechanisms (smart clients or the hybrid approach discussed in the next section) for load-balancing for traffic within their network.

3. Software Load Balancers

If we want to avoid the pain of creating a smart client, and since purchasing dedicated hardware is excessive, we can adopt a hybrid approach, called software load-balancers.

HAProxy is one of the popular open source software LB. Load balancer can be placed between client and server or between two server side layers. If we can control the machine where the client is running, HAProxy could be running on the same machine. Each service we want to load balance can have a locally bound port (e.g., localhost:9000) on that machine, and the client will use this port to connect to the server. This port is, actually, managed by HAProxy; every client request on this port will be received by the proxy and then passed to the backend service in an efficient way (distributing load). If we can’t manage client’s machine, HAProxy can run on an intermediate server. Similarly, we can have proxies running between different server side components. HAProxy manages health checks and will remove or add machines to those pools. It also balances requests across all the machines in those pools.

For most systems, we should start with a software load balancer and move to smart clients or hardware load balancing as need arises.

Principal-level deep dive

Load balancing at principal depth

The original lesson frames load balancing as "spread requests across servers, usually round-robin." At principal level the interesting decisions live elsewhere: where in the stack you balance, which algorithm preserves your invariants (cache locality, session state), how you detect and eject sick backends fast enough to matter, and how you stop the balancer itself from being the thing that takes you down. The round-robin default is almost never the right production choice.

L4 vs L7: balance on connections or on requests

An L4 (transport) load balancer makes its decision on the 5-tuple (src IP, src port, dst IP, dst port, protocol) and then forwards packets — it never parses HTTP. It is cheap, fast (millions of connections, line-rate with hardware/eBPF), and protocol-agnostic, so it handles raw TCP, TLS passthrough, gRPC, MQTT, anything. The cost: it pins an entire TCP connection to one backend, so with HTTP/2 or gRPC multiplexing, all streams on a long-lived connection land on the same server — a single chatty client can hot-spot one backend.

An L7 (application) load balancer terminates the connection, parses the request, and routes per request. This unlocks path/header routing, retries on idempotent methods, request-level metrics, header rewriting, TLS termination, and per-request rebalancing across an HTTP/2 connection pool. The cost is CPU (TLS + parse) and added latency. The principal heuristic:

Rule of thumbL4 for north-south raw throughput, TLS passthrough, non-HTTP protocols, and when you need the backend to see the original encryption. L7 when you need content-based routing, per-request fairness over multiplexed connections (gRPC/HTTP2), or smart retries. Many real stacks layer both: an L4 edge (anycast + DSR) fronting an L7 fleet (Envoy/NGINX/ALB).

Algorithms beyond round-robin

Round-robin assumes requests are uniform and backends identical — both false in practice. Better choices:

AlgorithmHow it picksWhen it wins / fails
Round-robin / weighted RRNext in rotation (weighted by capacity)Fine for uniform, short requests; blind to actual load — a stuck backend keeps getting traffic
Least-connectionsBackend with fewest active connectionsGood when request durations vary widely; can mis-route if connections aren't a proxy for work (keep-alive idle conns)
Least-response-time / EWMALowest in-flight × recent latencyBest signal for heterogeneous backends; needs accurate, low-noise latency telemetry
Power-of-two-choices (P2C)Sample 2 backends at random, pick the less loadedNear-optimal load spread with O(1) state and no global coordination; the default in modern meshes (Envoy, Finagle)
Consistent hashing (Maglev, ring/jump hash)Hash a key (client IP, session, cache key) to a backendPreserves affinity; minimal reshuffle on membership change — but can hot-spot on skewed keys

Why P2C is the modern default: classic least-connections requires the balancer to track global state across every backend, and with many balancer instances they all pile onto the same "least loaded" server (the herd problem). P2C — sample two, pick the better — gives provably exponential improvement in max load over pure random with almost no state, and degrades gracefully under stale information. It is why service meshes ship it as the standard policy.

Consistent hashing and why affinity matters

If your backends maintain a local cache, a per-tenant in-memory aggregate, or a stateful session, you want the same key to keep hitting the same backend. Naive hash(key) % N reshuffles almost everything when N changes (one backend added → ~100% cache miss storm). A consistent-hash ring moves only 1/N of keys on a membership change. Maglev hashing (Google's L4 LB) goes further: it builds a lookup table that gives near-perfect even distribution and minimal disruption, which matters when thousands of LB nodes must independently agree on the same mapping without coordination. The trade-off is always affinity vs even load — a single hot key (one whale tenant) can overload its assigned shard, so production systems add bounded-load consistent hashing (cap each backend, overflow to the next).

"Sticky sessions" are the degenerate, app-layer version of this: a cookie pins a user to one backend. It works but couples availability to a single node and breaks autoscaling/draining. Prefer externalizing session state (Redis) and using stateless backends, reserving stickiness for genuinely sticky workloads (in-progress uploads, WebSocket sessions).

Health checks and outlier ejection

A balancer is only as good as its picture of backend health. Two complementary mechanisms:

  • Active health checks — the LB proactively probes (e.g. GET /healthz every few seconds, with healthy/unhealthy thresholds and hysteresis). Catches dead backends but can be fooled by a shallow check that returns 200 while the app's dependency is down. Make the check deep enough to be meaningful but cheap enough to run often; expose separate liveness vs readiness signals.
  • Passive health checks / outlier detection — the LB watches real traffic and ejects a backend that returns a burst of 5xx, connection failures, or high latency (Envoy's outlier detection ejects on consecutive-5xx or consecutive-gateway-failure, with exponential ejection time and a max_ejection_percent cap). This catches grey failures an active probe misses, and reacts in real time.
Failure mode to design againstNever let health checking eject so aggressively that you cascade: if a dependency blip makes every backend return 5xx, ejecting them all routes 100% of traffic to the last node standing, which then dies. The max_ejection_percent cap and a "panic threshold" (when too many hosts are unhealthy, ignore health and spray to everyone) exist precisely to prevent this self-inflicted outage.

Connection draining on deploy

Rolling a deploy or scaling in must not 500 in-flight requests. Connection draining (a.k.a. graceful shutdown / lame-duck mode): mark the backend "draining," stop sending it new connections, but let existing requests finish up to a timeout, then close. The sequence that actually works in practice: (1) flip readiness to fail so the LB stops new traffic, (2) keep serving in-flight, (3) wait for the LB's de-registration to propagate before the process exits — the classic Kubernetes bug is the pod receiving SIGTERM and dying before the endpoint is removed from the LB, causing a few seconds of connection-refused errors. A preStop sleep covering the propagation delay fixes it.

Global server load balancing (GSLB)

Above a single region you steer users to the right location:

  • GeoDNS — the authoritative DNS server returns different A/AAAA records based on resolver geography/latency. Simple and works everywhere, but TTL caching and resolver behavior mean failover is slow (clients cling to stale records) and granularity is the resolver, not the user.
  • Anycast — the same IP is advertised from many sites via BGP; the network routes each client to the topologically nearest PoP. Failover is near-instant (withdraw the route) and there's no DNS-cache lag, which is why CDNs and DNS providers themselves run anycast. The catch: BGP can re-converge mid-connection and break long-lived TCP, so anycast pairs naturally with stateless/short requests at the edge (then hands off to unicast for stateful flows).

The LB as SPOF and bottleneck

A single load balancer is both a single point of failure and a throughput ceiling. Mitigations, roughly in order of scale:

  • Active-active pairs with a floating VIP (VRRP/keepalived) — a standby takes the virtual IP on failure. Cheap, but the surviving node must absorb full load.
  • ECMP + multiple LB nodes — routers spread flows across a fleet of equal-cost LB instances; combined with Direct Server Return (DSR), responses bypass the LB entirely (the LB sees only the inbound packet), so the LB stops being the bandwidth bottleneck. This is how hyperscale L4 tiers (Maglev, Katran/XDP) handle terabits.
  • DNS-level distribution across multiple independent LB clusters for the coarse first hop.

Client-side LB and the service-mesh sidecar

The modern internal alternative removes the centralized middlebox entirely. In client-side load balancing the caller gets the full backend list (from service discovery) and runs the algorithm itself — no extra network hop, no shared bottleneck, and per-call decisions with full knowledge. gRPC and Finagle ship this natively. The downside is fat clients: every language needs the logic, and policy changes require redeploying callers.

A service mesh (Istio/Linkerd) resolves that by moving the logic into a per-pod sidecar proxy (Envoy): the application makes a plain localhost call, and the sidecar does discovery, P2C balancing, mTLS, retries, circuit breaking, and outlier ejection — driven by a central control plane, so policy changes without app redeploys. You pay a per-hop latency and resource tax (a proxy beside every pod) for uniform, language-agnostic, observable L7 traffic management. This is the dominant pattern for east-west traffic in Kubernetes today.

Visual references

What the diagram hides

From a box-and-arrow to a system that actually runs

It takes one second to draw a load-balancer box with arrows to three server boxes. It takes a team to make that drawing real, because every arrow is a protocol and every box hides software the diagram never shows. Walk the questions the drawing silently skips — this is the leap from sketching systems to operating them.

1. How does the LB even know the servers' IP addresses?

The naive assumption is a hardcoded list of backend IPs in the LB's config. Real systems never do this, because servers are created and destroyed constantly. Instead they use service discovery:

  • A service registry — etcd, Consul, ZooKeeper, or your cloud's internal one — holds the live set of healthy backend IPs.
  • When a server boots it registers itself ("I am 10.0.3.7:8080, healthy"). The LB watches the registry and updates its backend pool automatically — within seconds, no human edits a file.
Why the registry is hard The registry is the source of truth for "who is alive," so it must stay correct and available even when the network is misbehaving — which is exactly the "no shared truth" tax. That is why registries run on consensus (Raft/ZAB): a majority must agree on membership. See also Leader Election & Coordination.

2. A server's hardware dies. What happens?

You are right that you cannot simply re-host the app on dead hardware — the box is gone. The sequence that makes this a non-event is automated:

  • The dead server stops answering health checks (the LB periodically opens a TCP connection or sends GET /health; no / 404 response → mark it down and stop routing to it — "outlier ejection").
  • It also stops sending heartbeats to the registry, whose lease/TTL expires the entry — so the registry, and therefore the LB, forget it.
  • An orchestrator (e.g. Kubernetes) notices the node is gone and reschedules the application onto a healthy node. The replacement comes up with a new IP, registers itself, and the LB picks it up — automatically.
The key shift You never "tell the LB the new IP" by hand. Servers tell a registry; the LB watches the registry; an orchestrator replaces dead nodes. The manual chore you imagined is exactly what all the extra software exists to eliminate.

3. What if the load balancer itself fails?

Drawing one LB box quietly reintroduces the single point of failure the LB was meant to remove. Real designs make the LB itself redundant, and the technique depends on the layer:

LayerHow the LB is made highly available
Within a datacenterRun two or more LB instances sharing a floating / virtual IP (VIP). A heartbeat protocol — VRRP (often via keepalived) — lets a standby instantly take over the VIP if the active one dies.
At the edge / globallyDNS with health checks (e.g. Route 53) hands out healthy LB addresses; anycast announces one IP from many locations via BGP, so the network itself routes around a dead site.
Managed cloud LBAWS ALB/NLB, Google GLB are themselves horizontally-scaled redundant fleets — "one box" is an illusion the cloud maintains for you.
The principle (from lesson 01) Redundancy never removes the single point of failure — it moves it. You push it from the LB to the VIP-failover mechanism, then to DNS, then to anycast/BGP. There is no bottom; you stop when the remaining failure probability is acceptable. Every "highly available" component is this trick applied recursively.

4. Which protocols are actually on the wire?

  • Client ↔ LB ↔ server: TCP for an L4 LB (forwards packets by IP/port); HTTP/HTTPS for an L7 LB (understands requests, routes by URL/header, terminates TLS). This is the L4-vs-L7 split above.
  • Health checks: periodic TCP connect or HTTP GET /health.
  • LB failover: VRRP / keepalived for the VIP.
  • Service registry: servers register over HTTP/gRPC; the registry replicates internally via Raft/ZAB; membership and failure detection often use a gossip protocol (SWIM).
  • The plumbing you never think about: DNS (name → IP), ARP (IP → MAC on the local network), BGP/OSPF (routing between networks), and NTP (clock sync — quietly critical, because so much breaks when clocks drift — see Time, Clocks & Ordering).

5. Who writes the load-balancing software, and what else runs on every node?

The LB is a real program someone built and operates: HAProxy, NGINX, or Envoy (the modern data plane behind most service meshes); hardware appliances like F5 BIG-IP; or a managed cloud LB. In a service mesh there is no central LB box at all — an Envoy sidecar runs next to every service and does client-side load balancing.

And the box labelled "server" is not just running your application. In production it also runs a quiet stack of supporting software — this is what the diagram hides most of all:

Software on a typical nodeWhy it's there
Service-registry agent (e.g. Consul agent)registers the node, runs local health checks
Gossip / membership daemon (SWIM)so nodes know who else is alive without a central poller
Service-mesh sidecar (Envoy/Linkerd)mTLS, retries, timeouts, client-side LB for all its traffic
Observability agentsmetrics exporter, log shipper, tracing agent (see Observability)
Orchestrator agent (kubelet)places/restarts containers, reports node health, drives the "node died → reschedule" loop
NTP, config/secrets agents, CNI networkingclock sync, configuration, and giving each container its IP in the first place
The takeaway Every box in a system diagram is a computer running far more than your app, and every arrow is a protocol with a failure mode. The drawing is a summary, not the system. The skill that separates building from sketching is being able to expand any box and any arrow into the operational reality underneath — service discovery, health checking, failover, and the agents that make "replace the dead node" automatic.

Mastery check

  • Can you explain the difference between an L4 and an L7 load balancer in terms of what decision is made on what data, and give one concrete case where each is the right choice (e.g. TLS passthrough / non-HTTP vs. content-based routing or per-request fairness over an HTTP/2 connection)?
  • Why does pinning a long-lived HTTP/2 or gRPC connection at L4 risk hot-spotting a single backend, and how does L7 (or per-request rebalancing) fix it?
  • Why should you avoid sticky sessions as a default — what does cookie-pinning couple availability to, and how does externalizing session state (e.g. Redis) restore stateless autoscaling and draining?
  • How do active health checks and passive outlier ejection complement each other, and why can aggressive ejection cause a cascading failure? What two safeguards (max_ejection_percent, panic threshold) prevent the self-inflicted outage?
  • Why does hash(key) % N trigger a cache-miss storm when N changes, and how does consistent hashing (or Maglev) limit reshuffle to ~1/N of keys while preserving cache affinity? What breaks this, and how does bounded-load consistent hashing patch it?
  • Why is power-of-two-choices the modern mesh default over global least-connections — what herd problem does it avoid with O(1) state?
  • In what ways is a single LB both a SPOF and a throughput ceiling, and how do a floating VIP (VRRP), ECMP + Direct Server Return, and client-side / sidecar balancing each remove it?
