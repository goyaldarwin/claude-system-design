Designing a Notification System

Send the right message to the right user on the right channel — push, SMS, email — at the scale of millions per second, reliably, without spamming anyone twice or melting a third-party gateway. The design is a fan-out pipeline of queues and workers; the hard parts are reliability (deliver despite a flaky external provider), not duplicating, and respecting user limits.

The crux Reliable, scalable, multi-channel delivery decoupled from the services that trigger it — in exchange for handling provider failures, deduplication, rate limits, and user preferences in the pipeline. A queue-based fan-out lets any service fire a notification and forget it, absorbing bursts. You pay by owning retries against unreliable external gateways, idempotent dedup so users aren't double-notified, and per-user throttling/preferences so you don't spam or get rate-limited.

1. Requirements

Functional: any service can trigger a notification to a user; support multiple channels (push (APNs+FCM), SMS, email, in-app); use templates with personalization; respect user preferences (opt-outs, quiet hours, per-channel settings); track delivery status.

Non-functional: high throughput (millions/sec at peak — a marketing blast or a breaking-news alert), reliable (don't lose a notification a service asked to send), no duplicates (don't notify a user twice for one event), low latency for urgent notifications, decoupled (triggering services don't wait on delivery).

2. Capacity estimate

Say 100M users, average 10 notifications/user/day, with bursty peaks. Work it:

  • Average rate: 100M × 10 = 1B/day ≈ ~11,500/sec average. But notifications are bursty — a "breaking news" or marketing send can target tens of millions at once.
  • Peak fan-out: a single send to 50M users in 5 minutes = ~167,000/sec — 14× the average. Consequence: the system must absorb huge bursts — a queue between trigger and delivery is mandatory, sized for peak not average.
  • Provider limits: external gateways (APNs, Twilio, SES) have their own rate limits. If you push 167k/sec at a provider rate-limited to 10k/sec, you'll be throttled/blocked. Consequence: you must rate-limit your outbound calls to each provider — the queue is also a buffer that smooths to the provider's pace.

3. The API

send({ userId, templateId, data, channels, priority, dedupeKey })
// async: returns immediately with a notificationId; delivery happens in the pipeline
getStatus(notificationId) -> { channel: state }   // queued | sent | delivered | failed

send is fire-and-forget (enqueue and return) so triggering services never block on delivery. dedupeKey is the caller-supplied idempotency key (see sub-problem 2); priority routes urgent vs bulk (sub-problem 4).

4. High-level design

A pipeline: (1) services call send → a notification service validates, applies templates + user preferences, and enqueues per-channel messages onto queues (one per channel/priority). (2) Per-channel worker pools consume their queue and call the external provider (APNs/FCM for push, Twilio for SMS, SES for email) — with retries, rate limiting, and dedup. (3) Delivery status (and provider webhooks for delivered/bounced) update a status store. The queue between trigger and workers is what decouples, absorbs bursts, and lets you throttle to each provider's pace.

5. Data model

StoreHolds
TemplatesPer-channel message templates with placeholders (versioned)
User preferences / devicesOpt-ins per channel, quiet hours, device tokens (APNs/FCM), verified email/phone
Notification log / statusEach notification's id, channel, state, provider response, timestamps
Dedup storeSeen dedupeKeys (TTL'd) for idempotency

Preferences and device tokens are read on every send (cache them); the status log is write-heavy and append-style (a time-series/wide-column store fits).

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — the SMS provider is down for 2 minutes Your SMS gateway (Twilio) returns errors for 2 minutes during a deploy on their side. During that window you must send 100,000 SMS. What happens to those notifications — lost? — and how does the pipeline guarantee they're eventually delivered without overwhelming the provider when it recovers?
Scenario 2 — the user got the same alert 3 times A triggering service retried its send call after a timeout (not sure the first worked), and a worker reprocessed a redelivered queue message — so the user received the identical "your order shipped" push three times. Where do the duplicates come from, and how do you ensure one event = one notification?
Scenario 3 — the marketing blast delayed the 2FA code A 40-million-user marketing campaign floods the queue. Meanwhile, a user's time-sensitive 2FA login code is stuck behind the marketing messages and arrives 4 minutes late — useless. What's this problem called, and how do you make urgent notifications jump the bulk ones?

Scenario 1 is reliability vs the provider. Scenario 2 is deduplication. Scenario 3 is fan-out & priority. Each maps to a sub-problem.

The four hard sub-problems

The pipeline is the easy part. These four are where notification systems live or die — each with a nested failure.

Sub-problem 1 — Reliability against an unreliable provider

The external gateways (APNs, Twilio, SES) will have outages, throttle you, and return transient errors. Your job is to deliver eventually anyway. The queue makes notifications durable (a provider outage doesn't lose them — they wait in the queue); workers retry with exponential backoff + jitter, and after repeated failures route to a dead-letter queue for inspection.

The failure that happens when you get this wrong — lost on provider error This is Scenario 1. If you call the provider synchronously from the trigger (no queue) and it's down, the notification is lost — and a retry storm of 100k synchronous calls would hammer the provider when it recovers (and your own threads while it's down, a cascade). The queue-based design fixes both: messages sit durably in the queue during the 2-minute outage; workers retry with backoff + jitter (so recovery isn't a synchronized stampede) and rate-limit to the provider's capacity; a circuit breaker stops hammering a clearly-down provider. When the provider recovers, the backlog drains at its allowed rate — nothing lost, nothing overwhelmed. Permanent failures (invalid token, bounced email) go to a DLQ, not infinite retry.

Sub-problem 2 — Deduplication (one event, one notification)

Duplicates arise from two places: the caller retries send (network timeout — did it go through?), and the queue redelivers (at-least-once delivery + a worker crash before ack). Both are inherent to a reliable distributed pipeline. You make delivery idempotent with a dedup key: the caller supplies a stable key per logical notification; the system records seen keys (with a TTL) and drops a repeat.

The failure that happens when you get this wrong — the double (triple) notification This is Scenario 2. Because both the caller and the queue can re-deliver (you want at-least-once so nothing is lost), without dedup the user gets the same alert multiple times — annoying for marketing, alarming for "payment received." The fix is an idempotency/dedup key derived from the event (e.g. orderId + "shipped"): before sending, atomically check-and-set the key in a dedup store (Redis with TTL); if it was already there, drop this one. This converts at-least-once delivery into one-notification-per-event (exactly-once effect, since exactly-once delivery is impossible). Choose the dedup key carefully — too broad and you suppress legitimately-distinct notifications.

Sub-problem 3 — Rate limiting & user preferences

Two kinds of limits. User-facing: respect opt-outs, quiet hours, and frequency caps (don't send a user 50 notifications in an hour) — checked against the preferences store before sending. Provider-facing: stay under each gateway's rate limit so you aren't throttled or banned — a token-bucket limiter per provider paces outbound calls.

The failure that happens when you get this wrong — spamming users & getting throttled This is Scenario 3's sibling. Skip user preferences/frequency caps and you spam people (they disable notifications entirely, or your domain gets marked spam — a lasting deliverability hit). Skip provider rate limiting and a burst (Scenario 3's 40M blast) exceeds the gateway's limit → the provider throttles or blocks you, delaying everything including urgent messages. Both fixes live in the pipeline: enforce per-user frequency caps + quiet hours + opt-outs at enqueue time, and token-bucket rate-limit each provider's worker pool so you send at exactly the provider's allowed pace. The queue buffers the difference between burst input and metered output.

Sub-problem 4 — Fan-out & priority

A single trigger may fan out to millions of users (a broadcast), and notifications have wildly different urgency (a 2FA code vs a "we miss you" email). You need to fan out efficiently and ensure urgent messages aren't stuck behind bulk ones.

The failure that happens when you get this wrong — head-of-line blocking (urgent stuck behind bulk) This is Scenario 3. With a single queue, a 40M-message marketing blast sits ahead of an urgent 2FA code — classic head-of-line blocking — and the time-sensitive notification arrives uselessly late. The fix: separate queues by priority (urgent / transactional / bulk), with workers servicing high-priority queues first (or with dedicated capacity), so a 2FA code skips the marketing backlog. For fan-out, expand a broadcast into per-user messages asynchronously (a fan-out worker reads "send to segment X" and enqueues individual sends) so the trigger returns instantly and the expansion is itself parallelized. Never let bulk volume share a lane with latency-critical sends.

At a glance — the design decisions

ConcernMechanism
Decouple trigger from deliveryFire-and-forget send → durable queue
Reliability vs flaky providerQueue durability + retry/backoff/jitter + circuit breaker + DLQ
No duplicatesCaller dedup key + check-and-set dedup store (TTL)
Don't spam / get throttledUser prefs + frequency caps; per-provider token-bucket limiting
Urgent beats bulkPriority queues; async fan-out for broadcasts
The rule in one lineMake send fire-and-forget into a durable queue; deliver via per-channel, per-priority workers that retry against flaky providers, dedup by key, respect user + provider rate limits, and keep urgent notifications in their own lane. The queue is the heart: it decouples, absorbs bursts, and meters output to each provider.

The takeawayManaged senders (SNS, SES, FCM, Twilio) handle the channel + provider relationship, and the cloud queues/functions give you the durable, bursty pipeline. But the orchestration — dedup keys, per-user preferences/frequency caps, priority lanes, and per-provider rate limiting — is your application logic; the platform delivers the message, it doesn't decide whether or not to duplicate it.

Famous implementations

Same facets — what it is, the model, and the failure to watch.

APNs / FCM (push providers)

  • What it is. Apple Push Notification service / Firebase Cloud Messaging — the OS-level gateways that actually deliver push to devices.
  • Model. You send to the provider with a device token; they deliver to the device and report success/failure (invalid token → prune it).
  • The failure to watch. Stale/invalid device tokens (clean them up from feedback), per-app rate limits, and provider outages — the reason your pipeline must retry + DLQ.

Twilio / SES (SMS & email)

  • What it is. Managed SMS (Twilio) and email (SES/SendGrid) providers.
  • Model. REST API to send; webhooks report delivered/bounced/complained; sender reputation matters for email.
  • The failure to watch. Rate limits + spam/reputation (mind frequency caps and opt-outs); transient errors need backoff retries.

In-house platforms (queue + workers)

  • What it is. The orchestration layer big products build atop providers (e.g. Uber/Netflix-style notification platforms).
  • Model. Kafka/queues + per-channel workers + dedup + preference service + priority lanes — exactly the design above.
  • The failure to watch. Dedup-key design, preference-store latency on the hot path (cache it), and priority isolation.

Build it yourself

Work each before opening the approach.

Guarantee one notification per event

A service may retry send, and your queue is at-least-once. Design dedup so the user gets exactly one "order shipped" notification per order.

Reveal the approach

Require a dedup key on send that's stable for the logical event — e.g. "shipped:" + orderId (not a random per-call id, or retries wouldn't match). Just before dispatching to the provider, do an atomic check-and-set in a dedup store: SET dedup:{key} NX EX {ttl} in Redis — if it succeeds (key was new), send; if it fails (key existed), drop as a duplicate. The TTL bounds memory and matches how long retries could plausibly arrive (e.g. hours). This collapses both caller retries and queue redeliveries into one delivery (Scenario 2). Edge case: do the check-and-set as close to the actual send as possible, and only mark "sent" after the provider accepts, so a crash before sending doesn't suppress a real delivery.

Keep 2FA codes ahead of marketing

A 40M marketing blast and time-critical 2FA codes share the system. Design so 2FA is never delayed by bulk.

Reveal the approach

Use separate queues by priority: urgent (2FA, security), transactional (order updates), bulk (marketing). Worker pools service higher-priority queues first — either strict priority (always drain urgent before bulk) or dedicated capacity per lane (a reserved pool for urgent that bulk can't starve). Now a 2FA code lands in the urgent queue and is sent within ms regardless of a 40M backlog in the bulk queue (fixing head-of-line blocking, Scenario 3). Expand the marketing broadcast asynchronously via a fan-out worker (reads "segment X," enqueues per-user bulk messages) so it never blocks the trigger or the urgent lane. Also apply per-provider rate limiting per lane so the bulk drain can't consume the provider quota the urgent lane needs.

Design-it-yourself: the open-ended ones

  • Add scheduled / quiet-hours delivery. Support "send at 9am in the user's timezone" and suppress during quiet hours. Design the scheduling. → Job Scheduler.
  • Reliable trigger → notification. A service must enqueue a notification atomically with its DB write (no dual-write drift). Design it. → Outbox, CDC.
  • Aggregate/digest notifications. Instead of 50 separate "X liked your post," send one digest. Design the batching window. → Newsfeed.
