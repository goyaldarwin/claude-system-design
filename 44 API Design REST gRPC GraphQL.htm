API Design: REST, gRPC, GraphQL

The contract is the product — how to model resources, evolve schemas without breaking callers, pick the right protocol, and make retries safe.

Why API design is a principal-level concern

An API is the most expensive thing your team will ship, because it is the hardest to take back. Internal code can be refactored freely; a published interface is a promise to every caller you do not control. Mobile clients sit in app stores for years. Partner integrations are written once and never touched. A poorly modeled or non-evolvable API becomes a permanent cost: every future change requires coordination across teams, versioned shims, and migration windows that drag for quarters.

The principal-level skill is not knowing REST verbs — it is designing a contract that can evolve under load without breaking callers, that makes safe retries trivial, and that does not leak your internal data model. Protocol choice (REST vs gRPC vs GraphQL) is downstream of those concerns. This lesson treats them as a toolkit, not a religion.

Key insightThe three styles are not competitors so much as fits for different boundaries: REST for public/partner edges and resource CRUD, gRPC for high-throughput service-to-service inside the mesh (see Microservices and Service Mesh), and GraphQL for client-facing aggregation across many backends. Large systems run all three at once.

REST and the Richardson maturity model

REST is an architectural style, not a protocol. Its leverage comes from a few constraints: resources identified by URIs, a uniform interface (the HTTP methods), statelessness, and cacheability. The Richardson Maturity Model grades how much of that you actually use:

  • Level 0 — one endpoint, RPC-over-HTTP (SOAP-style POST to /api). HTTP is just a tunnel.
  • Level 1 — multiple resources (/orders, /customers) but verbs encoded in the body or URL.
  • Level 2 — proper use of HTTP methods and status codes (GET, POST, PUT, DELETE; 200/201/404/409). This is where almost all good production APIs live.
  • Level 3 — HATEOAS: responses embed links to next actions. Powerful in theory, rarely worth the cost in practice because clients hardcode flows anyway.

Resource modeling

Model nouns, not verbs. POST /orders/123/cancel is pragmatic and common, but the purer form is to make cancellation a state transition on a sub-resource: POST /orders/123/cancellation creating a cancellation record. The test for a good resource model: can you express the operation as create/read/update/delete on a thing? When you cannot — "search", "transcode", "rebalance" — do not contort REST; model it as an action resource or a job (POST /transcoding-jobs returning 202 Accepted with a status URL). That async-job pattern is the standard way to expose long-running work over a request/response protocol.

PitfallLeaking your database schema as your API is the most common and most damaging mistake. Your tables normalize for storage; your API should denormalize for clients. Coupling them means every schema migration is now a breaking API change, and vice versa.

Idempotency, methods, and idempotency keys

Idempotency means applying an operation N times has the same effect as applying it once. It is the property that makes retries — the cornerstone of distributed resilience — safe. HTTP method semantics encode this:

MethodSafe (no side effects)IdempotentTypical use
GETYesYesRead a resource
PUTNoYesReplace at a known URI
DELETENoYesRemove a resource
POSTNoNoCreate under a collection
PATCHNoNot inherentlyPartial update

PUT /accounts/42 with the same body twice yields one account in one state — idempotent. POST /accounts twice creates two accounts. That asymmetry is exactly the problem when a client times out and retries a payment: did the first POST succeed before the connection dropped?

Idempotency keys

The fix, popularized by Stripe, is a client-supplied idempotency key — a UUID the caller generates per logical operation and sends in a header. The server records (key → result) and replays the stored response on any retry with the same key.

POST /v1/charges HTTP/1.1
Idempotency-Key: 7f3c1e9a-...-d21
Content-Type: application/json

{ "amount": 4999, "currency": "usd", "source": "tok_visa" }

// Server-side, atomically
key = request.header["Idempotency-Key"]
row = INSERT INTO idem_keys(key, status) VALUES(key,'PENDING')
        ON CONFLICT(key) DO NOTHING RETURNING *
if row is null:                       // key seen before
    existing = SELECT * FROM idem_keys WHERE key = ?
    if existing.status == 'DONE':  return existing.response
    else:                          return 409 Conflict  // still in flight (or block-and-replay; either way never lose/duplicate the result)
result = do_charge(...)                // the real, non-idempotent work
UPDATE idem_keys SET status='DONE', response=result WHERE key=?
return result
Key insightIdempotency keys must be scoped (per-endpoint, per-account), have a TTL (Stripe keeps 24h), and the lookup-or-insert must be atomic to handle two concurrent retries racing. This is the linchpin of any payment system and pairs directly with the retry guidance in Resilience Patterns.

Pagination, filtering, and versioning

Offset vs cursor pagination

Offset pagination (?limit=20&offset=40) is simple and supports jumping to arbitrary pages, but has two fatal flaws at scale. First, OFFSET 100000 LIMIT 20 forces the database to scan and discard 100k rows — O(offset) cost. Second, if rows are inserted or deleted between page fetches, the window shifts and the client sees duplicates or skips items.

Cursor (keyset) pagination encodes the position as an opaque token built from the last row's sort key: WHERE (created_at, id) > (:last_ts, :last_id) ORDER BY created_at, id LIMIT 20. This is O(limit) regardless of depth (it rides an index) and is stable under concurrent writes. The trade-off: no random page access and the sort columns must be unique and immutable enough to anchor on.

GET /events?limit=20
→ { "data": [...], "next_cursor": "eyJ0cyI6MTcwMCwiaWQiOjk5fQ" }
GET /events?limit=20&cursor=eyJ0cyI6MTcwMCwiaWQiOjk5fQ
Trade-offOffset wins on UX affordances (page numbers, "jump to last") and developer familiarity. Cursor wins on performance and correctness for large or live datasets — feeds, logs, infinite scroll. Default to cursor for anything that grows unboundedly; offset is acceptable for small, admin-style tables.

Versioning strategies

  • URI versioning (/v1/orders) — explicit, cache-friendly, trivially routable at the proxy/gateway. The pragmatic default for public APIs (Stripe, GitHub, Twitter all use it).
  • Header versioning (Accept: application/vnd.acme.v2+json) — cleaner URIs, content-negotiation purity, but harder to test in a browser and easy to forget.
  • Query-param versioning (?version=2) — easy but pollutes caching and feels bolted on.

The deeper move is to version as little as possible. A v1→v2 cutover means you now run two stacks and must migrate every caller. Prefer additive, non-breaking evolution (next section) so a single version lasts for years. Stripe famously pins each account to the API version active at signup and transforms responses internally — callers never have to migrate, which is a powerful (if expensive) strategy.

gRPC and Protocol Buffers

gRPC is a binary RPC framework over HTTP/2 with Protocol Buffers as the IDL and wire format. You define the service contract once; codegen produces strongly-typed clients and servers in every language. Where it wins:

  • Compactness and speed. Protobuf encodes integers as varints and uses field tags instead of field names, so payloads are a fraction of equivalent JSON and parsing skips reflection. For chatty service-to-service traffic this materially cuts CPU and bandwidth.
  • HTTP/2 multiplexing. Many concurrent calls share one connection without head-of-line blocking at the application layer, avoiding the connection-per-request overhead of HTTP/1.1.
  • Streaming. First-class server-streaming, client-streaming, and bidirectional streaming — ideal for telemetry, large result sets, and live updates (contrast with long-polling/WebSockets/SSE on the browser edge).
  • Contract-first discipline. The .proto is the single source of truth, generating consistent clients and enabling compile-time breakage detection.
syntax = "proto3";
package orders.v1;

service OrderService {
  rpc GetOrder(GetOrderRequest) returns (Order);
  rpc WatchOrders(WatchRequest) returns (stream OrderEvent);   // server stream
}

message Order {
  string id        = 1;
  int64  amount    = 2;
  Status status    = 3;
  // field 4 reserved for a future field; never reuse a retired tag
}

Schema evolution in protobuf

The wire format keys fields by their numeric tag, not name, which gives clean evolution rules: you may add new fields (old code ignores unknown tags), rename fields freely (the name is cosmetic), and remove fields — but you must never reuse a tag number and must mark removed numbers/names reserved. Changing a field's type or its tag is a breaking change. proto3 makes all scalars optional-with-defaults, which removes "required" footguns but means you cannot distinguish "absent" from "zero" without wrapper types or optional.

PitfallgRPC is awkward at the browser edge — browsers cannot speak raw gRPC, so you need gRPC-Web plus a proxy (Envoy) to translate. It is also harder to debug than text JSON (no curl + eyeball). Keep gRPC for east-west internal traffic; expose REST or GraphQL north-south to external clients.

GraphQL: over/under-fetching, N+1, cost limiting

GraphQL exists to solve a client problem: with REST, a mobile screen often needs data from several endpoints, causing under-fetching (many round trips) or, when the server pads responses, over-fetching (downloading fields the screen ignores). GraphQL lets the client send one query describing exactly the graph of fields it wants:

query {
  user(id: "42") {
    name
    orders(last: 3) { id total items { sku qty } }
  }
}

One request, exactly the requested shape, resolved server-side across whatever backends own each field. This is genuinely powerful for client-driven aggregation and is why it often sits in front of a microservice fleet as a federation layer.

The N+1 problem

The flexibility has a sharp edge. The query above resolves user, then for each order resolves items — naively, one DB call per order, the classic N+1 query explosion. The standard fix is DataLoader-style batching: within a single request tick, resolvers register the keys they need, and a batch function coalesces them into one query (SELECT * FROM items WHERE order_id IN (...)) with per-request caching. Without DataLoader, GraphQL will quietly hammer your databases.

Cost and complexity limiting

Because the client controls the query shape, a single request can ask for a deeply nested, expensive graph — an accidental or malicious denial of service. You cannot rate-limit by counting requests the way you would for REST (see Designing an API Rate Limiter); one request is unbounded work. Defenses:

  • Query depth limiting — reject queries nested beyond N levels.
  • Complexity/cost analysis — assign each field a cost, multiply by requested list sizes, and budget per caller. Rate-limit on accumulated cost, not request count.
  • Persisted queries — clients register approved queries by hash; the server only executes known-good shapes, killing arbitrary-query risk and shrinking payloads.
PitfallWhen NOT to use GraphQL: simple CRUD with one backend (REST is less machinery), high-throughput internal RPC (gRPC is faster and type-safe), and when HTTP caching matters — GraphQL typically POSTs to a single /graphql endpoint, so CDN and proxy caching by URL evaporates. You also inherit a complex server runtime, a harder authorization story (per-field, not per-endpoint), and opaque performance.

Backward and forward compatibility

The discipline that lets an API live for years is treating every change as either backward-compatible (old clients keep working against the new server) or forward-compatible (new clients work against an old server, vital during rolling deploys when both versions run at once — see Deployment Strategies).

Safe (additive)Breaking (avoid / version)
Add a new optional fieldRemove or rename a field
Add a new endpoint / RPCChange a field's type or units
Add a new enum value (if clients tolerate unknowns)Make an optional field required
Relax a validation constraintTighten validation / shrink ranges
Add a new optional request parameterChange default behavior / error codes

Two rules carry most of the weight. First, tolerant reader: clients must ignore unknown fields rather than reject them — this is what makes additive evolution safe and is built into protobuf's wire format and good JSON parsers. Second, never repurpose a field: a field's meaning and units are frozen for life; if semantics change, add a new field and deprecate the old. Adding a new enum value is the classic trap — it is only safe if every existing client has a sane default branch for unknown values, so design enums with a UNKNOWN = 0 sentinel from day one.

API gateways and the BFF pattern

An API gateway is the single entry point that sits between external clients and your internal services, consolidating cross-cutting concerns so each service does not re-implement them:

  • Routing — path/host-based dispatch to backend services; protocol translation (REST↔gRPC, gRPC-Web).
  • AuthN/AuthZ — validate JWT/OAuth tokens at the edge so services trust a verified identity (see Security in System Design).
  • Rate limiting and quotas — per-API-key throttling enforced centrally.
  • Aggregation — fan out one client call to several services and compose the result (the lightweight cousin of GraphQL).
  • Observability and resilience — centralized logging, metrics, request IDs, and circuit breaking.
Trade-offBe deliberate about what lives in the gateway. It is a shared component, so business logic there becomes a coupling chokepoint and a deployment bottleneck — the gateway should do generic edge concerns, not domain logic. The mesh and gateway overlap: a service mesh handles east-west service-to-service policy (mTLS, retries) while the gateway handles north-south client traffic.

The Backend-for-Frontend (BFF) pattern goes further: rather than one gateway for all clients, give each client type (web, iOS, Android, partner) its own thin aggregation layer tuned to that client's needs. This avoids a single API trying to please every consumer and lets the mobile team optimize payloads without negotiating with the web team. GraphQL is one common way to implement a BFF.

What strong engineers doThey refuse the false binary. A mature design uses REST for the public API and partner integrations, gRPC for the chatty internal call graph behind the gateway, and a GraphQL/BFF layer for rich clients — with one shared idempotency, error, and pagination convention across all of them so the org doesn't relearn the rules per protocol.

Principal-level mastery check

What strong engineers doThey lead with the evolution and idempotency story before naming a protocol. They reach for idempotency keys unprompted when discussing payments or any non-idempotent write over an unreliable network, default to cursor pagination for unbounded data, and treat the API contract as decoupled from the storage schema. They know GraphQL's N+1 and cost-limiting traps cold and can say crisply when not to use it. They distinguish north-south (gateway) from east-west (mesh) traffic and place each protocol accordingly.
  • A client POSTs a charge, the request times out, and it retries. Walk me through making this safe end-to-end — what does the server store, and how do you handle two retries racing concurrently?
  • Your feed API uses offset/limit. At page 5,000 latency spikes and users report seeing duplicate items. Diagnose both symptoms and design the fix, including the cursor token's contents.
  • A team wants to add a field to a protobuf message consumed by 40 services on independent deploy cadences. What are the rules, and what is the one operation that will silently corrupt old readers?
  • You expose GraphQL publicly. How do you stop a single query from taking down your databases, and how does rate limiting differ from the REST case?
  • When would you choose gRPC over REST for an internal service, and what do you lose at the browser edge and in debuggability by doing so?
  • Marketing wants to repurpose an existing discount field to mean something new in v2. Talk me through why that's dangerous and what you'd do instead, given mobile clients in the wild for 2+ years.
