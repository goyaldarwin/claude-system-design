Designing a Distributed Cache

Build a Redis/Memcached-class cache from first principles — eviction, sharding, replication, hot keys, and the failure modes that take down the database behind it.

The bottleneck Latency. building the cache itself — eviction, sharding, stampede control, all to keep reads near memory speed.

Requirements & scope

A distributed cache is an in-memory key-value store fronting a slower, authoritative datastore. The contract is simple but the engineering is dense: sub-millisecond reads at high throughput, predictable behavior under memory pressure, and graceful degradation when nodes die. This builds directly on Caching — here we design the cache itself, not just place one in an architecture.

Functional

  • get(key), set(key, value, ttl), delete(key), atomic incr/cas.
  • Per-key TTL with lazy + active expiration.
  • Eviction when memory is full (configurable policy).
  • Horizontal scale across many nodes; client routes keys to the right node.

Non-functional

  • Latency: p99 read < 1 ms intra-AZ; the whole point is to beat the DB.
  • Throughput: 100k–1M ops/sec per node (single-threaded Redis does ~100k–200k/core; Memcached scales multi-threaded higher).
  • Availability: a cache node loss should degrade hit rate, never lose correctness of the system of record.
  • Consistency: caches are intentionally weak-consistency. The hard question is always invalidation, not storage.
Key insightFrame the cache as a lossy, best-effort tier. The authoritative copy lives in the database. Every design decision (eviction, replication, persistence) trades hit rate and recovery speed — not durability of truth. If you find yourself treating the cache as a source of truth, you have built a slow, weird database.

Scale estimation

Anchor numbers so capacity, not vibes, drives the design. Suppose a read-heavy product: 1M DAU, 50 reads/user/day cache-eligible, 10:1 read:write.

  • Read QPS: 1M × 50 / 86,400 ≈ 580 avg; assume 5× peak → ~3k QPS sustained, bursts higher. (Real hyperscale caches run 10M+ QPS; scale the math.)
  • Working set: say 100M hot objects × 1 KB avg = 100 GB. Add ~50–75% overhead for keys, pointers, and allocator fragmentation → budget ~160–180 GB.
  • Node sizing: 64 GB usable RAM/node → ~3 shards for data; round up for replicas and headroom → ~6–9 nodes.
  • Network: 3k QPS × 1 KB = 3 MB/s — trivial. Hot keys or large values flip this fast; a 1 MB value at 5k QPS is 5 GB/s and saturates a single NIC.

The two numbers that actually decide the design: working-set size (drives shard count) and value-size distribution (drives hot-key and network risk).

Cache API & data model

Keep the wire protocol tiny and binary-friendly. Memcached's text/binary protocol and Redis's RESP are the references. A pragmatic surface:

GET key                 -> value | MISS
SET key value ttl flags -> OK
DEL key                 -> count
CAS key value cas_token -> OK | EXISTS   // compare-and-swap for optimistic concurrency
INCR key delta          -> new_value     // atomic counter
MGET k1 k2 ...          -> values        // pipelined multi-get, amortizes RTT

Data model is a flat keyspace of opaque bytes. The cache should not understand the value (no server-side joins). Two metadata fields per entry matter:

  • expireAt (absolute epoch ms) — for TTL.
  • cas/version — monotonic token for optimistic writes and for detecting stale invalidations.

Single-node engine: hashmap + eviction

Each node is a high-throughput in-memory map. Core structures:

  • A hash table (open-addressing or chained) for O(1) lookup. Redis uses incremental rehashing — it keeps two tables and migrates buckets a few at a time so a resize never causes a multi-millisecond stall.
  • An eviction index over entries (see below) to pick a victim when full.
  • An expiry structure — Redis keeps a separate dict of keys-with-TTL; expiration is lazy (checked on access) plus active (a background sampler deletes a random batch of expired keys ~10×/sec to bound stale memory).

Threading model

Two camps, and you should be able to defend either:

ModelExampleTrade-off
Single-threaded event loopRedisNo locks, simple reasoning, atomic commands for free; bounded by one core for the data path (offloads I/O/fsync to helper threads).
Multi-threaded, sharded locksMemcachedScales with cores on a big box; needs fine-grained locking and lock-free LRU tricks to avoid contention on the eviction list.
Trade-offSingle-threaded buys correctness simplicity at the cost of vertical scaling — which is fine because you scale horizontally across shards anyway. The common mistake is adding threads to a node before adding shards to the cluster.

Eviction policies: LRU, LFU, W-TinyLFU

LRU via hashmap + doubly-linked list

The canonical O(1) LRU: a hash map from key → node, plus a doubly-linked list ordered most-recently-used at the head. On access, splice the node to the head; on insert past capacity, drop the tail.

get(key):
  node = map[key]
  if not node: return MISS
  list.moveToHead(node)        // O(1) unlink + relink
  return node.value

set(key, val):
  if key in map: node=map[key]; node.value=val; list.moveToHead(node)
  else:
    if size == capacity:
      victim = list.tail
      list.remove(victim); map.remove(victim.key)   // evict LRU
    node = Node(key,val); list.addHead(node); map[key]=node

Two well-known weaknesses: (1) scan resistance — one large sequential scan evicts your whole hot set; (2) updating the list on every read is a write that hurts concurrency. Real systems approximate. Redis samples N random keys (default 5) and evicts the oldest — approximate LRU that avoids maintaining a global list.

LFU and its decay problem

LFU evicts the least-frequently used. Naive frequency counts never forget, so a key that was hot last week clings forever. Redis's LFU uses a logarithmic counter plus time-based decay so cold keys age out. LFU beats LRU on skewed, stable workloads but reacts slowly to shifts.

W-TinyLFU

W-TinyLFU (used by Caffeine) is the current state of the art and worth naming. It pairs a small window LRU (catches bursty recency) with a large main segment admitted via a frequency filter. Frequency is tracked cheaply with a Count-Min Sketch (see Probabilistic Data Structures) plus a periodic halving (aging). A new item is only admitted to the main cache if its estimated frequency beats the would-be victim's — giving near-optimal hit ratios with O(1) memory for counters and strong scan resistance.

What strong engineers doDon't just say "use LRU." State the workload (skewed? scan-heavy? bursty?), pick the matching policy, and acknowledge that production systems use approximate eviction (sampling, sketches) because maintaining exact global order is a concurrency bottleneck.

Sharding via consistent hashing

One node cannot hold 160 GB or serve 1M QPS, so partition the keyspace. Naive hash(key) % N remaps almost every key when N changes — a cold-cache catastrophe during scale-up. Use Consistent Hashing with virtual nodes: each physical node owns many points on the ring, so adding/removing a node only moves ~1/N of keys and keeps load balanced. This connects to Sharding / Data Partitioning.

Where does routing live?

ApproachExampleNotes
Client-side hashingMemcached clients, Redis Cluster smart clientsNo proxy hop — lowest latency; all clients must agree on the ring/topology.
Proxy / router tierTwemproxy, EnvoyCentralizes topology, simplifies clients; adds a hop and a thing to scale.
Server-side clusterRedis Cluster (16384 hash slots)Nodes own slot ranges, redirect clients via MOVED/ASK; topology gossiped between nodes.
PitfallRedis Cluster uses 16384 fixed hash slots, not a continuous ring — slots are explicitly assigned to nodes, which makes rebalancing a discrete, controllable migration rather than an emergent ring property. Conflating "consistent hashing" with "Redis Cluster slots" in practice signals you haven't read the implementation.

Replication & availability

A lost shard means a cold spot: every key it owned now misses and hammers the DB. Mitigate with replication — primary + N async replicas per shard (see Replication Deep Dive and Redundancy & Replication).

  • Async replication is standard for caches: the primary acks the client before replicas confirm. You accept a small window of replica staleness/loss because the DB is the source of truth anyway.
  • Failover: a sentinel/cluster process detects primary death and promotes a replica. Redis Sentinel and Cluster do leader election here — the same primitives covered in Leader Election & Coordination.
  • Read scaling: serve reads from replicas to spread hot shards — but accept stale reads. For a cache, usually fine.
Trade-offSynchronous replication would make failover lossless but doubles write latency and re-introduces the availability fragility you came to the cache to avoid. Per PACELC (Consistency Models & PACELC), caches almost always pick latency over consistency in the no-partition case. A cache that blocks on replica acks defeats its purpose.

Hot keys

Consistent hashing balances keyspace, not traffic. A single viral key (celebrity profile, flash-sale SKU) lands on one shard and melts it — the classic hot-key problem. Detection: per-key request sampling, or top-K sketches (Count-Min + heavy hitters). Mitigations, in escalating order:

  1. Local/L1 cache on the app node (near cache). A few-second TTL on the client absorbs almost all reads of a hot key with zero network. This is the single highest-leverage fix.
  2. Key splitting / replication: store hotkey#1..hotkey#N across shards and have clients pick a random copy — spreads read load N-way at the cost of N-way write fan-out.
  3. Dedicated replicas for the hot shard, serving reads round-robin.
  4. Request coalescing (see stampede below) so concurrent misses don't multiply backend load.
PitfallA 1 MB hot value isn't a QPS problem, it's a bandwidth problem — replication won't save a saturated NIC. The fix is a local cache or shrinking the value (store an ID, fetch the blob from a CDN/object store). Always ask for the value-size distribution.

Invalidation & write patterns

"There are only two hard things in computer science: cache invalidation and naming things." The write path determines correctness. Know all four patterns cold:

PatternMechanismUse when
Cache-aside (lazy)App reads cache; on miss, reads DB, populates cache. Writes go to DB then DEL the key.Default. Resilient to cache loss; only caches what's read.
Write-throughApp writes to cache, cache synchronously writes to DB.Read-heavy, need fresh cache, can tolerate write latency.
Write-back (write-behind)Write to cache, async-flush to DB later (batched).Write-heavy, can tolerate data loss on cache crash. Risky.
Write-aroundWrite straight to DB, skip cache; populate on next read.Write-once-read-rarely data (avoids cache pollution).
PitfallOn a write, delete the cache key rather than updating it. Update-on-write races with concurrent reads: a slow reader can write a stale value back after your update, leaving the cache permanently wrong until TTL. Delete-on-write forces a fresh read-through. Even deletes race — for strict correctness use versioned writes (CAS) or change-data-capture invalidation via Stream Processing & CDC.

TTL is your safety net: even if an invalidation is lost, the key self-heals at expiry. Set the shortest TTL the freshness SLA tolerates. TTL also bounds the blast radius of any invalidation bug.

Cache stampede / thundering herd

A popular key expires. In the same millisecond, 10,000 requests miss simultaneously, all stampede the database, and the DB falls over — the cache that was protecting it becomes the trigger for its collapse. Three layered defenses:

1. Request coalescing (single-flight)

Only the first miss for a key recomputes; concurrent callers wait on the same in-flight result. Per-node, a map of in-flight key → future:

get(key):
  v = cache.get(key); if v: return v
  // only one goroutine computes per key per node
  return inflight.do(key, () -> {
    v2 = cache.get(key)            // double-check after acquiring slot
    if v2: return v2
    v2 = db.load(key)
    cache.set(key, v2, ttl)
    return v2
  })

Go's singleflight is the reference implementation. This collapses a herd to one backend call per node; pair with a distributed lock (see Distributed Locking Service) to collapse it to one call cluster-wide if the recompute is genuinely expensive.

2. Probabilistic early expiration (XFetch)

Instead of everyone seeing the key expire at the same instant, each reader probabilistically refreshes slightly before expiry, with probability rising as expiry nears. This staggers recomputation across time so no single moment has a herd:

// recompute early if: now - delta*beta*ln(rand()) >= expiry
//   delta = time the last recompute took, beta ~ 1.0
if (now() - computeTime * beta * Math.log(random())) >= expireAt:
    value = recompute(); cache.set(key, value, ttl)

3. Stale-while-revalidate

Serve the stale value immediately and refresh asynchronously in the background. Users never see a miss; the DB sees one refresh, not a herd. Combine with jittered TTLs so a batch of keys set together don't all expire in the same second.

What strong engineers doThey distinguish three failure modes precisely: stampede (one hot key expires), cache penetration (queries for keys that don't exist bypass the cache — defend with a Bloom filter or negative caching), and cache avalanche (many keys expire at once or the cache tier dies — defend with TTL jitter, replication, and DB rate-limiting / circuit breakers from Resilience Patterns).

Persistence & memory management

Do you even need persistence?

A pure cache can be volatile — on restart it warms up cold. But a cold restart of a 100 GB cache under live traffic is an avalanche. Persistence options (Redis's two modes are the canonical reference):

  • Snapshot (RDB): periodic point-in-time fork-and-dump. Compact, fast restart, but loses everything since the last snapshot. Fork uses copy-on-write so the live process keeps serving.
  • Append-only log (AOF): append every write to a log, replay on restart. Durable to the last fsync, but larger and slower to replay; needs periodic rewrite/compaction. This is the same LSM-style append-then-compact idea in Storage Engines: LSM vs B-Tree.
  • Hybrid: RDB base + AOF tail — fast restart with bounded loss. Best of both.

For warm restarts, also consider replica-driven recovery: bring a new node up as a replica, let it sync from the primary, then promote — no DB stampede.

Memory management

  • Allocator: jemalloc-style slab/size-class allocators (Memcached's slabs, Redis on jemalloc) reduce external fragmentation that plagues general malloc under churn. Watch the fragmentation ratio (RSS / used-memory) — >1.5 means you're wasting RAM.
  • maxmemory + policy: set a hard cap and an eviction policy (allkeys-lru, volatile-lfu, noeviction). Hitting the cap with noeviction turns writes into errors — usually the wrong default for a cache.
  • Big keys are landmines: a single multi-GB value or a collection with millions of elements stalls the single-threaded loop on access/delete and skews shard memory. Detect and shard or reject them.

Principal-level mastery check

What strong engineers doThey start by pinning down the workload — read:write ratio, value-size distribution, key skew, freshness SLA — before drawing boxes, because every later decision (eviction policy, sharding, hot-key strategy, write pattern) follows from those four numbers. They treat the cache as a lossy tier protecting a system of record, name real systems and algorithms (approximate LRU, W-TinyLFU + Count-Min, Redis Cluster's 16384 slots, singleflight, XFetch), and proactively raise the failure trio — stampede, penetration, avalanche — with a distinct defense for each. Weak engineers say "add a Redis" and stop.
  • Your cache node holding a hot shard dies at peak. Walk me through the next 30 seconds — what does the DB see, how does failover work, and how do you avoid a cold-cache avalanche on recovery?
  • Why DEL the key on write instead of updating it in place? Construct the exact race that makes update-on-write leave a permanently stale value.
  • You're caching at 5M QPS and one key is 40% of traffic. Consistent hashing didn't help — why, and what are your options in priority order?
  • Compare LRU, LFU, and W-TinyLFU on a workload with a periodic batch scan over the entire keyspace. Which survives and why? How does each track frequency without unbounded memory?
  • Differentiate cache stampede, cache penetration, and cache avalanche. Give a distinct mitigation for each, and explain why probabilistic early expiry beats a plain lock for stampede.
  • When would you turn on AOF persistence for a cache, and when is it actively harmful? What does it cost you on the write path and on restart?
  • Design the invalidation path when the source-of-truth DB is updated by a service you don't control. How do CDC and versioned writes give you correctness that TTL alone can't?
