Designing a Distributed Cache

A single cache node (like one Redis) is fast but caps out at one machine's memory and dies as a single point of failure. A distributed cache spreads cached data across many nodes to scale capacity and survive failures — which forces you to solve where each key lives, how to add/remove nodes without a cache-wide miss storm, and how to keep hot keys from melting one node. We design Memcached/Redis-Cluster from first principles.

The bottleneck Cache capacity and availability beyond one machine — in exchange for the distribution problems: key placement, rebalancing, hot keys, and a cache that can itself fail. Spreading entries across N nodes scales memory and survives a node loss. You pay by solving consistent placement (so adding a node doesn't invalidate everything), defending hot keys, and protecting the database when the cache misses or falls over.

1. Requirements

Functional: get(key), set(key, value, ttl), delete(key); data spread across many nodes; data is in-memory and treated as disposable (a cache, not the system of record — a miss just falls through to the origin database).

Non-functional: very low latency (sub-millisecond reads — the whole point), high throughput (hundreds of thousands+ ops/sec), horizontally scalable (add nodes for more capacity), highly available (lose a node without losing the whole cache or hammering the database).

2. Capacity estimate — how many nodes?

Suppose the working set we want cached is 500 GB of hot data, average entry 1 KB, peak 1,000,000 reads/sec. Work it:

  • Memory: 500 GB of data. With a per-node usable cache of, say, 64 GB → 500/64 ≈ 8 nodes just for capacity. Add headroom + replication: call it ~12 nodes.
  • Throughput: 1,000,000 reads/sec across 12 nodes ≈ ~83,000 ops/sec/node — comfortable for an in-memory store (Redis/Memcached do 100k–1M+ ops/sec/node), so capacity (memory), not throughput, is the binding constraint here. Consequence: we size by memory.
  • The key consequence: with 12 nodes, losing or adding one node touches ~1/12 of the keyspace. If placement is naive (hash(key) % N), changing N from 12→13 remaps almost every key — a near-total cache miss. That single number is why we need consistent hashing.

3. The API

get(key) -> value | null         // null = miss, caller falls through to the database
set(key, value, ttl_seconds)     // store with an expiry
delete(key)                      // explicit invalidation

The client library does the heavy lifting: it knows the node topology and routes each key to the right node. A get returning null is a normal miss — the caller reads the origin database and sets the result (a cache-aside read).

4. High-level design

The classic shape (Memcached-style): a fleet of independent in-memory nodes, and a smart client that hashes each key to pick the owning node and talks to it directly — no central router (which would be a bottleneck/SPOF). The application calls the cache client; the client maps key → node via consistent hashing and issues the get/set there. Redis Cluster moves this routing partly server-side (the cluster knows its slot map and redirects), but the placement principle is identical.

For availability, each shard can have one or more replicas (Redis Cluster: a replica per master, promoted on failure). Memcached classically has no replication — a lost node just means those keys miss and refill from the database (acceptable for a pure cache, dangerous if it stampedes — sub-problem 4).

5. Placement model — consistent hashing

The "data model" of a cache is really its placement scheme: how key → node. Consistent hashing maps both nodes and keys onto a ring; a key is owned by the next node clockwise. Adding/removing a node only moves the keys in one arc (≈1/N of keys), not all of them. Virtual nodes (each physical node placed at many ring points) smooth out the distribution so no node gets an unfairly large arc. (Redis Cluster uses a fixed 16,384 hash slots instead of a ring — same goal: move slots, not rehash everything.)

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — you add one cache node and the database falls over Your cache fleet is at capacity, so you add a node. Moments later database load spikes 50× and latency craters — even though you added capacity. What did adding the node do to the keyspace, and what placement scheme would have moved only ~1/N of keys instead?
Scenario 2 — Facebook, 2010 (real-pattern incident) A single extremely popular key (a celebrity's profile, a global config) gets so many reads that the one node owning it is saturated while the rest idle — and when it briefly expires, thousands of requests miss simultaneously and stampede the database. (Facebook's famous memcache work tackled exactly this class of problem.) How do you stop one hot key from melting its node, and one expiry from stampeding the database?
Scenario 3 — the attacker querying keys that don't exist Requests pour in for keys that were never in the cache and don't exist in the database either (random user IDs). Every one misses the cache and misses the database — the cache provides zero protection, and the database takes the full flood. What is this called, and what cheap structure rejects these before they hit the database?

Scenario 1 is placement / the rehash storm. Scenario 2 is hot keys + stampede. Scenario 3 is cache penetration. Each maps to a sub-problem.

The four hard sub-problems

Walking the design was the easy part. These four are where distributed caches are won or lost — each with a nested failure.

Sub-problem 1 — Where does a key live? (placement & rebalancing)

You must map each key to a node deterministically (so any client finds the same key on the same node) and be able to add/remove nodes without remapping everything. The answer is consistent hashing (ring + virtual nodes) or fixed hash slots (Redis Cluster's 16,384 slots) — both move only the affected fraction of keys when topology changes.

The failure that happens when you get this wrong — the rehash storm This is Scenario 1. Naive placement — node = hash(key) % N — means changing N (add/remove a node) changes the result for almost every key: suddenly nearly the whole keyspace maps to different nodes, so virtually every read misses, and all that traffic stampedes the origin database at once (a self-inflicted outage from adding capacity). Consistent hashing fixes it: only the keys in the new node's arc (~1/N) move, so a topology change causes a small, survivable miss bump, not a cache-wide wipe. This is the single most important reason distributed caches use consistent hashing, not modulo.

Sub-problem 2 — The hot key

Even with perfect placement, one key can be vastly more popular than others (a viral post, a global feature flag). Since that key lives on one node, all its traffic hits that single node — which saturates while the rest of the fleet idles. Placement spreads keys evenly; it does nothing for skew within a key.

The failure that happens when you get this wrong — one node melts This is half of Scenario 2. A hot key concentrates load on its owner node — CPU/network saturate there, that node's latency spikes, and a fraction of all your cache traffic degrades. Fixes: replicate the hot key to multiple nodes and read from a random replica (spreads the read load); add a tiny per-host local cache (an L1 in the app process) for the hottest keys so most reads never even hit the cache fleet; or split the key into N sub-keys across nodes. The detection part matters too — track per-key request rates (a Count-Min sketch finds heavy hitters) so you know which keys to replicate.

Sub-problem 3 — Eviction & expiry

Memory is finite, so when a node fills, it must evict entries to make room (typically LRU — evict the least-recently-used), and entries carry a TTL so stale data expires. The eviction policy decides what stays cached, which directly sets your hit rate.

The failure that happens when you get this wrong — scan pollution A naive LRU is vulnerable to scan pollution: a one-off bulk scan (a batch job, an analytics crawl) reads a huge set of keys once, and pure LRU dutifully caches all of them — evicting the genuinely hot keys that your real traffic needs, so the hit rate collapses right after the scan. Fixes: a scan-resistant policy like LRU-2 / W-TinyLFU (admit a key to the main cache only if it's been seen more than once recently — the basis of Caffeine), so a one-time scan can't evict frequently-used data. Also separate large/rarely-accessed values from small/hot ones. The eviction policy is a real performance lever, not a detail.

Sub-problem 4 — Protecting the origin database

A cache exists to shield the database — but several failure modes turn the cache into a way to amplify load onto the database: stampede, penetration, and avalanche. Defending against them is the difference between a cache that helps and one that causes outages.

The failure that happens when you get this wrong — stampede, penetration, avalanche Three related database-killers (rest of Scenario 2 + Scenario 3): (1) Cache stampede — a hot key expires and thousands of concurrent requests all miss and hit the database at once (Facebook 2010). Fix: request coalescing / single-flight — only the first miss recomputes; others wait for its result; plus probabilistic early refresh before expiry. (2) Cache penetration — requests for keys that don't exist anywhere miss cache and database, every time (Scenario 3). Fix: cache the "not found" result (negative caching) and put a Bloom filter of known keys in front — "definitely not present" rejects the request before the database. (3) Cache avalanche — many keys with the same TTL all expire at once, mass-missing together. Fix: add random jitter to TTLs so expirations spread out. All three share a root: the cache must absorb load, not pass a thundering herd through to a database that can't take it.

The rule in one linePlace keys with consistent hashing (survive node changes), route from a smart client, defend hot keys by replicating them, keep the hit rate with scan-resistant eviction, and above all protect the database with single-flight + negative caching + Bloom filters + TTL jitter — a cache that lets a herd through is worse than no cache.

Famous implementations

Same facets — what it is, placement/replication, and the failure to watch.

Memcached

  • What it is. A multi-threaded, pure in-memory cache; no persistence, no replication (classically).
  • Placement. Client-side consistent hashing across independent nodes; a lost node just means those keys refill from the DB.
  • The failure to watch. No replication → a node loss is a miss burst — you must have stampede protection or it cascades to the database.

Redis Cluster

  • What it is. Sharded Redis with built-in replication and failover.
  • Placement. 16,384 hash slots (CRC16 mod 16384) mapped to masters; the cluster knows the slot map and redirects clients; replicas promote on master failure.
  • The failure to watch. Single-threaded core per node (a slow command blocks it); hot slots still concentrate on one master — hot-key handling is still on you.

Caffeine (in-process L1)

  • What it is. A per-process in-memory cache, often used as an L1 in front of the distributed cache (L2).
  • Placement. Local to each app node; W-TinyLFU admission makes it scan-resistant.
  • The failure to watch. Coherence — each process's L1 can hold stale data; use short TTLs / invalidation, and it's the natural home for the hottest keys (Scenario 2 fix).

Build it yourself

Work each before opening the approach.

Implement single-flight to stop a stampede

A hot key expires and 5,000 concurrent requests miss at once. Design so only one of them recomputes from the database and the other 4,999 don't pile on.

Reveal the approach

Request coalescing (single-flight): on a miss, a request tries to acquire a short-lived per-key lock (e.g. SET key:lock NX EX 5 in Redis). The winner recomputes from the database and populates the cache; the losers wait briefly and re-read the cache (now populated) instead of all hitting the database. So 5,000 misses cause one database query, not 5,000 (Scenario 2 fix). Complement it with probabilistic early recomputation: refresh a hot key slightly before its TTL expires (with a probability that rises as expiry nears), so the value is renewed in the background and ideally never has a hard-miss moment at all. Together: no thundering herd on expiry. (See Caching for the full treatment.)

Stop cache penetration with a Bloom filter

Attackers query random non-existent keys; every request misses cache and database. Design a defense that rejects them before the database.

Reveal the approach

Two layers. Negative caching: when a lookup finds the key doesn't exist in the database, cache a "not found" marker (with a short TTL) — so a repeat query for that key hits the cache and returns "not found" without touching the database. But attackers use random keys, so negative caching alone bloats. So add a Bloom filter of all known-existing keys in front of the cache: on a request, check the Bloom filter first — if it says "definitely not present," reject immediately (no cache, no DB). Its false positives just mean an occasional real lookup proceeds (harmless); its no false negatives mean it never wrongly rejects a real key. Result: the flood of nonexistent-key requests dies at the Bloom filter, and the database is shielded (Scenario 3 fix).
