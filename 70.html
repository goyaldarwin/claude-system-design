Designing a Recommendation System

"What should we show this user next?" out of a catalog of millions, in tens of milliseconds, for hundreds of millions of users — this is a systems problem at least as much as an ML one. The defining structure is a multi-stage funnel (cheaply narrow millions to hundreds, then expensively rank), and the parts that actually break in production are rarely the model: they're serving latency, training-serving skew, the cold-start problem, and the feedback loop.

The crux Relevant recommendations at scale and low latency — in exchange for a multi-stage approximation funnel, the systems plumbing of feature stores and serving, and managing skew, cold-start, and feedback bias. You can't score millions of items per request, so you approximate in stages. The cost: a retrieval→ranking pipeline, a feature store that keeps training and serving consistent, handling users/items with no history, and a feedback loop that can reinforce its own biases.

1. Requirements

Functional: given a user (and context — time, device, current item), return a ranked list of relevant items from a large catalog; incorporate user history and behavior; refresh as behavior changes; handle new users and new items.

Non-functional: low latency (recommendations in tens of ms — they're on the critical render path), large scale (catalog of millions, hundreds of millions of users), fresh (reflect recent behavior), relevant (the actual quality goal), and avoid degenerate behavior (over-narrow filter bubbles).

2. Capacity estimate — why you can't score everything

Say 10M items in the catalog, a request budget of ~100ms, and a ranking model that takes ~1ms to score one (user, item) pair. Work it:

  • Naive cost: scoring all 10M items with the ranking model = 10M × 1ms = 10,000 seconds per request. Absurd — off by ~5 orders of magnitude from the 100ms budget. Consequence: you fundamentally cannot run the good (expensive) model over the whole catalog — this single fact forces the multi-stage funnel.
  • The funnel math: a cheap retrieval stage narrows 10M → ~1,000 candidates in a few ms (using approximate methods); then the expensive ranking model scores only those ~1,000 = ~1,000ms... still too slow, so ranking uses a lighter model or batching to fit, narrowing to ~100; a final re-rank polishes those. Consequence: each stage trades accuracy for the ability to handle the volume the previous stage passes it — cheap-and-broad first, expensive-and-precise last.
  • Read volume: hundreds of millions of users × many requests/day = huge QPS, all needing <100ms. Consequence: heavy reliance on precomputation + fast retrieval indexes + caching; you can't compute everything live.

3. The API

getRecommendations({ userId, context, count }) -> [ itemId, score ]
logInteraction({ userId, itemId, action, timestamp })   // click/view/purchase -> feedback loop

getRecommendations must return within the latency budget — so internally it runs the funnel (retrieve → rank → re-rank). logInteraction feeds the feedback loop: every impression and click becomes training signal and updates the user's recent-behavior features.

4. High-level design — the multi-stage funnel

The architecture every large recommender (YouTube, Netflix, Amazon) shares: a funnel that progressively narrows and refines.

  1. Candidate generation (retrieval): cheaply select ~hundreds-to-thousands of plausible items from the millions — using collaborative filtering, embeddings + ANN, or two-tower models. Optimized for recall (don't miss good items) and speed, not precision.
  2. Ranking: a richer ML model scores each candidate with many features (user, item, context) to predict the target (click/watch/purchase probability), producing a precise ordering of the few hundred.
  3. Re-ranking: apply business rules, diversity (don't show 10 near-identical items), freshness, and policy (don't recommend out-of-stock) to the top results.

Behind it: a feature store supplies features consistently to both training and serving, and the three planes (offline/nearline/online) compute things at the right freshness/latency.

5. Data & the three planes

Recommendation computation happens across three planes, by latency:

PlaneLatencyDoes
Offline (batch)HoursTrain models, precompute embeddings & candidate lists over all data (batch)
Nearline (streaming)Seconds–minutesUpdate features from recent behavior (stream), refresh candidates
Online (request-time)MillisecondsRun the funnel: retrieve precomputed candidates, rank, re-rank

Plus a feature store: the system of record for features, serving the same feature values to offline training and online serving — the linchpin of correctness (sub-problem 2).

Scenario 3 — the brand-new user (and brand-new item) A user just signed up — zero history. A new video was just uploaded — zero interactions. Collaborative filtering (which relies on past behavior) has nothing to work with for either. How do you recommend to a new user and recommend a new item to others?

Scenario 1 is the multi-stage funnel (candidate generation). Scenario 2 is training-serving skew (feature store). Scenario 3 is the cold-start problem. Each maps to a sub-problem.

The four hard sub-problems

The funnel is the easy part to draw. These four are where recommenders are hard in production — each with a nested failure.

Sub-problem 1 — Candidate generation (retrieval)

Cheaply narrow millions of items to a few hundred-thousand candidates, fast. The main techniques: collaborative filtering ("users like you liked X" — from the user-item interaction matrix); embeddings + approximate nearest neighbor (ANN) (represent users and items as vectors; retrieve items whose vectors are near the user's — fast at scale via ANN indexes); and two-tower models (a neural net with a user tower and an item tower producing comparable embeddings, trained so a user's vector is close to items they'll like). Item embeddings are precomputed offline; retrieval is a fast ANN lookup online.

The failure that happens when you get this wrong — scoring the whole catalog This is Scenario 1, and it's the foundational constraint. Running your good ranking model over all 10M items per request is ~5 orders of magnitude too slow (the capacity math). The multi-stage funnel is the answer: a cheap retrieval stage (ANN over precomputed embeddings, or collaborative filtering) narrows millions → hundreds in a few ms, optimized for recall (don't drop good items), and only then does the expensive ranking model run — on hundreds, not millions. It's both a latency necessity (you literally can't score everything in time) and an accuracy strategy (spend your expensive compute where it matters — precisely ordering a good shortlist). Trying to do it in one stage either blows the latency budget or forces a model too cheap to be good. Cheap-and-broad first, expensive-and-precise last.

Sub-problem 2 — Ranking & the feature store

The ranking model scores each candidate using many features (user features, item features, context, cross features) to predict the objective (P(click), P(watch-time), etc.). Those features must be computed identically when training the model (offline, over historical data) and when serving it (online, in real time) — which is exactly what a feature store guarantees: it's the single source of feature definitions and values for both paths.

The failure that happens when you get this wrong — training-serving skew This is Scenario 2, the most insidious production ML bug. Training-serving skew is when a feature is computed differently in the training pipeline than in the serving path — e.g. training uses a clean batch aggregate while serving uses a slightly different real-time computation, or training accidentally uses data that wouldn't be available at serving time (a point-in-time correctness violation — leaking the future into training). The model then sees inputs in production that don't match what it learned on, so it performs far worse than offline metrics promised. The fix is a feature store: define each feature once and serve the same values to training and serving (and enforce point-in-time correctness — train only on features as they were at the time of the event, never later data). Without it, your great offline model is a production disappointment, and you waste weeks debugging the model when the bug is the data.

Sub-problem 3 — The cold-start problem

Collaborative filtering and behavior-based models need history — so they have nothing for a new user (no past interactions) or a new item (no one's interacted with it yet). Both are constant in real systems (signups, new content) and both need a fallback that doesn't depend on interaction history.

The failure that happens when you get this wrong — no recs for new users/items This is Scenario 3. A purely behavior-based recommender gives a new user nothing useful (so a bad first experience — they may churn) and never surfaces a new item (so good new content dies in obscurity, and creators leave). Fixes for each side: new user — fall back to popularity/trending, use any signal you do have (signup context, demographics, the first few clicks to bootstrap fast), or ask for explicit preferences during onboarding. New item — use content-based features (the item's own attributes/metadata/embeddings from its content, not its interactions) so it can be matched to users by similarity to items they like, and deliberately explore (show it to some users to gather interaction data — an explore/exploit trade-off). The principle: don't rely solely on collaborative signals; combine them with content features and exploration so the system works on day zero for any user or item.

Sub-problem 4 — The feedback loop

Recommendations shape behavior, and that behavior becomes the training data for the next model — a closed loop. This is powerful (the system learns and improves) but dangerous: the model can reinforce its own past choices, creating bias and degenerate dynamics.

The failure that happens when you get this wrong — the filter bubble / bias spiral The feedback loop can spiral: the model recommends what it already thinks you'll like → you click those (you weren't shown alternatives) → the model learns "they only like this" → it narrows further — a filter bubble, and a popularity bias where already-popular items get recommended more, get more clicks, and dominate, while the long tail never surfaces (related to the new-item problem). There's also position bias (users click the top result because it's on top, not because it's best — which the model then over-rewards). Mitigations: deliberate exploration (sometimes show items the model is uncertain about, to gather unbiased signal — explore/exploit), diversity constraints in re-ranking, debiasing the training data for position/popularity, and measuring long-term/diverse engagement rather than just immediate clicks. The loop must be actively counter-steered, or the system optimizes itself into a narrow, biased local optimum.

At a glance — the design decisions

ConcernMechanism
Can't score millions/requestMulti-stage funnel: cheap retrieval → rank → re-rank
Retrieve candidates fastEmbeddings + ANN, collaborative filtering, two-tower models (precomputed offline)
Model great offline, bad liveFeature store (consistent train/serve) + point-in-time correctness
New users/itemsPopularity fallback, content-based features, exploration
Biased/narrowing recsExploration (explore/exploit), diversity, debiasing
The rule in one lineIt's a systems problem: approximate with a multi-stage funnel (you can't score everything), serve features through a feature store to kill training-serving skew, handle cold start with content features + popularity + exploration, and actively counter the feedback loop with exploration and diversity. The model is one box; the plumbing around it is where production lives or dies.

Famous implementations

Same facets — what it is, how it works, the failure to watch.

YouTube / Netflix funnel

  • What it is. The canonical large-scale recommender: candidate generation → ranking → re-ranking, serving billions of recommendations daily.
  • How it works. Cheap retrieval (two-tower / embeddings + ANN) narrows the catalog; a deep ranking model scores the shortlist; re-ranking adds diversity + business rules (Netflix even re-ranks on-device).
  • The failure to watch. Skipping a stage (latency blows up) or letting the feedback loop narrow recommendations into a bubble.

Two-tower retrieval models

  • What it is. A neural architecture with a user tower and an item tower producing comparable embeddings; widely used (YouTube, Google) for candidate generation.
  • How it works. Train so a user's vector is close to items they engage with; precompute item embeddings offline; at serve time, embed the user and do an ANN lookup for nearby items.
  • The failure to watch. Training-serving skew in the towers' features; ANN recall vs latency tuning; cold-start items need content features to get an embedding.

Feature stores (Feast / SageMaker / Vertex)

  • What it is. The component that serves consistent features to training and serving.
  • How it works. Define features once; an offline store backfills training data (point-in-time correct), an online store serves low-latency features at request time.
  • The failure to watch. If you bypass it (compute features ad-hoc in serving), you reintroduce skew — the exact bug it exists to prevent.

Build it yourself

Work each before opening the approach.

Design the funnel for a video platform

10M videos, 100ms budget, hundreds of millions of users. Design the stages and roughly what each does to the candidate count, and why one stage can't do it all.

Reveal the approach

Three stages. (1) Candidate generation: precompute video embeddings offline; at request time embed the user (or use precomputed user embedding) and do an ANN lookup to retrieve ~1,000 candidates from 10M in a few ms — optimized for recall, cheap per item. Mix in sources (collaborative filtering, trending, subscriptions) for coverage. (2) Ranking: a richer model scores those ~1,000 with full features (user history, video metadata, context) to predict watch-probability/watch-time, ordering them precisely — expensive per item but only ~1,000 items, so it fits the budget (with a light enough model / batched inference). (3) Re-ranking: take the top ~50, apply diversity (don't show 10 near-identical videos), freshness, and rules (no already-watched, no policy-violating), produce the final list. One stage can't do it: a single model good enough to rank well is far too slow over 10M, and a single model cheap enough to run over 10M is too weak to rank well — the funnel resolves the tension by spending cheap compute broadly and expensive compute narrowly (Scenario 1).

Handle cold start for new users and items

A new user with no history and a freshly uploaded item with no interactions both need to work from minute one. Design the handling for each.

Reveal the approach

New user: don't rely on collaborative filtering (no history). Fall back to popularity/trending (globally or by any known segment — region, device, signup source), optionally ask a few onboarding preference questions, and learn fast from their first interactions (nearline updates so the 5th recommendation already reflects the first 4 clicks). New item: it has no interactions, so use content-based features — derive an embedding from the item's own attributes/metadata/content (title, category, creator, visual/text features), so it can be placed near similar items in embedding space and retrieved for users who like those. Then deliberately explore: show it to a sample of likely-interested users to gather real interaction data, after which collaborative signals take over (explore/exploit). The principle for both: blend collaborative (needs history) with content-based + popularity + exploration (don't), so the system never has "nothing to recommend" (Scenario 3).

Design-it-yourself: the open-ended ones

  • Kill training-serving skew. Design a feature store + pipeline so the exact same feature values train and serve the model, with point-in-time correctness. → Stream/CDC, Batch.
  • Counter the feedback loop. Design exploration + diversity + debiasing so the recommender doesn't collapse into a filter bubble or popularity spiral. → Newsfeed.
  • Serve at scale within budget. Decide what to precompute (offline candidate lists) vs compute live, and where caching fits, to hit the latency budget at huge QPS. → Caching, Cost-Aware Design.
