Containers, Kubernetes and Serverless

The runtime substrate underneath every modern service: what isolation you actually get, how Kubernetes reconciles desired state, where state breaks the model, and when serverless wins versus where the cost cliff bites.

The runtime substrate question

Every system design eventually has to answer: what runs my code, and how does it get there? The substrate you choose — raw VMs, containers on Kubernetes, or functions on a serverless platform — determines your unit of deployment, your failure granularity, your scaling latency, your cost model, and how much operational surface area your team owns. A principal engineer treats this as a first-class architectural decision, not a default inherited from whatever the platform team standardized on.

The three substrates are not a ladder of sophistication. They are points on a curve trading control against operational leverage. VMs give you maximum control and maximum toil. Serverless gives you minimum toil and minimum control. Kubernetes sits in the middle and, critically, lets you mix both inside one cluster. This lesson is about understanding the mechanics well enough to place a workload correctly.

Key insight The container image is the real interface, not Kubernetes. An OCI image is a portable, content-addressed filesystem bundle. Whether it runs on ECS, Kubernetes, Nomad, Fargate, or a laptop is an orchestration choice layered on top. Decouple "how do I package" from "how do I orchestrate" in your thinking.

Containers vs VMs: what isolation you get

A virtual machine virtualizes hardware. A hypervisor (KVM, Xen, Hyper-V) presents virtual CPUs, virtual NICs, and virtual block devices to a full guest OS with its own kernel. Isolation is strong because the boundary is the hardware interface itself; a guest kernel exploit does not directly reach the host kernel. The cost is density and start time: each VM carries a full kernel and OS, boots in tens of seconds, and consumes hundreds of MB to GB before your process starts.

A container does not virtualize hardware. It is a normal Linux process with a restricted view of the host, assembled from three kernel primitives:

  • Namespaces — partition what a process can see: PID (own process tree), network (own interfaces and routing table), mount (own filesystem view), UTS (hostname), IPC, user (UID mapping), and cgroup namespaces.
  • cgroups (v2) — limit and account what a process can use: CPU shares/quotas, memory limits (with the OOM killer enforcing them), I/O bandwidth, PIDs.
  • Capabilities, seccomp, and LSMs (AppArmor/SELinux) — restrict what a process can do: drop CAP_SYS_ADMIN, filter syscalls, confine file access.

The defining property: all containers on a host share one kernel. That is why a container starts in milliseconds and adds near-zero overhead — but it is also why container isolation is weaker than VM isolation. A kernel vulnerability or container-escape bug is a host-level compromise affecting every co-tenant.

Trade-off Density and speed vs blast radius. Containers give you 10–100× the density and ~100× faster startup, at the cost of a shared-kernel attack surface. For untrusted multi-tenant workloads (running customer code), this gap matters — which is why microVM runtimes like Firecracker (AWS Lambda, Fargate) and gVisor (a userspace kernel) exist: VM-grade isolation with container-grade startup (~125 ms for Firecracker).

Kubernetes: the reconciliation model

The single most important idea in Kubernetes is not pods or YAML — it is the control loop. Kubernetes is a declarative system built on level-triggered reconciliation. You write desired state into etcd via the API server; a set of controllers continuously observe actual state and take action to converge the two. There is no imperative "do X" command; there is only "the world should look like this," repeated forever.

# The reconciliation loop, conceptually
for {
    desired  := readSpecFromAPIServer()   # what you declared
    observed := readStatusOfWorld()        # what actually exists
    diff     := desired - observed
    if diff != 0 {
        takeActionToward(desired)          # create/delete/update
    }
    # no "done" state; it runs forever, idempotently
}

This level-triggered design (react to current state) rather than edge-triggered (react to events) is what makes Kubernetes self-healing. If a node dies and 5 pods vanish, the next loop iteration sees "I want 10, I have 5" and creates 5 more — no event needed to have been delivered reliably. A dropped event is harmless because the next reconciliation re-derives reality. This is the same robustness philosophy behind anti-entropy repair: continuously reconcile toward a target rather than trusting a stream of deltas.

The control plane components:

  • API server — the only thing that talks to etcd; the front door for all reads/writes, doing auth, admission control, and validation.
  • etcd — the consistent, Raft-backed key-value store holding all cluster state. Its consensus layer is exactly the Raft protocol; etcd's availability and write latency bound the whole control plane.
  • Controller manager — runs the built-in reconcilers (Deployment, ReplicaSet, Node, Job controllers…).
  • Scheduler — assigns unscheduled pods to nodes.
  • kubelet (per node) — the node agent that actually starts containers via the container runtime (containerd/CRI-O) and reports status.
Pitfall etcd is the bottleneck and the single point of correctness. It is a CP system (see CAP) — on a partition it sacrifices availability to preserve consistency. A wedged etcd quorum freezes the entire control plane: no scheduling, no scaling, no config changes. Running workloads keep running (kubelet caches their spec), but you are flying blind. Treat etcd capacity (it tops out around a few GB) and quorum health as a tier-0 concern.

Pods, Deployments, Services, and the scheduler

The object hierarchy maps onto the reconciliation model cleanly:

  • Pod — the atomic scheduling unit: one or more containers sharing a network namespace (same IP and port space) and able to share volumes. Sidecars (a service-mesh proxy, a log shipper) ride along in the pod. Pods are ephemeral — they are cattle, not pets, and get a new IP every time.
  • ReplicaSet — ensures N copies of a pod template exist.
  • Deployment — manages ReplicaSets to give you declarative rolling updates and rollbacks. Changing the pod template spins up a new ReplicaSet and scales the old one down per maxSurge/maxUnavailable.
  • Service — a stable virtual IP and DNS name fronting a dynamic set of pods (selected by labels), since pod IPs churn.

The scheduler runs a two-phase algorithm per pending pod: filtering (which nodes are feasible — enough allocatable CPU/memory, matching nodeSelector/affinity, tolerating taints, satisfying topology constraints) then scoring (rank feasible nodes — spread for availability, bin-pack for cost, prefer image locality). It binds the pod to the highest-scoring node. Note the scheduler decides placement only; the kubelet on the target node does the actual work, and the scheduler never blocks waiting for it.

resources:
  requests:                 # used for scheduling + autoscaling math
    cpu: "500m"             # 0.5 core; the scheduler reserves this
    memory: "512Mi"
  limits:                   # hard ceiling; exceeding memory -> OOMKill
    cpu: "1"               # CPU over-limit is throttled, not killed
    memory: "1Gi"
Key insight Requests drive scheduling and bin-packing; limits drive runtime enforcement. The gap between them is how you overcommit. Set memory request == limit for predictable behavior (memory is incompressible — overshoot means OOMKill). For CPU (compressible — overshoot means throttling), a request floor with a higher or absent limit lets bursty workloads borrow idle capacity. Getting these wrong is the most common cause of mysterious latency (CPU throttling) and crash loops (OOMKills) in production clusters.

Autoscaling: HPA, VPA, cluster autoscaler, headroom

Three distinct autoscalers operate at different layers, and conflating them causes pain:

ScalerAxisSignalLatency
HPA (Horizontal Pod Autoscaler)Pod countCPU/memory utilization, or custom/external metrics (RPS, queue depth)Tens of seconds
VPA (Vertical Pod Autoscaler)Pod size (requests)Observed usage over timeRequires pod restart (until in-place resize matures)
Cluster Autoscaler / KarpenterNode countUnschedulable pods (no room to place them)Minutes (provision + boot a VM)

The reflex is to scale on CPU, but for most request-driven and especially queue-driven services, CPU lags the signal you care about. Scale on the leading indicator: for an HTTP service, requests-per-second per pod or p99 latency; for a consumer, queue depth or Kafka consumer lag. Tools like KEDA bridge external event sources into HPA so you can scale a worker fleet on partition lag directly.

Pitfall Do not run HPA and VPA on the same resource metric — they fight: VPA shrinks requests, which raises utilization %, which makes HPA add pods, which lowers utilization, which makes VPA shrink again. Use HPA on a custom metric (RPS) and let VPA handle a different dimension, or pick one.

Capacity headroom is the trade-off that distinguishes a senior answer from a principal one. Autoscaling is not instantaneous: HPA reacts in tens of seconds, node provisioning in minutes. If your traffic can step-change faster than that (a flash sale, a thundering-herd retry storm, a cache stampede), reactive scaling arrives too late and you serve errors during the gap. You buy down that risk with headroom — running at 60% utilization instead of 90%, or pre-provisioned "pause pod" overprovisioning that holds spare node capacity warm. Headroom costs money continuously to avoid latency during bursts. State the number: "we hold 30% headroom so we can absorb a 1.5× spike during the ~3 min it takes to add nodes."

Stateful workloads: the hard problem

Kubernetes was designed for stateless cattle. State breaks the core assumption that pods are interchangeable and disposable. A stateless web tier can lose a pod with zero consequence; a database replica cannot — it has identity, durable storage, and ordering constraints.

StatefulSet is the answer for stateful pods, providing what Deployments deliberately don't:

  • Stable network identity — pods get ordinal, predictable names (mysql-0, mysql-1) and stable per-pod DNS, so a re-scheduled mysql-0 is still addressable as the same member.
  • Stable storage — each pod gets its own PersistentVolumeClaim that follows it across rescheduling; the data is not lost when the pod moves.
  • Ordered, controlled rollout — pods are created/updated/deleted one at a time in order, which respects quorum and bootstrap dependencies (you don't want to restart all 3 etcd members at once).
Trade-off A StatefulSet gives you orchestration primitives, but it does not understand your data. Kubernetes will happily reschedule postgres-0 to a healthy node, but it has no idea about WAL replication, failover, or quorum. That knowledge lives in an Operator — a custom controller encoding domain logic (promote a replica, re-replicate, run backups). The reconciliation pattern extends to your data via Operators; this is exactly where leader election and replication mechanics re-enter the picture.

Pitfall "Just run your database in Kubernetes" is a decision, not a default. You inherit the full operational burden of distributed storage on ephemeral compute: network-attached volume latency, failover correctness, backup/restore, and split-brain risk during partitions. For many teams a managed database (RDS, Cloud SQL, DynamoDB) is the right answer precisely because it takes the consistency-during-partition hard problems off your plate. Run stateful systems in-cluster only when you have a mature Operator and a team that understands the data plane.

Service discovery and in-cluster load balancing

Because pod IPs are ephemeral, you never address pods directly. A Service gives a stable identity, and discovery happens through cluster DNS (CoreDNS): orders.payments.svc.cluster.local resolves to the Service's virtual IP.

The default ClusterIP service is L4 load balancing implemented by kube-proxy programming the kernel. Classic mode uses iptables (a long DNAT rule chain, O(n) per packet for n endpoints); modern clusters use IPVS or eBPF (Cilium) for O(1) hashing and far better behavior at thousands of endpoints. The balancing is essentially random/round-robin per connection — crude. It does not understand request cost, in-flight load, or zones by default.

This crudeness is exactly why a service mesh exists. A sidecar proxy (Envoy) per pod upgrades you to L7: request-level load balancing (least-request, EWMA), retries with budgets, circuit breaking, mTLS, and locality-aware routing that keeps traffic in-zone to cut cross-AZ data charges and latency. Compare this with the dedicated load balancing lesson — the mesh is "load balancing pushed into every hop of the call graph." For traffic entering the cluster, an Ingress controller or Gateway API resource fronts a cloud L7 load balancer.

Serverless / FaaS: cold starts and cost cliffs

Serverless inverts the model: you ship a function, the platform owns everything below it — provisioning, scaling, patching, even scaling to zero. The defining properties are statelessness (no durable local state between invocations; persist everywhere else) and scale-to-zero (you pay only for execution time, typically billed per millisecond × allocated memory).

The signature failure mode is the cold start. When no warm execution environment exists, the platform must: allocate a microVM (Firecracker ~125 ms), pull/mount your code, initialize the runtime (JVM/Node/Python), and run your init code — before the first request executes. Penalties range from ~100 ms (Go, small) to several seconds (a fat JVM with heavy DI). Mitigations: provisioned concurrency (pre-warmed environments — but now you're paying continuously, eroding the core value prop), lighter runtimes, snapshot-restore (Lambda SnapStart, CRaC), and trimming init work.

Trade-off Serverless wins on spiky, low-average, event-driven, embarrassingly parallel workloads: cron jobs, glue/ETL between services, webhook handlers, infrequent APIs, fan-out processing. It is dominated by containers on steady, high-throughput workloads because of the cost cliff: per-invocation pricing is wonderful at 1M requests/month and ruinous at sustained thousands of RPS, where a reserved container fleet at high utilization is often 5–10× cheaper. Compute the crossover point explicitly; don't argue it by vibes.

The other serverless cost is vendor lock-in and the loss of low-level control. You inherit hard execution-time limits (e.g. 15 min on Lambda), constrained networking (cold-start penalties spike when attaching to a VPC), no control over the kernel or co-tenancy, and an event/trigger model wired to one cloud's primitives. Migrating a deep Lambda + Step Functions + EventBridge architecture to another cloud is a rewrite, not a redeploy. This is a real strategic cost for a system you expect to run for a decade. Note that serverless also reshapes your observability story — no long-lived process to attach to, so you lean entirely on structured logs, distributed traces, and platform metrics.

Key insight "Serverless vs containers" is a false binary. Mature architectures are hybrid: a Kubernetes core for steady stateful and high-throughput services, with serverless functions for the spiky edges — image-thumbnailing on upload, S3-event ETL, scheduled reports, and glue between systems. Place each workload by its traffic shape, not by ideology. Knative and AWS Fargate even blur the line, giving scale-to-zero serverless semantics on top of containers.

For multi-region and DR, the substrate choice has teeth: Kubernetes is regional (etcd quorum cannot straddle high-latency links), so multi-region means independent clusters with global traffic management on top — not one stretched cluster. Serverless inherits the provider's regional model and can make active-active simpler at the compute layer, while pushing the hard part down to the data layer where the consistency/latency trade-offs live.

Principal-level mastery check

What strong engineers do They place each workload on the substrate that fits its traffic shape and statefulness, and they say the number — headroom percentage, cold-start budget, the RPS crossover where serverless stops being cheap. They know the isolation boundary they're actually getting (shared kernel vs microVM) and why it matters for the threat model. They treat etcd and the reconciliation loop as the heart of Kubernetes rather than reciting object names, and they're explicit that "run the database in K8s" requires an Operator and earns its complexity. Weak engineers default everything to "Kubernetes" or "serverless" as identity, can't explain requests-vs-limits, and don't connect autoscaler latency to user-facing errors during bursts.
  • A container shares the host kernel. Walk me through what isolation a container actually gives you and where it fails — and when would you reach for Firecracker or gVisor instead?
  • Kubernetes is "self-healing." Explain the mechanism. Why does level-triggered reconciliation survive dropped events when an edge-triggered design wouldn't?
  • Your service has p99 latency spikes every morning at 9am. CPU never exceeds 50%. What's likely happening, and how do requests/limits, HPA, and the cluster autoscaler each contribute to or fix it?
  • We're considering moving our high-throughput event consumer (sustained 8k RPS) from a container fleet to Lambda to "save ops effort." Argue both sides with actual cost reasoning. Where's the crossover?
  • Walk me through what breaks when you run Postgres as a StatefulSet versus on RDS. What does the Operator have to know that Kubernetes itself doesn't?
  • etcd loses quorum for 10 minutes. What still works, what stops, and what's your blast radius for already-running workloads versus new deployments?
