Designing a Metrics & Monitoring System

Ingest a torrent of time-stamped numbers from thousands of services — CPU, request rates, latencies — store them efficiently, and answer "what's the p99 latency over the last 6 hours, by region?" in milliseconds, then alert when something's wrong. We design Prometheus/Datadog from first principles. The defining facts: it's a write-heavy time-series problem, and the hidden cost that sinks everyone is cardinality.

The bottleneck Cheap, queryable storage of a massive metric firehose with fast aggregate reads and alerting — in exchange for a strict cardinality budget and lossy long-term aggregation. You get observability over your whole fleet. You pay by bounding label cardinality (the cost driver that explodes time-series count), accepting downsampled history (you can't keep every data point at full resolution forever), and computing percentiles correctly (never by averaging).

1. Requirements

Functional: ingest time-series metrics (name + labels + timestamp + value) from many sources; store them; query with aggregation over time and dimensions (sum/avg/rate/percentile, group-by label); render dashboards; evaluate alert rules and notify.

Non-functional: very high write throughput (millions of data points/sec), fast aggregate reads (dashboards in ms), cost-efficient at scale (you keep a lot of data), scalable (more services → more metrics), reliable (monitoring must stay up when the monitored system is sick — it's your eyes during an incident).

2. Capacity estimate

Say 10,000 hosts, each emitting 1,000 distinct metrics, scraped every 10 seconds. Work it:

  • Active time-series: 10,000 hosts × 1,000 metrics = 10 million active series. (Each unique metric+label combination is one series.) Consequence: the system's size is measured in series count, and that count is set by labels — the cardinality problem is born here.
  • Write rate: 10M series / 10s = 1 million data points/sec, sustained. Consequence: a write-heavy workload — you need a write-optimized (LSM-style) time-series engine, not a B-tree OLTP database.
  • Storage: 1M points/sec × ~2 bytes/point (heavily compressed — timestamps and values delta-encode beautifully) ≈ ~170 GB/day raw at full resolution. Over a year that's ~60 TB — too much to keep at 10s resolution. Consequence: you must downsample old data (keep 10s for days, 1m for weeks, 1h for years) — retention tiers are mandatory.
  • The cardinality trap: add one label with 10,000 values (a user id, a request id) to a metric and you multiply its series count by 10,000 — 10M series becomes 100B+. Consequence: cardinality, not raw rate, is what actually kills these systems.

3. The API

ingest(metricName, labels{}, timestamp, value)        // or scraped from a /metrics endpoint
query(promQL-like expr, from, to, step)  -> series[]    // rate(), histogram_quantile(), sum by(label)
alert(rule: expr + threshold + for-duration)            // fires when condition holds

Two ingestion models: push (sources send metrics) and pull/scrape (the system periodically fetches each source's /metrics endpoint — Prometheus's model, which gives the monitor control and an implicit liveness check). Queries are aggregation-heavy (rates, sums, percentiles, group-by); the query language (PromQL-style) is built for that, not for point lookups.

4. High-level design

Four parts: (1) Collection — agents on each host expose/forward metrics; the system scrapes (or receives) them. (2) Ingestion + storage — a time-series database (TSDB) optimized for high-write, compressed, time-ordered data (an LSM-style engine). (3) Query engine — serves aggregate queries for dashboards (percentiles via histograms, rates, group-by). (4) Alerting — an evaluator runs alert rules on a schedule and fires to a notification system. For scale and HA, ingestion is sharded (by series) and replicated; old data tiers to cheaper storage. The monitoring system runs independently of what it monitors (so it survives when the fleet is sick).

5. Data model — the time-series & cardinality

A time-series is uniquely identified by a metric name plus a set of labels (key=value pairs): http_requests_total{service="api", route="/users", status="200", region="us"}. Each unique label-set is a separate series with its own stream of (timestamp, value) points. Total series count = the product of every label's distinct values — this product is the cardinality, and it is the single number that determines the system's cost and feasibility. Points compress extremely well (delta-of-delta on timestamps, XOR on values), so storage per point is tiny — but the number of series is everything.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to the comparison.

Scenario 1 — one label blows up the whole system (real-pattern incident) A developer adds a label to a metric — user_id — to "make debugging easier." Overnight the time-series count goes from millions to billions, ingestion falls behind, queries time out, and the monitoring bill explodes. What exploded, why does that one label multiply cost so violently, and what should carry per-user detail instead?
Scenario 2 — you can't keep everything forever, but you need history You want 10-second resolution for live dashboards and a year of history for capacity trends. Keeping every 10-second point for a year is petabytes — unaffordable. But you can't just delete old data; you need the trends. How do you keep useful history without keeping every raw point forever?
Scenario 3 — the "p99" that's a lie Each of 50 hosts reports its own p99 latency, and the dashboard shows the average of the 50 p99s as the global p99. A senior engineer says that number is meaningless. Why can't you average percentiles, and what must each host store so you can compute a correct global p99?

Scenario 1 is the cardinality budget. Scenario 2 is storage tiers & downsampling. Scenario 3 is querying percentiles correctly. Each maps to a sub-problem.

The four hard sub-problems

The collect→store→query→alert shape is standard. These four are where metrics systems are won or lost — each with a nested failure.

Sub-problem 1 — Ingesting the write firehose

1M+ points/sec, sustained and append-only, is a write-heavy workload that a general-purpose OLTP B-tree database handles badly (random in-place writes). The answer is a purpose-built time-series database with an LSM-style write path (sequential appends), aggressive compression (delta-encoded timestamps, XOR'd values — often <2 bytes/point), and sharding of series across nodes for horizontal write scale.

The failure that happens when you get this wrong — write overload Using the wrong storage engine (a row-oriented OLTP DB, or an under-provisioned single node) means the sustained 1M-points/sec firehose either falls behind (gaps in your data — blind spots during the incident you most need to see) or requires absurd hardware. The fix: a write-optimized TSDB — LSM-style sequential writes (Prometheus's TSDB, or scalable backends like Cortex/Mimir/VictoriaMetrics), heavy columnar/delta compression (metrics compress ~10× better than generic data because consecutive timestamps and values are highly regular), and sharding by series so writes spread across nodes. Recent data is held in memory + an append log and flushed to compressed blocks. Treating metrics like OLTP rows is the scale failure; treating them as compressible append-only time-series is the design.

Sub-problem 2 — The cardinality budget

This is the defining problem. Since total series = the product of all label values, every label is a multiplier. Labels must be bounded/low-cardinality (service, route, status, region — tens of values each) and never unbounded (user id, request id, full URL, email — millions of values). Enforcing this budget is the difference between a system that costs thousands and one that costs millions (or falls over).

The failure that happens when you get this wrong — the cardinality explosion This is Scenario 1. Total series count is the product of every label's distinct values, so adding one high-cardinality label (user_id with millions of values) multiplies the series count by millions — "label explosion." Ingestion can't keep up, the index of series bloats memory (TSDBs hold series metadata in RAM), queries that scan many series time out, and cost detonates. The rules: metric labels must be bounded — only the small set you'd legitimately group-by (service/route/status/region). Per-entity detail (which user, which request) belongs in logs or traces, which are indexed for high cardinality, not in metric labels. Watch hidden explosions: a route label containing IDs (/users/12345) is secretly unbounded — normalize to /users/:id. Guard with per-metric series limits that reject offending labels at ingest. This one decision dominates the whole system's viability.

Sub-problem 3 — Storage tiers & downsampling

You can't keep every raw point at full resolution forever — it's too much data. The answer is retention tiers with downsampling: keep recent data at high resolution (10s for a few days — needed for live debugging), then downsample older data to coarser resolution (1m for weeks, 1h for a year), discarding the fine-grained points you no longer need. Old, downsampled data moves to cheaper storage (object storage).

The failure that happens when you get this wrong — unbounded retention cost This is Scenario 2. Keeping all data at 10-second resolution forever grows storage (and query cost) without bound — petabytes, most of which nobody queries at fine resolution (you don't need per-10-second data from 8 months ago, you need the trend). Downsampling resolves it: as data ages, roll it up to coarser intervals (compute 1-minute and 1-hour aggregates — min/max/avg/count, and crucially the histogram for percentiles — from the raw points, then drop the raw). So you keep useful history (the trends, at appropriate resolution) at a fraction of the cost, and tier cold rollups to cheap object storage. The trap is either keeping everything (cost explosion) or deleting old data outright (losing trends) — downsampling keeps the signal while shedding the volume. Be careful to downsample percentile-capable data (histograms, not pre-computed percentiles — see sub-problem 4).

Sub-problem 4 — Querying percentiles & alerting

Dashboards live on aggregate queries — rates, sums, group-by, and especially percentiles (p50/p99/p999 latency, the numbers that reflect user experience). Percentiles can't be computed from averages or summed across series; you need the distribution. So latency is recorded as a histogram (bucketed counts) per series, and percentiles are computed from merged histograms. Alerting then evaluates rules (often SLO burn-rate) on these queries and fires to a notification system.

The failure that happens when you get this wrong — averaging percentiles This is Scenario 3, one of the most common metrics mistakes. You cannot average percentiles — the mean of 50 hosts' p99s is not the global p99, because a percentile is a property of the whole distribution, not a value you can linearly combine (a host with few requests and a host with millions can't be averaged). The correct design: each host records latency as a histogram (counts per latency bucket) per series; to get the global p99, you merge the histograms (sum the bucket counts across hosts) and compute the percentile from the merged distribution. This is exactly why metrics systems store histograms, not pre-computed per-host percentiles — histograms are mergeable, percentiles are not. (Same lesson as t-digest / HdrHistogram.) Storing or downsampling pre-computed percentiles destroys your ability to aggregate them — always keep the histogram.

At a glance — the design decisions

ConcernMechanism
Absorb 1M+ points/secWrite-optimized TSDB (LSM), delta/XOR compression, shard by series
Control costCardinality budget — bounded labels only; per-entity detail to logs/traces

The rule in one lineStore metrics in a write-optimized, heavily-compressed TSDB, ruthlessly enforce a cardinality budget (bounded labels; per-entity detail goes to logs/traces), keep history via downsampling tiers, and compute percentiles from mergeable histograms (never averaged). Cardinality — not write rate — is what kills these systems.

The shape to remember: 4 sub-problems — ingest firehose (TSDB), cardinality budget, downsampling tiers, percentiles via histograms.

Famous implementations

Same facets — what it is, the model, and the failure to watch.

Prometheus

  • What it is. The de-facto open-source metrics system — pull-based scraping, label model, PromQL, its own TSDB.
  • Model. Scrapes /metrics endpoints (implicit liveness check), stores compressed series locally, evaluates alert rules via Alertmanager.
  • The failure to watch. Cardinality explosion can OOM the server (series index in RAM); single-node storage — use Cortex/Thanos/Mimir for HA + long-term.

Time-series databases (VictoriaMetrics / InfluxDB / Druid)

  • What it is. Purpose-built TSDBs optimized for the metric write firehose + aggregate reads.
  • Model. LSM-style writes, columnar/delta compression, downsampling/retention policies, fast aggregation queries.
  • The failure to watch. Same cardinality limits; choosing the right one depends on cardinality, retention, and query patterns.

Datadog / Grafana (dashboards + SaaS)

  • What it is. All-in-one observability (Datadog) or vendor-neutral dashboards (Grafana) over many backends.
  • Model. Ingest metrics/logs/traces; correlate; alert; bill on volume + cardinality.
  • The failure to watch. Cardinality- and volume-based billing makes uncontrolled labels financially ruinous — the budget is a cost decision.

Build it yourself

Work each before opening the approach.

Design a metric with a safe cardinality budget

You want to monitor HTTP request rate, errors, and latency, sliced usefully, without a cardinality explosion. Define the metric(s) and labels — and say what you deliberately exclude.

Reveal the approach

Emit RED metrics: a request counter, an error counter (or status label), and a latency histogram. Label with bounded dimensions only: service (tens), route normalized (/users/:id, never the raw ID — that's the hidden explosion), method (handful), status_class (2xx/4xx/5xx, not every code), region (handful). Series count = product of those = maybe a few thousand per service — cheap and groupable. Deliberately exclude: user_id, request_id, session_id, full URLs, email — all unbounded; putting any of them in a metric label multiplies series count by their cardinality (Scenario 1). When you need "which user/request," that lives in logs/traces (keyed by trace id), which are built for high cardinality. Enforce it with a per-metric series limit at ingest that rejects offending labels before they explode the index.

Compute a correct global p99 across hosts

50 hosts each measure request latency. Design how you get an accurate fleet-wide p99 — and explain why the obvious approach is wrong.

Reveal the approach

The wrong (common) approach: each host computes its own p99 and you average the 50 — meaningless, because a percentile isn't linearly combinable (a host serving 10 requests and one serving 10M can't be averaged, and the mean of percentiles is not the percentile of the union; Scenario 3). The right approach: each host records latency as a histogram — counts of requests per latency bucket — as a metric (e.g. http_latency_bucket{le="0.1"}, {le="0.3"}, …). To get the fleet p99, the query engine sums the bucket counts across all 50 hosts (histograms are mergeable) to form the global distribution, then reads the p99 off the merged buckets (e.g. PromQL histogram_quantile(0.99, sum by(le)(rate(http_latency_bucket[5m])))). This is exact-within-bucket-resolution and correctly weighted by each host's request volume. The principle: store and merge histograms, never pre-computed percentiles — and remember it when downsampling too (keep the histogram, not the percentile).

Design-it-yourself: the open-ended ones

  • Add long-term storage + HA. Single-node Prometheus can't hold a year or survive a node loss. Design horizontal storage + replication (Thanos/Cortex-style) with downsampling tiers. → Object Storage, Replication.
  • Build SLO burn-rate alerting on top. Wire alert rules to error-budget burn rate so pages mean "users hurt fast," routed through a notification system. → SLI/SLO/SLA.
  • Cut the metrics bill in half. Given an exploding cost, find the cardinality and retention offenders and redesign labels + downsampling. → Cost-Aware System Design.
