CAP Theorem

The classic Consistency/Availability/Partition-tolerance trade-off — and why modern practice extends it to PACELC.

The bottleneck No shared truth. when the network partitions you must give up consistency or availability — you cannot keep both.

In this lesson

  • The three guarantees
  • Why you can only pick two — during a partition
  • Why “CA” is not a real choice
  • CP systems — consistency over availability
  • AP systems — availability over consistency

CAP theorem states that it is impossible for a distributed software system to simultaneously provide more than two out of three of the following guarantees. When we design a distributed system, trading off among these three guarantees is almost the first thing we want to consider. The theorem — originally conjectured by Eric Brewer in 2000 and later formally proven by Gilbert and Lynch — gives us the vocabulary to talk precisely about what a system gives up when something goes wrong.

The three guarantees

The three properties at the heart of the theorem are:

  • Consistency (C): All nodes see the same data at the same time. Concretely, a read is guaranteed to return the most recent write for a given piece of data (or an error). Consistency is achieved by updating several nodes before allowing further reads.
  • Availability (A): Every request received by a non-failing node in the system must result in a non-error response. Availability is achieved by replicating the data across different servers — but note that an available response carries no guarantee that it reflects the most recent write.
  • Partition tolerance (P): The system continues to function and upholds its consistency guarantees in spite of network partitions. A partition is any communication breakdown — arbitrary message loss, delay, or failure — between nodes in the system. Distributed systems guaranteeing partition tolerance can gracefully recover from partitions once the partition heals.

Why you can only pick two — during a partition

The crux of CAP is that you cannot have all three at once. More precisely: in the presence of a network partition, you must choose between consistency and availability. When the network splits and two nodes can no longer talk to each other, a node that receives a request faces a binary decision with no third option:

  1. Reject the request (or block) to preserve consistency. If a node cannot confirm with its peers that it holds the latest data, it refuses to serve a possibly-stale read or accept a write that it cannot replicate. It sacrifices availability to keep consistency — this is a CP system.
  2. Serve the request anyway to preserve availability. The node responds with whatever data it has, accepting that it may be stale, and accepts writes locally to reconcile later. It sacrifices consistency to keep availability — this is an AP system.
Key insightCAP is not a menu you order from once at design time. The trade-off only forces itself on you when a partition actually occurs. When the network is healthy, a well-built system can offer both strong consistency and high availability. CAP describes degradation behavior under failure, not steady-state capability.

Why “CA” is not a real choice

The Venn diagram suggests three pairs — CP, AP, and CA — but the CA corner is a trap. In any real distributed system that communicates over a network, partitions will happen: switches fail, cables get cut, garbage-collection pauses freeze a node, packets are dropped under load. Network partitions are a fact of life, not an optional condition you can design away.

Because a partition is inevitable, partition tolerance is not negotiable for a distributed system — you must tolerate partitions, which means the real choice is always between C and A. A system that claims to be “CA” is really just a CP or AP system that has not yet decided how it behaves under a partition, or a single-node system where the question does not arise. So in distributed systems, partition tolerance is a given, and we are left choosing between consistency and availability when (not if) the network fails.

CP systems — consistency over availability

A CP system chooses consistency over availability when a partition occurs. If the system cannot guarantee that a read reflects the most recent write — because the nodes it needs to confirm with are unreachable — it will return an error or time out rather than serve potentially stale data. The minority side of a partition effectively stops serving.

  • Banking and financial ledgers are the canonical example: you would rather a transfer fail and ask the user to retry than allow two partitions to independently spend the same balance.
  • Coordination and metadata stores such as ZooKeeper, etcd, and Consul are deliberately CP — they back a consensus protocol and would rather become unavailable than hand out conflicting answers about who holds a lock or which node is the leader.
  • HBase and MongoDB (in their default configurations) lean CP: a region/primary that cannot reach the rest of the cluster stops accepting writes.

AP systems — availability over consistency

An AP system chooses availability over consistency. Every node keeps answering requests even when partitioned, returning the best data it has locally and accepting writes that it will reconcile later (often via versioning, vector clocks, or last-write-wins). The system stays up everywhere, but two clients on opposite sides of a partition may briefly see different values. Once the partition heals, the replicas converge — this is eventual consistency.

  • Amazon DynamoDB and Apache Cassandra are the textbook AP stores, built on the Dynamo lineage where staying available for shopping carts mattered more than perfectly current reads.
  • DNS and web caches / CDNs are everyday AP systems: they serve possibly-stale answers globally rather than refusing to resolve a name.
  • Couchbase and Riak similarly favor availability and resolve conflicts after the fact.
The trade-off in one line: CP sacrifices uptime to never lie about data; AP sacrifices freshness to never go down. The right answer depends entirely on whether a stale answer or no answer is worse for your domain — a stale “likes” count is harmless, a stale account balance is not.
What strong engineers doThey never just say “it’s a CP system.” They name the operation, the failure, and the user-visible consequence: “During a partition, writes to the minority side are rejected with a 503, so the client must retry; reads on the majority side stay strongly consistent. We accept a few seconds of write unavailability per region failover because double-spending is unacceptable here.” They also point out that CAP only bites during a partition and immediately reach for PACELC to describe the normal-case latency cost.
Principal-level deep dive

From CAP to PACELC and the consistency spectrum

CAP is necessary vocabulary but insufficient as a design tool, and over-reliance on it is a tell-tale sign of a junior answer. Its two weaknesses are well known. First, it treats consistency and availability as binary, when both are spectrums. Second, and more importantly, it only says anything about the system during a partition — and partitions are rare. A datacenter network is healthy the overwhelming majority of the time. CAP is silent about the trade-off you make on every single request the other 99.9% of the time, which is precisely the trade-off that dominates your latency budget and your user experience.

PACELC: the else-branch CAP forgets

Daniel Abadi's PACELC (2010) fixes this by spelling out both regimes. Read it as two conditional clauses:

PACELCIf there is a Partition, choose between Availability and Consistency (the CAP question); Else — in normal operation — choose between Latency and Consistency.

The ELC leg is the common case CAP ignores. Even with a perfectly healthy network, a strongly consistent system must coordinate across replicas before acknowledging — a quorum write, a leader round-trip, a consensus log append — and that coordination costs latency. To go faster you must relax consistency (read from a follower, accept eventual convergence). So every read and write carries a latency-vs-consistency decision, partition or not. This is why a principal engineer reaches for PACELC: it forces you to state behavior in both regimes instead of only the rare one.

Classifying real systems

PACELC gives a two-letter classification, one letter per leg:

SystemPACELC classReading
Google SpannerPC/ECConsistent under partition and in normal operation; it pays the latency (TrueTime commit-wait) to stay externally consistent everywhere.
DynamoDB / CassandraPA/EL (tunable)Available under partition, low-latency in normal operation by default — but per-request quorum knobs let you dial toward consistency when you need it.
MongoDBPA/ECUnder partition it keeps the majority available, while in normal operation it favors consistency by routing reads to the primary.

The tunable cases matter most in practice. Cassandra's R + W > N quorum math is literally the ELC leg exposed as a per-statement dial: QUORUM/QUORUM buys consistency at higher latency, ONE/ONE buys latency at the cost of staleness. There is no single answer for the whole datastore — you choose per operation.

Consistency is a spectrum, not a switch

“Consistent vs eventually consistent” hides a whole ladder of models, ordered from strongest (and most expensive) to weakest (and cheapest):

  • Linearizable — the system behaves as if there is a single copy of the data and every operation takes effect at one instant between its call and return. The gold standard; requires consensus or synchronous quorums.
  • Sequential — all nodes see operations in the same total order, but that order need not match real (wall-clock) time. Weaker than linearizable, which adds the real-time constraint.
  • Causal — operations that are causally related (a reply after a comment) are seen in order everywhere; concurrent, unrelated operations may be ordered differently on different nodes. Often the sweet spot: it preserves the orderings users actually notice without paying for global agreement.
  • Eventual — if writes stop, all replicas eventually converge to the same value, with no guarantee about intermediate reads or ordering. Cheapest and most available.

Choosing a point on this ladder is the real design decision; CAP's binary collapses it into a single yes/no. See Consistency Models & PACELC for the full ladder and how each level is enforced.

Why CA isn't a real choice, restated

Worth saying once more in PACELC terms: there is no CA class. PACELC has no notation for “ignores partitions” because a distributed system cannot. Every system must have a P-clause answer, even if its designers never consciously chose one — in which case the partition behavior is whatever the implementation accidentally does, which is far worse than a deliberate choice. A claimed CA system is a single-node system, or a CP/AP system in denial.

How consensus systems land on CP

Systems built on consensus — Raft and Paxos backing ZooKeeper, etcd, Consul, and the metadata layers of Spanner, CockroachDB, and Kafka's controller — deliberately choose CP. A consensus protocol requires a majority quorum to commit any entry. When a partition splits the cluster, at most one side can hold a majority. The majority side keeps making progress and stays consistent; the minority side cannot commit and therefore stalls — it refuses writes and (for linearizable reads) refuses reads rather than serve stale state. This is the CP choice made concrete: the system sacrifices availability on the minority partition to guarantee that it never commits two conflicting histories. When the partition heals, the minority catches up by replaying the leader's log. This is exactly why you put a coordination service on the critical path only where you can tolerate that stall, and keep high-availability data paths on AP stores. See Consensus & Raft for the quorum mechanics and Replication for how leader/follower topologies turn these choices into routing decisions.
