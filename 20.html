Designing Twitter

A read-heavy social feed: ~1,150 tweets written per second, but ~325,000 timeline reads per second. We walk the full design — requirements, capacity math, API, high-level architecture, data model — then deep-dive the four hard parts. The thread running through all of it: optimize relentlessly for reads, and the trick that makes reads cheap (fan-out on write) is exactly what breaks for celebrities.

The crux Reads outnumber writes ~280:1, so you do more work on the rare write to make the common read cheap. Pre-compute each user's timeline when a tweet is posted, so a read is a lookup, not a giant query — but pre-computing for a user with 100M followers is its own failure, which is why the real answer is a hybrid.

1. Requirements

Pin down scope before designing — a design is only "right" relative to what it must do.

Functional (what it must do):

  • Post a tweet (text, up to 140 chars, optional photo/video).
  • Follow other users.
  • Favorite (like) a tweet.
  • See a home timeline — the merged, recency-sorted tweets of everyone you follow.

Non-functional (how well):

  • Highly available — the system stays up; a brief stale timeline is acceptable.
  • Low latency — ~200 ms to generate a timeline.
  • Eventual consistency is fine — not seeing a tweet for a few seconds is OK (this is a deliberate CAP choice: availability over strict consistency).

Extended (note but defer): search, replies, trending topics, tagging, notifications, who-to-follow. We design the core and point to where the rest is covered.

2. Capacity estimate

The numbers aren't busywork — the read:write ratio is the architecture. Assume 1 billion total users, 200M daily active, 100M new tweets/day, average 200 follows each.

Traffic: the ratio that decides everything

  • Writes: 100M tweets / 86,400s ≈ 1,150 tweets/sec.
  • Reads: assume each of 200M users loads their timeline twice and views 5 other pages, 20 tweets each: 200M × (2+5) × 20 ≈ 28B views/day ≈ 325,000 reads/sec.
  • Read:write ratio ≈ 280:1. This single number is why every decision below favors reads.

Storage

  • Text: 140 chars × 2 bytes + ~30 bytes metadata ≈ 310 bytes/tweet. 100M × 310 ≈ 30 GB/day of text.
  • Media: assume every 5th tweet has a 200 KB photo, every 10th a 2 MB video: (100M/5 × 200KB) + (100M/10 × 2MB) ≈ 24 TB/day of media. Media dominates storage by ~800× — which is why it never lives in the tweet database.

Bandwidth

  • Ingress: 24 TB/day ≈ 290 MB/s in.
  • Egress (28B views/day, showing every photo but only every 3rd video watched): ~93 MB/s text + ~13 GB/s photos + ~22 GB/s video ≈ 35 GB/s out. Egress is ~120× ingress — the system is overwhelmingly about serving data, which is what pushes media to a CDN.
What the numbers tell usThree facts fall out and shape the whole design: (1) reads dominate writes 280:1 → pre-compute on write (fan-out); (2) media is ~800× the text volume → object store + CDN, never the tweet DB; (3) egress is ~120× ingress → the system is a read-serving machine.

3. System API

Define the contract before the internals. The core write endpoint:

tweet(api_dev_key, tweet_data, tweet_location,
      user_location, media_ids)
  → returns the URL of the created tweet, or an HTTP error
  • api_dev_key — identifies the caller; also used to rate-limit per quota.
  • tweet_data — the text (≤140 chars).
  • tweet_location, user_location — optional lat/long.
  • media_ids — IDs of photos/videos uploaded separately (media never travels inline through the tweet path — see the bandwidth numbers).

And the read endpoint that everything optimizes for:

getHomeTimeline(api_dev_key, user_id, count, max_id)
  → returns the next `count` tweets in the user's home timeline

4. High-level architecture

Given 1,150 writes/sec and 325K reads/sec, this is fundamentally a read-heavy system. The high-level shape:

  • Stateless application servers behind load balancers — scale horizontally with traffic; the LB sits at the client↔app edge (and again app↔cache and app↔DB).
  • A write path: post a tweet → assign an ID → persist to the tweet store → fan out into followers' timelines (the expensive step, done on write).
  • A read path: get home timeline → read the pre-computed timeline from a cache (a cheap lookup) → hydrate tweet bodies.
  • A separate media path: upload/serve photos & video via an object store + CDN, never the tweet DB.
  • Read replicas on the tweet store to absorb the 280:1 read load; the primary takes writes.
The shape in one sentenceStateless app servers behind load balancers, a write path that pre-computes timelines via fan-out, a read path that serves them from cache, and media offloaded to a CDN — everything arranged so the 325K/sec reads stay cheap.

5. Data model

Four core entities. The relationships are simple; the interesting choices are the indexes and how this gets sharded (Sub-problem 2).

TableKey fieldsNotes
UseruserID (PK), name, email, created_atThe account.
TweettweetID (PK), userID, content, media_ids, created_at, locationtweetID is time-encoded (Sub-problem 4), so it sorts by recency for free.
FollowfollowerID, followeeIDThe social graph; "who do I follow" drives fan-out.
FavoriteuserID, tweetID, created_atLikes.

SQL vs NoSQL here follows the same reasoning as Instagram: the relationships are simple and the access patterns are known, so the model itself isn't the hard part — scale is. That's why the weight of the design is in sharding, caching, and ID generation, which we take next. (How to make this choice in general: Choosing a Database.)

Three scenarios — the hard parts, as questions

The walkthrough above is the routine design. The interesting engineering is in four sub-problems where the obvious approach breaks. Decide your answer to these before reading on.

Scenario 1 — the celebrity posts Your write path pushes each new tweet into every follower's pre-computed timeline (fan-out on write). A celebrity with 100 million followers posts one tweet. How many writes does that single tweet trigger, and what happens to your write pipeline?
Scenario 2 — the hot shard You shard the tweet store by UserID so all of a user's tweets live on one server. One account goes viral and its tweets are read millions of times an hour. Which server feels it, and why doesn't adding more shards help that one account?
Scenario 3 — the 64-bit ID Two tweets are posted in the same millisecond on two different servers. You need every tweet to have a globally unique ID that also sorts by time (so timeline merges are cheap). How do you generate IDs that never collide and stay time-ordered, without a single central counter becoming a bottleneck?

Scenario 1 is the celebrity fan-out problem — the single most important idea here. Scenario 2 is the hot shard. Scenario 3 is ID generation (Snowflake). All three are sub-problems below.

6. The four hard sub-problems

This is where the design earns its keep. Each has a characteristic failure, shown nested beneath it.

Sub-problem 1 — Timeline generation (fan-out)

A home timeline is the merged recent tweets of everyone you follow. Two pure strategies at opposite ends of the read/write trade:

  • Fan-out on read (pull). Store each user's tweets with them; build a timeline by querying everyone they follow and merging. Cheap writes, but a read does hundreds of queries + a merge — far too slow at 325K reads/sec.
  • Fan-out on write (push). When a user posts, copy the tweet into every follower's pre-computed timeline. A read is then one cheap lookup. The right default because reads dominate.
The failure when you get this wrong — the celebrity fan-out This is Scenario 1. Fan-out on write means one tweet = one write per follower. A celebrity with 100M followers turns a single tweet into 100 million writes — write amplification that floods the pipeline and delays everyone's tweets.
How Twitter actually solves it — the hybrid. Twitter fans out on write for ordinary users but excepts high-follower celebrity accounts: their tweets are not pushed. When you load your timeline, the system merges your pre-computed timeline (normal follows) with a pull of the latest tweets from the few celebrities you follow. The expensive path runs only for the handful of accounts where push is ruinous. (At its peak, Justin Bieber's account alone was reported to be ~3% of all Twitter traffic — the celebrity case is load-bearing, not a corner case. This hybrid is the canonical answer in Kleppmann's Designing Data-Intensive Applications.)

Sub-problem 2 — Tweet storage & sharding

1,150 writes/sec and billions of reads means tweets must spread across many machines. Three sharding keys, each flawed:

Shard byHow it worksThe catch
UserIDAll of a user's tweets on one serverA hot/celebrity user overloads one server; uneven growth
TweetID (random)Each tweet on a random serverEven load, but a timeline read must query every shard and aggregate — higher latency
Creation timeRecent tweets togetherAll writes hit the newest shard (hotspot); old shards idle
The combined answerShard by TweetID, but make the TweetID encode the timestamp (Sub-problem 4). You get even write distribution and fast "find the latest" — the benefits of both, which is exactly what a Snowflake-style ID buys.
The failure when you get this wrong — the hot shard This is Scenario 2. Shard by UserID and one viral account concentrates all its read traffic on the single server that owns it — adding shards doesn't help, because that user still lives on one shard. It's the same hot-key problem from the caching lesson, and the fixes are the same family (cache the hot account, replicate it, or split it).

Sub-problem 3 — The read cache

Because reads dominate, you cache aggressively. Two observations shape it:

  • 80-20: ~20% of tweets generate ~80% of reads — cache the hot 20% per shard and you absorb most traffic.
  • Recency: ~80% of users only look at the last ~3 days. Caching just the last 3 days of tweets is ~100 GB — small enough to hold in memory and replicate for read scaling.

The timeline cache is a hash table keyed by OwnerID whose value is a capped list of recent tweets, newest at the head — a per-user LRU list.

The failure when you get this wrong — the cold-timeline storm If the timeline cache is cold or restarts, every read falls through to the database's expensive merge at once — the cache stampede, now aimed at your tweet store. Warm the cache and bound the fallback (single-flight, last-3-days replicas) so a cache blip doesn't take down the database.

Sub-problem 4 — Tweet ID generation

This is Scenario 3. You need IDs that are (a) globally unique across many servers and (b) roughly time-sortable, so merging timelines and finding "latest" is cheap — without a single counter everyone contends on.

The answer is a time-ordered 64-bit ID: high bits = timestamp (epoch ms), low bits = a per-node sequence + a node identifier. ~31 bits hold 50 years of seconds; ~17 bits hold a per-second sequence (130K IDs/sec/node). Because the timestamp is most significant, IDs sort chronologically; because each node owns its sequence and ID, no two nodes collide and there's no central bottleneck.

The failure when you get this wrong — collisions or lost order A naive per-server auto-increment collides across servers; a single central counter is a SPOF and a throughput ceiling. Put the timestamp in the low bits and your IDs stop sorting by time, making merges expensive. The structure (timestamp high, node+sequence low) is the whole trick — this is Snowflake.

Self-check — can you reconstruct the design?

Close the page and rebuild it — both the walkthrough and the hard parts.

  • The walkthrough: from the assumptions (200M DAU, 100M tweets/day, 200 follows), derive the write rate, read rate, and ratio — and say what each of the three big numbers (280:1, media 800×, egress 120×) forces in the design.
  • Scenario 1 (celebrity): Why does fan-out on write turn one celebrity tweet into 100M writes, and how does the hybrid fix it without slowing everyone's reads?
  • Scenario 2 (hot shard): Why does sharding by UserID concentrate a viral account on one server, why doesn't adding shards help, and what caching-lesson problem is this?
  • Scenario 3 (IDs): What goes in the high vs low bits of a Snowflake ID, why does that make merges cheap, and why is a central counter wrong?
  • Why shard by a time-encoded TweetID rather than UserID, random TweetID, or raw creation time?
  • Why does media never travel through the tweet database or the tweet API?

The shape to remember: walk the design (requirements → estimate → API → high-level → data model), then solve 4 hard parts — timeline generation, sharding, read cache, IDs — with the hybrid fan-out as the load-bearing insight.

Famous implementations

How real systems implement the pieces. Same facets — what it is, the key idea, the failure to watch.

Twitter's hybrid timeline

  • What it is. The production answer to Sub-problem 1: fan-out on write for ordinary users, fan-out on read for high-follower celebrities, merged at read time.
  • Key idea. Run the expensive path only where the cheap one fails. The vast majority of accounts are cheap to push; the tiny number of mega-accounts are pulled.
  • The failure to watch. The read-time merge adds latency for users who follow many celebrities — bounded because celebrity tweets are read by everyone and cache extremely well.

Redis-backed timelines

  • What it is. Each home timeline stored as a Redis list of tweet IDs, newest at the head, capped at a few hundred (nobody scrolls back 10,000 tweets).
  • Key idea. Store IDs, not full tweets — hydrate bodies from the tweet cache on read. The timeline is tiny; writes are O(1) LPUSH + LTRIM.
  • The failure to watch. Per-user lists are data to keep coherent and rebuild after a Redis loss — you need a way to regenerate a timeline from source.

Snowflake IDs

  • What it is. Twitter's open-sourced 64-bit ID: timestamp (high bits) + datacenter/worker ID + per-node sequence. The answer to Sub-problem 4.
  • Key idea. Time in the high bits makes IDs k-sortable; node ID + sequence in the low bits removes any central coordinator.
  • The failure to watch. Clock skew — if a node's clock moves backwards, IDs can collide or mis-order; production Snowflake refuses to issue IDs while the clock is rewound. (Full treatment in Lesson 58.)

Build it yourself

Designing is a skill you practice. Work each before opening the approach.

Redo the capacity estimate from scratch

From 200M DAU, 100M tweets/day, 200 follows: derive write rate, read rate, ratio, storage, and egress — without looking back. Then say which design decision each number forces.

Reveal the approach

Writes: 100M / 86,400 ≈ 1,150/s. Reads: 200M × (2+5) × 20 ≈ 28B/day ≈ 325K/s → ratio ~280:1 → fan-out on write. Storage: ~310 B/tweet → 30 GB/day text; media (every 5th photo 200KB, every 10th video 2MB) → 24 TB/day → object store + CDN. Egress ~35 GB/s vs ingress ~290 MB/s → ~120× → read-serving machine. The discipline: every number ends in a design consequence, never a bare figure.

Implement hybrid fan-out

Write the post-tweet logic: decide push vs pull per author, and the read-path merge. Where's the threshold?

Reveal the approach

On post: if the author's follower count is below a threshold (~10K–100K), enqueue a fan-out job that LPUSHes the tweet ID into each follower's capped timeline. If above (a celebrity), skip the push and mark them a "pull" author. On read: take the user's pre-computed timeline, then for the few celebrities they follow, pull those authors' latest tweets and merge-sort by the time-sortable tweet ID. The threshold trades write amplification against read-merge cost; celebrities are rare, so the merge stays small.
