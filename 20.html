Designing Twitter

A read-heavy social feed: ~1,150 tweets written per second, but ~325,000 timeline reads per second. We walk the full design — requirements, capacity math, API, high-level architecture, data model — then deep-dive the four hard parts. The thread running through all of it: optimize relentlessly for reads, and the trick that makes reads cheap (fan-out on write) is exactly what breaks for celebrities.

The crux Reads outnumber writes ~280:1, so you do more work on the rare write to make the common read cheap. Pre-compute each user's timeline when a tweet is posted, so a read is a lookup, not a giant query — but pre-computing for a user with 100M followers is its own failure, which is why the real answer is a hybrid.

1. Requirements

Pin down scope before designing — a design is only "right" relative to what it must do.

Functional (what it must do):

  • Post a tweet (text, up to 140 chars, optional photo/video).
  • Follow other users.
  • Favorite (like) a tweet.
  • See a home timeline — the merged, recency-sorted tweets of everyone you follow.

Non-functional (how well):

  • Highly available — the system stays up; a brief stale timeline is acceptable.
  • Low latency — ~200 ms to generate a timeline.
  • Eventual consistency is fine — not seeing a tweet for a few seconds is OK (this is a deliberate CAP choice: availability over strict consistency).

Extended (note but defer): search, replies, trending topics, tagging, notifications, who-to-follow. We design the core and point to where the rest is covered.

2. Capacity estimate

The numbers aren't busywork — the read:write ratio is the architecture. Assume 1 billion total users, 200M daily active, 100M new tweets/day, average 200 follows each.

Traffic: the ratio that decides everything

  • Writes: 100M tweets / 86,400s ≈ 1,150 tweets/sec.
  • Reads: assume each of 200M users loads their timeline twice and views 5 other pages, 20 tweets each: 200M × (2+5) × 20 ≈ 28B views/day ≈ 325,000 reads/sec.
  • Read:write ratio ≈ 280:1. This single number is why every decision below favors reads.

Storage

  • Text: 140 chars × 2 bytes + ~30 bytes metadata ≈ 310 bytes/tweet. 100M × 310 ≈ 30 GB/day of text.
  • Media: assume every 5th tweet has a 200 KB photo, every 10th a 2 MB video: (100M/5 × 200KB) + (100M/10 × 2MB) ≈ 24 TB/day of media. Media dominates storage by ~800× — which is why it never lives in the tweet database.

Bandwidth

  • Ingress: 24 TB/day ≈ 290 MB/s in.
  • Egress (28B views/day, showing every photo but only every 3rd video watched): ~93 MB/s text + ~13 GB/s photos + ~22 GB/s video ≈ 35 GB/s out. Egress is ~120× ingress — the system is overwhelmingly about serving data, which is what pushes media to a CDN.
What the numbers tell usThree facts fall out and shape the whole design: (1) reads dominate writes 280:1 → pre-compute on write (fan-out); (2) media is ~800× the text volume → object store + CDN, never the tweet DB; (3) egress is ~120× ingress → the system is a read-serving machine.

3. System API

Define the contract before the internals. The core write endpoint:

tweet(api_dev_key, tweet_data, tweet_location,
      user_location, media_ids)
  → returns the URL of the created tweet, or an HTTP error
  • api_dev_key — identifies the caller; also used to rate-limit per quota.
  • tweet_data — the text (≤140 chars).
  • tweet_location, user_location — optional lat/long.
  • media_ids — IDs of photos/videos uploaded separately (media never travels inline through the tweet path — see the bandwidth numbers).

And the read endpoint that everything optimizes for:

getHomeTimeline(api_dev_key, user_id, count, max_id)
  → returns the next `count` tweets in the user's home timeline

4. High-level architecture

Given 1,150 writes/sec and 325K reads/sec, this is fundamentally a read-heavy system. The high-level shape:

  • Stateless application servers behind load balancers — scale horizontally with traffic; the LB sits at the client↔app edge (and again app↔cache and app↔DB).
  • A write path: post a tweet → assign an ID → persist to the tweet store → fan out into followers' timelines (the expensive step, done on write).
  • A read path: get home timeline → read the pre-computed timeline from a cache (a cheap lookup) → hydrate tweet bodies.
  • A separate media path: upload/serve photos & video via an object store + CDN, never the tweet DB.
  • Read replicas on the tweet store to absorb the 280:1 read load; the primary takes writes.
The shape in one sentenceStateless app servers behind load balancers, a write path that pre-computes timelines via fan-out, a read path that serves them from cache, and media offloaded to a CDN — everything arranged so the 325K/sec reads stay cheap.

5. Data model

Four core entities. The relationships are simple; the interesting choices are the indexes and how this gets sharded (Sub-problem 2).

SQL vs NoSQL here follows the same reasoning as Instagram: the relationships are simple and the access patterns are known, so the model itself isn't the hard part — scale is. That's why the weight of the design is in sharding, caching, and ID generation, which we take next. (How to make this choice in general: Choosing a Database.)

Three scenarios — the hard parts, as questions

The walkthrough above is the routine design. The interesting engineering is in four sub-problems where the obvious approach breaks. Decide your answer to these before reading on.

Scenario 1 — the celebrity posts Your write path pushes each new tweet into every follower's pre-computed timeline (fan-out on write). A celebrity with 100 million followers posts one tweet. How many writes does that single tweet trigger, and what happens to your write pipeline?
Scenario 2 — the hot shard You shard the tweet store by UserID so all of a user's tweets live on one server. One account goes viral and its tweets are read millions of times an hour. Which server feels it, and why doesn't adding more shards help that one account?
Scenario 3 — the 64-bit ID Two tweets are posted in the same millisecond on two different servers. You need every tweet to have a globally unique ID that also sorts by time (so timeline merges are cheap). How do you generate IDs that never collide and stay time-ordered, without a single central counter becoming a bottleneck?

Scenario 1 is the celebrity fan-out problem — the single most important idea here. Scenario 2 is the hot shard. Scenario 3 is ID generation (Snowflake). All three are sub-problems below.

6. The four hard sub-problems

This is where the design earns its keep. Each has a characteristic failure, shown nested beneath it.

Sub-problem 1 — Timeline generation (fan-out)

A home timeline is the merged recent tweets of everyone you follow. Two pure strategies at opposite ends of the read/write trade:

  • Fan-out on read (pull). Store each user's tweets with them; build a timeline by querying everyone they follow and merging. Cheap writes, but a read does hundreds of queries + a merge — far too slow at 325K reads/sec.
  • Fan-out on write (push). When a user posts, copy the tweet into every follower's pre-computed timeline. A read is then one cheap lookup. The right default because reads dominate.
The failure when you get this wrong — the celebrity fan-out This is Scenario 1. Fan-out on write means one tweet = one write per follower. A celebrity with 100M followers turns a single tweet into 100 million writes — write amplification that floods the pipeline and delays everyone's tweets.
How Twitter actually solves it — the hybrid. Twitter fans out on write for ordinary users but excepts high-follower celebrity accounts: their tweets are not pushed. When you load your timeline, the system merges your pre-computed timeline (normal follows) with a pull of the latest tweets from the few celebrities you follow. The expensive path runs only for the handful of accounts where push is ruinous. (At its peak, Justin Bieber's account alone was reported to be ~3% of all Twitter traffic — the celebrity case is load-bearing, not a corner case. This hybrid is the canonical answer in Kleppmann's Designing Data-Intensive Applications.)

Sub-problem 2 — Tweet storage & sharding

1,150 writes/sec and billions of reads means tweets must spread across many machines. Three sharding keys, each flawed:

Shard byHow it worksThe catch
UserIDAll of a user's tweets on one serverA hot/celebrity user overloads one server; uneven growth
TweetID (random)Each tweet on a random serverEven load, but a timeline read must query every shard and aggregate — higher latency
Creation timeRecent tweets togetherAll writes hit the newest shard (hotspot); old shards idle
The combined answerShard by TweetID, but make the TweetID encode the timestamp (Sub-problem 4). You get even write distribution and fast "find the latest" — the benefits of both, which is exactly what a Snowflake-style ID buys.
The failure when you get this wrong — the hot shard This is Scenario 2. Shard by UserID and one viral account concentrates all its read traffic on the single server that owns it — adding shards doesn't help, because that user still lives on one shard. It's the same hot-key problem from the caching lesson, and the fixes are the same family (cache the hot account, replicate it, or split it).

Sub-problem 3 — The read cache

Because reads dominate, you cache aggressively. Two observations shape it:

  • 80-20: ~20% of tweets generate ~80% of reads — cache the hot 20% per shard and you absorb most traffic.
  • Recency: ~80% of users only look at the last ~3 days. Caching just the last 3 days of tweets is ~100 GB — small enough to hold in memory and replicate for read scaling.

The timeline cache is a hash table keyed by OwnerID whose value is a capped list of recent tweets, newest at the head — a per-user LRU list.

The failure when you get this wrong — the cold-timeline storm If the timeline cache is cold or restarts, every read falls through to the database's expensive merge at once — the cache stampede, now aimed at your tweet store. Warm the cache and bound the fallback (single-flight, last-3-days replicas) so a cache blip doesn't take down the database.

Sub-problem 4 — Tweet ID generation

This is Scenario 3. You need IDs that are (a) globally unique across many servers and (b) roughly time-sortable, so merging timelines and finding "latest" is cheap — without a single counter everyone contends on.

The answer is a time-ordered 64-bit ID: high bits = timestamp (epoch ms), low bits = a per-node sequence + a node identifier. ~31 bits hold 50 years of seconds; ~17 bits hold a per-second sequence (130K IDs/sec/node). Because the timestamp is most significant, IDs sort chronologically; because each node owns its sequence and ID, no two nodes collide and there's no central bottleneck.

The failure when you get this wrong — collisions or lost order A naive per-server auto-increment collides across servers; a single central counter is a SPOF and a throughput ceiling. Put the timestamp in the low bits and your IDs stop sorting by time, making merges expensive. The structure (timestamp high, node+sequence low) is the whole trick — this is Snowflake.

Self-check — can you reconstruct the design?

Close the page and rebuild it — both the walkthrough and the hard parts.

  • The walkthrough: from the assumptions (200M DAU, 100M tweets/day, 200 follows), derive the write rate, read rate, and ratio — and say what each of the three big numbers (280:1, media 800×, egress 120×) forces in the design.
  • Scenario 1 (celebrity): Why does fan-out on write turn one celebrity tweet into 100M writes, and how does the hybrid fix it without slowing everyone's reads?
  • Scenario 2 (hot shard): Why does sharding by UserID concentrate a viral account on one server, why doesn't adding shards help, and what caching-lesson problem is this?
  • Scenario 3 (IDs): What goes in the high vs low bits of a Snowflake ID, why does that make merges cheap, and why is a central counter wrong?
  • Why shard by a time-encoded TweetID rather than UserID, random TweetID, or raw creation time?
  • Why does media never travel through the tweet database or the tweet API?

The shape to remember: walk the design (requirements → estimate → API → high-level → data model), then solve 4 hard parts — timeline generation, sharding, read cache, IDs — with the hybrid fan-out as the load-bearing insight.

Famous implementations

How real systems implement the pieces. Same facets — what it is, the key idea, the failure to watch.

Twitter's hybrid timeline

  • What it is. The production answer to Sub-problem 1: fan-out on write for ordinary users, fan-out on read for high-follower celebrities, merged at read time.
  • Key idea. Run the expensive path only where the cheap one fails. The vast majority of accounts are cheap to push; the tiny number of mega-accounts are pulled.
  • The failure to watch. The read-time merge adds latency for users who follow many celebrities — bounded because celebrity tweets are read by everyone and cache extremely well.

Redis-backed timelines

  • What it is. Each home timeline stored as a Redis list of tweet IDs, newest at the head, capped at a few hundred (nobody scrolls back 10,000 tweets).
  • Key idea. Store IDs, not full tweets — hydrate bodies from the tweet cache on read. The timeline is tiny; writes are O(1) LPUSH + LTRIM.
  • The failure to watch. Per-user lists are data to keep coherent and rebuild after a Redis loss — you need a way to regenerate a timeline from source.

Snowflake IDs

  • What it is. Twitter's open-sourced 64-bit ID: timestamp (high bits) + datacenter/worker ID + per-node sequence. The answer to Sub-problem 4.
  • Key idea. Time in the high bits makes IDs k-sortable; node ID + sequence in the low bits removes any central coordinator.
  • The failure to watch. Clock skew — if a node's clock moves backwards, IDs can collide or mis-order; production Snowflake refuses to issue IDs while the clock is rewound. (Full treatment in Lesson 58.)

Build it yourself

Designing is a skill you practice. Work each before opening the approach.

Redo the capacity estimate from scratch

From 200M DAU, 100M tweets/day, 200 follows: derive write rate, read rate, ratio, storage, and egress — without looking back. Then say which design decision each number forces.

Reveal the approach

Writes: 100M / 86,400 ≈ 1,150/s. Reads: 200M × (2+5) × 20 ≈ 28B/day ≈ 325K/s → ratio ~280:1 → fan-out on write. Storage: ~310 B/tweet → 30 GB/day text; media (every 5th photo 200KB, every 10th video 2MB) → 24 TB/day → object store + CDN. Egress ~35 GB/s vs ingress ~290 MB/s → ~120× → read-serving machine. The discipline: every number ends in a design consequence, never a bare figure.

Implement hybrid fan-out

Write the post-tweet logic: decide push vs pull per author, and the read-path merge. Where's the threshold?

Reveal the approach

On post: if the author's follower count is below a threshold (~10K–100K), enqueue a fan-out job that LPUSHes the tweet ID into each follower's capped timeline. If above (a celebrity), skip the push and mark them a "pull" author. On read: take the user's pre-computed timeline, then for the few celebrities they follow, pull those authors' latest tweets and merge-sort by the time-sortable tweet ID. The threshold trades write amplification against read-merge cost; celebrities are rare, so the merge stays small.

What is Twitter?

Twitter is an online social networking service where users post and read short 140-character messages called "tweets". Registered users can post and read tweets, but those who are not registered can only read them. Users access Twitter through their website interface, SMS or mobile app.

Requirements and Goals of the System

We will be designing a simpler version of Twitter with following requirements:

Functional Requirements

  1. Users should be able to post new tweets.
  2. A user should be able to follow other users.
  3. Users should be able to mark tweets favorite.
  4. The service should be able to create and display user’s timeline consisting of top tweets from all the people the user follows.
  5. Tweets can contain photos and videos.

Non-functional Requirements

  1. Our service needs to be highly available.
  2. Acceptable latency of the system is 200ms for timeline generation.
  3. Consistency can take a hit (in the interest of availability), if a user doesn’t see a tweet for a while, it should be fine.

Extended Requirements

  1. Searching tweets.
  2. Reply to a tweet.
  3. Trending topics – current hot topics/searches.
  4. Tagging other users.
  5. Tweet Notification.
  6. Who to follow? Suggestions?
  7. Moments.

Capacity Estimation and Constraints

Let’s assume we have one billion total users, with 200 million daily active users (DAU). Also, we have 100 million new tweets every day, and on average each user follows 200 people.

How many favorites per day? If on average each user favorites five tweets per day, we will have:

200M users * 5 favorites => 1B favorites

How many total tweet-views our system will generate? Let’s assume on average a user visits their timeline two times a day and visits five other people’s pages. On each page if a user sees 20 tweets, total tweet-views our system will generate:

200M DAU * ((2 + 5) * 20 tweets) => 28B/day

Storage Estimates Let’s say each tweet has 140 characters and we need two bytes to store a character without compression. Let’s assume we need 30 bytes to store metadata with each tweet (like ID, timestamp, user ID, etc.). Total storage we would need:

100M * (280 + 30) bytes => 30GB/day

What would be our storage needs for five years? How much storage we would need for users’ data, follows, favorites? We will leave this for exercise.

Not all tweets will have media, let’s assume that on average every fifth tweet has a photo and every tenth has a video. Let’s also assume on average a photo is 200KB and a video is 2MB. This will lead us to have 24TB of new media every day.

(100M/5 photos * 200KB) + (100M/10 videos * 2MB) ~= 24TB/day

Bandwidth Estimates Since total ingress is 24TB per day, this would translate into 290MB/sec.

Remember that we have 28B tweet views per day. We must show the photo of every tweet (if it has a photo), but let’s assume that the users watch every 3rd video they see in their timeline. So, total egress will be:

(28B * 280 bytes) / 86400s of text => 93MB/s
+ (28B/5 * 200KB ) / 86400s of photos => 13GB/s
+ (28B/10/3 * 2MB ) / 86400s of Videos => 22GB/s
Total ~= 35GB/s

System APIs

Once we've finalized the requirements, it's always a good idea to define the system APIs. This would explicitly state what is expected from the system.

We can have SOAP or REST APIs to expose the functionality of our service. Following could be the definition of the API for posting a new tweet:

tweet(api_dev_key, tweet_data, tweet_location, user_location, media_ids, maximum_results_to_return)

Parameters:
api_dev_key (string): The API developer key of a registered account. This will be used to, among other things, throttle users based on their allocated quota.
tweet_data (string): The text of the tweet, typically up to 140 characters.
tweet_location (string): Optional location (longitude, latitude) this Tweet refers to.
user_location (string): Optional location (longitude, latitude) of the user adding the tweet.
media_ids (number[]): Optional list of media_ids to be associated with the Tweet. (All the media photo, video, etc.) need to be uploaded separately.

Returns: (string)
A successful post will return the URL to access that tweet. Otherwise, an appropriate HTTP error is returned.

High Level System Design

We need a system that can efficiently store all the new tweets, 100M/86400s => 1150 tweets per second and read 28B/86400s => 325K tweets per second. It is clear from the requirements that this will be a read-heavy system.

At a high level, we need multiple application servers to serve all these requests with load balancers in front of them for traffic distributions. On the backend, we need an efficient database that can store all the new tweets and can support a huge number of reads. We would also need some file storage to store photos and videos.

Database Schema

We need to store data about users, their tweets, their favorite tweets, and people they follow.

For choosing between SQL and NoSQL databases to store the above schema, please see ‘Database schema’ under Designing Instagram.

Data Sharding

Since we have a huge number of new tweets every day and our read load is extremely high too, we need to distribute our data onto multiple machines such that we can read/write it efficiently. We have many options to shard our data; let’s go through them one by one:

Sharding based on UserID: We can try storing all the data of a user on one server. While storing, we can pass the UserID to our hash function that will map the user to a database server where we will store all of the user’s tweets, favorites, follows, etc. While querying for tweets/follows/favorites of a user, we can ask our hash function where can we find the data of a user and then read it from there. This approach has a couple of issues:

  1. What if a user becomes hot? There could be a lot of queries on the server holding the user. This high load will affect the performance of our service.
  2. Over time some users can end up storing a lot of tweets or have a lot of follows compared to others. Maintaining a uniform distribution of growing user’s data is quite difficult.

To recover from these situations either we have to repartition/redistribute our data or use consistent hashing.

Sharding based on TweetID: Our hash function will map each TweetID to a random server where we will store that Tweet. To search tweets, we have to query all servers, and each server will return a set of tweets. A centralized server will aggregate these results to return them to the user. Let’s look into timeline generation example, here are the number of steps our system has to perform to generate a user’s timeline:

  1. Our application (app) server will find all the people the user follows.
  2. App server will send the query to all database servers to find tweets from these people.
  3. Each database server will find the tweets for each user, sort them by recency and return the top tweets.
  4. App server will merge all the results and sort them again to return the top results to the user.

This approach solves the problem of hot users, but in contrast to sharding by UserID, we have to query all database partitions to find tweets of a user, which can result in higher latencies.

We can further improve our performance by introducing cache to store hot tweets in front of the database servers.

Sharding based on Tweet creation time: Storing tweets based on recency will give us the advantage of fetching all the top tweets quickly, and we only have to query a very small set of servers. But the problem here is that the traffic load will not be distributed, e.g., while writing, all new tweets will be going to one server, and the remaining servers will be sitting idle. Similarly while reading, the server holding latest data will have a very high load as compared to servers holding old data.

What if we can combine sharding by TweetID and Tweet creation time? If we don’t store tweet creation time separately and use TweetID to reflect that, we can get benefits of both the approaches. This way it will be quite quick to find latest Tweets. For this, we must make each TweetID universally unique in our system, and each TweetID should contain timestamp too.

We can use epoch time for this. Let’s say our TweetID will have two parts; the first part will be representing epoch seconds and the second part will be an auto-incrementing sequence. So, to make a new TweetID, we can take the current epoch time and append an auto-incrementing number to it. We can figure out shard number from this TweetID and store it there.

What could be the size of our TweetID? Let’s say our epoch time starts today, how many bits we would need to store the number of seconds for next 50 years?

86400 sec/day * 365 (days a year) * 50 (years) => 1.6B

We would need 31 bits to store this number. Since on average we are expecting 1150 new tweets per second, we can allocate 17 bits to store auto incremented sequence; this will make our TweetID 48 bits long. So, every second we can store (2^17 => 130K) new tweets. We can reset our auto incrementing sequence every second. For fault tolerance and better performance, we can have two database servers to generate auto-incrementing keys for us, one generating even numbered keys and the other generating odd numbered keys.

1483228800 000001
1483228800 000002
1483228800 000003
1483228800 000004
…

If we make our TweetID 64bits (8 bytes) long, we can easily store tweets for next 100 years and also store them for mili-seconds granularity.

Cache

We can introduce a cache for database servers to cache hot tweets and users. We can use an off-the-shelf solution like Memcache that can store the whole tweet objects. Application servers before hitting database can quickly check if the cache has desired tweets. Based on clients’ usage pattern we can determine how many cache servers we need.

Which cache replacement policy would best fit our needs? When the cache is full, and we want to replace a tweet with a newer/hotter tweet, how would we choose? Least Recently Used (LRU) can be a reasonable policy for our system. Under this policy, we discard the least recently viewed tweet first.

How can we have more intelligent cache? If we go with 80-20 rule, that is 20% of tweets are generating 80% of read traffic which means that certain tweets are so popular that majority of people read them. This dictates that we can try to cache 20% of daily read volume from each shard.

What if we cache the latest data? Our service can benefit from this approach. Let’s say if 80% of our users see tweets from past three days only; we can try to cache all the tweets from past three days. Let’s say we have dedicated cache servers that cache all the tweets from all users from past three days. As estimated above, we are getting 100 million new tweets or 30GB of new data every day (without photos and videos). If we want to store all the tweets from last three days, we would need less than 100GB of memory. This data can easily fit into one server, but we should replicate it onto multiple servers to distribute all the read traffic to reduce the load on cache servers. So whenever we are generating a user’s timeline, we can ask the cache servers if they have all the recent tweets for that user, if yes, we can simply return all the data from the cache. If we don’t have enough tweets in the cache, we have to query backend to fetch that data. On a similar design, we can try caching photos and videos from last three days.

Our cache would be like a hash table, where ‘key’ would be ‘OwnerID’ and ‘value’ would be a doubly linked list containing all the tweets from that user in past three days. Since we want to retrieve most recent data first, we can always insert new tweets at the head of the linked list, which means all the older tweets will be near the tail of the linked list. Therefore, we can remove tweets from the tail to make space for newer tweets.

Replication and Fault Tolerance

Since our system is read-heavy, we can have multiple secondary database servers for each DB partition. Secondary servers will be used for read traffic only. All writes will first go to the primary server and then will be replicated to secondary servers. This scheme will also give us fault tolerance, as whenever the primary server goes down, we can failover to a secondary server.

Load Balancing

We can add Load balancing layer at three places in our system 1) Between Clients and Application servers 2) Between Application servers and database replication servers and 3) Between Aggregation servers and Cache server. Initially, a simple Round Robin approach can be adopted; that distributes incoming requests equally among servers. This LB is simple to implement and does not introduce any overhead. Another benefit of this approach is if a server is dead, LB will take it out of the rotation and will stop sending any traffic to it. A problem with Round Robin LB is, it won’t take server load into consideration. If a server is overloaded or slow, the LB will not stop sending new requests to that server. To handle this, a more intelligent LB solution can be placed that periodically queries backend server about their load and adjusts traffic based on that.

Monitoring

Having the ability to monitor our systems is crucial. We should constantly collect data to get an instant insight into how our system is doing. We can collect following metrics/counters to get an understanding of the performance of our service:

  1. New tweets per day/second, what is the daily peak?
  2. Timeline delivery stats, how many tweets per day/second our service is delivering.
  3. Average latency that is seen by the user to refresh timeline.

By monitoring these counters, we will realize if we need more replication or load balancing or caching, etc.

Extended Requirements

How to serve feeds? Get all the latest tweets from the people someone follows and merge/sort them by time. Use pagination to fetch/show tweets. Only fetch top N tweets from all the people someone follows. This N will depend on the client’s Viewport, as on mobile we show fewer tweets compared to a Web client. We can also cache next top tweets to speed things up.

Alternately, we can pre-generate the feed to improve efficiency, for details please see ‘Ranking and timeline generation’ under Designing Instagram.

Retweet: With each Tweet object in the database, we can store the ID of original Tweet and not store any contents on this retweet object.

Trending Topics: We can cache most frequently occurring hashtags or searched queries in the last N seconds and keep updating them after every M seconds. We can rank trending topics based on the frequency of tweets or search queries or retweets or likes. We can give more weight to topics which are shown to more people.

Who to follow? How to give suggestions? This feature will improve user engagement. We can suggest friends of people someone follows. We can go two or three level down to find famous people for the suggestions. We can give preference to people with more followers.

As only a few suggestions can be made at any time, use Machine Learning (ML) to shuffle and re-prioritize. ML signals could include people with recently increased follow-ship, common followers if the other person is following this user, common location or interests, etc.

Moments: Get top news for different websites for past 1 or 2 hours, figure out related tweets, prioritize them, categorize them (news, support, financials, entertainment, etc.) using ML – supervised learning or Clustering. Then we can show these articles as trending topics in Moments.

Principal-level deep dive

Deeper analysis: trade-offs & failure modes

The lesson presents timeline generation as a merge-on-read across TweetID shards. That model is correct for a small service but inverts the actual economics at scale: Twitter is read-heavy by ~280:1 (the lesson's own numbers: 28B reads/day vs 100M writes/day). You almost never want to do the expensive work (gather + sort + rank) at read time when reads outnumber writes by two orders of magnitude. The real design pushes work to write time. This is the single most important insight the original lesson under-develops.

Fan-out: write vs read vs hybrid

Fan-out-on-write (push)When a user tweets, the tweet ID is pushed into a precomputed, per-follower home-timeline list. Reads become an O(1) cache lookup of an already-materialized list. This is what makes the 200ms timeline SLA achievable for the median user. Cost: a single tweet from a user with N followers triggers N list inserts.
Fan-out-on-read (pull)Store tweets once per author; at read time gather the latest tweets from everyone the viewer follows, merge-sort by time/rank. No write amplification, but read latency scales with follow count and requires scatter-gather across shards. This is the lesson's TweetID-sharding model.
The celebrity problemPure push breaks for high-fanout accounts. A celebrity with 100M followers (Cristiano Ronaldo, Elon Musk) would trigger 100M list writes per tweet. At even a modest tweet rate this is a write storm that creates a "thundering herd" of cache mutations and unbounded fan-out latency — a follower could see the tweet minutes after others. This is the canonical failure mode of naive push.

The production answer is hybrid fan-out: push for the vast majority of users, pull for a small set of celebrity accounts. At read time, a viewer's home timeline = (precomputed pushed list) merged with (a small pull-and-merge of the handful of celebrities they follow). Because a viewer follows only a few mega-accounts, the read-time merge stays cheap.

Optimization: don't push to the deadOf 1B accounts, only 200M are daily active. Pushing a tweet into the home-timeline list of a dormant account wastes memory and write IOPS. Production systems skip fan-out to inactive followers and lazily reconstruct (pull) their timeline if they return.

The home-timeline cache

The lesson's cache section caches tweet objects keyed by OwnerID. The more important cache is the home timeline itself — the per-user materialized list of tweet IDs the push path writes into. Critically, you cache IDs, not bodies: a timeline entry is a tweet ID + author ID (~16 bytes), and bodies are hydrated from the tweet cache on read. This keeps each user's list tiny and lets one tweet body be shared across millions of timelines instead of duplicated.

Bounded listsHome-timeline lists are capped (e.g. ~800 most-recent entries). You don't materialize an infinite history — older pages fall back to pull. This bounds memory: 200M active users × 800 IDs × ~16 bytes is in the low-terabyte range, fits across a Redis fleet, and bounds the cost of a celebrity de-fan-out should one occur.

Redis-backed timelines

Real implementations (Twitter's was Redis-based) store each home timeline as a Redis list/sorted-set. A tweet fan-out is an LPUSH + LTRIM (or ZADD + ZREMRANGEBYRANK) per follower; a timeline read is a single LRANGE/ZREVRANGE. Sorted sets let you rank by score (recency or an ML rank signal) rather than pure insertion order. Lists are replicated and sharded by user ID; because each list is independent, the cache shards near-perfectly. See 04 Caching for eviction and replication mechanics.

Tweet storage and sharding

The lesson lands on the right answer for IDs: a Snowflake-style 64-bit TweetID = timestamp ‖ machine/sequence bits. The key property it under-states is that time-prefixed IDs are roughly sortable, so "latest tweets" is a range scan and the ID itself encodes creation time (no separate timestamp index needed). Shard tweet bodies by TweetID (spreads write load, avoids the hot-user problem of UserID sharding) but keep the per-author tweet list separately so celebrity pulls are a single-shard read rather than a scatter-gather.

Failure mode: clock skew & sequence exhaustionSnowflake IDs assume monotonic clocks. A backwards NTP step on an ID-generator node can produce duplicate or out-of-order IDs; production generators refuse to issue IDs until the clock catches up. Sequence-bit exhaustion within a millisecond (more tweets/ms than the sequence space) must block-and-wait, not wrap.

How the social graph drives fan-out

Fan-out is fundamentally a graph traversal: "who follows this author?" must be answered for every tweet. This makes the follower-list store a first-class system, not an afterthought. It must answer "followers of X" (for push) and "followees of Y" (for pull) cheaply — typically an adjacency-list store sharded by user, often denormalized in both directions. The follower-count on this graph is also the routing signal that decides push vs pull per account.

Trending and search

Trending is a streaming heavy-hitters problem, not a batch one: you can't sort all hashtags by frequency at scale. Use approximate counting (Count-Min Sketch) over sliding windows to find top-K terms in near-real-time, weighted by velocity (rate of change) rather than raw volume so a term that's suddenly spiking beats an always-popular one. Search is a separate inverted-index pipeline. Both are deep topics — see 24 Twitter Search.
