Designing BookMyShow

Movie/event ticket booking. The defining challenge isn't scale of data — it's concurrency on a scarce resource: when a blockbuster opens, 50,000 people fight for the same few hundred seats in the same second. Each seat must go to exactly one person, with a fair "hold while you pay" window, and no double-booking ever.

The crux Sell each seat to exactly one buyer under massive concurrent contention — hold it during payment, never double-book. Temporary seat holds, the choice of concurrency control, and surviving a thundering herd on a hot show are the three hard parts.

1. Requirements

Functional: browse shows/seats; select seats and hold them while paying; confirm on payment, release on timeout; no seat ever sold twice. Non-functional: strong consistency on seat state (this is money + a physical seat — correctness over availability here), low latency, survive huge spikes for popular shows. Extended: waitlists, dynamic pricing, partial-group booking.

2. Capacity estimate

  • Contention, not volume, is the problem. Data is small (shows, seats), but a hit show concentrates tens of thousands of concurrent buyers on a few hundred seats in seconds — extreme write contention on a tiny hot set.
  • Read spike too: everyone refreshes the seat map at once; reads must not overwhelm the system or show stale "available" seats that are actually taken.
  • Correctness is non-negotiable: unlike a feed, a wrong answer here means a customer with a ticket and no seat — so this leans CP (consistency over availability) for the booking path.

3. System API

getSeatMap(show_id) → seats + availability
holdSeats(show_id, [seat_ids], user_id) → hold_id (TTL ~5-10 min) | conflict
confirmBooking(hold_id, payment) → ticket | failure (hold expired)
releaseHold(hold_id)   // explicit or on timeout

4. High-level architecture

  • Inventory/seat service — the source of truth for seat state (available / held / booked), with the concurrency control.
  • Hold store — temporary seat holds with TTL (often Redis).
  • Booking + payment — confirms a held seat on successful payment.
  • Read path — cached seat maps, refreshed as state changes.
The shape in one sentenceA seat-inventory service is the single source of truth; selecting a seat places a TTL'd hold, payment confirms it, timeout releases it — and the concurrency control guarantees exactly-one-owner throughout.

Three scenarios — the hard parts, as questions

Scenario 1 — pay now, or lose the seat? A user picks seat 14F and starts entering their card. Meanwhile someone else wants 14F too. You can't book it until payment succeeds, but you can't let two people pay for it either. What has to happen the moment they select it?
Scenario 2 — two clicks, same seat, same millisecond Two users both click "book 14F" at the same instant. Both requests read "14F available." What must the system guarantee, and what mechanism stops both from succeeding?
Scenario 3 — tickets drop at 10:00am A blockbuster's booking opens; 50,000 people hit the same show's seat map and booking endpoint in the same minute. What melts first, and how do you keep the system correct (and fair) under that herd?

Scenario 1 is the seat hold. Scenario 2 is concurrency control. Scenario 3 is the thundering herd.

6. The hard sub-problems

Sub-problem 1 — Holding a seat during payment

Payment takes seconds-to-minutes, but you can't leave a seat "available" during it (someone else grabs it) nor mark it "booked" before payment succeeds (it might fail). The answer is a third state: a temporary hold with a TTL.

The failure that happens when you get this wrong — the no-hold race This is Scenario 1. Without a hold, two users both see 14F available, both proceed to pay, and both succeed — one ends up without a seat. The fix: the instant a user selects 14F, place a hold (state → held, with held_by + a TTL of ~5–10 min, often in Redis). Others now see it unavailable. If payment confirms within the TTL → booked; if the user abandons or the TTL lapses → auto-released back to available. The TTL is essential: it prevents abandoned carts from locking seats forever (the same role as the lock TTL in Uber matching).

Sub-problem 2 — Concurrency control

Placing the hold itself must be atomic — two simultaneous holds on 14F can't both win. Two classic approaches:

  • Pessimistic locking — lock the seat row (DB SELECT … FOR UPDATE) or a distributed lock (Redis SET NX) before changing state. Simple and certain; holds a lock (contention) for the operation.
  • Optimistic concurrency — read the seat's version, then update only if the version still matches (compare-and-set). No lock held; on a version mismatch, the loser retries/fails. Great when conflicts are rare — but on a hot seat conflicts are common, so retries pile up.
The failure that happens when you get this wrong — double-booking This is Scenario 2, and it's the cardinal sin. A non-atomic read-then-write (read "available", write "held") lets two requests both pass the check — the same read-modify-write race as everywhere else, but here it means a sold-twice seat. You must make the seat claim atomic: either pessimistic (lock the seat, change state, release) or optimistic (conditional update on version — UPDATE … SET status='held' WHERE seat_id=? AND status='available', which succeeds for exactly one). For a highly-contended hot seat, a short pessimistic lock (or a Redis NX claim) avoids the retry storm optimistic control would cause; for low-contention seats, optimistic is cheaper. The DB's own ACID guarantees on that conditional update are what ultimately enforce exactly-one.

Sub-problem 3 — The opening-night herd

A hot show concentrates enormous traffic on one show's seats in seconds — on both the read (seat map) and write (booking) paths.

The failure that happens when you get this wrong — the hot-show meltdown This is Scenario 3. 50,000 buyers on a few hundred seats means the seat-inventory rows for that one show are a hot spot, and naive handling melts it. Defenses: a virtual waiting room / queue that admits buyers to the booking page at a controlled rate (so the inventory service sees manageable concurrency, not a wall); cache the seat map aggressively and push updates as seats change (don't let 50k clients poll the DB); keep the per-show inventory in a fast store (in-memory) that can do atomic claims quickly; and accept that most of the herd will find seats gone — fail them fast and clearly rather than letting them queue on locks. Fairness (first-come) comes from the admission queue, not from letting everyone race the database at once.

Famous implementations

Ticketmaster / BookMyShow-style

  • What it is. High-demand ticketing platforms; the canonical "virtual waiting room" pattern came from exactly this problem.
  • Key idea. Admission queue throttles entry to the booking flow; seat holds + atomic confirmation enforce exactly-one.
  • The failure to watch. Bots/scalpers and fairness — the waiting room is also an abuse-control surface.

DB conditional writes / distributed locks

  • What it is. The atomic-claim primitive: UPDATE … WHERE status='available' (optimistic) or row/Redis lock (pessimistic).
  • Key idea. Lean on ACID / atomic ops for exactly-one; don't hand-roll it. (Distributed Locking.)
  • The failure to watch. Lock TTL + failover correctness; choose pessimistic for hot rows to avoid optimistic retry storms.

Build it yourself

Implement an atomic seat hold

Two users try to hold seat 14F simultaneously. Implement the hold so exactly one wins, the seat auto-frees if payment never completes, and the loser is told immediately.

Reveal the approach

Make the hold a single atomic conditional write: UPDATE seats SET status='held', held_by=:user, hold_expiry=now()+10min WHERE seat_id='14F' AND status='available'. Exactly one of the two concurrent requests will match status='available' and update one row; the other matches zero rows and is told "seat taken" immediately — no double-hold, no lock held longer than the statement. (Equivalently, a Redis SET seat:14F NX PX 600000.) A background sweep (or the TTL itself) flips expired holds back to available, so an abandoned cart can't freeze the seat. On payment success, a second conditional write moves held-by-this-user → booked; on failure/timeout, release. The whole guarantee rests on that one atomic compare-and-set — the DB does the hard part.

Design-it-yourself

  • Group booking. Hold 5 adjacent seats atomically — all-or-nothing. What changes about the claim?
  • Virtual waiting room. Admit buyers at a controlled rate for a hot show, fairly. → Queues, Rate Limiter.

What is an online movie ticket booking system?

A movie ticket booking system provides its customers the ability to purchase theatre seats online. E-ticketing systems allow the customers to browse through movies currently being played and to book seats, anywhere and anytime.

Requirements and Goals of the System

Our ticket booking service should meet the following requirements:

Functional Requirements:

  1. Our ticket booking service should be able to list down different cities where its affiliate cinemas are located.
  2. Once the user selects the city, the service should display the movies released in that particular city.
  3. Once the user selects the movie, the service should display the cinemas running that movie and its available shows.
  4. The user should be able to select the show at a particular cinema and book their tickets.
  5. The service should be able to show the user the seating arrangement of the cinema hall. The user should be able to select multiple seats according to their preference.
  6. The user should be able to distinguish available seats from the booked ones.
  7. Users should be able to put a hold on the seats for five minutes before they make a payment to finalize the booking.
  8. The user should be able to wait if there is a chance that seats might become available – e.g. when holds by other users expire.
  9. Waiting customers should be serviced fairly in a first come first serve manner.

Non-Functional Requirements:

  1. The system would need to be highly concurrent. There will be multiple booking requests for the same seat at any particular point in time. The service should handle this gracefully and fairly.
  2. The core thing of the service is ticket booking which means financial transactions. This means that the system should be secure and the database ACID compliant.

Some Design Considerations

  1. For simplicity, let’s assume our service doesn’t require user authentication.
  2. The system will not handle partial ticket orders. Either user gets all the tickets they want, or they get nothing.
  3. Fairness is mandatory for the system.
  4. To stop system abuse, we can restrict users not to book more than ten seats.
  5. We can assume that traffic would spike on popular/much-awaited movie releases, and the seats fill up pretty fast. The system should be scalable, highly available to cope up with the surge in traffic.

Capacity Estimation

Traffic estimates: Let’s assume that our service has 3 billion page views per month and sells 10 million tickets a month.

Storage estimates: Let’s assume that we have 500 cities and on average each city has ten cinemas. If there are 2000 seats in each cinema and on average, there are two shows every day.

Let’s assume each seat booking needs 50 bytes (IDs, NumberOfSeats, ShowID, MovieID, SeatNumbers, SeatStatus, Timestamp, etc.) to store in the database. We would also need to store information about movies and cinemas, let’s assume it’ll take 50 bytes. So, to store all the data about all shows of all cinemas of all cities for a day:

500 cities * 10 cinemas * 2000 seats * 2 shows * (50+50) bytes = 2GB / day

To store five years of this data, we would need around 3.6TB.

Database Design

Here are a few observations about the data we are going to store:

  1. Each City can have multiple Cinemas.
  2. Each Cinema will have multiple halls.
  3. Each Movie will have many Shows, and each Show will have multiple Bookings.
  4. A user can have multiple bookings.

High Level Design

At a high level, our web servers will manage Users’ sessions, and application servers will handle all the ticket management, storing data in the databases and work with cache servers to process reservations.

Detailed Component Design

First, let’s try to build our service assuming if it is being served from a single server.

Ticket Booking Workflow: Following would be a typical ticket booking workflow:

  1. User searches for a movie.
  2. User selects a movie.
  3. User is shown the available shows of the movie.
  4. User selects a show.
  5. User selects the number of seats to be reserved.
  6. If the required number of seats are available, the user is shown a map of the theater to select seats. If not, the user is taken to ‘step 8’ below.
  7. Once the user selects the seat, the system will try to reserve those selected seats.
  8. If seats can’t be reserved, we have following options:
  • Show is full; the user is shown the error message.
  • The seats user wants to reserve are no longer available, but there are other seats available, so the user is taken back to the theater map to choose different seats.
  • There are no seats available to reserve, but all the seats are not booked yet as there are some seats that other users are holding in reservation pool and have not booked yet. The user will be taken to a waiting page where they can wait until required seats get freed from the reservation pool. This waiting could result in following options:
    • If the required number of seats become available, the user is taken to the theater map page where they can choose the seats.
    • While waiting if all seats get booked, or there are fewer seats in the reservation pool than the user intend to book, the user is shown the error message.
    • User cancels the waiting and is taken back to the movie search page.
    • At maximum, a user can wait one hour, after that user’s session gets expired and the user is taken back to the movie search page.
  1. If seats are reserved successfully, the user has five minutes to pay for the reservation. After payment, booking is marked complete. If the user is not able to pay within five minutes, all their reserved seats are freed to become available to other users.

How would the server keep track of all the active reservation that haven’t been booked yet? And how would the server keep track of all the waiting customers?
We need two daemon services, one to keep track of all active reservations and to remove any expired reservation from the system, let’s call it ActiveReservationsService. The other service would be keeping track of all the waiting user requests, and as soon as the required number of seats become available, it’ll notify the (the longest waiting) user to choose the seats, let’s call it WaitingUsersService.

a. ActiveReservationsService

We can keep all the reservations of a ‘show’ in memory in a Linked HashMap in addition to keeping all the data in the database. We would need a Linked HashMap so that we can jump to any reservation to remove it when the booking is complete. Also, since we will have expiry time associated with each reservation, the head of the linked HashTable will always point to the oldest reservation record, so that the reservation can be expired when the timeout is reached.

To store every reservation for every show, we can have a HashTable where the ‘key’ would be ‘ShowID’ and the ‘value’ would be the Linked HashMap containing ‘BookingID’ and creation ‘Timestamp’.

In the database, we’ll store the reservation in the ‘Booking’ table, and the expiry time will be in the Timestamp column. The ‘Status’ field will have a value of ‘Reserved (1)’ and as soon as a booking is complete, the system will update the ‘Status’ to ‘Booked (2)’ and remove the reservation record from the Linked HashMap of the relevant show. When the reservation is expired, we can either remove it from the Booking table or mark it ‘Expired (3)’ in addition to removing it from memory.

ActiveReservationsService will also work with the external financial service to process user payments. Whenever a booking is completed, or a reservation gets expired, WaitingUsersService will get a signal, so that any waiting customer can be served.

ActiveReservationsService keeping track of all active reservations

b. WaitingUsersService

Just like ActiveReservationsService, we can keep all the waiting users of a show in memory in a Linked HashMap. We need a Linked HashMap so that we can jump to any user to remove them from the HashMap when the user cancels their request. Also, since we are serving in a first-come-first-serve manner, the head of the Linked HashMap would always be pointing to the longest waiting user, so that whenever seats become available, we can serve users in a fair manner.

We will have a HashTable to store all the waiting users for every Show. The ‘key’ would be 'ShowID, and the ‘value’ would be a Linked HashMap containing ‘UserIDs’ and their wait-start-time.

Clients can use Long Polling for keeping themselve updated for their reservation status. Whenever seats become available, the server can use this request to notify the user.

Reservation Expiration
On the server, ActiveReservationsService keeps track of expiry (based on reservation time) of active reservations. As the client will be shown a timer (for the expiration time) which could be a little out of sync with the server, we can add a buffer of five seconds on the server to safeguard from a broken experience - such that – the client never times out after the server, preventing a successful purchase.

Concurrency

How to handle concurrency; such that no two users are able to book same seat? We can use transactions in SQL databases to avoid any clashes. For example, if we are using SQL server we can utilize Transaction Isolation Levels to lock the rows before we can update them. Here is the sample code:

SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

BEGIN TRANSACTION;

    -- Suppose we intend to reserve three seats (IDs: 54, 55, 56) for ShowID=99
    Select * From Show_Seat where ShowID=99 && ShowSeatID in (54, 55, 56) && Status=0 -- free

    -- if the number of rows returned by the above statement is three, we can update to
    -- return success otherwise return failure to the user.
    update Show_Seat ...
    update Booking ...

COMMIT TRANSACTION;

‘Serializable’ is the highest isolation level and guarantees safety from Dirty, Nonrepeatable and Phantoms reads. One thing to note here, within a transaction if we read rows we acquire a shared/range lock on them that blocks other transactions from updating them.

Once the above database transaction is successful, we can start tracking the reservation in ActiveReservationsService.

Fault Tolerance

What happens when ActiveReservationsService or WaitingUsersService crashes?
Whenever ActiveReservationsService crashes, we can read all the active reservations from the ‘Booking’ table. Remember that we keep the ‘Status’ column as ‘Reserved (1)’ until a reservation gets booked. Another option is to have master-slave configuration so that when master crashes slave can take over. Since we are not storing the waiting users in the database, so when WaitingUsersService crashes, we don’t have any means to recover that data unless we have a master-slave setup.

Similarly, we’ll have a master-slave setup for databases to make them fault tolerant.

Data Partitioning

Database partitioning: If we partition by ‘MovieID’ then all the Shows of a movie will be on a single server. For a very hot movie, this could cause a lot of load on that server. A better approach would be to partition based on ShowID, this way the load gets distributed among different servers.

ActiveReservationsService and WaitingUsersService partitioning: Our web servers will manage all the active users’ sessions and handle all the communication with the users. We can use Consistent Hashing to allocate application servers for both ActiveReservationsService and WaitingUsersService based upon the ‘ShowID’. This way, all reservations and waiting users of a particular show will be handled by a certain set of servers. Let’s assume for load balancing our Consistent Hashing allocates three servers for any Show, so whenever a reservation is expired, the server holding that reservation will do following things:

  1. Update database to remove the Booking (or mark it expired) and update the seats’ Status in ‘Show_Seats’ table.
  2. Remove the reservation from the Linked HashMap.
  3. Notify the user that their reservation has expired.
  4. Broadcast a message to all WaitingUsersService servers that are holding waiting users of that Show to figure out the longest waiting user. Consistent Hashing scheme will tell what servers are holding these users.
  5. Send a message to the WaitingUsersService server holding the longest waiting user to process their request if required seats have become available.

Whenever a reservation is successful, following things will happen:

  1. The server holding that reservation sends a message to all servers holding waiting users of that Show so that they can expire all those waiting users that need more seats than the available seats.
  2. Upon receiving above message, all servers holding the waiting users will query the database to find how many free seats are available now. Database cache would greatly help here to run this query only once.
  3. Expire all waiting users who want to reserve more seats than the available seats. For this, WaitingUsersService has to iterate through the Linked HashMap of all the waiting users.
Principal-level deep dive

Deeper analysis: trade-offs & failure modes

The entire system collapses to one hard problem: two users must never own the same seat, and this must hold under a burst of thousands of requests hitting the same few rows the instant a hot show opens. Everything else — search, catalogue, the seat map — is read-heavy and trivially cacheable. The booking path is the only place real money and real contention collide. Get the concurrency model wrong and you either double-sell (a refund, an angry customer, a chargeback) or you over-lock and the whole show grinds to a halt while one slow client holds a lock.

The core invariant A seat for a given ShowID is in exactly one of three states at any instant: FREE, HELD (by one session, with an expiry), or BOOKED (paid, permanent). Every transition must be atomic and observed by all readers. The bug to fear is the lost update / phantom: two transactions both read FREE, both write HELD.

Three ways to guarantee the invariant

StrategyHow it worksWinsCosts / failure mode
Pessimistic locking (the lesson's SERIALIZABLE approach)SELECT ... FOR UPDATE on the seat rows inside a txn; concurrent txns block until commit.Simple, correct, no retry logic. Fine at moderate QPS.Under a thundering herd the hot rows become a serialization point. Lock held across a slow app round-trip = head-of-line blocking. SERIALIZABLE can also deadlock or abort with serialization failures you must retry.
Optimistic concurrency (OCC)Read seat + a version; on write do UPDATE ... WHERE id=? AND version=?. Zero rows affected → someone else won, retry.No locks held during user think-time; great when conflicts are rare.On a hot show conflicts are the common case, so you get a retry storm — OCC degrades exactly when you need it most.

The seat-hold pattern in practice The hold is the only step that must be globally serialized, and it is cheap — a single conditional write. A common high-throughput implementation: a Redis key per seat, SET show:99:seat:54 <sessionId> NX PX 300000. NX makes it atomic (only one session can create the key); PX sets the 5-minute TTL so a hold self-expires even if the client vanishes. The relational Booking row is written on confirm, not on hold — so the DB never sees the contention. The lesson's in-memory Linked HashMap is the single-server version of exactly this idea; Redis (or a partitioned hold service) is the distributed version. See 62 Distributed Locking for why a naive Redis lock needs fencing tokens to survive a primary failover.

Thundering herd on a hot show

When tickets for a blockbuster (or a Taylor Swift tour — Ticketmaster's 2022 Eras Sale fielded ~3.5 billion requests and melted) open at 10:00:00, the load is not uniform: it is a vertical spike onto a few hundred rows. Defenses, layered:

Waiting room / virtual queue Don't let the herd reach the booking tier at all. Admit users through a token-issuing queue (a la Ticketmaster's "Smart Queue", or a Cloudflare Waiting Room). Each admitted user gets a signed token granting a time window; the booking service rejects anyone without a valid token. This converts an uncontrollable spike into a steady, capacity-matched drip and is the single most important scaling decision for this system.

  1. Front the seat-map reads with a cache + CDN. 99% of the herd is just looking. Serve the seat-availability snapshot from Redis/edge with a 1–2s TTL; accept slight staleness on reads, enforce truth only at hold-time.
  2. Shard the hold service by ShowID. (The lesson's consistent-hashing partitioning.) One show's frenzy can't starve another's. A single mega-show may still need its seats spread across shards or its own dedicated capacity.
  3. Fail fast and clearly. A losing hold attempt should return instantly ("seat just taken, pick another") rather than queue behind a lock — the lesson's three-way ReserveSeats response is exactly this.
  4. First-come-first-serve waiting list (the WaitingUsersService) absorbs users for seats that might free up when holds expire, served fairly via the head of the Linked HashMap.

Hold → pay → confirm: a saga, not a transaction

You cannot wrap "hold the seat" and "charge the card" in one ACID transaction — the payment lives in an external gateway (Stripe/Razorpay), takes seconds-to-minutes, and may involve a 3-D Secure redirect. This is a textbook distributed transaction handled as a saga with compensations rather than a 2PC:

StepForward actionCompensation if a later step fails
1. HoldFREE→HELD with 5-min TTLLet TTL expire (or release explicitly) → back to FREE
2. ChargeCall payment gateway with an idempotency keyRefund / void the authorization
3. ConfirmHELD→BOOKED, write Booking row, issue ticket(terminal) on confirm failure after a successful charge → auto-refund + alert
Idempotency is non-negotiable The dangerous window is between a successful charge and a successful confirm. A client retry, a duplicated message, or a gateway webhook delivered twice must not double-charge or double-book. Two keys do the work: (1) an idempotency key on the payment call so the gateway dedupes a retried charge; (2) a unique constraint (ShowID, SeatID) with Status=BOOKED in the DB so a duplicate confirm is rejected at the storage layer, not just in app code. Prefer authorize-on-hold + capture-on-confirm so an unpaid expiry never leaves a real charge. See 59 Payment System for exactly-once charge semantics and 32 Distributed Transactions for saga orchestration vs. choreography.

The "stranded HELD seat" failure mode If the hold lives only in an in-memory map (as in the base lesson) and that server crashes, the seat is logically held with no owner and no expiry to fire. Mitigations: persist the hold's expiry in the Booking row (the lesson's recovery path), use a TTL-backed store (Redis) as the source of truth so expiry is intrinsic, and run a reconciliation sweeper that releases HELD rows whose timestamp is past. The lesson's 5-second server-side buffer (client never times out after the server) is a smaller version of the same defensive instinct — never let two parties disagree about whether a hold is still alive.
