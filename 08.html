Queues

A queue decouples a producer from a consumer: the producer drops work and moves on, the consumer takes it when ready. You buy elasticity, resilience, and smoothing of spikes — and you pay in eventual consistency, ordering headaches, and a new failure mode where the queue silently grows until something dies.

The trade-off Decoupling and spike-absorption in exchange for async complexity and a backlog that can grow unbounded. Putting a buffer between producer and consumer lets each scale and fail independently and absorbs bursts. In return the work is now asynchronous (eventual, out-of-order, maybe-duplicated), and an unbounded backlog is a real way to take the system down.

What a queue is

A queue sits between a producer (who creates work) and a consumer (who does it). The producer appends a message and returns immediately; the consumer pulls messages and processes them at its own pace. That single buffer buys a lot:

  • Decoupling — producer and consumer scale, deploy, and fail independently.
  • Spike absorption — a burst of work queues up instead of overwhelming the consumer.
  • Resilience — if the consumer is down, work waits rather than being lost.
  • Load leveling — consumers process at a steady rate regardless of bursty input.
The one-line frameA queue doesn't make work disappear — it delays it. That's its power (smooth the spike) and its danger: if consumers can't keep up on average, the backlog grows without bound. A queue absorbs bursts, not sustained overload.

Three scenarios — decide before you read

Scenario 1 — the consumer falls behind forever Your service runs 200 workers pulling from an in-memory queue, writing to a database. The database slows down. Producers keep sending at the same rate. What happens to the queue over the next ten minutes, and how does this end?
Scenario 2 — the payment processed twice A consumer pulls a "charge card" message, charges the card, then crashes before acknowledging it. The queue redelivers the message to another worker. What just happened to the customer, and how do you prevent it without a perfect "exactly-once" guarantee that doesn't really exist?
Scenario 3 — the events arrive out of order You emit "account created" then "account deleted." With multiple consumers/partitions, the delete is processed before the create. Why did ordering break, and what does guaranteeing order cost you?

Scenario 1 is the unbounded backlog. Scenario 2 is delivery semantics (and idempotency). Scenario 3 is ordering.

A queue is defined by four decisions

Decision 1 — Queue or log? (delete-on-ack vs retain-and-replay)

Two fundamentally different models share the word "queue":

  • Broker / queue (delete-on-ack) — a message is delivered to one consumer and deleted once acknowledged. Work-distribution semantics; once consumed, it's gone. (SQS, RabbitMQ.)
  • Commit log (retain & replay) — messages are appended to an ordered, durable log and retained; consumers track their own offset and can replay from any point. Many independent consumers can read the same stream. (Kafka, Kinesis — see Kafka & Log Messaging.)
The failure that happens when you get this wrong — you can't replay Pick a delete-on-ack broker for an event stream that a new consumer (a fresh analytics job, a rebuilt cache) needs to read from the beginning, and you're stuck — the data is gone the moment the first consumer acked it. If you need multiple independent readers, replay after a bug, or to re-derive state, you need a log, not a queue. Conversely, using a log for simple work-distribution adds offset-management overhead you didn't need.

Decision 2 — Delivery semantics

Three guarantees, and one of them is mostly a myth:

  • At-most-once — deliver and don't retry; messages can be lost. Rarely what you want.
  • At-least-once — retry until acked; messages are never lost but can be duplicated (the default for most queues).
  • Exactly-once — the holy grail, and largely a misnomer: true end-to-end exactly-once delivery is impossible across a network. What systems actually provide is exactly-once processing via at-least-once delivery + idempotent consumers (or transactional offset commits, as Kafka does within its boundary).
The failure that happens when you get this wrong — duplicate processing This is Scenario 2. Under at-least-once (the norm), a consumer that does work then crashes before acking will see the message redelivered — so the card gets charged twice. You can't wish this away with an "exactly-once" checkbox. The real fix is idempotency: give each message a unique ID and make processing safe to repeat (record "charge 12345 done"; on redelivery, see it's done and skip). Design every at-least-once consumer to tolerate duplicates.

Decision 3 — Ordering

Strict global ordering and high throughput are in tension. A single consumer preserves order but can't scale; multiple consumers (or partitions) scale but interleave. The usual compromise is per-key ordering: messages with the same key (e.g. same account) go to the same partition and stay ordered, while different keys process in parallel.

The failure that happens when you get this wrong — out-of-order effects This is Scenario 3. Spread related events across partitions/consumers and "delete" can be processed before "create" — corrupting state. Global ordering would fix it but forces a single consumer (no parallelism). The pragmatic answer: partition by the entity key so all events for one account are ordered, and let unrelated accounts run concurrently. Guaranteeing order always costs throughput — buy only as much ordering as the domain needs.

Decision 4 — What happens under sustained overload?

A queue smooths bursts, but if consumers can't keep up on average, the backlog grows. You must decide what happens then — before it happens.

The failure that happens when you get this wrong — the unbounded backlog (OOM) This is Scenario 1. An unbounded queue absorbs backpressure instead of enforcing it: producers keep sending, the queue grows, and either memory exhausts (an in-memory queue or a RabbitMQ with no x-max-length will accept messages until the broker OOMs and crashes) or latency balloons past usefulness. A documented pattern: 200 workers on an unbounded queue, the database slows, tasks pile up, retries amplify the load, and 503s hit the gateway within ~12 minutes. The fixes:
  • Bound the queue and apply backpressure — reject or slow producers when full (a full queue should push back, not swallow).
  • Consumer-lag-driven autoscaling — scale consumers on queue depth / message age; if lag crosses a threshold, shed or throttle ingress.
  • Dead-letter queue (DLQ) — route messages that repeatedly fail (poison messages) aside so one bad message doesn't block the line.
  • Retry with backoff + jitter — so retries don't synchronize into a storm.

The dual-write trapWriting to your database and publishing to a queue in one operation isn't atomic — a crash between them loses the event or emits a phantom. The fix is the outbox pattern: write the event to an outbox table in the same DB transaction as the data, then a separate process (often CDC) ships outbox rows to the queue. (More in Event-Driven Architecture.)

Self-check — can you reconstruct the three scenarios?

  • Scenario 1 (backlog): Why does an unbounded queue end in OOM or runaway latency, and what are the four guardrails (bound+backpressure, lag autoscale, DLQ, backoff+jitter)?
  • Scenario 2 (duplicates): Why does at-least-once cause double-processing, why isn't "exactly-once delivery" real, and how does idempotency give you exactly-once processing?
  • Scenario 3 (ordering): Why do multiple consumers break ordering, and why does partition-by-key beat global ordering?
  • Queue vs log — when do you specifically need retain-and-replay?
  • What is the dual-write problem and how does the outbox pattern solve it?

The shape to remember: a queue is 4 decisions — queue vs log, delivery semantics, ordering, overload behavior — with failures: can't-replay, duplicates, disorder, unbounded backlog.

Famous implementations

Amazon SQS

  • Model. Managed broker (delete-on-ack); standard (at-least-once, best-effort order) or FIFO (ordered, dedup window).
  • Key idea. Fully managed, infinite scale, zero ops; visibility timeout redelivers if not acked.
  • The failure to watch. At-least-once means design idempotent consumers; no replay after delete (use a DLQ for failures).

RabbitMQ

  • Model. Broker with rich routing (exchanges, bindings); delete-on-ack.
  • Key idea. Flexible routing topologies and per-message control; great for complex job distribution.
  • The failure to watch. No x-max-length = the broker accepts until it OOMs — always bound queues and apply backpressure.

Kafka

  • Model. Commit log (retain & replay); partitioned, consumers track offsets.
  • Key idea. Many independent consumers, replay from any offset, per-partition ordering, huge throughput — the backbone of event-driven systems.
  • The failure to watch. Ordering is only per-partition; a hot/badly-chosen partition key skews load. (Deep dive: Lesson 39.)

Build it yourself

Make a consumer idempotent

Your queue is at-least-once, so any message may arrive twice. Make "charge this card" safe to process repeatedly. What do you store, and where's the race?

Reveal the approach

Give each message a unique idempotency key (the message ID, or a business key like order_id+attempt). Before doing the side-effect, atomically record "processing key K" — e.g. an INSERT into a processed-keys table with a unique constraint, in the same transaction as the effect (or a conditional write). If the key already exists, skip. The race: checking "have I seen K?" and doing the work must be atomic, or two concurrent redeliveries both pass the check — so rely on a unique constraint / conditional write, not a read-then-write. This turns at-least-once delivery into exactly-once processing.

Design backpressure for a slow consumer

The database behind your consumers slows down. Design the system so the queue doesn't grow without bound and the whole thing doesn't fall over.

Reveal the approach

Bound the queue (max length). When it fills, producers get a clear signal — reject with a retryable error, or block/slow them — so pressure propagates back to the source instead of accumulating. Export per-partition lag and message age as metrics; autoscale consumers on lag, and when lag crosses a ceiling, shed or throttle ingress at the gateway (e.g. reduce accept rate by 30%). Add a DLQ for poison messages and backoff+jitter on retries so they don't synchronize. The principle: a queue must enforce backpressure, not silently absorb it.

Design-it-yourself

  • Outbox pattern. Emit an event exactly when (and only when) a DB write commits, with no dual-write hole. → Event-Driven Architecture, CDC.
  • Build a distributed queue. Design SQS from scratch — visibility timeout, at-least-once, DLQ. → Distributed Message Queue.
