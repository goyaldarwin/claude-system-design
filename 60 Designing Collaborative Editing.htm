Designing Collaborative Editing (Google Docs)

How dozens of people type into the same document at the same time and converge on identical, intention-preserving state — the canonical hard problem of conflict resolution at low latency.

Requirements & scope

The problem: "design Google Docs." The trap is jumping to OT or CRDT vocabulary in the first minute. A principal scopes first, because the answer changes dramatically with the requirements.

Functional

  • Real-time multi-user editing: many users type into the same document; each sees others' edits within ~100–200 ms.
  • Conflict-free convergence: after all edits propagate, every replica shows byte-identical content. No edit silently lost, no character duplicated.
  • Intention preservation: if Alice bolds "hello" while Bob inserts "X" before it, both intentions survive — not just "some" consistent state.
  • Offline support: edit on a plane, reconnect, and have local changes merge cleanly.
  • Presence / awareness: live cursors, selections, "who is here" avatars.
  • History & undo: per-user undo (undo my last edit, not the last global edit) and a revision timeline.

Non-functional & back-of-envelope

  • Latency budget: keystroke echo locally must be 0 ms (optimistic local apply); remote propagation P99 < 200 ms.
  • Concurrency: a single doc rarely has >50 simultaneous active editors; the system has millions of docs, mostly idle. This shapes sharding: partition by document, one coordinator per active doc.
  • Edit volume: a fast typist emits ~8–10 ops/sec. 50 editors × 10 ops/sec = 500 ops/sec per hot doc — tiny per doc, but the fan-out and the persistence log dominate.
  • Durability: an acknowledged edit must never be lost. This pushes us toward an append-only operation log before we ack the client.
Key insightThe defining constraint is the combination of optimistic local editing (zero perceived latency) with guaranteed eventual convergence. You cannot lock the document and wait for a round trip per keystroke — that is what makes naive last-write-wins or pessimistic locking unacceptable, and what forces OT or CRDTs.

The core problem: concurrent conflicting edits

Forget the network for a second. The whole difficulty is one scenario. The document is "abc". Two users act on it concurrently, each against their own local copy:

Start:  "abc"
Alice:  insert("X", pos=0)   ->  "Xabc"
Bob:    delete(pos=2)        ->  "ab"   (deleting 'c')

Each user applies their own op locally with no delay, then ships it to the other. If the server (or peer) naively replays the raw ops in the order received, you get garbage. Bob's delete(pos=2) was authored against "abc" where index 2 was 'c'. But after Alice's insert shifts everything right by one, index 2 in "Xabc" is 'b'. Applying Bob's op verbatim deletes the wrong character, and the two replicas diverge permanently.

This is the heart of it: an operation's positional intent is relative to the state it was authored against. When concurrent operations interleave, positions become stale. Two families of algorithms solve this differently:

  • Operational Transformation (OT) — rewrite each operation so it applies correctly against the state the receiver actually has.
  • CRDTs — redesign the data model so operations commute and no rewriting is ever needed.

Operational Transformation (OT)

OT is the technique that originally powered Google Docs, Etherpad, and Apache Wave. The idea: maintain a function transform(opA, opB) that takes two concurrent operations and rewrites opA so it can be applied after opB while preserving both intentions.

The transform function

For our example, the server has already applied Alice's insert("X", 0). Bob's delete(2) arrives, authored against the pre-insert state. We transform it:

transform(delete(pos=2), insert("X", pos=0)):
    # an insert at or before delete's position shifts it right
    if insert.pos <= delete.pos:
        return delete(pos = 2 + len("X"))   # -> delete(pos=3)
    else:
        return delete(pos = 2)              # unaffected

Now delete(pos=3) applied to "Xabc" removes 'c' — exactly Bob's intent. Both replicas converge to "Xab". The transform must satisfy the TP1 property: apply(apply(s, a), transform(b, a)) == apply(apply(s, b), transform(a, b)) — transforming in either order yields the same state. Systems that allow peer-to-peer reordering also need TP2, which is notoriously hard to get right and was a major reason Google Wave's OT engine was so complex.

Why OT effectively needs a central server

OT in its practical form (Jupiter algorithm, used by Google Docs) sidesteps TP2 by funneling everything through a single server that imposes a total order. Each client keeps a revision number; the server is the authority on "what revision are we at." The protocol per client looks like this:

client state: { revision, pending[], buffer[] }

on local edit op:
    apply op locally (optimistic, 0 latency)
    if no op is in-flight:  send op tagged with current revision; move to pending
    else:                   queue op in buffer

on server ACK (revision r):
    revision = r
    flush next buffered op as pending

on remote op from server (authored at revision r):
    op' = op
    for each o in (pending + buffer):     # transform against my unacked ops
        op', o = transform(op', o)
    apply op' locally

The server does the symmetric transform against every op it has accepted since the client's stated revision. Because the server linearizes all ops, every client transforms against the same canonical history and convergence is guaranteed without TP2. The price: OT does not work well for true peer-to-peer or fully decentralized topologies — you need that authoritative serialization point. (This is closely related to why ordering is so fundamental — see Time, Clocks & Ordering.)

PitfallOT's correctness lives entirely in the transform functions, and the matrix grows with the number of operation types (insert, delete, retain/format, attribute changes). Each pair must satisfy TP1, and bugs manifest as rare, data-corrupting divergence that is brutal to reproduce. Treat the transform suite as the highest-risk code in the system and fuzz it relentlessly with randomized concurrent op sequences.

CRDTs for text

CRDTs (Conflict-free Replicated Data Types) attack the same problem from the data-model side: design the structure so that concurrent operations commute by construction. Apply them in any order and you converge — no transform function, no central serializer required. See CRDTs for the general theory; here we focus on the sequence variant for text.

The key idea: stable, dense identifiers instead of integer positions

The reason OT needs transforms is that integer positions are relative. CRDT text types replace them with globally unique, immutable, totally-orderable identifiers for each character. A character is never "position 2"; it is "the element with id (0.5, siteA, 7), which sorts between these two neighbors." Insertion means "create an id strictly between my left and right neighbor's ids." Deletion is a tombstone (mark removed, don't physically delete) so concurrent inserts referencing it still resolve.

RGA and the common sequence CRDTs

  • RGA (Replicated Growable Array): each element holds a unique (timestamp, siteId) and a reference to the element it was inserted after. Concurrent inserts at the same spot are ordered deterministically by timestamp, breaking ties by siteId. Widely used because operations are compact and it handles the insert-after model cleanly.
  • LSEQ / Logoot / Treedoc: use fractional or path-based positions — an id is a list of digits forming a dense, infinitely divisible order between neighbors.
  • Yjs (YATA) and Automerge: the dominant production libraries. Yjs uses a doubly-linked-list CRDT with an integer-pair clock and aggressive run-length encoding of contiguous inserts; Automerge is columnar-encoded and history-complete.
// RGA-style insert: place "X" between left and right neighbors
insert(value, leftId):
    id = (lamportClock++, mySiteId)        // unique, monotone
    node = { id, value, after: leftId, deleted: false }
    // among elements sharing the same 'after', order by id descending
    linkInOrder(node)
    broadcast(node)                        // commutes: any apply order converges

delete(targetId):
    nodes[targetId].deleted = true         // tombstone, not physical removal
    broadcast(tombstone(targetId))
Key insightBecause every element id is globally unique and the merge rule is deterministic, CRDT ops are idempotent and commutative. That means you can deliver them out of order, duplicate them, gossip them peer-to-peer, or apply a batch after a week offline — convergence still holds. This is what makes CRDTs the natural fit for true offline-first and decentralized editing (see Anti-Entropy & Gossip).
PitfallTombstones never go away on their own. A heavily-edited document accumulates deleted-but-retained elements, bloating memory and the on-wire payload. Production CRDTs need a garbage-collection / compaction strategy — safe only once you can prove no concurrent op could still reference a tombstone (e.g., once a causal stability watermark passes it). Underestimating tombstone growth is the classic CRDT scaling mistake.

OT vs CRDT trade-offs

DimensionOperational TransformationCRDT (sequence)
TopologyNeeds an authoritative server (Jupiter) to serializeWorks peer-to-peer, server-relay, or hybrid
Correctness burdenO(n²) transform-function matrix; TP1/TP2 proofsCorrectness proven once in the data type
Per-character overheadLow — ops are tiny, state is plain textHigher — unique id + metadata per element, plus tombstones
Offline / long divergencePainful — must transform against large op historyNatural — merge any time, any order
Memory footprintJust the document stringDocument + ids + tombstones (needs GC)
Maturity in practiceGoogle Docs, Etherpad, ShareDBFigma, Yjs ecosystem, Automerge, Apple Notes
Trade-offThe decision is really about where complexity lives and what topology you need. If you already have a strong central server and want minimal per-op bytes — OT (Google's bet for Docs in 2010). If you want offline-first, peer-to-peer, or to avoid an ever-growing transform matrix and are willing to pay metadata + GC cost — CRDTs (the modern default for new systems like Figma multiplayer and the Yjs ecosystem). Neither is "more consistent"; both target strong eventual consistency — see Consistency Models & PACELC.
What strong engineers doThey don't religiously pick one. They say: "I'd use a server-authoritative CRDT (Yjs-style) — it gives me the offline and out-of-order resilience of CRDTs while still routing through a relay server I control for auth, persistence, and presence fan-out. I get the best of both rather than forcing a pure topology."

Real-time transport & presence

Transport choice

Editing is bidirectional and low-latency, so the transport must push server->client without polling. The right primitive is WebSockets — full-duplex, persistent, low per-message overhead. SSE is one-directional (server->client only) so it can't carry the client's keystrokes; long-polling adds a round trip per message. See Long-Polling vs WebSockets vs SSE for the full comparison. Production reality: keep a WebSocket per active editor, with an HTTP fallback for hostile proxies, and reconnect-with-backoff carrying the client's last-known revision/vector so the server can replay the gap.

Presence / awareness

Cursor positions, selections, and "who's online" are ephemeral, high-frequency, and disposable — you do not want them in the durable op log. Model presence as a separate channel with last-write-wins-per-user semantics and a TTL (heartbeat every few seconds; expire a cursor after ~30 s of silence). Yjs ships this as a dedicated "awareness" protocol precisely so cursor spam never touches the document CRDT. Cursor positions are expressed as relative positions (anchored to a character id) so that a remote insert above your cursor moves it correctly instead of leaving it pointing at a stale integer offset.

PitfallBroadcasting raw cursor moves to all N peers is O(N²) chatter on a busy doc. Throttle/debounce presence to ~10–20 Hz, batch into a single periodic frame, and never persist it. Conflating presence durability with edit durability is a common design smell.

Persistence, snapshots & scaling

The operation log is the source of truth

Before the server ACKs an edit, it must durably append the op to a per-document append-only log (think a partitioned commit log, conceptually like Kafka keyed by docId). Reconstructing a document = replay the log from revision 0. This gives you the revision history and timeline "for free," and is the durability boundary: ACK only after the log write is fsynced/replicated.

Snapshotting

Replaying millions of ops to open a doc is too slow, so periodically write a snapshot of materialized state (the plain text for OT, or the compacted CRDT structure) plus the revision it corresponds to. Open = load latest snapshot + replay only the tail of ops after it. Snapshots also enable CRDT tombstone GC (a snapshot past a causal-stability watermark can drop tombstones) and OT history truncation.

Scaling to many documents

  • Partition by document. Each active doc has exactly one authoritative coordinator (a single process/actor) that owns serialization, presence fan-out, and write batching. This single-owner-per-doc model is what makes OT's total order trivial and prevents two servers from racing on the same doc.
  • Route via consistent hashing. Map docId → coordinator node with consistent hashing so the same doc always lands on the same owner; a thin gateway holds the WebSocket and forwards to the owner.
  • Handle owner failover. The hard part. If a coordinator dies, a new one must take ownership without two owners existing simultaneously (split-brain). Use a lease/lock from a coordination service (see Distributed Locking Service and Leader Election). The new owner rebuilds state from snapshot + log tail; in-flight clients reconnect and replay from their last revision.
  • Cold docs cost nothing. Idle docs hold no coordinator — they are just a snapshot + log at rest. Spin up a coordinator lazily on first connect.

End-to-end architecture

             WebSocket (edits + presence)
 [Browser] <-----------------------------------> [Edge Gateway]
   |  local CRDT/OT state, optimistic apply           |
   |                                                   v  (route by docId, consistent hash)
   |                                          [Doc Coordinator (1 per active doc)]
   |                                            - serializes / merges ops
   |                                            - transforms (OT) or merges (CRDT)
   |                                            - presence fan-out (ephemeral)
   |                                            - batches writes
   |                                            |              |
   |                                            v              v
   |                                   [Op Log: append-only]  [Snapshot store]
   |                                    (durable, replicated)  (object storage)

Request path for one keystroke: apply locally (0 ms) → send op + revision over WebSocket → gateway forwards to the doc's coordinator → coordinator transforms/merges, appends to op log, fans out the canonical op to all other connected clients and ACKs the author. Each receiving client transforms/merges the incoming op against its own unacked ops and applies it.

What strong engineers doThey separate the three planes explicitly: the edit plane (durable, ordered, converging), the presence plane (ephemeral, lossy, TTL'd), and the transport plane (WebSocket with reconnect/replay). Muddling them — e.g., persisting cursors, or routing edits without a single per-doc owner — is what produces divergence and split-brain in real systems.

Principal-level mastery check

What strong engineers doThey scope before naming algorithms, reproduce the concurrent-edit divergence with a concrete 3-character example to prove they understand why the problem is hard, then justify OT vs CRDT against the actual requirements (topology, offline, team familiarity) rather than reciting a preference. They explicitly call out the single-owner-per-doc invariant and how failover preserves it, treat presence as a separate ephemeral plane, and name the durability boundary (ACK only after the op log persists). They acknowledge the unglamorous risks — transform-matrix correctness, tombstone GC, cursor chatter — instead of hand-waving "and then it just converges."
  • Walk me through two concurrent ops — an insert and a delete at overlapping positions — and show exactly how OT transforms them and how a CRDT avoids the transform entirely. Where does the determinism come from in each?
  • You picked CRDTs. Tombstones grow without bound. When is it provably safe to garbage-collect one, and how do you compute that watermark across clients that may be offline for days?
  • A document coordinator crashes mid-edit while 40 people are connected. Walk me through failover without ever allowing two owners. What's your fencing mechanism and how do reconnecting clients avoid losing or duplicating their in-flight ops?
  • How do you implement per-user undo (undo my last edit, not the global last edit) on top of an OT or CRDT log? Why is this harder than it sounds?
  • Why can't you use SSE here, and what specifically breaks if you tried last-write-wins per field instead of OT/CRDT? Tie it to the consistency model you're actually guaranteeing.
  • How does cursor/presence propagation differ from edit propagation in durability, ordering, and fan-out cost — and what goes wrong if you treat them the same way?
