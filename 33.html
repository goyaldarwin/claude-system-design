Time, Clocks & Ordering

"What happened first?" sounds trivial — until the two events happened on different machines whose wall clocks disagree by milliseconds and can jump backwards. Physical time can't order distributed events reliably. So we build logical clocks that capture causality instead, and only the richest engineering effort (atomic clocks + GPS) earns back real-time ordering.

The crux A reliable order for events across machines — in exchange for either giving up real wall-clock time (logical clocks) or buying expensive bounded-error time (TrueTime). You cannot trust physical timestamps to order distributed events. Logical clocks capture "happened-before" cheaply but lose real time; TrueTime restores real-time order at the cost of GPS/atomic hardware and a deliberate wait.

Why ordering is the real problem

You almost never need to know the absolute time an event happened. What you need is order: did this write come before that one? Did the comment happen after the post it replies to? Distributed systems make ordering hard because the only obvious tool — each machine's wall clock — is unreliable: clocks drift, are corrected by NTP in steps that can move time backwards, and never agree to better than a few milliseconds across a fleet.

So we separate two things a "clock" can give you:

  • Causal order — "A could have influenced B." Captured by logical clocks (Lamport, vector) without any reference to wall time.
  • Real-time order — "A finished before B started, by the wall." Only recoverable with tightly-bounded physical clocks (TrueTime) or a coordination round-trip.
The one-line frameOrdering trades real-time meaning for cheap causality, or money for restored real time. Logical clocks order causally-related events for free but tell you nothing about wall-clock order of concurrent events. TrueTime buys back real-time order with atomic/GPS hardware and a deliberate "commit-wait." Pick what your correctness actually needs.

Three scenarios — decide before you read

Don't scroll past these. Decide first. If you know all three cold, skip to choosing a clock.

Scenario 1 — the leap second, 2012 & 2016 (real incidents) On 30 June 2012 a leap second was inserted (23:59:60 UTC). Reddit, Mozilla, and others went down; Linux hosts spun their CPUs to 100% trying to reconcile a clock that didn't advance normally. In 2016, Cloudflare's DNS software computed the gap between two timestamps, got a negative number — because time had effectively gone backwards — and panicked, breaking 0.2% of DNS queries. What core assumption did all this code make, and why is it false in any real system?
Scenario 2 — last-write-wins eats a write Two replicas accept concurrent writes to the same key. Conflict resolution is "keep the one with the larger timestamp." Replica A's clock is 30 ms ahead of B's. A's write happened causally later in reality but carries a smaller timestamp because... actually it's the other way around — the point is you can't tell. How can last-write-wins silently discard the write that should have won, and what kind of clock would at least detect that the two writes were concurrent?
Scenario 3 — Spanner's deliberate wait Google Spanner gives you globally consistent transactions with real-time ordering across continents. To do it, every commit waits a few milliseconds on purpose before acknowledging — deliberately making itself slower. What is it waiting for, and why does waiting make the system more correct rather than just slower?

Scenario 1 is why wall-clock time lies. Scenario 2 is why you need vector clocks to detect concurrency. Scenario 3 is TrueTime and commit-wait. Each maps to an idea below.

The five ideas to understand

Theory, not knobs — ideas to hold correctly, each guarded by a misconception.

Idea 1 — Physical wall-clock time lies

Two failure modes make wall-clock timestamps unsafe for ordering. Clock skew: no two machines' clocks agree exactly — they drift apart and NTP only resyncs them periodically, to within milliseconds at best. Non-monotonicity: NTP corrections and leap seconds can step the clock backwards, so a later reading can be smaller than an earlier one. Code that subtracts two timestamps and assumes the result is non-negative is a bug waiting for a leap second.

The misconception to avoid — "timestamps order events" This is Scenario 1. The 2012/2016 leap-second outages all came from one assumption: time only moves forward, at a steady rate. It doesn't. For elapsed-time measurement, use a monotonic clock (a counter that never goes backwards, e.g. CLOCK_MONOTONIC) — not the wall clock. For ordering events across machines, don't use physical time at all unless you have bounded-error clocks; use logical clocks.

Idea 2 — Happens-before and Lamport logical clocks

Leslie Lamport (1978) defined the happens-before relation (→): A → B if A and B are on the same node with A first, or A is a send and B is its receive, or by transitivity. If neither A → B nor B → A, they are concurrent. A Lamport clock is one integer per node: increment on each event; on send, attach your counter; on receive, set counter = max(local, received) + 1. This guarantees: if A → B then timestamp(A) < timestamp(B).

The misconception to avoid — "a smaller Lamport timestamp means it really happened first" The implication only runs one way. A → B implies LC(A) < LC(B), but LC(A) < LC(B) does NOT imply A happened before B — they might be concurrent and just got different numbers. Lamport clocks give a total order consistent with causality (useful for breaking ties deterministically) but they cannot tell you whether two events were concurrent. For that you need vector clocks.

Idea 3 — Vector clocks detect concurrency

A vector clock keeps one counter per node: [a, b, c] for nodes A, B, C. Each node increments its own slot on an event; on receive, take the element-wise max then increment your own slot. Now you can compare two vectors: if every element of V1 ≤ V2 (and at least one is strictly less), then V1 happened-before V2; if neither dominates the other, the events are concurrent — a genuine conflict. This is exactly what AP stores need to flag write conflicts instead of silently dropping one.

The misconception to avoid — "vector clocks are free to use everywhere" This is the fix for Scenario 2 — vector clocks detect the concurrent writes that last-write-wins silently loses. But the cost is real: the vector grows with the number of nodes that have ever written, so metadata can balloon in large or churning clusters. Production systems prune them (per-client version vectors, dotted version vectors) to bound the size. They detect conflicts; they don't resolve them — you still need an app-level merge or a CRDT.

Idea 4 — Hybrid Logical Clocks (HLC)

Logical clocks lose real time; physical clocks lie. HLC (Kulkarni et al., 2014) combines them: a timestamp is a pair (physical_time, logical_counter). It tracks physical time closely (so timestamps roughly match the wall clock and are human-meaningful) and preserves happens-before (so it never goes backwards causally). When physical time stalls or jumps back, the logical counter takes over to keep order monotonic.

The misconception to avoid — "you need atomic clocks like Spanner to get ordered timestamps" HLC gives you causality-respecting, roughly-wall-clock timestamps on commodity hardware with ordinary NTP — no GPS or atomic clocks. That's why CockroachDB and MongoDB use it: it's the practical middle ground between pure logical clocks (no real-time meaning) and TrueTime (needs special hardware). It does not give the tight error bounds TrueTime does, so it can't offer Spanner-grade external consistency without extra coordination.

Idea 5 — TrueTime and commit-wait

Google's TrueTime equips every datacenter with GPS receivers and atomic clocks, and instead of returning a single timestamp it returns an interval [earliest, latest] guaranteed to contain the true time — with the uncertainty (ε) kept to single-digit milliseconds. Spanner uses it for commit-wait: after picking a commit timestamp, it waits out the uncertainty interval before acknowledging, guaranteeing that any transaction that starts later gets a strictly greater timestamp. That's how it achieves global external consistency.

The misconception to avoid — "TrueTime is exact, perfectly-synced time" This is Scenario 3. TrueTime is not exact — it's honest about being uncertain, returning a bounded interval rather than a point. The cleverness is in the wait: by deliberately pausing for the uncertainty window, Spanner ensures no two transactions can disagree about order. The "slower" commit-wait is what buys the correctness — you trade a few milliseconds of latency for globally consistent ordering. Shrinking ε (better clocks) directly shrinks that wait.

The rule in one lineFor elapsed time use a monotonic clock; for ordering across machines never trust the wall clock — use logical clocks (Lamport to order, vector to detect concurrency), HLC when you want timestamps that also look like real time, and TrueTime only when you genuinely need global real-time order and can afford the hardware.

Famous implementations

Clocks made concrete. Same facets — the clock used, what it enables, and the failure to watch.

Vector clocks — Dynamo / Riak

  • Clock used. Version vectors (vector clocks) per key.
  • What it enables. Detecting concurrent writes so the system can return both siblings to the client (or a resolver) instead of silently picking one.
  • The failure to watch. Vector size grows with writers; Riak uses dotted version vectors to bound it, and unresolved siblings can pile up if the app never merges them.

HLC — CockroachDB / MongoDB

  • Clock used. Hybrid Logical Clocks — (physical, logical) pairs.
  • What it enables. Causally-correct, roughly-wall-clock timestamps on ordinary cloud hardware with NTP — ordering transactions without GPS.
  • The failure to watch. No tight ε bound like TrueTime, so they need extra mechanisms (uncertainty intervals + retries, or read refreshes) to approach strict serializability; large clock skew degrades performance.

TrueTime — Spanner

  • Clock used. TrueTime — GPS + atomic clocks returning bounded intervals.
  • What it enables. Global external consistency (strict serializability) via commit-wait, across continents.
  • The failure to watch. Requires Google's clock infrastructure; commit-wait adds latency proportional to clock uncertainty — you pay milliseconds per commit for the guarantee, and a clock-sync degradation widens ε and slows everything.

Build it yourself

Clocks click when you trace them by hand. Work each before opening the approach.

Compute Lamport timestamps across two nodes

Node A does events a1, a2, then sends a message to B. B has done b1, then receives A's message as b2. Assign Lamport timestamps and state which events are ordered and which are concurrent.

Reveal the approach

A: a1=1, a2=2, send carries 2. B: b1=1; on receiving (with stamp 2), b2 = max(1,2)+1 = 3. So LC: a1=1, a2=2, b1=1, b2=3. Ordered: a1→a2→(send)→b2, and b1→b2. Concurrent: a1 and b1 (both =1, no causal path) — and a2 vs b1 are also concurrent even though LC(b1)=1 < LC(a2)=2. That last pair is the trap: a smaller Lamport timestamp did not mean it happened first. To know a2 and b1 are concurrent you'd need vector clocks.

Detect a conflict with vector clocks

Three nodes A,B,C. A writes (vector [1,0,0]), B independently writes [0,1,0]. Are these concurrent? Now A's write propagates to B, which writes again. Show the new vector and why it's no longer a conflict with A's.

Reveal the approach

[1,0,0] vs [0,1,0]: neither dominates (A's 1st slot is bigger, B's 2nd slot is bigger) → concurrent — a conflict the system must surface. After B receives A's write, B merges (element-wise max → [1,1,0]) then increments its own slot → [1,2,0]. Now [1,0,0] ≤ [1,2,0] element-wise → A's write happened-before B's latest — no conflict, B's value supersedes. The dominance test is the whole mechanism: comparable → ordered; incomparable → concurrent.

Design-it-yourself: the open-ended ones

No single right answer — reason it out, then follow the link.

  • Replace last-write-wins. Your key-value store loses writes under LWW. Redesign conflict handling using vector clocks to detect concurrency, then a merge strategy that doesn't drop data. → CRDTs, Replication.
  • Order a global event log without TrueTime. You need a near-real-time, causally-correct ordering of events across regions on commodity hardware. Design it with HLC and explain where it falls short of strict serializability. → Consistency Models.
  • Make a leap second a non-event. Audit a service for code that assumes time only moves forward. Decide where to use a monotonic clock and how to handle a backwards step. → Resilience Patterns.
